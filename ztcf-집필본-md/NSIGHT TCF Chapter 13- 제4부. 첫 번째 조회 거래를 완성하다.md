# 제4부. 첫 번째 조회 거래를 완성하다

## 도입 전 안내말

제1부에서는 NSIGHT TCF의 전체 구조와 거래 처리 흐름을 이해했다.

제2부에서는 개발환경을 준비하고 소스를 빌드·실행·디버깅했다.

제3부에서는 기존 프로그램을 정방향과 역방향으로 추적하고, 변경하기 전에 영향 범위를 분석하는 방법을 배웠다.

제4부에서는 이제 실제로 새로운 거래를 설계하고 구현한다.

다만 코드를 바로 작성하지 않는다.

업무 담당자가 요구사항을 설명한다.
↓
개발자가 화면부터 만든다.
↓
필요한 필드를 DTO에 추가한다.
↓
Service와 Mapper를 작성한다.
↓
화면은 동작한다.
↓
업무 규칙·권한·오류·운영 기준이 뒤늦게 발견된다.
↓
프로그램과 설계서를 다시 수정한다.

이 방식은 개발 시작은 빨라 보이지만, 실제 완료 시점은 늦어진다.

조회 기능 하나를 개발하더라도 다음 사항이 먼저 결정되어야 한다.

누가 요청하는가?

어떤 조건으로 조회하는가?

무엇을 반환하는가?

조회 결과가 없다는 것은 무슨 뜻인가?

누가 이 데이터를 볼 수 있는가?

개인정보를 어떻게 표시하는가?

어느 업무가 데이터를 소유하는가?

어떤 ServiceId로 실행하는가?

몇 초 안에 끝나야 하는가?

어떤 오류를 화면에 반환하는가?

운영에서 어떤 로그로 찾을 수 있는가?

무엇을 테스트해야 완료인가?

제4부에서는 가장 단순한 조회 거래 한 건을 기준으로 다음 과정을 수행한다.

요구사항 구조화
↓
요청·응답 DTO 설계
↓
입력 검증
↓
Handler·Facade·Service·Rule 설계
↓
DAO·Mapper·SQL 구현
↓
ServiceId·OM 등록
↓
로컬 거래 실행
↓
정상·경계·실패 검증

NSIGHT 프로젝트의 기능 요구사항은 단순한 화면 기능 목록이 아니라 업무요구사항 ID → 유스케이스 → 화면·이벤트 → ServiceId → 거래코드 → 프로그램 → SQL·DB → 테스트케이스의 연결 구조로 관리해야 한다.

# 제13장. 요구사항을 개발 단위로 바꾸기

## 이 장을 시작하며

업무 담당자는 일반적으로 Java 클래스나 ServiceId 형식으로 요구사항을 설명하지 않는다.

다음과 같이 말한다.

고객번호를 입력하면 고객의 기본정보와 등급을 보여 주세요.

우리 지점에서 관리하는 고객만 조회되어야 합니다.

고객이 없을 때는 오류 메시지를 보여 주세요.

VIP 고객은 화면에서 눈에 잘 띄게 표시해 주세요.

조회가 너무 오래 걸리지 않게 해 주세요.

고객 이름은 권한에 따라 마스킹해야 합니다.

이 요구사항을 그대로 개발자에게 전달하면 사람마다 다른 프로그램을 만들 수 있다.

한 개발자는 고객이 없을 때 빈 객체를 반환할 수 있다.

다른 개발자는 업무 오류를 발생시킬 수 있다.

또 다른 개발자는 모든 지점의 고객을 조회할 수 있다.

같은 문장
→ 서로 다른 해석
→ 서로 다른 구현
→ 서로 다른 테스트
→ 운영 결과 불일치

따라서 개발 전 요구사항을 다음 수준으로 구조화해야 한다.

자연어 요구사항
↓
업무 목적
↓
사용자·권한
↓
선행조건
↓
입력
↓
업무 규칙
↓
출력
↓
오류·예외
↓
데이터
↓
비기능 조건
↓
거래 식별자
↓
완료 기준

원본 제13장은 요구사항을 업무 규칙·거래 설계·구현·테스트·추적성으로 연결하며, 코드를 작성하기 전에 사실·가정·결정·미결사항을 구분하도록 정의한다.

## 핵심 관점

요구사항 분석의 결과는
긴 설명문이 아니다.

서로 다른 사람이 읽어도
같은 입력,
같은 업무 판단,
같은 결과,
같은 실패,
같은 완료조건을 이해할 수 있는
실행 가능한 거래 정의다.

# 13.1 원시 요구사항 구조화

## 13.1.1 원시 요구사항이란

원시 요구사항은 아직 개발 기준으로 구조화되지 않은 최초 입력자료다.

대표 출처:

업무 인터뷰
회의록
화면 스케치
엑셀 업무 목록
기존 시스템 화면
법규·내규
장애 개선 요청
사용자 불편사항
DB 테이블 정의
보고서
외부 연계 문서

원시 요구사항에는 다음 내용이 섞여 있을 수 있다.

사실
희망
추측
기존 관행
설계 결정
기술 제약
개인 의견
미확정 사항

따라서 문장을 그대로 프로그램으로 옮겨서는 안 된다.

## 13.1.2 원시 요구사항 예

고객 종합정보 화면에서 고객번호를 입력하고 조회하면
고객 기본정보, 고객등급과 보유상품을 한 번에 보여 준다.

본인 지점 고객만 볼 수 있어야 한다.

VIP 고객은 강조해서 보여 주고,
고객이 없으면 안내 메시지를 표시한다.

조회가 느리면 안 된다.

이 문장에는 다음 미확정 사항이 있다.

| 영역 | 미확정 질문 |
| --- | --- |
| 사용자 | 상담직원만 사용하는가 |
| 고객번호 | 필수인가, 다른 조건도 가능한가 |
| 지점 권한 | 소속 지점만인가, 관할 지점도 가능한가 |
| 고객 기본정보 | 정확히 어떤 필드인가 |
| 고객등급 | 현재 등급인가, 기준일 등급인가 |
| 보유상품 | 전체인가, 활성상품만인가 |
| VIP 강조 | 서버 규칙인가, 화면 표현인가 |
| 고객 없음 | 빈 결과인가, 업무 오류인가 |
| 느리면 안 됨 | 목표 응답시간은 몇 초인가 |
| 감사 | 고객조회 기록을 남겨야 하는가 |
| 마스킹 | 고객명·연락처 표시 기준은 무엇인가 |

## 13.1.3 요구사항 구조화 기본 문장

다음 형식을 사용하면 모호성을 줄일 수 있다.

\[사용자\]가
\[선행조건\]에서
\[입력\]을 제공하면
시스템은 \[업무 규칙\]을 적용하여
\[처리 대상\]을 처리하고
\[출력\]을 반환해야 한다.

\[실패 조건\]인 경우
\[오류 결과\]를 반환해야 하며,
\[비기능 조건\]을 만족해야 한다.

예:

고객조회 기능권한을 가진 영업점 상담직원이
유효한 JWT 인증 상태에서 고객번호를 입력하면,
시스템은 사용자의 지점 데이터권한과 고객 상태를 확인한 후
고객 기본정보·현재 등급·활성 보유상품 요약을 반환해야 한다.

고객번호가 없거나 형식이 잘못된 경우 입력 오류를 반환하고,
고객이 존재하지 않는 경우 고객 미존재 업무 오류를 반환한다.

개인정보는 권한에 따라 마스킹하며,
거래는 p95 3초 이내에 완료되고 고객조회 감사로그를 기록해야 한다.

## 13.1.4 업무 목적과 구현 방법을 구분한다

### 업무 목적

상담직원이 고객 상담 전에 핵심 정보를 한 화면에서 확인한다.

### 구현 방법

SV\_CUSTOMER와 SV\_PRODUCT 테이블을 JOIN한다.

업무 목적은 비교적 안정적이지만 구현 방법은 변경될 수 있다.

현재 구현
→ DB Join

향후 구현
→ Read Model

다른 선택
→ 다른 WAR ServiceId 호출

요구사항 문서에는 업무 목적과 결과를 우선 기록하고, 구현 방식은 설계 결정으로 분리한다.

## 13.1.8 선행조건

사용자가 로그인되어 있어야 한다.

고객조회 기능권한이 있어야 한다.

사용자의 소속 지점이 유효해야 한다.

ServiceId가 OM Catalog에서 사용 상태여야 한다.

거래가 운영상 차단되지 않아야 한다.

고객 데이터가 기준시점까지 적재되어 있어야 한다.

선행조건이 충족되지 않으면 Handler 이전 STF에서 차단할지, 업무 Service에서 오류를 반환할지 책임을 구분해야 한다.

# 13.2 입력·출력·업무규칙 정의

## 13.2.1 입력은 화면 필드 목록이 아니다

입력은 거래가 업무 판단을 위해 필요로 하는 정보다.

화면에는 표시되지만 서버로 보낼 필요가 없는 값이 있을 수 있다.

반대로 화면에는 없지만 인증 Context에서 가져와야 하는 값도 있다.

| 입력 유형 | 예 | 출처 |
| --- | --- | --- |
| 사용자 입력 | 고객번호 | 화면 |
| 선택 조건 | 기준일 | 화면 |
| 인증 정보 | 사용자 ID | JWT Context |
| 조직 정보 | 지점코드 | 인증 Context |
| 채널 정보 | WEBTOP | 표준 Header |
| 추적 정보 | GUID·TraceId | TCF |
| 운영 정책 | Timeout | OM |

클라이언트가 보낸 userId와 branchId를 그대로 신뢰하지 않는다.

| 지점 권한 | 업무 데이터 범위 | Service·Rule |
| 중복 | DB 상태 필요 | Service·Rule |
| 상태 전이 | 현재 상태 필요 | Service·Rule |

원본은 필수·길이·형식 검증과 중복·상태·권한 같은 업무 규칙이 서로 다른 실패 의미와 처리 계층을 가진다고 설명한다.

# 13.3 결정·가정·미결사항 구분

## 13.3.1 왜 구분해야 하는가

회의에서 나온 모든 문장을 확정 요구사항으로 취급하면 안 된다.

“아마 3초면 될 것 같습니다.”

“기존에도 그렇게 했던 것 같습니다.”

“일단 고객번호만 하면 되지 않을까요?”

“운영팀이 알아서 설정할 겁니다.”

이 문장들은 결정이 아닐 수 있다.

구분하지 않으면 다음 문제가 발생한다.

가정을 사실로 구현

미결사항을 개발자가 임의 결정

결정 근거 소실

팀별 다른 해석

테스트 기준 불일치

## 13.4.11 수용 기준

### 정상

다른 도메인의 테이블을 소유자 확인 없이 직접 조회한다.

사용자가 보낸 userId·branchId를 신뢰한다.

권한과 마스킹을 UI에서만 처리한다.

“빠르게”를 응답시간 요구사항으로 사용한다.

Timeout을 개발 완료 후 운영팀이 정하게 한다.

ServiceId만 만들고 OM Catalog를 등록하지 않는다.

정상 시나리오만 정의하고 오류 시나리오를 생략한다.

코드 작성 완료를 개발 완료로 처리한다.

요구사항과 테스트 ID를 연결하지 않는다.

미결사항의 책임자와 결정기한을 지정하지 않는다.

# 시사점

## 핵심 아키텍처 판단

첫째, 자연어 요구사항은 아직 개발 단위가 아니다.

자연어
→ 구조화
→ 거래 설계
→ 실행 계약

과정을 거쳐야 한다.

둘째, 입력·출력만 정의해서는 부족하다.

입력
\+ 업무 규칙
\+ 오류
\+ 권한
\+ 성능
\+ 운영

이 함께 정의되어야 한다.

셋째, 가정과 미결사항을 숨기면 코드가 임의의 정책이 된다.

넷째, 하나의 개발 단위는 Java 클래스가 아니라 요구사항부터 운영 증적까지 연결된 거래다.

다섯째, 완료 기준은 “화면이 동작한다”가 아니라 정상·경계·실패·운영 추적이 증명되는 것이다.

## 주요 위험

| 위험 | 결과 |
| --- | --- |
| 자연어 그대로 구현 | 개발자별 해석 차이 |
| 업무 목적·구현 혼용 | 기술 변경 시 요구사항 왜곡 |
| 입력만 정의 | 결과·오류 불명확 |
| 가정 미표시 | 임시값 영구정책화 |
| 미결 책임자 없음 | 개발 지연·임의결정 |
| 거래 과대 | 복잡한 트랜잭션·권한 |
| 거래 과소 | 호출·운영 복잡도 증가 |
| 데이터 소유권 미확정 | 도메인 강결합 |
| 성능 목표 없음 | 구조 대안 판단 불가 |

| OM 등록 누락 | 운영 실행 실패 |
| 완료조건 부재 | 끝나지 않는 개발 |

## 우선 보완 과제

1.  요구사항 표준 템플릿을 공식화한다.
2.  사실·가정·결정·미결사항 대장을 운영한다.
3.  요구사항 ID에서 ServiceId까지 자동 연결한다.
4.  ServiceId 명명·중복 검사를 자동화한다.
5.  데이터 소유권 승인 절차를 추가한다.
6.  요청·응답 Schema를 계약 저장소로 관리한다.
7.  업무 규칙과 오류코드를 연결한다.
8.  비기능 요구사항을 거래별로 등록한다.
9.  OM Catalog 등록을 개발 완료조건에 포함한다.
10.  RTM 누락검사를 CI/CD Gate에 추가한다.
11.  Pull Request에 요구사항·영향·테스트를 의무화한다.
12.  Blocker 미결사항이 있는 거래의 개발 착수를 차단한다.
