**집필 설계서 및 원고 골격 v0.1**

대상 독자 · NSIGHT TCF를 처음 접하는 Java·Spring 개발자

# 1\. 집필 목표와 편집 원칙

이 책은 기존 자료를 축약한 매뉴얼이 아니라, 초보 개발자가 화면 요청 한 건을 따라가며 TCF의 구조·개발·보안·운영을 단계적으로 이해하도록 재구성한 학습서다.

-   ‘왜 필요한가’를 먼저 설명하고 용어와 구현을 뒤에 배치한다.
-   한 장에서는 하나의 학습 질문만 해결한다.
-   개념은 ztcfbook-m, 표준은 ztcfbook·znsight-man, 기술 사실은 zman과 실제 설정 자료로 교차 검증한다.
-   설계와 구현이 다르면 차이를 숨기지 않고 ‘설계 의도 / 현행 구현 / 적용 판단’으로 구분한다.
-   모든 실습은 고객 종합정보 조회와 고객 메모 CRUD라는 하나의 사례로 연결한다.

# 2\. 독자 경험 설계

| **독자** | **출발점** | **책을 통해 도달할 수준** |
| --- | --- | --- |
| 완전 초보 | 웹·Java 용어가 낯섦 | 전체 요청 흐름을 설명하고 예제를 실행 |
| 업무 개발자 | Spring 경험, TCF는 처음 | ServiceId와 6계층으로 CRUD 거래 구현 |
| 운영 담당자 | 배포·로그 경험 | OM·Timeout·자원 지표로 장애를 진단 |
| 예비 아키텍트 | 개발 경험 보유 | 책임 경계와 Architecture Gate를 판단 |

# 3\. 자료 체계와 적용 우선순위

참고자료는 서로 목적과 상세 수준이 다르므로 동일한 권위로 합치지 않는다. 집필 시 다음 우선순위를 적용한다.

| **우선순위** | **자료** | **집필에서의 역할** |
| --- | --- | --- |
| 1 | 목차안 DOCX | 61개 장과 14개 부록의 최종 구조 |
| 2 | ztcfbook-m | 초보자 눈높이, 비유, 실습 진입 |
| 3 | ztcfbook / ztcfbook-h | 개발 표준과 아키텍처 판단 근거 |
| 4 | zman / znsight-man / znsight-guide-word | 설계-코드 대조, 절차, 체크리스트 |
| 5 | config / capacity / 구축방법론 | 설정값, 운영, 용량, 프로젝트 수행 기준 |

# 5\. 장별 공통 집필 포맷

| **순서** | **구성 요소** | **작성 기준** |
| --- | --- | --- |
| 1 | 이번 장에서 해결할 질문 | 독자가 장을 읽어야 하는 이유를 한 문장으로 제시 |
| 2 | 먼저 그림으로 보기 | 요청 흐름·책임 경계를 한 장의 그림으로 표현 |
| 3 | 쉬운 설명 | 비유 1개와 핵심 용어 3~7개 |
| 4 | NSIGHT에서는 | 일반 개념을 실제 TCF 구조에 연결 |
| 5 | 코드 또는 설정 따라 하기 | 최소 실행 예제와 예상 결과 |
| 6 | 자주 하는 실수 | 금지 예시와 실패 증상 |
| 7 | 직접 해보기 | 10~30분 단위 실습 |
| 8 | 확인 문제·체크리스트 | 이해 확인과 현장 적용 점검 |
| 9 | 근거 자료 | 장 말미에 출처와 적용 기준 기록 |

# 7\. 제0부 원고 초안

## 0장. 이 책을 읽기 전에

### 0.1 이 책을 만든 이유

기업의 업무 시스템을 처음 접한 개발자는 코드보다 먼저 ‘전체가 어떻게 연결되는가’에서 막힌다. 화면, 웹 서버, 업무 WAR, 프레임워크, 데이터베이스, 운영관리 화면이 각각 무엇을 하는지 모르면 파일 하나를 수정하는 일도 위험한 추측이 된다.

NSIGHT TCF는 온라인 거래를 같은 방식으로 처리하기 위한 프레임워크다. 일반적인 REST 예제처럼 기능마다 URL과 Controller를 늘리는 방식이 아니라, 공통 진입점과 serviceId를 사용해 실행할 업무를 식별한다. 요청은 STF 전처리, Dispatcher, Handler와 업무 6계층, ETF 후처리를 통과하며 보안·통제·로그·Timeout이 공통 규칙으로 적용된다.

이 책의 목적은 이 구조를 외우게 하는 것이 아니다. 고객 종합정보 조회라는 하나의 거래를 설계하고 구현하고 배포한 뒤 운영 화면과 로그에서 확인하는 경험을 통해, 독자가 새로운 거래도 스스로 분석할 수 있게 만드는 것이다.

### 0.2 이 책에서 만들 시스템

실습 시스템은 고객 종합정보 조회를 중심으로 시작한다. 먼저 고객 목록과 상세 정보를 조회하고, 고객 메모를 등록·변경·삭제한다. 이후 상품정보 연계, JWT 검증, 데이터 권한, 개인정보 마스킹, Timeout과 표준 오류 응답을 단계적으로 추가한다.

| **단계** | **기능** | **학습 목표** |
| --- | --- | --- |
| 1 | 고객 목록·상세 조회 | 표준 요청, ServiceId, Handler~Mapper 흐름 |
| 2 | 고객 메모 CRUD | 트랜잭션, Validation, 업무 오류 |
| 3 | 상품정보 연계 | WAR 간 호출, Timeout, 오류 전파 |
| 4 | 보안 적용 | JWT, 권한, 개인정보 마스킹, 감사로그 |
| 5 | 배포·운영 | WAR, Tomcat, OM 등록, 로그와 장애 진단 |

### 0.3 대상 독자와 선행지식

주 독자는 Java와 Spring Boot를 조금 사용해 보았지만 NSIGHT TCF는 처음인 개발자다. Java 클래스와 메서드, JSON, SQL의 기본 문법을 알면 좋지만, 익숙하지 않은 독자를 위해 제1부에서 필요한 기초를 다시 설명한다.

-   Spring의 @RestController 예제는 알지만 엔터프라이즈 거래 프레임워크는 처음인 개발자
-   업무 화면과 설계서는 보았지만 전체 실행 흐름을 설명하기 어려운 개발자
-   운영·테스트·아키텍처까지 시야를 넓히려는 업무 개발자

### 0.4 이 책을 읽는 핵심 관점

기능을 찾을 때 URL부터 찾지 않는다. 먼저 업무코드, serviceId, 거래코드, Handler를 확인한다. 처리 중 문제가 생기면 한 클래스만 보지 않고 STF부터 ETF까지 거래의 전체 경로와 OM 기준정보, Timeout, 로그를 함께 본다.

| **일반적인 첫 질문** | **TCF에서 바꿔야 할 질문** |
| --- | --- |
| 어떤 URL을 추가하지? | 어떤 serviceId와 거래코드로 식별하지? |
| Controller에 로직을 넣을까? | Handler와 6계층의 어느 책임에 둘까? |
| 오류를 catch해서 문자열로 보낼까? | 공통 오류 매핑과 표준 응답을 어떻게 적용할까? |
| 느리면 Thread를 늘릴까? | 어느 구간에서 대기하며 병목 자원은 무엇인가? |

# 9\. 집필 시 사실 검증 원칙

-   클래스명·serviceId·포트·설정 키는 현행 코드나 설정 파일과 대조한다.
-   설계서의 목표 구조와 현재 구현을 구분해 표시한다.
-   용량 수치는 가정·산식·적용 범위를 함께 제시하며 단일 정답처럼 쓰지 않는다.
-   보안 정보와 실제 운영 비밀값은 예제에 포함하지 않는다.
-   각 장 말미에 근거 자료와 검증 상태를 기록한다.
