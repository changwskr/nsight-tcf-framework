# **제14부. 아키텍처 설계·검토·의사결정·품질 Gate 실전**

# **1\. 도입 전 안내말**

제13부에서는 tcf-core, tcf-web, tcf-jwt, tcf-gateway, tcf-cache, tcf-eai, tcf-batch, tcf-om 등 TCF 공통모듈의 역할과 책임경계를 배웠습니다.

이제 새로운 NSIGHT 프로젝트를 처음 시작한다고 가정해 보겠습니다.

아직 다음 내용이 확정되지 않았습니다.

업무 WAR를 몇 개로 나눌 것인가?

하나의 Tomcat에 여러 WAR를 배포할 것인가?

업무그룹별로 Tomcat을 분리할 것인가?

Gateway를 사용할 것인가?

Session을 유지할 것인가, JWT 중심으로 전환할 것인가?

DB Pool은 업무별로 얼마를 설정할 것인가?

사용자 36,000명을 어떻게 수용할 것인가?

장애가 발생하면 어느 단위로 격리할 것인가?

OM에서 무엇을 관측하고 통제할 것인가?

배포와 Rollback은 어떻게 할 것인가?

이때 개발자는 곧바로 다음 작업을 시작하고 싶어 할 수 있습니다.

프로젝트 생성

Package 생성

Controller 작성

DB Table 생성

화면 개발

그러나 아키텍처 전제가 정리되지 않은 상태에서 개발을 시작하면 이후 다음 문제가 발생합니다.

업무별 구조가 서로 달라진다.

WAR 분리기준이 계속 변경된다.

DB Pool 합계가 DB 허용량을 초과한다.

Session과 JWT가 혼합되어 인증 흐름이 불명확해진다.

운영 모니터링 기능이 뒤늦게 추가된다.

배포 시 다른 업무까지 중단된다.

성능시험 직전에 용량 부족이 발견된다.

장애가 발생해도 책임 주체가 불명확하다.

아키텍처는 개발 이후에 문서로 정리하는 것이 아닙니다.

아키텍처
\= 프로젝트가 어떤 원칙과 구조로 만들어지고
어떤 조건을 만족해야 하며
어떤 위험을 어떻게 통제할지 결정하는 활동

좋은 아키텍처 설계는 다음 질문에 답합니다.

왜 이 구조를 선택했는가?

다른 대안은 무엇이었는가?

어떤 요구사항을 만족하는가?

어떤 제약을 받아들였는가?

어떤 위험이 남아 있는가?

누가 무엇을 책임지는가?

언제 무엇을 검증하는가?

조건을 만족하지 못하면 어떻게 조치하는가?

제14부에서는 신규 프로젝트 아키텍처를 다음 순서로 수립합니다.

프로젝트 목표 확인

→ 현행 구조 분석

→ 문제 정의

→ 기능·비기능 요구사항

→ 제약조건

→ 설계 원칙

→ 대안 도출

→ 비교·의사결정

→ 목표 아키텍처

→ 용량·보안·운영 설계

→ Architecture Gate

→ 운영전환·변경관리

제14부의 핵심 원칙은 다음과 같습니다.

구성도보다 먼저 문제를 정의한다.

요구사항과 제약조건을 구분한다.

비기능 요구사항은 수치로 표현한다.

대안 없이 결론만 제시하지 않는다.

의사결정에는 근거와 포기한 것을 기록한다.

검증되지 않은 아키텍처는 확정된 아키텍처가 아니다.

Architecture Gate는 문서 검토가 아니라
진입·진출 조건을 확인하는 품질 통제다.

# **2\. 제14부 개요**

## **2.1 목적**

제14부의 목적은 초보 개발자와 아키텍트가 신규 NSIGHT 프로젝트의 기술구조를 단계적으로 수립하고, 대안을 객관적으로 비교하며, 단계별 Architecture Gate를 통해 설계와 구현의 정합성을 검증하도록 돕는 것입니다.

학습을 마친 뒤에는 다음 작업을 수행할 수 있어야 합니다.

1.  프로젝트 목표와 아키텍처 범위를 정의한다.
2.  현행 구조와 문제점을 사실·증상·원인으로 구분한다.
3.  기능 요구사항과 비기능 요구사항을 구분한다.
4.  사용자 수·TPS·응답시간·가용성 요구를 수치화한다.
5.  요구사항과 제약조건을 구분한다.
6.  설계 원칙을 Architecture Principle로 정의한다.
7.  복수의 아키텍처 대안을 도출한다.
8.  대안을 비용·운영·성능·보안·확장성으로 비교한다.
9.  ADR을 작성하여 의사결정을 기록한다.
10.  목표 애플리케이션·배포·데이터·보안 아키텍처를 작성한다.
11.  Tomcat·WAR·Thread·DB Pool·JVM 용량을 정합성 있게 검토한다.
12.  정상·Timeout·장애·Rollback 흐름을 설계한다.
13.  단계별 Architecture Gate를 정의한다.
14.  아키텍처 검토회의를 의사결정 중심으로 운영한다.
15.  아키텍처 예외와 잔여 위험을 관리한다.
16.  운영전환과 폐기까지 포함한 생명주기를 설계한다.

## **2.3 대상 독자**

-   신규 프로젝트 아키텍처를 처음 수립하는 개발자
-   구성도는 그릴 수 있지만 결정근거를 작성하기 어려운 개발자
-   기능 요구사항만 받고 개발을 시작하는 개발자
-   서버 수와 Thread·DB Pool 관계가 어려운 개발자
-   여러 아키텍처 대안을 비교해야 하는 아키텍트
-   Architecture Review 회의를 준비하는 담당자
-   PMO에 설계 근거와 검증결과를 제출해야 하는 담당자
-   기존 시스템을 NSIGHT TCF 구조로 전환해야 하는 개발자

## **2.4 선행조건**

다음 내용을 이해하고 있어야 합니다.

TCF는 ServiceId 기반 거래 Framework다.

업무기능은 Handler 이하 계층에서 구현한다.

업무 WAR는 도메인과 배포경계를 가진다.

Gateway는 인증과 Route를 담당한다.

OM은 운영정책과 관측을 담당한다.

Tomcat Thread와 DB Pool은 상호 연관된다.

JWT와 Session은 서로 다른 상태관리 방식이다.

CI/CD와 Rollback은 운영 아키텍처의 일부다.

# **제124장. 신규 프로젝트 아키텍처 시작하기**

학습 목표 | 124장. 신규 프로젝트 아키텍처 시작하기의 핵심 개념을 설명하고, 프로젝트에서 적용 위치와 검증 방법을 스스로 판단한다.

핵심 관점 | 근거 기반 의사결정: 설계안은 장점만 설명하지 말고 제약, 대안, 비용, 위험과 결정 근거를 함께 기록해야 합니다.

읽는 순서 | 먼저 용어와 책임 경계를 확인하고, 이어지는 정상 예시와 금지 예시를 비교한 뒤 실제 설정·코드·로그에서 근거를 찾아봅니다.

## **124.1 아키텍처 작업의 첫 질문**

첫 질문은 다음이 아닙니다.

서버를 몇 대 사용할 것인가?

첫 질문은 다음이어야 합니다.

이 프로젝트가 해결해야 하는 문제는 무엇인가?

예:

사용자 36,000명이 고객·마케팅 정보를 조회한다.

여러 업무 시스템의 데이터를 한 화면에서 제공한다.

업무별 기능을 독립 개발·배포할 수 있어야 한다.

운영자가 ServiceId와 SQL을 추적할 수 있어야 한다.

특정 업무 장애가 전체 시스템 장애로 확산되지 않아야 한다.

## **124.2 프로젝트 목표**

목표는 기술이 아니라 업무 결과 중심으로 작성합니다.

나쁜 예:

Spring Boot를 사용한다.

Tomcat을 사용한다.

JWT를 적용한다.

좋은 예:

영업점 사용자가 고객정보를 p95 3초 안에 조회한다.

업무기능을 ServiceId 기준으로 통제·감사한다.

업무별 장애와 배포 영향을 최소화한다.

운영자가 장애 원인을 10분 이내에 특정한다.

운영 중 무중단 또는 최소중단 배포를 지원한다.

## **124.4 범위 제외**

아키텍처 문서에는 하지 않을 것도 기록합니다.

예:

원장계 업무로직은 변경하지 않는다.

외부 SSO 서버 자체는 구축범위가 아니다.

전사 네트워크 재설계는 제외한다.

모든 Legacy 시스템을 MSA로 전환하지 않는다.

1차 구축에서는 Redis를 도입하지 않는다.

범위 제외가 없으면 이후 모든 문제를 프로젝트가 해결해야 하는 것으로 오해할 수 있습니다.

보안취약점이 있는 상태에서는 운영 Gate를 통과시키지 않는다.

직접 DB 접근과 관리 Endpoint를 통제한다.
