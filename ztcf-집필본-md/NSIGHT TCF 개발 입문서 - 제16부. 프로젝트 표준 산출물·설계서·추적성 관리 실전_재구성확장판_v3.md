# **제16부. 프로젝트 표준 산출물·설계서·추적성 관리 실전**

# **1\. 도입 전 안내말**

제15부에서는 운영 장애를 ServiceId, TraceId, Tomcat Thread, JVM, DB Pool, SQL과 외부연계 정보로 추적하고 복구하는 방법을 배웠습니다.

그런데 운영자가 이러한 정보를 연결하려면 개발 단계에서부터 다음 관계가 정리되어 있어야 합니다.

요구사항
→ 화면
→ 화면 이벤트
→ ServiceId
→ 거래코드
→ Handler
→ Facade
→ Service
→ Rule
→ DAO
→ Mapper
→ SQL
→ DB 객체
→ 테스트
→ OM 기준정보
→ 배포 버전

이 관계가 문서와 실제 소스에 일관되게 반영되지 않으면 다음과 같은 문제가 발생합니다.

요구사항에는 있지만 구현되지 않은 기능이 생긴다.

화면 버튼이 어느 ServiceId를 호출하는지 알 수 없다.

ServiceId는 존재하지만 처리하는 Handler가 없다.

프로그램 설계서의 클래스명과 실제 소스가 다르다.

Mapper XML에 존재하는 SQL이 설계서에 없다.

DB 컬럼이 변경되었지만 화면 영향이 분석되지 않는다.

테스트가 어떤 요구사항을 검증하는지 알 수 없다.

신규 ServiceId가 OM에 등록되지 않아 운영에서 차단된다.

폐기한 화면과 거래가 운영 설정에 계속 남아 있다.

산출물을 프로젝트 종료 시점에 한꺼번에 작성하면 실제 구현과 차이가 커지기 쉽습니다.

나쁜 흐름:

개발 완료
→ 테스트 완료
→ 운영 배포 직전
→ 설계서 작성

권장 흐름:

요구사항 정의
→ 설계서 초안
→ 구현
→ 자동 정합성 검증
→ 테스트 결과 반영
→ 운영정보 등록
→ 변경 이력 유지

산출물의 목적은 문서 수를 늘리는 것이 아닙니다.

산출물
\= 프로젝트 구성요소의 의미·책임·연결관계·변경이력을
공식적으로 관리하는 기준정보

좋은 산출물은 다음 질문에 바로 답할 수 있어야 합니다.

이 요구사항은 어느 화면에 구현되었는가?

이 화면 버튼은 어느 ServiceId를 호출하는가?

이 ServiceId는 어느 프로그램과 SQL을 실행하는가?

이 SQL은 어느 테이블을 조회하거나 변경하는가?

테이블 컬럼을 변경하면 어떤 화면이 영향을 받는가?

이 거래의 권한과 Timeout은 무엇인가?

어떤 테스트가 이 기능을 검증하는가?

운영에서 어느 Metric과 경보를 확인해야 하는가?

기능을 폐기하려면 무엇을 함께 제거해야 하는가?

제16부의 핵심 원칙은 다음과 같습니다.

산출물은 실제 구현보다 먼저 초안을 작성한다.

하나의 기준정보는 하나의 공식 소유자를 가진다.

같은 정보를 여러 문서에 중복 소유하지 않는다.

산출물 간 연결에는 공식 식별자를 사용한다.

문서와 소스의 정합성을 자동검증한다.

변경 시 수정 대상과 영향범위를 함께 기록한다.

운영 배포 후에도 산출물을 최신 상태로 유지한다.

# **2\. 제16부 개요**

## **2.1 목적**

제16부의 목적은 NSIGHT TCF 프로젝트에서 작성해야 할 표준 산출물의 구조와 책임을 정의하고, 요구사항부터 운영까지의 추적성을 지속적으로 유지하는 방법을 설명하는 것입니다.

학습을 마친 뒤에는 다음 작업을 수행할 수 있어야 합니다.

1.  프로젝트 산출물의 전체 계층을 설명한다.
2.  산출물별 목적과 공식 소유자를 구분한다.
3.  요구사항 정의서와 요구사항 추적 매트릭스를 작성한다.
4.  화면 설계서와 화면 이벤트 정의서를 작성한다.
5.  거래 설계서에 ServiceId와 정상·오류 흐름을 정의한다.
6.  프로그램 설계서에 계층별 책임과 메서드를 기록한다.
7.  SQL 설계서와 DB 설계서를 연결한다.
8.  인터페이스·Batch·파일·운영 설계서를 작성한다.
9.  화면에서 DB와 테스트까지 통합 추적성을 구성한다.
10.  산출물의 Version·상태·검토·승인 이력을 관리한다.
11.  변경 영향분석과 폐기 절차를 수행한다.
12.  문서와 소스·OM·DB의 정합성을 자동검증한다.
13.  Architecture Gate에서 산출물 완성도를 검증한다.
14.  AI 개발도구가 사용할 수 있는 구조화된 설계정보를 만든다.

# **제149장. 프로젝트 산출물 전체 체계**

학습 목표 | 149장. 프로젝트 산출물 전체 체계의 핵심 개념을 설명하고, 프로젝트에서 적용 위치와 검증 방법을 스스로 판단한다.

핵심 관점 | 근거 기반 의사결정: 설계안은 장점만 설명하지 말고 제약, 대안, 비용, 위험과 결정 근거를 함께 기록해야 합니다.

읽는 순서 | 먼저 용어와 책임 경계를 확인하고, 이어지는 정상 예시와 금지 예시를 비교한 뒤 실제 설정·코드·로그에서 근거를 찾아봅니다.

2\. 아키텍처 산출물

## **149.6 산출물 상태**

DRAFT
→ IN\_REVIEW
→ REVIEWED
→ APPROVED
→ BASELINED
→ SUPERSEDED
→ RETIRED

APPROVED와 BASELINED를 같은 상태로 단순화할 수도 있지만, 승인 후 기준본 확정 여부를 구분하면 변경관리가 명확해집니다.

# **7\. 정상 처리 흐름**

1\. 요구사항을 등록하고 ID를 발급한다.

2\. 업무규칙과 수용기준을 작성한다.

3\. 화면 ID와 이벤트 ID를 발급한다.

4\. 이벤트별 ServiceId를 정의한다.

5\. 거래코드·권한·Timeout을 연결한다.

6\. 계층별 프로그램을 설계한다.

7\. Mapper·SQL·DB 객체를 연결한다.

8\. 정상·오류·Timeout 흐름을 작성한다.

9\. 요구사항별 테스트를 정의한다.

10\. 산출물을 검토·승인한다.

11\. Source와 산출물 정합성을 자동검증한다.

12\. OM 기준정보와 Release를 등록한다.

13\. 운영 결과와 변경사항을 산출물에 반영한다.

# **21\. 마무리말**

제16부에서 가장 중요하게 기억해야 할 산출물 연결구조는 다음과 같습니다.

요구사항

→ 업무규칙

→ 화면 ID

→ 화면 이벤트 ID

→ ServiceId

→ 거래코드

→ Handler

→ Facade

→ Service

→ Rule

→ DAO

→ Mapper

→ SQL ID

→ DB 객체

→ Test ID

→ OM 정책

→ Release ID

→ 운영 Log·Metric·Trace

각 산출물의 핵심 역할은 다음과 같습니다.

요구사항 정의서
\= 무엇을 구현하고 어떻게 완료를 판단하는가

화면 설계서
\= 사용자가 무엇을 보고 어떤 이벤트를 실행하는가

거래 설계서
\= ServiceId가 어떤 조건과 흐름으로 실행되는가

프로그램 설계서
\= 각 계층이 어떤 책임으로 구현되는가

SQL 설계서
\= 어떤 데이터 처리를 어떤 성능으로 수행하는가

DB 설계서
\= 어느 업무가 어떤 데이터를 소유하는가

테스트 설계서
\= 무엇을 어떤 조건으로 검증하는가

운영 설계서
\= 어떻게 관측·통제·복구하는가

추적성 매트릭스
\= 이 모든 요소를 어떻게 연결하는가

초보 개발자와 산출물 담당자가 문서를 완료하기 전에 마지막으로 확인해야 할 질문은 다음과 같습니다.

이 문서의 공식 ID와 소유자는 누구인가?

이 정보의 공식 SoT는 어느 산출물인가?

요구사항에 수용기준과 검증방법이 있는가?

화면 이벤트가 ServiceId와 연결되는가?

ServiceId의 권한·Timeout·오류 흐름이 정의되는가?

프로그램 계층별 책임이 명확한가?

Mapper Method와 SQL ID가 일치하는가?

SQL이 사용하는 DB 객체와 예상건수가 기록되는가?

DB 객체의 소유 업무와 개인정보가 정의되는가?

각 요구사항에 검증 Test가 존재하는가?

OM 정책과 운영 Runbook이 연결되는가?

Source·DB·OM과 문서가 실제로 일치하는가?

변경 영향이 화면부터 운영까지 분석되었는가?

폐기 기능의 호출량과 데이터 보관을 확인했는가?

검토의견과 조건부 항목에 책임자와 기한이 있는가?

이 질문에 답할 수 있다면 산출물을 단순히 작성하는 수준을 넘어, 프로젝트의 요구사항·구현·운영 상태를 연결하고 변경과 장애에도 흔들리지 않는 기준정보 체계를 구축할 수 있습니다.
