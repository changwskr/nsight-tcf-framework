# 03. 애플리케이션 6계층 아키텍처

> **범위:** Handler → Facade → Service → Rule → DAO/Mapper 업무 계층 설계  
> **관련:** [zman/08-업무Handler개발.md](../zman/08-업무Handler개발.md) · [zdoc/어플리케이션계층.md](../zdoc/어플리케이션계층.md)

---

## 1. 개요

NSIGHT 업무 WAR·tcf-om·tcf-jwt는 **동일한 6계층 패키지 구조**를 따른다. TCF(STF/ETF) 바깥의 **업무 코드**는 이 계층 안에서만 작성한다.

### 1.1 계층 다이어그램

```mermaid
flowchart LR
    TCF[TCF Dispatcher] --> H[Handler<br/>entry/handler]
    H --> F[Facade<br/>entry/facade]
    F --> S[Service<br/>application/service]
    S --> R[Rule<br/>application/rule]
    R --> D[DAO<br/>persistence/dao]
    D --> M[Mapper<br/>persistence/mapper]
    M --> DB[(Database)]

    H -.->|금지| D
    S -.->|금지| M
    F -->|@Transactional| S
```

---

## 2. 계층별 책임

### 2.1 Handler (entry/handler)

| ✅ 담당 | ❌ 금지 |
|---------|---------|
| serviceId 분기 (switch) | Header·세션·거래통제 |
| Body → Map/DTO 변환 | SQL 실행 |
| Facade 호출 | StandardResponse 조립 |
| TransactionContext 전달 | 다른 WAR Java import |

**역할:** TCF ↔ 업무 **Adapter**

**명명:** `{Business}{Domain}Handler` — 예: `SvCustomerHandler`, `OmAuthHandler`

**등록:** `@Component` + `TransactionHandler` + `serviceIds()`

### 2.2 Facade (entry/facade)

| ✅ 담당 | ❌ 금지 |
|---------|---------|
| **@Transactional** 경계 | 복잡 비즈니스 규칙 |
| Service 오케스트레이션 | SQL |
| 다중 Service 조합 | HTTP 외부 호출 (→ client) |

**역할:** 트랜잭션 단위 **Use Case 진입점**

**명명:** `{Business}{Domain}Facade`

### 2.3 Service (application/service)

| ✅ 담당 | ❌ 금지 |
|---------|---------|
| 업무 흐름·시퀀스 | @Transactional (Facade에) |
| Rule 호출 | SQL 직접 |
| DAO 호출 | |

**명명:** `{Business}{Domain}Service` 또는 `{Feature}Service`

### 2.4 Rule (application/rule)

| ✅ 담당 | ❌ 금지 |
|---------|---------|
| 검증·계산·정책 | DB 접근 |
| 순수 비즈니스 규칙 | HTTP |

**명명:** `{Business}{Domain}Rule`

예: `EbUserRule` — 사용자 등록 유효성, `EbEventRule` — 이벤트 상태 전이

### 2.5 DAO (persistence/dao)

| ✅ 담당 | ❌ 금지 |
|---------|---------|
| Mapper 호출 래핑 | 비즈니스 규칙 |
| SQL ID ↔ Step Log | |

**명명:** `{Business}{Entity}Dao`

### 2.6 Mapper (persistence/mapper)

| ✅ 담당 | ❌ 금지 |
|---------|---------|
| MyBatis `@Mapper` | Java 로직 |
| SQL (XML/Annotation) | |

**명명:** [zdoc/mybatisNaming.md](../zdoc/mybatisNaming.md) 준수

## 3. 패키지 구조 (표준)

```
com.nh.nsight.marketing.{code}/
├── Nsight{Code}ServiceApplication.java   # extends NsightWarBootstrap
├── config/                               # Spring @Configuration
├── entry/
│   ├── handler/                          # TransactionHandler
│   └── facade/                           # @Transactional Facade
├── application/
│   ├── service/                          # 업무 Service
│   ├── rule/                             # 검증·규칙
│   └── scheduler/                        # @Scheduled (eb 등)
├── client/                               # tcf-eai, 외부 HTTP Client
├── persistence/
│   ├── dao/
│   └── mapper/
└── support/                              # enum, helper, properties
```

tcf-om, tcf-jwt도 동일 패턴 (`com.nh.nsight.marketing.om`, `com.nh.nsight.auth.jwt`)

---

## 4. 호출 규칙

### 4.1 허용 호출 방향

```
Handler → Facade → Service → Rule → DAO → Mapper
Handler → Facade → Service → client (tcf-eai)
Facade → Service (복수)
Service → Rule (복수)
Service → DAO (복수)
```

### 4.2 금지 호출

| 금지 | 대안 |
|------|------|
| Handler → DAO | Facade 경유 |
| Handler → Service (Facade 생략) | Facade 경유 |
| Service → Handler | 없음 |
| WAR A → WAR B Java class | tcf-eai HTTP |
| 업무 Controller for /online | tcf-web 공통 |

## 5. 신규 거래 추가 절차

### 5.1 체크리스트

| # | 작업 | 위치 |
|---|------|------|
| 1 | Service 메서드 | application/service |
| 2 | Rule (필요 시) | application/rule |
| 3 | DAO/Mapper (필요 시) | persistence |
| 4 | Facade @Transactional | entry/facade |
| 5 | Handler serviceIds + switch | entry/handler |
| 6 | **OM Catalog** | tcf-om data.sql |
| 7 | **거래통제** | TCF_TRANSACTION_CONTROL |
| 8 | **Timeout** | TCF_SERVICE_TIMEOUT_POLICY |
| 9 | 샘플 JSON | tcf-ui/sample-requests |
| 10 | tcf-ui 화면 (선택) | static/{code}/ |

### 5.2 serviceId 설계

```
{BC}.{Domain}.{action}
BC = businessCode (SV, IC, OM, …)
Domain = PascalCase 도메인 (Customer, Sample, Auth)
action = camelCase 동사 (selectSummary, inquiry, create)
```

---

## 6. 트랜잭션 경계

- **Facade**에 `@Transactional` — readOnly 또는 read-write
- Service·Rule·DAO는 Facade 트랜잭션에 참여
- WAR 간 호출(tcf-eai)은 **별도 HTTP 트랜잭션** — 분산 TX 없음 (Saga/Outbox는 eb-service 참고)

---

## 7. 페이징 아키텍처

SV 고객 목록 등 페이징 샘플:

- body: `pageNo`, `pageSize`, `sortField`
- Service: totalCount + list 조회
- Rule: pageSize 상한 검증

상세: [zdoc/sv-service-페이징-가이드.md](../zdoc/sv-service-페이징-가이드.md) · [docs/architecture/27-paging.md](../docs/architecture/27-paging.md)

## 8. 예외 처리

```java
// Service/Rule에서
throw new BusinessException(ErrorCode.CUSTOMER_NOT_FOUND, "고객 없음");
```

- BusinessException → ETF businessFail
- ErrorCode → OM_ERROR_CODE 연계
- Rule에서 검증 실패 시 BusinessException

상세: [zdoc/예외처리.md](../zdoc/예외처리.md)
