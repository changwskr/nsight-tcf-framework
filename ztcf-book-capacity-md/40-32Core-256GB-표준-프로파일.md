## 1. 문서 목적과 적용 전제

본 문서는 NSIGHT 마케팅플랫폼을 32 Core / 256GB IaaS VM 기준으로 구성할 때, 각 영역별로 실제 작업자가 적용해야 할 환경설정값, 설정 위치, 설정 예시, 검증 기준을 상세히 정리한 실행 가이드이다.32 Core / 256GB는 8 Core / 32GB 표준 Scale-Out 단위를 단순히 키운 값이 아니다. 고부하 조회, SingleView 집중 처리, EBM Rule 처리, 분석성·배치성 Java 업무, 또는 BI/리포팅 성격이 강한 특수 AP에 적용하는 검토 기준으로 보아야 한다.

항목32 Core / 256GB 기준 적용 원칙작업 시 주의사항서버 성격고부하·특수 AP 기준일반 온라인 AP 전체를 대형 VM에 무조건 통합하지 않는다.

처리량 기준이론상 1,000 TPS/VM 검토 가능실측 전까지는 업무복잡도, DB SQL, 외부연계 대기시간을 반영하여 보수 적용한다.

메모리 기준Heap 48~64GB 중심, 특수 업무는 96GB 이내 검토Heap을 크게 잡는 것이 항상 좋은 것은 아니며 GC Pause와 Native Memory를 같이 본다.

Thread 기준Tomcat maxThreads 1,500~1,800Thread 수 증가가 처리량 증가를 보장하지 않는다. DB Pool과 연계 Timeout이 함께 맞아야 한다.

DB Pool 기준일반 120~150, SingleView 150~180AP 수 × Pool 수가 DB Session 한계를 넘지 않도록 DBA와 사전 검증한다.

장애 기준AP A-A, DB A-S, 영역별 격리센터 장애나 VM 장애 시 잔여 처리량을 반드시 계산한다.2. 32 Core / 256GB 기준 총괄 설정표아래 표는 전체 영역의 1차 기준값이다. 실제 운영 반영 전에는 선도개발, 대표 SingleView 거래, 연계 호출, RDW SQL을 포함한 성능테스트로 보정해야 한다.

## 3. 용량산정 및 서버 배치 작업 가이드32 Core / 256GB 서버를 사용할 때도 세션 수와 TPS를 혼동하면 안 된다. 세션은 로그인 유지 규모이고, TPS·Thread·DB Pool은 동시에 요청을 발생시키는 사용자 기준으로 산정한다.

작업 지침32 Core / 256GB 서버 1대가 720 TPS를 처리할 수 있어도 운영 구성은 최소 2대 이상으로 설계한다.

AP 수량 산정 후 AP 수 × Hikari Pool 크기로 DB Session 총량을 계산하고 DBA와 한계치를 확인한다.

### 40.1 적용 대상

본 절(**적용 대상**)은 32C/256G 영역에서 **배치·ETL** 관점의 NSIGHT 1차 표준을 정의합니다. 피크 **720 TPS**·p95 **3초 이하**·8 vCPU / 32GB 기준으로 권고값을 제시하며, 최종값은 선도개발·성능시험(360/720/1080) 실측으로 확정합니다.

#### NSIGHT 권고값

| 항목 | 배치·ETL 권고 |
|------|----------------|
| 기준 VM | 8 vCPU / 32GB |
| TPS | 360/720/1080 |
| 설정 파일 | ETL AP |
| 핵심 | JVM 분리 |

#### 설정 예시

**설정 파일**: `ETL AP` · **핵심 항목**: JVM 분리

#### 검증 기준

| 시나리오 | 합격 기준 |
|----------|----------|
| 360 TPS | p95≤3s, CPU≤70%, 오류율≤1% |
| 720 TPS | Thread/Pool 고갈 없음, Hikari Pending=0 |
| 1,080 TPS | Fail-fast·Timeout 정상, CB 동작 확인 |
| AP 1대 Down | L4 제외·잔여 TPS≥목표 |
| 센터 장애 | 잔여 센터 TPS≥720 |

#### 주의사항

- **피크 기준** 설계 — 평균 TPS만으로 설정 확정 금지
- **세션 ≠ TPS** — 세션 증가는 Heap·복제 부담, TPS와 별도 산정
- **Thread ≠ Pool** — Pool ≤ Thread×30~40%, DB Session 총량 검증
- **안쪽 Timeout 짧게** — SQL(3s) < Tx(5s) < Proxy(10s) < Web(15s)

#### 운영 참고

DEV→STG→PRD 동일 항목·환경별 수치만 변경. PRD 확정값은 변경관리(53장) 준수.

본 절(**업무별 Bulkhead**)은 32C/256G 영역에서 **배치·ETL** 관점의 NSIGHT 1차 표준을 정의합니다. 피크 **720 TPS**·p95 **3초 이하**·8 vCPU / 32GB 기준으로 권고값을 제시하며, 최종값은 선도개발·성능시험(360/720/1080) 실측으로 확정합니다.

#### NSIGHT 권고값

| 패턴 | 권고 |
|------|------|
| Circuit Breaker | 필수 |
| Bulkhead | 업무별 |
| Rate Limit | Gateway |

#### 설정 예시

```yaml
resilience4j.circuitbreaker.failure-rate-threshold: 50
```

#### 검증 기준

| 시나리오 | 합격 기준 |
|----------|----------|
| 360 TPS | p95≤3s, CPU≤70%, 오류율≤1% |
| 720 TPS | Thread/Pool 고갈 없음, Hikari Pending=0 |
| 1,080 TPS | Fail-fast·Timeout 정상, CB 동작 확인 |
| AP 1대 Down | L4 제외·잔여 TPS≥목표 |
| 센터 장애 | 잔여 센터 TPS≥720 |

#### 주의사항

- **피크 기준** 설계 — 평균 TPS만으로 설정 확정 금지
- **세션 ≠ TPS** — 세션 증가는 Heap·복제 부담, TPS와 별도 산정
- **Thread ≠ Pool** — Pool ≤ Thread×30~40%, DB Session 총량 검증
- **안쪽 Timeout 짧게** — SQL(3s) < Tx(5s) < Proxy(10s) < Web(15s)

## 32Core·256GB 표준 프로파일 — 실무 요약

본 장은 **32C/256G**(배치·ETL)의 핵심을 23장(JVM)과 동일한 깊이로 요약합니다.

### E2E 용량산정 체인

```
[1] 사용자 21,600 → [2] 동시요청자 1,080~3,240
→ [3] TPS 360/720/1080 → [4] AP 8대 권장
→ [5] Thread 400~500 → [6] Pool 50/60
→ [7] DB Session Σ(AP×Pool) → [8] 장애 잔여 TPS≥720
```

### NSIGHT 1차 표준 요약

| 항목 | NSIGHT 1차 권고 |
|------|----------------|
| 사용자 | 21,600명 (3600×6) |
| 세션 | 26,000~28,000 |
| TPS | 360 / 720 / 1080 |
| VM | 8 vCPU / 32GB |
| VM당 TPS | 250 |
| p95 | 3초 이하 |

### Timeout 계층

| 계층 | 권고 | 설정 위치 |
|------|------|----------|
| SQL | 3초 | MyBatis / Mapper |
| Pool 획득 | 3초 | Hikari connectionTimeout |
| Transaction | 5초 | @Transactional |
| TCF ServiceId | 4~5초 | TCF 설정 |
| Proxy | 10초 | Apache / Gateway |
| WebTop | 15초 | 단말 |
| L4 Idle | 120초 | L4 Profile |

### 산정 공식

```
동시요청자 = 전체사용자 × 동시요청률
TPS = 동시요청자 ÷ 목표응답시간(3초)
AP 대수 = ⌈TPS ÷ 250⌉ (A-A 배치)
Thread = AP당TPS × 1.2초 × 1.2
Pool = max(30, min(TPS×0.15×1.3, Thread×30%))
```
