## 원문 기반 본문

NSIGHT 시스템 용량산정 및 아키텍처 설정 표준 가이드아키텍처 파트요약 보고 (Executive Summary)본 문서는 NSIGHT 시스템 안정성과 확장성을 보장하기 위한 세션 아키텍처 설계 및 시스템 용량산정 기준을 정리한 보고서이다. 핵심 전략은 다음과 같다.

세션 아키텍처: GSLB 기반의 센터 선택과 WebTopSuite의 센터 L4 직접 접근 방식을 채택한다. 센터 내부 AP 간에는 Tomcat DeltaManager를 통해 세션을 복제하여 가용성을 확보하되, 센터 간 복제는 미적용하여 복잡도를 최소화한다.

용량산정 표준: 8 vCPU / 32GB 사양의 VM 1대당 250 TPS 처리를 기준으로 삼는다. 전체 사용자 21,600명을 기준으로 설계 세션은 26,000~28,000개로 산정하며, 피크 타임 시 720 TPS 처리를 목표로 한다.

권장 구성: 피크 부하 및 센터 장애 대응(DR)을 고려하여 센터당 4대, 총 8대의 AP 서버 구성을 1차 권고한다. 이는 한 개 센터 장애 시에도 잔여 센터가 피크 부하(720 TPS)를 안정적으로 감당할 수 있는 구조이다.

성능 목표: 목표 응답시간은 p95 기준 3초 이하로 설정하며, 이를 위해 평균 SQL 처리 속도 100~300ms 유지와 타임아웃(Timeout) 체계의 정합성 확보를 필수 원칙으로 한다.

## 1. 세션 관리 아키텍처 설계

### 1.1 기본 설계 원칙NSIGHT 세션 아키텍처는 센터 간 독립성을 유지하면서 센터 내부의 가용성을 극대화하는 방향으로 설계되었다.

항목설계 내용센터 선택GSLB DNS Lookup을 통해 접속 가능한 센터 L4 VIP 식별센터 유지WebTopSuite 단말이 식별된 센터 정보를 보관하고 해당 L4로 직접 접근세션 복제Tomcat DeltaManager를 사용하여 동일 센터 내부 AP Cluster 간 세션 복제장애 정책센터 장애 시 타 센터로 전환하되, 기본적으로 재로그인 정책 적용데이터 최소화세션에는 인증 및 권한 정보만 저장하고, 업무 데이터 저장은 엄격히 금지

### 1.2 세션 데이터 관리 정책세션 비대화로 인한 복제 부하 및 성능 저하를 방지하기 위해 저장 항목을 엄격히 제한한다.

저장 가능 항목: User ID, Branch/Dept ID, Role ID, 메뉴 권한 요약, 인증 상태, CSRF 토큰 등 최소 정보 (목표 크기 2KB 이하).

저장 금지 항목: 고객 Single View 조회 결과, 거래 목록, 캠페인 대상자 목록, 파일 데이터, RDW ResultSet 등 대용량 데이터.

## 6. Runtime별 용량산정 및 관리 원칙

### 6.1 자원 경합 차단NSIGHT는 업무 유형에 따라 Runtime을 분리하여 상호 간섭을 최소화한다.

RDW: 현행성·정합성 중심, Single View 조회 전용.

ADW: 분석·집계·통계 및 BI 조회 전용.

자원 분리: 온라인 조회와 대량 배치/분석 SQL이 동일 자원을 점유하지 않도록 Pool 및 인스턴스를 분리한다.

### 6.2 데이터 흐름 기반 산정FAST 흐름 (CDC/Kafka): 지연시간(Latency)과 Consumer Lag을 중심으로 산정.

DEEP 흐름 (ETL/DataStage): 적재 데이터량과 배치 윈도우(시간 내 완료 여부)를 중심으로 산정.

데이터 보관: 초기 데이터량 외에 5년 성장률 및 인덱스/파티션 오버헤드를 반영하여 저장 공간을 확보한다.

1. RDW와 ADW의 분리는 데이터 균형을 설계하는 일이다농협 상호금융 정보계에서 RDW와 ADW를 분리하는 것은 단순한 데이터베이스 분리가 아니다.

이것은실시간성, 정합성, 분석 성능, 운영 안정성 사이의 균형을 설계하는 결정이다.

RDW는 현행성, 정합성, 빠른 조회가 중요한 영역이다.

ADW는 분석, 집계, 전략 의사결정, 대량 데이터 처리가 중요한 영역이다.

만약 RDW와 ADW를 하나의 구조로 섞어버리면 어떻게 되는가.

실시간 조회와 대량 분석이 같은 자원을 사용하게 된다.

마케팅 조회와 BI 분석이 서로 성능에 영향을 준다.

운영성 데이터와 분석성 데이터의 책임이 흐려진다.

장애가 발생했을 때 영향 범위를 분리하기 어렵다.

그래서 우리는 RDW와 ADW를 분리한다.

이 분리는 단순한 물리 분리가 아니다.

데이터의 역할을 분리하고, 업무의 책임을 분리하고, 성능의 충돌을 막기 위한 아키텍처적 균형이다.

RDW는 빠르고 정확해야 한다.

ADW는 깊고 넓게 분석할 수 있어야 한다.

이 두 가지를 동시에 만족시키기 위해농협 상호금융 아키텍처는 저장소를 나누는 것이 아니라,데이터의 성격에 맞는 결정 구조를 나누는 것이다.

## 2. 실시간과 배치의 분리는 처리 방식의 균형을 설계하는 일이다정보계 시스템에서 모든 것을 실시간으로 처리하는 것은 이상적으로 보일 수 있다.

그러나 현실의 시스템은 그렇게 단순하지 않다.

어떤 데이터는 즉시 반영되어야 한다.

어떤 데이터는 검증과 정제가 필요하다.

어떤 데이터는 대량으로 모아서 처리해야 한다.

어떤 데이터는 분석 품질을 위해 일정한 기준 시점이 필요하다.

그래서 농협 상호금융 아키텍처는실시간과 배치를 대립 관계로 보지 않고, 병행 구조로 설계한다.

실시간은 빠른 판단을 위한 흐름이다.

배치는 깊은 분석과 정합성 확보를 위한 흐름이다.

CDC와 Kafka를 활용한 실시간 이벤트 흐름은고객 변화, 거래 변화, 마케팅 반응과 같은 빠른 업무 대응을 가능하게 한다.

반면 ETL과 DataStage 기반의 배치 흐름은대량 데이터 정제, 집계, 분석, 검증, 기준 데이터 생성을 담당한다.

즉, 농협 상호금융 아키텍처에서 실시간과 배치의 분리는“무엇이 더 좋은가”를 선택하는 문제가 아니다.

빠르게 흘러야 할 데이터는 빠르게 흐르게 하고,정확하게 쌓여야 할 데이터는 정확하게 쌓이게 하는 균형의 문제이다.

- **Thread ≠ Pool** — Pool ≤ Thread×30~40%, DB Session 총량 검증
- **안쪽 Timeout 짧게** — SQL(3s) < Tx(5s) < Proxy(10s) < Web(15s)

### 42.3 VM·Tomcat·WAR 배치

본 절(**VM·Tomcat·WAR 배치**)은 목표 아키텍처 영역에서 **To-Be** 관점의 NSIGHT 1차 표준을 정의합니다. 피크 **720 TPS**·p95 **3초 이하**·8 vCPU / 32GB 기준으로 권고값을 제시하며, 최종값은 선도개발·성능시험(360/720/1080) 실측으로 확정합니다.

#### NSIGHT 권고값

| 시나리오 | 동시요청률 | 동시요청자 | TPS |
|----------|-----------|-----------|-----|
| 기본 | 5% | 1,080 | 360 |
| 피크 | 10% | 2,160 | 720 |
| 스트레스 | 15% | 3,240 | 1080 |

| VM | VM당 TPS | 피크 AP 권장 |
|----|----------|-------------|
| 8C/32G | 250 | 8대(A-A) |

#### 설정 예시

```
동시요청자 = 전체사용자 × 동시요청률
TPS = 동시요청자 ÷ 목표응답시간(3초)
AP 대수 = ⌈TPS ÷ 250⌉ (A-A 배치)
Thread = AP당TPS × 1.2초 × 1.2
Pool = max(30, min(TPS×0.15×1.3, Thread×30%))
```

#### 검증 기준

| 시나리오 | 합격 기준 |
|----------|----------|
| 360 TPS | p95≤3s, CPU≤70%, 오류율≤1% |
| 720 TPS | Thread/Pool 고갈 없음, Hikari Pending=0 |
| 1,080 TPS | Fail-fast·Timeout 정상, CB 동작 확인 |
| AP 1대 Down | L4 제외·잔여 TPS≥목표 |
| 센터 장애 | 잔여 센터 TPS≥720 |

#### 주의사항

- **피크 기준** 설계 — 평균 TPS만으로 설정 확정 금지
- **세션 ≠ TPS** — 세션 증가는 Heap·복제 부담, TPS와 별도 산정
- **Thread ≠ Pool** — Pool ≤ Thread×30~40%, DB Session 총량 검증
- **안쪽 Timeout 짧게** — SQL(3s) < Tx(5s) < Proxy(10s) < Web(15s)
