80~100개·SingleView AP 100~120개를 1차 기준으로 설정한다. 공통적으로 Spring Transaction Timeout은 4~5초, DB Query Timeout은 2~3초, WebTopSuite Request Timeout은 6~8초, L4 Idle Timeout은 10~15초 이상으로 설정한다.세션은 기본 90분으로 관리하고, Tomcat DeltaManager는 센터 내부 AP Cluster에 한정해 적용한다.별첨농협 상호금융 정보계에서 8CORE VM 기반이 더 적합한 이유- 16CORE 대비 운영 안정성과 장애격리 중심의 아키텍처 전략 -1. 개요농협 상호금융 정보계(NSIGHT)는 일반적인 웹서비스가 아니라 다음과 같은 특성을 가진다.전국 단위 동시 사용자 처리실시간 고객조회 및 SingleView 중심 구조RDW 기반 온라인 조회 구조WebTopSuite 기반 장시간 세션AP Active-Active 운영 구조센터 장애 시 즉시 Fail-Over 필요무중단 배포 및 운영 지속성 요구장애 영향 최소화가 핵심 목표따라서 단순히 “서버 수를 줄이는 구조”보다,운영 가능한 안정성, 장애 격리, 무중단 운영, 점진적 확장성이 더 중요하다.이 기준으로 보면 농협 정보계는 16CORE보다 8CORE 기반 Scale-Out 구조가 더 적합하다.2. 8CORE와 16CORE의 근본적인 차이구분8CORE / 32GB16CORE / 64GB구조Scale-Out 중심Scale-Up 중심VM 1대 처리량약 250 TPS약 500 TPS운영 철학작은 단위 분산큰 단위 집중장애 영향 범위작음큼배포 영향 범위작음큼확장 방식VM 추가VM 증설운영 복구빠름상대적으로 큼장애 격리유리불리농협 정보계 적합성높음보조적즉,8CORE = 운영 안정성 중심16CORE = 자원 효율 중심이다.3. 농협 정보계에서 중요한 것은 “TPS”보다 “운영 가능성”많은 프로젝트가 TPS만 보고 큰 VM을 선호한다.하지만 정보계는 TPS보다 다음이 더 중요하다.핵심 요소중요 이유장애 영향 최소화전국 업무 영향 방지무중단 배포영업시간 운영 지속장애 격리특정 AP 장애의 전체 전파 방지점진적 확장업무 증가 대응Session 안정성장시간 사용자 유지RDW 보호DB Session 폭증 방지운영 단순화장애 분석과 복구 속도 향상즉,“한 대가 얼마나 빠른가”보다 “문제가 생겼을 때 얼마나 작게 터지는가”가 중요하다.4. 16CORE가 위험한 이유4.1 장애 영향 범위 증가16CORE VM은 VM 1대가 처리하는 TPS가 크다.기준8CORE16COREVM당 TPS250 TPS500 TPSVM 장애 시 손실 TPS250 TPS500 TPS즉,16CORE VM 1대 장애=8CORE VM 2대 장애와 유사이다.농협 정보계는 전국 사용자 기반이므로,VM 한 대의 영향 범위가 커지는 것은 매우 위험하다.4.2 GC Pause 위험 증가Heap이 커질수록 GC 영향 범위가 커진다.구분8CORE16COREHeap 크기12~14GB24~28GB객체 관리량작음큼GC Pause 영향작음큼장애 분석 난이도낮음높음정보계는 응답시간 p95 3초를 유지해야 하므로,큰 Heap보다 “작고 안정적인 Heap 여러 개”가 더 안전하다.4.3 Thread 증가에 따른 Context Switching 증가16CORE는 Thread를 많이 둘 수 있지만 무조건 좋은 것은 아니다.기준8CORE16CORE권장 maxThreads400~500800~1000Thread Stack 증가보통큼Context Switching적음많음CPU Burst 영향작음큼특히 WebTopSuite 기반 장시간 연결 환경에서는많은 Thread→ 더 많은 KeepAlive→ 더 많은 Session→ 더 많은 GC/메모리 사용구조가 된다.5. 8CORE가 농협 정보계에 유리한 이유5.1 장애 격리(Blast Radius)가 작다8CORE 구조는 장애 단위가 작다.8CORE VM 장애→ 전체 영향 작음→ 빠른 우회 가능→ 운영 복구 용이특히 AP Active-Active 구조에서는 이 안정성유리운영 분석 난이도낮음농협 정보계 적합성매우 높음8. 최종 결론농협 상호금융 정보계(NSIGHT)는 단순 TPS 처리 시스템이 아니라 전국 단위 사용자, WebTopSuite 장시간 세션, RDW 기반 SingleView 조회, AP Active-Active, 무중단 운영을 요구하는 고안정성 정보계 시스템이다. 따라서 단순히 서버 수를 줄이는 Scale-Up 방식보다 장애 영향 최소화, 운영 안정성, 점진적 확장성, 무중단 배포가 더 중요하다.16CORE / 64GB VM은 VM 수를 줄이고 TPS 효율을 높이는 장점은 있으나, VM 1대 장애 시 영향 범위 증가, GC Pause 확대, Thread 및 DB Session 급증, RDW 부하 증가 등의 위험이 존재한다.반면 8CORE / 32GB VM 기반 Scale-Out 구조는 VM 장애 영향 범위가 작고, AP Active-Active 구조에서 장애 격리와 Fail-Over가 용이하며, Rolling 배포와 점진적 확장이 쉽고, RDW 보호와 운영 분석 측면에서도 안정적이다.따라서 농협 상호금융 정보계의 온라인 AP 및 SingleView AP는 8CORE / 32GB VM을 표준 단위로 채택하고, 필요 시 VM 수를 수평 확장하는 Scale-Out 구조를 기본 전략으로 적용하는 것이 가장 현실적이고 운영 가능한 아키텍처 전략이다.
