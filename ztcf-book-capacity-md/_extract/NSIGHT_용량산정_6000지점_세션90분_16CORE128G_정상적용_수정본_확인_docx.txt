Active-Active 구조에서 장애 격리와 Fail-Over가 용이하며, Rolling 배포와 점진적 확장이 쉽고, RDW 보호와 운영 분석 측면에서도 안정적이다.따라서 농협 상호금융 정보계의 온라인 AP 및 SingleView AP는 8CORE / 32GB VM을 표준 단위로 채택하고, 필요 시 VM 수를 수평 확장하는 Scale-Out 구조를 기본 이것입니다.“16CORE / 128GB VM인데 왜 JVM Heap을 32GB 정도만 쓰고, 나머지 메모리는 어디에 쓰는가?”결론부터 말하면, 128GB 전체를 JVM Heap으로 크게 잡는 것이 좋은 운영 설계가 아닙니다.NSIGHT 같은 은행 정보계 온라인 AP에서는 Heap을 크게 잡아 TPS를 올리는 방식보다, Heap은 안정적인 크기로 제한하고 Heap으로 쓰지 않는 이유2.1 Heap을 키운다고 TPS가 자동 증가하지 않는다TPS는 대략 다음 요소가 결정합니다.TPS ≒ CPU Core 수 × Core당 처리능력 × 응답시간 안정성반면 Heap은 다음에 영향을 줍니다.Heap ≒ 객체 보관량, 세션 객체, 캐시, 응답 조립 데이터, GC 안정성따라서 16CORE/64GB와 제한하는 이유는 전체 메모리를 사용하지 못해서가 아니라, 온라인 AP의 운영 안정성을 확보하기 위한 의도적 설계이다. NSIGHT 온라인 AP의 TPS는 CPU Core 수, 평균 응답시간, DB SQL 시간, Tomcat Busy Thread, Hikari Pool 사용률에 의해 결정되며, 128GB 메모리는 TPS를 직접 증가시키는 자원이 아니다. 따라서 Heap은 일반 AP 24~32GB, SingleView AP 32~40GB 범위에서 관리하고, 나머지 메모리는 OS 영역, 보안 Agent, APM/Monitoring Agent, Thread Stack, Direct Memory, File System Cache, 로그·덤프 생성, 장애 분석 및 피크 예비 영역으로 확보한다. 이 구조가 GC Pause, Heap Dump 분석, 장애 복구, 운영 감시, 보안 정책 적용 측면에서 더 안정적이다.한 줄로 정리하면:16CORE/128GB에서 JVM 32GB는 메모리를 적게 쓰는 것이 아니라, Heap을 통제 가능한 크기로 제한하고 나머지 메모리를 운영 안정성과 장애 대응 여유로 남기는 금융권 온라인 AP형 설계입니다.
