**16Core·64GB 프로파일** — 특수·고부하 검토용.

## 원문 기반 본문

농협 상호금융 NSIGHT 정보계 온라인 AP VM 표준 전략 보고서전체 64 Core / 256GB 자원 기준: 8CORE × 8대 vs 16CORE × 4대 운영 안정성 비교

## 1. 검토 목적

본 문서는 농협 상호금융 NSIGHT 마케팅플랫폼과 같은 은행 정보계 온라인 AP를 운영할 때, 동일한 총 자원인 64 Core / 256GB Memory를 8CORE VM 8대 구조로 구성하는 방안과 16CORE VM 4대 구조로 구성하는 방안을 운영 안정성 관점에서 비교·검토하기 위한 전략 문서이다.

검토의 핵심은 단순 처리량이 아니라 장애 영향도, RDW 보호, 장애 분석 용이성, GC 안정성, 배포·복구 유연성이다. NSIGHT 용량산정은 평균 사용량이 아니라 Runtime별 피크 부하, 자원 경합, 장애 전환, 확장성을 기준으로 산정해야 하며, 온라인 AP와 DB Connection Pool을 별도로 관리해야 한다.

## 2. 결론 요약결론적으로 NSIGHT 온라인 AP와 SingleView AP는 16CORE 대형 VM으로 적게 구성하는 방식보다, 8CORE VM을 표준 단위로 여러 대 구성하는 Scale-Out 방식이 더 적합하다.16CORE는 서버 수를 줄이는 장점이 있지만, 온라인 AP 운영에서는 VM 1대 장애 시 영향 범위가 커지고, DB Connection 압력이 커지며, 장애 분석과 GC 리스크가 증가한다. 반면 8CORE는 장애 영향 단위가 작고, Rolling 배포와 장애 격리가 쉽고, RDW 보호와 운영 분석 측면에서 안정적이다.

구분8CORE × 8대16CORE × 4대판단전체 Core/Memory64 Core / 256GB64 Core / 256GB총 자원은 동일운영 방식작은 단위 Scale-Out큰 단위 Scale-Up정보계는 Scale-Out 유리VM 1대 장애 영향전체 용량의 12.5% 손실전체 용량의 25% 손실8CORE 우위DB Pool 압력작게 분산VM당 대형 Pool 필요8CORE 우위장애 분석Heap/Thread Dump 분석 용이Dump 크고 분석 복잡8CORE 우위GC 리스크상대적으로 낮음Heap 증가로 영향 확대8CORE 우위배포/롤백1대씩 점진 배포 용이1대 제외 영향 큼8CORE 우위적합 영역온라인 AP, SingleView AP배치, BI, 분석, ETL 일부역할 분리 권장

## 5. 백엔드 DB(RDW) 보호 및 Session 관리정보계 온라인 AP의 핵심 병목은 AP CPU만이 아니다. 실제 운영에서는 RDW SQL 시간, Hikari Pool 사용률, DB Session 총량이 서비스 안정성에 큰 영향을 준다. 8CORE 구조는 DB Connection Pool이 작은 단위로 분산되어 특정 VM의 폭주가 전체 RDW를 급격히 압박하지 않도록 제어하기 쉽다.

항목8CORE × 8대16CORE × 4대판단Pool 설계VM당 일반 50 / SingleView 60VM당 일반 80~100 / SingleView 100~12016CORE는 VM당 Pool 큼DB Session 증가 속도작은 단위로 완만소수 VM에서 급격8CORE가 RDW 보호 유리Pool 격리업무별·VM별 통제 용이한 VM 내 집중 가능8CORE 유리장애 시 DB 압력분산되어 완화잔여 VM Pool에 집중8CORE 유리따라서 온라인 AP의 DB Pool은 Thread와 1:1로 맞추는 것이 아니라, RDW를 보호하기 위한 통제 장치로 설계해야 한다. 특히 SingleView는 고객정보, 거래요약, 캠페인 이력 등 다중 조회가 발생할 수 있으므로 Pool을 크게 여는 방식보다 SQL 품질, Query Timeout, Pool 사용률 감시가 더 중요하다.

## 6. 장애 원인 분석 및 사후 대응금융권 운영에서는 장애를 막는 것만큼 빠른 원인 분석과 복구가 중요하다. JVM 기반 온라인 AP에서는 Heap Dump, Thread Dump, GC Log, APM Trace 분석 속도가 장애 대응 시간을 좌우한다. 8CORE/32GB 구조는 Heap 크기가 상대적으로 작아 덤프 추출과 분석이 수월하다.

항목8CORE / 32GB16CORE / 64GB운영 영향Heap 크기12~14GB 수준24~28GB 이상8CORE 분석 용이Heap Dump상대적으로 작음대용량 Dump16CORE 분석 지연 가능Thread DumpThread 수 적음Thread 수 많음8CORE 직관적GC 로그 분석단순복잡8CORE 유리장애 복구 판단빠름느림운영 안정성은 8CORE 우위

## 7. GC Stop-The-World 리스크온라인 AP에서 GC Pause는 사용자 응답시간 안에 포함된다. 목표 응답시간이 p95 3초라면 GC Pause가 500ms 이상 반복되는 것만으로도 응답시간 SLA를 크게 흔들 수 있다. 16CORE 구조는 대형 Heap을 사용하기 때문에 객체 생명주기와 GC 튜닝이 잘못되면 Stop-The-World 영향이 커질 수 있다.

항목8CORE / 32GB16CORE / 64GB판단Heap 관리 단위작음큼8CORE 안정GC Pause 영향 범위VM 단위 작음VM 단위 큼8CORE 유리STW 발생 시 영향 사용자상대적으로 적음상대적으로 많음8CORE 유리튜닝 난이도낮음높음16CORE는 고급 튜닝 필요따라서 NSIGHT 온라인 AP는 G1GC 기준으로 운영하되, Heap을 과도하게 키우기보다 8CORE 단위로 분산하여 GC 영향 범위를 작게 유지하는 것이 안정적이다.

## 8. 배포·오토스케일링·복구 동적 안정성은행 정보계는 영업시간 중에도 긴급 패치, 설정 변경, 장애 우회, Rolling 배포가 필요할 수 있다. 8CORE VM은 단위가 작아 1대씩 L4에서 제외하고 배포·검증·복귀하기 쉽다. 반면 16CORE VM은 1대 제외만으로도 전체 용량의 25%가 빠지므로 배포 리스크가 커진다.

항목8CORE × 8대16CORE × 4대판단Rolling 배포1대 제외 영향 12.5%1대 제외 영향 25%8CORE 유리긴급 증설작은 단위 증설큰 단위 증설8CORE 유연L4 분산 안정성점진적 분산큰 단위 변동8CORE 유리Fail-Back점진 복귀 가능복귀 영향 큼8CORE 유리

## 9. 전략적 권장안

NSIGHT 마케팅플랫폼의 온라인 AP와 SingleView AP는 8CORE / 32GB VM을 표준 단위로 채택하는 것이 가장 타당하다. 이는 총 자원을 줄이기 위한 결정이 아니라, 동일 자원을 더 운영 가능한 단위로 나누어 장애 영향과 운영 리스크를 낮추는 결정이다.

구분권장 전략설명온라인 AP8CORE / 32GB 표준전국 사용자, 세션, RDW 조회, 장애 격리 관점에서 유리SingleView AP8CORE 기반 Scale-OutDB Pool과 SQL 병목을 작은 단위로 통제16CORE 적용 영역배치/BI/AI/ETL 일부장시간 처리, 대용량 메모리 필요 영역에 제한 적용운영 기준작은 VM 다수 구조Rolling 배포, 장애 격리, 복구 용이성 확보검증 기준성능테스트 필수TPS, p95, GC, Hikari, SQL, Busy Thread 확인

## 10. 최종 결론동일한 전체 64 Core / 256GB 자원을 기준으로 볼 때, 16CORE × 4대 구성은 서버 수를 줄이고 단위 처리량을 높일 수 있는 장점이 있다. 그러나 농협 상호금융 NSIGHT와 같은 은행 정보계 온라인 AP에서는 단순 자원 효율보다 운영 안정성, 장애 격리, RDW 보호, 무중단 배포, 장애 분석 용이성이 더 중요하다.8CORE × 8대 구성은 VM 1대 장애 시 영향 범위가 12.5%로 작고, L4 부하 재분산이 완만하며, DB Connection Pool을 작은 단위로 제어할 수 있고, Heap/Thread/GC 분석이 상대적으로 쉽다. 또한 Rolling 배포와 장애 복구 시 한 번에 빠지는 용량이 작아 운영 안정성이 높다.

따라서 NSIGHT 마케팅플랫폼의 온라인 AP 및 SingleView AP는 8CORE / 32GB VM을 표준 운영 단위로 채택하고, 필요 시 VM 수를 수평 확장하는 Scale-Out 전략을 적용하는 것이 가장 현실적이다. 16CORE / 64GB VM은 온라인 AP 표준이 아니라, 배치·BI·AI·ETL 등 대용량 처리나 고메모리 요구가 명확한 영역에 제한적으로 적용하는 것이 바람직하다.

부록. 보고용 핵심 메시지핵심 메시지보고 문장운영 철학NSIGHT 온라인 AP는 크게 묶는 구조보다 작게 나누어 안정적으로 운영하는 구조가 적합하다.

장애 영향16CORE 1대 장애는 전체 용량의 25% 손실이지만, 8CORE 1대 장애는 12.5% 손실에 그친다.

DB 보호8CORE 구조는 DB Connection Pool을 작은 단위로 제어해 RDW Session 폭증을 방지하기 쉽다.

장애 분석8CORE는 Heap Dump, Thread Dump, GC Log 분석이 상대적으로 빠르고 단순하다.

최종 권고온라인 AP와 SingleView AP는 8CORE 표준, Batch/BI/ETL은 필요 시 16CORE 제한 적용이 합리적이다.

| 시나리오 | 합격 기준 |
|----------|----------|

#### 주의사항

- **피크 기준** 설계 — 평균 TPS만으로 설정 확정 금지
- **세션 ≠ TPS** — 세션 증가는 Heap·복제 부담, TPS와 별도 산정
- **Thread ≠ Pool** — Pool ≤ Thread×30~40%, DB Session 총량 검증
- **안쪽 Timeout 짧게** — SQL(3s) < Tx(5s) < Proxy(10s) < Web(15s)

## 16Core·64GB 표준 프로파일 — 실무 요약

본 장은 **16C/64G**(특수 프로파일)의 핵심을 23장(JVM)과 동일한 깊이로 요약합니다.

### E2E 용량산정 체인

```
[1] 사용자 21,600 → [2] 동시요청자 1,080~3,240
→ [3] TPS 360/720/1080 → [4] AP 8대 권장
→ [5] Thread 400~500 → [6] Pool 50/60
→ [7] DB Session Σ(AP×Pool) → [8] 장애 잔여 TPS≥720
```

### NSIGHT 1차 표준 요약

| 항목 | NSIGHT 1차 권고 |
|------|----------------|
| 사용자 | 21,600명 (3600×6) |
| 세션 | 26,000~28,000 |
| TPS | 360 / 720 / 1080 |
| VM | 8 vCPU / 32GB |
| VM당 TPS | 250 |
| p95 | 3초 이하 |

| 계층 | 권고 | 설정 위치 |

### 산정 공식

```
동시요청자 = 전체사용자 × 동시요청률
TPS = 동시요청자 ÷ 목표응답시간(3초)
AP 대수 = ⌈TPS ÷ 250⌉ (A-A 배치)
Thread = AP당TPS × 1.2초 × 1.2
Pool = max(30, min(TPS×0.15×1.3, Thread×30%))
```
