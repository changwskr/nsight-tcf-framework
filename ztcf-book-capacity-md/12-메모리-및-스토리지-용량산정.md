질문하신 핵심은 이것입니다.“16CORE / 128GB VM인데 왜 JVM Heap을 32GB 정도만 쓰고, 나머지 메모리는 어디에 쓰는가?”결론부터 말하면, 128GB 전체를 JVM Heap으로 크게 잡는 것이 좋은 운영 설계가 아닙니다.

NSIGHT 같은 은행 정보계 온라인 AP에서는 Heap을 크게 잡아 TPS를 올리는 방식보다, Heap은 안정적인 크기로 제한하고 나머지 메모리는 OS, 보안, APM Agent, Native Memory, Thread Stack, 장애 분석 여유, 파일/네트워크 버퍼, 운영 예비 영역으로 남기는 방식이 더 안전합니다.

## 1. 기본 결론구분판단VM 사양16CORE / 128GBJVM Heap 권장일반 AP 24~32GB, SingleView AP 32~40GB 이내128GB 전체 Heap 사용 여부비권장VM당 TPS16CORE/64GB와 동일하게 500 TPS 기준128GB의 의미TPS 증가가 아니라 운영 안정성·메모리 여유·장애 대응 여유핵심 원칙Heap은 필요한 만큼만, 나머지는 운영 여유로 남긴다16CORE/128GB에서 JVM을 32GB 수준으로 잡는 이유는 Core 수가 16개로 동일하면 TPS의 1차 한계는 CPU와 거래당 처리시간이 결정하고, 메모리는 그 TPS를 더 안정적으로 유지하게 해주는 보조 자원이기 때문입니다. 기존 NSIGHT 기준에서도 VM당 처리량은 TPS, 평균 응답시간, Tomcat Busy Thread, DB SQL 시간, Hikari Pool 사용률, GC Pause가 함께 맞아야 한다고 정리되어 있습니다.2. 128GB 메모리를 전부 JVM Heap으로 쓰지 않는 이유

### 2.1 Heap을 키운다고 TPS가 자동 증가하지 않는다TPS는 대략 다음 요소가 결정합니다.

TPS ≒ CPU Core 수 × Core당 처리능력 × 응답시간 안정성반면 Heap은 다음에 영향을 줍니다.

Heap ≒ 객체 보관량, 세션 객체, 캐시, 응답 조립 데이터, GC 안정성따라서 16CORE/64GB와 16CORE/128GB는 CPU Core 수가 같으므로 기준 TPS는 동일하게 보는 것이 맞습니다.128GB는 더 많은 TPS를 보장하는 자원이 아니라, GC 여유, 캐시 여유, 장애 시 분석 여유, Native 영역 여유를 확보하는 자원입니다.

### 2.2 Heap이 너무 크면 GC와 장애 분석이 어려워진다Heap을 64GB, 80GB, 100GB처럼 크게 잡으면 다음 문제가 생깁니다.

문제설명GC 관리 부담 증가JVM이 관리해야 할 객체 공간이 커짐Old 영역 누적 시 회수 부담 증가오래 살아 있는 객체가 많아질수록 GC 판단 비용 증가Heap Dump 크기 증가장애 시 덤프 파일이 수십 GB 이상으로 커짐장애 분석 시간 증가MAT, JProfiler, Mission Control 등 분석 도구 로딩 시간이 증가Full GC 발생 시 영향 증가발생 가능성은 낮출 수 있어도, 발생 시 영향이 커짐장애 복구 지연Dump 생성, 파일 전송, 압축, 분석 시간이 길어짐온라인 AP는 “메모리를 많이 쓰는 것”보다 일정한 응답시간을 유지하는 것이 중요합니다. NSIGHT 기준에서도 GC Pause는 응답시간 3초 안에 포함되므로 100~200ms 수준으로 관리하고, Full GC는 운영 중 미발생을 원칙으로 봅니다.3. 16CORE / 128GB VM의 메모리 사용 구조예를 들어 128GB VM에서 일반 AP Heap을 32GB로 설정하면 나머지 약 96GB는 놀고 있는 메모리가 아닙니다. 아래 영역들이 사용하거나, 장애와 피크를 대비한 여유 영역으로 남습니다.

## 4. 영역별로 왜 메모리가 필요한가

### 4.1 OS 영역항목설명Kernel Memory네트워크, 파일시스템, 프로세스 관리System Daemonsystemd, ssh, cron, 로그 서비스 등Network StackTCP 연결, 소켓 버퍼, 패킷 처리File System Cache자주 읽는 파일, 로그, 라이브러리 캐시Page Cache디스크 I/O 성능 안정화운영 중에는 JVM만 메모리를 쓰지 않습니다. Linux OS는 남는 메모리를 파일시스템 캐시와 네트워크 버퍼로 활용합니다. 특히 로그, 클래스 로딩, 설정 파일, APM 파일, 인증서, 라이브러리 접근이 많은 운영 서버에서는 OS 캐시가 응답 안정성에 도움이 됩니다.

### 4.2 보안 영역금융권 운영 서버에는 보통 여러 보안 기능이 같이 올라갑니다.

보안 영역메모리 사용 이유백신 / EDR Agent프로세스 감시, 파일 감시, 행위 탐지접근통제 Agent계정, 명령어, 파일 접근 감시DB 접근통제 연동 Agent접속 이력, 통제 로그 연계암호화 모듈키 관리, 암복호화 버퍼취약점 점검 Agent구성정보 수집, 정책 점검로그 수집 Agent감사로그, 보안로그 수집·전송이 영역은 평상시에는 작게 보일 수 있지만, 보안 점검·패턴 업데이트·로그 폭증·장애 시 수집 작업이 겹치면 메모리 사용량이 순간적으로 커질 수 있습니다. 따라서 운영 서버에서는 보안 Agent용 여유 메모리를 별도로 남겨두는 것이 안전합니다.

### 4.3 APM / Monitoring Agent 영역APM Agent는 단순 로그 수집기가 아닙니다. 거래를 추적하기 위해 다음 정보를 메모리에 잡습니다.

항목설명Transaction TraceURL, Controller, Service, SQL, 외부 호출 추적SQL TraceSQL ID, 실행시간, 호출 횟수Method Profiling특정 메서드 수행시간 수집Error TraceException, Stack Trace 수집Metric BufferCPU, Memory, Thread, GC, Pool, TPS 수집전송 BufferAPM 서버로 보내기 전 임시 저장NSIGHT에서는 GUID, 로그, 오류코드, 모니터링 지표로 거래 흐름을 추적하는 관측 가능성이 중요합니다. 이런 관측성은 운영 필수 기능이지만, 그만큼 APM/로그 Agent가 메모리를 사용합니다. 기존 NSIGHT 원칙에서도 모든 거래와 오류는 GUID, 로그, 오류코드, 모니터링 지표로 추적 가능해야 한다고 정리되어 있습니다.

### 4.4 Thread Stack 영역Tomcat의 maxThreads가 800~1,000이면 Thread Stack 메모리도 무시할 수 없습니다.

예를 들어 -Xss512k 기준이면 다음과 같습니다.1,000 Thread × 512KB = 약 512MB여기에 Tomcat Worker Thread 외에도 다음 Thread들이 추가됩니다.

### 4.5 Direct Memory / Native Memory 영역Java 애플리케이션은 Heap 밖의 메모리도 많이 사용합니다.

영역설명Direct ByteBufferNIO, HTTP 통신, 파일 I/OTLS BufferHTTPS 암복호화 처리Compression BufferGZIP, 로그 압축, 응답 압축JDBC Driver Native 영역Oracle Driver, 네트워크 BufferClass Metadata클래스 정보, Reflection 정보Code CacheJIT 컴파일된 Native 코드GC Native 구조G1GC Region 관리 등특히 금융권 AP는 HTTPS, JDBC, APM, 로그, 전문 파싱, JSON 직렬화가 많기 때문에 Heap 외 Native Memory 여유가 필요합니다.

### 4.6 장애 분석 여유 영역장애가 발생하면 평상시보다 메모리와 디스크, CPU 사용량이 더 커질 수 있습니다.

장애 상황추가 메모리/공간 필요 이유Heap Dump 생성Heap 크기만큼 대용량 덤프 파일 생성Thread Dump 반복 수집다수 Thread 상태 저장GC Log 증가장애 시 GC 로그 급증APM Trace 폭증오류 거래가 늘면 Trace Buffer 증가로그 폭증Timeout, DB 오류, 연계 오류 로그 증가압축/전송Dump 압축과 전송 과정에서 추가 자원 사용원인 분석 도구 실행운영 중 임시 분석 스크립트 실행 가능즉, 128GB VM이라고 해서 Heap을 100GB로 잡으면 장애 시 덤프 생성 자체가 장애를 더 악화시킬 수 있습니다. 반대로 Heap을 32GB 수준으로 제한하면 장애 시 덤프 분석과 복구가 훨씬 현실적입니다.5. 128GB VM에서 JVM 32GB 적용 예시

## 6. 왜 64GB Heap이 아니라 32GB Heap인가판단 항목32GB Heap64GB 이상 HeapGC 안정성관리 가능튜닝 부담 증가Heap Dump분석 가능 범위분석 부담 큼장애 복구상대적으로 빠름복구 지연 가능Full GC 영향제한 가능발생 시 영향 큼운영 표준화쉬움업무별 튜닝 필요OS/Agent 여유충분줄어듦온라인 AP 적합성높음낮음대용량 분석/캐시 적합성보통일부 적합32GB는 작게 잡는 것이 아니라, 온라인 AP에서 운영자가 통제 가능한 상한선에 가깝습니다.

SingleView처럼 메모리 사용이 큰 업무도 40GB 이내에서 시작하고, 실제 성능테스트에서 GC 로그, Heap 사용률, Old 영역 증가, 응답시간, Dump 분석 가능성을 보고 보정하는 것이 맞습니다.

## 7. 문서용 결론 문장문서에는 아래처럼 쓰면 가장 명확합니다.16CORE/128GB VM에서 JVM Heap을 32GB 수준으로 제한하는 이유는 전체 메모리를 사용하지 못해서가 아니라, 온라인 AP의 운영 안정성을 확보하기 위한 의도적 설계이다. NSIGHT 온라인 AP의 TPS는 CPU Core 수, 평균 응답시간, DB SQL 시간, Tomcat Busy Thread, Hikari Pool 사용률에 의해 결정되며, 128GB 메모리는 TPS를 직접 증가시키는 자원이 아니다. 따라서 Heap은 일반 AP 24~32GB, SingleView AP 32~40GB 범위에서 관리하고, 나머지 메모리는 OS 영역, 보안 Agent, APM/Monitoring Agent, Thread Stack, Direct Memory, File System Cache, 로그·덤프 생성, 장애 분석 및 피크 예비 영역으로 확보한다. 이 구조가 GC Pause, Heap Dump 분석, 장애 복구, 운영 감시, 보안 정책 적용 측면에서 더 안정적이다.

한 줄로 정리하면:16CORE/128GB에서 JVM 32GB는 메모리를 적게 쓰는 것이 아니라, Heap을 통제 가능한 크기로 제한하고 나머지 메모리를 운영 안정성과 장애 대응 여유로 남기는 금융권 온라인 AP형 설계입니다.

### 12.1 OS Memory

본 절(**OS Memory**)은 메모리·스토리지 영역에서 **Heap·디스크** 관점의 NSIGHT 1차 표준을 정의합니다. 피크 **720 TPS**·p95 **3초 이하**·8 vCPU / 32GB 기준으로 권고값을 제시하며, 최종값은 선도개발·성능시험(360/720/1080) 실측으로 확정합니다.

#### NSIGHT 권고값

| 구분 | 8C/32G |
|------|--------|
| OS·Cache | 6~8GB |
| APM/Agent | 2~4GB |
| 여유 | ≥4GB |

#### 설정 예시

32GB VM에서 Heap 24GB+ **금지**.

#### 검증 기준

| 시나리오 | 합격 기준 |
|----------|----------|
| 360 TPS | p95≤3s, CPU≤70%, 오류율≤1% |
| 720 TPS | Thread/Pool 고갈 없음, Hikari Pending=0 |
| 1,080 TPS | Fail-fast·Timeout 정상, CB 동작 확인 |
| AP 1대 Down | L4 제외·잔여 TPS≥목표 |
| 센터 장애 | 잔여 센터 TPS≥720 |

#### 주의사항

- **피크 기준** 설계 — 평균 TPS만으로 설정 확정 금지
- **세션 ≠ TPS** — 세션 증가는 Heap·복제 부담, TPS와 별도 산정
- **Thread ≠ Pool** — Pool ≤ Thread×30~40%, DB Session 총량 검증
- **안쪽 Timeout 짧게** — SQL(3s) < Tx(5s) < Proxy(10s) < Web(15s)
