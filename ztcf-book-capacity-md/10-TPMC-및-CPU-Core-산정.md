## 원문 기반 본문

TPS와 TPMC의 관계 분석 및 실효 처리량 산정 가이드요약 보고 (Executive Summary)본 문서는 Spring Boot 기반 정보계 시스템의 성능 산정 시 핵심 지표인 TPS(Transaction Per Second)와 TPMC(Transaction Per Minute Benchmark) 간의 상관관계를 분석한 결과이다. 핵심 분석 결과에 따르면, 단순히 이론적인 산식(1 TPS = 60 TPMC)을 적용하는 것은 실제 업무 복잡도를 반영하지 못해 위험하다.

Spring Boot 업무 거래는 전문 파싱, 인증, 로깅, DB 조회 등 다단계 프로세스를 포함하므로, 1 TPS당 약 2,500~3,50

0 TPMC의 성능 소모를 적용하는 것이 적정하다. 이를 기준으로 Core당 106,932 TPMC 성능을 보유한 서버를 산정할 경우, Core당 실효 처리량은 약 30~40 TPS로 도출된다.

## 1. 성능 산정의 기본 개념 및 이론적 배경

### 1.1 Core당 벤치마크 성능기준 성능: 1 Core당 106,932 TPMC이론적 TPS: 106,932 ÷ 60 = 약 1,782 TPS/Core한계점: 위 수치는 이론적인 벤치마크 값일 뿐이며, 실제 Spring Boot 기반의 복잡한 업무 거래 환경을 반영하지 못한다.

### 1.2 Spring Boot 업무 거래의 복잡성실제 정보계 거래는 단순한 수치 이상의 자원을 소모하며, 하나의 업무 TPS가 처리되기 위해 다음과 같은 일련의 단계를 거친다.

전문 수신 및 파싱인증 및 권한 처리공통 유효값 검증 및 공통 로깅비즈니스 업무 처리 (DB/RDW 조회 포함)추가 공통 로깅 및 전문 조립클라이언트 반환이러한 단계적 처리로 인해 실제 1 업무 TPS는 이론적 수치보다 10배에서 40배 수준의 TPMC 성능을 소모하게 된다.

## 절별 상세

### 10.1 TPS와 TPMC의 차이

본 절(**TPS와 TPMC의 차이**)은 TPMC·Core 영역에서 **CPU 교차검증** 관점의 NSIGHT 1차 표준을 정의합니다. 피크 **720 TPS**·p95 **3초 이하**·8 vCPU / 32GB 기준으로 권고값을 제시하며, 최종값은 선도개발·성능시험(360/720/1080) 실측으로 확정합니다.

#### NSIGHT 권고값

| 시나리오 | 동시요청률 | 동시요청자 | TPS |
|----------|-----------|-----------|-----|
| 기본 | 5% | 1,080 | 360 |
| 피크 | 10% | 2,160 | 720 |
| 스트레스 | 15% | 3,240 | 1080 |

| VM | VM당 TPS | 피크 AP 권장 |
|----|----------|-------------|
| 8C/32G | 250 | 8대(A-A) |

#### 설정 예시

```
동시요청자 = 전체사용자 × 동시요청률
TPS = 동시요청자 ÷ 목표응답시간(3초)
AP 대수 = ⌈TPS ÷ 250⌉ (A-A 배치)
Thread = AP당TPS × 1.2초 × 1.2
Pool = max(30, min(TPS×0.15×1.3, Thread×30%))
```

#### 검증 기준

| 시나리오 | 합격 기준 |
|----------|----------|
| 360 TPS | p95≤3s, CPU≤70%, 오류율≤1% |
| 720 TPS | Thread/Pool 고갈 없음, Hikari Pending=0 |
| 1,080 TPS | Fail-fast·Timeout 정상, CB 동작 확인 |
| AP 1대 Down | L4 제외·잔여 TPS≥목표 |
| 센터 장애 | 잔여 센터 TPS≥720 |

#### 주의사항

- **피크 기준** 설계 — 평균 TPS만으로 설정 확정 금지
- **세션 ≠ TPS** — 세션 증가는 Heap·복제 부담, TPS와 별도 산정
- **Thread ≠ Pool** — Pool ≤ Thread×30~40%, DB Session 총량 검증
- **안쪽 Timeout 짧게** — SQL(3s) < Tx(5s) < Proxy(10s) < Web(15s)
