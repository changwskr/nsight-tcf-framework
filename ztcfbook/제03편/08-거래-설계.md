## 8.1 ServiceId 설계 · Catalog 등록

신규 온라인 거래 개발은 코드 작성 이전에 **ServiceId 설계**부터 시작한다. ServiceId는 Dispatcher 실행, OM Catalog, 권한, Timeout, 거래로그, 오류코드가 연결되는 중심 식별자이기 때문이다.

ServiceId 설계 절차는 다음과 같다. 첫째, 업무 요건에서 도메인(업무대상)과 행위(처리행위)를 추출한다. 둘째, `{BC}.{Domain}.{action}` 형식으로 ServiceId 후보를 작성한다. 셋째, 기존 Catalog와 중복·유사 여부를 확인한다. 넷째, Handler 클래스명·Facade·Service·Mapper·SQL ID와 1:1 추적 가능한지 검증한다. 다섯째, OM Service Catalog 등록 티켓을 발행한다.

```text
요건: SV 화면에서 고객 요약 정보 조회
  ↓
도메인: Customer / 행위: selectSummary
  ↓
ServiceId: SV.Customer.selectSummary
  ↓
Handler: SvCustomerSummaryHandler
  ↓
OM Catalog 등록 (Timeout, 권한, 거래유형 포함)
```

OM Service Catalog(또는 TCF_SERVICE_MASTER) 등록 항목은 ServiceId, serviceName, businessCode, processingType(INQUIRY/CREATE/UPDATE/DELETE), timeoutMs, useYn, 등록일시이다. 미등록 ServiceId는 STF 또는 AuthorizationValidator에서 차단된다.

등록 절차: 업무 개발자가 설계서에 ServiceId 정의 → OM 담당자가 Catalog에 등록 → 개발자가 Handler에 `serviceIds()` 구현 → 통합 테스트 → 운영 반영. 개발 환경에서 Catalog 검증을 비활성화할 수 있으나, 통합·운영 환경에서는 반드시 활성화된다.

Catalog 등록 시 `processingType`은 INQUIRY·CREATE·UPDATE·DELETE 중 하나이다. STF·권한·감사 정책이 processingType에 연동된다. 조회 거래에 CREATE를 잘못 지정하면 불필요한 감사로그·트랜잭션 오버헤드가 발생한다.

## 8.2 거래코드 설계

거래코드(transactionCode)는 거래로그·감사로그·재처리·통계에서 거래를 식별한다. ServiceId 설계와 동시에 거래코드를 할당한다.

설계 절차: 업무 유형(조회/등록/변경/삭제/배치)을 결정 → `{BC}-{TYPE}-{NNNN}` 형식으로 발급 → 설계서·화면 명세·OM 거래통제에 동시 기재한다.

| 유형 코드 | 의미 | 예시 |
| --- | --- | --- |
| INQ | 조회 | SV-INQ-0001 |
| REG | 등록 | SV-REG-0001 |
| UPD | 변경 | SV-UPD-0001 |
| DEL | 삭제 | SV-DEL-0001 |
| BAT | 배치 | OM-BAT-0001 |

거래코드는 Header에 클라이언트(화면·WebTop)가 설정하여 전송한다. 서버가 임의 생성하지 않는다. 화면 설계서에 "이 버튼 클릭 시 transactionCode=SV-INQ-0001"을 명시한다. 동일 화면에서 조회와 엑셀 다운로드가 다르면 거래코드도 분리한다.

일련번호는 업무별로 독립 관리한다. SV-INQ-0001과 IC-INQ-0001은 공존 가능하다. 신규 발급 시 OM 담당자에게 다음 번호를 요청하여 충돌을 방지한다.

---

## 8.3 Header 7항 · 거래통제 설계

Header 7항은 거래통제(Transaction Control) Allow-List의 핵심 필드이다. STF가 요청 Header를 검증할 때 필수 항목 존재·형식·업무 정합성을 확인한다.

Header 7항(핵심): `businessCode`, `serviceId`, `transactionCode`, `channelId`, `userId`, `guid`(또는 서버 생성), `traceId`(또는 서버 생성)이다. 추가로 `serviceName`, `branchId`, `clientIp`, `requestDateTime`이 설계서에 정의된다.

거래통제 설계는 다음을 포함한다. 허용 채널 목록(channelId Allow-List), 허용 사용자·역할, businessCode와 serviceId prefix 정합성, transactionCode와 ServiceId 매핑, IP 제한(필요 시), 거래 시간대 제한(필요 시)이다.

```text
거래통제 검증 흐름 (STF)
─────────────────────────
1. businessCode = SV ?
2. serviceId starts with SV. ?
3. transactionCode matches SV-INQ-* ?
4. channelId in [WEBTOP, TCFUI, API] ?
5. userId has permission for serviceId ?
6. serviceId registered in OM Catalog ?
7. transactionCode registered in 거래통제 ?
```

OM 거래통제 등록 시 ServiceId, transactionCode, channelId, 권한 그룹, 사용 여부를 설정한다. 설계 단계에서 "이 거래는 WEBTOP 채널만 허용" 같은 정책을 문서화하고, OM 등록 시 반영한다.

외부 API 채널(`channelId=EXTAPI`)은 IP Allow-List와 함께 설계한다. 내부 WebTop 채널과 동일한 ServiceId를 공유할 수 있으나, 거래코드를 분리하면 채널별 통계가 용이하다.

## 8.4 Timeout 정책 설계

Timeout은 장시간 거래로 인한 Thread·Connection Pool 고갈을 방지한다. 설계 단계에서 거래별 Timeout을 정의하고 OM Catalog에 등록한다.

Timeout 계층은 세 가지이다. **Online Timeout**은 HTTP 요청 전체(STF~ETF) 제한 시간이다. 기본 30초, 조회 10초, 복합 60초 등 거래 유형별로 다르다. **Service Timeout**은 OM Catalog의 `timeoutMs`로 ServiceId별 설정한다. **Query Timeout**은 MyBatis SQL 실행 제한이다. `defaultStatementTimeout` 또는 SQL별 설정이다.

| 거래 유형 | Online Timeout | Query Timeout | 비고 |
| --- | --- | --- | --- |
| 단순 조회 | 10초 | 5초 | 인덱스 필수 |
| 목록·페이징 | 15초 | 10초 | COUNT 분리 |
| 등록·변경 | 30초 | 10초 | 트랜잭션 포함 |
| 외부 EAI 연동 | 60초 | — | EAI Client 별도 Timeout |
| 배치성 온라인 | 120초 | 30초 | 사전 승인 필요 |

설계서에 Timeout 초과 시 사용자 메시지(`E-COM-TIMEOUT`)와 운영 대응(로그 확인 항목)을 명시한다. Timeout은 운영 환경에서 OM으로 변경 가능하나, 개발 시 Catalog 등록값과 일치시켜야 한다.

EAI 연동이 포함된 거래는 **내부 처리 시간 + EAI 왕복 시간**을 합산하여 Online Timeout을 산정한다. 내부 10초 + EAI 30초이면 최소 45초 이상 버퍼를 둔다.

## 8.5 화면–ServiceId–거래코드 정합성

화면 설계, ServiceId, 거래코드는 **삼각 정합**을 유지해야 한다. 하나라도 어긋나면 화면에서 호출한 거래와 서버 로그·권한·감사가 불일치한다.

설계서 템플릿 예시:

| 화면번호 | 기능 | functionCode | ServiceId | 거래코드 | 비고 |
| --- | --- | --- | --- | --- | --- |
| SV0101 | 고객요약 조회 | SV0101_INQ | SV.Customer.selectSummary | SV-INQ-0001 | |
| SV0101 | 엑셀 다운로드 | SV0101_EXCEL | SV.Customer.exportSummary | SV-INQ-0002 | 파일 UD |
| SV0102 | 고객상세 조회 | SV0102_INQ | SV.Customer.selectDetail | SV-INQ-0003 | |

정합성 검증 체크리스트:

```text
□ 화면 JS/프론트의 serviceId = 설계서 ServiceId
□ 화면 JS의 transactionCode = 설계서 거래코드
□ businessCode = 담당 WAR BC
□ OM Catalog에 ServiceId 등록됨
□ OM 거래통제에 transactionCode 등록됨
□ menuId·functionCode 권한 매핑 완료
□ Handler.serviceIds()에 ServiceId 포함
```

개발 완료 후 tcf-ui로 표준 전문을 직접 호출하여 Header 값이 설계서와 일치하는지 검증한다. WebTop 연동 전에 tcf-ui 검증을 통과하는 것이 품질 게이트이다.

설계 리뷰 회의 체크 질문: (1) ServiceId Catalog 중복 없음? (2) 거래코드 OM 등록? (3) Header 7항·채널·권한 정의? (4) Timeout·processingType 적절? (5) 화면 functionCode 매핑 완료? 확인 후 제9~10장 구현으로 진행한다.

거래 설계 산출물 템플릿: ServiceId 정의서, 거래코드 할당표, Header·Body 스키마(JSON), 화면–거래 매핑표, OM 등록 체크리스트. 이 다섯 문서가 갖춰지면 구현 착수 승인을 받는다.

---

## 장 요약

거래 설계 단계에서 ServiceId·거래코드·Header 7항·Timeout·화면 정합성을 먼저 정의하고 OM에 등록한 뒤 코드를 작성한다. ServiceId는 Catalog 등록 후에만 운영 실행이 가능하며, 거래코드는 로그·감사 추적의 기준이다. 화면 functionCode별 ServiceId·거래코드 매핑을 설계서에 명시하고 개발·테스트·OM 등록 전 구간에서 일치를 검증한다.
