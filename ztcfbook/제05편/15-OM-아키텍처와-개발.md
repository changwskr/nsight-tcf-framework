## 15.1 tcf-om vs om-service (레거시)

OM(Operations Management)은 NSIGHT TCF의 **운영 기준정보 원장(System of Record for Operations)**이다. 사용자·권한·메뉴, ServiceId Catalog, 거래통제, Timeout, 공통코드·오류코드, 세션 레지스트리, 거래·감사 로그 조회, Cache·Batch·Deploy·Dashboard 등 플랫폼 운영에 필요한 마스터 데이터와 관리 API를 제공한다. 업무 도메인 로직(고객·상품·계약 처리)은 OM이 담당하지 않는다.

현행 표준 모듈은 **`tcf-om`**이다. 업무코드 `OM`, 포트 8097(bootRun), Context `/om`, WAR `om.war`/`tcf-om.war`이다. Gradle 멀티 모듈·CI/CD·ztomcat 배포 파이프라인에 포함되어 있으며, OM Admin UI(`tcf-ui`/`tcf-uj`)가 Relay로 `/om/online`을 호출한다.

**`om-service`는 레거시**이다. Gradle·CI에 포함되지 않으며 샘플·참고용으로만 존재한다. 신규 개발·운영·배포는 반드시 tcf-om 기준으로 진행한다. 설계서에 83 Handler로 기술된 OM 영역은 코드베이스에서 **24 Handler(도메인 통합)**로 구현되어 있으며, 도메인당 1 Handler + `serviceIds()` + switch 패턴을 따른다.

OMDB는 로컬 H2 `./data/nsight-txlog/nsight_om`에 위치한다. SESSIONDB(`SPRING_SESSION`)와 LOGDB(`TCF_TX_LOG` 등)와 논리적으로 분리되나 로컬에서는 동일 H2 파일군을 공유하는 경우가 많다. 운영 Oracle에서는 OMDB·SESSIONDB·LOGDB 스키마 분리가 원칙이다.

tcf-om은 업무 WAR와 동일하게 TCF 6계층(Handler → Facade → Service → Rule → DAO → Mapper)을 따른다. 차이점은 도메인이 "운영·기준정보"이며, RDW 업무 SQL 대신 OMDB·SESSIONDB·LOGDB 조회 SQL을 사용한다는 점이다. Gateway Route·JWT 발급은 tcf-om이 **원천 인증**만 담당하고, 토큰 서명은 tcf-jwt가 수행한다.

| 구분 | tcf-om | om-service (레거시) |
| --- | --- | --- |
| Gradle 포함 | O | X |
| CI/CD bootWar | O | X |
| Handler 수 | 24 (도메인 통합) | 구조 상이 |
| OM Admin UI 연동 | O | 미사용 |
| SESSIONDB | 공유 | - |

## 15.2 OM Handler·화면·serviceId

OM 거래는 업무 WAR와 동일하게 `POST /om/online` + TCF 파이프라인(STF → Dispatcher → Handler → Facade → Service → Rule → DAO)으로 처리된다. `header.serviceId` 형식은 `OM.{Domain}.{action}`이다. 예: `OM.Auth.login`, `OM.ServiceCatalog.inquiry`, `OM.TransactionControl.save`.

24 Handler 목록(요약):

| Handler | serviceId prefix | 영역 |
| --- | --- | --- |
| OmAuthHandler | OM.Auth.* | login, logout, session, ssoLogin |
| OmUserHandler | OM.User.* | 사용자 CRUD |
| OmAuthGroupHandler | OM.AuthGroup.* | 권한그룹 |
| OmMenuHandler | OM.Menu.* | 메뉴 |
| OmFunctionAuthHandler | OM.FunctionAuth.* | 기능권한 |
| OmServiceCatalogHandler | OM.ServiceCatalog.* | ServiceId Catalog |
| OmTransactionControlHandler | OM.TransactionControl.* | 거래통제 |
| OmTimeoutPolicyHandler | OM.TimeoutPolicy.* | Timeout |
| OmSessionHandler | OM.Session.* | 세션 관리 |
| OmBatchHandler | OM.Batch.* | 배치 Job |
| OmDashboardHandler | OM.Dashboard.* | 대시보드 |
| … | … | Cache, Deploy, File, Health 등 |

OM Admin UI는 `tcf-ui`(8099)에서 `/om/admin/*.html`로 제공된다. 공통 JS `om-admin.js`의 `relayFetch()`, `callViaGateway()`가 Cookie 또는 Bearer Relay를 수행한다. 테스트 계정 예: `admin01` / `nsight01!`. 화면별 serviceId는 Catalog에 등록되어 있어야 하며, 메뉴·기능권한과 연동된다.

신규 OM 기능 개발 절차: serviceId 설계 → Handler/Facade/Service/DAO 구현 → Catalog·TC·Timeout seed → HTML·JS 화면 → 기능권한·메뉴 등록 → Relay·Gateway 경로 테스트.

Handler 내부는 `doHandle`에서 action 분기한다. 1 Handler = N serviceId 패턴이 표준이며, Facade에 `@Transactional`을 두어 OMDB 변경과 감사로그 기록을 한 트랜잭션으로 묶는다. OM 거래도 STF 거래통제·권한 검증을 거치므로, `OM.Auth.login` 등 면제 목록을 제외한 Admin 기능은 기능권한(`OM_FUNCTION_AUTH`)이 필요하다.

```text
tcf-ui /om/admin/users.html
  │ relayFetch → POST /api/relay/om/online
  ▼
tcf-om OnlineTransactionController
  ▼ TCF.process()
  OmUserHandler → OmUserFacade → OmUserService → OmUserDao
```

## 15.3 Service Catalog · 거래통제 · Timeout

`OM_SERVICE_CATALOG`는 업무·OM 모든 serviceId의 **마스터 레지스트리**이다. STF는 Catalog에 없는 serviceId를 거부하거나 경고하며, 운영 환경에서는 미등록=실행 불가 정책을 적용한다. Catalog Row는 `SERVICE_ID`, `BUSINESS_CODE`, `HANDLER_CLASS`, `SERVICE_NAME`, `USE_YN`을 포함한다.

거래통제(`TCF_TRANSACTION_CONTROL`)와 Timeout(`TCF_SERVICE_TIMEOUT_POLICY`)은 Catalog와 **별도 테이블**이지만 등록 절차는 한 세트로 진행한다. 신규 업무 serviceId 오픈 체크리스트: ① Catalog INSERT ② Header 7 TC Row ③ Timeout Policy ④ OM_ERROR_CODE(필요 시) ⑤ 기능·데이터 권한.

Seed 데이터는 `tcf-om/src/main/resources/data.sql`과 `ServiceCatalogSeedData.java`, `TransactionControlSeedData.java`로 초기화된다. 로컬 bootRun 시 자동 적용되며, 운영은 OM Admin 또는 migration 스크립트로 반영한다. `USE_YN=N`으로 Catalog를 내리면 해당 serviceId는 운영에서 닫힌다.

STF·Gateway·업무 WAR는 OMDB(또는 cache)를 조회하므로, OM 변경 후 **cache Evict**(`OM.Cache.*` 또는 tcf-cache API)를 잊지 않는다. Catalog·TC·Timeout 변경은 즉시 또는 TTL 이내 전파되어야 하며, 긴급 차단은 BLOCK_Y TC Row + cache Evict 조합이 가장 빠르다.

OM이 SoT인 기준정보와 소비자 관계:

| 테이블 | 소비 모듈 | 갱신 경로 |
| --- | --- | --- |
| OM_SERVICE_CATALOG | 모든 WAR STF | OM Admin / seed |
| TCF_TRANSACTION_CONTROL | STF | OM.TransactionControl.* |
| TCF_SERVICE_TIMEOUT_POLICY | STF TimeoutExecutor | OM.TimeoutPolicy.* |
| OM_COMMON_CODE | UI, Header, SQL | OM.CommonCode.* |

업무 WAR 개발자는 Catalog 등록을 OM 팀·플랫폼 MR과 함께 진행한다. Handler `serviceIds()` 추가만 하고 Catalog 미등록 시 운영 smoke에서 `E-COM-SERVICE_NOT_FOUND` 또는 Catalog 경고가 발생한다.

## 15.4 환경설정 조회·배포관리

`OM.SystemConfig.*`는 환경별 설정 키·값을 OMDB에 등록·조회한다. application.yml에 하드코딩된 운영 값 대신 OM에서 조회해 Feature Flag·한도·URL을 바꾸는 패턴을 지원한다. 다만 보안 Secret(DB password, JWT private key)은 OM에 평문 저장하지 않고 외부 Secret Manager·tcf-cicd profile을 사용한다.

배포관리(`OM.Deploy.*`)는 CI/CD와 OM을 연결한다. OM은 배포 **요청·승인·이력·상태 조회·롤백 지시**를 담당하고, CI/CD는 **빌드·WAR 생성·Tomcat 반영·Health Check·Rollback 실행**을 담당한다. `tcf-batch`의 `DeployStatusCollectService`가 각 WAR 버전·기동 상태를 수집해 `OM_DEPLOY_STATUS`에 적재하고, OM Dashboard가 이를 표시한다.

배포 이력은 Git Tag·Artifact 버전·WAR 파일명·배포 시각·실행자·Health Check 결과를 한 Row로 추적 가능해야 한다. 운영 전환 체크리스트(부록 J)와 연계해 "누가 어떤 버전을 언제 운영에 올렸는가"를 OM에서 조회할 수 있어야 감사 요구를 충족한다.

배포관리 흐름:

```text
개발자 MR merge → CI bootWar → Artifact 저장
  → OM.Deploy.request (선택) → 승인
  → deploy script → Tomcat WAR 교체
  → Health Check → OM.Deploy.complete
  → tcf-batch BAT-BATCH-004 → OM_DEPLOY_STATUS
```

환경설정 조회는 런타임 Feature Toggle에 사용한다. 예: 특정 채널 maintenance 모드, 조회 한도, 외부 URL. 변경 시 cache Evict 없이 즉시 반영되는 키와, 재기동이 필요한 키를 문서화해 운영 혼선을 방지한다.

## 15.5 운영 대시보드·헬스체크

OM Dashboard(`OM.Dashboard.*`)는 `tcf-batch`가 수집한 스냅샷 테이블을 **조회만**한다. `OM_AP_STATUS`(AP health·CPU·Heap·Thread), `OM_DB_STATUS`(DB·Pool), `OM_SESSION_STATUS`(세션), `OM_DEPLOY_STATUS`(배포)가 주요 소스이다. 배치 Job ID: `BAT-BATCH-001`~`004`.

헬스체크(`OM.HealthCheck.*`)는 개별 WAR Actuator `/actuator/health` 또는 커스텀 probe 결과를 OM에서 통합 조회한다. 장애 시 개발자 확인 항목(매뉴얼 69장)과 guid 상관 로그(`TCF_TX_LOG`, `TCF_GATEWAY_TX_LOG`)를 함께 본다.

관측성 원칙: 동일 `guid`로 Gateway log ↔ 업무 TX log ↔ Step log를 연결한다. MDC traceId·userId·serviceId·jti(JWT)가 로그에 남도록 STF·ETF 설정을 유지한다. Prometheus·Actuator metric은 tcf-batch AP 수집 Job과 중복되지 않게 역할을 나눈다.

OM 운영자는 Dashboard에서 이상 징후(AP DOWN, SESSION WARN, DB Pool 고갈)를 확인한 뒤 Session 강제 종료, Cache Evict, Batch 수동 실행(`/batch/jobs/*/run`), Deploy 롤백 지시로 대응한다. 플랫폼 개발자는 Handler·수집 Client·Mapper를 확장해 새 지표를 추가한다.

Dashboard 화면은 `/om/admin/dashboard.html`. AP·DB·Session·Deploy 탭별로 최근 스냅샷 시각(`COLLECTED_AT`)을 표시하며, batch 미기동 시 빈 화면이므로 `OM_BATCH_HISTORY`를 먼저 확인한다.

| 지표 | WARN 기준(예) | 1차 조치 |
| --- | --- | --- |
| AP health DOWN | Actuator DOWN | WAR 재기동·롤백 |
| DB pool active/max > 90% | HikariCP | connection leak 점검 |
| Session uniqueUser 급증 | OM_SESSION_STATUS | 세션 정책·공격 탐지 |
| Deploy version drift | Artifact ≠ 수집값 | 재배포·batch run |

---

## 장 요약

tcf-om은 운영 기준정보 원장으로 24 Handler·OMDB·OM Admin UI로 Catalog·통제·Timeout·권한·세션·로그·배포·대시보드를 관리한다. om-service는 레거시이며 신규는 tcf-om만 사용한다. OM 변경은 seed·Admin·cache Evict·STF 조회로 런타임에 반영되며, tcf-batch 수집 + OM Dashboard로 운영 가시성을 확보한다.
