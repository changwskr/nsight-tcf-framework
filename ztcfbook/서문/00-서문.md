## 0.1 이 책의 범위와 읽는 법

**NSIGHT TCF Framework — 완전 개발 가이드**(`ztcfbook/`)는 NSIGHT 마케팅 플랫폼의 TCF 기반 개발 표준을 **한 권의 책 형태**로 재구성한 원고이다. 단순히 기존 Markdown·Word 문서를 링크 모음으로 나열한 것이 아니라, 개발자가 ServiceId 하나를 신규 등록할 때부터 OM Catalog 등록, Handler·6계층 구현, SQL 작성, 테스트, 배포·운영 전환까지 **끊김 없이 따라갈 수 있는 내러티브**를 목표로 한다.

본 책은 REST Resource 중심 Spring MVC 튜토리얼이 아니다. NSIGHT TCF의 핵심 계약은 다음과 같다.

```text
POST /{businessCode}/online
  → StandardRequest (header + body)
  → TCF.process() : STF → Dispatcher → Handler → ETF
  → StandardResponse (header + result + body)
```

실제 업무 실행은 URL이 아니라 Header의 `serviceId`로 결정된다. Handler는 Facade만 호출하고, 세션·권한·거래통제·Timeout·거래로그는 TCF 공통 파이프라인이 담당한다. 이 원칙을 전제로 32장 본문과 14개 부록이 구성되어 있다.

| 구분 | 장·부록 | 역할 |
| --- | --- | --- |

| **제1~4편** | 1~14장 | TCF 이해, 표준·명명, 거래 개발, 보안·통제 |
| **제5~7편** | 15~21장 | OM·플랫폼, 환경·배포, 테스트 |

예상 분량은 약 730~800페이지(인쇄 기준)이며, 본문 각 장은 절별 설명·장 요약·이전·다음 네비게이션·**말미 출처 색인** 형식을 따른다. 본문 중간에 원본 문서 링크를 끼워 넣지 않고, 근거 자료는 장·부록 끝의 출처 색인에서 일괄 확인한다.

### 0.1.3 역할별 권장 읽기 경로

처음 NSIGHT TCF에 접하는 경우 **서문 → 제1~4편 → 제8편 실습 → 부록 A~G** 순서를 권장한다. 이미 Spring Boot 경험이 있어도 REST Controller 패턴을 그대로 가져오지 말고, 제1장·제10장(Handler)부터 읽어 TCF 계약을 먼저 익히는 것이 안전하다.

각 장 파일 상단의 **상태**가 `집필 완료`이면 본문이 작성된 장이다. `00-목차.md`는 전체 색인·역할별 경로·원본 자료 맵을 담은 **목차 정의서**이며, 장별 상세 절 목록과 원본 출처 매핑을 확인할 때 사용한다.

### 0.1.4 비적용 범위

본 책은 **애플리케이션 개발·TCF 거래 실행·운영 연계**에 초점을 맞춘다. Apache·Tomcat·L4·GSLB 물리 구성, RDW/ADW DBA 운영, GitLab Runner 설치, OS·네트워크 보안 Agent 설정 등 **인프라·시스템 운영**은 별도 운영 매뉴얼·인프라 설계서를 따른다. 다만 개발자는 Timeout, Connection Pool, Health Check, WAR 배포 단위 등을 설계할 때 운영 기준을 위반하지 않도록 본 책의 제6편·부록 G·J를 참고해야 한다.

## 0.2 문서 계층과 역할

NSIGHT TCF 관련 문서는 역할과 상세 수준이 다르다. `ztcfbook`은 이들을 **통합·재편집한 학습용 책**이며, 원본 저장소의 문서는 **근거 자료·상세 레퍼런스**로 유지된다. 혼동을 줄이기 위해 계층을 다음과 같이 이해하면 된다.

```text
[설계·의사결정]
  zman/              Word 설계서(1~25장) 장별 요약 — "설계서는 무엇을 말하는가"
  zarchitecture/     영역별 아키텍처(16편) — "시스템은 어떻게 구성되는가"

[구현·운영 상세]
  docs/architecture/ 기술 정의서(53편) — 클래스·설정·DB·API 상세
  zdoc/              코딩 규칙·샘플·운영 메모

[개발 실무]
  znsight-man/       개발 매뉴얼(78장 + 부록 10 + 명명 21) — 절차·체크리스트·Word 동기
  zguide/            모듈별 개발 가이드(22종) — "이 모듈은 어떻게 만드는가"

[통합 원고]
  ztcfbook/          본 책 — 위 자료를 편·장·부록으로 재구성한 완전 개발 가이드
```

### 0.2.1 계층별 활용

| 계층 | 경로 | 주요 독자 | 특징 |
| --- | --- | --- | --- |
| 설계서 요약 | `zman/` | 아키텍트, 리뷰어 | Word 설계서 25장 요약, Gap·대조표(00) |
| 영역 아키텍처 | `zarchitecture/` | 아키텍트, 플랫폼 | GSLB→Gateway→WAR→TCF, OM·Gateway·UI·배치 |

### 0.2.2 ztcfbook과 znsight-man의 관계

`znsight-man`은 78장 본문 + 부록 10 + 명명규칙 21주제로 **개발 Manual 전체**를 담는다. `ztcfbook`은 동일 주제를 **10편 32장 + 부록 14**로 재배치하여, 개념(제1~4편) → 표준(제2~3편) → 실무(제3~8편) → 레퍼런스(제9~10편·부록) 순으로 읽도록 설계했다. 내용이 충돌할 때 우선순위는 다음과 같다.

1. **동작·클래스·serviceId** — `develop` 브랜치 **코드·OM Catalog·`zman/00-대조표`**
2. **구현·설정 상세** — `docs/architecture`
3. **절차·체크리스트** — `znsight-man`
4. **설계 의도·원칙** — Word 설계서·`zman` 장별 요약
5. **학습용 통합 설명** — `ztcfbook` (위 자료를 재편집한 결과)

본 책의 출처 색인은 각 장·부록 말미에 원본 경로를 적어 두었으므로, 더 깊은 절차나 DDL·소스 위치가 필요하면 색인 링크로 이동하면 된다.

### 0.2.3 Word 원본과 Markdown

공식 Word 매뉴얼은 `znsight-guide-word/`에 있다. Markdown(`znsight-man`, `ztcfbook`)은 Word와 동기화·검색·PR 리뷰를 위해 유지하며, 명명규칙 21주제 등은 Word docx에서 추출·재생성 스크립트(`znsight-man/_extract-docx.cjs` 등)로 갱신할 수 있다. **현장 적용 시** 부록 A~G 표준표·템플릿을 복사해 팀 Wiki나 프로젝트 템플릿으로 사용하는 것을 권장한다.

## 0.3 설계서 vs 코드베이스 차이

NSIGHT Framework **Word 설계서**(docx)와 **`develop` 브랜치 코드**는 대체로 같은 아키텍처를 공유하지만, 구현 과정에서 **Handler 구조·OM Handler 수·모듈 명칭** 등에서 차이가 있다. 본 책과 `zman`은 설계 **원칙**은 설계서를 따르고, **클래스·serviceId·포트**는 코드·Catalog를 기준으로 기술한다. 상세 대조표는 `zman/00-설계서-코드베이스-대조표.md`에 있으며, 서문에서는 핵심만 정리한다.

### 0.3.1 일치하는 핵심

| 주제 | 설계 | 코드 (`develop`) |
| --- | --- | --- |
| 표준 전문 | StandardRequest/Response | `tcf-core/message/*` |
| Online Endpoint | `POST /{businessCode}/online` | `OnlineTransactionController` |
| 처리 파이프라인 | STF → Dispatcher → ETF | `tcf-core` processor |
| 6계층 | Handler → Facade → … → Mapper | `*-service`, `tcf-om` 패키지 |
| OM 본체 | 운영관리 WAR | **`tcf-om`** (8097, `/om`) — `om-service`는 레거시 |
| Gateway | 업무코드 라우팅 | `tcf-gateway` (8100) |
| JWT | SSO·토큰 | `tcf-jwt` (8110) |
| WAR 간 연동 | HTTP 원칙 | **`tcf-eai`**, `TcfServiceClient` |
| Cache | 공통 Cache | `tcf-cache`, `TcfCacheSupport` |

### 0.3.2 주의할 차이 — Handler

설계서 초안은 **serviceId당 Handler 클래스 1개**(`SvCustomerSummaryInquiryHandler` 등)를 가정하는 경우가 많다. 현재 코드베이스는 **도메인(업무 객체)당 Handler 1개**가 여러 `serviceId`를 `serviceIds()`로 등록하고, `doHandle` 내부에서 분기하는 패턴을 사용한다.

| 구분 | Word 설계서 경향 | `develop` 코드 |
| --- | --- | --- |
| Handler 단위 | serviceId 1 : Handler 1 | **도메인 1 : Handler 1** |
| 등록 API | `serviceId()` 단일 | **`serviceIds()`** 목록 |
| tcf-om Handler 수 | 83개 수준 가정 | **약 24 Handler** (기능 통합) |
| Registry | Catalog HANDLER_CLASS | Spring Bean Registry + Catalog |

```java
// 코드 패턴 예 (sv-service)
@Component
public class SvCustomerHandler implements TransactionHandler {
  @Override
  public List<String> serviceIds() {
    return List.of("SV.Customer.selectSummary", "SV.Customer.selectDetail");
  }
  @Override
  public Object doHandle(StandardRequest request, TransactionContext ctx) {
    return switch (request.getHeader().getServiceId()) {
      case "SV.Customer.selectSummary" -> facade.selectSummary(request);
      case "SV.Customer.selectDetail"   -> facade.selectDetail(request);
      default -> throw new BusinessException("E-TCF-DSP-0001", "...");
    };
  }
}
```

신규 개발 시 **설계서의 클래스 이름 예시를 그대로 복사하지 말고**, 팀 표준 Handler 패턴(도메인 통합 vs serviceId 단독)을 정한 뒤 OM Catalog의 `HANDLER_CLASS`·`SERVICE_ID`와 일치시킨다. `SV.Customer.selectSummary`처럼 대표 serviceId는 설계·코드·Catalog **삼자가 일치**하는지 반드시 확인한다.

### 0.3.3 주의할 차이 — OM·EAI·WAR

| 주제 | 설계서·초안 | 코드베이스 |
| --- | --- | --- |
| OM 모듈 | `om-service` 언급 | **`tcf-om`**이 운영 OM 본체 |
| 업무 WAR | 17 WAR 목표 | **9 WAR 배포** (ic, pc, ms, sv, pd, eb, ep, ss, mg) + 플랫폼 |
| WAR 간 호출 | HTTP 직접 호출 원칙 | **`tcf-eai`** 모듈로 표준화 (ic↔sv 등) |
| ztomcat Context | 단일 포트 다중 WAR | Gateway **`/gw`**, JWT **`/jwt`** Context 주의 |

Gap·향후 과제(17 WAR 확대, CI/CD Pipeline, Session 규모, IdP SSO 등)는 **제32장**과 `zman` Gap 절에서 다룬다. 본 책을 읽을 때 "설계서에만 있고 코드에 없는 것"과 "코드에만 있는 것"을 구분하려면 **제31장(공식 설계안 매핑)·제32장(Gap)**을 함께 보면 된다.

### 0.3.4 문서 갱신 원칙

`zman` 갱신 시 권장 원칙은 다음과 같다.

1. **원칙·흐름** — Word 설계서·아키텍처 결정 유지  
2. **클래스·serviceId·포트** — **코드·`OM_SERVICE_CATALOG`·seed `data.sql`**

4. **학습용 통합본** — `ztcfbook`은 위 변경을 반영해 장·부록 단위로 재집필

## 0.4 용어·약어 표

본 책 전반에서 반복되는 용어를 정리한다. 명명·표기 상세는 `docs/architecture/53-naming-conventions.md`, 부록 A~C, `znsight-man/명명규칙-00-목차`(21주제)를 참고한다.

### 0.4.1 프레임워크·처리 계층

| 용어 | 영문 | 설명 |
| --- | --- | --- |
| **TCF** | Transaction Control Framework | 거래 실행·통제 프레임워크 전체. Dispatcher·Registry 포함 |
| **STF** | Standard Transaction Framework | **전처리** — Header 검증, GUID, 세션·권한·거래통제, 거래 시작 로그 |
| **ETF** | End Transaction Framework | **후처리** — StandardResponse 조립, 오류 매핑, 거래·감사 로그 |
| **Dispatcher** | TransactionDispatcher | `serviceId` → `TransactionHandler` 라우팅 |
| **Handler** | TransactionHandler | 업무 진입 Adapter. Facade만 호출 |
| **Online Endpoint** | — | `POST /{businessCode}/online` — 업무 WAR 공통 HTTP 진입점 |

### 0.4.2 식별자·전문

| 용어 | 형식·예 | 설명 |
| --- | --- | --- |
| **업무코드** | `SV`, `OM` (대문자 2자) | Context, WAR, Package, ServiceId Prefix의 기준 |
| **businessCode** | Header 필드 | URL Path·Gateway Route와 일치해야 함 |
| **serviceId** | `SV.Customer.selectSummary` | **실행 Handler** 식별. `{BC}.{Domain}.{action}` |
| **transactionCode** | `SV-INQ-0001` | **거래로그·감사·재처리** 식별. `{BC}-{TYPE}-{NNNN}` |
| **processingType** | `INQUIRY`, `CREATE`, … | Header 처리유형. 거래유형과 정합 필요 |
| **StandardRequest** | header + body | TCF 요청 표준 전문 |
| **StandardResponse** | header + result + body | TCF 응답 표준 전문 |
| **GUID** | `GUID-…` | End-to-End 거래 추적 ID |
| **TraceId** | `TRC-…` | 시스템 내부 추적 ID |

### 0.4.3 애플리케이션 6계층

| 계층 | 접미사·역할 |
| --- | --- |
| **Handler** | `*Handler` — serviceId 진입, Facade 호출만 |
| **Facade** | `*Facade` — 유스케이스 조립, **트랜잭션 경계** |
| **Service** | `*Service` — 업무 흐름·조합 |
| **Rule** | `*Rule` — 검증·업무 규칙·재처리 판단 |
| **DAO** | `*Dao` — Mapper 호출, DB 예외 변환 |
| **Mapper** | `*Mapper` + XML — SQL 실행·결과 매핑 |

### 0.4.4 플랫폼·인프라

| 용어 | 설명 |
| --- | --- |
| **OM** | Operation Management — `tcf-om`, Service Catalog·거래통제·권한·배치 Admin |
| **Gateway** | `tcf-gateway` — GSLB/L4 하류, 업무 WAR 라우팅·인증 |
| **tcf-eai** | WAR 간 표준 HTTP 연동 클라이언트 |
| **tcf-ui / tcf-uj** | 브라우저·Relay 채널 (표준 전문 호출) |
| **bootRun** | Gradle로 모듈별 독립 포트 기동 (로컬 개발) |

## 장 요약

`ztcfbook`은 NSIGHT TCF 개발 표준을 32장·14부록으로 통합한 **완전 개발 가이드**이다. REST가 아닌 `POST /{businessCode}/online` + `serviceId` + STF/TCF/ETF 파이프라인을 전제로 하며, `zman`·`zarchitecture`·`docs`·`znsight-man`·`zguide`는 계층별 근거 자료이다. 설계서와 코드의 Handler·OM·EAI 차이는 본 서문 0.3절과 제31~32장에서 정리한다. 역할별 읽기 경로(0.1.3)에 따라 제1편부터 순차적으로 진행하거나, 부록 A~G·H~J를 현장 체크리스트로 활용하면 된다.
