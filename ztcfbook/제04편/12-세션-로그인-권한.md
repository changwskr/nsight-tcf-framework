## 12.1 Spring Session · SESSIONDB

NSIGHT TCF에서 세션은 단일 저장소에 모든 역할을 맡기지 않는다. **인증 세션**, **거래 헤더 세션 검증**, **운영 모니터링 세션 지표**가 목적에 따라 분리되어 설계되어 있으며, 이 세 관점을 함께 이해해야 업무 개발자와 OM 운영자가 같은 용어로 대화할 수 있다.

인증 세션의 핵심 저장소는 **SESSIONDB**에 위치한 `SPRING_SESSION` 테이블이다. `tcf-om`은 Spring Session JDBC를 사용하며, `spring.session.store-type: jdbc`, `spring.session.jdbc.table-name: SPRING_SESSION`, 세션 만료 60분을 기본으로 한다. 브라우저에는 `JSESSIONID` 쿠키가 발급되며 `HttpOnly: true`, `SameSite: Lax`가 적용된다. WAS 메모리에 세션을 두지 않고 JDBC에 영속화하기 때문에 Tomcat 인스턴스를 교체하거나 bootRun과 WAR 배포를 전환해도 동일 SESSIONDB를 바라보면 로그인 상태가 유지된다.

거래 헤더 세션 검증은 `tcf-core` STF 단계의 `SessionValidator`가 담당한다. `nsight.tcf.session-validation-enabled`가 `true`일 때 `header.userId`가 비어 있으면 `E-COM-AUTH-0001`(`SESSION_INVALID`)로 거래가 차단된다. 이 검증은 SPRING_SESSION을 직접 조회하는 방식이 아니라 **헤더 기반 최소 검증**이며, Gateway나 UI Relay가 이미 세션을 확인한 뒤 헤더에 사용자 정보를 실어 보낸다는 전제 위에 동작한다.

운영 모니터링 관점에서는 `tcf-batch`가 `SPRING_SESSION`과 각 WAR의 Actuator 세션 metric을 주기적으로 수집해 `OM_SESSION_STATUS`에 적재한다. OM Admin 대시보드는 이 스냅샷을 조회만 하며, 배치가 직접 화면을 그리지 않는다. 세션 정리(`OmSessionCleanupService`)는 만료 행을 주기적으로 삭제해 SESSIONDB 누적을 방지한다.

로컬 개발 시 SESSIONDB는 H2 파일 `./data/nsight-txlog/nsight_om`에 위치하며, Gateway의 `session-datasource`도 동일 DB를 참조해야 4단계 세션 검증이 정상 동작한다. 세션 만료 시간과 온라인 거래 Timeout은 별개 정책임을 기억해야 한다. 세션은 "누가 로그인했는가"를, Timeout은 "이 요청을 얼마나 기다릴 것인가"를 결정한다.

`SPRING_SESSION`과 `SPRING_SESSION_ATTRIBUTES` 테이블 구조는 Spring Session JDBC 표준을 따른다. `SESSION_ID`, `CREATION_TIME`, `LAST_ACCESS_TIME`, `MAX_INACTIVE_INTERVAL`, `EXPIRY_TIME`, `PRINCIPAL_NAME` 컬럼이 핵심이며, OM 세션 관리 화면(`OM.Session.inquiry`)은 `OmSessionService`가 이 테이블을 직접 조회한다. 강제 종료(`OM.Session.delete`) 시 대상 세션 존재 확인 후 삭제하고, 현재 세션을 지운 경우 `sessionSupport.invalidateSession()`을 호출한다. 실패하면 `E-OM-BIZ-0002`가 반환된다.

전체 세션 흐름을 ASCII로 정리하면 다음과 같다.

```text
Browser ──Cookie/JWT──► tcf-ui / tcf-uj
           │
           ├─ /api/relay/OM/online ──► tcf-om (Cookie)
           └─ /api/gateway/om/online ──► tcf-gateway ──► tcf-om (Bearer)
tcf-om: Spring Session JDBC + OmAuthSessionSupport
tcf-gateway: SESSIONDB 4단계 + TCF_USER_SESSION
업무 WAR STF: SessionValidator (header.userId)
tcf-batch: SPRING_SESSION 집계 → OM_SESSION_STATUS
```

운영 환경에서는 HTTPS와 `secure cookie=true`를 적용하고, `nsight.om.session-cleanup.fixed-rate-ms`(예: 10000ms)로 만료 세션 정리 주기를 점검한다. UI Relay 경로에서 `Cookie`·`Set-Cookie` 누락은 로그인 직후 API 호출 실패로 나타나므로, 브라우저 개발자 도구 Network 탭에서 Relay 응답 헤더를 확인하는 것이 1차 진단 방법이다.

향후 강화 방향으로 STF `SessionValidator`가 SPRING_SESSION 직접 조회형으로 확장될 수 있다. 현행은 헤더 기반이지만, 운영 보안 요구가 높아지면 Gateway 4단계와 STF DB 조회를 병행하는 설계를 검토할 수 있다. 업무 개발자는 이 확장과 무관하게 SessionContext 사용 원칙을 지키면 된다.

## 12.2 로그인·사용자 정보

OM 로그인은 REST Controller가 아니라 **TCF 표준 거래**로 처리된다. 브라우저의 `login.html`에서 `OmAdmin.login(userId, password)`가 호출되면 `POST /api/relay/OM/online`으로 표준 전문이 전달되고, `header.serviceId`는 `OM.Auth.login`, `transactionCode`는 `OM-AUT-0002`이다. 이 요청은 `tcf-ui`의 `TransactionRelayService`를 거쳐 `tcf-om`의 `/om/online`에 도달한다.

`tcf-ui`는 세션을 저장하지 않는 **쿠키 전달자** 역할만 한다. 요청 시 브라우저의 `Cookie` 헤더를 그대로 downstream에 넘기고, 응답의 `Set-Cookie`를 `RelayResult`에 담아 `TcfApiController.applySetCookies()`가 브라우저 응답에 재설정한다. 로그인 성공 후 발급된 `JSESSIONID`가 이 경로를 통해 유지되므로, 이후 `OM.Auth.session`, 메뉴 조회, Catalog 관리 등 모든 OM Admin API가 동일 세션으로 연속 호출된다.

`OmAuthService.login`의 처리 순서는 다음과 같다. 필수값 검증 → 사용자 조회 → `useYn` 확인 → `PasswordEncoder.matches`로 비밀번호 검증 → `lastLoginTime` 갱신 → `sessionSupport.createSession(user)` → `sessionId`, `userId`, `branchId`, `authGroupId` 등을 응답 body에 반환한다. 로그인 실패 시 `E-OM-AUTH-0001`을 반환하며, 계정 존재 여부를 구분하지 않는 통일 메시지로 응답해 정보 노출을 줄인다.

로그아웃(`OM.Auth.logout`)은 `sessionSupport.invalidateSession()`으로 현재 세션을 무효화하고, 세션 조회(`OM.Auth.session`)는 UI가 페이지 진입 시 자동 로그인 여부를 판단하는 데 사용한다. 성공 시 `sessionStorage(nsight.om.session)`에 사용자 요약 정보가 저장되며, 이는 화면 표시용이고 실제 인증 근거는 여전히 `JSESSIONID`와 SESSIONDB이다.

JWT 로그인은 보조 경로로 `tcf-jwt`가 Access/Refresh Token을 발급한다. OM Admin에서 JWT 모드를 사용하면 `om-admin.js`의 `callViaGateway`가 `Authorization: Bearer`로 `/api/gateway/om/online`을 호출한다. 쿠키 세션과 JWT는 공존하며, Gateway는 Bearer가 있으면 JWT, 없으면 세션으로 분기한다. 두 방식 모두 STF에서 Header 7항과 권한·거래통제 검증을 거친다.

로그인 관련 표준 거래 식별자는 OM Admin·업무 테스트 JSON·Catalog seed에서 동일해야 한다.

| serviceId | transactionCode | Handler | 용도 |
| --- | --- | --- | --- |
| OM.Auth.login | OM-AUT-0002 | OmAuthHandler | 로그인 |
| OM.Auth.logout | OM-AUT-0003 | OmAuthHandler | 로그아웃 |
| OM.Auth.session | OM-AUT-0004 | OmAuthHandler | 세션 조회 |

`om-admin.js`가 조립하는 요청 Header 예시는 `businessCode: OM`, `serviceId: OM.Auth.login`, `processingType: EXECUTE`, `channelId: WEBTOP`이다. Body는 `{ "userId": "...", "password": "..." }` 형태다. 성공 응답에서 `payload.body.loggedIn === true`와 `sessionId`를 확인한 뒤 UI 상태를 갱신한다.

로그인 거래도 STF → Dispatcher → Handler → Facade → Service → ETF 파이프라인을 그대로 탄다. 따라서 GUID·TraceId·거래로그가 일반 업무 거래와 동일하게 `TCF_TX_LOG`에 남는다. `nsight.tcf.session-validation-enabled=true` 환경에서 로그인 거래에 `header.userId`를 비워 두면 STF에서 차단될 수 있으므로, 로그인·비로그인 면제 거래 목록을 운영 설정과 seed에서 일치시켜야 한다.

비밀번호는 DB `passwordHash`와 Spring `PasswordEncoder`로만 검증한다. 평문 비밀번호는 로그·응답·세션 attribute 어디에도 남기지 않는다. 로컬 테스트 계정 `admin01` / `nsight01!`은 개발용이며 운영에서는 초기 비밀번호 강제 변경·잠금 정책을 OM User 관리와 연동한다.

로그인 실패·성공 모두 OM_ACCESS_LOG·감사 정책 대상이 될 수 있다. 반복 실패 IP·계정에 대한 잠금은 OM 확장 기능이며, 최소한 거래로그와 감사로그로 추적 가능해야 한다. SSO 연계(`OM.Auth.ssoLogin`)는 제13장 JWT·SSO 절과 연결해 읽는다.

## 12.3 기능권한·메뉴

OM 권한 모델은 **기능권한(Function Auth)**, **데이터권한(Data Auth)**, **메뉴(Menu)** 세 축으로 구성된다. 사용자는 `OM_AUTH_GROUP`에 소속되고, 그룹별로 메뉴 접근과 기능 실행 권한이 매핑된다. OM Admin 화면은 메뉴 트리를 `OM.Menu.*` 거래로 조회하며, 화면 진입 전 `OmAdmin` 공통 JS가 세션·권한 상태를 확인한다.

기능권한은 `OM.FunctionAuth.*` Handler로 CRUD되며, `functionCode` 단위로 "이 사용자가 특정 OM 기능을 실행할 수 있는가"를 판단한다. 데이터권한은 `OM.DataAuth.*`로 지점·조직·업무 범위를 제한한다. 업무 WAR에서 STF의 `AuthorizationValidator`는 `header.userId`와 권한 테이블을 대조해 기능 실행 가능 여부를 검사한다. 권한이 없어도 거래통제에 등록되어 있으면 실행될 수 없고, 권한이 있어도 거래통제 미등록이면 차단될 수 있다는 점이 제14장과 연결된다.

메뉴 구조는 `menuId`, `functionCode`, 화면 URL을 OMDB에 등록하고 seed 또는 OM Admin에서 관리한다. `tcf-ui`의 `/om/admin/` 하위 HTML은 메뉴 권한에 따라 노출·숨김이 결정된다. 신규 OM 화면을 추가할 때는 Handler·serviceId·Catalog·기능권한·메뉴 항목을 함께 등록해야 운영 포털에서 접근 가능하다.

권한 변경 이력은 `OM.AuthHistory.*`로 추적한다. 세션 강제 종료(`OM.Session.delete`)나 권한 그룹 변경 시 감사로그(`OM_AUDIT_LOG`) 기록이 필수이다. 운영 감사 관점에서 "누가, 언제, 어떤 권한으로, 어떤 거래를 실행했는가"는 Header 7항 + 세션 + 권한 테이블 + `TCF_TX_LOG`의 조합으로 재구성할 수 있어야 한다.

OM Admin 주요 화면과 serviceId 매핑은 다음과 같다.

| 화면 | 경로 | serviceId 예 |
| --- | --- | --- |
| 로그인 | /om/admin/login.html | OM.Auth.login |
| 대시보드 | /om/admin/dashboard.html | OM.Dashboard.* |
| ServiceId | /om/admin/service-catalog.html | OM.ServiceCatalog.* |
| 거래로그 | /om/admin/transaction-log.html | OM.TransactionLog.* |
| 세션 | /om/admin/session.html | OM.Session.* |
| Cache | /om/admin/cache.html | OM.Cache.* |

기능권한 `functionCode`는 화면번호·menuId·ServiceId 연결 명명규칙(제6장)과 정합해야 한다. 예를 들어 Service Catalog 저장 기능과 화면 버튼이 다른 functionCode를 가리키면 "화면은 보이나 저장 거래는 권한 없음" 불일치가 발생한다.

데이터권한은 SQL WHERE를 OM에서 대신해 주지 않는다. Rule·Service가 SessionContext의 branchId·authGroupIds를 읽어 DAO 파라미터로 넘기고, Mapper XML에서 `AND BRANCH_ID = #{branchId}` 형태로 적용한다. DAO가 세션을 직접 읽으면 테스트 불가·계층 위반·Connection Pool 혼선이 생긴다.

메뉴는 트리 depth·정렬·사용 여부(`USE_YN`)를 OMDB에서 관리한다. WebTopSuite 대체로 tcf-ui 정적 HTML + om-admin.js가 메뉴 API 결과를 DOM에 렌더링한다. 신규 메뉴 추가 시 HTML 파일·menuId·functionCode·Catalog·기능권한·메뉴 seed 다섯 가지를 한 MR에서 다루는 것이 운영 drift를 줄인다.

권한 그룹 변경이 기존 세션에 즉시 반영되지 않을 수 있다. 설계 선택지는 (1) 권한 변경 시 해당 사용자 세션 강제 종료 (2) STF가 매 요청 OMDB/cache 재조회 (3) TTL 캐시 + 운영 수동 Evict이다. NSIGHT는 cache + OM Session 관리 조합을 기본으로 하며, 고보안 환경에서는 (1)을 추가 적용한다.

## 12.4 세션 사용 기준 (업무 WAR)

업무 개발자의 핵심 원칙은 **HttpSession을 Service·Rule·DAO에서 직접 사용하지 않는다**는 것이다. 세션 검증은 STF.preProcess()에서 수행되고, Handler 이하 계층은 검증이 끝난 `SessionContext` 또는 Header에 실린 `userId`, `branchId`, `channelId`만 참조한다.

| 계층 | 세션 사용 |
| --- | --- |
| Controller / Gateway | Cookie 전달만, 세션 로직 금지 |
| STF | 세션·Header 정합성 검증 |
| Handler | SessionContext 전달 |
| Facade | 등록자·수정자·감사정보 설정 |
| Service / Rule | SessionContext 파라미터 참조 |
| DAO / Mapper | 세션 직접 참조 금지 |

세션에 저장할 수 있는 정보는 사용자 ID, 이름, 지점, 채널, 권한그룹, 로그인 유형 등 **인증·운영 최소 정보**로 한정한다. 고객 조회 결과, 대량 목록, 파일 원본, 비밀번호, JWT 원문은 세션 attribute에 넣지 않는다. Idle Timeout과 Absolute Timeout을 병행 적용하는 것이 설계 기준이며, OM.Session.* 화면에서 활성 세션 조회·강제 종료·중복 로그인 관리가 가능하다.

업무 WAR에서 `nsight.tcf.session-validation-enabled`를 환경별로 명확히 설정해야 한다. 로컬 개발에서는 완화할 수 있으나 dev 이상에서는 Header userId 검증을 켜 두는 것이 권장된다. Gateway 4단계 세션 검증과 STF 검증이 중복되더라도 Gateway는 HTTP 관문, STF는 거래 실행 최종 관문이라는 역할 분리를 유지한다.

`SessionContext`는 업무 로직이 참조하는 표준 DTO이다. `sessionId`, `userId`, `userName`, `branchId`, `branchName`, `channelId`, `loginType`, `authGroupIds`, `roleCodes`, `loginTime` 등을 포함하며 Handler에서 Facade·Service로 전달한다. Facade는 `@Transactional` 경계 안에서 등록자·수정자·감사 필드를 SessionContext에서 채운다.

세션 attribute에 허용·금지 데이터를 다시 정리하면 다음과 같다.

| 구분 | 예 |
| --- | --- |
| 허용 | userId, userName, branchId, channelId, authGroupId, loginType |
| 금지 | password, JWT 원문, 고객 조회 결과, 대량 List, 파일 byte[] |

Cookie 이름은 `JSESSIONID` 또는 환경에 따라 `NSIGHTSID`를 사용할 수 있다. Gateway 4단계 1단계 검증이 Cookie 존재를 확인하므로, API 테스트 도구(Postman)에서 Cookie jar를 유지하지 않으면 연속 호출이 401로 실패한다. curl 사용 시 `-b`/`-c` 옵션으로 Cookie 파일을 지정한다.

업무 WAR bootRun 단독 테스트 시 Gateway·UI Relay 없이 `POST /sv/online`에 Header userId를 수동 기입하면 STF session validation을 통과할 수 있다. 이는 **개발 편의**이며, 통합·staging 테스트에서는 Gateway·세션·JWT 경로를 반드시 재현한다. ztomcat 8080 통합 검증은 제19장에서 다룬다.

ETF는 감사·거래로그에 `userId`, `branchId`, `channelId`를 SessionContext·Header에서 기록한다. DAO까지 세션을 전달하지 말고, ETF가 필요한 값은 이미 Header·MDC에 있다. MDC 키는 명명규칙-21 Header 항목과 일치시킨다.

## 12.5 권한 검증 기준

권한 검증과 거래통제는 질문이 다르다. **권한**은 "이 사용자가 이 기능을 쓸 수 있는가"이고, **거래통제**는 "이 Header 7조합의 거래가 허용 등록되었는가"이다. 권한이 있어도 Catalog·거래통제에 없으면 실행되지 않으며, 반대로 거래통제만 등록하고 기능권한이 없어도 차단된다.

STF.preProcess()에서 권한 검증은 Header 7항 필수값 확인 → Catalog 존재 확인 → `AuthorizationValidator` → `TransactionControlService` 순으로 진행된다. `OM.Auth.login`, `logout`, `session`은 거래통제 검사에서 제외된다. 로그인 전·후 경계 거래는 `GatewayAuthExemptions`와 STF 면제 목록을 함께 관리해야 한다.

실패 시 오류코드는 계층별로 구분된다. 세션 무효는 `E-COM-AUTH-0001`, 권한 부족은 `E-COM-AUTH-0002` 계열, 거래통제 차단은 `E-TCF-CTL-001`이다. HTTP 상태는 대부분 200이고 `result.resultCode`와 `errorCode`로 판단하는 TCF 표준 규약을 따른다. Gateway에서 401이 반환되는 경우는 세션·JWT 1차 관문 실패이며, 업무 WAR까지 도달한 요청은 STF 오류 형식으로 응답한다.

운영 체크리스트: 신규 serviceId 등록 시 Catalog + 거래통제 + Timeout + 기능권한을 동시에 반영했는가, 채널·지점별 통제 시나리오를 테스트했는가, OM 권한 변경 후 기존 세션 사용자에게 즉시 반영되는지(강제 재로그인 정책) 확인했는가. `tcf-cache`가 Catalog·통제 정책을 캐시하므로 OM 변경 후 Evict 정책도 함께 점검한다.

STF.preProcess() 전체 순서는 세션·권한·통제·Timeout·멱등성·거래로그 생성을 포함한다. 세션·권한 단계만 발췌하면 다음과 같다.

```text
STF.preProcess()
  → Header normalize (guid, traceId)
  → SessionValidator (header.userId, 옵션)
  → AuthorizationValidator (기능권한)
  → AuthenticationContextValidator (JWT claim ↔ Header, 옵션)
  → TransactionControlService (Header 7)
  → TimeoutPolicy 조회
  → TCF_TX_LOG 시작
```

오류 응답 Contract: HTTP 200 + `result.resultCode=E0001` + `errorCode` + `errorMessage`. Gateway 401 JSON과 구분해야 한다. 채널 UI는 Gateway 실패와 STF 실패를 다른 UX로 표시하는 것이 좋다(재로그인 vs 거래통제·권한 안내).

| 상황 | 오류코드 | 처리 위치 |
| --- | --- | --- |
| header userId 없음 | E-COM-AUTH-0001 | SessionValidator |
| 기능권한 없음 | E-COM-AUTH-0002 계열 | AuthorizationValidator |
| 로그인 실패 | E-OM-AUTH-0001 | OmAuthService |
| 거래통제 차단 | E-TCF-CTL-001 | TransactionControlService |

Gateway `GatewayAuthExemptions`와 STF 면제 목록은 **동기화**가 필요하다. Gateway에서만 면제하고 STF에서 막히면 "Gateway 통과 후 업무 403/200+error"가 발생한다. 신규 공개 API(헬스·로그인) 추가 시 두 목록을 함께 MR에 포함한다.

침해·권한 오남용 대응: OM.Session.delete로 세션 강제 종료, AuthHistory·AuditLog 확인, 필요 시 JWT jti Denylist(제13장). 장기적으로는 비정상 channelId·branchId 조합 알람을 OM Dashboard 또는 SIEM 연계로 확장한다.

개발 완료 체크(부록 H): OM 로그인·세션조회·로그아웃 정상, UI Relay Cookie 유지, 업무 거래 Header userId·branchId·channelId 실림, STF session-validation 환경별 설정 문서화, 권한 없는 functionCode 호출 차단 테스트 통과.

---

## 장 요약

NSIGHT TCF 세션은 Spring Session JDBC와 SESSIONDB를 중심으로 인증·거래 검증·운영 모니터링이 분리된 3계층 구조이다. OM 로그인은 `OM.Auth.*` TCF 거래로 처리되며, `tcf-ui` Relay가 Cookie를 투명 전달한다. 업무 WAR는 HttpSession을 직접 쓰지 않고 STF에서 검증된 SessionContext만 사용한다. 권한(기능·데이터·메뉴)과 거래통제(Header 7 Allow-List)는 STF에서 순차 검증되며, JWT는 쿠키 세션과 공존하는 보조 인증 경로이다.
