## 7.1 Java Class · Method · Field

Java 클래스·메서드·필드 명명은 NSIGHT TCF 6계층 구조와 ServiceId 체계에 맞춰 일관되게 작성한다.

**클래스 명명**은 계층별 접미사를 사용한다.

| 계층 | 접미사 | 예시 |
| --- | --- | --- |
| Handler | Handler | SvCustomerSummaryHandler |
| Facade | Facade | SvCustomerFacade |
| Service | Service | SvCustomerService |
| Rule | Rule | SvCustomerRule |
| DAO | Dao | SvCustomerDao |
| Mapper | Mapper | SvCustomerMapper |
| Request DTO | Req | SvCustomerSummaryReq |
| Response DTO | Res | SvCustomerSummaryRes |

클래스명은 PascalCase이다. 업무코드 접두사를 붙인다. `Sv` + `Customer` + `Summary` + `Handler` = `SvCustomerSummaryHandler`이다. Application 클래스는 `Nsight{Code}Application`이다. `NsightSvApplication`, `NsightOmApplication` 등이다.

**메서드 명명**은 lowerCamelCase 동사로 시작한다. Handler의 `doHandle`, Facade의 `selectSummary`, Service의 `findCustomerByNo`, Rule의 `validateSummaryRequest`, DAO의 `selectCustomerSummary` 등이다. 조회는 `select*`, `find*`, `get*`, 등록은 `insert*`, `save*`, `create*`, 변경은 `update*`, 삭제는 `delete*`를 사용한다.

**필드 명명**은 lowerCamelCase이다. `customerNo`, `customerName`, `branchId` 등이다. 상수는 `UPPER_SNAKE_CASE`이다. `MAX_PAGE_SIZE`, `DEFAULT_TIMEOUT_MS` 등이다.

```java
@Component
public class SvCustomerSummaryHandler implements TransactionHandler {
  private final SvCustomerFacade customerFacade;
  // ...
}

@Service
public class SvCustomerService {
  private final SvCustomerDao customerDao;
  private static final int MAX_PAGE_SIZE = 100;
}
```

## 7.2 DTO 유형·작성 규칙

DTO(Data Transfer Object)는 계층 간 데이터 전달을 위해 사용한다. NSIGHT TCF는 Request DTO, Response DTO, 내부 Command/Query 객체를 분리한다.

**Request DTO** (`*Req`)는 Handler가 `StandardRequest.body`를 변환하는 입력 객체이다. `entry` 패키지 또는 `entry/dto/request`에 위치한다. Jackson 역직렬화를 위해 기본 생성자·getter/setter 또는 record를 사용한다.

**Response DTO** (`*Res`)는 Facade·Service가 반환하고, Handler가 `StandardResponse.body`에 담는 출력 객체이다. `entry/dto/response`에 위치한다.

**내부 Command/Query**는 Service·DAO 간 내부 전달용이다. 외부 API(표준 전문)에 노출되지 않는다. `application/dto` 또는 `persistence/dto`에 위치할 수 있다.

```java
// Request DTO
public class SvCustomerSummaryReq {
  @NotBlank(message = "고객번호는 필수입니다.")
  private String customerNo;
}

// Response DTO
public class SvCustomerSummaryRes {
  private String customerNo;
  private String customerName;
  private String grade;
}
```

DTO 작성 원칙: 표준 전문 Body 필드명과 DTO 필드명을 일치시킨다. Validation 어노테이션(`@NotBlank`, `@Size`, `@Pattern`)을 Request DTO에 선언한다. Response DTO에 민감 정보를 포함하지 않거나 마스킹한다. DTO 간 불필요한 상속을 피하고, 조합(composition)을 사용한다.

## 7.3 MyBatis Mapper · SQL ID

MyBatis Mapper는 `persistence.mapper` 패키지의 인터페이스와 `resources/mapper/{code}/*.xml`이 쌍을 이룬다.

**Mapper 인터페이스** 명명은 `{Bc}{Domain}Mapper`이다. `SvCustomerMapper`, `OmOperationMapper` 등이다. `@Mapper` 어노테이션을 선언한다.

**SQL ID**는 ServiceId와 동일한 `{BC}.{Domain}.{action}` 형식을 권장한다. `SV.Customer.selectSummary`, `SV.Customer.selectDetail` 등이다. XML의 `id` 속성과 Mapper 인터페이스 메서드명을 일치시킨다.

```xml
<!-- SvCustomerMapper.xml -->
<mapper namespace="com.nh.nsight.marketing.sv.persistence.mapper.SvCustomerMapper">

  <select id="SV.Customer.selectSummary"
          parameterType="SvCustomerSummaryReq"
          resultType="SvCustomerSummaryRes">
    SELECT customer_no, customer_name, grade
    FROM sv_customer
    WHERE customer_no = #{customerNo}
  </select>

</mapper>
```

```java
@Mapper
public interface SvCustomerMapper {
  SvCustomerSummaryRes selectSummary(SvCustomerSummaryReq req);
}
```

Mapper XML 파일명은 `{Bc}{Domain}Mapper.xml`이다. `SvCustomerMapper.xml`을 `mapper/sv/` 디렉터리에 둔다. Namespace는 Mapper 인터페이스 FQCN과 정확히 일치해야 한다. SQL ID 중복은 빌드 시 또는 런타임에 오류를 유발한다.

## 7.4 DB 테이블·컬럼·인덱스

DB 객체 명명은 snake_case를 사용한다. 업무코드 prefix를 테이블명에 포함한다.

**테이블** 명명은 `{bc}_{domain}` 또는 `tcf_{purpose}`이다. `sv_customer`, `sv_product_holding`, `tcf_tx_log`, `om_service_catalog` 등이다. TCF 공통 테이블은 `tcf_` prefix를 사용한다.

**컬럼** 명명은 snake_case이다. `customer_no`, `service_id`, `transaction_code`, `created_at`, `updated_by` 등이다. PK는 `{table_singular}_id` 또는 복합키 컬럼명을 사용한다.

**인덱스** 명명은 `idx_{table}_{columns}`이다. `idx_sv_customer_no`, `idx_tcf_tx_log_service_id` 등이다. Unique 인덱스는 `uk_{table}_{columns}`이다.

| 구분 | 패턴 | 예시 |
| --- | --- | --- |
| 업무 테이블 | {bc}_{domain} | sv_customer |
| TCF 공통 | tcf_{purpose} | tcf_tx_log |
| OM 테이블 | om_{domain} | om_service_catalog |
| PK 컬럼 | {name}_id 또는 업무키 | customer_no |
| 감사 컬럼 | created_at, updated_at, created_by, updated_by | 공통 |

RDW(Real-time Data Warehouse)와 ADW(Analytical Data Warehouse) 접근 경계를 준수한다. 실시간 거래 조회·등록·변경은 RDW, 대량 분석·배치 집계는 ADW를 사용한다. Mapper XML에 RDW/ADW DataSource를 혼용하지 않는다.

## 7.5 오류코드 · 메시지코드

오류코드는 `{범위}-{업무}-{일련번호}` 형식이다. 프레임워크 공통은 `E-COM-*`, 업무 오류는 `E-{BC}-*`이다.

| 범위 | 패턴 | 예시 |
| --- | --- | --- |
| 공통 | E-COM-{NNNN} | E-COM-0001 (HEADER_INVALID) |
| 업무 | E-{BC}-{NNNN} | E-SV-0001 (고객 미존재) |
| 시스템 | E-SYS-{NNNN} | E-SYS-0001 (DB 연결 실패) |

```java
// BusinessException 발생
throw new BusinessException("E-SV-0001", "고객번호를 찾을 수 없습니다.");

// ErrorCode 상수 (tcf-core)
public static final String HEADER_INVALID = "E-COM-0001";
```

메시지코드는 OM 메시지 관리 테이블에 등록된 사용자 표시 메시지 ID이다. `MSG-SV-0001` 등이다. 오류코드와 메시지코드는 1:1 또는 N:1 매핑이 가능하나, 운영 환경에서는 OM에서 메시지 문구를 변경할 수 있도록 메시지코드를 분리한다.

오류코드 발급은 OM 담당자와 협의하여 중복 없이 할당한다. 임의 문자열(`"ERROR"`, `"FAIL"`)을 오류코드로 사용하면 ETF 결과 매핑·운영 모니터링이 불가능하다.

## 7.6 로그·감사로그 항목

로그 항목 명명은 MDC 키, 거래로그 DB 컬럼, 감사로그 필드에서 일관성을 유지한다.

**애플리케이션 로그** (SLF4J)는 MDC를 통해 자동으로 추적 키가 포함된다. 로그 메시지는 한국어 또는 영문으로 명확하게 작성하고, 민감 정보는 마스킹한다.

```java
log.info("고객 요약 조회 완료. customerNo={}", maskCustomerNo(req.getCustomerNo()));
```

**거래로그** 항목은 STF·ETF가 자동 기록한다. `guid`, `trace_id`, `service_id`, `transaction_code`, `business_code`, `user_id`, `channel_id`, `start_time`, `end_time`, `result_code`, `elapsed_ms` 등이다.

**감사로그** 항목은 등록·변경·삭제 거래에서 추가 기록한다. `audit_type`(CREATE/UPDATE/DELETE), `target_table`, `target_key`, `before_value`, `after_value`, `user_id`, `audit_time` 등이다.

| 로그 유형 | 기록 주체 | 주요 항목 |
| --- | --- | --- |
| 거래로그 | STF·ETF 자동 | guid, serviceId, transactionCode, resultCode |
| 애플리케이션 로그 | 개발자 SLF4J | MDC + 업무 메시지 |
| 감사로그 | Service/Facade 호출 | 변경 전후 값, 사용자 |

감사로그는 `AuditLogService`를 통해 기록한다. 직접 LOGDB에 INSERT하지 않는다. 조회 거래는 감사로그 대상이 아닌 경우가 많으나, 개인정보 조회는 감사 대상일 수 있다. 업무 요건에 따라 Rule·Service에서 감사 호출 여부를 결정한다.

---

## 장 요약

Java 클래스는 계층별 접미사(Handler, Facade, Service, Rule, Dao, Mapper, Req, Res)를 사용하고, DTO는 Request·Response·내부 객체를 분리한다. MyBatis SQL ID는 ServiceId와 동일 패턴을 권장하며, DB 테이블·컬럼은 snake_case와 업무 prefix를 따른다. 오류코드 E-{BC}-{NNNN}와 메시지코드는 OM 등록 후 사용하고, 거래로그·감사로그 항목은 MDC·DB 컬럼명 일관성을 유지한다.
