# TCF 트랜잭션 타임아웃 관리 정리

NSIGHT TCF의 트랜잭션 타임아웃은 **한 곳에서만 관리하지 않습니다**.  
**정책 테이블 + 3계층 적용 + Facade 선언**으로 나뉩니다.

| 관점 | 역할 | 핵심 |
|------|------|------|
| **거래통제** | 이 거래를 허용할지 | `TCF_TRANSACTION_CONTROL` |
| **Timeout 정책** | 몇 초 안에 끝낼지 | `TCF_SERVICE_TIMEOUT_POLICY` |
| **Online Timeout** | Dispatcher 전체 구간 | `OnlineTransactionTimeoutExecutor` |
| **Spring TX Timeout** | Facade DB 트랜잭션 | `@Transactional` + AOP 덮어쓰기 |
| **DB Query Timeout** | SQL 1건 실행 | `PolicyDrivenQueryTimeoutInterceptor` |
| **운영 관리** | serviceId별 초 조정 | OM `OM.TimeoutPolicy.*` |

---

## 1. 개념: 두 가지를 분리

```text
TCF_TRANSACTION_CONTROL     = 이 거래를 허용할 것인가?
TCF_SERVICE_TIMEOUT_POLICY  = 이 거래를 몇 초 안에 끝내야 하는가?
```

| 구분 | 테이블/역할 |
|------|-------------|
| **거래통제** | `TCF_TRANSACTION_CONTROL` — 허용/차단 (7필드 Header 매칭) |
| **Timeout 정책** | `TCF_SERVICE_TIMEOUT_POLICY` — Online/TX/Query 초 |

**Timeout 정책 PK:** `serviceId` + `transactionCode` + `businessCode`

거래통제 7필드(`serviceId`, `transactionCode`, `businessCode`, `serviceName`, `userId`, `channelId`, `branchId`)는 **허용/차단만** 담당합니다. Timeout은 정책 테이블에서 별도 관리합니다.

---

## 2. 전체 구조

```text
POST /online
   │
   ▼
STF.preProcess()
   ├─ transactionControlService.check()     ← TCF_TRANSACTION_CONTROL
   └─ timeoutPolicyService.resolveAndApply() ← TCF_SERVICE_TIMEOUT_POLICY
         → TimeoutContextHolder + TransactionContext 바인딩
   │
   ▼
TCF.process()
   ├─ OnlineTransactionTimeoutExecutor      ← onlineTimeoutSec (Dispatcher 전체)
   │     └─ Handler → Facade @Transactional ← txTimeoutSec (AOP 덮어쓰기)
   │           └─ Service → DAO → MyBatis   ← dbQueryTimeoutSec (Interceptor)
   └─ ETF
         └─ TimeoutExceptionResolver → E-TCF-TIME-001~006
```

---

## 3. Timeout 3종 + 기본값

| 종류 | 컬럼 | 기본값 | 적용 지점 |
|------|------|--------|-----------|
| **Online** | `ONLINE_TIMEOUT_SEC` | 5초 | TCF → Dispatcher 전체 구간 |
| **Spring TX** | `TX_TIMEOUT_SEC` | 5초 | Facade `@Transactional` |
| **DB Query** | `DB_QUERY_TIMEOUT_SEC` | 3초 | MyBatis SQL 1건 |

**추가 컬럼**

| 컬럼 | 기본값 | 용도 |
|------|--------|------|
| `EXTERNAL_CONNECT_TIMEOUT_MS` | 3000ms | 외부 API 연결 |
| `EXTERNAL_READ_TIMEOUT_MS` | 5000ms | 외부 API 읽기 |
| `TIMEOUT_ACTION` | `FAIL` | Timeout 발생 시 처리 방침 |

**TIMEOUT_ACTION 값**

| 값 | 의미 |
|----|------|
| `FAIL` | 즉시 실패 응답 (조회 등) |
| `UNKNOWN` | 결과 불명 — 등록·변경 시 검토 |
| `STATUS_CHECK` | 상태 확인 후 처리 (EXECUTE·외부연계) |

> Hikari `connectionTimeout`은 SQL 실행 시간이 아니라 **Pool에서 Connection을 빌리기 위해 기다리는 시간**입니다. (별도 레이어)

## 4. 처리 흐름 (요청 1건 기준)

```text
POST /online
 → STF.preProcess()
      ① 거래통제 (TCF_TRANSACTION_CONTROL)
      ② TimeoutPolicyService.resolveAndApply()
           → TCF_SERVICE_TIMEOUT_POLICY 조회 (없으면 기본값)
           → TimeoutContextHolder + TransactionContext에 바인딩
 → TCF.process()
      ③ OnlineTransactionTimeoutExecutor
           → 별도 스레드에서 dispatch, future.get(onlineTimeoutSec)
      ④ Handler → Facade @Transactional
           → PolicyDrivenTransactionAttributeSource가 TX timeout을 정책값으로 덮어씀
      ⑤ Service → DAO → MyBatis
           → PolicyDrivenQueryTimeoutInterceptor가 dbQueryTimeoutSec 적용
 → ETF
      ⑥ TimeoutExceptionResolver → E-TCF-TIME-001~006 표준 오류
```

**STF에서 정책을 조회·바인딩합니다.**

```77:79:tcf-core/src/main/java/com/nh/nsight/tcf/core/processor/STF.java
        System.out.println(
                " =====================================================[STF.preProcess] timeoutPolicyService.resolveAndApply");
        timeoutPolicyService.resolveAndApply(header, context);
```

**정책 조회 순서** (`TimeoutPolicyResolver`):

1. Header의 `serviceId` + `transactionCode` + `businessCode`로 DB 조회
2. 없으면 `TimeoutPolicy` 기본값 객체 반환 (Online 5초 / TX 5초 / Query 3초)

## 5. 계층별 구현

### 5.1 ① Online Timeout (전체 거래)

`TCF.process()`의 **Dispatcher 구간**을 감쌉니다.  
정책의 `onlineTimeoutSec`만큼 기다린 뒤 초과 시 **`E-TCF-TIME-001`** 입니다.

```30:48:tcf-core/src/main/java/com/nh/nsight/tcf/core/timeout/OnlineTransactionTimeoutExecutor.java
    public Object execute(Supplier<Object> action) {
        if (!properties.isTimeoutPolicyEnabled() || executorService == null) {
            return action.get();
        }
        TimeoutPolicy policy = TimeoutContextHolder.get();
        int timeoutSec = policy != null && policy.getOnlineTimeoutSec() > 0
                ? policy.getOnlineTimeoutSec()
                : TcfServiceTimeoutConstants.DEFAULT_ONLINE_TIMEOUT_SEC;
        // ...
            return future.get(timeoutSec, TimeUnit.SECONDS);
        } catch (TimeoutException e) {
            throw new BusinessException(
                    ErrorCode.TIMEOUT_ONLINE,
                    "온라인 거래 처리 시간(" + timeoutSec + "초)을 초과했습니다.");
```

- 별도 스레드 풀(`onlineTransactionTimeoutThreadPool`)에서 dispatch 실행
- `TimeoutThreadContext`로 MDC·Context 스냅샷 전파
- `nsight.tcf.timeout-policy-enabled=false` 또는 풀 미구성 시 → 동기 실행 (timeout 없음)

### 5.2 ② Spring Transaction Timeout (Facade)

Facade에는 **코딩 시점**에 `@Transactional(timeout=…)`을 선언합니다.

```17:19:sv-service/src/main/java/com/nh/nsight/marketing/sv/entry/facade/SvSampleFacade.java
    @Transactional(readOnly = true, timeout = 5)
    public Map<String, Object> inquiry(Map<String, Object> body, TransactionContext context) {
        return service.inquiry(body, context);
```

`nsight.tcf.timeout-policy-enabled=true`(기본)이면 `PolicyDrivenTransactionAttributeSource`가 **실행 시점**에 `TX_TIMEOUT_SEC`로 덮어씁니다.  
Facade에 적은 값은 **fallback·문서 역할**에 가깝습니다.

```32:47:tcf-web/src/main/java/com/nh/nsight/tcf/web/application/rule/PolicyDrivenTransactionAttributeSource.java
    private TransactionAttribute applyPolicyTimeout(@Nullable TransactionAttribute attr) {
        if (attr == null || !properties.isTimeoutPolicyEnabled()) {
            return attr;
        }
        TimeoutPolicy policy = TimeoutContextHolder.get();
        // ...
        return copyWithTimeout(rule, policyTimeout);
    }
```

**Facade 밖에서 수동 TX**가 필요하면 `PolicyDrivenTransactionExecutor`를 사용합니다.

| 용도 | 컴포넌트 |
|------|----------|
| Facade AOP | `PolicyDrivenTransactionAttributeSource` |
| 프로그램 TX | `PolicyDrivenTransactionExecutor` |

### 5.3 ③ MyBatis Query Timeout

`PolicyDrivenQueryTimeoutInterceptor`가 `dbQueryTimeoutSec`을 **MappedStatement timeout**으로 적용합니다.

```40:57:tcf-web/src/main/java/com/nh/nsight/tcf/web/support/PolicyDrivenQueryTimeoutInterceptor.java
        if (!properties.isTimeoutPolicyEnabled()) {
            return invocation.proceed();
        }
        TimeoutPolicy policy = TimeoutContextHolder.get();
        // ...
        args[0] = MappedStatementSupport.copyWithTimeout(mappedStatement, policyTimeout);
        return invocation.proceed();
```

- `TcfOnlineTimeoutConfiguration`에서 MyBatis `ConfigurationCustomizer`로 등록
- 정책 비활성 시 → `mybatis.configuration.default-statement-timeout` 등 기본값만 사용

## 6. 테이블 `TCF_SERVICE_TIMEOUT_POLICY`

| 컬럼 | 설명 |
|------|------|
| `SERVICE_ID`, `TRANSACTION_CODE`, `BUSINESS_CODE` | PK |
| `SERVICE_NAME` | 서비스명 (PK 제외) |
| `ONLINE_TIMEOUT_SEC` | 온라인 전체 Timeout |
| `TX_TIMEOUT_SEC` | Spring Transaction Timeout |
| `DB_QUERY_TIMEOUT_SEC` | MyBatis Query Timeout |
| `EXTERNAL_CONNECT_TIMEOUT_MS` | 외부 연계 connect |
| `EXTERNAL_READ_TIMEOUT_MS` | 외부 연계 read |
| `TIMEOUT_ACTION` | FAIL / UNKNOWN / STATUS_CHECK |
| `USE_YN`, `DESCRIPTION` | 사용·설명 |

**시드:** `TimeoutPolicySeedData` — `OM_SERVICE_CATALOG` 기준 MERGE

| processingType | Online (시드) | TIMEOUT_ACTION (시드) |
|----------------|---------------|------------------------|
| INQUIRY | 5초 | FAIL |
| CREATE/UPDATE/DELETE | 10초 | UNKNOWN |
| EXECUTE | 10초 | STATUS_CHECK |

## 7. 운영 관리 (OM)

| 항목 | 내용 |
|------|------|
| **화면** | `/ui/om/admin/timeout-policy.html` |
| **ServiceId** | `OM.TimeoutPolicy.inquiry/save/update/delete` |
| **시드** | `OM_SERVICE_CATALOG` 기준 MERGE |
| **설정 끄기** | `nsight.tcf.timeout-policy-enabled: false` → Facade `@Transactional`·MyBatis 기본값만 사용 |

### serviceId (OM.TimeoutPolicy.*)

| serviceId | transactionCode | 역할 |
|-----------|-----------------|------|
| `OM.TimeoutPolicy.inquiry` | `OM-TMO-0001` | 정책 목록 조회 |
| `OM.TimeoutPolicy.save` | `OM-TMO-0002` | 정책 등록 |
| `OM.TimeoutPolicy.update` | `OM-TMO-0003` | 정책 수정 |
| `OM.TimeoutPolicy.delete` | `OM-TMO-0004` | 정책 삭제 |

**구현:** `OmTimeoutPolicyFacade` → `OmTimeoutPolicyService` → `OmOperationDao`

### 운영 원칙

1. 기본 온라인 Timeout 5초, DB Query 3초.
2. 등록·변경 Timeout 시 `FAIL` 단독 처리 지양 — `UNKNOWN`/`STATUS_CHECK` 검토.
3. 정책 변경은 OM 화면 + 권한이력 기록.
4. 10초 초과 온라인 Timeout은 예외 승인 대상.

## 8. 오류 코드

| 코드 | 의미 | 발생 지점 |
|------|------|-----------|
| `E-TCF-TIME-001` | Online timeout | `OnlineTransactionTimeoutExecutor` |
| `E-TCF-TIME-002` | Transaction timeout | Spring `TransactionTimedOutException` |
| `E-TCF-TIME-003` | DB Query timeout | MyBatis / `SQLTimeoutException` |
| `E-TCF-TIME-004` | DB Connection 획득 timeout | Hikari 등 |
| `E-TCF-TIME-005` | External Connect timeout | 외부 API |
| `E-TCF-TIME-006` | External Read timeout | 외부 API |

`TCF.process()`의 catch 블록에서 `TimeoutExceptionResolver`가 Spring TX·SQL timeout을 `BusinessException`으로 변환합니다.

```18:31:tcf-core/src/main/java/com/nh/nsight/tcf/core/timeout/TimeoutExceptionResolver.java
    public static Optional<BusinessException> toBusinessException(Throwable throwable) {
        // ...
            if (TX_TIMED_OUT.equals(className) || isTransactionTimedOut(current)) {
                return Optional.of(new BusinessException(
                        ErrorCode.TIMEOUT_TRANSACTION, "트랜잭션 처리 시간을 초과했습니다.", current));
```

→ ETF `businessFail` 경로 → HTTP 200 + `result.resultCode = E0001`

## 9. 설정 키

| 설정 | 기본값 | 설명 |
|------|--------|------|
| `nsight.tcf.timeout-policy-enabled` | `true` | 정책 기반 3계층 적용 on/off |
| `nsight.tcf.timeout-policy-schema-auto-init` | `true` | 정책 테이블 자동 생성 |
| `nsight.tcf.timeout-policy-table-name` | `TCF_SERVICE_TIMEOUT_POLICY` | 테이블명 |

정책 **비활성** 시:

- Online — 동기 dispatch, executor timeout 없음
- TX — Facade `@Transactional(timeout=…)` 그대로
- Query — MyBatis global default만

## 10. 거래로그와의 관계

```text
STF transactionLogService.start()  ← Facade TX 시작 전
   ...
Facade @Transactional              ← TX timeout 적용 구간
   ...
ETF transactionLogService.end()    ← Facade TX 종료 후
```

**거래로그는 Facade TX와 분리**되어 기록됩니다.  
Online timeout으로 dispatch가 중단되어도 STF에서 start된 로그는 ETF/end에서 마무리됩니다.

## 11. 처리 유형별 권장 기준

| 거래 유형 | Online/TX | DB Query | TIMEOUT_ACTION |
|-----------|-----------|----------|----------------|
| 조회 | 3~5초 / 5초 | 3초 | FAIL |
| 등록·변경 | 5~10초 / 5초 | 3초 | UNKNOWN 또는 STATUS_CHECK |
| 발송·외부연계 | 10초 / 5초 | 3초 | STATUS_CHECK |
| 배치 트리거 | 30초 (예외 승인) | 3초 | STATUS_CHECK |

| Facade 선언 (fallback) | 용도 |
|------------------------|------|
| `@Transactional(readOnly = true, timeout = 5)` | 단순 조회 |
| `@Transactional(timeout = 10)` | 일반 CUD |
| `@Transactional(timeout = 30)` | 배치 트리거·대기형 |

## 12. 개발자 체크리스트

1. **Facade** — `@Transactional(timeout=…)` 선언 (TX 경계 + fallback)
2. **운영** — `TCF_SERVICE_TIMEOUT_POLICY`로 serviceId별 Online/TX/Query 초 조정
3. **계층 분리** — Online은 Facade 바깥(TCF dispatch), TX는 Facade AOP, Query는 MyBatis interceptor
4. **거래통제와 혼동 금지** — 허용/차단 ≠ Timeout 초
5. **PK 3키** — 정책 조회는 `serviceId` + `transactionCode` + `businessCode`
6. **수동 TX** — Facade 밖이면 `PolicyDrivenTransactionExecutor` 사용
7. **로컬 테스트** — `timeout-policy-enabled: false`로 Facade 선언값만 검증 가능
