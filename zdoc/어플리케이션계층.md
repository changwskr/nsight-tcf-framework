# TCF 어플리케이션 계층 구조 정리

NSIGHT TCF의 **어플리케이션 계층**은 TCF 엔진(`tcf-core`) 파이프라인 **이후**에 위치하는 업무 코드 영역입니다.

| 관점 | 역할 | 핵심 |
|------|------|------|
| **거래 진입** | serviceId ↔ Handler | `entry.handler` — TCF와 업무 연결 |
| **유스케이스** | 트랜잭션 경계 | `entry.facade` — `@Transactional` |
| **도메인 처리** | 업무 로직·응답 조립 | `application.service` |
| **입력 검증** | Body·업무 규칙 | `application.rule` |
| **영속** | DB 접근 | `persistence.dao` → `persistence.mapper` |
| **부가** | 스케줄·외부연동·설정 | `scheduler`, `client`, `config`, `support` |

---

## 1. 프레임워크 vs 어플리케이션

```text
[프레임워크 — tcf-web / tcf-core]
  HTTP 진입 → STF → Dispatcher → ETF
        │
        │ serviceId 라우팅
        ▼
[어플리케이션 — *-service / tcf-om]
  entry/handler → entry/facade → application/service
                                    ↓
                              application/rule
                                    ↓
                         persistence/dao → mapper
```

**목표**

1. **거래 진입점 통일** — 도메인(application Service)당 Handler 1개, `serviceIds()`로 다중 거래 처리
2. **책임 분리** — Facade(오케스트레이션), Service(도메인), Rule(검증), DAO(영속)
3. **프레임워크 비침투** — `StandardRequest` / `TransactionContext` 계약만 의존

## 3. 패키지 구조 (6계층 + 부가)

업무 WAR(`sv-service` 등), `tcf-om`, `tcf-jwt`, `tcf-gateway`는 동일한 패키지 규칙을 따릅니다.

```text
com.nh.nsight.marketing.{code}
├── Nsight{Xxx}Application.java    # extends NsightWarBootstrap
├── entry/
│   ├── handler/       TransactionHandler
│   ├── facade/        @Transactional 유스케이스
│   └── web/           REST Controller (해당 시)
├── application/
│   ├── service/       도메인 Service
│   ├── rule/          검증 Rule
│   └── scheduler/     @Scheduled (해당 시)
├── persistence/
│   ├── dao/           @Repository
│   └── mapper/        MyBatis @Mapper
├── client/            외부 WAS·API (해당 시)
├── config/            모듈 Spring 설정
└── support/           시드·마이그레이션·헬퍼

src/main/resources/mapper/{code}/   MyBatis XML
```

### WAR Bootstrap

```8:17:sv-service/src/main/java/com/nh/nsight/marketing/sv/NsightSvServiceApplication.java
@SpringBootApplication(scanBasePackages = "com.nh.nsight")
@MapperScan("com.nh.nsight.marketing.sv.persistence.mapper")
public class NsightSvServiceApplication extends NsightWarBootstrap {
    public NsightSvServiceApplication() {
        super(NsightSvServiceApplication.class);
    }
```

| 패키지 | Spring | 역할 |
|--------|--------|------|
| `entry.handler` | `@Component` | TCF 거래 진입 |
| `entry.facade` | `@Service` | 트랜잭션·유스케이스 |
| `entry.web` | `@RestController` | 비표준 REST |
| `application.service` | `@Service` | 도메인 로직 |
| `application.rule` | `@Component` | 검증 |
| `application.scheduler` | `@Component` | 배치 스케줄 |
| `persistence.dao` | `@Repository` | Mapper 위임 |
| `persistence.mapper` | `@Mapper` | SQL 인터페이스 |

## 4. 계층별 책임

### 4.1 Handler — 거래 어댑터

| 항목 | 내용 |
|------|------|
| **역할** | `serviceId`와 TCF 파이프라인 연결 — **업무 유일 진입점** |
| **인터페이스** | `TransactionHandler` |
| **등록** | `@Component` → `TransactionDispatcher` 기동 시 맵 등록 |
| **금지** | SQL 직접 호출, 복잡 비즈니스, `@Transactional` |

Handler는 **얇게(thin)** 유지 — `body` + `context`를 Facade에 넘깁니다.

```18:34:sv-service/src/main/java/com/nh/nsight/marketing/sv/entry/handler/SvSampleHandler.java
    private static final String INQUIRY = "SV.Sample.inquiry";

    @Override
    public Collection<String> serviceIds() {
        return List.of(INQUIRY);
    }

    @Override
    public Object doHandle(StandardRequest<Map<String, Object>> request, TransactionContext context) {
        String serviceId = context.getHeader().getServiceId();
        return switch (serviceId) {
            case INQUIRY -> facade.inquiry(request.getBody(), context);
            default -> throw new BusinessException(ErrorCode.SERVICE_NOT_FOUND, serviceId);
        };
    }
```

- `serviceId` 중복 등록 시 **기동 실패**
- 형식: `{BC}.{Domain}.{action}` — [네이밍.md](네이밍.md)

### 4.2 Facade — 유스케이스·트랜잭션 경계

| 항목 | 내용 |
|------|------|
| **역할** | 하나의 거래(유스케이스) 흐름 조율 |
| **트랜잭션** | **`@Transactional`은 Facade에 선언** |
| **호출** | Service, Support |
| **반환** | ETF로 전달될 `Map` (응답 body) |

```17:19:sv-service/src/main/java/com/nh/nsight/marketing/sv/entry/facade/SvSampleFacade.java
    @Transactional(readOnly = true, timeout = 5)
    public Map<String, Object> inquiry(Map<String, Object> body, TransactionContext context) {
        return service.inquiry(body, context);
```

**tcf-om Facade 예**

| Facade | 담당 |
|--------|------|
| `OmAuthFacade` | 로그인·로그아웃 |
| `OmUserFacade` | 사용자 CRUD |
| `OmDashboardFacade` | 운영 대시보드 |
| `OmErrorCodeFacade` | 오류코드 CRUD |
| `OmBatchFacade` | 배치 Job 실행 |

복수 Service 조합·외부 HTTP(tcf-batch)는 Facade에서 묶습니다.

### 4.3 Service — 도메인 로직

| 항목 | 내용 |
|------|------|
| **역할** | 업무 처리·데이터 조립 |
| **호출** | Rule → DAO |
| **컨텍스트** | `TransactionContext` — guid, userId, serviceId |
| **예외** | `BusinessException` |

```20:28:sv-service/src/main/java/com/nh/nsight/marketing/sv/application/service/SvSampleService.java
    public Map<String, Object> inquiry(Map<String, Object> body, TransactionContext context) {
        rule.validateInquiry(body);
        Map<String, Object> data = dao.selectSample(body);
        Map<String, Object> result = new LinkedHashMap<>();
        result.put("businessCode", "SV");
        result.put("serviceId", context.getHeader().getServiceId());
        result.put("guid", context.getHeader().getGuid());
        result.put("data", data);
        return result;
```

**Facade vs Service**

| 구분 | Facade | Service |
|------|--------|---------|
| 트랜잭션 | `@Transactional` | 없음 (기본) |
| 범위 | 유스케이스 1건 | 도메인 연산 |
| 재사용 | Handler 전용 조합 | Facade·다른 Service에서 재사용 |

### 4.4 Rule — 입력·업무 검증

| 항목 | 내용 |
|------|------|
| **역할** | Body·조건값 검증, 업무 불가 판단 |
| **예외** | `BusinessException` + `errorCode` |
| **위치** | Service 호출 **전** |
| **금지** | DAO 호출 |

**Rule vs STF**

| 계층 | 검증 대상 |
|------|-----------|
| STF (`tcf-core`) | Header, 세션, 권한, 멱등성, 거래통제, Timeout |
| Rule (어플리케이션) | Body 필드, 업무 제약, 도메인 규칙 |

### 4.5 DAO · Mapper — 영속

| 계층 | 역할 |
|------|------|
| **DAO** | Mapper 위임, 조건 Map 조립 — **비즈니스 판단 없음** |
| **Mapper** | MyBatis 인터페이스 — method name == SQL id |

```text
resources/mapper/sv/SvSampleMapper.xml
resources/mapper/om/OmOperationMapper.xml
```

## 5. 호출 규칙 (의존성)

### 5.1 허용 방향

```text
Handler     → Facade
Facade      → Service, Support
Service     → Rule, DAO, (주의) 다른 Service
DAO         → Mapper
Rule        → (없음 — 순수 검증)
```

### 5.2 금지

| 금지 | 이유 |
|------|------|
| Handler → DAO | 계층 우회, TX 경계 붕괴 |
| Rule → DAO | 검증·조회 혼재 |
| Mapper → Service | 역방향 의존 |
| Service → Handler | 순환 참조 |
| Handler catch 후 StandardResponse 직접 생성 | ETF·거래로그 우회 |

### 5.3 트랜잭션 가이드

| 처리 유형 | Facade 어노테이션 |
|-----------|-------------------|
| 조회 | `@Transactional(readOnly = true, timeout = 5)` |
| CUD | `@Transactional(timeout = 10)` |
| 배치·대기형 | `@Transactional(timeout = 30)` (예외 승인) |

운영 Timeout은 `TCF_SERVICE_TIMEOUT_POLICY`가 실행 시 덮어씁니다 — [타임아웃관리.md](타임아웃관리.md)

## 6. 프레임워크 경계

어플리케이션 계층은 TCF를 **호출하지 않습니다**. Dispatcher가 Handler를 **피호출**합니다.

```text
OnlineTransactionController
        │
        ▼
      TCF.process()
   ┌────┴────┐
   ▼         ▼
  STF    Dispatcher ──→ Handler (어플리케이션)
   │         │
   └────┬────┘
        ▼
       ETF  ← Handler 반환 Map → StandardResponse.body
```

### TransactionContext

STF가 생성·`TransactionContextHolder`에 등록, TCF `finally`에서 clear.

| 제공 정보 | 용도 |
|-----------|------|
| `StandardHeader` | serviceId, userId, guid, businessCode |
| TimeoutPolicy 등 | `context` attributes |

Handler·Facade·Service 전 구간에서 `context` 인자로 전달합니다.

### TcfGateway (비표준 진입)

REST·multipart도 TCF 파이프라인을 태우려면 `TcfGateway.invoke()` 사용.  
표준 JSON 거래는 **Handler 경로 우선**입니다.

## 7. 요청·응답 데이터 흐름

```text
1. Client POST /sv/online + StandardRequest JSON
2. OnlineTransactionController — businessCode·clientIp 보정
3. TCF.process()
4. STF — Header 검증, Context 생성
5. Dispatcher — "SV.Sample.inquiry" → SvSampleHandler
6. Handler — facade.inquiry(body, context)
7. Facade — @Transactional, service.inquiry()
8. Service — rule.validate → dao.select → result Map 조립
9. ETF — StandardResponse { header, result, body }
10. Client — result.resultCode == "S0000"
```

| 데이터 | 타입 | 책임 |
|--------|------|------|
| 요청 body | `Map<String, Object>` | 채널·화면별 유연 필드 |
| 응답 body | Service 조립 `Map` | ETF가 그대로 래핑 |
| Header | `TransactionContext` | Handler는 body만, Header는 context |

## 8. 모듈 유형별 적용

### 8.1 업무 서비스 (`*-service`)

```text
sv-service, eb-service, ep-service …
표준 6계층: Handler → Facade → Service → Rule → DAO → Mapper
```

- 도메인(Service) 1개 = Handler 1개 (`serviceIds()`로 다중 거래)
- `tcf-web` + `tcf-core` 의존
- `scanBasePackages = "com.nh.nsight"`

### 8.2 운영 모듈 (`tcf-om`)

```text
Handler (40+) → Facade (도메인별) → Service → Rule/DAO/Mapper
         ├─ support/   시드, 마이그레이션
         ├─ config/    Spring Session 등
         └─ entry/web  UD 파일 REST (TCF 미경유)
```

- OM Admin 전 기능 → `OM.*` serviceId
- `OM_SERVICE_CATALOG` — Handler·권한·Timeout 시드

### 8.3 UI Relay (`tcf-ui`) — 별도

```text
브라우저 → om-admin.js → Relay → 업무 WAR /online
```

어플리케이션 6계층과 **별도** — 비즈니스 로직 없음.

### 8.4 배치 (`tcf-batch`) — 별도

```text
Scheduler / REST → CollectService → JDBC MERGE
```

Handler 패턴 대신 **Job + CollectService**. OM Dashboard는 결과 테이블 **조회만**.

### 8.5 예외 모듈

| 모듈 | 구조 |
|------|------|
| `tcf-jwt` | 6계층 (Handler → Facade → Service) |
| `tcf-gateway` | catalog, service, persistence — 프록시 라우팅 |

## 10. 신규 거래 추가 절차

1. **`serviceId`** — `{BC}.{Domain}.{action}` 정의
2. **Handler** — 도메인 핸들러 `serviceIds()`에 추가 + `serviceId` 분기 → Facade 위임 (새 도메인일 때만 신규 생성)
3. **Facade** — `@Transactional` 유스케이스 메서드
4. **Service** — 도메인 로직·응답 Map 조립
5. **Rule** — 입력 검증 (필요 시)
6. **DAO / Mapper / XML** — DB 연동 (필요 시)
7. **카탈로그** — `OM_SERVICE_CATALOG` MERGE
8. **샘플 JSON** — `tcf-ui/.../sample-requests/`
9. **검증** — curl 또는 tcf-ui 거래 화면

## 11. 개발자 체크리스트

1. Handler는 **얇게** — Facade만 호출
2. **`@Transactional`은 Facade**에만 (Service X)
3. **검증** — Header는 STF, Body는 Rule
4. **응답 body** — Service가 Map 조립
5. **예외** — `BusinessException` + `errorCode`, Handler catch 금지
6. **serviceId** — 도메인당 Handler 1개, `serviceIds()` 목록에 거래 등록
7. **계층 우회** — Handler→DAO, Rule→DAO 금지
8. **네이밍** — 도메인 이름 Handler~Mapper 일치 — [네이밍.md](네이밍.md)
