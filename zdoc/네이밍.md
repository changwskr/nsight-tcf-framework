# TCF 네이밍 정리

NSIGHT TCF의 네이밍은 **거래 식별자(`serviceId`)를 중심**으로 계층별 클래스·패키지·SQL ID가 1:1로 추적되도록 설계합니다.

| 관점 | 역할 | 핵심 |
|------|------|------|
| **거래 식별** | 라우팅·카탈로그 키 | `serviceId` — `{BC}.{Domain}.{action}` |
| **화면·거래 코드** | Header·카탈로그 | `transactionCode` — `{BC}-{ACT3}-{NNNN}` |
| **계층 클래스** | 역할 즉시 파악 | `{Business}{Domain}{Role}` 접미사 |
| **영속 계층** | SQL 추적 | Mapper method == SQL id |
| **오류 코드** | 예외·마스터 | `E-{BC}-{CATEGORY}-{NNNN}` |
| **리소스** | UI·샘플 JSON | kebab-case 파일명 |

---

## 1. 목표

| 목표 | 설명 |
|------|------|
| **일관성** | 코드만 보고 Controller/Facade/Service 등 역할 즉시 파악 |
| **추적성** | `serviceId` → Handler → Facade → Service → DAO → Mapper SQL |
| **검색성** | IDE 검색(`OM.ErrorCode.update`)으로 관련 파일 빠른 발견 |
| **확장성** | 업무코드(OM/SV/EB…)가 늘어도 동일 패턴 유지 |

---

## 2. 전체 추적 구조

```text
serviceId: OM.ErrorCode.update
   │
   ├─ Handler  : OmErrorCodeUpdateHandler.serviceId()
   ├─ Facade   : OmErrorCodeFacade.update(...)
   ├─ Service  : OmErrorCodeService.update(...)
   ├─ Rule     : OmOperationRule.requireField(...)
   ├─ DAO      : OmOperationDao.updateErrorCode(...)
   ├─ Mapper   : OmOperationMapper.updateErrorCode(...)
   ├─ SQL id   : updateErrorCode
   ├─ transactionCode : OM-ERR-0004
   ├─ errorCode       : E-OM-BIZ-0002 (실패 시)
   └─ sample JSON     : om-error-code-update.json (선택)
```

## 3. 공통 표기 원칙

| 항목 | 규칙 | 예 |
|------|------|-----|
| 패키지/경로 | 소문자 | `entry.handler`, `mapper/om` |
| 클래스/인터페이스 | `PascalCase` | `SvSampleService` |
| 메서드/필드 | `camelCase` | `inquiry`, `saveUser` |
| 상수 | `UPPER_SNAKE_CASE` | `DEFAULT_PAGE_SIZE` |
| 업무 약어 | 대문자 2자리 | `OM`, `SV`, `EB`, `UD` |
| 역할 접미사 | 고정 | `Controller`, `Handler`, `Facade`, `Service`, `Rule`, `Dao`, `Mapper` |

### 금지

| 금지 | 이유 |
|------|------|
| `CommonService`, `Util2`, `EtcService` | 책임 모호 |
| 역할 접미사 생략 (`OmUser`) | 계층 식별 불가 |
| 계층 혼합 (`OmUserDaoService`) | 단일 책임 위반 |
| 복수 도메인 접두 (`OmBatchUserService`) | 응집도 저하 |
| SQL id 축약 (`upd01`, `updErrCd`) | 검색·추적 불가 |

## 4. 모듈·패키지 네이밍

### 4.1 Gradle 모듈

| 패턴 | 예 | 역할 |
|------|-----|------|
| `tcf-{name}` | `tcf-core`, `tcf-web`, `tcf-om`, `tcf-ui` | 프레임워크·공통 |
| `{bc}-service` | `sv-service`, `eb-service`, `ep-service` | 업무 WAR |
| `tcf-{role}` | `tcf-gateway`, `tcf-jwt`, `tcf-batch` | 인프라·부가 |

### 4.2 루트 패키지

```text
com.nh.nsight.marketing.{businessCodeLower}   ← 업무 WAR
com.nh.nsight.tcf.{area}                      ← tcf-core, tcf-web
com.nh.nsight.auth.jwt                        ← tcf-jwt
com.nh.nsight.gateway                         ← tcf-gateway
```

예: `com.nh.nsight.marketing.sv`, `com.nh.nsight.marketing.om`

### 4.3 6계층 패키지

| 패키지 | 역할 | 예 |
|--------|------|-----|
| `entry.handler` | TCF 거래 진입 (`serviceId`) | `OmErrorCodeUpdateHandler` |
| `entry.facade` | 트랜잭션 경계·유스케이스 | `OmErrorCodeFacade` |
| `entry.web` | REST·비표준 HTTP | `OmUpdownloadFileController` |
| `application.service` | 도메인 처리 | `OmErrorCodeService` |
| `application.rule` | 검증 규칙 | `OmOperationRule` |
| `application.scheduler` | `@Scheduled` | `EbEventPublishScheduler` |
| `persistence.dao` | 영속 추상화 | `OmOperationDao` |
| `persistence.mapper` | MyBatis 인터페이스 | `OmOperationMapper` |
| `client` | 외부 WAS·API | `EpOnlineClient` |
| `config` | Spring 설정 | `OmUpdownloadConfiguration` |
| `support` | 시드·마이그레이션 | `TimeoutPolicySeedData` |

> TCF 표준 거래(`POST /online`)는 `entry.handler`가 진입점입니다.  
> `entry.web`는 파일 업로드·UI Relay API 등 **비표준 REST**에서 주로 사용합니다.

## 5. 계층별 클래스 네이밍

| 계층 | 패턴 | 예 |
|------|------|-----|
| **Controller** | `{Domain}{Resource}{Action?}Controller` | `OnlineTransactionController`, `TcfApiController` |
| **Handler** | `{Business}{Domain}{Action}Handler` | `OmErrorCodeUpdateHandler` |
| **Facade** | `{Business}{Domain}Facade` | `OmErrorCodeFacade`, `SvSampleFacade` |
| **Service** | `{Business}{Domain}Service` | `OmAuthService`, `EbUserService` |
| **Rule** | `{Business}{Domain?}Rule` | `SvSampleRule`, `OmOperationRule` |
| **DAO** | `{Business}{Domain}Dao` | `OmOperationDao`, `SvSampleDao` |
| **Mapper** | `{Business}{Domain}Mapper` | `OmOperationMapper` |

### Handler 규칙

- `Action`은 `serviceId` 마지막 토큰과 의미 일치 (`inquiry`, `save`, `update`, `delete`, `execute`)
- Handler 1개 = `serviceId()` 1개 (원칙)

```java
@Component
public class SvSampleInquiryHandler implements TransactionHandler {
    @Override
    public String serviceId() { return "SV.Sample.inquiry"; }

    @Override
    public Object doHandle(StandardRequest<Map<String, Object>> request, TransactionContext context) {
        return facade.inquiry(request.getBody(), context);
    }
}
```

### Facade 규칙

- 유스케이스(도메인) 경계 단위 — `User`, `Batch`, `ErrorCode`
- 클래스명에 `Transactional` 등 기술 용어 금지

## 6. 메서드 네이밍

### 6.1 권장 액션 어휘

| 용도 | 권장 메서드명 |
|------|---------------|
| 단건 조회 | `detail`, `get`, `findBy...` |
| 목록 조회 | `inquiry`, `list`, `search` |
| 등록 | `save`, `create`, `insert` |
| 수정 | `update`, `modify` |
| 삭제/비활성 | `delete`, `remove`, `disable` |
| 실행형 | `execute`, `run` |

### 6.2 계층별 패턴

| 계층 | 패턴 | 예 |
|------|------|-----|
| Handler | `doHandle(...)` 고정 | TCF 인터페이스 |
| Facade | 유스케이스 동사 | `inquiry(...)`, `save(...)` |
| Service | 도메인 동사 | `searchErrorCodes(...)`, `deleteSession(...)` |
| Rule | `validate*`, `require*`, `normalize*` | `validateInquiry`, `requireField` |
| DAO / Mapper | `select*`, `insert*`, `update*`, `merge*`, `count*` | `selectErrorCodeByCode` |

## 7. serviceId (핵심)

### 7.1 형식

```text
{BC}.{Domain}.{action}
```

| 파트 | 규칙 | 예 |
|------|------|-----|
| `{BC}` | 대문자 업무코드 | `OM`, `SV`, `EB` |
| `{Domain}` | PascalCase 도메인 | `ErrorCode`, `Auth`, `Sample` |
| `{action}` | 소문자 동사 | `inquiry`, `save`, `update`, `delete`, `execute` |

**예**

```text
OM.ErrorCode.inquiry
OM.ErrorCode.update
SV.Sample.inquiry
EB.User.create
OM.TimeoutPolicy.save
```

### 7.2 processingType과의 관계

| 항목 | 역할 |
|------|------|
| `serviceId` | **Dispatcher 라우팅 키** (필수) |
| `processingType` | INQUIRY/CREATE/UPDATE/DELETE/EXECUTE — 감사·로그·규칙 참조 (라우팅 키 아님) |

### 7.3 OM_SERVICE_CATALOG

Header의 `serviceId`·`transactionCode`는 **`OM_SERVICE_CATALOG`** 에 등록합니다.  
Handler 실제 등록은 Spring `@Component` + `TransactionDispatcher` 기동 시 `serviceId()` 맵입니다.

---

## 8. transactionCode

### 8.1 형식

```text
{BC}-{ACTION3}-{NNNN}
```

| 파트 | 설명 | 예 |
|------|------|-----|
| `{BC}` | 업무코드 | `OM`, `SV`, `EB` |
| `{ACTION3}` | 도메인·동작 식별 3자 | `ERR`, `INQ`, `USR`, `TMO` |
| `{NNNN}` | 4자리 일련번호 | `0001` |

**예**

| serviceId | transactionCode |
|-----------|-----------------|
| `OM.ErrorCode.inquiry` | `OM-ERR-0001` |
| `OM.ErrorCode.update` | `OM-ERR-0004` |
| `SV.Sample.inquiry` | `SV-INQ-0001` |
| `EB.User.create` | `EB-USR-0001` |
| `OM.TimeoutPolicy.save` | `OM-TMO-0002` |

### 8.2 규칙

- 카탈로그 기준 **유일성** 보장
- `serviceId`와 1:1 대응 권장
- 샘플 JSON Header·화면 코드와 **일치** 유지

---

## 9. errorCode

### 9.1 형식

```text
E-{BC}-{CATEGORY}-{NNNN}
```

| 파트 | 의미 | 예 |
|------|------|-----|
| `E` | Error prefix | 고정 |
| `{BC}` | COM / OM / SV / UD / TCF | `OM` |
| `{CATEGORY}` | VALID, BIZ, AUTH, SYS, TIME, CTL … | `BIZ` |
| `{NNNN}` | 4자리 일련번호 | `0002` |

**예**

```text
E-COM-VALID-0001    ← Header 검증
E-COM-SYS-0001      ← 시스템 오류
E-OM-BIZ-0002       ← OM 업무 — 데이터 없음
E-OM-VAL-0002       ← OM 필수값 누락
E-TCF-TIME-001      ← Online timeout
E-UD-VAL-0001       ← UD REST 검증
```

### 9.2 OM_ERROR_CODE 마스터

운영 메시지는 `OM_ERROR_CODE` + `OM.ErrorCode.*` CRUD로 관리합니다.  
상세: [예외처리.md](예외처리.md)

## 10. SQL · Mapper 네이밍

### 10.1 파일 경로

```text
src/main/resources/mapper/{bc}/{Business}{Domain}Mapper.xml
```

예: `mapper/sv/SvSampleMapper.xml`, `mapper/om/OmOperationMapper.xml`

### 10.2 SQL ID 규칙

| 구분 | 규칙 | 예 |
|------|------|-----|
| 단건 조회 | `select{Domain}{ByCondition}` | `selectErrorCodeByCode` |
| 목록 조회 | `search{DomainPlural}` | `searchErrorCodes` |
| 건수 | `count{DomainPlural}` | `countErrorCodes` |
| 등록 | `insert{Domain}` | `insertErrorCode` |
| 수정 | `update{Domain}` | `updateErrorCode` |
| 병합 | `merge{Domain}` | `mergeErrorCode` |
| 삭제/비활성 | `delete{Domain}` / `disable{Domain}` | `disableErrorCode` |

### 10.3 Mapper method == SQL id (강제)

```java
// OmOperationMapper.java
int updateErrorCode(Map<String, Object> params);
```

```xml
<!-- OmOperationMapper.xml -->
<update id="updateErrorCode">
  <!-- SQL -->
</update>
```

## 12. structCode · 테이블 (전문·Timeout)

전문구조·Timeout 정책 등 **프레임워크 메타** 네이밍:

| 영역 | 패턴 | 예 |
|------|------|-----|
| 전문 structCode | `STD-{REQ\|RES}-{SEGMENT}` | `STD-REQ-HEADER` |
| Timeout 테이블 | `TCF_SERVICE_TIMEOUT_POLICY` | PK: serviceId + transactionCode + businessCode |
| 거래통제 테이블 | `TCF_TRANSACTION_CONTROL` | 7필드 Header 매칭 |

상세: [전문관리.md](전문관리.md), [타임아웃관리.md](타임아웃관리.md)

## 14. 신규 거래 추가 체크리스트

1. **`serviceId`** — `{BC}.{Domain}.{action}` 정의
2. **Handler** — `{Business}{Domain}{Action}Handler`, `serviceId()` 1:1
3. **Facade / Service / Rule** — 동일 도메인 이름
4. **DAO / Mapper** — 메서드명 == SQL id
5. **`transactionCode`** — `{BC}-{ACT3}-{NNNN}`, `OM_SERVICE_CATALOG` 등록
6. **`errorCode`** — `E-{BC}-*` 사전 정의, `OM_ERROR_CODE` 등록 (OM)
7. **샘플 JSON** — `{bc}-{domain}-{action}.json` 추가
8. **Timeout 시드** — `TCF_SERVICE_TIMEOUT_POLICY` 자동 MERGE 확인

---

## 15. 개발자 체크리스트

1. 클래스명만 보고 **계층·도메인**이 드러나는가?
2. `serviceId`로 Handler부터 SQL까지 **한 줄 추적** 가능한가?
3. Handler `serviceId()` 와 Header `serviceId`가 **일치**하는가?
4. `transactionCode`가 카탈로그·샘플 JSON과 **일치**하는가?
5. 업무 오류 `errorCode`가 **`E-{BC}-*`** 규칙을 따르는가?
6. Mapper XML `id`와 Java 메서드명이 **동일**한가?
7. `Common*`, `Util*`, 축약 SQL id를 **사용하지 않았**는가?
