# TCF 세션 관리 정리

NSIGHT TCF의 세션은 **한 가지 방식이 아니라**, 목적에 따라 역할이 나뉩니다.

| 관점 | 역할 | 핵심 |
|------|------|------|
| **인증 세션** | OM 로그인 상태 유지 | `SPRING_SESSION` (Spring Session JDBC) |
| **거래 검증** | TCF 온라인 거래 시 로그인 확인 | STF `SessionValidator` |
| **Gateway 관문** | gateway 경유 시 SESSIONDB 검증 | `TCF_USER_SESSION` + `SPRING_SESSION` |
| **운영 모니터링** | 대시보드 세션 집계 | `tcf-batch` → `OM_SESSION_STATUS` |

---

## 1. 전체 구조

```text
Browser (JSESSIONID 쿠키)
   │
   ├─ tcf-ui / tcf-uj  ── Cookie Relay ──► tcf-om / *-service
   │
   └─ tcf-uj ──► tcf-gateway ── Cookie 전달 ──► downstream WAS
                    │
                    └─ SESSIONDB 4단계 검증 (세션 소유 X)

tcf-om
   ├─ Spring Session JDBC  → SPRING_SESSION
   ├─ OmAuthSessionSupport → 로그인/로그아웃/세션 조회
   └─ OM.Session.*         → 운영 세션 관리

tcf-batch
   └─ SPRING_SESSION + Actuator metric → OM_SESSION_STATUS
```

---

## 2. 인증 세션 (OM 포털)

### 저장 방식

`tcf-om`은 **Spring Session JDBC**를 사용합니다.

| 설정 | 기본값 |
|------|--------|
| `spring.session.store-type` | `jdbc` |
| `spring.session.jdbc.table-name` | `SPRING_SESSION` |
| `spring.session.timeout` | `60m` |
| `server.servlet.session.timeout` | `60m` |
| 쿠키 이름 | `JSESSIONID` (HttpOnly, SameSite=Lax) |

**공유 DB:** bootRun·ztomcat에서 `data/nsight-txlog/nsight_om` H2를 OM·업무 WAR·gateway가 **SESSIONDB**로 공유합니다.

### 로그인 관련 serviceId

| serviceId | 역할 |
|-----------|------|
| `OM.Auth.login` | 인증 → 세션 생성 |
| `OM.Auth.logout` | 현재 세션 무효화 |
| `OM.Auth.session` | 현재 세션 사용자 정보 조회 |

**로그인 흐름:**

```text
login.html → POST /api/relay/OM/online (OM.Auth.login)
  → tcf-ui/tcf-uj Relay (Cookie·Set-Cookie 중계)
  → tcf-om TCF.process()
  → OmAuthService.login()
  → OmAuthSessionSupport.createSession(user)
  → SPRING_SESSION 저장 + JSESSIONID 쿠키
```

`OmAuthSessionSupport`는 HttpSession에 `OM_SESSION_USER`(userId, branchId, authGroupId 등)를 저장합니다.

---

## 3. UI Relay — 쿠키 전파

`tcf-ui` / `tcf-uj`는 **세션을 저장하지 않고 Cookie 중계**만 합니다.

| 방향 | 처리 |
|------|------|
| 요청 | 브라우저 `Cookie` → 대상 WAS(gateway/om)로 전달 |
| 응답 | WAS `Set-Cookie` → 브라우저에 재설정 |

브라우저는 UI를 경유해도 `JSESSIONID`를 유지하므로, OM Admin 로그인 후 연속 API 호출이 가능합니다.

---

## 4. TCF STF 세션 검증

STF 전처리에서 `SessionValidator`가 동작합니다 (설정 시).

```java
// tcf-core/.../security/SessionValidator.java
public void validate(StandardHeader header) {
    if (!properties.isSessionValidationEnabled()) {
        return;
    }
    if (!StringUtils.hasText(header.getUserId())) {
        throw new BusinessException(ErrorCode.SESSION_INVALID, "로그인 세션이 유효하지 않습니다.");
    }
}
```

| 설정 | 기본값 | 동작 |
|------|--------|------|
| `nsight.tcf.session-validation-enabled` | `false` | `true`일 때 `header.userId` 필수 |

- 현재는 **Header 기반 최소 검증** (`userId` 존재 여부)
- `SPRING_SESSION` 직접 조회는 아님 — gateway·OM Admin 환경에서는 gateway/header 보정과 함께 사용

실패: `E-COM-AUTH-0001` (`SESSION_INVALID`)

---

## 5. tcf-gateway 세션 관문

Gateway는 **세션을 소유하지 않습니다.** SESSIONDB만 참조하는 **관문** 역할입니다.

```text
클라이언트 ──Cookie(JSESSIONID)──► Gateway ──동일 Cookie──► 업무 WAS
                ▲                              │
                └── Set-Cookie (downstream) ───┘
```

**GSF 4단계 검증:**

| 단계 | 내용 |
|------|------|
| 1 | Cookie 존재 (`JSESSIONID` / `NSIGHTSID`) |
| 2 | `SPRING_SESSION` 존재·만료 여부 |
| 3 | `TCF_USER_SESSION` STATUS (ACTIVE 등) |
| 4 | header.userId vs SESSIONDB userId |

| 테이블 | 역할 |
|--------|------|
| `SPRING_SESSION` | Spring Session 저장소 (공유 SESSIONDB) |
| `TCF_USER_SESSION` | 사용자 세션 레지스트리 (gateway 관리) |

- Gateway 자체 HttpSession **미생성** (`session.tracking-modes: []`)
- **이중 인증:** `Authorization: Bearer` 있으면 Gateway JWT 검증(JWKS), 없으면 SESSIONDB 쿠키 4단계
- OM 로그인 성공 시 `TCF_USER_SESSION` 등록 + `GatewayUserSessionSyncScheduler`로 주기 동기화

## 6. OM 운영 세션 관리

| serviceId | 역할 |
|-----------|------|
| `OM.Session.inquiry` | `SPRING_SESSION` 목록·활성 건수 조회 |
| `OM.Session.delete` | 세션 강제 종료 |

**화면:** `/ui/om/admin/session.html`

- 만료 세션 정리: `OmSessionCleanupScheduler` (`nsight.om.session-cleanup.fixed-rate-ms`, 기본 10초)

---

## 7. 세션 모니터링 (tcf-batch)

`tcf-batch`가 `OM_SESSION_STATUS`에 스냅샷을 적재합니다.

| scope | source | 의미 |
|-------|--------|------|
| `OM-PORTAL` | `spring-session` | `SPRING_SESSION` 직접 집계 |
| `{CODE}-AP` | Actuator | 각 WAR `tomcat.sessions.active.current` |

집계: `activeCount`, `expiredCount`, `totalCount`, `uniqueUserCount`, `healthStatus`

---

## 8. 핵심 테이블

| 테이블 | 용도 |
|--------|------|
| `SPRING_SESSION` | Spring Session 본체 |
| `SPRING_SESSION_ATTRIBUTES` | 세션 속성(serialized) |
| `TCF_USER_SESSION` | Gateway 사용자 세션 레지스트리 |
| `OM_SESSION_STATUS` | 대시보드 세션 모니터링 스냅샷 |

---

## 9. 세션 vs Timeout (구분)

| 구분 | 의미 | 설정 |
|------|------|------|
| **세션 만료** | 로그인 유지 시간 | `spring.session.timeout` (60m) |
| **온라인 Timeout** | 거래 1건 처리 상한 | `TCF_SERVICE_TIMEOUT_POLICY` |

세션 만료와 거래 Timeout은 **별개**입니다.

---

## 10. 배포 모드별 요약

| 모드 | 로그인 | 세션 검증 |
|------|--------|-----------|
| **bootRun + tcf-ui** | UI → tcf-om 직접 Relay | STF `SessionValidator` (설정 시) |
| **bootRun + tcf-uj** | UI → gateway → WAS | Gateway 4단계 + Cookie 전달 |
| **ztomcat** | `/ui` 또는 `/uj` → `/gw` → WAS | 동일, context path만 다름 |

로컬 테스트: `admin01` / `nsight01!`  
OM Admin: http://localhost:8099/om/admin/login.html (bootRun tcf-ui)

---

## 11. 개발자 체크리스트

1. 로그인·세션 API는 **`OM.Auth.*`** serviceId 사용 (직접 HttpSession API 남용 금지)
2. UI 경유 시 **Cookie Relay**에 의존 — `userId`를 header에 수동 넣을 때는 gateway enrich 흐름 확인
3. 업무 WAS에서 세션 필수 거래는 **`session-validation-enabled: true`** + header `userId` 보장
4. gateway 경유 시 **tcf-om 기동 + SESSIONDB 공유** 선행
