# TCF 예외 처리 정리

NSIGHT TCF의 예외 처리는 **HTTP 상태 코드가 아닌 표준 응답 전문(`StandardResponse.result`)** 으로 성공·실패를 전달하는 것이 기본 원칙입니다.

| 관점 | 역할 | 핵심 |
|------|------|------|
| **응답 계약** | 성공·실패 판별 | `result.resultCode` — `S0000` / `E0001` |
| **업무 예외** | 예상 가능한 오류 | `BusinessException` → `ETF.businessFail()` |
| **시스템 예외** | 비의도적 오류 | `Exception` → `ETF.systemError()` |
| **파이프라인 밖** | TCF 진입 전 오류 | `GlobalStandardExceptionHandler` |
| **오류코드 마스터** | 운영·화면 메시지 | `OM_ERROR_CODE` + `OM.ErrorCode.*` |
| **비표준 경로** | TCF 미경유 REST | UD 파일 API 등 — 수동 Map 조립 |

---

## 1. 기본 원칙

| 원칙 | 설명 |
|------|------|
| **HTTP 200 우선** | TCF 파이프라인 내 오류도 대부분 HTTP 200 + `resultCode: E0001` |
| **이중 코드** | `resultCode`(S0000/E0001) + `errorCode`(상세) |
| **의도적 vs 비의도적** | `BusinessException` = 업무·검증, 그 외 = 시스템 |
| **스택 미노출** | 클라이언트에는 `errorMessage`·`errorDetail`(요약)만 |
| **계층 위임** | Handler는 catch하지 않고 TCF·ETF에 위임 |

```text
업무 코드 throw BusinessException
        │
        ▼
TCF.process() catch
        │
        ├─ BusinessException  → ETF.businessFail()  → errorCode = 예외 코드
        ├─ Timeout (변환됨)   → ETF.businessFail()  → E-TCF-TIME-*
        └─ Exception          → ETF.systemError()   → E-COM-SYS-0001
        │
        ▼
StandardResponse JSON (HTTP 200)
```

---

## 2. 전체 구조

```text
[Client] POST StandardRequest
   │
   ▼
OnlineTransactionController ──(예외)──► GlobalStandardExceptionHandler
   │
   ▼
TCF.process()
   │
   ├─ STF ──────────────► BusinessException? ──┐
   ├─ Dispatcher ───────► BusinessException? ──┤
   ├─ Handler → Facade → Service → Rule         │
   │              │         └── BusinessException? ──┤
   │              └── @Transactional 롤백              │
   └─ Exception (NPE, SQL…) ──► systemError ────────┤
                                                    ▼
                                         ETF → StandardResponse
                                                    │
[Client] ◄──────────────────────────────────────────┘
   result.resultCode / errorCode / errorMessage
```

## 3. 예외 유형

### 3.1 BusinessException (표준 업무 예외)

**위치:** `tcf-core/.../error/BusinessException.java`

```java
throw new BusinessException(ErrorCode.BUSINESS_ERROR, "요청 Body가 비어 있습니다.");
throw new BusinessException("E-OM-BIZ-0002", "사용자를 찾을 수 없습니다.");
```

| 항목 | 내용 |
|------|------|
| 용도 | 검증 실패, 데이터 없음, 중복, 권한 부족 등 **예상 가능한** 오류 |
| 발생 위치 | STF, Dispatcher, Rule, Service, IdempotencyChecker … |
| 처리 | `TCF` → `ETF.businessFail()` |
| 응답 | `resultCode=E0001`, `errorCode=예외 코드`, `errorMessage=예외 message` |

### 3.2 SystemException

cause를 보존하는 래퍼로 정의되어 있으나, 현재 TCF는 **`Exception`을 직접 catch**합니다.  
신규 코드에서는 unchecked 예외를 그대로 던져 `ETF.systemError()`로 처리하거나, 인프라 오류에 `SystemException`을 사용합니다.

### 3.3 기타 Exception (비의도적 시스템 오류)

| 항목 | 값 |
|------|-----|
| 예시 | `NullPointerException`, `DataAccessException` |
| 처리 | `ETF.systemError()` |
| `errorCode` | `E-COM-SYS-0001` 고정 |
| `errorMessage` | `시스템 오류가 발생했습니다.` |
| `errorDetail` | 예외 클래스 단순명 |
| 서버 로그 | SLF4J `error` + stack trace |

### 3.4 Timeout 예외 (BusinessException으로 변환)

`TimeoutExceptionResolver`가 Spring TX·SQL timeout을 `BusinessException`으로 변환한 뒤 `businessFail` 경로로 처리합니다.

| 코드 | 의미 |
|------|------|
| `E-TCF-TIME-001` | Online timeout |
| `E-TCF-TIME-002` | Transaction timeout |
| `E-TCF-TIME-003` | DB Query timeout |

상세: [타임아웃관리.md](타임아웃관리.md)

## 4. TCF.process() catch 분기

```59:79:tcf-core/src/main/java/com/nh/nsight/tcf/core/processor/TCF.java
        } catch (BusinessException e) {
            StandardResponse<Object> response = etf.businessFail(request, e, context, clientHeader);
            return response;
        } catch (Exception e) {
            var timeoutError = TimeoutExceptionResolver.toBusinessException(e);
            if (timeoutError.isPresent()) {
                StandardResponse<Object> response = etf.businessFail(request, timeoutError.get(), context,
                        clientHeader);
                return response;
            }
            StandardResponse<Object> response = etf.systemError(request, e, context, clientHeader);
            return response;
```

| catch 순서 | 예외 | ETF |
|------------|------|-----|
| 1 | `BusinessException` | `businessFail()` |
| 2 | Timeout 계열 (변환 후) | `businessFail()` |
| 3 | 그 외 `Exception` | `systemError()` |

## 5. ETF — 응답 조립 + 종료 처리

| ETF 메서드 | 트리거 | errorCode | 부가 처리 |
|------------|--------|-----------|-----------|
| `success()` | 정상 | — | 멱등성 success, 거래로그 S0000, 감사, 메트릭 |
| `businessFail()` | `BusinessException` | `e.getErrorCode()` | 멱등성 fail, 거래로그 E0001 |
| `systemError()` | `Exception` | `E-COM-SYS-0001` | SLF4J error 로그 + 동일 종료 |

```56:72:tcf-core/src/main/java/com/nh/nsight/tcf/core/processor/ETF.java
    public StandardResponse<Object> businessFail(...) {
        // ...
            transactionLogService.end(context, "E0001", e.getErrorCode());
        // ...
        return StandardResponse.fail(header, e.getErrorCode(), e.getMessage(), null);
    }
```

**실패 시에도** 멱등성 종료·거래로그·감사·메트릭을 기록합니다.  
업무 DB는 Facade `@Transactional` 롤백되어도 **거래로그는 FAIL로 별도 커밋**됩니다.

## 6. Result · 응답 전문 매핑

### 6.1 Result 필드

| 필드 | 성공 | 실패 |
|------|------|------|
| `resultCode` | `S0000` | `E0001` (고정) |
| `resultMessage` | `정상 처리되었습니다.` | `처리 중 오류가 발생했습니다.` (고정) |
| `errorCode` | — | `E-COM-...` / `E-OM-...` |
| `errorMessage` | — | `BusinessException.getMessage()` |
| `errorDetail` | — | systemError 시 클래스명 등 |
| `errorSystemId` | — | `NSIGHT-MP` |
| `errorDateTime` | — | KST ISO-8601 |

`Result.fail()` 구현:

```23:32:tcf-core/src/main/java/com/nh/nsight/tcf/core/message/Result.java
        result.errorCode = errorCode;
        // ...
    }
```

## 7. 처리 경로 3종

### 7.1 ① TCF 파이프라인 (주 경로)

```text
STF → Dispatcher → Handler → Facade → Service → Rule
  → BusinessException / Exception
  → ETF.businessFail / systemError
  → StandardResponse (HTTP 200)
```

### 7.2 ② 파이프라인 밖 — GlobalStandardExceptionHandler

`TCF.process()` **이전**에 터지는 예외용 안전망.

**위치:** `tcf-web/.../GlobalStandardExceptionHandler.java`

| `@ExceptionHandler` | errorCode | 비고 |
|---------------------|-----------|------|
| `BusinessException` | 예외 코드 | TCF 밖에서 던진 경우 |
| `MethodArgumentNotValidException` | `E-COM-VALID-0001` | Bean Validation |
| `HttpRequestMethodNotSupportedException` | `E-COM-VALID-0001` | GET 접근 차단 안내 |
| `Exception` | `E-COM-SYS-0001` | 기타 |

`header`가 없을 수 있어 `StandardResponse.fail(null, ...)` 형태입니다.

### 7.3 ③ TCF 미경유 — UD REST 등

파일 업·다운로드는 Controller `try/catch` → `OmUpdownloadResponseSupport.fail()`.  
TCF STF/ETF·거래로그·멱등성 검사는 적용되지 않습니다.

## 8. 오류 코드 체계

### 8.1 명명 규칙

```text
E-{영역}-{분류}-{일련번호}

영역  : COM(공통), OM, SV, UD, TCF …
분류  : VALID, BIZ, AUTH, SYS, VAL, TIME, CTL, HDR …
```

| 예 | 의미 |
|----|------|
| `E-COM-VALID-0001` | Header 검증 실패 |
| `E-COM-DISP-0001` | 미등록 serviceId |
| `E-OM-BIZ-0002` | OM — 대상 데이터 없음 |
| `E-OM-VAL-0002` | OM — 필수값 누락 |
| `E-TCF-CTL-001` | 거래통제 차단 |
| `E-TCF-TIME-001` | Online timeout |

### 8.2 프레임워크 공통 코드 (`ErrorCode`)

**위치:** `tcf-core/.../error/ErrorCode.java`

| 상수 | 코드 | 발생 위치 |
|------|------|-----------|
| `INVALID_HEADER` | `E-COM-VALID-0001` | `StandardHeaderValidator`, Dispatcher |
| `SERVICE_NOT_FOUND` | `E-COM-DISP-0001` | `TransactionDispatcher` |
| `SESSION_INVALID` | `E-COM-AUTH-0001` | `SessionValidator` |
| `AUTHORIZATION_DENIED` | `E-COM-AUTH-0002` | `AuthorizationValidator` |
| `DUPLICATE_REQUEST` | `E-COM-IDEMP-0001` | `IdempotencyChecker` |
| `TXCTRL_NOT_ALLOWED` | `E-TCF-CTL-001` | `TransactionControlService` |
| `TIMEOUT_ONLINE` ~ `TIMEOUT_EXTERNAL_READ` | `E-TCF-TIME-001~006` | Timeout 계층 |
| `BUSINESS_ERROR` | `E-COM-BIZ-0001` | 업무 Rule 범용 |
| `SYSTEM_ERROR` | `E-COM-SYS-0001` | `ETF.systemError()` |

### 8.3 업무·OM 상세 코드

- **OM**: `E-OM-*` — Service·Rule에서 사용, `OM_ERROR_CODE` 테이블 마스터
- **샘플 업무**(SV, EB …): `ErrorCode.BUSINESS_ERROR` 또는 `E-{BC}-BIZ-*` 권장
- **UD REST**: `E-UD-VAL-0001`, `E-UD-BIZ-0001`, `E-UD-SYS-0001` (TCF 미경유)

### 8.4 OM_ERROR_CODE 마스터

| 컬럼 | 용도 |
|------|------|
| `ERROR_CODE` | PK — `E-OM-BIZ-0002` 등 |
| `ERROR_CATEGORY` | BIZ, AUTH, VALIDATION, DB, SYS … |
| `USER_MESSAGE` | 최종 사용자 안내 |
| `OPERATOR_MESSAGE` | 운영자 설명 |
| `ACTION_GUIDE` | 조치 가이드 |

**OM Admin:** `/ui/om/admin/error-code.html` — `OM.ErrorCode.inquiry/save/update/delete`

런타임 `BusinessException.errorMessage`는 **개발 시점 메시지**이며, 장기적으로 마스터 `USER_MESSAGE`와 연동할 수 있습니다.

## 9. 계층별 throw 가이드

| 계층 | 예외 | errorCode 예 | 비고 |
|------|------|--------------|------|
| **STF / Validator** | `BusinessException` | `E-COM-VALID-0001`, `E-COM-AUTH-*` | Header·세션·권한 |
| **Dispatcher** | `BusinessException` | `E-COM-DISP-0001` | 미등록 serviceId |
| **Rule** | `BusinessException` | `E-COM-BIZ-0001`, `E-{BC}-VAL-*` | 입력 검증 |
| **Service** | `BusinessException` | `E-{BC}-BIZ-*` | 없음, 중복, 상태 오류 |
| **DAO** | (권장) 예외 전파 | — | `DataAccessException` → systemError |
| **Handler** | **던지지 않음** | — | TCF에 위임 |
| **Facade** | (권장) Service 위임 | — | `@Transactional` 롤백 + 전파 |

### Rule 예시 (SV 샘플)

```java
if (body == null || body.isEmpty()) {
    throw new BusinessException(ErrorCode.BUSINESS_ERROR, "요청 Body가 비어 있습니다.");
}
```

### Service 예시 (OM)

```java
if (row == null) {
    throw new BusinessException("E-OM-BIZ-0002", "오류코드를 찾을 수 없습니다.");
}
```

### Handler·Facade와 트랜잭션

```text
Service throw BusinessException
  → Facade @Transactional 롤백
  → Handler 예외 전파 (catch 금지)
  → TCF catch → ETF.businessFail()
  → TCF_TX_LOG FAIL 기록 (별도 커밋)
```

## 10. 릴레이·클라이언트 오류 처리

### 10.1 릴레이 연결 오류 (TCF 체계 밖)

`tcf-ui` `TransactionRelayService` 연결 실패:

```json
{"error":"...","targetUrl":"...","hint":"대상 WAS가 기동 중인지 확인하세요."}
```

### 10.2 클라이언트 판별 순서

```text
1. relay.httpStatus < 400  (릴레이 사용 시)
2. payload.result.resultCode === "S0000"
3. 실패 시 result.errorCode, result.errorMessage 표시
```

```javascript
if (payload.result?.resultCode !== 'S0000') {
  const msg = payload.result.errorMessage || payload.result.resultMessage;
  const code = payload.result.errorCode;
}
```

**HTTP 상태만으로 성공을 판단하지 않습니다.**

## 11. 오류와 로그·감사

| 이벤트 | 기록 |
|--------|------|
| `BusinessException` | ETF `transactionLog.end(E0001, errorCode)`, audit, metric |
| `systemError` | SLF4J error + 동일 종료 로그 |
| MDC | `guid`, `serviceId` — 로그 상관 |

`errorDetail`은 클라이언트용 요약이며, 상세 진단은 **서버 application 로그**와 `TCF_TX_LOG.ERROR_CODE`를 확인합니다.

## 12. 권장 · 금지

### 12.1 권장

| 항목 | 내용 |
|------|------|
| 업무 오류 | `BusinessException(errorCode, userMessage)` |
| errorCode | `E-{BC}-*` 규칙 + OM 마스터 등록 |
| 메시지 | 사용자·운영자가 이해할 수 있는 한글 |
| 검증 | Rule 계층 선행, Service는 비즈니스 판단 |
| 로깅 | 시스템 오류만 `log.error` + stack trace |

### 12.2 금지

| 금지 | 이유 |
|------|------|
| Handler에서 catch 후 `StandardResponse` 직접 생성 | ETF·거래로그·멱등성 우회 |
| `/online`에서 HTTP 4xx/5xx로 업무 실패 반환 | 채널 계약 위반 |
| `errorCode` 없이 `RuntimeException`만 던지기 | `E-COM-SYS-0001`로 뭉개짐 |
| 스택 트레이스를 `errorMessage`에 포함 | 보안·UX 저하 |
| Rule에서 DAO 호출 | 계층 혼재 |

## 13. 신규 오류코드 등록 절차

1. **코드 할당** — `E-{BC}-{분류}-{번호}` 규칙 준수
2. **OM_ERROR_CODE 등록** — OM Admin 또는 `data.sql` (개발)
3. **코드 throw** — Rule/Service에서 `BusinessException` 사용
4. **Service Catalog** — transactionCode·Handler와 함께 문서화
5. **샘플·테스트** — `tcf-ui/sample-requests` 또는 거래 테스트 UI로 실패 케이스 확인

---

## 14. 개발자 체크리스트

1. 업무 오류는 **`BusinessException`** + 명시적 **`errorCode`**
2. Handler는 **catch하지 않고** TCF에 위임
3. 클라이언트는 **`result.resultCode == S0000`** 으로 성공 판별
4. 시스템 오류는 **`E-COM-SYS-0001`** — 원인은 서버 로그에서 확인
5. Timeout 오류는 **`E-TCF-TIME-*`** — [타임아웃관리.md](타임아웃관리.md) 참조
6. OM 업무는 **`OM_ERROR_CODE`** 마스터에 등록
7. UD REST 등 TCF 미경유 API는 **별도 응답 규약** 확인
