# TCF 캐시 관리 정리

NSIGHT TCF의 캐시는 `tcf-cache` 모듈이 제공하는 **Spring Cache + Ehcache 3 (JCache)** 기반 공통 구조입니다.  
**애플리케이션 캐시(Ehcache)** 와 **운영 관리(OM)** 를 분리합니다.

| 관점 | 역할 | 핵심 |
|------|------|------|
| **캐시 엔진** | 메모리 조회 가속 | Ehcache 3 + JCache |
| **선언적 캐시** | AOP `@Cacheable` | `OmCommonCodeCacheService` |
| **프로그램 API** | snapshot/evict | `TcfCacheSupport` |
| **운영 관리** | 조회·무효화 | `OM.Cache.inquiry/delete` |
| **정합성** | DB가 기준 | 변경 시 **evict** |
| **모듈** | 라이브러리 | `tcf-cache` — `tcf-om` 의존 |

AOP 동작: [AOP.md](AOP.md)

---

## 1. 개요

| 목적 | 설명 |
|------|------|
| **성능** | 반복 DB 조회 감소 |
| **안정성** | DB 부하 완충 |
| **운영** | OM 화면에서 캐시 상태 조회·삭제 |

```text
Service
  ├─ cache hit  → 메모리 즉시 반환
  └─ cache miss → DAO → DB → cache put
```

캐시는 DB를 **대체하지 않습니다**. 영속 정합성 기준은 **DB**이며, 변경 시 **명시적 evict**로 일치시킵니다.

---

## 2. 모듈 구조

```text
tcf-cache (라이브러리)
  ├─ ehcache.xml                 캐시 영역·TTL·heap
  ├─ TcfCacheNames               캐시 alias 상수
  ├─ TcfCacheSupport             snapshot / evict API
  └─ AutoConfiguration           Spring Cache + @EnableCaching

tcf-om
  ├─ OmCommonCodeCacheService    @Cacheable / @CacheEvict
  ├─ OmCommonCodeService         조회 시 캐시, CUD 후 evict
  ├─ OmCacheService              OM.Cache.* 운영
  └─ /ui/om/admin/cache.html     운영 화면
```

---

## 3. 기술 스택

| 계층 | 기술 |
|------|------|
| 추상화 | Spring Cache (`@Cacheable`, `@CacheEvict`) |
| 구현체 | Ehcache 3 |
| API | JCache (`javax.cache`) |
| 운영 | `TcfCacheSupport` + `OM.Cache.*` |

**커스텀 `@Aspect` 캐시 없음** — Spring 내장 `CacheInterceptor`만 사용.

---

## 4. 설정

### 4.1 tcf-om (예)

```yaml
spring:
  cache:
    type: jcache
    jcache:
      config: classpath:ehcache.xml

nsight:
  tcf:
    cache:
      enabled: true
      config-location: classpath:ehcache.xml
```

### 4.2 비활성화

```yaml
nsight:
  tcf:
    cache:
      enabled: false
```

- `TcfCacheSupport` 빈 미생성
- `@Cacheable` 미적용 → 매번 DB 조회
- `OM.Cache.inquiry` → DAO fallback (`fromEhCache=false`)

## 5. 캐시 영역 (Region)

`ehcache.xml` + `TcfCacheNames`:

| alias | TTL | heap | 용도 | AOP 사용 |
|-------|-----|------|------|----------|
| `commonCode` | 30분 | 200 | OM 공통코드 | ● `@Cacheable` |
| `serviceCatalog` | 60분 | 100 | ServiceId 카탈로그 | ○ 예약 (프로그램 API) |
| `sessionRegion` | 10분 | 100 | 세션 보조 | ○ 예약 |

템플릿 `tcf-default`: TTL 30분, heap 500 entries.

`TcfCacheSupport` TTL 표준(초):

- `commonCode` = 1800
- `serviceCatalog` = 3600
- `sessionRegion` = 600

## 6. 접근 패턴

### 6.1 선언적 캐시 — `@Cacheable` (AOP)

**`OmCommonCodeCacheService`** — 프로젝트 내 유일한 `@Cacheable` 사용처.

```27:38:tcf-om/src/main/java/com/nh/nsight/marketing/om/application/service/OmCommonCodeCacheService.java
    @Cacheable(cacheNames = TcfCacheNames.COMMON_CODE, key = "#codeGroup")
    public List<Map<String, Object>> loadByCodeGroup(String codeGroup) {
        // miss 시 DAO.searchCommonCodes
    }

    @Cacheable(cacheNames = TcfCacheNames.COMMON_CODE, key = "'__ALL_GROUPS__'")
    public List<String> loadAllCodeGroupNames() {
        // miss 시 DAO.selectDistinctCodeGroups
    }
```

```text
Cache hit  → EhCache 반환 (DAO 호출 X)
Cache miss → DAO → DB → put
```

**Self-invocation 주의:** `findInGroup()` 내부의 `this.loadByCodeGroup()` 은 프록시를 우회할 수 있음 — 외부 빈에서 호출 권장.

### 6.2 무효화 — `@CacheEvict`

공통코드 CUD 후 `OmCommonCodeService`가 호출:

```53:58:tcf-om/src/main/java/com/nh/nsight/marketing/om/application/service/OmCommonCodeCacheService.java
    @Caching(evict = {
            @CacheEvict(cacheNames = TcfCacheNames.COMMON_CODE, key = "#codeGroup"),
            @CacheEvict(cacheNames = TcfCacheNames.COMMON_CODE, key = "'__ALL_GROUPS__'")
    })
    public void evictCodeGroup(String codeGroup) {
```

- 해당 `codeGroup` + 그룹 인덱스(`__ALL_GROUPS__`) 동시 evict
- 상세·목록 불일치 방지

### 6.3 프로그램 방식 — `TcfCacheSupport`

| API | 용도 |
|-----|------|
| `evict(cacheName, key)` | 단일 키 삭제 |
| `evictAll(cacheName)` | 영역 전체 clear |
| `snapshot()` | 영역별 entryCount 요약 |
| `snapshotEntries(nameFilter, keyFilter)` | 엔트리 상세 |

`serviceCatalog` 등은 **AOP 없이** 필요 시 이 API로 직접 제어.

## 7. 운영 관리 (OM)

### 7.1 serviceId

| serviceId | transactionCode | 역할 |
|-----------|-----------------|------|
| `OM.Cache.inquiry` | `OM-CCH-0001` | 캐시 목록/엔트리 조회 |
| `OM.Cache.delete` | `OM-CCH-0002` | 캐시 evict (키 또는 전체) |

**화면:** `/ui/om/admin/cache.html`  
**권한:** `ROLE_OM_CACHE`

### 7.2 OmCacheService 동작

**inquiry**

1. `TcfCacheSupport` 있음 → `snapshotEntries()` — **`fromEhCache=true`**
2. 없음 → `dao.searchCacheStatus()` fallback

**delete**

1. `cacheName` 필수, `deleteReason` 필수 (감사)
2. `cacheKey` 있음 → `evict(name, key)` / 없음 → `evictAll(name)`
3. 권한·감사 이력 기록

**응답 필드**

| 필드 | 의미 |
|------|------|
| `fromEhCache` | 실제 Ehcache 조회 여부 |
| `cacheName` / `cacheKey` | 영역·키 |
| `entryCount` | 엔트리 수 |
| `ttlSec` | TTL(초) |
| `source` | `ehcache` |

## 8. End-to-End — 공통코드 조회

```text
OM.CommonCode.inquiry
  → OmCommonCodeFacade
  → OmCommonCodeService.inquiry()
       → OmCommonCodeCacheService.loadByCodeGroup()  [AOP CacheInterceptor]
            hit  → List 반환
            miss → OmOperationDao.searchCommonCodes → put
  → 응답 body 조립
```

**공통코드 변경 (save/update/delete)**

```text
OmCommonCodeService.save/update/delete
  → DAO CUD
  → commonCodeCacheService.evictCodeGroup(codeGroup)  [AOP CacheEvict]
```

## 9. DB · 거래 계층과의 관계

| 원칙 | 내용 |
|------|------|
| 1 | 영속 **정합성 기준 = DB** |
| 2 | 변경 시 **evict** (자동 또는 OM.Cache.delete) |
| 3 | miss 경로 = 기존 **DAO** 재사용 |
| 4 | 캐시에 **비밀번호·토큰** 저장 금지 |

DAO 계층: [DAO처리.md](DAO처리.md)

---

## 10. 장애 · 운영 시나리오

| 시나리오 | 동작 | 대응 |
|----------|------|------|
| `cache.enabled=false` | DB 직접 조회 증가 | 성능 모니터링 후 재활성 |
| 오래된 데이터 | stale hit | CUD evict 점검, `OM.Cache.delete` |
| Ehcache 빈 없음 | `fromEhCache=false` | `spring.cache.type=jcache` 확인 |
| heap 초과 | LRU eviction | TTL/heap 튜닝, 키 카디널리티 점검 |

---

## 11. 캐시 키 설계 원칙

1. **최소 식별자** — `codeGroup` 수준 (GUID 등 고카디널리티 금지)
2. **예약 키** — `__ALL_GROUPS__` (그룹 인덱스)
3. **alias 일치** — `ehcache.xml` ↔ `TcfCacheNames` ↔ `@Cacheable cacheNames`
4. **evict 쌍** — 상세 키 + 인덱스 키 함께 evict

---

## 12. 보안 · 감사

- `OM.Cache.delete` — `deleteReason` 필수, `OmChangeRecorder` 감사·권한 이력
- 민감정보 캐시 **금지**
- 환경별 TTL 차이는 운영 문서와 함께 관리

---

## 13. 개발자 체크리스트

1. `spring.cache.type=jcache` + `ehcache.xml` 적용 확인
2. `TcfCacheNames`와 **ehcache alias** 일치
3. DB **변경 거래** 후 대응 **evict** 구현
4. `@Cacheable`은 **별도 Service 클래스** + 외부 빈 호출 (self-invocation 방지)
5. `OM.Cache.inquiry/delete`로 운영 evict 테스트
6. `cache.enabled=false` fallback 동작 확인
7. **serviceCatalog** 등 신규 영역 — ehcache.xml + TcfCacheNames + evict 전략
