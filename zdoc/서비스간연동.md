# 서비스 간 연동 (tcf-eai)

> NSIGHT TCF 프레임워크에서 업무 서비스끼리 **HTTP/JSON 표준 전문 + serviceId** 방식으로 호출하기 위한 공통 모듈 `tcf-eai` 설계/사용 가이드.
> 예시 연동: `ic-service` → `sv-service` (IC 고객상세 조회 시 SV 고객요약 연동).

관련 문서: [SV 고객요약 샘플](SV고객요약샘플.md) · [타임아웃관리](타임아웃관리.md) · [환경구성](환경구성.md)

---

## 1. 핵심 원칙

- 업무 서비스는 **다른 서비스의 Java Service 를 직접 참조하지 않는다.**
- 서비스 간 호출은 외부 사용자 호출과 동일하게 `POST /{businessCode}/online` + `header.serviceId` 로 한다.
- 호출 로직은 업무 Service 내부에 `WebClient/RestClient` 를 직접 심지 않고 **Integration Adapter → tcf-eai** 계층으로 분리한다.
- **서비스 간 분산 트랜잭션은 금지.** 이번 모듈은 우선 **조회성 연동**에 사용한다.

```
[ic-service]
  IcCustomerFacade → IcCustomerService
        ↓
  SvIntegrationClient            (adapter: ic-service 내부)
        ↓
[tcf-eai]
  TcfServiceClient → DefaultTcfServiceClient
  (StandardRequestBuilder / HeaderPropagation / ResponseResultValidator / Timeout / ErrorMapping / Log)
        ↓  HTTP POST /sv/online
[sv-service]
  TCF 전처리 → Dispatcher → SV.Customer.selectSummary Handler → Facade → Service → DAO → RDW
```

## 2. 모듈 구조 (`tcf-eai`)

패키지 base: `com.nh.nsight.tcf.eai`

```
tcf-eai
├── client
│   ├── TcfServiceClient.java          (연동 호출 인터페이스)
│   └── DefaultTcfServiceClient.java   (RestClient 기반 구현)
├── config
│   ├── TcfIntegrationProperties.java  (nsight.integration.*)
│   └── TcfIntegrationConfiguration.java (빈 등록 + @EnableConfigurationProperties)
├── model
│   ├── IntegrationCallRequest.java    (대상 businessCode/serviceId/txCode/body)
│   └── IntegrationCallResult.java     (success/resultCode/message/body/elapsedMs)
├── support
│   ├── StandardRequestBuilder.java    (표준 전문 header/body 조립)
│   ├── HeaderPropagationHelper.java   (GUID/TraceId/user/branch/caller 전파)
│   └── ResponseResultValidator.java   (resultCode=S0000 검증, 업무오류 전파)
└── exception
    ├── IntegrationErrorCode.java       (E-TCF-IF-0001/0002, E-TCF-MSG-0001)
    ├── IntegrationException.java       (시스템성 연동 오류)
    ├── IntegrationTimeoutException.java(Timeout)
    └── IntegrationBusinessException.java(대상 업무 실패)
```

> 각 서비스별 Adapter(`SvIntegrationClient` 등)는 **호출하는 쪽 서비스 모듈**(ic-service)에 둔다.
> 공통 모듈(tcf-eai)이 특정 업무 응답 타입에 의존하지 않도록 하기 위함이며, 응답은 `Map<String,Object>` 로 다룬다.

의존 방향: `tcf-core → tcf-eai → *-service`
`settings.gradle` 에 `include 'tcf-eai'`, 호출 서비스 `build.gradle` 에 `implementation project(':tcf-eai')`.

## 3. 설정 (`application-*.yml`)

```yaml
nsight:
  integration:
    default-timeout-ms: 3000
    services:
      SV:
        base-url: http://127.0.0.1:8086   # 운영: http://sv-service:8080
        context-path: /sv
        online-path: /online
        connect-timeout-ms: 1000
        read-timeout-ms: 3000
```

최종 호출 URL = `base-url + context-path + online-path` = `http://127.0.0.1:8086/sv/online`

- `read-timeout-ms` 미지정 시 `default-timeout-ms` 적용.
- 대상 businessCode(key, 예: `SV`) 로 endpoint 를 조회한다.

## 4. 호출 방식

### 4.0 Adapter 없이 한 줄 호출 (범용 헬퍼)

대상별 Adapter 를 만들지 않고 `TcfServiceClient` 편의 메서드로 바로 호출할 수 있다.

```java
@Service
public class SomeService {
    private final TcfServiceClient tcfServiceClient;   // 생성자 주입

    public Map<String, Object> callSv(String customerNo, TransactionContext ctx) {
        // 성공 시 대상 body(Map) 반환. 업무실패/Timeout/시스템오류는 IntegrationException 계열로 전파
        return tcfServiceClient.callForBody(
                "SV", "SV.Customer.selectSummary", "SV-INQ-0002",
                Map.of("customerNo", customerNo), ctx);
    }
}
```

- `call(bizCode, serviceId, txCode, body, ctx)` → `IntegrationCallResult`(resultCode/message/body/elapsedMs)
- `callForBody(bizCode, serviceId, txCode, body, ctx)` → 대상 body(Map) 직접 반환
- 세밀한 제어(serviceName/processingType 지정)가 필요하면 `IntegrationCallRequest.builder()` + `call(request, ctx)` 사용.

> **정적/범용**: `tcf-eai` 는 특정 업무에 묶이지 않는다. `nsight.integration.services.{코드}` 만 등록하면
> IC→SV, IC→MG, SV→PC 등 **모든 업무 조합**을 동일 방식으로 호출할 수 있다.
> 반복 호출이 많은 대상만 선택적으로 Adapter(예: `SvIntegrationClient`)로 감싸 가독성을 높인다.

### 4.1 Adapter (ic-service) — 편의 래퍼 예시

```java
@Component
public class SvIntegrationClient {
    private final TcfServiceClient tcfServiceClient;

    public Map<String, Object> selectCustomerSummary(String customerNo, TransactionContext ctx) {
        return tcfServiceClient.callForBody(
                "SV", "SV.Customer.selectSummary", "SV-INQ-0002",
                Map.of("customerNo", customerNo), ctx);
    }
}
```

### 4.2 업무 Service 에서 사용 (ic-service)

```java
Map<String,Object> svSummary = svIntegrationClient.selectCustomerSummary(customerNo, context);
// IC 자체 정보 + svSummary 조립 후 반환
```

### 4.3 실제 전송 전문 (tcf-eai 가 조립)

```json
{
  "header": {
    "systemId": "NSIGHT-MP",
    "businessCode": "SV",
    "serviceId": "SV.Customer.selectSummary",
    "transactionCode": "SV-INQ-0002",
    "processingType": "INQUIRY",
    "guid": "<원 거래 GUID 유지>",
    "traceId": "<원 거래 TraceId 유지>",
    "channelId": "IC",
    "userId": "U123456",
    "branchId": "001234",
    "callerBusinessCode": "IC",
    "callerServiceId": "IC.Customer.inquiry"
  },
  "body": { "customerNo": "CUST00000001" }
}
```

## 5. 호출 위치 기준

| 계층 | 허용 | 기준 |
|---|---|---|
| Controller | 금지 | HTTP 수신만 |
| Handler | 제한 | Facade 호출만 |
| Facade | 허용 | 유스케이스 조립, 트랜잭션 경계 |
| Service | 허용 | 업무 흐름상 외부 조회 |
| Rule | 제한 | 순수 업무 규칙 (외부 호출 금지) |
| DAO/Mapper | 금지 | DB 접근만 |
| Integration Adapter | 권장 | 서비스 간 호출 전담 |

---

## 6. 트랜잭션 주의사항

- IC DB 트랜잭션과 SV DB 트랜잭션을 **하나로 묶지 않는다.**
- 조회성 연동(IC 조회 중 SV 요약 참조)만 실시간 HTTP 호출로 처리.
- 저장+저장 동시 처리 필요 시 → 이벤트/Outbox/보상처리/Saga 검토.

예제(`IcCustomerFacade`)는 `@Transactional(readOnly = true)` 조회 경계 안에서 SV 를 호출한다.

## 7. 표준 오류 처리

| 상황 | 예외 | 오류코드 |
|---|---|---|
| 대상 응답 Timeout | `IntegrationTimeoutException` | `E-TCF-IF-0001` |
| 대상 연결 불가/다운/설정 누락 | `IntegrationException` | `E-TCF-IF-0002` |
| 응답 전문 형식 오류 | `IntegrationException` | `E-TCF-MSG-0001` |
| 대상 업무 실패(resultCode≠S0000) | `IntegrationBusinessException` | 대상 resultCode 전파 |

호출 측(ic-service)은 이 연동 예외를 잡아 `BusinessException(errorCode, message)` 로 변환하며,
최종 응답은 ETF 후처리에서 표준 오류코드/메시지로 노출한다(기술 예외 그대로 노출 금지).

오류코드 시드는 `tcf-om/src/main/resources/data.sql` 의 `OM_ERROR_CODE` 에 등록:
`E-TCF-IF-0001`, `E-TCF-IF-0002`, `E-TCF-MSG-0001`.

## 8. 카탈로그 등록

`OM_SERVICE_CATALOG` (data.sql + `ServiceCatalogSeedData.java`):

| CATALOG_ID | serviceId | txCode | Handler | timeout |
|---|---|---|---|---|
| CAT-074 | IC.Customer.inquiry | IC-INQ-0002 | IcCustomerHandler | 5s |
| CAT-002 | SV.Customer.selectSummary | SV-INQ-0002 | SvCustomerHandler | 3s |
| CAT-075 | SV.Integration.icSample | SV-INT-0001 | SvIntegrationHandler | 5s |

> **필수 헤더 주의**: 대상 서비스의 거래통제는 `serviceName` 을 필수로 요구한다(`E-TCF-HDR-004`).
> 편의 메서드(`call`/`callForBody`)는 `serviceName` 인자가 없으므로, `StandardRequestBuilder` 가
> 미지정 시 `serviceId` 로 자동 보완한다. 표시용 이름을 명시하려면 `IntegrationCallRequest.builder().serviceName(..)` 사용.

## 8-1. 구현된 데모 조합 (양방향 검증 완료)

| 방향 | 진입 serviceId | 연동 대상 | Adapter |
|---|---|---|---|
| IC → SV | IC.Customer.inquiry | SV.Customer.selectSummary | `ic-service` `SvIntegrationClient` |
| SV → IC | SV.Integration.icSample | IC.Sample.inquiry | `sv-service` `IcIntegrationClient` |

두 조합 모두 `tcf-eai` 공통 모듈만 사용하며, 서비스별 설정은 `nsight.integration.services.{대상코드}` 등록뿐이다.
(IC 는 `services.SV`, SV 는 `services.IC` 를 각각 자신의 `application-local.yml` 에 등록.)

## 9. 로컬 테스트 절차

1. `gradle :sv-service:bootRun` (포트 8086)
2. `gradle :ic-service:bootRun` (포트 8082)
3. IC 호출:

```
POST http://127.0.0.1:8082/ic/online
Content-Type: application/json
```
body = `tcf-ui/src/main/resources/sample-requests/ic-customer-inquiry.json` (IC.Customer.inquiry)

또는 tcf-ui 멀티 화면의 IC → "Customer Inquiry (SV 연동)" 샘플 사용.
IC 응답 body 에 `customerGrade / totalBalance / lastTransactionDate` (SV 연동 결과)가 포함되면 정상.

## 10. 최종 권장안

| 구분 | 기준 |
|---|---|
| 모듈명 | `tcf-eai` |
| 통신 | HTTP/JSON 표준 전문 |
| 호출 기준 | targetBusinessCode + targetServiceId |
| URL | `/{businessCode}/online` |
| 구현 | Spring `RestClient` (기존 EB→EP 패턴과 동일) |
| Header 전파 | GUID, TraceId, userId, branchId, channelId, callerServiceId |
| Timeout | 기본 3초 (서비스별 override) |
| DB 트랜잭션 | 서비스 간 분산 금지 |
| 호출 위치 | Integration Adapter 계층 |
| 운영 로그 | callerServiceId, targetServiceId, elapsedMs 기록 |
