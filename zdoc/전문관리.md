# TCF 전문(준문) 관리 정리

NSIGHT TCF에서 **전문(준문)** 이란 온라인 거래를 주고받는 **표준 HTTP/JSON 메시지**입니다. 구조 정의·검증·운영·테스트가 **코드(tcf-core) + OM 마스터 + 샘플 JSON** 으로 나뉩니다.

| 관점 | 역할 | 핵심 |
|------|------|------|
| **표준 계약** | 요청/응답 JSON 스키마 | `StandardRequest` / `StandardResponse` |
| **프레임워크 카탈로그** | Header·Result 필드 SoT | `TcfStandardMessageCatalog` (tcf-core) |
| **OM 전문구조** | 업무별 Body·필드 메타 CRUD | `OM_MESSAGE_STRUCT` / `OM_MESSAGE_FIELD` |
| **ServiceId 카탈로그** | 거래 식별·Handler 매핑 | `OM_SERVICE_CATALOG` |
| **샘플·테스트** | 로컬·UI 거래 테스트 | `tcf-ui/.../sample-requests/` |

---

## 1. 전체 구조

```text
Client (브라우저 / curl / 채널)
   │  POST /{code}/online  (application/json)
   ▼
┌─ Transport ─────────────────────────────────────────┐
│  OnlineTransactionController / TcfGateway / Relay   │
└────────────────────────┬────────────────────────────┘
                         │ StandardRequest JSON
┌─ Semantic (tcf-core) ──┴────────────────────────────┐
│  STF  → Header 검증·normalize·guid/traceId          │
│  Dispatcher → Handler (serviceId)                   │
│  BTF  → Service가 body Map 조립                     │
│  ETF  → StandardResponse (header + result + body)   │
└─────────────────────────────────────────────────────┘

전문 정의·운영
   ├─ TcfStandardMessageCatalog     ← 프레임워크 표준 Header/Result
   ├─ OM_MESSAGE_STRUCT/FIELD       ← OM 전문구조 마스터
   ├─ OM_SERVICE_CATALOG            ← serviceId · transactionCode
   └─ sample-requests/*.json          ← tcf-ui 거래 테스트
```

---

## 2. 표준 전문 구조

### 2.1 요청 (StandardRequest)

```json
{
  "header": { ... },
  "body": { ... }
}
```

| 필드 | Java 타입 | 설명 |
|------|-----------|------|
| `header` | `StandardHeader` | 거래 공통 메타 (필수 필드 STF 검증) |
| `body` | `Map<String, Object>` | 업무별 가변 데이터 |

### 2.2 응답 (StandardResponse)

```json
{
  "header": { ... },
  "result": { ... },
  "body": { ... }
}
```

| 필드 | 설명 |
|------|------|
| `header` | 요청 guid/traceId 유지 |
| `result` | `resultCode` — **`S0000` = 성공**, `E0001` = 실패 |
| `body` | 업무 응답 (실패 시 null 가능) |

**성공 판별:** HTTP 200이어도 `result.resultCode == "S0000"` 으로 판단합니다.

### 2.3 전송 규약

| 항목 | 값 |
|------|-----|
| Method | **POST** |
| Content-Type | `application/json; charset=UTF-8` |
| Endpoint | `/online` 또는 `/{businessCode}/online` |
| HTTP Status | 성공·실패 모두 주로 **200 OK** |

---

## 3. 요청 Header (StandardHeader)

### 3.1 필수 필드 (STF 검증)

| 필드 | 설명 |
|------|------|
| `businessCode` | 업무 코드 (`SV`, `OM` …) |
| `serviceId` | 거래 식별자 — Dispatcher 라우팅 키 |
| `transactionCode` | 화면·거래 코드 (`SV-INQ-0001`) |
| `processingType` | `INQUIRY`, `CREATE`, `UPDATE`, `DELETE`, `EXECUTE` … |
| `channelId` | 채널 (`WEBTOP`, `OM-PORTAL` …) |

### 3.2 선택 필드 (프레임워크 보완)

| 필드 | 보완 주체 |
|------|-----------|
| `guid`, `traceId` | STF — 빈 값이면 자동 생성 |
| `clientIp` | Controller — `X-Forwarded-For` / remoteAddr |
| `businessCode` | Controller — URL path `/{code}/online` |
| `userId`, `branchId` | 클라이언트 또는 Gateway enrich |
| `idempotencyKey` | 멱등성 검사 (설정 시) |

### 3.3 serviceId 형식

```text
{BusinessCode}.{Domain}.{action}

예) SV.Sample.inquiry
    OM.MessageStructure.inquiry
    OM.Auth.login
```

---

## 4. Result · 오류 코드

### 4.1 Result 필드

| 필드 | 성공 | 실패 |
|------|------|------|
| `resultCode` | `S0000` | `E0001` |
| `resultMessage` | 정상 처리 메시지 | 오류 요약 |
| `errorCode` | — | `E-COM-...` / `E-OM-...` |
| `errorMessage` | — | 화면용 설명 |

### 4.2 프레임워크 공통 오류 (ErrorCode)

| 코드 | 의미 |
|------|------|
| `E-COM-VALID-0001` | Header 검증 실패 |
| `E-COM-DISP-0001` | 미등록 serviceId |
| `E-COM-AUTH-0001` | 세션 무효 |
| `E-COM-IDEMP-0001` | 멱등 중복 요청 |
| `E-COM-BIZ-0001` | 업무 오류 (BusinessException) |
| `E-COM-SYS-0001` | 시스템 오류 |

업무·OM 상세 오류는 **`OM_ERROR_CODE`** 마스터에서 관리합니다.

## 5. TcfStandardMessageCatalog (프레임워크 SoT)

**위치:** `tcf-core/.../message/catalog/TcfStandardMessageCatalog.java`

tcf-core message 패키지의 **Single Source of Truth**입니다. STF `StandardHeaderValidator`·OM 시드·전문구조 frameworkInquiry가 이 정의를 따릅니다.

| structCode | 구분 | 설명 |
|------------|------|------|
| `STD-REQ-HEADER` | REQUEST / HEADER | 표준 요청 Header 필드 목록 |
| `STD-RES-HEADER` | RESPONSE / HEADER | 표준 응답 Header |
| `STD-RES-RESULT` | RESPONSE / RESULT | result.resultCode 등 |
| `STD-REQ-BODY` | REQUEST / BODY | Body 샘플 필드 예시 (업무 가변) |

기동 시 `MessageStructureSeedData`가 위 4개 템플릿을 **`OM_MESSAGE_STRUCT`** 에 MERGE합니다.

## 6. OM 전문구조 관리

### 6.1 테이블

| 테이블 | 용도 |
|--------|------|
| `OM_MESSAGE_STRUCT` | 전문 구조(세그먼트) 마스터 |
| `OM_MESSAGE_FIELD` | 구조별 필드 정의 (키·타입·필수·샘플값) |

**OM_MESSAGE_STRUCT 주요 컬럼:** `STRUCT_CODE`, `BUSINESS_CODE`, `SERVICE_ID`, `TRANSACTION_CODE`, `MESSAGE_TYPE`, `SEGMENT_TYPE`, `STRUCT_NAME`, `SAMPLE_JSON`, `USE_YN`

**OM_MESSAGE_FIELD 주요 컬럼:** `FIELD_KEY`, `DATA_TYPE`, `REQUIRED_YN`, `MAX_LENGTH`, `SAMPLE_VALUE`, `VALIDATION_RULE`, `SORT_ORDER`

### 6.2 serviceId (OM.MessageStructure.*)

| serviceId | 역할 |
|-----------|------|
| `OM.MessageStructure.inquiry` | 전문구조 목록 조회 |
| `OM.MessageStructure.detail` | 구조 + 필드 상세 |
| `OM.MessageStructure.frameworkInquiry` | TCF 표준 템플릿 (`TcfStandardMessageCatalog`) |
| `OM.MessageStructure.save` | 전문구조 등록 |
| `OM.MessageStructure.update` | 전문구조 수정 |
| `OM.MessageStructure.delete` | 전문구조 삭제 |

**구현:** `OmMessageStructureFacade` → `OmMessageStructureService` → `OmOperationDao`

### 6.3 OM Admin 화면

| 화면 | 경로 |
|------|------|
| 전문 조립 / 테스트 | `/ui/om/admin/message-composer.html` |
| ServiceId 카탈로그 | `/ui/om/admin/service-catalog.html` |
| 오류코드 | `/ui/om/admin/error-code.html` |

`message-composer.html`에서 표준 Header/Body를 조립하고 Relay로 거래 테스트를 수행합니다.

## 7. ServiceId 카탈로그 (OM_SERVICE_CATALOG)

전문 Header의 `serviceId`·`transactionCode`는 **`OM_SERVICE_CATALOG`** 에 등록됩니다.

| 항목 | 설명 |
|------|------|
| `SERVICE_ID` | `{BC}.{Domain}.{action}` |
| `TRANSACTION_CODE` | `{BC}-{ACTION3}-{NNNN}` |
| `HANDLER_CLASS` | Handler 클래스명 (참조) |
| `PROCESSING_TYPE` | INQUIRY / CREATE / … |
| `TIMEOUT_SEC` | 기본 Timeout (정책 시드 연계) |

Handler 실제 등록은 Spring `@Component` + `TransactionDispatcher` 기동 시 `serviceId()` 맵입니다. 카탈로그는 **운영·문서·Timeout 시드** 용도입니다.

## 8. 샘플 JSON (tcf-ui)

**경로:** `tcf-ui/src/main/resources/sample-requests/`

| 파일 패턴 | 예 |
|-----------|-----|
| `{bc}-sample-inquiry.json` | `sv-sample-inquiry.json` |
| `{bc}-user-create.json` | `eb-user-create.json` |
| `{bc}-transactions.json` | `om-transactions.json`, `sv-transactions.json` |

**로컬 호출:**

```bash
curl -X POST http://localhost:8086/sv/online \
  -H "Content-Type: application/json" \
  -d @tcf-ui/src/main/resources/sample-requests/sv-sample-inquiry.json
```

**UI 테스트:** http://localhost:8099/sv/index.html (bootRun tcf-ui)

> 샘플 JSON의 `transactionIntime`, `systemDate` 등 **StandardHeader에 없는 필드**는 Jackson이 무시합니다. 서버가 인식하는 Header는 `StandardHeader` 정의 필드만 해당합니다.

## 9. 진입점 유형

| 유형 | 컴포넌트 | TCF 파이프라인 |
|------|----------|----------------|
| **A. 표준 JSON** | `OnlineTransactionController` | ● STF → Handler → ETF |
| **B. TcfGateway** | REST/multipart → `StandardRequest` 변환 | ● 동일 |
| **C. UI Relay** | `TransactionRelayService` | ○ JSON 투명 전달 (대상 WAS에서 TCF) |
| **D. 비표준 REST** | `OmUpdownloadFileController` 등 | ✕ TCF 미경유 |

## 10. 전문 처리 흐름 (한 건)

```text
1. Client — StandardRequest JSON 작성
2. Controller — businessCode · clientIp 보완
3. STF — Header 검증 · guid/traceId · 세션·통제·Timeout
4. Dispatcher — serviceId → Handler
5. Handler → Facade → Service — body 처리·응답 Map 조립
6. ETF — StandardResponse (result + body)
7. Client — result.resultCode == "S0000" 확인
```

**Body 조립:** Service 계층 책임. ETF는 조립된 Map을 `StandardResponse.body`에 넣을 뿐 추가 가공하지 않습니다.

---

## 11. GUID / TraceId · 로깅

| 필드 | 생성 | 용도 |
|------|------|------|
| `guid` | STF (`GuidGenerator`) | 거래 단위 ID · 거래로그 · MDC |
| `traceId` | STF | 분산 추적 |

MDC 키: `guid`, `traceId`, `serviceId`, `userId`, `branchId` — logback 패턴과 `TX_START`/`TX_END` 로그 상관.

---

## 12. 개발자 체크리스트

1. **serviceId** — `{BC}.{Domain}.{action}` 형식, Handler `serviceId()` 와 **1:1**
2. **transactionCode** — `OM_SERVICE_CATALOG` / 샘플 JSON과 일치
3. **Header 필수 5필드** — businessCode, serviceId, transactionCode, processingType, channelId
4. **Body** — 업무별 자유 정의, Rule에서 검증
5. **응답** — Service가 Map 조립, 성공 시 `resultCode=S0000`
6. **샘플 JSON** — `tcf-ui/.../sample-requests/` 추가·갱신
7. **전문구조** — 신규 업무 Body 필드 메타는 OM `OM_MESSAGE_STRUCT` 등록 (운영 환경)
