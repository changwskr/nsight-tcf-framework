# TCF 온라인 처리 정리

NSIGHT TCF의 **온라인 처리**는 HTTP JSON 전문 1건을 **단일 파이프라인**(`TCF.process()`)으로 처리하는 동기 거래 모델입니다.

| 관점 | 역할 | 핵심 |
|------|------|------|
| **진입점** | HTTP → 표준 전문 | `OnlineTransactionController`, Gateway, Relay |
| **TCF 파이프라인** | 전처리 → 업무 → 후처리 | STF → Dispatcher → ETF |
| **업무 계층** | serviceId별 로직 | Handler → Facade → Service → DAO |
| **거래 계약** | 성공/실패 판별 | HTTP 200 + `result.resultCode` (`S0000` / `E0001`) |
| **추적** | 상관키·거래로그 | `guid`, `traceId`, `TCF_TX_LOG` |

---

## 1. 개념: 두 가지 "트랜잭션" 구분

```text
┌──────────────────────────────────────────────────────────────┐
│  TCF 온라인 거래 (1 HTTP Request = 1 TCF Transaction)         │
│  STF ──→ Handler ──→ ETF                                     │
│         └─ Facade @Transactional (DB TX, 선택)                 │
└──────────────────────────────────────────────────────────────┘
```

| 용어 | 의미 | 담당 |
|------|------|------|
| **TCF 온라인 거래** | JSON 전문 1건의 처리 파이프라인 | `TCF.process()` — STF → Dispatcher → ETF |
| **DB 트랜잭션** | 업무 DB 커밋/롤백 단위 | Facade `@Transactional` (Spring) |

- 비즈니스 예외 시: DB 롤백 + ETF 실패 응답이 **동시** 수행
- 거래 로그(`TCF_TX_LOG`)는 업무 TX와 **별도 DS**로 커밋되어 추적 가능

---

## 2. 전체 구조

```text
Client / Channel
   │
   ├─ 직접: POST /{businessCode}/online
   ├─ UI Relay: /api/relay/{code}/online (tcf-ui)
   └─ Gateway: tcf-gateway → downstream /online
           │
           ▼
OnlineTransactionController (또는 TcfGateway)
   │  header 보정 (businessCode, clientIp)
   ▼
TCF.process()
   ├─ STF.preProcess()           전처리·검증·컨텍스트
   ├─ OnlineTransactionTimeoutExecutor
   │     └─ TransactionDispatcher.dispatch()
   │           └─ TransactionHandler.doHandle()
   │                 └─ Facade → Service → Rule → DAO
   └─ ETF.success() / businessFail() / systemError()
           │
           ▼
StandardResponse JSON
```

### TCF 엔진 컴포넌트

| 약어 | 클래스 | 역할 |
|------|--------|------|
| **STF** | `STF` | **S**tandard **T**ransaction **F**ront — 전처리 |
| **TCF** | `TCF` | **T**ransaction **C**ontrol **F**ramework — 오케스트레이션 |
| **ETF** | `ETF` | **E**nd **T**ransaction **F**ramework — 후처리·응답 |

## 3. 진입점

### 3.1 표준: `OnlineTransactionController`

| 엔드포인트 | 설명 |
|------------|------|
| `POST /online` | businessCode는 header에만 존재 |
| `POST /{businessCode}/online` | path의 businessCode를 header에 보정 |

Controller가 수행하는 보정:

1. `header`가 없으면 빈 `StandardHeader` 생성
2. path `{businessCode}` → header에 설정 (header에 없을 때)
3. `clientIp` → `X-Forwarded-For` 또는 `remoteAddr`

이후 `tcf.process(request)` 호출.

### 3.2 UI Relay (`tcf-ui` / `tcf-uj`)

```text
Browser → POST /api/relay/{code}/online
       → TransactionRelayService
       → {target}/online
       → RelayResult(responseBody, httpStatus, elapsedMs, setCookies)
```

- 요청/응답 본문은 **변환 없이** 투명 전달
- Cookie/Set-Cookie 중계로 세션 유지 → [세션관리.md](세션관리.md), [로그인.md](로그인.md)

### 3.3 Gateway (`tcf-gateway`)

multipart/REST 등 비표준 요청을 `TcfInvokeRequest`로 표준 전문 변환 후 TCF 호출.

### 3.4 배포 토폴로지

| 환경 | 경로 예 |
|------|---------|
| bootRun | UI(:8099) → OM(:8097)/SV(:8086) 직접 포트 |
| ztomcat | UI(:8080/ui) → Relay → :8080/{context}/online |

## 4. TCF.process() — 핵심 오케스트레이션

```35:86:tcf-core/src/main/java/com/nh/nsight/tcf/core/processor/TCF.java
    public StandardResponse<Object> process(StandardRequest<Map<String, Object>> request) {
        TransactionContext context = null;
        StandardHeader clientHeader = StandardHeader.copyOf(request == null ? null : request.getHeader());
        // ...
        try {
            context = stf.preProcess(request, clientHeader);
            Object body = onlineTransactionTimeoutExecutor.execute(
                    () -> dispatcher.dispatch(request, dispatchContext));
            return etf.success(request, body, context, clientHeader);
        } catch (BusinessException e) {
            return etf.businessFail(request, e, context, clientHeader);
        } catch (Exception e) {
            // TimeoutExceptionResolver → businessFail (timeout)
            // 그 외 → systemError
        } finally {
            TransactionContextHolder.clear();
            TimeoutContextHolder.clear();
            MDC.clear();
        }
    }
```

| 단계 | 예외 | ETF 분기 | resultCode |
|------|------|----------|------------|
| 정상 | — | `success()` | `S0000` |
| 업무 오류 | `BusinessException` | `businessFail()` | `E0001` + 업무 errorCode |
| 타임아웃 | Timeout → `BusinessException` | `businessFail()` | `E0001` + TIMEOUT errorCode |
| 시스템 오류 | 기타 `Exception` | `systemError()` | `E0001` + `E-COM-SYS-0001` |

**설계 포인트**

- 예외를 HTTP 500으로 올리지 않고 **표준 응답 전문**으로 변환
- `finally`에서 `TransactionContextHolder`·`TimeoutContextHolder`·`MDC` 정리
- `clientHeader`는 클라이언트가 보낸 header 스냅샷 — 응답 header echo에 사용

## 5. STF — 전처리

```47:87:tcf-core/src/main/java/com/nh/nsight/tcf/core/processor/STF.java
    public TransactionContext preProcess(...) {
        headerValidator.validate(request);
        // guid, traceId 부여
        TransactionContext context = new TransactionContext(header, clientHeader);
        TransactionContextHolder.set(context);
        putMdc(header);
        sessionValidator.validate(header);
        authorizationValidator.validate(header);
        transactionControlService.check(header);
        timeoutPolicyService.resolveAndApply(header, context);
        idempotencyChecker.checkAndMarkProcessing(header);
        transactionLogService.start(context);
        return context;
    }
```

| 순서 | 처리 | 실패 시 (대표) |
|------|------|----------------|
| 1 | Header 필수값 검증 | `E-COM-VALID-0001` |
| 2 | `guid`, `traceId` 생성 | — |
| 3 | `TransactionContext` + MDC | — |
| 4 | 세션 검증 (설정 시) | `E-COM-AUTH-0001` |
| 5 | 권한 검증 (설정 시) | `E-COM-AUTH-0002` |
| 6 | 거래통제 | `TCF_TRANSACTION_CONTROL` → [타임아웃관리.md](타임아웃관리.md) |
| 7 | Timeout 정책 적용 | `TCF_SERVICE_TIMEOUT_POLICY` |
| 8 | 멱등성 검사 | `E-COM-IDEMP-0001` |
| 9 | 거래 시작 로그 (`TX_START`) | — |

### Header 필수 필드

- `serviceId`, `businessCode`, `transactionCode`, `processingType`, `channelId`

상세: [전문관리.md](전문관리.md)

### 세션/권한 (설정 기반, 기본 비활성)

| 프로퍼티 | 기본값 |
|----------|--------|
| `nsight.tcf.session-validation-enabled` | `false` |
| `nsight.tcf.authorization-validation-enabled` | `false` |

OM Admin 등 로그인 필수 환경에서는 `true`.

## 6. Dispatcher — serviceId 라우팅

```text
기동 시:  모든 @Component TransactionHandler → Map<serviceId, Handler>
요청 시:  handlerMap.get(serviceId) → handler.handle(request, context)
```

| 상황 | 오류 |
|------|------|
| `serviceId` 없음 | `E-COM-VALID-0001` |
| 미등록 serviceId | `E-COM-DISP-0001` |
| 동일 serviceId 중복 빈 | 기동 시 `IllegalStateException` |

Handler는 `handle()`에서 로깅 후 `doHandle()` 호출 — [어플리케이션계층.md](어플리케이션계층.md)

## 7. Online Timeout — Dispatcher 구간

STF에서 정책을 `TimeoutContextHolder`에 적재한 뒤, **Dispatcher 전체**(Handler → Facade → DAO)를 `OnlineTransactionTimeoutExecutor`가 감쌉니다.

```30:48:tcf-core/src/main/java/com/nh/nsight/tcf/core/timeout/OnlineTransactionTimeoutExecutor.java
    public Object execute(Supplier<Object> action) {
        // timeoutPolicyEnabled + threadPool 있을 때만 적용
        int timeoutSec = policy.getOnlineTimeoutSec() ...;
        return future.get(timeoutSec, TimeUnit.SECONDS);
        // 초과 → BusinessException(TIMEOUT_ONLINE)
    }
```

3계층 Timeout 상세: [타임아웃관리.md](타임아웃관리.md)

## 8. Handler 이후 — 업무 계층

```text
TransactionHandler.doHandle()
  → Facade (@Transactional)     ← DB TX 경계
    → Service (도메인 로직)
      → Rule (검증)
      → DAO/Mapper (영속)
```

- Handler는 **얇게** — 파라미터 변환 + Facade 위임
- `BusinessException`은 TCF가 catch → ETF `businessFail()`
- Facade `@Transactional`, 캐시, 정책 AOP: [AOP.md](AOP.md)

## 9. ETF — 후처리

| 분기 | resultCode | 공통 후처리 |
|------|------------|-------------|
| `success()` | `S0000` | idempotency SUCCESS, TX_END, audit, metric |
| `businessFail()` | `E0001` + errorCode | idempotency FAIL, TX_END, audit, metric |
| `systemError()` | `E0001` + `E-COM-SYS-0001` | 동일 + error 로그 |

## 10. 온라인 계약 (Contract)

### 요청/응답 형식

```json
{ "header": { "serviceId": "SV.Sample.inquiry", ... }, "body": { ... } }
```

```json
{ "header": { ... }, "result": { "resultCode": "S0000", ... }, "body": { ... } }
```

### HTTP vs resultCode 이중 규약

| 항목 | 규칙 |
|------|------|
| HTTP Method | POST only |
| HTTP Status | 성공/업무오류 모두 주로 **200** |
| 실제 성공 여부 | `result.resultCode == "S0000"` |
| 상세 원인 | `errorCode`, `errorMessage`, `errorDetail` |

## 11. TransactionContext — 거래 컨텍스트

요청 1건당 STF에서 1개 생성.

| 속성 | 설명 |
|------|------|
| `header` | 정규화·보완된 `StandardHeader` |
| `clientHeader` | 클라이언트 원본 header 스냅샷 |
| `startTimeMillis` | 거래 시작 시각 |
| `attributes` | 거래 범위 임의 속성 Map |
| `elapsedMillis()` | 현재까지 소요 시간 |

| 접근 | 용도 |
|------|------|
| `TransactionContextHolder` (ThreadLocal) | Handler·Service·DAO 전 구간 |
| 메서드 인자 전달 | Handler → Facade → Service 명시적 전달 |
