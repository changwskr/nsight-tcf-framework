# 4. 개발 표준 전체 요약

> **NSIGHT TCF 개발 Manual** · 원본: [`znsight-guide-word`](../znsight-guide-word/) · 갱신: 2026-07-05

## 4. 개발 표준 전체 요약

### 4.1 도입 전 안내말

본 장은 NSIGHT TCF Framework 기반 개발자가 반드시 따라야 할 개발 표준 전체 기준을 한눈에 정리한다.
NSIGHT 개발 표준은 단순히 Java 코딩 스타일만 의미하지 않는다. 업무코드, URL Context, WAR, Package, ServiceId, 거래코드, Handler, Mapper, SQL ID, 오류코드, 로그, 테스트, 배포 단위가 하나의 기준으로 연결되어야 한다.
NSIGHT Java 코딩 스타일 가이드도 Java/Spring Boot 기반 정보계 애플리케이션의 개발 표준, 코딩 스타일, 품질 기준, MyBatis SQL Mapper 기준을 통합 정의하고 있으며, 적용 대상에는 NSIGHT 마케팅플랫폼, SingleView, 공통모듈, 배치·연계 Java 모듈, MyBatis SQL Mapper가 포함된다.또한 NSIGHT MyBatis 표준에서는 MyBatis를 DAO/Mapper 계층에 두고, SQL 표준화, RDW/ADW 접근 경계, Query Timeout, Mapper 표준, 관측성 통제를 위한 메커니즘으로 정의한다.
핵심은 다음이다.
NSIGHT 개발 표준은 “코드 작성 규칙”이 아니라 “운영 가능한 거래를 만드는 규칙”이다.

### 4.2 개발 표준 전체 결론

NSIGHT 개발 표준은 다음 12개 축으로 정리한다.
| No | 표준 영역 | 핵심 기준 |
| --- | --- | --- |
| 1 | 업무코드 표준 | 모든 개발 산출물은 업무코드를 기준으로 시작한다. |
| 2 | URL Context 표준 | 업무코드와 Context Path를 일치시킨다. |
| 3 | WAR 표준 | 업무별 독립 WAR 단위로 배포한다. |
| 4 | Package 표준 | 업무코드 기반 Package 구조를 사용한다. |
| 5 | ServiceId 표준 | 실제 실행 기준은 URL이 아니라 header.serviceId이다. |
| 6 | 거래코드 표준 | 거래 추적을 위해 업무코드 + 유형 + 일련번호 체계를 사용한다. |
| 7 | 계층구조 표준 | Handler → Facade → Service → Rule → DAO/Mapper 구조를 따른다. |
| 8 | DTO 표준 | Request DTO, Response DTO, 내부 Command/Query 객체를 분리한다. |
| 9 | MyBatis 표준 | Mapper Interface와 XML을 업무별로 분리하고 SQL ID를 표준화한다. |
| 10 | 예외·오류코드 표준 | BusinessException과 표준 오류코드를 사용한다. |
| 11 | 로그·감사 표준 | GUID, TraceId, ServiceId, 거래코드 기준으로 추적 가능하게 한다. |
| 12 | 테스트·배포 표준 | 단위/통합/거래 테스트와 CI/CD 품질 게이트를 통과해야 한다. |

### 4.3 개발 표준의 기본 연결 구조

NSIGHT 개발 산출물은 다음 순서로 연결되어야 한다.
```text
업무코드
↓
URL Context
↓
WAR
↓
Package
↓
ServiceId
↓
TransactionHandler
↓
Facade
↓
Service
↓
Rule
↓
DAO / Mapper
↓
MyBatis XML / SQL ID
↓
거래로그 / 감사로그
↓
오류코드 / 운영관리 OM
```

예를 들어 SV 고객 요약 조회 거래는 다음처럼 연결된다.
| 구분 | 표준 예시 |
| --- | --- |
| 업무코드 | SV |
| 업무명 | Single View |
| Context Path | /sv |
| WAR | sv.war |
| Package | com.nh.nsight.sv |
| Endpoint | POST /sv/online |
| ServiceId | SV.Customer.selectSummary |
| 거래코드 | SV-INQ-0001 |
| Handler | SvCustomerSelectSummaryHandler |
| Facade | SvCustomerFacade |
| Service | SvCustomerService |
| Rule | SvCustomerRule |
| DAO | SvCustomerDao |
| Mapper | SvCustomerMapper |
| Mapper XML | mapper/sv/SvCustomerMapper.xml |
| SQL ID | selectCustomerSummary |
| 오류코드 | E-SV-BIZ-0001 |

### 4.4 업무코드 / Context / WAR 표준 요약

업무코드는 개발 표준의 최상위 기준이다.업무코드가 정해지면 URL Context, WAR명, Package, ServiceId, Mapper 위치가 함께 결정된다.
| 구분 | 표준 규칙 | 예시 |
| --- | --- | --- |
| 업무코드 | 대문자 2자리 또는 2~3자리 | SV, IC, OM |
| Context Path | 업무코드 소문자 | /sv, /ic, /om |
| WAR명 | 업무코드 소문자 + .war | sv.war, ic.war, om.war |
| Package | 업무코드 소문자 포함 | com.nh.nsight.sv |
| ServiceId Prefix | 업무코드 대문자 | SV.Customer.selectSummary |
| Mapper 경로 | 업무코드 소문자 기준 | mapper/sv |
| 오류코드 Prefix | 업무코드 대문자 | E-SV-BIZ-0001 |

기본 원칙은 다음이다.
업무코드 = 업무 경계
Context = 호출 경로
WAR = 배포 단위
Package = 소스 경계
ServiceId = 실행 단위

### 4.5 ServiceId 표준 요약

ServiceId는 NSIGHT 온라인 거래의 실행 기준이다.Controller Method나 URL이 아니라 header.serviceId를 기준으로 Handler가 선택된다.
```text
POST /sv/online
↓
header.serviceId = SV.Customer.selectSummary
↓
TransactionDispatcher
↓
SvCustomerSelectSummaryHandler 실행
```

ServiceId 명명 규칙은 다음과 같다.
{업무코드}.{업무도메인}.{행위}

| 구분 | 작성 기준 | 예시 |
| --- | --- | --- |
| 업무코드 | 대문자 | SV |
| 업무도메인 | PascalCase 또는 명확한 영문명 | Customer |
| 행위 | camelCase 동사 + 목적어 | selectSummary |
| 전체 형식 | 업무코드.도메인.행위 | SV.Customer.selectSummary |

권장 행위명은 다음과 같다.
| 처리유형 | 권장 행위명 | 예시 |
| --- | --- | --- |
| 단건 조회 | select, selectDetail | SV.Customer.selectDetail |
| 요약 조회 | selectSummary | SV.Customer.selectSummary |
| 목록 조회 | searchList, selectList | CM.Campaign.searchList |
| 등록 | create, register, save | OM.User.create |
| 수정 | update, modify | OM.User.update |
| 삭제 | delete, remove | OM.User.delete |
| 승인 | approve | OM.Deploy.approve |
| 실행 | execute, run | BAT.Job.execute |
| 다운로드 | download | UD.File.download |
| 업로드 | upload | UD.File.upload |

### 4.6 거래코드 표준 요약

거래코드는 운영 추적과 거래 분류를 위한 식별자이다.ServiceId가 프로그램 실행 기준이라면, 거래코드는 운영·감사·로그 기준의 거래 식별자이다.
권장 형식은 다음과 같다.
{업무코드}-{처리유형}-{일련번호}

| 처리유형 | 의미 | 예시 |
| --- | --- | --- |
| INQ | 조회 | SV-INQ-0001 |
| REG | 등록 | OM-REG-0001 |
| UPD | 수정 | OM-UPD-0001 |
| DEL | 삭제 | OM-DEL-0001 |
| EXE | 실행 | BAT-EXE-0001 |
| DWN | 다운로드 | UD-DWN-0001 |
| UPL | 업로드 | UD-UPL-0001 |
| APR | 승인 | OM-APR-0001 |
| SND | 발송 | MG-SND-0001 |

거래코드는 반드시 다음 정보와 함께 관리한다.
| 관리 항목 | 설명 |
| --- | --- |
| 거래코드 | SV-INQ-0001 |
| ServiceId | SV.Customer.selectSummary |
| 거래명 | 고객 요약 조회 |
| 업무코드 | SV |
| 처리유형 | 조회 |
| Timeout | 3초 또는 5초 |
| 감사로그 여부 | Y/N |
| 권한 필요 여부 | Y/N |
| 사용 여부 | Y/N |

### 4.7 Package 표준 요약

업무 WAR 내부 Package는 계층 책임이 드러나도록 구성한다.
com.nh.nsight.{businessCode}
```text
 ├─ entry
 │   ├─ handler
 │   └─ facade
 ├─ application
 │   ├─ service
 │   └─ rule
 ├─ persistence
 │   ├─ dao
 │   └─ mapper
 ├─ domain
 │   ├─ model
 │   └─ dto
 ├─ integration
 │   └─ client
 ├─ support
 │   └─ util
 └─ config
```

| Package | 역할 | 예시 |
| --- | --- | --- |
| entry.handler | ServiceId별 업무 진입점 | SvCustomerSelectSummaryHandler |
| entry.facade | 유스케이스 조립, 트랜잭션 경계 | SvCustomerFacade |
| application.service | 업무 처리 절차 | SvCustomerService |
| application.rule | 업무 규칙, 조건 판단 | SvCustomerRule |
| persistence.dao | Mapper 호출 캡슐화 | SvCustomerDao |
| persistence.mapper | MyBatis Mapper Interface | SvCustomerMapper |
| domain.dto | Request/Response DTO | SvCustomerSummaryRequest |
| domain.model | 업무 도메인 객체 | SvCustomer |
| integration.client | 외부·타업무 호출 Client | SvIntegrationClient |
| support.util | 업무 내부 Utility | SvMaskingUtil |
| config | 업무별 Spring 설정 | SvServiceConfig |

### 4.9 계층별 개발 표준 요약

| 계층 | 해야 할 일 | 하지 말아야 할 일 |
| --- | --- | --- |
| Handler | Body를 DTO로 변환하고 Facade 호출 | SQL 실행, 복잡한 업무 로직, 응답 전문 직접 조립 |
| Facade | 트랜잭션 경계 설정, 유스케이스 조립 | SQL 직접 실행 |
| Service | 업무 처리 절차 수행, Rule/DAO/Client 호출 | HTTP 응답 직접 생성 |
| Rule | 업무 규칙, 조건 판단, 계산, 검증 | Mapper 직접 호출 남발 |
| DAO | Mapper 호출 캡슐화 | 업무 판단 로직 작성 |
| Mapper | SQL 실행 Interface | 업무 로직 작성 |
| Mapper XML | SQL 작성 | 화면 로직, 업무 규칙 과다 포함 |
| ETF | 표준 응답 조립, 오류코드 매핑 | 업무별 임의 응답으로 대체 금지 |

### 4.10 DTO 작성 표준 요약

DTO는 목적에 따라 분리한다.
| DTO 유형 | 용도 | 예시 |
| --- | --- | --- |
| Request DTO | 화면/API 요청 Body 수신 | SvCustomerSummaryRequest |
| Response DTO | 화면/API 응답 Body 반환 | SvCustomerSummaryResponse |
| Query DTO | 조회 조건 내부 전달 | SvCustomerSearchQuery |
| Command DTO | 등록·수정 내부 전달 | SvCustomerUpdateCommand |
| Result DTO | DAO/Mapper 조회 결과 | SvCustomerSummaryResult |
| Page DTO | 목록 조회 페이징 정보 | PageRequest, PageResult |

DTO 작성 원칙은 다음과 같다.
| 원칙 | 설명 |
| --- | --- |
| 요청과 응답 분리 | Request DTO와 Response DTO를 같은 클래스로 쓰지 않는다. |
| 화면 DTO와 DB Result 분리 | DB 컬럼 구조가 화면 응답에 직접 노출되지 않게 한다. |
| 불필요한 상속 금지 | DTO는 단순하고 명확하게 유지한다. |
| 민감정보 포함 주의 | 주민번호, 계좌번호, 전화번호 등은 마스킹 기준을 적용한다. |
| Validation 명확화 | 필수값, 길이, 형식, 코드값 검증 기준을 둔다. |

### 4.11 MyBatis / SQL 표준 요약

MyBatis는 DAO/Mapper 계층에 위치하며, 업무 로직을 수행하는 곳이 아니라 SQL 실행 책임만 갖는다. NSIGHT MyBatis 표준에서도 DAO/Mapper는 업무 로직을 수행하지 않고 SQL 실행 책임만 갖는다고 정의한다.
| 구분 | 표준 기준 |
| --- | --- |
| Mapper Interface | persistence.mapper 패키지에 위치 |
| Mapper XML | src/main/resources/mapper/{업무코드} 하위 위치 |
| SQL ID | 기능을 알 수 있게 동사 + 대상 형태 사용 |
| ResultMap | 복잡한 결과는 명시적 ResultMap 사용 |
| RDW/ADW 구분 | 실시간 조회는 RDW, 분석/대량 조회는 ADW |
| Query Timeout | 온라인 SQL은 Timeout 기준 적용 |
| Paging | 목록 조회는 반드시 Paging 적용 |
| Full Scan | 운영 SQL에서 원칙적으로 금지 |
| 동적 SQL | 과도한 조건 분기는 지양 |
| SQL 로그 | SQL ID와 ServiceId 추적 가능해야 함 |

SQL ID 예시는 다음과 같다.
| 유형 | SQL ID 예시 |
| --- | --- |
| 단건 조회 | selectCustomerSummary |
| 상세 조회 | selectCustomerDetail |
| 목록 조회 | selectCustomerList |
| Count 조회 | countCustomerList |
| 등록 | insertCustomer |
| 수정 | updateCustomer |
| 삭제 | deleteCustomer |

### 4.12 예외처리 / 오류코드 표준 요약

예외는 업무별로 임의 응답을 만들지 않고, 표준 예외와 오류코드로 처리한다.
| 구분 | 표준 |
| --- | --- |
| 업무 오류 | BusinessException |
| 시스템 오류 | SystemException |
| 입력값 오류 | ValidationException |
| 인증 오류 | AuthenticationException 또는 표준 인증 예외 |
| 권한 오류 | AuthorizationException 또는 표준 권한 예외 |
| Timeout 오류 | TimeoutException 또는 TCF Timeout 예외 |
| 오류 응답 | ETF에서 StandardResponse.result로 조립 |
| 오류코드 관리 | OM 오류코드 관리 기준 |

오류코드 형식은 다음을 권장한다.
E-{DOMAIN}-{CATEGORY}-{NNNN}

| 오류코드 | 의미 |
| --- | --- |
| E-TCF-HDR-0001 | Header 필수값 누락 |
| E-TCF-CTL-0001 | 거래통제 미등록 |
| E-TCF-AUTH-0001 | 권한 없음 |
| E-SV-BIZ-0001 | SV 업무 오류 |
| E-OM-CODE-0001 | OM 공통코드 오류 |
| E-UD-FILE-0001 | 파일 처리 오류 |

### 4.13 로그 / 감사 / 추적 표준 요약

로그는 단순 디버깅 메시지가 아니라 운영 추적 수단이다.
| 로그 항목 | 설명 |
| --- | --- |
| GUID | End-to-End 거래 추적 ID |
| TraceId | 내부 처리 흐름 추적 ID |
| ServiceId | 실행 서비스 식별자 |
| TransactionCode | 거래코드 |
| BusinessCode | 업무코드 |
| User | 사용자 ID |
| Branch | 지점 코드 |
| ChannelId | 채널 ID |
| SQL ID | 실행 SQL 식별자 |
| Elapsed Time | 처리시간 |
| Result Code | 성공/실패/Timeout 결과 |

거래 상태는 다음 기준으로 관리한다.
```text
PROCESSING
↓
SUCCESS / FAIL / TIMEOUT / UNKNOWN
```

거래로그 설계 기준에서도 거래 시작 시 PROCESSING 상태를 생성하고, 종료 시 SUCCESS / FAIL / TIMEOUT / UNKNOWN 상태로 갱신하는 구조를 권장한다.

### 4.14 보안 개발 표준 요약

| 보안 영역 | 개발 기준 |
| --- | --- |
| 인증 | SSO 또는 JWT 인증 결과를 신뢰하되, TCF에서 세션/JWT를 검증한다. |
| 권한 | ServiceId, 메뉴, 기능권한 기준으로 통제한다. |
| 세션 | Spring Session JDBC 기준으로 관리한다. |
| 입력값 검증 | 필수값, 길이, 형식, 코드값을 검증한다. |
| SQL Injection 방지 | MyBatis 바인딩 파라미터를 사용하고 문자열 결합 SQL을 금지한다. |
| 개인정보 보호 | 응답과 로그에 마스킹을 적용한다. |
| 민감정보 로그 금지 | 비밀번호, 토큰, 주민번호, 계좌번호 원문 로그 금지 |
| 파일 보안 | 업로드 확장자, 크기, 저장 위치, 다운로드 권한을 검증한다. |
| 외부연계 보안 | 내부 호출 인증, Timeout, 로그, 재시도 제한을 적용한다. |

### 4.15 Timeout / 트랜잭션 표준 요약

Timeout은 거래통제 테이블과 분리된 별도 정책으로 관리한다. 기존 Timeout 설계에서도 거래 허용 여부는 TCF_TRANSACTION_CONTROL, 실행 제한시간은 TCF_SERVICE_TIMEOUT_POLICY로 분리하는 구조를 권장한다.
| 구분 | 표준 기준 |
| --- | --- |
| 온라인 거래 Timeout | ServiceId별 정책 기준 |
| DB Transaction Timeout | Facade 트랜잭션 기준 |
| MyBatis Query Timeout | SQL 실행 기준 |
| Hikari Connection Timeout | DB Connection 대기 제한 |
| Gateway Timeout | Downstream 업무 호출 제한 |
| 외부연계 Timeout | 외부 API 호출 제한 |
| ReadOnly | 조회 거래는 ReadOnly 적용 |
| Rollback | 업무 오류와 시스템 오류 기준 명확화 |
| 장시간 거래 | 별도 배치 또는 비동기 처리 검토 |

트랜잭션 적용 기준은 다음과 같다.
| 거래 유형 | 트랜잭션 기준 |
| --- | --- |
| 단순 조회 | ReadOnly |
| 단건 등록 | Required |
| 단건 수정 | Required |
| 복합 업무 | Facade 단위 Required |
| 외부연계 포함 | 보상처리 또는 상태관리 필요 |
| 대량 처리 | 온라인 금지, Batch 분리 권장 |

### 4.16 Cache / 공통코드 표준 요약

Cache는 업무 데이터를 무분별하게 저장하는 기능이 아니다.기준정보와 정책정보를 빠르고 안정적으로 조회하기 위한 공통 구조이다.
| Cache 대상 | 캐시 여부 | 설명 |
| --- | --- | --- |
| 공통코드 | 허용 | 화면 코드, 업무 분기 기준 |
| ServiceId Catalog | 허용 | Dispatcher/OM 조회 기준 |
| 거래통제 정책 | 허용 | Allow-List 조회 성능 확보 |
| Timeout 정책 | 허용 | 서비스별 제한시간 조회 |
| 권한 정책 | 제한적 허용 | 변경 즉시 Evict 필요 |
| 사용자 세션 | 원칙적으로 SESSIONDB 기준 | Cache 남용 금지 |
| 업무 조회 결과 | 원칙적으로 금지 | 데이터 정합성 문제 |
| 대량 목록 | 금지 | Heap 사용량 증가 |

Cache 관리 설계에서도 업무 데이터는 캐시하지 않고, 기준정보·정책정보·카탈로그성 데이터만 캐시하는 것을 핵심 원칙으로 둔다.

### 4.17 테스트 표준 요약

| 테스트 유형 | 대상 | 기준 |
| --- | --- | --- |
| 단위 테스트 | Rule, Service | 업무 규칙과 처리 흐름 검증 |
| Handler 테스트 | Handler | Body 변환과 Facade 호출 검증 |
| Mapper 테스트 | MyBatis Mapper | SQL, ResultMap, Paging 검증 |
| 통합 테스트 | Handler → Mapper | 전체 업무 흐름 검증 |
| TCF 거래 테스트 | StandardRequest/Response | 표준 전문 처리 검증 |
| 오류 테스트 | 예외 발생 거래 | 오류코드와 메시지 검증 |
| 권한 테스트 | 미권한 사용자 | 차단 여부 검증 |
| Timeout 테스트 | 지연 거래 | Timeout 처리 검증 |
| 성능 테스트 | 주요 조회 거래 | 응답시간, SQL 시간 검증 |
| 배포 테스트 | WAR, Health Check | 기동과 정상 응답 확인 |

개발 완료 기준은 다음이다.
코드 작성 완료
  ≠ 개발 완료

개발 완료
  = 코드 작성
  + 단위 테스트
  + Mapper 테스트
  + TCF 거래 테스트
  + 오류/권한/Timeout 테스트
  + 코드 리뷰
  + CI/CD 빌드 통과
  + OM 기준정보 등록 확인

### 4.18 CI/CD / 배포 표준 요약

NSIGHT 구축·운영 표준서에서는 형상관리를 GitLab, CI/CD를 GitLab Runner와 Gradle 기준으로 정의하고 있다. 또한 CI/CD 운영 매뉴얼은 GitLab, GitLab Runner, Gradle, 13 WAR 배포 (deploy-wars.sh)를 기준으로 개발 표준화, 형상관리 표준화, 빌드 자동화, 배포 자동화, 롤백 자동화, 운영 안정성 확보를 목표로 둔다.
| 구분 | 표준 기준 |
| --- | --- |
| 형상관리 | GitLab |
| Build Tool | Gradle |
| 배포 산출물 | 업무별 WAR |
| 배포 단위 | 업무 WAR 단위 |
| 빌드 검증 | Compile, Unit Test, Code Inspection |
| 배포 검증 | Health Check, Smoke Test |
| 롤백 | 이전 WAR 복구 |
| 이력관리 | OM 배포이력 또는 CI/CD 이력 |
| 운영 반영 | 승인 기반 배포 |

배포 흐름은 다음과 같다.
```text
개발자 Commit
↓
Merge Request
↓
Code Review
↓
Gradle Build
↓
Unit Test / Quality Gate
↓
WAR 생성
↓
Artifact 저장
↓
배포 승인
↓
Tomcat 배포
↓
Health Check
↓
서비스 반영
```

### 4.20 마무리말

NSIGHT 개발 표준의 핵심은 모든 개발 산출물을 하나의 거래 단위로 연결하는 것이다.
개발자는 Java Class 하나, Mapper XML 하나, SQL 하나를 작성하더라도 그것이 ServiceId, 거래코드, 거래통제, Timeout, 거래로그, 감사로그, 오류코드, 배포 단위와 연결된다는 점을 이해해야 한다.
정리하면 다음과 같다.
개발 표준
= 업무코드 표준
+ ServiceId 표준
+ 계층구조 표준
+ MyBatis 표준
+ 오류/로그 표준
+ 보안/Timeout 표준
+ 테스트/배포 표준

### 4.21 시사점

NSIGHT 개발 표준을 지키면 다음 효과가 있다.
| 효과 | 설명 |
| --- | --- |
| 개발 일관성 확보 | 업무팀이 달라도 동일한 구조로 개발할 수 있다. |
| 운영 추적성 확보 | GUID, ServiceId, 거래코드, SQL ID 기준으로 장애 분석이 가능하다. |
| 보안 통제 강화 | 세션, 권한, 거래통제, 마스킹 기준을 공통 적용할 수 있다. |
| 장애 확산 방지 | Timeout, Query Timeout, Pool 보호 기준을 개발 단계에서 반영할 수 있다. |
| 배포 안정성 확보 | WAR 단위 빌드, 테스트, Health Check 기준이 명확해진다. |
| 유지보수성 향상 | Handler, Facade, Service, Rule, DAO/Mapper 책임이 분리되어 수정 영향이 줄어든다. |
| PMO 대응력 강화 | 표준화된 개발 기준, 산출물, 체크리스트로 개발 품질을 설명할 수 있다. |

최종 문장은 다음이다.
NSIGHT 개발 표준은 개발자를 통제하기 위한 규칙이 아니라, 개발자가 만든 거래를 운영 가능한 서비스로 완성하기 위한 기준이다.
