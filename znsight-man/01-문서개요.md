# 1. 문서 개요

> **NSIGHT TCF 개발 Manual** · 원본: [`znsight-guide`](../znsight-guide/) · 갱신: 2026-07-05

## 1. 문서 개요

### 1.1 도입 전 안내말

본 문서는 NSIGHT TCF Framework 개발자 가이드이다.
NSIGHT 개발자 가이드는 단순한 코딩 규칙 문서가 아니다. 개발자가 신규 업무 서비스를 만들 때 어떤 ServiceId를 등록하고, 어떤 Handler를 작성하며, Facade·Service·Rule·DAO·Mapper를 어떤 책임으로 나누고, 예외·로그·거래통제·Timeout·테스트·배포 기준을 어떻게 지켜야 하는지를 설명하는 실행형 개발 표준서이다.
기존 NSIGHT Java 코딩 스타일 가이드는 Java/Spring Boot 기반 정보계 애플리케이션의 개발 표준, 코딩 스타일, 품질 기준, 실제 소스 개선 기준을 통합 정의하는 문서이며, 적용 대상도 NSIGHT 마케팅플랫폼, SingleView, 공통모듈, 배치·연계성 Java 모듈, MyBatis SQL Mapper를 포함한다.또한 NSIGHT TCF Framework는 REST Resource 방식이 아니라 HTTP/JSON 표준 전문 + ServiceId Dispatcher 방식을 기준으로 하며, 업무 처리는 Handler → Facade → Service → Rule → DAO/Mapper 구조를 따른다.

### 1.2 문서 목적

본 문서의 목적은 NSIGHT TCF Framework 기반 업무 개발자가 동일한 구조, 동일한 명명, 동일한 전문, 동일한 오류·로그·SQL·테스트 기준으로 개발하도록 표준 개발 절차를 제공하는 것이다.
| 목적 | 설명 |
| --- | --- |
| 개발 표준화 | 업무 개발자가 동일한 패키지, 클래스, DTO, Mapper, SQL 구조로 개발하도록 기준을 제공한다. |
| 거래 처리 일관성 확보 | 모든 온라인 거래가 POST /{businessCode}/online과 header.serviceId 기준으로 실행되도록 한다. |
| 계층 책임 분리 | Handler, Facade, Service, Rule, DAO, Mapper의 책임을 명확히 하여 업무 로직과 공통 통제를 분리한다. |
| 운영 추적성 확보 | GUID, TraceId, ServiceId, TransactionCode, SQL ID 기준으로 장애 추적이 가능하도록 한다. |
| 품질 확보 | 예외처리, 오류코드, 로그, 테스트, 코드리뷰, SQL 검증 기준을 개발 단계에서 적용한다. |
| 보안 내재화 | 인증, 권한, 세션, 마스킹, 입력값 검증, 민감정보 로그 금지 기준을 개발 표준에 포함한다. |
| 배포 안정성 확보 | 개발 산출물이 CI/CD, WAR 배포, Health Check, 운영 전환 기준을 통과할 수 있도록 한다. |
| 업무 확산 지원 | SV, IC, OM, CM, MG 등 업무별 서비스를 같은 방식으로 확장할 수 있도록 샘플과 템플릿을 제공한다. |

### 1.3 작성 배경

NSIGHT 마케팅플랫폼은 단순 Spring MVC 애플리케이션 구조가 아니라, 다음과 같은 공통 실행 구조를 가진다.
```text
사용자 / WebTopSuite / 외부 API
↓
GSLB
↓
L4
↓
Apache
↓
tcf-gateway 또는 업무 WAR
↓
TCF Framework
↓
STF 전처리
↓
TransactionDispatcher
↓
업무 Handler
↓
Facade → Service → Rule → DAO / Mapper
↓
RDW / ADW / SESSIONDB / LOGDB
↓
ETF 후처리
↓
표준 응답 전문 반환
```

따라서 개발자가 임의로 Controller, Service, DAO를 구성하면 다음 문제가 발생할 수 있다.
| 문제 | 설명 |
| --- | --- |
| 거래 식별 불일치 | URL, ServiceId, 거래코드, Handler가 서로 맞지 않아 운영 추적이 어려워진다. |
| 계층 책임 혼재 | Controller나 Handler에 업무 로직, SQL, 트랜잭션 처리가 섞일 수 있다. |
| 오류처리 불일치 | 업무별로 오류코드, 사용자 메시지, 운영 메시지 처리 방식이 달라진다. |
| 로그 추적 불가 | GUID, TraceId, ServiceId가 일관되게 남지 않아 장애 분석이 어려워진다. |
| 보안 기준 누락 | 인증, 권한, 세션, 마스킹, 입력값 검증이 업무별로 다르게 적용될 수 있다. |
| SQL 품질 저하 | MyBatis Mapper, SQL ID, Paging, Timeout 기준이 일관되지 않을 수 있다. |
| 운영 반영 지연 | CI/CD, 테스트, 배포, Health Check 기준을 충족하지 못할 수 있다. |

본 개발자 가이드는 이러한 문제를 방지하기 위해 업무 개발자가 반드시 따라야 할 개발 기준과 예제를 제공한다.

### 1.5 적용 범위

본 개발자 가이드는 NSIGHT TCF Framework 기반 개발 전 과정을 적용 범위로 한다.
| 구분 | 적용 범위 |
| --- | --- |
| 개발 구조 | 업무 WAR, OM, Gateway, Batch, JWT, Cache, Integration 모듈 |
| 온라인 거래 | POST /{businessCode}/online 기반 HTTP/JSON 표준 전문 거래 |
| 실행 기준 | header.serviceId 기반 TransactionHandler 실행 |
| 업무 계층 | Handler, Facade, Service, Rule, DAO, Mapper |
| 전문 구조 | StandardRequest, StandardHeader, StandardResponse, Result |
| 공통 처리 | STF 전처리, TransactionDispatcher, ETF 후처리 |
| 데이터 접근 | MyBatis Mapper, Mapper XML, SQL ID, RDW/ADW/SESSIONDB/LOGDB 접근 |
| 예외 처리 | BusinessException, SystemException, ValidationException, 오류코드 매핑 |
| 로그 / 감사 | GUID, TraceId, MDC, 거래로그, 감사로그, 성능로그 |
| 보안 | 세션, SSO, JWT, 권한, 마스킹, 입력값 검증, 민감정보 보호 |
| Timeout | 온라인 거래 Timeout, DB Query Timeout, 외부연계 Timeout |
| Cache | 공통코드, ServiceId, 정책정보 Cache 조회 및 Evict |
| 파일 | 업로드, 다운로드, 파일 메타, 다운로드 감사로그 |
| Batch | Scheduler, Job, 실행이력, 실패 재처리 |
| 테스트 | 단위 테스트, 통합 테스트, SQL 테스트, 거래 전문 테스트 |
| 배포 | Gradle Build, WAR 생성, CI/CD, Health Check, Rollback |

전처리·후처리 요구사항 정의서에서도 적용 계층은 Controller, Filter, Interceptor, Facade, Service, DAO, Integration Adapter까지 포함하고, 공통기능은 전문, GUID, 인증, 권한, 검증, 오류, 로그, 감사, Timeout, 재처리, 모니터링을 포함한다.

### 1.6 비적용 범위

본 문서는 개발자가 따라야 할 애플리케이션 개발 기준을 정의한다. 다음 항목은 본 문서의 직접 적용 범위에서 제외하고, 별도 운영 표준서 또는 인프라 설계서를 따른다.
| 제외 항목 | 별도 기준 문서 |
| --- | --- |
| Apache 설치 및 상세 운영 설정 | Apache 운영 매뉴얼 |
| Tomcat 설치, Connector, JVM, 세션 복제 상세 | Tomcat 운영 매뉴얼 |
| L4 / GSLB 물리 구성 | 인프라 아키텍처 설계서 |
| RDW / ADW 물리 DB 운영 | RDW/ADW 운영 매뉴얼 |
| GitLab Runner 서버 설치 | CI/CD 운영 매뉴얼 |
| 운영 서버 OS, 보안 Agent, 네트워크 설정 | 시스템 운영 표준서 |
| 성능시험 장비 구성 | 성능·장애 검증 계획서 |

다만 개발자는 위 항목을 직접 설정하지 않더라도, 개발 결과물이 해당 운영 기준을 위반하지 않도록 Timeout, Connection Pool, SQL, 로그, Health Check, 배포 단위를 고려해야 한다.

### 1.9 개발 산출물 기준

업무 개발자는 하나의 온라인 거래를 개발할 때 최소한 다음 산출물을 작성해야 한다.
| 산출물 | 설명 |
| --- | --- |
| ServiceId 정의 | 실행할 업무 서비스 식별자 |
| 거래코드 정의 | 업무 거래를 추적하기 위한 거래코드 |
| Handler 클래스 | TCF Dispatcher가 호출하는 업무 진입점 |
| Facade 클래스 | 유스케이스 조립 및 트랜잭션 경계 |
| Service 클래스 | 업무 처리 절차 구현 |
| Rule 클래스 | 업무 규칙, 조건 판단, 검증 로직 |
| DAO 클래스 | Mapper 호출 캡슐화 |
| Mapper Interface | MyBatis Mapper Java Interface |
| Mapper XML | SQL 작성 위치 |
| Request DTO | 요청 Body 구조 |
| Response DTO | 응답 Body 구조 |
| 오류코드 | 업무 오류 및 시스템 오류 코드 |
| 테스트 코드 | 단위 테스트, 통합 테스트, Mapper 테스트 |
| 거래 테스트 전문 | StandardRequest JSON 샘플 |
| 운영 등록 정보 | ServiceId, 거래통제, Timeout, 권한, 메뉴 매핑 |

### 1.10 문서 작성 원칙

본 개발자 가이드는 다음 원칙으로 작성한다.
| 원칙 | 설명 |
| --- | --- |
| 실행 가능성 | 개발자가 문서를 보고 바로 클래스를 만들 수 있어야 한다. |
| 표준 우선 | 개인 개발 습관보다 NSIGHT 표준 구조를 우선한다. |
| 예제 중심 | 설명만 하지 않고 실제 패키지, 클래스, JSON, SQL 예시를 제공한다. |
| 운영 연계 | 개발 결과물이 거래로그, 감사로그, OM, Timeout, 권한, 배포와 연결되어야 한다. |
| 책임 분리 | 각 계층의 역할과 금지사항을 명확히 제시한다. |
| 검증 가능성 | 테스트, 코드 리뷰, 품질 게이트로 확인 가능한 기준을 제시한다. |
| 확장 가능성 | SV 샘플을 IC, PC, CM, MG, OM 등 다른 업무에도 복제 가능하게 작성한다. |

### 1.11 주요 용어 정의

| 용어 | 설명 |
| --- | --- |
| TCF | Transaction Control Framework. NSIGHT 온라인 거래를 표준 전문 방식으로 처리하는 공통 프레임워크 |
| STF | Standard Transaction Front. Header 검증, 세션, 권한, 거래통제, Timeout 정책을 수행하는 전처리 영역 |
| ETF | End Transaction Framework. 표준 응답 조립, 오류코드 매핑, 감사로그, 거래로그 갱신을 수행하는 후처리 영역 |
| StandardRequest | NSIGHT 표준 요청 전문. header + body 구조 |
| StandardResponse | NSIGHT 표준 응답 전문. header + result + body 구조 |
| ServiceId | 업무 Handler 실행 기준이 되는 서비스 식별자. 예: SV.Customer.selectSummary |
| TransactionCode | 거래 추적과 운영 통제를 위한 거래코드. 예: SV-INQ-0001 |
| BusinessCode | 업무 구분 코드. 예: SV, IC, OM, CM |
| TransactionHandler | ServiceId에 매핑되어 실제 업무 처리를 시작하는 진입 클래스 |
| Dispatcher | 요청 Header의 serviceId를 기준으로 실행할 Handler를 찾는 컴포넌트 |
| Facade | 유스케이스 조립과 트랜잭션 경계를 담당하는 계층 |
| Service | 업무 처리 절차를 수행하는 계층 |
| Rule | 업무 규칙, 조건 판단, 검증 로직을 담당하는 계층 |
| DAO | Mapper 호출을 캡슐화하는 DB 접근 계층 |
| Mapper | MyBatis SQL 실행 Interface 및 XML |
| OM | Operation Management. 사용자, 권한, 메뉴, ServiceId, 거래통제, Timeout, 오류코드 등을 관리하는 운영관리 모듈 |
| GUID | End-to-End 거래 추적을 위한 고유 식별자 |
| TraceId | 내부 처리와 외부 연계 흐름을 추적하기 위한 식별자 |
| MDC | 로그에 GUID, TraceId, ServiceId 등을 자동 포함하기 위한 Logging Context |
| Idempotency | 동일 요청이 중복 처리되지 않도록 통제하는 방식 |
| SESSIONDB | Spring Session JDBC 기반 세션 저장 DB |
| LOGDB | 거래로그, 감사로그, 성능로그 저장 DB |

### 1.12 마무리말

본 개발자 가이드는 NSIGHT 업무 개발자가 자유롭게 개발하되, 실행 구조는 표준화하기 위한 기준 문서이다.
업무 개발자는 화면이나 업무 요건에 따라 다양한 서비스를 만들 수 있다. 그러나 모든 거래는 동일한 방식으로 수신되고, 동일한 Header 기준으로 검증되며, 동일한 Dispatcher를 통해 Handler가 실행되고, 동일한 후처리 기준으로 응답과 로그가 남아야 한다.
핵심은 다음 한 문장으로 정리할 수 있다.
개발자는 업무 로직을 구현하고, TCF는 거래 실행을 통제한다.
따라서 본 문서는 개발자에게 단순 코딩 방법을 알려주는 문서가 아니라, NSIGHT 운영 가능한 애플리케이션을 만드는 방법을 제시하는 개발 표준서이다.
