> **NSIGHT TCF 개발 Manual** · 원본: [`znsight-guide-word`](../znsight-guide-word/) · 갱신: 2026-07-05

## 44. 파일 업다운로드 기준

### 44.1 도입 전 안내말

파일 업다운로드는 단순히 파일을 서버에 올리고 내려받는 기능이 아니다.NSIGHT TCF Framework에서는 파일이 고객정보, 엑셀 다운로드, 리포트, 배치 결과, 운영 증적, 첨부자료, 감사 대상 데이터와 연결될 수 있으므로 보안·권한·감사·성능 기준을 함께 설계해야 한다.
NSIGHT 기준에서 파일관리는 OM(Operation Management) 안에 포함된 UD, Upload/Download 기능으로 설계하는 것이 맞다. 설계 방향은 OM 파일관리 = 파일 메타 관리 + 업로드 + 다운로드 + 다운로드 감사로그 + 권한통제 + 보관/삭제 정책, UD 파일 API = 실제 파일 Stream 처리로 분리한다. 또한 파일 원본은 DB와 Session에 저장하지 않고, DB에는 파일 메타정보만 저장하며 실제 파일은 전용 Storage, 파일시스템, NAS 또는 Object Storage에 저장하는 것이 핵심 원칙이다.
핵심 문장은 다음이다.
파일 원본은 DB와 Session에 저장하지 않는다.DB에는 메타정보와 감사이력만 저장하고, 파일 Stream은 UD 전용 API에서 처리한다.

### 44.2 파일 업다운로드 기준 결론

| 구분 | 기준 | 관리 주체 |
| --- | --- | --- |
| OM(Operation Management) | 실행 기능 | UD(Upload/Download) |
| 화면 위치 | OM 파일관리 화면 | Stream 처리 |
| /ud/files 전용 API | 메타 관리 | UD_FILE_META |
| 다운로드 이력 | OM_FILE_DOWNLOAD_LOG | 감사로그 |
| OM_AUDIT_LOG, TCF_TX_LOG 연계 | 파일 저장 | Local/dev는 파일시스템, 운영은 NAS/Object Storage 권장 |
| DB 저장 | 파일 원본 저장 금지, 메타정보만 저장 | Session 저장 |
| 파일 원본, 조회 결과, 다운로드 데이터 저장 금지 | 권한 기준 | 조회, 업로드, 다운로드, 수정, 삭제, 이력조회 권한 분리 |
| 보안 기준 | 확장자, MIME, 파일명, 크기, 경로, 악성코드 검사 | 대용량 기준 |
| 일반 업로드와 대용량 다운로드 경로 분리 | 운영 기준 | 보관기간, 논리삭제, 물리삭제 배치, 감사이력 관리 |

### 44.3 전체 구조

```text
[사용자 / 운영자 / 업무화면]
        ↓
[WebTopSuite / tcf-ui]
        ↓

```

```text
[Apache / Gateway]
        ↓
[tcf-om / UD API]
        ├─ 파일 메타 관리
        ├─ 업로드 검증
        ├─ 다운로드 권한 검증
        ├─ 다운로드 감사로그
        ├─ 보관기간 / 삭제정책
        ↓

```

```text
[Storage]
        ├─ Local FileSystem, dev
        ├─ NAS, prod
        └─ Object Storage, 선택
        ↓
[DB]
        ├─ UD_FILE_META
        ├─ OM_FILE_DOWNLOAD_LOG
        ├─ OM_AUDIT_LOG
        └─ TCF_TX_LOG

```

전처리·후처리 기준에서도 적용 거래에는 다운로드가 포함되며, 적용 공통기능은 전문, GUID, 인증, 권한, 검증, 오류, 로그, 감사, Timeout, 재처리, 모니터링까지 포함된다. 즉 파일 업다운로드도 일반 부가기능이 아니라 TCF 운영통제 대상 거래로 관리해야 한다.

### 44.6 다운로드 기준

다운로드는 단순히 fileId로 파일을 찾아 내려주는 방식이면 안 된다.다운로드 전에는 반드시 권한, 파일 상태, 보관기간, 감사 대상 여부를 확인한다.
```text
[화면]
  다운로드 클릭
        ↓
[Download API]
  fileId 수신
        ↓

```

```text
[Download Validator]
  fileId 존재 확인
  USE_YN 확인
  DELETE_YN 확인
  만료 여부 확인
  다운로드 권한 확인
  데이터권한 확인
  다운로드 사유 확인
        ↓

  파일 존재 확인

        ↓
```

### 44.10 파일 크기 기준

| 구분 | 권장 크기 | 처리 기준 |
| --- | --- | --- |
| 일반 첨부 | 10MB 이하 | 동기 업로드 가능 |
| 일반 운영 파일 | 50MB 이하 | 기본 UD API 처리 |
| 중간 규모 파일 | 50MB ~ 500MB | 별도 Timeout, Stream 처리 |
| 대용량 다운로드 | 500MB 이상 | 비동기 생성 후 다운로드 |
| 초대용량 파일 | 수 GB 이상 | WAS 직접 처리 지양, 전용 파일서버 검토 |

현재 기본 업로드 제한은 50MB 수준으로 두는 것이 적절하다. 다만 대량 엑셀, 리포트, 배치 결과 파일은 일반 온라인 거래와 분리해 비동기 생성 후 다운로드하는 구조가 안전하다.

| 대용량 파일 | 온라인 Thread 장시간 점유 금지 |
| Heap 사용 | 파일 byte[] 전체 보관 금지 |
| DB 저장 | BLOB 저장 지양, 메타만 저장 |
| Timeout | 일반 파일과 대용량 파일 Timeout 분리 |
| 다운로드 속도 | 필요 시 Rate Limit 적용 |
| 동시 다운로드 | 사용자·업무별 동시 건수 제한 |
| 파일 생성 | 대량 엑셀은 비동기 Job 처리 |
| 모니터링 | 파일 처리시간, 실패율, 크기, 건수 수집 |
| 금지 코드 | byte[] bytes = multipartFile.getBytes(); |

대용량 파일에서 getBytes()는 Heap을 급격히 사용하므로 지양한다.
권장 코드
try (InputStream in = multipartFile.getInputStream()) {
    storageService.store(fileId, in, multipartFile.getSize());
}

### 44.14 대용량 다운로드 기준

대량 조회 결과를 즉시 파일로 만들어 응답하면 WAS Thread와 Heap을 오래 점유한다.따라서 대용량 다운로드는 다음 구조를 권장한다.
```text
[화면]
  다운로드 요청
        ↓
[TCF 거래]
  다운로드 Job 등록
        ↓

```

```text
[Batch / Async Worker]
  데이터 조회
  파일 생성
  Storage 저장
  UD_FILE_META 저장
        ↓
[화면]
  생성 상태 조회
        ↓

```

```text
[다운로드]
  생성 완료된 파일 Stream 다운로드
```

| 구분 | 기준 |
| --- | --- |
| 소량 다운로드 | 즉시 생성 가능 |
| 대량 다운로드 | 비동기 생성 |
| 생성 상태 | WAIT, PROCESSING, SUCCESS, FAIL |
| 다운로드 가능 시점 | SUCCESS 이후 |
| 파일 보관기간 | 업무별 1~30일 |
| 감사로그 | 요청 시점과 다운로드 시점 모두 기록 |

### 44.18 Java 구현 예시

#### 44.18.1 Upload Service

```java
@Service
@RequiredArgsConstructor
public class UdFileUploadService {
    private final FileUploadValidator validator;
    private final FileStorageService storageService;
    private final UdFileMetaMapper fileMetaMapper;
    private final AuditLogService auditLogService;
    @Transactional(timeout = 10)
    public FileUploadResponse upload(
            MultipartFile file,
            FileUploadCommand command,
            TransactionContext context
    ) {
        validator.validate(file, command, context);
        String fileId = UUID.randomUUID().toString();
        String originalName = validator.sanitizeFileName(file.getOriginalFilename());
        String storedName = storageService.generateStoredName(originalName);
        StoredFile storedFile = storageService.store(
                fileId,
                storedName,
                file
        );
        UdFileMeta meta = UdFileMeta.builder()
                .fileId(fileId)
                .originalName(originalName)
                .storedName(storedName)
                .storageType(storedFile.getStorageType())
                .storagePath(storedFile.getStoragePath())
                .contentType(file.getContentType())
                .fileSize(file.getSize())
                .fileHash(storedFile.getSha256Hash())
                .description(command.getDescription())
                .businessCode(command.getBusinessCode())
                .uploadUser(context.getSessionContext().getUserId())
                .useYn("Y")
                .deleteYn("N")
                .build();
        fileMetaMapper.insert(meta);
        auditLogService.writeFileUpload(meta, context);
        return FileUploadResponse.from(meta);
    }
}
```

#### 44.18.2 Download Service

```java
@Service
@RequiredArgsConstructor
public class UdFileDownloadService {
    private final UdFileMetaMapper fileMetaMapper;
    private final FileStorageService storageService;
    private final FileDownloadValidator validator;
    private final FileDownloadLogService downloadLogService;
    @Transactional(timeout = 5)
    public FileDownloadResource download(
            String fileId,
            FileDownloadCommand command,
            TransactionContext context
    ) {
        UdFileMeta meta = fileMetaMapper.selectByFileId(fileId);
        validator.validateDownload(meta, command, context);
        InputStreamResource resource =
                storageService.loadAsResource(meta);
        downloadLogService.insertSuccess(meta, command, context);
        return FileDownloadResource.builder()
                .fileName(meta.getOriginalName())
                .contentType(meta.getContentType())
                .contentLength(meta.getFileSize())
                .resource(resource)
                .build();
    }
}
```

### 44.22 마무리말

파일 업다운로드 기준의 핵심은 파일 Stream 처리와 운영통제를 분리하는 것이다.파일 원본은 UD API와 Storage가 처리하고, OM은 메타·권한·감사·보관·삭제 정책을 관리해야 한다.
파일은 고객정보와 운영증적을 포함할 수 있기 때문에 일반 데이터 조회보다 더 강한 보안과 감사 기준이 필요하다. 따라서 NSIGHT에서는 파일 업다운로드를 단순 API가 아니라 권한검증, 감사로그, 보관정책, 삭제정책이 결합된 운영관리 기능으로 설계해야 한다.
### 44.23 시사점

| 관점 | 시사점 | 개발 관점 |
| --- | --- | --- |
| 업무 개발자는 파일 원본을 DB나 Session에 넣지 않고 UD 공통 API를 사용한다 | 운영 관점 | OM에서 파일 목록, 다운로드 이력, 삭제, 보관기간, 감사로그를 관리해야 한다 |
| 보안 관점 | 파일명·확장자·MIME·권한·감사·악성코드 검증이 필수이다 | 아키텍처 관점 |
| 대용량 파일은 온라인 거래와 분리하고, Storage + Metadata + Audit 구조로 관리해야 한다 |  |  |
