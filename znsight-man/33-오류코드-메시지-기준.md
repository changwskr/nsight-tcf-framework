# 33. 오류코드 / 메시지 기준

> **NSIGHT TCF 개발 Manual** · 원본: [`znsight-guide-word`](../znsight-guide-word/) · 갱신: 2026-07-05

## 33. 오류코드 / 메시지 기준

### 33.1 도입 전 안내말

오류코드와 메시지는 단순히 화면에 “오류가 발생했습니다”를 보여주기 위한 문자열이 아니다.NSIGHT TCF Framework에서는 오류코드와 메시지를 거래 통제, 장애 추적, 사용자 안내, 운영자 조치, 감사 대응을 연결하는 운영 표준 체계로 관리한다.
NSIGHT 오류코드 설계 기준에서는 오류 발생 시 Exception → TCF/ETF 표준 오류코드 변환 → OM_ERROR_CODE 메시지 조회 → StandardResponse 조립 → 거래로그/감사로그/모니터링 기록 흐름으로 처리하도록 정의되어 있다.
오류 발생
```text
  ↓
BusinessException / SystemException 발생
  ↓
ETF 또는 GlobalExceptionHandler 수신
  ↓

```

ErrorCode 확인
```text
  ↓
```

OM_ERROR_CODE 조회
```text
  ↓
```

사용자 메시지 / 운영자 메시지 분리
```text
  ↓
StandardResponse 조립
  ↓
거래로그 / 감사로그 / 메트릭 기록
  ↓
화면 또는 호출 시스템에 표준 응답 반환

```

핵심 문장은 다음이다.
오류코드는 개발자가 임의로 만드는 문자열이 아니라, OM에서 관리되는 운영 기준정보이다.사용자 메시지는 쉽게, 운영자 메시지는 원인 추적 가능하게 작성한다.

### 33.2 오류코드 / 메시지 기준 결론

| 구분 | 기준 | 관리 주체 |
| --- | --- | --- |
| OM, Operation Management | 저장 테이블 | OM_ERROR_CODE |
| 처리 위치 | ETF 후처리 + Global Exception Handler | 오류코드 형식 |
| E-{DOMAIN}-{CATEGORY}-{NNNN} | 결과코드 | S0000, E0001, E0002, T0001, U0001, E9999 |
| 메시지 분리 | 사용자 메시지 / 운영자 메시지 / 조치 가이드 | 사용자 메시지 |
| 짧고 이해 가능하게, 기술정보 제거 | 운영자 메시지 | GUID, TraceId, ServiceId, SQL ID, 대상 시스템 등 포함 |
| 조치 가이드 | 운영자가 확인할 화면·테이블·로그 위치 안내 | 상세 오류 |
| 기본 마스킹 | 응답 구조 | StandardResponse.header + result + body |
| 로그 연계 | 거래로그, 감사로그, 오류로그, APM | 변경관리 |

OM 화면에서 등록·수정·사용중지·이력관리
오류코드 설계 기준에서도 오류코드와 메시지는 ETF에서 일원화하고, 운영관리는 OM 오류코드 관리 화면에서 수행하는 구조가 적합하다고 정리되어 있다. 또한 오류코드는 E-{DOMAIN}-{CATEGORY}-{NNNN} 형식을 사용한다.

### 33.3 오류코드 명명 규칙

오류코드는 다음 형식을 따른다.
E-{DOMAIN}-{CATEGORY}-{NNNN}

예시는 다음과 같다.
E-TCF-HDR-0001
E-TCF-CTL-0001
E-SV-BIZ-0001
E-OM-CODE-0001
E-COM-DB-0001
E-COM-TIME-0001
E-UD-FILE-0001

| 구분 | 의미 | 예시 |
| --- | --- | --- |
| E | Error | 오류 |
| DOMAIN | 오류 발생 영역 | TCF, COM, SV, OM, GW, JWT, UD |
| CATEGORY | 오류 분류 | HDR, VAL, BIZ, DB, IF, TIME, AUTHZ |
| NNNN | 일련번호 | 0001, 0002 |

### 33.6 ResultCode와 ErrorCode 분리 기준

오류코드 기준에서 가장 중요한 점은 결과코드와 오류코드를 분리하는 것이다.
resultStatus = 성공/실패/Timeout/상태불명확
resultCode   = 응답 결과의 큰 분류
errorCode    = 상세 오류 식별자
messageCode  = 메시지 식별자

| 구분 | 역할 | 예시 |
| --- | --- | --- |
| resultStatus | 처리 상태 | SUCCESS, FAIL, TIMEOUT, UNKNOWN |
| resultCode | 응답 결과 코드 | S0000, E0001, E0002, T0001 |
| errorCode | 상세 오류 식별자 | E-TCF-HDR-0001 |
| messageCode | 메시지 식별자 | MSG-TCF-HDR-0001 |
| userMessage | 사용자 안내 메시지 | 입력값을 확인해 주십시오. |
| operatorMessage | 운영자 상세 메시지 | Header.serviceId is blank |
| actionGuide | 조치 가이드 | 요청 전문 Header를 확인하십시오. |

### 33.7 메시지 분리 기준

오류 메시지는 반드시 사용자 메시지와 운영자 메시지를 분리한다. NSIGHT 오류코드 설계 기준에서도 사용자 메시지는 쉽고 짧게 작성하고 기술정보를 제거하며, 운영자 메시지는 OM·로그·APM에서 원인 추적이 가능하도록 작성하도록 정의되어 있다. 또한 SQL 전문, DB 계정·테이블명 상세, StackTrace, 서버 IP, 개인정보는 사용자 메시지에 노출하지 않아야 한다.
메시지 구분
표시 대상
작성 기준
| 예시 | 사용자 메시지 | 화면 사용자 |
| --- | --- | --- |
| 쉽고 짧게, 기술정보 제거 | 입력값을 확인해 주십시오. | 운영자 메시지 |
| OM, 로그, APM | 원인 추적 가능하게 작성 | Header.serviceId is blank |

### 33.8 표준 응답 전문 기준

오류 응답도 성공 응답과 동일하게 표준 전문 구조를 따른다.
```json
{
  "header": {
    "guid": "7f9c9d10-1111-2222-3333-111122223333",
    "traceId": "TRC202606270001",
    "transactionId": "TX202606270001",
    "serviceId": "SV.Customer.selectSummary",
    "transactionCode": "SV-INQ-0001",
    "businessCode": "SV",
    "channelId": "WEBTOP",
    "userId": "U123456",
    "branchId": "001234",
    "responseDateTime": "2026-06-27T13:45:00"
  },
  "result": {
    "resultStatus": "FAIL",
    "resultCode": "E0001",
    "resultMessage": "처리할 수 없습니다.",
    "errorCode": "E-TCF-CTL-0001",
    "messageCode": "MSG-TCF-CTL-0001",
    "userMessage": "해당 거래를 수행할 수 없습니다.",
    "errorDateTime": "2026-06-27T13:45:00"
  },
  "body": null
}
```

운영 로그와 OM 상세 화면에는 다음 항목을 추가로 보여줄 수 있다.
{
  "operatorMessage": "TCF_TRANSACTION_CONTROL allow-list not found",
  "actionGuide": "OM 거래통제 관리 화면에서 serviceId, 거래코드, 사용자, 채널, 지점 등록 여부를 확인하십시오.",
  "errorSystemId": "NSIGHT-MP",
  "nodeId": "ap01",
  "sqlId": null,
  "targetServiceId": null
}

주의할 점은 사용자 응답과 운영자 상세를 분리하는 것이다.사용자 응답에는 operatorMessage, sqlId, stackTrace, nodeId를 기본적으로 포함하지 않는다.

### 33.9 OM_ERROR_CODE 테이블 기준

오류코드는 OM 기준정보로 관리한다.
CREATE TABLE OM_ERROR_CODE (
    ERROR_CODE          VARCHAR2(50)   NOT NULL,
    ERROR_NAME          VARCHAR2(100)  NOT NULL,
    ERROR_CATEGORY      VARCHAR2(30)   NOT NULL,
    ERROR_LEVEL         VARCHAR2(20)   DEFAULT 'ERROR' NOT NULL,
    RESULT_CODE         VARCHAR2(20)   DEFAULT 'E0001' NOT NULL,
    HTTP_STATUS         NUMBER(3)      DEFAULT 200 NOT NULL,

    USER_MESSAGE        VARCHAR2(500)  NOT NULL,
    OPERATOR_MESSAGE    VARCHAR2(1000),
    ACTION_GUIDE        VARCHAR2(1000),
    NOTIFY_TARGET       VARCHAR2(100),

    RETRY_YN            CHAR(1)        DEFAULT 'N' NOT NULL,
    AUDIT_YN            CHAR(1)        DEFAULT 'N' NOT NULL,
    MASK_DETAIL_YN      CHAR(1)        DEFAULT 'Y' NOT NULL,
    USE_YN              CHAR(1)        DEFAULT 'Y' NOT NULL,

    CREATED_BY          VARCHAR2(50),
    CREATED_AT          DATE           DEFAULT SYSDATE,
    UPDATED_BY          VARCHAR2(50),
    UPDATED_AT          DATE,

    CONSTRAINT PK_OM_ERROR_CODE PRIMARY KEY (ERROR_CODE)
);

CREATE INDEX IX_OM_ERROR_CODE_CATEGORY
ON OM_ERROR_CODE (ERROR_CATEGORY, USE_YN);

NSIGHT 오류코드 설계 기준에서도 OM_ERROR_CODE 테이블은 ERROR_CODE, ERROR_NAME, ERROR_CATEGORY, ERROR_LEVEL, RESULT_CODE, HTTP_STATUS, USER_MESSAGE, OPERATOR_MESSAGE, ACTION_GUIDE, NOTIFY_TARGET, RETRY_YN, AUDIT_YN, MASK_DETAIL_YN, USE_YN 등을 관리하도록 정의되어 있다.

### 33.15 ErrorCodeResolver 기준

ETF는 오류코드를 기준으로 메시지를 해석한다.
```java
@Component
@RequiredArgsConstructor
public class ErrorCodeResolver {
    private final ErrorCodeRepository errorCodeRepository;
    public ResolvedErrorMessage resolve(String errorCode) {
        return errorCodeRepository.findActiveByErrorCode(errorCode)
                .map(this::toResolvedMessage)
                .orElseGet(() -> defaultSystemError(errorCode));
    }
    private ResolvedErrorMessage toResolvedMessage(OmErrorCode row) {
        return ResolvedErrorMessage.builder()
                .errorCode(row.getErrorCode())
                .resultCode(row.getResultCode())
                .httpStatus(row.getHttpStatus())
                .userMessage(row.getUserMessage())
                .operatorMessage(row.getOperatorMessage())
                .actionGuide(row.getActionGuide())
                .retryYn(row.getRetryYn())
                .auditYn(row.getAuditYn())
                .maskDetailYn(row.getMaskDetailYn())
                .build();
    }
    private ResolvedErrorMessage defaultSystemError(String errorCode) {
        return ResolvedErrorMessage.builder()
                .errorCode(errorCode)
                .resultCode("E9999")
                .httpStatus(200)
                .userMessage("시스템 처리 중 오류가 발생했습니다.")
                .operatorMessage("error code is not registered. errorCode=" + errorCode)
                .actionGuide("OM 오류코드 관리 화면에서 오류코드 등록 여부를 확인하십시오.")
                .retryYn("N")
                .auditYn("Y")
                .maskDetailYn("Y")
                .build();
    }
}
```

### 33.16 ETF 오류 응답 조립 기준

ETF는 예외를 받아 표준 응답으로 변환한다.
```java
public StandardResponse<?> error(StandardRequest<?> request, Throwable throwable) {
    String errorCode = extractErrorCode(throwable);
    ResolvedErrorMessage message = errorCodeResolver.resolve(errorCode);
    StandardResult result = StandardResult.builder()
            .resultStatus(resolveResultStatus(throwable))
            .resultCode(message.getResultCode())
            .resultMessage("처리할 수 없습니다.")
            .errorCode(message.getErrorCode())
            .userMessage(message.getUserMessage())
            .errorDateTime(LocalDateTime.now())
            .build();
    transactionLogService.updateFail(
            request.getHeader().getGuid(),
            message.getErrorCode(),
            message.getOperatorMessage()
    );
    if ("Y".equals(message.getAuditYn())) {
        auditLogService.writeErrorAudit(request, message);
    }
    return StandardResponse.builder()
            .header(buildResponseHeader(request.getHeader()))
            .result(result)
            .body(null)
            .build();
}
```

### 33.19 업무별 오류코드 등록 기준

업무별 오류코드는 업무팀이 임의로 추가하지 않고, 다음 절차를 따른다.
| 순서 | 작업 | 설명 |
| --- | --- | --- |
| 1 | 오류 상황 식별 | 업무 오류, 입력값 오류, DB 오류, 연계 오류 구분 |
| 2 | DOMAIN 결정 | 업무코드 또는 공통 영역 선택 |
| 3 | CATEGORY 결정 | VAL, BIZ, DB, IF, TIME 등 |
| 4 | 일련번호 채번 | 동일 DOMAIN/CATEGORY 내 증가 |
| 5 | 사용자 메시지 작성 | 기술정보 제거 |
| 6 | 운영자 메시지 작성 | 원인 추적 가능하게 작성 |
| 7 | 조치 가이드 작성 | 확인 화면·로그·테이블 명시 |
| 8 | 감사 여부 결정 | 권한, 고객정보, 다운로드 관련 여부 |
| 9 | 재시도 여부 결정 | 조회성·연계성 거래 여부 |
| 10 | OM 등록 | OM_ERROR_CODE 등록 |
| 11 | 개발 코드 반영 | BusinessException, SystemException 사용 |
| 12 | 테스트 | 표준 응답, 거래로그, 감사로그 확인 |

### 33.20 메시지 변경관리 기준

오류 메시지는 운영 중 변경될 수 있으므로 변경이력을 관리한다.
| 항목 | 기준 |
| --- | --- |
| 사용자 메시지 변경 | 업무팀 요청 + 운영 승인 |
| 운영자 메시지 변경 | 운영팀 또는 개발팀 승인 |
| 조치 가이드 변경 | 운영 절차 변경 시 수정 |
| 오류코드 삭제 | 원칙 금지, 사용중지 처리 |
| 오류코드 재사용 | 금지 |
| 메시지 이력 | 변경 전/후, 변경자, 변경일시 기록 |
| 긴급 변경 | 장애 대응 후 사후 승인 |
| 다국어 | 필요 시 메시지 테이블 분리 검토 |

권장 이력 테이블은 다음과 같다.
CREATE TABLE OM_ERROR_CODE_HISTORY (
    HISTORY_ID          VARCHAR2(50)   NOT NULL,
    ERROR_CODE          VARCHAR2(50)   NOT NULL,
    CHANGE_TYPE         VARCHAR2(20)   NOT NULL,
    BEFORE_USER_MESSAGE VARCHAR2(500),
    AFTER_USER_MESSAGE  VARCHAR2(500),
    BEFORE_OPERATOR_MESSAGE VARCHAR2(1000),

    CHANGED_AT          DATE DEFAULT SYSDATE,
    CHANGE_REASON       VARCHAR2(1000),

### 33.21 로그 연계 기준

오류 발생 시 다음 항목을 거래로그 또는 오류로그에 남긴다.
| 로그 항목 | 설명 |
| --- | --- |
| guid | 거래 식별자 |
| traceId | End-to-End 추적 ID |
| serviceId | 실행 서비스 |
| transactionCode | 거래코드 |
| businessCode | 업무코드 |
| errorCode | 표준 오류코드 |
| resultCode | 결과코드 |
| resultStatus | FAIL / TIMEOUT / UNKNOWN |
| userMessage | 사용자 메시지 |
| operatorMessage | 운영자 메시지 |
| sqlId | DB 오류 시 SQL ID |
| targetServiceId | 연계 오류 시 대상 ServiceId |
| elapsedTime | 처리시간 |
| nodeId | AP 노드 |
| exceptionType | 예외 클래스 |

예외처리 설계 기준에서도 장애 발생 시 GUID → 거래로그 → ServiceId/TransactionCode → ErrorCode → OM_ERROR_CODE → 운영자 메시지/조치 가이드 → 담당팀 조치 순서로 추적 가능해야 한다고 정리한다.

### 33.25 마무리말

오류코드와 메시지 기준의 핵심은 오류를 사용자에게 보여주는 것이 아니라, 오류를 운영 가능한 정보로 바꾸는 것이다.
NSIGHT에서는 다음 구조를 표준으로 삼는다.
```text
Exception
↓
ErrorCode
↓
OM_ERROR_CODE
↓
UserMessage / OperatorMessage / ActionGuide
↓
StandardResponse
↓
TransactionLog / AuditLog / Metrics
↓
```

OM 오류코드 관리

따라서 업무 개발자는 throw new RuntimeException("오류")처럼 임의 예외를 던지지 않는다.모든 오류는 표준 오류코드로 표현하고, 메시지는 OM에서 관리한다.

### 33.26 시사점

오류코드와 메시지가 표준화되면 운영자는 장애 발생 시 다음 순서로 원인을 추적할 수 있다.
사용자 오류 화면
```text
  ↓
GUID / TraceId
  ↓
거래로그
  ↓
ServiceId / TransactionCode
  ↓

```

```text
ErrorCode
↓
OM_ERROR_CODE
↓
운영자 메시지 / 조치 가이드
↓
```

담당팀 조치

즉, 오류코드와 메시지 기준은 단순 개발 규칙이 아니라 운영 안정성, 장애 추적성, 보안 통제, 감사 대응, PMO 설명 가능성을 확보하는 핵심 개발 표준이다.
