# 58. 보안 테스트 기준

> **NSIGHT TCF 개발 Manual** · 원본: [`znsight-guide`](../znsight-guide/) · 갱신: 2026-07-05

## 58. 보안 테스트 기준

### 58.1 도입 전 안내말

NSIGHT TCF Framework에서 보안 테스트는 단순히 로그인 성공 여부를 확인하는 테스트가 아니다.보안 테스트는 표준 전문 Header, 세션, SSO/JWT, 권한, 거래통제, 입력값 검증, 마스킹, 감사로그, 파일 업다운로드, 환경설정 Secret 관리까지 전체 거래 실행 구조가 안전한지 검증하는 절차이다.
NSIGHT 전처리·후처리 기준에서도 보안 통제는 인증, 권한, 세션, 마스킹, 감사 기준을 적용하는 것으로 정의되어 있으며, 모든 거래는 표준 전문으로 수신되고 GUID·TraceId 기반으로 추적되어야 한다. 또한 전처리 요구사항에는 세션 검증, 인증 검증, 권한 검증, 입력값 검증, 중복요청 방지, 감사 대상 식별이 필수 또는 중요 항목으로 포함되어 있다.
보안 테스트
= 인증 테스트
+ 세션 테스트
+ JWT / SSO 테스트
+ 권한 테스트
+ 거래통제 테스트
+ 입력값 검증 테스트
+ SQL Injection 테스트
+ XSS / 응답 마스킹 테스트
+ 파일 업다운로드 테스트
+ 감사로그 테스트
+ Secret / 환경설정 테스트
+ Gateway / Actuator 접근통제 테스트

### 58.2 보안 테스트 기준 결론

| 구분 | 기준 | 테스트 목적 |
| --- | --- | --- |
| 인증된 사용자가 허용된 거래만 안전하게 실행하는지 검증 | 테스트 범위 | Client → Gateway/Controller → STF → Dispatcher → Handler → 업무계층 → DB → ETF |
| 인증 기준 | Session, SSO Token, JWT Access Token 검증 | 권한 기준 |
| 메뉴권한, 기능권한, 데이터권한, 고객정보 접근권한 검증 | 거래통제 기준 | 등록된 ServiceId와 허용된 Header 조합만 실행 |
| 입력값 기준 | 필수값, 길이, 날짜, 코드값, 금지문자, 범위 검증 | 응답 기준 |
| 개인정보 마스킹, 오류 상세정보 비노출 | 로그 기준 | GUID/TraceId는 남기되 개인정보·Token·Secret은 남기지 않음 |
| 감사 기준 | 고객정보 조회, 다운로드, 권한 위반, 관리자 기능은 감사로그 필수 | 파일 기준 |
| 확장자, 크기, 경로, 권한, 다운로드 감사 검증 | 환경설정 기준 | Password, Secret, Key는 Git 저장 금지 |

완료 기준
인증/권한/입력/마스킹/감사/로그/Secret 테스트 통과
Java 코딩 스타일 가이드에서도 보안 코딩 기준과 테스트·품질 게이트를 별도 장으로 두고 있으며, 코드 인스펙션에서는 예외·로그·보안·SQL 작성 기준의 중대 위반 0건, 통합 테스트에서는 전문 Header, GUID, 오류코드, 권한, 마스킹 End-to-End 시나리오 통과를 기준으로 한다.

### 58.3 보안 테스트 전체 구조

```text
[사용자 / WebTopSuite / tcf-ui / 외부 API]
        ↓
[Gateway / Apache / Controller]
        ↓

```

```text
[STF.preProcess]
 - Header 위변조 검증
 - GUID / TraceId 검증
 - 세션 검증
 - SSO / JWT 인증 검증
 - 사용자 / 지점 / 채널 검증
 - 메뉴권한 / 기능권한 / 데이터권한 검증
 - 거래통제 Allow-List 검증
 - 입력값 / 금지문자 / 길이 검증
 - Idempotency 중복요청 검증
 - 감사 대상 식별
        ↓
[Dispatcher]
 - serviceId 기준 Handler 검증
        ↓

```

```text
[업무 Handler / Facade / Service / Rule / DAO]
 - 업무 권한 재확인
 - SQL Injection 방어
 - 외부연계 인증정보 보호
        ↓
[ETF.postProcess]
 - 표준 오류코드 변환
 - 개인정보 마스킹
 - Token / Secret 응답 제거
 - 감사로그 기록
 - 거래로그 갱신
 - MDC / ThreadLocal 정리
        ↓

```

[StandardResponse]

전처리·후처리 구조에서도 업무 처리 전에는 Header 검증, 세션 검증, 인증·권한 확인, 입력값 검증, 중복요청 확인, 거래상태 PROCESSING 저장을 수행하고, 후처리에서는 결과코드 변환, 오류코드 매핑, 응답 전문 조립, 데이터 마스킹, 감사로그, 성능 메트릭, 거래상태 갱신을 수행하도록 정의되어 있다.

### 58.4 보안 테스트 유형

| 테스트 유형 | 목적 | 주요 검증 대상 |
| --- | --- | --- |
| 인증 테스트 | 로그인 사용자만 거래 가능한지 검증 | Session, SSO, JWT |
| 세션 테스트 | 세션 위변조·만료·탈취 대응 검증 | JSESSIONID, SESSIONDB |
| JWT 테스트 | Token 위변조·만료·권한 검증 | Access Token, Refresh Token |
| 권한 테스트 | 허용된 사용자만 기능 실행 가능 여부 | 메뉴권한, 기능권한, 데이터권한 |
| 거래통제 테스트 | 등록된 거래만 실행되는지 검증 | ServiceId, TransactionCode |
| Header 위변조 테스트 | 전문 Header 변조 차단 | businessCode, userId, branchId |
| 입력값 검증 테스트 | 비정상 입력 차단 | 필수값, 길이, 형식, 금지문자 |
| SQL Injection 테스트 | SQL 조작 방지 | MyBatis #{} 바인딩 |
| XSS 테스트 | 스크립트 응답·저장 차단 | 화면 출력값, 메시지 |
| 파일 보안 테스트 | 악성 파일·경로 조작 방지 | 확장자, 크기, 경로 |
| 마스킹 테스트 | 개인정보 노출 방지 | 고객명, 계좌번호, 전화번호 |
| 오류응답 테스트 | StackTrace·SQL·서버경로 비노출 | StandardResponse.error |
| 로그 보안 테스트 | 로그에 민감정보 미기록 | Token, Password, 개인정보 |
| 감사로그 테스트 | 보안 이벤트 추적 가능 | 고객조회, 다운로드, 권한위반 |
| Secret 테스트 | 중요정보 설정 보호 | DB Password, JWT Key |
| Actuator 테스트 | 운영 Endpoint 접근통제 | /actuator/* |
| Gateway 테스트 | 라우팅 우회·업무코드 변조 차단 | /gw/{businessCode}/online |

### 58.5 인증 테스트 기준

#### 58.5.1 인증 테스트 원칙

| 테스트 항목 | 기준 |
| --- | --- |
| 세션 없음 | 업무 거래 차단 |
| 만료 세션 | 인증 오류 반환 |
| 위조 세션 ID | 인증 오류 반환 |
| 다른 사용자 세션 | Header userId와 세션 userId 불일치 차단 |
| SSO Token 없음 | SSO 로그인 차단 |
| SSO Token 위변조 | 인증 실패 |
| JWT 없음 | JWT 기반 API 호출 차단 |
| JWT 만료 | 인증 오류 |
| JWT 서명 위변조 | 인증 오류 |
| 로그아웃 후 호출 | 인증 오류 |

SSO, Session, JWT, TCF는 책임을 섞지 않아야 한다. SSO는 외부 인증기관의 사용자 확인 결과, Session은 NSIGHT 화면 사용 상태, JWT는 API 호출 인증 수단, TCF는 이 사용자가 해당 거래를 실행해도 되는지 통제하는 실행 프레임워크로 구분된다.

### 58.8 권한 테스트 기준

NSIGHT 권한 테스트는 단순 Role 확인이 아니다.메뉴권한, 기능권한, 데이터권한, 고객정보 접근권한을 분리해서 검증해야 한다.
권한 유형
검증 기준
| 예시 | 메뉴권한 | 사용자가 메뉴에 접근 가능한가 | OM 사용자관리 메뉴 |
| --- | --- | --- | --- |
| 기능권한 | 조회/등록/수정/삭제/다운로드 가능 여부 | btnDelete, download | 데이터권한 |
| 지점/센터/업무범위 데이터 접근 가능 여부 | 타 지점 고객 조회 | ServiceId 권한 | 해당 ServiceId 호출 가능 여부 |
| OM.User.delete | 관리자 권한 | OM 운영 기능 접근 가능 여부 | 거래통제 수정 |

### 58.26 마무리말

NSIGHT 보안 테스트의 핵심은 로그인 성공 여부가 아니라,인증된 사용자가 허용된 거래를 허용된 데이터 범위 안에서만 수행하고, 그 결과가 마스킹·감사·로그 기준에 맞게 남는지 검증하는 것이다.
보안 테스트 완료
= 인증 통과/실패 검증
+ 세션 위변조 차단
+ JWT/SSO 위조 차단
+ 권한 없는 기능 차단
+ 미등록 거래 차단
+ 악성 입력 차단
+ 개인정보 마스킹
+ 오류정보 비노출
+ 감사로그 생성
+ Secret 비노출

### 58.27 시사점

NSIGHT TCF는 모든 거래가 표준 전문과 ServiceId를 기준으로 실행되므로, 보안 테스트도 화면 단위가 아니라 ServiceId 거래 단위로 수행해야 한다.
보안 테스트의 최종 기준은 “이 사용자가 이 ServiceId를 이 지점·이 채널·이 권한으로 실행해도 되는가를 TCF가 설명 가능하게 차단하거나 허용하는가”이다.
