# 35. 거래로그 / 감사로그 기준

> **NSIGHT TCF 개발 Manual** · 원본: [`znsight-guide-word`](../znsight-guide-word/) · 갱신: 2026-07-05

## 35. 거래로그 / 감사로그 기준

### 35.1 도입 전 안내말

거래로그와 감사로그는 단순히 “로그를 남긴다”는 의미가 아니다.NSIGHT TCF Framework에서는 거래가 언제 시작되었고, 어떤 ServiceId가 실행되었으며, 어느 구간에서 지연·오류·권한 위반이 발생했는지 추적하기 위한 운영 증적이다.
거래로그는 장애 추적과 성능 분석을 위한 로그이고, 감사로그는 보안·권한·개인정보 접근 증적을 위한 로그이다.
기존 NSIGHT 거래로그 / 감사로그 구조에서도 모든 온라인 거래는 GUID, TraceId, TransactionId 기반으로 추적되어야 하며, 거래 시작 시 PROCESSING 로그를 생성하고 종료 시 SUCCESS / FAIL / TIMEOUT / UNKNOWN 상태로 갱신하는 구조를 기준으로 한다. 또한 전처리·후처리 기준에서도 모든 온라인 거래는 표준 전문으로 수신되고, 거래로그·성능로그·감사로그·모니터링 지표를 제공해야 한다고 정의되어 있다.
```text
요청 수신
  ↓
STF 전처리
  - GUID / TraceId 생성
  - Header 검증
  - 세션 / 권한 / 거래통제 확인
  - 거래로그 PROCESSING 생성
  ↓
```

업무 Handler 실행
```text
  ↓
Facade → Service → Rule → DAO/Mapper
  ↓
ETF 후처리
  - 결과코드 / 오류코드 매핑
  - 거래로그 SUCCESS / FAIL / TIMEOUT 갱신
  - 감사로그 기록
  - 성능로그 기록
  - 표준응답 조립

```

핵심 문장은 다음이다.
거래로그는 “이 거래가 어떻게 처리되었는가”를 남긴다.감사로그는 “누가 민감한 행위를 했는가”를 남긴다.

### 35.2 거래로그 / 감사로그 기준 결론

| 구분 | 기준 |
| --- | --- |
| 거래로그 기록 주체 | TCF 전처리·후처리 공통 모듈 |
| 감사로그 기록 주체 | TCF 후처리 + OM 감사 정책 |
| 거래 시작 기록 위치 | STF.preProcess() |
| 거래 종료 기록 위치 | ETF.postProcess() |
| 거래 식별자 | GUID, TraceId, TransactionId |
| 실행 식별자 | businessCode, serviceId, transactionCode |
| 사용자 식별자 | userId, branchId, channelId, sessionId |
| 거래 상태 | PROCESSING, SUCCESS, FAIL, TIMEOUT, UNKNOWN, BLOCKED |
| 거래로그 테이블 | TCF_TX_LOG |
| 거래상세 로그 테이블 | TCF_TX_STEP_LOG |
| 성능로그 테이블 | TCF_PERFORMANCE_LOG |
| 감사로그 테이블 | OM_AUDIT_LOG |
| 오류 연계 | errorCode, resultCode, operatorMessage |
| SQL 연계 | sqlId, mapperId, dataSource, dbElapsedMs |
| 연계 연동 | callerServiceId, targetServiceId, targetSystem, integrationElapsedMs |
| 개인정보 기준 | 원문 저장 금지, 마스킹 저장 |
| 운영 조회 | OM 거래로그 / 감사로그 화면 |
| 보관 정책 | 거래로그, 상세로그, 감사로그별 분리 보관 |

### 35.3 거래로그와 감사로그의 차이

| 구분 | 거래로그 | 감사로그 |
| --- | --- | --- |
| 목적 | 거래 처리 상태 추적 | 보안·권한·민감행위 증적 |
| 관심 질문 | 이 거래가 성공했는가, 실패했는가? | 누가 어떤 민감 행위를 했는가? |
| 주요 사용자 | 운영자, 개발자, DBA, PMO | 보안담당자, 감사담당자, 운영관리자 |
| 주요 키 | GUID, TraceId, ServiceId | userId, actionType, targetResource |
| 기록 시점 | 모든 온라인 거래 | 감사 대상 행위 |
| 예시 | 고객요약조회 성공, Timeout, DB 오류 | 고객정보 조회, 권한 변경, 파일 다운로드 |
| 보관기간 | 운영 기준에 따라 3~13개월 | 감사 기준에 따라 장기 보관 |
| 상세정보 | 처리시간, 오류코드, SQL ID | 사용자, 대상, 행위, 변경 전/후 |

### 35.4 거래로그 작성 원칙

| 원칙 | 설명 |
| --- | --- |
| 모든 온라인 거래 기록 | 성공·실패·차단·Timeout 모두 기록 |
| 시작과 종료 분리 | 시작 시 PROCESSING, 종료 시 최종 상태 갱신 |
| GUID 중심 추적 | 사용자 문의 한 건을 GUID로 추적 가능해야 함 |
| ServiceId 중심 분석 | 특정 서비스의 오류율·응답시간을 분석 가능해야 함 |
| 오류코드 연계 | errorCode 기준으로 OM 오류코드와 연결 |
| SQL ID 연계 | DB 오류 또는 지연 시 SQL ID를 남김 |
| 연계 호출 추적 | 내부 서비스 호출 시 caller/target 관계 기록 |
| 대용량 Body 저장 금지 | 요청/응답 전문 전체 저장 금지 |
| 개인정보 마스킹 | 고객번호, 계좌번호, 주민번호 등 원문 저장 금지 |
| 상태 불명확 관리 | Timeout 또는 후처리 실패는 UNKNOWN으로 관리 |

### 35.5 감사로그 작성 원칙

| 원칙 | 설명 |
| --- | --- |
| 민감행위 중심 기록 | 모든 거래가 아니라 감사 대상 행위를 기록 |
| 사용자 행위 증적 | 누가, 언제, 어디서, 무엇을 했는지 기록 |
| 변경 전/후 관리 | 운영 기준정보 변경은 변경 전/후 값을 기록 |
| 개인정보 원문 금지 | 감사로그도 원문 개인정보 저장 금지 |
| 권한 위반 기록 | 권한 없음, 데이터권한 위반은 반드시 감사로그 기록 |
| 다운로드 기록 필수 | 엑셀, 파일, 대량조회 다운로드는 감사로그 필수 |
| 관리자 기능 기록 | 사용자, 권한, 메뉴, ServiceId, 오류코드 변경 기록 |
| 강제조치 기록 | 강제 로그아웃, 토큰 폐기, 배포 승인 기록 |
| 삭제보다 사용중지 | 감사 대상 기준정보는 삭제보다 사용중지 우선 |
| 장기 보관 검토 | 내부 감사·보안 정책에 따라 장기 보관 |

### 35.6 거래 상태 기준

상태
| 의미 | 발생 시점 | PROCESSING | 거래 처리 중 |
| --- | --- | --- | --- |
| STF 전처리 시작 | SUCCESS | 정상 처리 완료 | ETF 성공 후처리 |
| FAIL | 업무 오류 또는 시스템 오류 | BusinessException / SystemException | TIMEOUT |
| 제한시간 초과 | 온라인, DB, 연계 Timeout | UNKNOWN | 처리상태 불명확 |
| 응답 실패, 후처리 실패, Timeout 후 상태 불명 | BLOCKED | 거래통제 또는 권한으로 차단 | STF 거래통제 / 권한 검증 |
| DUPLICATED | 중복요청 | Idempotency 검증 | CANCELLED |
| 사용자 또는 운영자 취소 | 배치·파일·비동기 업무 | RETRY_TARGET | 재처리 대상 |

UNKNOWN 또는 연계 실패 후 상태 확인 필요
기존 거래로그 설계 기준에서도 거래 시작 시점에 PROCESSING 거래로그를 생성하고, 종료 시점에 SUCCESS / FAIL / TIMEOUT / UNKNOWN 상태로 갱신하는 구조를 권장한다.

### 35.9 감사로그 대상

감사로그는 모든 거래가 아니라 감사 대상 행위에 대해 기록한다.
감사 대상
| 예시 | 감사 필요성 | 고객정보 조회 |
| --- | --- | --- |
| 고객요약, 고객상세, 거래내역 조회 | 개인정보 접근 증적 | 고객정보 다운로드 |
| 엑셀 다운로드, 파일 다운로드 | 대량 개인정보 반출 통제 | 권한 위반 |

### 35.11 거래로그 테이블 기준

대표 거래로그 테이블은 TCF_TX_LOG로 둔다.
CREATE TABLE TCF_TX_LOG (
    GUID                 VARCHAR2(64)   NOT NULL,
    TRACE_ID             VARCHAR2(64),
    TRANSACTION_ID       VARCHAR2(64),

    BUSINESS_CODE        VARCHAR2(20)   NOT NULL,
    SERVICE_ID           VARCHAR2(100)  NOT NULL,
    TRANSACTION_CODE     VARCHAR2(50),
    SERVICE_NAME         VARCHAR2(200),

    USER_ID              VARCHAR2(50),
    BRANCH_ID            VARCHAR2(20),
    CHANNEL_ID           VARCHAR2(30),
    SESSION_ID           VARCHAR2(128),
    CLIENT_IP            VARCHAR2(50),
    AP_NODE_ID           VARCHAR2(50),

    TX_STATUS            VARCHAR2(20)   NOT NULL,
    RESULT_CODE          VARCHAR2(20),
    ERROR_CODE           VARCHAR2(50),

    START_TIME           TIMESTAMP      NOT NULL,
    END_TIME             TIMESTAMP,
    ELAPSED_MS           NUMBER,

    DB_ELAPSED_MS        NUMBER,
    INTEGRATION_ELAPSED_MS NUMBER,

    SQL_ID               VARCHAR2(200),
    TARGET_SERVICE_ID    VARCHAR2(100),

    REQUEST_SUMMARY      VARCHAR2(1000),
    RESPONSE_SUMMARY     VARCHAR2(1000),
    OPERATOR_MESSAGE     VARCHAR2(2000),

    CREATED_AT           TIMESTAMP      DEFAULT SYSTIMESTAMP NOT NULL,
    UPDATED_AT           TIMESTAMP,

    CONSTRAINT PK_TCF_TX_LOG PRIMARY KEY (GUID)
);

CREATE INDEX IX_TCF_TX_LOG_01
ON TCF_TX_LOG (START_TIME, SERVICE_ID);

CREATE INDEX IX_TCF_TX_LOG_02
ON TCF_TX_LOG (USER_ID, START_TIME);

CREATE INDEX IX_TCF_TX_LOG_03
ON TCF_TX_LOG (TX_STATUS, START_TIME);

CREATE INDEX IX_TCF_TX_LOG_04
ON TCF_TX_LOG (ERROR_CODE, START_TIME);

### 35.12 거래상세 로그 테이블 기준

단계별 처리시간과 오류 위치를 추적하기 위해 TCF_TX_STEP_LOG를 둔다.
CREATE TABLE TCF_TX_STEP_LOG (
    STEP_LOG_ID          VARCHAR2(64)   NOT NULL,
    GUID                 VARCHAR2(64)   NOT NULL,
    TRACE_ID             VARCHAR2(64),

    STEP_SEQ             NUMBER         NOT NULL,
    STEP_NAME            VARCHAR2(100)  NOT NULL,
    STEP_TYPE            VARCHAR2(30)   NOT NULL,

    START_TIME           TIMESTAMP      NOT NULL,
    END_TIME             TIMESTAMP,
    ELAPSED_MS           NUMBER,

    RESULT_STATUS        VARCHAR2(20),
    ERROR_CODE           VARCHAR2(50),
    DETAIL_MESSAGE       VARCHAR2(2000),

    SQL_ID               VARCHAR2(200),
    MAPPER_ID            VARCHAR2(200),
    TARGET_SERVICE_ID    VARCHAR2(100),
    TARGET_SYSTEM        VARCHAR2(100),

    CREATED_AT           TIMESTAMP      DEFAULT SYSTIMESTAMP NOT NULL,

    CONSTRAINT PK_TCF_TX_STEP_LOG PRIMARY KEY (STEP_LOG_ID)
);

CREATE INDEX IX_TCF_TX_STEP_LOG_01
ON TCF_TX_STEP_LOG (GUID, STEP_SEQ);

단계 유형은 다음과 같이 정의한다.
STEP_TYPE

| 설명 | STF | 전처리 |

| DISPATCHER | ServiceId Handler 매핑 | HANDLER |

| Handler 실행 | FACADE | 트랜잭션 경계 |
| --- | --- | --- |
| SERVICE | 업무 서비스 | RULE |

### 35.13 성능로그 테이블 기준

    PERF_LOG_ID          VARCHAR2(64)   NOT NULL,
    GUID                 VARCHAR2(64)   NOT NULL,
    TRACE_ID             VARCHAR2(64),

    BUSINESS_CODE        VARCHAR2(20),
    SERVICE_ID           VARCHAR2(100),
    TRANSACTION_CODE     VARCHAR2(50),

### 35.28 금지 규칙

| 금지 항목 | 사유 |
| --- | --- |
| 거래로그 미기록 | 장애 추적 불가 |
| 성공 거래만 기록 | 실패·Timeout 분석 불가 |
| 요청 전문 전체 저장 | 개인정보·대용량 위험 |
| 응답 전문 전체 저장 | 개인정보·대용량 위험 |
| 고객정보 원문 저장 | 보안·감사 위험 |
| JWT 원문 저장 | 인증정보 유출 위험 |
| 비밀번호 로그 저장 | 보안 사고 |
| SQL 전문 저장 | 내부 구조 노출 |
| UNKNOWN 상태 방치 | 재처리·상태확인 불가 |
| 권한 위반 감사로그 누락 | 감사 대응 실패 |
| 관리자 변경 이력 누락 | 운영통제 실패 |
| 감사로그 물리 삭제 | 증적 훼손 |
| 수동 로그 삭제 | 감사·장애 분석 훼손 |

### 35.29 마무리말

거래로그와 감사로그 기준의 핵심은 로그를 많이 남기는 것이 아니다.운영자가 거래를 추적하고, 보안 담당자가 행위를 증명하며, 개발자가 장애 원인을 찾을 수 있도록 필요한 정보를 표준 구조로 남기는 것이다.
NSIGHT에서는 다음 구조를 표준으로 삼는다.
```text
GUID / TraceId
  ↓
TCF_TX_LOG
  ↓
TCF_TX_STEP_LOG
  ↓
TCF_PERFORMANCE_LOG
  ↓
OM_AUDIT_LOG
  ↓
OM 거래로그 / 감사로그 조회
  ↓
```

장애 분석 / 감사 대응 / 재처리 판단

### 35.30 시사점

사용자 문의 발생
```text
  ↓

  ↓

  ↓
```

개인정보 접근 여부를 증명할 수 없음

  ↓

```text
  ↓
```

SQL ID / 연계 대상 확인
```text
  ↓
```

```text
  ↓
```

담당팀 조치
