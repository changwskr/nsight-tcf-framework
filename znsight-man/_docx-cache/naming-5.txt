명명규칙 - 모듈 설계기준
1. 도입 전 안내말
모듈 명명규칙은 단순히 디렉터리 이름을 정하는 기준이 아니다.NSIGHT TCF Framework에서는 모듈명 → 업무코드 → Context Path → WAR명 → Package명 → ServiceId → Handler → Mapper → SQL ID → 배포 단위가 하나의 기준으로 연결되어야 한다.
현재 NSIGHT TCF Framework는 하나의 Spring Boot 애플리케이션이 아니라, 공통 프레임워크 모듈, 업무 WAR 모듈, 운영관리 모듈, Gateway, JWT, Batch, UI, CI/CD 설정 모듈이 함께 구성된 멀티 모듈 Gradle 프로젝트로 정의되어 있다.또한 TCF 실행 구조는 POST /{businessCode}/online으로 요청을 받고, 실제 실행 대상은 URL이 아니라 header.serviceId로 결정하는 구조이다.
따라서 모듈 설계기준의 핵심은 다음이다.
공통 기능은 TCF 공통 모듈에 두고, 업무 기능은 업무 WAR에 두며, 운영 기준정보는 OM에서 관리한다.

7. 모듈 의존성 기준
모듈 의존 방향은 아래 기준을 따른다.
tcf-util
   ↓
tcf-core
   ↓
tcf-web
   ↓
tcf-cache 선택
   ↓
업무 서비스 / tcf-om / tcf-batch / tcf-jwt / tcf-gateway

현재 설계 기준에서도 모듈 의존 방향은 tcf-util → tcf-core → tcf-web → tcf-cache(선택) → 업무 서비스 / tcf-om / tcf-batch / tcf-jwt / tcf-gateway로 정리되어 있다.

13. 모듈 설계 품질 체크리스트
점검 항목
확인
모듈명이 tcf-* 또는 {업무코드소문자}-service 규칙을 따르는가
□
업무코드, Context, WAR, Package가 같은 기준으로 연결되는가
□
공통 기능이 업무 WAR에 중복 구현되지 않았는가
□
업무 로직이 tcf-core, tcf-web, tcf-util에 들어가지 않았는가
□
업무 WAR 간 Java 직접 참조가 없는가
□
업무 간 호출은 HTTP/JSON 표준 전문 방식으로 설계되었는가
□
ServiceId Prefix가 업무코드와 일치하는가
□
Handler, Facade, Service, Rule, DAO, Mapper 계층 책임이 분리되었는가
□
Mapper XML 위치가 mapper/{업무코드소문자} 기준을 따르는가
□
운영 기준정보는 tcf-om에서 관리하도록 설계되었는가
□
환경설정은 local/dev/stg/prd로 분리되었는가
□
배포 단위와 Rollback 단위가 명확한가
□

14. 마무리말
모듈 설계기준은 소스 디렉터리 정리 기준이 아니라 운영 가능한 아키텍처 경계를 정하는 기준이다.
NSIGHT TCF Framework에서는 다음 원칙을 반드시 지킨다.
공통은 tcf-* 모듈로,
업무는 {업무코드}-service로,
운영관리는 tcf-om으로,
라우팅은 tcf-gateway로,
인증토큰은 tcf-jwt로,
배치는 tcf-batch로 분리한다.

이 기준을 지키면 신규 업무가 추가되더라도 업무코드 → 모듈 → Context → WAR → Package → ServiceId → Handler → Mapper → 배포까지 일관되게 확장할 수 있다.
