8. Gradle 멀티 모듈 구조
8.1 도입 전 안내말
본 장은 NSIGHT TCF Framework의 Gradle 멀티 모듈 구조를 정의한다.
NSIGHT TCF Framework는 하나의 Spring Boot 프로젝트가 아니다.공통 프레임워크, 업무 서비스, 운영관리, Gateway, JWT, Batch, UI, Cache, CI/CD 설정이 함께 동작하는 구조이다. 따라서 Gradle도 단일 build.gradle 중심이 아니라, 루트 프로젝트 + 하위 모듈 구조로 관리한다.
기존 TCF 설계 기준에서도 NSIGHT TCF Framework는 tcf-util, tcf-core, tcf-web, tcf-cache, tcf-om, tcf-batch, tcf-ui, tcf-gateway, tcf-jwt, 업무 서비스 WAR들로 구성되는 멀티 모듈 구조로 정리되어 있다. 또한 CI/CD 운영 기준도 GitLab, GitLab Runner, Gradle, WAR 배포를 기준으로 빌드 자동화와 배포 자동화를 수행하도록 정의한다.

8.4 모듈 의존성 방향
NSIGHT Gradle 멀티 모듈에서 가장 중요한 원칙은 의존성 방향을 한쪽으로만 흐르게 하는 것이다.
tcf-util
   ↑
tcf-core
   ↑
tcf-web
   ↑
업무 서비스 / tcf-om / tcf-gateway / tcf-batch / tcf-jwt

더 상세히 표현하면 다음과 같다.
[업무 서비스 WAR]
 sv-service / ic-service / pc-service
        │
        ├─ depends on tcf-web
        ├─ depends on tcf-core
        ├─ depends on tcf-cache
        └─ depends on tcf-integration
                    │
                    ▼
              [tcf-core]
                    │
                    ▼
              [tcf-util]

원칙
설명
공통 모듈은 업무 모듈을 참조하지 않는다
tcf-core가 sv-service를 참조하면 안 됨
업무 모듈은 공통 모듈을 참조한다
업무 서비스는 tcf-web, tcf-core 사용
업무 모듈끼리 직접 참조하지 않는다
ic-service가 sv-service Java 클래스를 직접 참조하면 안 됨
업무 간 호출은 HTTP/JSON으로 한다
tcf-integration Client 사용
DB 접근은 업무 모듈 내부에서 처리한다
업무별 DAO/Mapper 분리
공통 모듈 변경은 전체 영향도를 검토한다
tcf-core, tcf-web 변경은 승인 필요

8.11 모듈 의존성 금지 규칙
Gradle 멀티 모듈에서 다음 의존성은 금지한다.
금지 의존성
사유
tcf-core → sv-service
공통 모듈이 업무 모듈에 종속됨
tcf-web → tcf-om
Web 공통이 운영관리 기능에 종속됨
ic-service → sv-service
업무 간 직접 결합 발생
sv-service → tcf-om
업무 서비스가 운영관리 구현에 종속됨
tcf-cache → tcf-om
Cache 공통이 OM 구현에 종속됨
tcf-util → Spring Boot Web
Util 모듈이 무거워짐
업무 서비스 → Gateway
Gateway는 앞단 라우터이며 업무가 참조하면 안 됨
업무 서비스 → JWT 구현체 직접 참조
인증은 Filter 또는 Client 계층에서 처리
허용되는 방향은 다음과 같다.
업무 서비스
   → tcf-web
   → tcf-core
   → tcf-util

업무 서비스
   → tcf-cache

업무 서비스
   → tcf-integration

8.12.2 tcf-core
tcf-core는 TCF Framework의 핵심 모듈이다.
항목
기준
역할
표준 전문, TCF 처리, STF, ETF, Dispatcher, Context, 예외
주요 클래스
StandardRequest, StandardResponse, TransactionContext
사용 주체
tcf-web, 업무 서비스, OM

TCF 설계 기준에서 모든 온라인 거래는 STF → TransactionDispatcher → ETF 흐름으로 처리되며, serviceId에 매핑된 Handler를 실행한다.

8.12.4 tcf-cache
tcf-cache는 Cache 공통 모듈이다.
항목
기준
역할
Spring Cache, EhCache, Cache Manager
Cache 대상
공통코드, ServiceId, 거래통제, Timeout 정책
금지
대량 업무 데이터 Cache
관리 주체
OM Cache 관리 화면
Cache 구조 설계에서도 tcf-cache는 Spring Cache + EhCache 기반 공통 모듈이고, 업무 데이터가 아니라 기준정보와 정책정보를 Cache 대상으로 둔다.

8.12.5 tcf-integration
tcf-integration은 서비스 간 호출 공통 모듈이다.
항목
기준
역할
업무 간 HTTP/JSON 표준 전문 호출
호출 방식
businessCode, serviceId, transactionCode 기준
금지
업무 서비스 Java 클래스 직접 호출
사용 예
IC → SV, OM → JWT
서비스 간 연동은 업무 서비스의 Java 클래스를 직접 참조하지 않고, HTTP/JSON 표준 전문과 serviceId 호출 방식으로 수행하는 것이 기준이다.

   ↓
SV.Customer.selectSummary

이 기준을 지키면 업무 모듈 간 결합을 줄이고, 각 업무 WAR를 독립 배포할 수 있다.

8.15 빌드 명령 기준
전체 프로젝트 빌드는 다음 명령으로 수행한다.
./gradlew clean build

특정 모듈만 빌드할 경우 다음과 같이 수행한다.
./gradlew :sv-service:clean :sv-service:build

업무 WAR만 생성할 경우 다음과 같이 수행한다.
./gradlew :sv-service:bootWar

테스트만 수행할 경우 다음과 같다.
./gradlew test

특정 모듈 테스트는 다음과 같다.
./gradlew :sv-service:test


8.16 로컬 실행 기준
로컬 개발 시에는 bootRun으로 업무 모듈을 실행한다.
./gradlew :sv-service:bootRun --args='--spring.profiles.active=local'

OM 실행 예시는 다음과 같다.
./gradlew :tcf-om:bootRun --args='--spring.profiles.active=local'

Gateway 실행 예시는 다음과 같다.
./gradlew :tcf-gateway:bootRun --args='--spring.profiles.active=local'

운영 배포는 bootRun이 아니라 WAR를 생성하여 Tomcat에 배포한다.
로컬 개발 = bootRun
운영 배포 = bootWar → Tomcat 배포

8.24 마무리말
Gradle 멀티 모듈 구조는 단순히 프로젝트 폴더를 여러 개로 나누는 방식이 아니다.NSIGHT TCF Framework에서는 멀티 모듈 구조가 곧 공통과 업무의 경계, 개발과 운영의 경계, 빌드와 배포의 경계를 결정한다.
따라서 개발자는 다음 기준을 반드시 지켜야 한다.
공통 기능 = tcf-* 모듈
업무 기능 = 업무코드-service 모듈
업무 실행 = ServiceId Handler
공통 처리 = tcf-core / tcf-web
업무 간 호출 = tcf-integration
배포 단위 = 업무별 WAR


8.25 시사점
NSIGHT TCF Framework에서 Gradle 멀티 모듈 구조의 핵심은 “재사용은 하되 결합하지 않는 것”이다.
핵심 원칙
의미
공통은 아래에 둔다
tcf-util, tcf-core, tcf-web은 업무를 몰라야 한다
업무는 독립시킨다
각 업무 서비스는 독립 WAR로 빌드·배포되어야 한다
업무 간 호출은 HTTP/JSON으로 한다
Java 직접 참조가 아니라 표준 전문 호출을 사용한다
빌드는 자동화한다
Gradle Task와 CI/CD Pipeline으로 통제한다
배포 산출물은 명확히 한다
sv.war, om.war, gateway.war처럼 식별 가능해야 한다
결론적으로 Gradle 멀티 모듈 구조는 다음 한 문장으로 정리할 수 있다.
공통 프레임워크는 JAR로 재사용하고, 업무 서비스는 WAR로 독립 배포하며, 모든 업무 실행은 TCF 표준 전문과 ServiceId Dispatcher 기준으로 통제한다.
