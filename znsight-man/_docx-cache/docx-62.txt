64. WAR 생성 기준
64.1 도입 전 안내말
WAR 생성은 개발자가 작성한 업무 소스와 TCF 공통 모듈을 Tomcat에 배포 가능한 하나의 실행 산출물로 패키징하는 절차이다.
NSIGHT TCF Framework는 하나의 Spring Boot 애플리케이션이 아니라 tcf-util, tcf-core, tcf-web, tcf-cache 같은 공통 프레임워크 모듈과 업무 WAR, tcf-om, tcf-gateway, tcf-jwt, tcf-batch, tcf-ui가 함께 구성된 멀티 모듈 Gradle 프로젝트이다. 기존 모듈 설계 기준에서도 업무 서비스는 독립 WAR로 구성하고, 공통 모듈은 업무 WAR 내부에 포함되는 구조로 정리되어 있다.
따라서 WAR 생성 기준의 핵심은 다음이다.
업무별로 독립 실행 가능한 WAR를 생성하되, TCF 공통 JAR는 Tomcat 공용 lib에 넣지 않고 각 WAR의 WEB-INF/lib 안에 포함한다.

64.5 Gradle WAR 생성 명령
전체 WAR 생성은 루트에서 수행한다.
./gradlew clean bootWar

Windows는 다음과 같다.
gradlew.bat clean bootWar

특정 업무 WAR만 생성할 경우에는 모듈명을 지정한다.
./gradlew :sv-service:bootWar

운영관리 WAR 생성:
./gradlew :tcf-om:bootWar

Gateway WAR 생성:
./gradlew :tcf-gateway:bootWar

JWT WAR 생성:
./gradlew :tcf-jwt:bootWar

Batch WAR 생성:
./gradlew :tcf-batch:bootWar

64.11 WAR와 Profile 관계
WAR는 가능하면 환경에 종속되지 않게 생성하고, 환경 차이는 외부 설정으로 분리한다.
동일 WAR
  + application-local.yml → local
  + application-dev.yml   → dev
  + application-prod.yml  → prod

운영에서는 다음 방식으로 외부 설정을 주입한다.
-Dspring.profiles.active=prod
-Dspring.config.additional-location=/app/nsight/config/sv/

환경별 기준은 다음과 같다.
환경
실행 방식
설정 기준
local
Gradle bootRun
모듈 내부 application-local.yml
dev
ztomcat / 개발 Tomcat
tcf-cicd/dev
prod
운영 Tomcat + Apache
tcf-cicd/prod 또는 외부 Mount 설정
기존 환경 설정 정의서도 local은 bootRun, dev는 ztomcat, prod는 운영 Tomcat 기준으로 구분하고, 운영 설정은 외부 yml Mount 방식으로 관리할 수 있다고 정리한다.

64.16 마무리말
WAR 생성은 단순한 압축 파일 생성이 아니라 NSIGHT 업무 서비스를 Tomcat에서 실행 가능한 배포 단위로 확정하는 절차이다.
개발자는 로컬에서 WAR 생성 여부를 확인할 수 있지만, 운영에 반영되는 공식 WAR는 반드시 CI/CD에서 생성되어야 한다. 또한 공통 TCF 모듈은 Tomcat 공용 lib에 배치하지 않고, 각 업무 WAR 내부에 포함하여 업무별 배포 독립성과 롤백 가능성을 보장해야 한다.

64.17 시사점
NSIGHT에서 WAR 생성 기준이 흔들리면 배포 기준, 롤백 기준, 장애 분석 기준도 함께 흔들린다.
따라서 WAR 생성의 핵심 기준은 다음 한 문장으로 정리할 수 있다.
WAR는 업무별 독립 배포 단위이며, 생성은 Gradle bootWar, 공식 산출물은 CI/CD, 실행 검증은 Health Check와 Smoke Test로 완료한다.
