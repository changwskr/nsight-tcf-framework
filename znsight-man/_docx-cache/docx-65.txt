68. 운영 전환 체크리스트
68.1 도입 전 안내말
운영 전환 체크리스트는 NSIGHT TCF Framework를 개발·검증 환경에서 운영 환경으로 넘기기 전에 서비스 가능 상태, 장애 대응 가능 상태, 롤백 가능 상태, 운영 감시 가능 상태를 최종 확인하는 표준 점검표이다.
NSIGHT는 Apache → Tomcat → Spring Boot → MyBatis/HikariCP → RDW/ADW/SESSIONDB 구조를 기준으로 하며, 표준 목표는 사용자 36,000명, 세션 43,200개, TPS 720, P95 3초, 가용성 99.99%로 정의되어 있다. 또한 통합 운영 기준에서는 Busy Thread 70% 이하, Heap 70% 이하, Hikari Pool 70~80% 이하를 운영 목표로 둔다.
운영 전환의 핵심 문장은 다음과 같다.
운영 전환은 개발 완료 선언이 아니라, 운영자가 장애 없이 서비스하고 장애 시 복구할 수 있음을 확인하는 절차이다.

68.15 Health Check 체크리스트
헬스체크는 /actuator/health 단일 확인이 아니라 Liveness, Readiness, Deep, Smoke Check의 4단계 구조로 수행한다.
단계
점검명
확인 내용
확인
1
Liveness
JVM / Spring Boot 프로세스 생존

2
Readiness
업무 요청 수신 가능

3
Deep Check
DB, SESSIONDB, Cache, 외부연계 정상

4
Smoke Check
대표 ServiceId 거래 성공

예시:
curl -f http://localhost:8080/sv/actuator/health/liveness
curl -f http://localhost:8080/sv/actuator/health/readiness
curl -f http://localhost:8080/sv/actuator/health

68.22 마무리말
운영 전환 체크리스트는 형식적인 확인 문서가 아니라 운영 전환 실패를 막는 마지막 안전장치이다.
NSIGHT TCF Framework는 업무 WAR, Gateway, JWT, OM, Batch, Cache, SessionDB, RDW/ADW, Apache, L4가 함께 동작한다. 따라서 하나의 항목이라도 누락되면 로그인 실패, 거래통제 오류, 세션 불일치, DB 연결 오류, Timeout 증가, 배포 후 장애로 이어질 수 있다.

68.23 시사점
NSIGHT의 운영 전환 기준은 다음 한 문장으로 정리할 수 있다.
운영 전환은 “배포 완료”가 아니라, 업무·보안·세션·DB·로그·모니터링·롤백까지 운영자가 책임질 수 있는 상태로 넘기는 절차이다.
