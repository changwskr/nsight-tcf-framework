41. JWT / SSO 연계 기준
41.1 도입 전 안내말
JWT / SSO 연계는 단순히 “로그인 후 토큰을 발급하는 기능”이 아니다.NSIGHT TCF Framework에서는 SSO 인증, OM 사용자 매핑, Spring Session 생성, JWT 발급, Gateway/API 인증, TCF 권한·거래통제 검증이 하나의 흐름으로 연결되어야 한다.
NSIGHT 기준에서 JWT는 TCF를 대체하지 않는다. JWT는 사용자가 누구인지 증명하는 인증 수단이고, TCF는 인증된 사용자가 해당 serviceId 거래를 실행해도 되는지 판단하는 거래 실행 통제 프레임워크이다. 기존 JWT/SSO 설계에서도 SSO 검증은 tcf-om, JWT 발급은 tcf-jwt, 세션 원천은 Spring Session JDBC + SESSIONDB, 업무 실행 기준은 JWT가 아니라 serviceId로 정리되어 있다.
핵심 문장은 다음이다.
SSO는 사용자를 인증하고, JWT는 인증 결과를 전달하며, TCF는 그 사용자가 해당 거래를 실행할 수 있는지 판단한다.

41.19.1 SSO 로그인 Handler
@Component
@RequiredArgsConstructor
public class OmAuthSsoLoginHandler
        implements TransactionHandler<OmSsoLoginRequest, OmSsoLoginResponse> {

    private final OmAuthFacade omAuthFacade;

    @Override
    public OmSsoLoginResponse handle(
            OmSsoLoginRequest request,
            TransactionContext context
    ) {
        return omAuthFacade.ssoLogin(request, context);
    }
}

41.19.2 OM Auth Facade
@Service
@RequiredArgsConstructor
public class OmAuthFacade {

    private final SsoTokenVerifier ssoTokenVerifier;
    private final OmUserService omUserService;
    private final OmSessionService omSessionService;
    private final JwtIssueClient jwtIssueClient;

    @Transactional(timeout = 5)
    public OmSsoLoginResponse ssoLogin(
            OmSsoLoginRequest request,
            TransactionContext context
    ) {
        SsoUserInfo ssoUser =
                ssoTokenVerifier.verify(request.getSsoToken(), request.getNonce());

        OmUser user =
                omUserService.findEnabledUserBySsoSubject(ssoUser.getSubject());

        SessionContext session =
                omSessionService.createSession(user, context);

        JwtTokenPair tokenPair =
                jwtIssueClient.issueSsoTokenPair(user, session, context);

        return OmSsoLoginResponse.builder()
                .loggedIn(true)
                .loginType("SSO")
                .userId(user.getUserId())
                .userName(user.getUserName())
                .branchId(user.getBranchId())
                .channelId(session.getChannelId())
                .accessToken(tokenPair.getAccessToken())
                .expiresIn(tokenPair.getExpiresIn())
                .build();
    }
}

41.19.3 JWT Claim과 Header 비교
@Component
public class JwtHeaderConsistencyValidator {

    public void validate(
            StandardHeader header,
            JwtPrincipal principal,
            SessionContext session
    ) {
        if (!header.getUser().equals(principal.getUserId())) {
            throw new BusinessException(
                    "E-JWT-AUTH-0004",
                    "Token 사용자와 Header 사용자가 일치하지 않습니다."
            );
        }

        if (!header.getBranch().equals(principal.getBranchId())) {
            throw new BusinessException(
                    "E-JWT-AUTH-0004",
                    "Token 지점과 Header 지점이 일치하지 않습니다."
            );
        }

        if (!header.getChannelId().equals(principal.getChannelId())) {
            throw new BusinessException(
                    "E-JWT-AUTH-0004",
                    "Token 채널과 Header 채널이 일치하지 않습니다."
            );
        }

        if (!session.getUserId().equals(principal.getUserId())) {
            throw new BusinessException(
                    "E-SES-0004",
                    "세션 사용자와 Token 사용자가 일치하지 않습니다."
            );
        }
    }
}

41.23 마무리말
JWT / SSO 연계 기준의 핵심은 책임 분리이다.SSO는 외부 인증기관의 사용자 인증 결과이고, Session은 NSIGHT 화면 사용 상태이며, JWT는 API 호출 인증 수단이고, TCF는 인증된 사용자가 실제 거래를 실행할 수 있는지 판단하는 프레임워크이다.
따라서 NSIGHT에서는 로그인 성공만으로 업무를 실행해서는 안 된다.SSO 검증, OM 사용자 매핑, 세션 생성, JWT 발급, Token Claim 검증, Header 비교, 권한검증, 거래통제, 감사로그까지 하나의 인증 수명주기로 묶어야 한다.
41.24 시사점
관점
시사점
개발 관점
업무 개발자는 SSO/JWT 로직을 직접 구현하지 않고 TCF 인증 결과와 SessionContext를 사용한다
운영 관점
OM에서 사용자, 세션, 토큰, 강제 로그아웃, 감사 이력을 통합 관리해야 한다
보안 관점
Token은 짧게, Refresh Token은 Hash 저장, 폐기는 Denylist와 DB 상태로 통제한다
아키텍처 관점
JWT는 인증 수단이고, 최종 업무 실행 판단은 ServiceId + 권한 + 거래통제 기준으로 수행한다
