28. MyBatis Mapper 개발
28.1 도입 전 안내말
MyBatis Mapper는 NSIGHT TCF Framework에서 Java 업무 프로그램과 SQL을 연결하는 표준 데이터 접근 계층이다.
NSIGHT의 업무 처리는 다음 흐름을 따른다.
Handler
  ↓
Facade
  ↓
Service
  ↓
Rule
  ↓
DAO
  ↓
Mapper Interface
  ↓
Mapper XML
  ↓
RDW / ADW / SESSIONDB / LOGDB

MyBatis Mapper는 업무 로직을 처리하는 곳이 아니다.Mapper의 책임은 SQL 실행, 파라미터 바인딩, 결과 매핑이다.
NSIGHT Java 코딩 스타일 기준에서도 기존 금융권 DU/DM 구조를 Spring Boot의 DAO + Mapper 구조로 매핑하고, DB 접근은 DAO/Mapper, MyBatis XML, SQL ID 기준으로 표준화하도록 정리하고 있다. 또한 NSIGHT 운영 기준에서는 MyBatis를 SQL 위치, Query Timeout, RDW/ADW 접근 경계, Mapper 명명, SQL 추적성을 통제하는 표준 메커니즘으로 정의한다.

28.3 Mapper 표준 위치
SV 업무를 기준으로 Mapper 구조는 다음과 같다.
sv-service
└── src/main/java/com/nh/nsight/marketing/sv
    └── persistence
        ├── dao
        │   └── SvCustomerDao.java
        └── mapper
            └── SvCustomerMapper.java

sv-service
└── src/main/resources
    └── mapper
        └── sv
            └── SvCustomerMapper.xml

업무코드, URL, WAR, Package, Mapper, SQL ID, 거래코드, 에러코드는 하나의 기준으로 연결되어야 한다. NSIGHT 업무코드 기준에서도 Mapper Interface, Mapper XML, SQL ID를 표준 정의 대상으로 두고 있으며, 예시로 SvSingleViewCustomerMapper, mapper/sv/SvSingleViewCustomerMapper.xml, SV.Customer.selectSummary를 제시한다.

package com.nh.nsight.marketing.sv.persistence.mapper;

import com.nh.nsight.marketing.sv.dto.SvCustomerQuery;
import com.nh.nsight.marketing.sv.dto.SvCustomerSummaryRow;

import java.util.List;

@Mapper
public interface SvCustomerMapper {

    SvCustomerSummaryRow selectCustomerSummary(SvCustomerQuery query);

MyBatis에서는 두 가지 식별자를 구분해서 관리한다.
구분

    SELECT
          C.CUSTOMER_NO
        , C.CUSTOMER_NAME
        , C.BRANCH_CODE
        , C.TOTAL_BALANCE
    FROM RDW_CUSTOMER_SUMMARY C
    WHERE C.CUSTOMER_NO = #{customerNo}
</select>

이렇게 하면 Java Method, Mapper XML, 업무 SQL ID, 거래로그를 모두 추적할 수 있다.

public String resolveSortColumn(String sortKey) {
    return switch (sortKey) {
        case "customerNo" -> "C.CUSTOMER_NO";
        case "customerName" -> "C.CUSTOMER_NAME";
        default -> "C.CUSTOMER_NO";
    };
}

그리고 Mapper에는 검증된 값만 넘긴다.
ORDER BY ${sortColumn}

이 경우에도 ${} 사용은 코드 리뷰 대상이다.

28.22 마무리말
MyBatis Mapper 개발의 핵심은 SQL을 빠르게 작성하는 것이 아니라, SQL을 운영 가능한 형태로 표준화하는 것이다.
NSIGHT에서는 Mapper 하나가 다음 운영 요소와 연결된다.
Mapper XML
  ↓
SQL ID
  ↓
ServiceId
  ↓
거래로그
  ↓
GUID / TraceId
  ↓
성능 분석
  ↓
장애 추적

따라서 Mapper 개발자는 단순히 “조회 결과가 나오는지”만 확인하면 안 된다.해당 SQL이 어느 DB를 사용하는지, 몇 초 안에 끝나는지, 몇 건까지 조회 가능한지, Full Scan은 없는지, 장애 발생 시 어떤 SQL ID로 추적할 수 있는지까지 확인해야 한다.

28.23 시사점
NSIGHT에서 MyBatis Mapper는 개발 편의 기능이 아니라 DB 부하와 장애 확산을 통제하는 운영 표준 계층이다.
특히 RDW, ADW, SESSIONDB, LOGDB가 분리된 구조에서는 Mapper 표준이 흐트러지면 실시간 조회 DB에 분석 SQL이 들어가거나, 온라인 거래에서 대량 조회가 실행되거나, Hikari Pool 고갈이 발생할 수 있다.
따라서 MyBatis Mapper 개발 기준은 반드시 다음 세 가지로 요약되어야 한다.
첫째, SQL은 Mapper XML에만 둔다.
둘째, Mapper는 업무코드와 DB 목적별로 분리한다.
셋째, 모든 목록 조회는 Paging과 Timeout을 전제로 작성한다.
