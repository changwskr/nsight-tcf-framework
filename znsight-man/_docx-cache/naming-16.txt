로그 / 감사로그 명명규칙 설계기준
1. 도입 전 안내말
로그와 감사로그 명명규칙은 단순히 로그 파일명을 정하는 기준이 아니다.NSIGHT TCF에서는 로그명이 업무코드, ServiceId, 거래코드, GUID, TraceId, 사용자, 화면번호, 오류코드, 감사 이벤트와 연결되어야 한다.
NSIGHT 거래로그는 TCF 전처리·후처리 공통 모듈에서 자동 기록하는 구조가 적합하며, 거래 시작 시 PROCESSING 상태를 만들고, 종료 시 SUCCESS / FAIL / TIMEOUT / UNKNOWN 상태로 갱신하는 흐름을 가진다. 또한 운영 조회는 GUID / TraceId / ServiceId / 거래코드 기준으로 추적해야 한다.
따라서 로그 명명규칙은 다음 흐름을 기준으로 잡는다.
업무코드
  ↓
ServiceId
  ↓
TransactionCode
  ↓
GUID / TraceId
  ↓
LogType
  ↓
Log File / Log Table / Audit Event
  ↓
운영 조회 / 장애 분석 / 감사 대응


2. 로그 명명 최상위 원칙
원칙
설계 기준
업무코드 우선
업무 로그는 반드시 SV, CM, OM, MG 등 업무코드를 포함한다
로그유형 분리
애플리케이션, 거래, 감사, 오류, 성능, SQL, 연계, 파일, 배치 로그를 분리한다
추적ID 포함
로그 내용에는 GUID, TraceId, TransactionId를 반드시 포함한다
ServiceId 중심
온라인 거래 로그는 serviceId와 transactionCode를 포함한다
사용자 추적
감사성 로그는 userId, branchId, channelId, clientIp를 포함한다
개인정보 보호
로그 파일명과 로그 본문에 개인정보 원문을 남기지 않는다
DB 로그와 파일 로그 병행
운영 조회·감사용 DB 로그와 장애 분석용 파일 로그를 병행한다
명칭 일관성
파일명, 테이블명, 컬럼명, Logger명, Appender명을 같은 용어로 맞춘다
운영 검색성
파일명만 보고 업무, 로그유형, 일자를 알 수 있어야 한다
폐기·보관 기준 연계
로그명은 보관주기, 압축, 삭제 정책과 연결되어야 한다

26. 마무리말
로그 / 감사로그 명명규칙의 핵심은 다음과 같다.
로그명은 업무를 식별하고,
거래로그는 상태를 설명하며,
감사로그는 누가 무엇을 했는지 증명해야 한다.

NSIGHT에서는 좋은 로그명이 짧은 로그명이 아니다.운영자가 로그 파일명과 GUID, ServiceId, screenId, errorCode만 보고도 어느 업무의 어떤 화면에서 어떤 사용자가 어떤 거래를 수행했고, 어디서 실패했는지 추적할 수 있어야 한다.
따라서 로그 파일은 nsight-{envCode}-{businessCode}-{logType}.log 형식으로 관리하고, DB 로그 테이블은 TCF_TRANSACTION_LOG, TCF_AUDIT_LOG, TCF_PERFORMANCE_LOG, UD_DOWNLOAD_AUDIT_LOG처럼 목적별로 분리한다. 감사 이벤트는 AUDIT-{DOMAIN}-{ACTION} 형식으로 관리하여 운영, 보안, 감사 대응이 가능한 구조로 고정한다.
