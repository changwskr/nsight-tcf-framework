Java DTO 명명규칙 설계기준
1. 도입 전 안내말
Java DTO 명명규칙은 단순히 Request, Response 이름을 붙이는 기준이 아니다.NSIGHT TCF Framework에서는 DTO가 표준 전문, ServiceId, Handler, Facade, Service, DAO/Mapper, SQL Parameter, 응답 Body를 연결하는 데이터 전달 기준이 된다.
NSIGHT Java 코딩 스타일 가이드에서도 데이터 전달 구조를 기존 IDataSet, IOnlineContext, 전문 Header 개념에서 Request/Response DTO, StandardHeader, RequestContext, GUID/MDC 구조로 전환하도록 정의하고 있다.또한 NSIGHT TCF 개발자 가이드에서는 DTO를 외부 요청, 외부 응답, 내부 처리 요청, 내부 처리 결과로 분리하고 Request, Response, Command, Result, Criteria, Entity, Item 유형을 구분하도록 제시한다.
핵심 원칙은 다음이다.
외부 전문 DTO와 내부 처리 DTO를 분리하고, DTO 이름만 보고 데이터의 사용 위치와 책임을 알 수 있어야 한다.

3. DTO와 표준 전문 관계
NSIGHT TCF 표준 전문은 요청은 StandardRequest, 응답은 StandardResponse로 구성한다. 요청 전문은 header + body, 응답 전문은 header + result + body 구조를 가진다.

   ↓

   ↓

4. DTO Package 배치 기준
앞 절에서 정한 Package Root 기준에 맞춰 DTO는 다음 위치에 둔다.
com.nh.nsight.marketing.{업무코드소문자}.dto
│
├── request
├── response
├── command
├── result
├── query
├── criteria
├── entity
└── item

예를 들어 SV 업무는 다음과 같다.
com.nh.nsight.marketing.sv.dto.request
com.nh.nsight.marketing.sv.dto.response
com.nh.nsight.marketing.sv.dto.command
com.nh.nsight.marketing.sv.dto.result
com.nh.nsight.marketing.sv.dto.query
com.nh.nsight.marketing.sv.dto.entity
com.nh.nsight.marketing.sv.dto.item

NSIGHT TCF 개발자 가이드에서도 DTO Package를 {업무}.dto.{request/response/command/result} 구조로 분리하도록 제시한다.

8. Request / Response / Command / Result 변환 기준
DTO 변환은 다음 흐름으로 고정한다.
Request
   ↓ from(request, header)
Command
   ↓ Service 처리
Result
   ↓ from(result)
Response

권장 Method명은 다음과 같다.
변환
Method
예시
Request → Command
toCommand() 또는 from()
CustomerSummaryCommand.from(request, header)
Result → Response
from()
CustomerSummaryResponse.from(result)
Entity → Result
from()
CustomerSummaryResult.from(entity)
Criteria → Query
toQuery()
CustomerSearchCriteria.toQuery()
예시는 다음과 같다.
public class CustomerSummaryCommand {

    private String customerNo;
    private LocalDate baseDate;
    private String userId;
    private String branchCode;

    public static CustomerSummaryCommand from(
            CustomerSummaryRequest request,
            StandardHeader header
    ) {
        CustomerSummaryCommand command = new CustomerSummaryCommand();
        command.customerNo = request.getCustomerNo();
        command.baseDate = request.getBaseDate();
        command.userId = header.getUserId();
        command.branchCode = header.getBranchCode();
        return command;
    }
}

public class CustomerSummaryResponse {

    private String customerNo;
    private String customerName;
    private String customerGrade;

    public static CustomerSummaryResponse from(CustomerSummaryResult result) {
        CustomerSummaryResponse response = new CustomerSummaryResponse();
        response.customerNo = result.getMaskedCustomerNo();
        response.customerName = result.getCustomerName();
        response.customerGrade = result.getCustomerGrade();
        return response;
    }
}

12. Validation DTO 기준
Validation Annotation은 Request DTO에 우선 적용한다.
public class CustomerSummaryRequest {

    @NotBlank
    private String customerNo;

    @NotNull
    private LocalDate baseDate;
}

17. DTO 정합성 검증 기준
검증 항목
정상 기준
DTO명이 사용 목적을 표현하는가
Request / Response / Command / Result / Query 구분
DTO Package가 유형별로 분리되었는가
dto.request, dto.response, dto.command
Request DTO가 외부 입력값만 포함하는가
Header 중복 없음
Command DTO가 내부 처리 기준을 포함하는가
userId, branchCode 등 서버 보정값 가능
Query DTO가 SQL 조건만 포함하는가
화면 표시 전용값 없음
Entity가 외부 응답으로 직접 반환되지 않는가
Response로 변환
Response DTO가 마스킹 기준을 반영하는가
민감정보 평문 없음
목록 응답이 Item DTO와 Page DTO를 사용하는가
List<Item> + PageResponse
DTO Field가 lowerCamelCase인가
customerNo, branchCode
DTO에 업무 로직이나 DB 접근 코드가 없는가
데이터 전달만 수행
Map<String, Object> 사용이 제한되는가
명시적 DTO 우선
ServiceId 의미와 DTO명이 연결되는가
SV.Customer.selectSummary ↔ CustomerSummaryRequest

18. DTO 설계 체크리스트
점검 항목
확인
Request / Response / Command / Result DTO가 분리되었는가
□
DTO명이 {업무대상}{행위}{DTO유형} 형식인가
□
업무 내부 전용 DTO와 공통 DTO Prefix 기준이 명확한가
□
DTO Package가 dto.request, dto.response, dto.command, dto.result로 분리되었는가
□
Request DTO에 Header 항목이 중복 포함되지 않았는가
□
Response DTO에 오류코드나 StackTrace가 포함되지 않았는가
□
Entity가 외부 Response로 직접 반환되지 않는가
□
Query DTO가 Mapper SQL 조건과 일치하는가
□
목록 응답은 Item + PageResponse 구조인가
□
민감정보는 Response 생성 전에 마스킹되는가
□
DTO Field가 의미 있는 업무 용어를 사용하는가
□
CustomerDTO, Param, Data, Info 같은 모호한 이름이 없는가
□
DTO 변환 Method가 from, toCommand, toQuery 등으로 일관되는가
□

19. 마무리말
Java DTO 명명규칙은 NSIGHT TCF Framework의 데이터 흐름을 명확히 하는 기준이다.
StandardRequest<Request>
   ↓
Command
   ↓
Query / Entity / Result
   ↓
Response
   ↓
StandardResponse<Response>

따라서 NSIGHT에서는 다음 원칙을 고정한다.
외부 전문 DTO는 Request/Response로, 내부 처리 DTO는 Command/Result/Query/Entity로 분리한다.
이 기준을 지키면 개발자는 DTO의 사용 위치를 혼동하지 않고, 운영자는 ServiceId와 거래로그를 기준으로 요청값, 처리값, SQL 조건, 응답값을 일관되게 추적할 수 있다.
