52. 배포관리 연계
52.1 도입 전 안내말
배포관리는 NSIGHT TCF Framework에서 개발된 업무 WAR, OM, Gateway, Batch, JWT, UI 모듈을 운영환경에 안전하게 반영하고, 배포 이력·승인·Health Check·Rollback 상태를 OM에서 추적하는 절차이다.
중요한 원칙은 다음과 같다.
OM은 배포 요청·승인·이력·상태를 관리하고, 실제 빌드와 서버 배포 실행은 CI/CD 파이프라인이 수행한다.
기존 배포관리 설계에서도 OM은 배포 요청, 승인, 이력, 상태 조회, 롤백 지시를 담당하고, CI/CD는 빌드, 패키징, 서버 배포, Health Check, 롤백 실행을 담당하는 구조가 적합하다고 정리되어 있다. 또한 NSIGHT CI/CD 운영 기준은 GitLab, GitLab Runner, Gradle, 17 WAR 배포를 기준으로 개발 표준화, 형상관리 표준화, 빌드 자동화, 배포 자동화, 롤백 자동화, 운영 안정성 확보를 목표로 한다.

52.15 Health Check / Smoke Check 기준
배포 후 단순히 프로세스가 떠 있는지만 보면 안 된다.
기존 헬스체크 설계에서도 NSIGHT는 /actuator/health 한 개로 끝내지 않고, Liveness, Readiness, Deep Check, Smoke Check의 4단계 구조로 설계하는 것이 적절하다고 정리되어 있다.

목적

52.21 마무리말
배포관리 연계의 핵심은 OM과 CI/CD의 역할 분리이다.
OM은 배포를 통제하고, CI/CD는 배포를 실행한다.OM이 승인·상태·이력·감사·Rollback 요청을 관리하고, GitLab Runner가 빌드·테스트·WAR 생성·서버 반영·Health Check를 수행해야 운영 안정성과 감사 대응성을 동시에 확보할 수 있다.
정리하면 다음과 같다.
OM
= 배포 요청 / 승인 / 이력 / 상태 / Rollback 요청 관리

CI/CD
= Build / Test / Package / Deploy / Health Check / Rollback 실행

Tomcat
= 업무 WAR 실행

Apache / L4
= 배포 중 트래픽 우회와 서비스 안정성 지원


52.22 시사점
배포관리는 개발 산출물을 운영에 올리는 단순 작업이 아니라 운영 변경관리 절차이다.
관점
시사점
개발 관점
배포 전 ServiceId, 거래통제, Timeout, 오류코드 등록을 함께 확인해야 한다
운영 관점
배포 상태와 Health Check 결과가 OM에서 보여야 한다
보안 관점
승인되지 않은 배포는 실행되지 않아야 한다
감사 관점
누가, 언제, 어떤 버전을, 어느 서버에 배포했는지 남아야 한다
아키텍처 관점
OM은 통제 지점, CI/CD는 실행 지점으로 분리해야 한다
