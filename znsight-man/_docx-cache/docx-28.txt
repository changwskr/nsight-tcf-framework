23. TransactionHandler 개발 기준
23.1 도입 전 안내말
NSIGHT TCF Framework에서 TransactionHandler는 serviceId로 식별되는 실제 업무 진입 클래스이다.
화면 또는 WebTopSuite가 POST /{businessCode}/online으로 표준 전문을 호출하면, TCF Framework는 Header의 serviceId를 읽고 해당 serviceId에 매핑된 TransactionHandler를 찾아 실행한다.
POST /sv/online
   ↓
StandardRequest.header.serviceId = SV.Customer.selectSummary
   ↓
TransactionDispatcher
   ↓
SvCustomerSummaryHandler
   ↓
Facade → Service → Rule → DAO / Mapper

따라서 TransactionHandler는 Controller가 아니고, Service도 아니고, DAO도 아니다.TransactionHandler는 TCF 공통 처리 영역과 업무 계층 사이를 연결하는 업무 진입 Adapter이다.

23.4 TransactionHandler 기본 인터페이스
TCF 표준 Handler 인터페이스는 다음 형태를 기준으로 한다.
public interface TransactionHandler {

    /**
     * Handler가 처리할 ServiceId를 반환한다.
     */
    String serviceId();

    /**
     * 표준 요청 전문과 거래 Context를 받아 업무 처리를 수행한다.
     */
    Object handle(StandardRequest<?> request, TransactionContext context);
}

메서드
설명
serviceId()
이 Handler가 담당하는 ServiceId 반환
handle()
표준 전문 Body를 업무 DTO로 변환하고 Facade 호출
반환값
업무 응답 DTO 또는 업무 결과 객체
예외
검증 오류, 업무 오류, 시스템 오류는 표준 예외로 전파

23.5 TransactionHandler 기본 작성 예시
SV 고객 요약 조회 Handler 예시는 다음과 같다.
@Component
@RequiredArgsConstructor
public class SvCustomerSummaryHandler implements TransactionHandler {

    private final SvCustomerFacade svCustomerFacade;
    private final TcfRequestValidator tcfRequestValidator;
    private final ObjectMapper objectMapper;

    @Override
    public String serviceId() {
        return "SV.Customer.selectSummary";
    }

    @Override
    public Object handle(
            StandardRequest<?> request,
            TransactionContext context
    ) {
        SvCustomerSummaryRequest body =
                objectMapper.convertValue(
                        request.getBody(),
                        SvCustomerSummaryRequest.class
                );

        tcfRequestValidator.validate(body);

        return svCustomerFacade.selectCustomerSummary(body, context);
    }
}

@Component
@RequiredArgsConstructor
public class BadSvCustomerHandler implements TransactionHandler {

    private final SvCustomerDao svCustomerDao;

    @Override
    public String serviceId() {
        return "SV.Customer.selectSummary";
    }

    @Override
    public Object handle(StandardRequest<?> request, TransactionContext context) {

        Map<String, Object> body = (Map<String, Object>) request.getBody();
        String customerNo = (String) body.get("customerNo");

        // 금지: Handler에서 DB 직접 조회

        Map<String, Object> response = new HashMap<>();
        response.put("resultCode", "SUCCESS");
        response.put("data", result);

        return response;
    }
}

23.15 Handler 등록 기준
Handler는 Spring Bean으로 등록한다.
@Component
@RequiredArgsConstructor
public class SvCustomerSummaryHandler implements TransactionHandler {
    ...
}

Dispatcher는 Spring Bean으로 등록된 TransactionHandler 목록을 수집하여 serviceId 기준 Map을 구성한다.
@Component
public class TransactionHandlerRegistry {

    private final Map<String, TransactionHandler> handlerMap;

    public TransactionHandlerRegistry(List<TransactionHandler> handlers) {
        this.handlerMap = handlers.stream()
                .collect(Collectors.toMap(
                        TransactionHandler::serviceId,
                        Function.identity()
                ));
    }

    public TransactionHandler getHandler(String serviceId) {
        TransactionHandler handler = handlerMap.get(serviceId);

        if (handler == null) {
            throw new BusinessException(
                    "E-TCF-DSP-0004",
                    "ServiceId에 해당하는 Handler가 없습니다."
            );
        }

        return handler;
    }
}

중복 ServiceId가 있으면 애플리케이션 기동 시점에 오류로 처리해야 한다.
동일 ServiceId를 가진 Handler가 2개 이상이면 기동 실패 처리

}

23.28 마무리말
TransactionHandler는 NSIGHT TCF Framework에서 ServiceId와 업무 계층을 연결하는 핵심 진입점이다.
하지만 Handler가 핵심이라고 해서 모든 로직을 Handler에 넣으면 안 된다.Handler가 비대해지면 TCF Framework의 계층 구조가 무너지고, 테스트와 운영 추적이 어려워진다.
따라서 Handler는 다음 원칙을 지켜야 한다.
Handler는 얇게 만든다.
Handler는 DTO 변환과 Validation 호출만 수행한다.
Handler는 Facade를 호출한다.
업무 판단은 Rule에서 한다.
DB 접근은 DAO/Mapper에서 한다.
응답 전문 조립은 ETF에서 한다.
