38. Idempotency / 중복요청 처리
38.1 도입 전 안내말
Idempotency, 즉 멱등성은 같은 요청이 여러 번 들어와도 업무 결과가 한 번만 반영되도록 보장하는 처리 기준이다.
NSIGHT TCF에서는 사용자가 버튼을 여러 번 클릭하거나, 화면이 Timeout 후 재전송하거나, Gateway·서비스 간 연동 Client가 재시도할 때 동일 등록·변경·승인·파일처리 거래가 중복 실행되지 않도록 해야 한다.
기존 TCF 처리 구조에서도 STF.preProcess() 단계에서 Header 검증, GUID/TraceId, 세션, 권한, 거래통제 이후 Idempotency 중복요청 확인을 수행하고, 그 다음 Timeout 정책 조회와 거래 시작 로그 저장으로 이어지는 구조로 정리되어 있다. 또한 거래 상태는 PROCESSING, SUCCESS, FAIL, TIMEOUT, UNKNOWN 등으로 관리해야 하므로, 중복요청 처리도 거래로그 상태와 함께 판단해야 한다.
핵심 문장은 다음이다.
UI 중복클릭 방지만으로는 부족하다.NSIGHT는 서버의 TCF 전처리에서 Idempotency Key 기준으로 중복요청을 차단해야 한다.

        v

        v

TCF 공통 파이프라인은 STF → TransactionDispatcher → ETF 흐름에서 Header 검증, 세션·권한, 거래통제, Timeout, 로깅, 응답 조립을 담당하므로, Idempotency도 업무 Handler가 아니라 STF 전처리에서 우선 처리하는 것이 맞다.

38.8 중복 판단 기준
중복 판단은 단순히 idempotencyKey만 보지 않는다. 최소한 다음 조합으로 판단한다.
businessCode
+ serviceId
+ transactionCode
+ user
+ channelId
+ branch
+ idempotencyKey

    private String idempotencyKey;

38.15.2 STF 적용 예시
@Component
@RequiredArgsConstructor
public class STF {

    private final IdempotencyService idempotencyService;
    private final TimeoutPolicyResolver timeoutPolicyResolver;
    private final TransactionLogWriter transactionLogWriter;

    public TransactionContext preProcess(StandardRequest<?> request) {

        StandardHeader header = request.getHeader();

        // 1. Header, 세션, 권한, 거래통제 검증은 앞 단계에서 수행했다고 가정

        // 2. Idempotency 중복요청 확인
        IdempotencyCheckResult idempotencyResult =
                idempotencyService.checkAndReserve(request);

        if (idempotencyResult.isDuplicated()) {
            throw new BusinessException(
                    "E-TCF-IDEMP-0002",
                    "동일한 요청이 처리 중입니다."
            );
        }

        // 3. Timeout 정책 조회
        TimeoutContext timeoutContext =
                timeoutPolicyResolver.resolve(header.getServiceId());

        // 4. 거래 컨텍스트 생성
        TransactionContext context = TransactionContext.builder()
                .guid(header.getGuid())
                .traceId(header.getTraceId())
                .serviceId(header.getServiceId())
                .transactionCode(header.getTransactionCode())
                .businessCode(header.getBusinessCode())
                .userId(header.getUser())
                .channelId(header.getChannelId())
                .branchCode(header.getBranch())
                .idempotencyKey(header.getIdempotencyKey())
                .timeoutContext(timeoutContext)
                .build();

        // 5. 거래 시작 로그 저장
        transactionLogWriter.insertProcessing(context);

        return context;
    }
}

38.15.3 Idempotency Service 예시
@Service
@RequiredArgsConstructor
public class IdempotencyService {

    private final IdempotencyMapper idempotencyMapper;
    private final RequestHashGenerator requestHashGenerator;

    public IdempotencyCheckResult checkAndReserve(StandardRequest<?> request) {

        StandardHeader header = request.getHeader();

        boolean required = isIdempotencyRequired(header.getTransactionCode());

        if (required && isBlank(header.getIdempotencyKey())) {
            throw new BusinessException(
                    "E-TCF-IDEMP-0001",
                    "중복 방지 키가 없습니다."
            );
        }

        if (!required && isBlank(header.getIdempotencyKey())) {
            return IdempotencyCheckResult.notRequired();
        }

        String requestHash = requestHashGenerator.generate(request.getBody());

        try {
            idempotencyMapper.insertProcessing(
                    header.getBusinessCode(),
                    header.getServiceId(),
                    header.getTransactionCode(),
                    header.getUser(),
                    header.getChannelId(),
                    header.getBranch(),
                    header.getIdempotencyKey(),
                    requestHash,
                    header.getGuid(),
                    header.getTraceId()
            );

            return IdempotencyCheckResult.firstRequest();

        } catch (DuplicateKeyException e) {

            IdempotencyRecord existing =
                    idempotencyMapper.selectByKey(
                            header.getBusinessCode(),
                            header.getServiceId(),
                            header.getTransactionCode(),
                            header.getUser(),
                            header.getChannelId(),
                            header.getBranch(),
                            header.getIdempotencyKey()
                    );

            if (!existing.getRequestHash().equals(requestHash)) {
                throw new BusinessException(
                        "E-TCF-IDEMP-0003",
                        "같은 중복 방지 키로 다른 요청이 들어왔습니다."
                );
            }

            return switch (existing.getTxStatus()) {
                case "PROCESSING" -> IdempotencyCheckResult.duplicatedProcessing(existing);
                case "SUCCESS" -> IdempotencyCheckResult.alreadySuccess(existing);
                case "TIMEOUT", "UNKNOWN" -> IdempotencyCheckResult.unknown(existing);
                case "FAIL" -> IdempotencyCheckResult.previousFail(existing);
                default -> IdempotencyCheckResult.invalidStatus(existing);
            };
        }
    }

    private boolean isIdempotencyRequired(String transactionCode) {
        return transactionCode.contains("-SAV-")
                || transactionCode.contains("-UPD-")
                || transactionCode.contains("-DEL-")
                || transactionCode.contains("-APR-")
                || transactionCode.contains("-EXT-");
    }

    private boolean isBlank(String value) {
        return value == null || value.trim().isEmpty();
    }
}

38.18 마무리말
Idempotency / 중복요청 처리는 단순한 편의 기능이 아니다. NSIGHT TCF에서는 사용자의 중복 클릭, 네트워크 재시도, Timeout 후 재전송, 서비스 간 연동 재호출을 모두 통제하기 위한 거래 안정성 장치이다.
개발자는 업무 프로그램 안에서 중복요청을 개별적으로 막으려 하지 말고, TCF의 STF.preProcess()에서 idempotencyKey 기준으로 공통 처리되도록 해야 한다.
