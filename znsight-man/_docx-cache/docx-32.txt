27. DAO 개발
27.1 도입 전 안내말
DAO는 NSIGHT TCF Framework에서 DB 접근을 담당하는 표준 계층이다.하지만 DAO는 업무 로직을 처리하는 곳이 아니다.
NSIGHT의 표준 흐름은 다음과 같다.
Handler
  ↓
Facade
  ↓
Service
  ↓
Rule
  ↓
DAO
  ↓
Mapper Interface
  ↓
MyBatis XML
  ↓
RDW / ADW / SESSIONDB / LOGDB

DAO의 핵심 역할은 Mapper 호출을 캡슐화하고, DB 접근 기준을 일관되게 유지하는 것이다. 기존 DAO 설계 기준에서도 DAO 위치는 persistence.dao, Mapper 위치는 persistence.mapper, Mapper XML 위치는 src/main/resources/mapper/{업무코드소문자}로 정의되어 있으며, SQL은 Java 코드가 아니라 Mapper XML에 작성하도록 정리되어 있다.
NSIGHT Java 코딩 스타일 기준도 기존 금융권 DU/DM 구조를 Spring Boot의 DAO + Mapper 구조로 매핑하고, DB 접근은 MyBatis XML과 SQL ID 기준으로 표준화하도록 정의한다.

27.7 DAO 구현 예시
27.7.1 Query DTO
package com.nh.nsight.marketing.sv.dto;

public class SvCustomerQuery {

    private String customerNo;
    private int pageNo = 1;
    private int pageSize = 100;
    private int offset = 0;

    public String getCustomerNo() {
        return customerNo;
    }

    public int getPageNo() {
        return pageNo;
    }

    public int getPageSize() {
        return pageSize;
    }

    public int getOffset() {
        return offset;
    }

    public void normalizePaging() {
        if (pageNo < 1) {
            pageNo = 1;
        }

        if (pageSize < 1) {
            pageSize = 100;
        }

        if (pageSize > 500) {
            pageSize = 500;
        }

        offset = (pageNo - 1) * pageSize;
    }
}

27.7.4 DAO Class
package com.nh.nsight.marketing.sv.persistence.dao;

import com.nh.nsight.marketing.sv.dto.SvCustomerQuery;
import com.nh.nsight.marketing.sv.dto.SvCustomerSummaryRow;
import com.nh.nsight.marketing.sv.persistence.mapper.SvCustomerMapper;
import lombok.RequiredArgsConstructor;
import org.springframework.stereotype.Repository;

import java.util.List;
import java.util.Optional;

@Repository
@RequiredArgsConstructor
public class SvCustomerDao {

    private final SvCustomerMapper mapper;

    public Optional<SvCustomerSummaryRow> selectCustomerSummary(SvCustomerQuery query) {
        return Optional.ofNullable(mapper.selectCustomerSummary(query));
    }

    public List<SvCustomerSummaryRow> selectCustomerList(SvCustomerQuery query) {
        query.normalizePaging();
        return mapper.selectCustomerList(query);
    }

    public int countCustomerList(SvCustomerQuery query) {
        return mapper.countCustomerList(query);
    }
}


27.7.5 Mapper XML
<?xml version="1.0" encoding="UTF-8" ?>
<!DOCTYPE mapper
  PUBLIC "-//mybatis.org//DTD Mapper 3.0//EN"
  "https://mybatis.org/dtd/mybatis-3-mapper.dtd">

<mapper namespace="com.nh.nsight.marketing.sv.persistence.mapper.SvCustomerMapper">

    <resultMap id="SvCustomerSummaryRowMap"
               type="com.nh.nsight.marketing.sv.dto.SvCustomerSummaryRow">
        <result property="customerNo" column="CUSTOMER_NO"/>
        <result property="customerName" column="CUSTOMER_NAME"/>
        <result property="branchCode" column="BRANCH_CODE"/>
        <result property="totalBalance" column="TOTAL_BALANCE"/>
    </resultMap>

    <select id="selectCustomerSummary"
            parameterType="com.nh.nsight.marketing.sv.dto.SvCustomerQuery"
            resultMap="SvCustomerSummaryRowMap">
        SELECT
              C.CUSTOMER_NO
            , C.CUSTOMER_NAME
            , C.BRANCH_CODE
            , C.TOTAL_BALANCE
        FROM RDW_CUSTOMER_SUMMARY C
        WHERE C.CUSTOMER_NO = #{customerNo}
    </select>

    <select id="selectCustomerList"
            parameterType="com.nh.nsight.marketing.sv.dto.SvCustomerQuery"
            resultMap="SvCustomerSummaryRowMap">
        SELECT
              C.CUSTOMER_NO
            , C.CUSTOMER_NAME
            , C.BRANCH_CODE
            , C.TOTAL_BALANCE
        FROM RDW_CUSTOMER_SUMMARY C
        WHERE 1 = 1
        <if test="customerNo != null and customerNo != ''">
            AND C.CUSTOMER_NO = #{customerNo}
        </if>
        ORDER BY C.CUSTOMER_NO
        OFFSET #{offset} ROWS FETCH NEXT #{pageSize} ROWS ONLY
    </select>

    <select id="countCustomerList"
            parameterType="com.nh.nsight.marketing.sv.dto.SvCustomerQuery"
            resultType="int">
        SELECT COUNT(1)
        FROM RDW_CUSTOMER_SUMMARY C
        WHERE 1 = 1
        <if test="customerNo != null and customerNo != ''">
            AND C.CUSTOMER_NO = #{customerNo}
        </if>
    </select>

</mapper>

27.9 트랜잭션 기준
DAO에는 원칙적으로 @Transactional을 붙이지 않는다.
Facade = 트랜잭션 경계
Service = 업무 흐름
Rule = 업무 규칙
DAO = Mapper 호출
Mapper XML = SQL 실행

예시는 다음과 같다.
@Service
@RequiredArgsConstructor
public class SvCustomerFacade {

    private final SvCustomerService customerService;

    @Transactional(readOnly = true, timeout = 3)
    public SvCustomerSummaryResponse selectCustomerSummary(SvCustomerSummaryRequest request) {
        return customerService.selectCustomerSummary(request);
    }
}

DAO는 위 트랜잭션 안에서 호출된다. DAO가 직접 트랜잭션을 시작하면 거래 단위가 깨지고, 여러 DAO 호출을 하나의 업무 유스케이스로 묶기 어렵다.

27.13 마무리말
DAO 개발의 핵심은 DB 접근을 단순화하고 표준화하는 것이다.
DAO가 업무 판단, 응답 조립, 트랜잭션 제어, SQL 문자열 생성까지 모두 담당하면 처음에는 편해 보이지만, 시간이 지나면 장애 추적과 유지보수가 어려워진다.
NSIGHT에서는 다음 원칙을 지켜야 한다.
업무 판단은 Rule
업무 흐름은 Service
트랜잭션 경계는 Facade
DB 접근은 DAO
SQL 실행은 Mapper XML
응답 조립은 ETF


27.14 시사점
DAO는 개발자가 가장 자주 작성하는 계층이지만, 운영 관점에서는 가장 위험한 계층이기도 하다. 잘못 작성된 DAO 하나가 RDW 부하, ADW 대량조회, HikariCP 고갈, Tomcat Thread 대기, p95 응답시간 지연으로 이어질 수 있다.
따라서 DAO 개발 표준은 단순 코딩 규칙이 아니라 성능·장애·보안·감사 기준을 코드에 반영하는 운영 표준으로 관리해야 한다.
