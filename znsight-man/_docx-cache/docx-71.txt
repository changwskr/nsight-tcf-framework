77. 오류처리 샘플
77.1 도입 전 안내말
NSIGHT TCF에서 오류처리는 try-catch를 많이 넣는 방식이 아니다.오류처리는 업무 예외, 입력값 오류, 권한 오류, DB 오류, 외부연계 오류, Timeout, 시스템 오류를 표준 오류코드로 변환하고, ETF 후처리에서 표준 응답 전문으로 조립하는 구조로 통일해야 한다.
NSIGHT 전처리·후처리 기준에서도 모든 거래는 표준 전문으로 수신되고, 전처리에서 Header 검증, GUID/Trace 생성, 세션·인증·권한 확인, 거래 시작 로그를 수행한 뒤, 후처리에서 오류코드 매핑, 응답 전문 조립, 거래 종료 로그, 감사로그, 성능 메트릭, 거래상태 저장을 수행하도록 정의되어 있다.

오류코드와 메시지는 TCF 공통 후처리 ETF에서 일원화하고, 운영관리는 OM 오류코드 관리 화면에서 수행하는 구조가 적합하다. 오류 발생 시 Exception을 표준 오류코드로 변환하고, OM_ERROR_CODE에서 사용자 메시지와 운영자 메시지를 조회한 뒤 StandardResponse로 조립하는 방식이다.

77.6 TcfException 기본 클래스
@Getter
public abstract class TcfException extends RuntimeException {

    private final String errorCode;
    private final String userMessage;
    private final String operatorMessage;
    private final boolean retryable;
    private final boolean auditRequired;

    protected TcfException(
            String errorCode,
            String userMessage,
            String operatorMessage,
            boolean retryable,
            boolean auditRequired
    ) {
        super(operatorMessage);
        this.errorCode = errorCode;
        this.userMessage = userMessage;
        this.operatorMessage = operatorMessage;
        this.retryable = retryable;
        this.auditRequired = auditRequired;
    }

    protected TcfException(
            String errorCode,
            String userMessage,
            String operatorMessage,
            Throwable cause,
            boolean retryable,
            boolean auditRequired
    ) {
        super(operatorMessage, cause);
        this.errorCode = errorCode;
        this.userMessage = userMessage;
        this.operatorMessage = operatorMessage;
        this.retryable = retryable;
        this.auditRequired = auditRequired;
    }
}

@Service
@RequiredArgsConstructor
public class SvCustomerService {

    private final SvCustomerDao svCustomerDao;

    public SvCustomerSummaryResponse selectCustomerSummary(String customerNo) {

        SvCustomerInfo customerInfo =
                svCustomerDao.selectCustomerInfo(customerNo);

        if (customerInfo == null) {
            throw new BusinessException(
                    "E-SV-BIZ-0001",
                    "조회 결과가 없습니다.",
                    "SV 고객요약조회 결과 없음. customerNo=" + customerNo
            );
        }

        return SvCustomerSummaryResponse.from(customerInfo);
    }
}

@Component
public class SvCustomerRequestValidator {

    public void validate(SvCustomerSummaryRequest request) {

        List<FieldErrorDetail> errors = new ArrayList<>();

        if (request.getCustomerNo() == null || request.getCustomerNo().isBlank()) {
            errors.add(new FieldErrorDetail(
                    "customerNo",
                    "고객번호는 필수입니다."
            ));
        }

        if (request.getBaseDate() != null
                && !request.getBaseDate().matches("\\d{8}")) {
            errors.add(new FieldErrorDetail(
                    "baseDate",
                    "기준일자는 yyyyMMdd 형식이어야 합니다."
            ));
        }

        if (!errors.isEmpty()) {
            throw new ValidationException(
                    "E-TCF-VAL-0001",
                    "입력값을 확인해 주십시오.",
                    errors
            );
        }
    }
}

입력값 검증은 필수값, 타입, 길이, 코드값, 범위, 금지문자, 페이지, 다운로드 가능 건수까지 포함해야 한다.

@Service
@RequiredArgsConstructor
public class CustomerDataAccessPolicyService {

    private final UserPermissionService userPermissionService;

    public void validateCustomerAccess(String userId, String branchId, String customerBranchId) {

        boolean sameBranch = branchId.equals(customerBranchId);
        boolean hasCrossBranchPermission =
                userPermissionService.hasPermission(userId, "CUSTOMER_CROSS_BRANCH_VIEW");

        if (!sameBranch && !hasCrossBranchPermission) {
            throw new AuthorizationException(
                    "E-TCF-AUTHZ-0001",
                    "접근 권한이 없습니다.",
                    "타 지점 고객 조회 권한 없음. userId=" + userId
                            + ", branchId=" + branchId
                            + ", customerBranchId=" + customerBranchId
            );
        }
    }
}

권한 오류는 감사 대상이다.사용자에게는 내부 권한 구조를 노출하지 않고, 감사로그에는 사용자ID, 요청 메뉴, 요청 기능, 실패사유를 남긴다.

77.12 Timeout 오류 처리 샘플
Timeout은 실패인지 성공인지 즉시 알 수 없는 경우가 있다.따라서 Timeout 거래는 FAIL로 단순 처리하지 말고, 거래 유형에 따라 TIMEOUT 또는 UNKNOWN 상태로 남긴다.
public class TcfTimeoutException extends TcfException {

    public TcfTimeoutException(String operatorMessage, Throwable cause) {
        super(
                "E-TCF-TIME-0001",
                "처리 상태를 확인해 주십시오.",
                operatorMessage,
                cause,
                true,
                true
        );
    }
}

@Component
@RequiredArgsConstructor
public class OnlineTransactionTimeoutExecutor {

    private final ExecutorService executorService;

    public Object execute(Callable<Object> task, Duration timeout) {

        Future<Object> future = executorService.submit(task);

        try {
            return future.get(timeout.toMillis(), TimeUnit.MILLISECONDS);

        } catch (TimeoutException e) {
            future.cancel(true);

            throw new TcfTimeoutException(
                    "온라인 거래 Timeout 발생. timeoutMs=" + timeout.toMillis(),
                    e
            );

        } catch (Exception e) {
            throw new SystemException(
                    "E-TCF-SYS-9999",
                    "시스템 처리 중 오류가 발생했습니다.",
                    "온라인 거래 실행 중 시스템 오류",
                    e
            );
        }
    }
}

Timeout 설계 기준에서도 Timeout은 WAS Thread 보호, DB Connection Pool 보호, 장시간 SQL 차단, 장애 전파 방지, 거래 상태 불명확 방지를 위한 핵심 운영 정책으로 정의되어 있다.

77.13 ETF 오류 후처리 샘플
@Component
@RequiredArgsConstructor
public class EtfErrorProcessor {

    private final ErrorMessageResolver errorMessageResolver;
    private final StandardErrorResponseAssembler responseAssembler;
    private final TransactionLogService transactionLogService;
    private final AuditLogService auditLogService;
    private final MetricsService metricsService;

    public StandardResponse handleError(
            StandardRequest request,
            Exception exception,
            TransactionContext context
    ) {
        TcfException tcfException = convertToTcfException(exception);

        ErrorMessage errorMessage =
                errorMessageResolver.resolve(tcfException.getErrorCode());

        StandardResponse response =
                responseAssembler.assemble(
                        request,
                        context,
                        tcfException,
                        errorMessage
                );

        transactionLogService.updateFail(
                context.getGuid(),
                resolveResultStatus(tcfException),
                errorMessage.getResultCode(),
                tcfException.getErrorCode(),
                tcfException.getOperatorMessage()
        );

        if (tcfException.isAuditRequired() || errorMessage.isAuditRequired()) {
            auditLogService.recordErrorAudit(request, context, tcfException);
        }

        metricsService.incrementErrorCount(
                request.getHeader().getServiceId(),
                tcfException.getErrorCode()
        );

        return response;
    }

    private TcfException convertToTcfException(Exception exception) {

        if (exception instanceof TcfException tcfException) {
            return tcfException;
        }

        return new SystemException(
                "E-TCF-SYS-9999",
                "시스템 처리 중 오류가 발생했습니다.",
                "Uncaught exception: " + exception.getClass().getName(),
                exception
        );
    }

    private String resolveResultStatus(TcfException exception) {

        if (exception instanceof TcfTimeoutException) {
            return "UNKNOWN";
        }

        return "FAIL";
    }
}

후처리 요구사항은 모든 응답을 표준 응답 전문으로 반환하고, 예외 발생 시 표준 오류코드와 사용자 메시지로 변환하며, 사용자 메시지와 운영 상세 메시지를 분리하고, 거래 종료 로그·성능 로그·감사 로그·거래상태 갱신을 수행하도록 요구한다.

77.14 ErrorMessageResolver 샘플
@Component
@RequiredArgsConstructor
public class ErrorMessageResolver {

    private final OmErrorCodeRepository omErrorCodeRepository;

    public ErrorMessage resolve(String errorCode) {

        ErrorMessage message =
                omErrorCodeRepository.selectErrorMessage(errorCode);

        if (message == null) {
            return ErrorMessage.builder()
                    .errorCode("E-TCF-SYS-9999")
                    .resultCode("E9999")
                    .resultStatus("SYSTEM_ERROR")
                    .userMessage("시스템 처리 중 오류가 발생했습니다.")
                    .operatorMessage("OM_ERROR_CODE 미등록 오류코드. errorCode=" + errorCode)
                    .actionGuide("OM 오류코드 관리 화면에 오류코드를 등록하십시오.")
                    .httpStatus(200)
                    .auditRequired(true)
                    .maskDetail(true)
                    .build();
        }

        return message;
    }
}


77.15 표준 오류 응답 조립 샘플
@Component
public class StandardErrorResponseAssembler {

    public StandardResponse assemble(
            StandardRequest request,
            TransactionContext context,
            TcfException exception,
            ErrorMessage message
    ) {
        return StandardResponse.builder()
                .header(StandardResponseHeader.builder()
                        .guid(context.getGuid())
                        .traceId(context.getTraceId())
                        .transactionId(context.getTransactionId())
                        .serviceId(request.getHeader().getServiceId())
                        .transactionCode(request.getHeader().getTransactionCode())
                        .businessCode(request.getHeader().getBusinessCode())
                        .channelId(request.getHeader().getChannelId())
                        .userId(maskUserId(request.getHeader().getUserId()))
                        .branchId(request.getHeader().getBranchId())
                        .responseDateTime(LocalDateTime.now())
                        .build())
                .body(null)
                .error(StandardError.builder()
                        .resultStatus(message.getResultStatus())
                        .resultCode(message.getResultCode())
                        .resultMessage(message.getUserMessage())
                        .errorCode(exception.getErrorCode())
                        .messageCode(message.getMessageCode())
                        .userMessage(message.getUserMessage())
                        .operatorMessage(resolveOperatorMessage(exception, message))
                        .actionGuide(message.getActionGuide())
                        .errorSystemId("NSIGHT-MP")
                        .errorDateTime(LocalDateTime.now())
                        .build())
                .build();
    }

    private String resolveOperatorMessage(TcfException exception, ErrorMessage message) {

        if (message.isMaskDetail()) {
            return "운영 상세 메시지는 로그에서 GUID 기준으로 확인하십시오.";
        }

        return exception.getOperatorMessage();
    }

    private String maskUserId(String userId) {
        if (userId == null || userId.length() < 3) {
            return userId;
        }
        return userId.substring(0, 2) + "***";
    }
}

77.17 TCF.process 오류 처리 샘플
@Component
@RequiredArgsConstructor
public class TcfProcessor {

    private final StandardTransactionFront stf;
    private final TransactionDispatcher dispatcher;
    private final StandardTransactionEnd etf;
    private final OnlineTransactionTimeoutExecutor timeoutExecutor;

    public StandardResponse process(StandardRequest request) {

        TransactionContext context = null;

        try {
            context = stf.preProcess(request);

            Object result = timeoutExecutor.execute(
                    () -> dispatcher.dispatch(request, context),
                    context.getOnlineTimeout()
            );

            return etf.success(request, result, context);

        } catch (BusinessException e) {
            return etf.businessFail(request, e, context);

        } catch (ValidationException e) {
            return etf.validationFail(request, e, context);

        } catch (TcfTimeoutException e) {
            return etf.timeout(request, e, context);

        } catch (Exception e) {
            return etf.systemError(request, e, context);

        } finally {
            TcfMdc.clear();
            TransactionContextHolder.clear();
        }
    }
}

TCF 구조는 STF.preProcess()에서 Header·세션·권한·거래통제·Timeout을 처리하고, TransactionDispatcher가 serviceId → TransactionHandler를 선택하며, ETF가 표준 응답 조립, 오류코드 매핑, 데이터 마스킹, 감사로그, 성능 메트릭, 거래 종료 로그를 처리하는 구조다.

77.18 Global Exception Handler 샘플
TCF 바깥에서 발생한 오류도 표준 응답으로 맞춘다.
@RestControllerAdvice
@RequiredArgsConstructor
public class TcfGlobalExceptionHandler {

    private final EtfErrorProcessor etfErrorProcessor;

    @ExceptionHandler(Exception.class)
    public ResponseEntity<StandardResponse> handleException(
            Exception exception,
            HttpServletRequest servletRequest
    ) {
        StandardRequest fallbackRequest =
                StandardRequestFallbackFactory.from(servletRequest);

        TransactionContext context =
                TransactionContextFallbackFactory.from(servletRequest);

        StandardResponse response =
                etfErrorProcessor.handleError(
                        fallbackRequest,
                        exception,
                        context
                );

        return ResponseEntity.ok(response);
    }
}

운영 기준상 HTTP Status를 항상 200으로 둘지, 시스템 오류만 500으로 둘지는 프로젝트 표준으로 확정해야 한다.금융권 내부 전문 방식에서는 업무 실패도 정상 HTTP 응답으로 보고 resultCode로 판단하는 방식이 일반적으로 운영하기 쉽다.

77.20 거래로그 갱신 샘플
@Service
@RequiredArgsConstructor
public class TransactionLogService {

    private final TransactionLogMapper transactionLogMapper;

    public void updateFail(
            String guid,
            String resultStatus,
            String resultCode,
            String errorCode,
            String errorMessage
    ) {
        transactionLogMapper.updateTransactionFail(
                guid,
                resultStatus,
                resultCode,
                errorCode,
                sanitize(errorMessage),
                LocalDateTime.now()
        );
    }

    private String sanitize(String message) {
        if (message == null) {
            return null;
        }

        return message
                .replaceAll("(?i)password=[^\\s]+", "password=***")
                .replaceAll("(?i)token=[^\\s]+", "token=***");
    }
}

77.25 마무리말
오류처리 샘플의 핵심은 다음이다.
업무 개발자
= 오류 상황에서 표준 예외를 발생시킨다.

TCF / ETF
= 예외를 표준 오류코드로 변환한다.

OM_ERROR_CODE
= 사용자 메시지, 운영자 메시지, 조치 가이드를 관리한다.

StandardResponse
= 화면에 동일한 구조로 오류를 반환한다.

거래로그 / 감사로그
= GUID 기준으로 장애 원인과 처리상태를 추적한다.

따라서 NSIGHT TCF 오류처리는 다음 한 문장으로 정리할 수 있다.
업무 개발자는 예외를 던지고, TCF는 표준 오류 응답을 만든다.
