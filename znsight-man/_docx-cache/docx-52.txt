49. Timeout 정책 등록 절차
49.1 도입 전 안내말
Timeout 정책은 NSIGHT TCF Framework에서 온라인 거래가 어느 시간 안에 끝나야 하는지 정의하는 실행 통제 기준이다.
Timeout은 단순히 “몇 초 기다릴 것인가”를 정하는 설정이 아니다. NSIGHT에서는 Timeout을 WAS Thread 보호, DB Connection Pool 보호, 장시간 SQL 차단, 외부 연계 지연 차단, 장애 전파 방지, 거래 상태 명확화를 위한 핵심 운영 정책으로 본다.
기존 설계 기준에서도 Timeout은 거래통제 테이블에 넣지 않고 별도 TCF_SERVICE_TIMEOUT_POLICY로 분리하는 것이 적절하다고 정리되어 있다. 즉, TCF_TRANSACTION_CONTROL은 “이 거래를 허용할 것인가”를 판단하고, TCF_SERVICE_TIMEOUT_POLICY는 “이 거래를 몇 초 안에 끝내야 하는가”를 판단한다. 또한 Spring Boot 설정 기준에서도 온라인 거래 Timeout, DB Query Timeout, 외부 연계 Timeout을 구분해서 관리하도록 정의되어 있다.
핵심 문장은 다음과 같다.
거래통제는 실행 허용 여부를 판단하고, Timeout 정책은 실행 제한시간을 판단한다.

49.17 운영 변경 원칙
Timeout 정책은 운영 중 장애 대응을 위해 변경될 수 있지만, 무분별하게 늘리면 안 된다.

Timeout을 늘리는 것은 장애 해결이 아니라 장애를 더 오래 기다리게 만드는 것일 수 있다. 따라서 Timeout 증가는 반드시 SQL, DB Pool, 외부연계, Thread 사용률과 함께 검토해야 한다.

49.20 마무리말
Timeout 정책 등록 절차는 NSIGHT TCF Framework의 운영 안정성을 확보하기 위한 필수 절차이다.
ServiceId가 “무엇을 실행할 것인가”를 정의하고, 거래통제가 “실행해도 되는가”를 정의한다면, Timeout 정책은 그 거래가 얼마 안에 끝나야 하는가를 정의한다.
정리하면 다음과 같다.
ServiceId 등록
= 실행할 프로그램 정의

거래통제 등록
= 실행 허용 조건 정의

Timeout 정책 등록
= 실행 제한시간 정의

거래로그 / 감사로그
= 실행 결과와 변경 이력 추적


49.21 시사점
Timeout 정책은 성능 설정이 아니라 장애 확산 방지 정책이다.
관점
시사점
개발 관점
신규 ServiceId 개발 시 Timeout 정책을 함께 정의해야 한다
운영 관점
장애 거래의 Timeout을 ServiceId 단위로 통제할 수 있어야 한다
성능 관점
Query Timeout, DB Pool, Thread 사용률과 함께 관리해야 한다
보안/감사 관점
Timeout 정책 변경 이력이 남아야 한다
아키텍처 관점
Timeout 정책은 거래통제와 분리하고 TCF 전처리에서 적용해야 한다
