MyBatis Mapper / XML / SQL ID 명명규칙 설계기준
1. 도입 전 안내말
MyBatis 명명규칙은 단순히 Mapper 파일명을 정하는 기준이 아니다.NSIGHT TCF Framework에서는 Mapper 이름이 업무코드, Package, DAO, Mapper XML, SQL ID, ServiceId, 거래로그, 성능분석과 연결되어야 한다.
NSIGHT 명명규칙은 업무코드 → Context Path → WAR → Gradle Module → Java Package → ServiceId → Handler → Facade / Service / Rule / DAO / Mapper → Mapper XML → SQL ID → 거래로그 / 오류코드 / 감사로그가 하나로 이어지는 구조를 기준으로 한다.
따라서 MyBatis 명명규칙의 핵심은 다음이다.
Mapper는 테이블 단위가 아니라 업무코드 + 업무기능 단위로 관리하고, Mapper XML과 SQL ID는 운영 추적이 가능해야 한다.

13. Mapper XML 작성 예시
13.1 Mapper Interface
package com.nh.nsight.marketing.sv.persistence.mapper;

import com.nh.nsight.marketing.sv.dto.query.CustomerSummaryQuery;
import com.nh.nsight.marketing.sv.dto.result.CustomerSummaryResult;
import org.apache.ibatis.annotations.Mapper;

@Mapper
public interface SvCustomerMapper {

    CustomerSummaryResult selectCustomerSummary(CustomerSummaryQuery query);

    int countCustomerProductList(CustomerSummaryQuery query);
}

13.2 Mapper XML
<?xml version="1.0" encoding="UTF-8" ?>
<!DOCTYPE mapper
  PUBLIC "-//mybatis.org//DTD Mapper 3.0//EN"
  "http://mybatis.org/dtd/mybatis-3-mapper.dtd">

<mapper namespace="com.nh.nsight.marketing.sv.persistence.mapper.SvCustomerMapper">

    <resultMap id="customerSummaryResultMap"
               type="com.nh.nsight.marketing.sv.dto.result.CustomerSummaryResult">
        <result property="customerNo" column="CUSTOMER_NO"/>
        <result property="customerName" column="CUSTOMER_NAME"/>
        <result property="branchCode" column="BRANCH_CODE"/>
        <result property="customerGrade" column="CUSTOMER_GRADE"/>
    </resultMap>

    <sql id="baseCustomerColumnList">
          C.CUSTOMER_NO
        , C.CUSTOMER_NAME
        , C.BRANCH_CODE
        , C.CUSTOMER_GRADE
    </sql>

    <select id="selectCustomerSummary"
            parameterType="com.nh.nsight.marketing.sv.dto.query.CustomerSummaryQuery"
            resultMap="customerSummaryResultMap">
        SELECT
            <include refid="baseCustomerColumnList"/>
        FROM RDW_CUSTOMER C
        WHERE C.CUSTOMER_NO = #{customerNo}
          AND C.USE_YN = 'Y'
    </select>

    <select id="countCustomerProductList"
            parameterType="com.nh.nsight.marketing.sv.dto.query.CustomerSummaryQuery"
            resultType="int">
        SELECT COUNT(1)
        FROM RDW_CUSTOMER_PRODUCT P
        WHERE P.CUSTOMER_NO = #{customerNo}
    </select>

</mapper>

mybatis:
  mapper-locations:
    - classpath:/mapper/**/*.xml
  type-aliases-package: com.nh.nsight.marketing
  configuration:
    map-underscore-to-camel-case: true

23. 설계 체크리스트
점검 항목
확인
Mapper Interface명이 {업무Prefix}{업무대상}Mapper 형식인가
□
Mapper Package가 persistence.mapper 하위인가
□
Mapper XML이 resources/mapper/{업무코드} 하위에 있는가
□
XML 파일명이 Mapper Interface명과 동일한가
□
XML namespace가 Mapper Interface FQCN과 동일한가
□
SQL ID가 Mapper Method명과 동일한가
□
SQL ID가 select, count, exists, insert, update, delete, merge 등 표준 동사로 시작하는가
□
목록 조회 SQL에 Paging이 적용되어 있는가
□
Count SQL이 목록 SQL과 쌍으로 관리되는가
□
SELECT *를 사용하지 않는가
□
${} 사용 시 Whitelist 검증이 있는가
□
RDW / ADW / SESSIONDB / LOGDB Mapper가 필요 시 분리되어 있는가
□
Mapper가 업무 판단, 권한 판단, 메시지 조립을 하지 않는가
□
Query DTO와 Result / Entity DTO가 명확히 분리되어 있는가
□
ServiceId, Mapper, SQL ID, 거래로그가 추적 가능하게 연결되어 있는가
□

24. 마무리말
MyBatis Mapper / XML / SQL ID 명명규칙은 DB 접근을 표준화하고 운영 추적성을 확보하기 위한 기준이다.
업무코드
   ↓
Mapper Interface
   ↓
Mapper XML
   ↓
SQL ID
   ↓
Statement ID
   ↓
거래로그 / 성능로그 / 장애분석

NSIGHT에서는 다음 기준을 고정한다.
Mapper Interface는 {업무Prefix}{업무대상}Mapper, Mapper XML은 mapper/{업무코드}/{Mapper명}.xml, SQL ID는 Mapper Method명과 동일하게 작성한다.
이 기준을 지키면 개발자는 SQL 위치를 쉽게 찾을 수 있고, 운영자는 ServiceId와 거래로그만으로 Mapper XML과 SQL ID까지 일관되게 추적할 수 있다.
