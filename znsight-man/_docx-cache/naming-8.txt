거래코드(Transaction Code) 명명규칙 설계기준
1. 도입 전 안내말
거래코드(Transaction Code)는 NSIGHT TCF Framework에서 온라인 거래를 로그·감사·거래통제·재처리 관점으로 식별하는 코드이다.
ServiceId가 “어떤 Handler를 실행할 것인가”를 나타낸다면, 거래코드는 “이 요청이 어떤 업무 거래인가”를 나타낸다.
businessCode      = 어느 업무인가
serviceId         = 어느 Handler를 실행할 것인가
transactionCode   = 어떤 거래로 기록·통제·감사할 것인가

NSIGHT 표준 전문 Header에서도 serviceId, transactionCode, businessCode는 핵심 업무 Header 항목으로 사용되며, 거래통제 기준에서도 transactionCode는 필수 판단 항목으로 정의된다.

2. 거래코드 설계 결론
거래코드는 다음 형식을 표준으로 한다.
{업무코드}-{거래유형}-{일련번호}

예시는 다음과 같다.
SV-INQ-0001
CM-REG-0001
PD-UPD-0001
MG-SND-0001
OM-ADM-0001

구성요소
표기 기준
설명
예시
업무코드
대문자 2자리
17개 업무코드 기준
SV, CM, OM
거래유형
대문자 2~4자리
조회, 등록, 수정, 삭제, 실행 등 처리유형
INQ, REG, UPD
일련번호
숫자 4자리
업무코드 + 거래유형 내 순번
0001, 0002
기존 NSIGHT 개발자 가이드에서도 거래코드는 {BUSINESS_CODE}-{TYPE}-{NNNN} 형식으로 정의하고, SV-INQ-0001, CM-REG-0001, OM-UPD-0001 등을 예시로 제시한다.

7. 거래코드와 Header 정합성 기준
표준 전문 Header에서는 transactionCode가 필수 항목이다.
{
  "header": {
    "serviceId": "SV.Customer.selectSummary",
    "transactionCode": "SV-INQ-0001",
    "businessCode": "SV",
    "serviceName": "고객요약조회",
    "user": "U123456",
    "channelId": "WEBTOP",
    "branch": "001234"
  },
  "body": {
    "customerId": "CUST0000001"
  }
}

Header 항목
정합성 기준
businessCode
거래코드 Prefix와 일치해야 함
serviceId
Service Catalog에 등록된 거래코드와 매핑되어야 함
transactionCode
{업무코드}-{거래유형}-{일련번호} 형식이어야 함
serviceName
거래코드의 업무 의미와 일치해야 함
user, channelId, branch
거래통제 Allow-List 검증 대상
TCF 설계서에서도 표준 전문 검증 단계에서 transactionCode의 형식과 등록 여부를 검증하고, 오류코드는 E-TCF-TRX-0001 유형으로 관리하는 기준을 제시한다.

8. 거래코드와 거래통제 관계
거래코드는 거래통제 테이블의 핵심 Key 중 하나이다.
거래통제는 다음 Header 조합을 기준으로 허용 여부를 판단한다.
serviceId
+ transactionCode
+ businessCode
+ serviceName
+ user
+ channelId
+ branch

거래통제 설계안에서도 TCF_TRANSACTION_CONTROL 테이블의 기본 Key에 SERVICE_ID, TRANSACTION_CODE, BUSINESS_CODE, SERVICE_NAME, USER_ID, CHANNEL_ID, BRANCH_ID를 포함하도록 정의하고 있다.
거래통제 항목
예시

따라서 거래코드는 단순 로그용 코드가 아니라, 거래 허용 여부를 판단하는 통제 Key이다.

10. 거래코드와 ServiceId 매핑 기준
거래코드와 ServiceId의 관계는 다음 세 가지 방식으로 정의할 수 있다.
매핑 방식
설명
예시
권장
1:1
하나의 ServiceId에 하나의 거래코드
SV.Customer.selectSummary ↔ SV-INQ-0001
기본 권장
1:N
하나의 ServiceId가 채널/처리유형별 여러 거래코드 사용
MG.Message.send ↔ MG-SND-0001, MG-SND-0002
제한 허용
N:1
여러 ServiceId가 하나의 거래코드 공유
여러 조회가 SV-INQ-0001 공유
비권장
권장 기준은 다음이다.
운영 추적성을 위해 ServiceId와 거래코드는 원칙적으로 1:1로 매핑한다.다만 동일 Handler가 채널별·업무구분별로 별도 감사 기준을 가져야 할 경우 1:N을 허용한다.

11. 거래코드 관리 테이블 설계
거래코드는 OM 기준정보로 관리하는 것이 좋다.
11.1 OM_TRANSACTION_CODE 예시
CREATE TABLE OM_TRANSACTION_CODE (
    TRANSACTION_CODE     VARCHAR2(30)  NOT NULL,
    BUSINESS_CODE        VARCHAR2(10)  NOT NULL,
    SERVICE_ID           VARCHAR2(100) NOT NULL,
    TRANSACTION_NAME     VARCHAR2(200) NOT NULL,
    TRANSACTION_TYPE     VARCHAR2(20)  NOT NULL,
    DESCRIPTION          VARCHAR2(500),

    AUDIT_YN             CHAR(1) DEFAULT 'N' NOT NULL,
    MASKING_YN           CHAR(1) DEFAULT 'N' NOT NULL,
    REPROCESS_YN         CHAR(1) DEFAULT 'N' NOT NULL,
    DOWNLOAD_YN          CHAR(1) DEFAULT 'N' NOT NULL,

    USE_YN               CHAR(1) DEFAULT 'Y' NOT NULL,
    STATUS               VARCHAR2(20) DEFAULT 'DRAFT' NOT NULL,

    CREATED_BY           VARCHAR2(50),
    CREATED_AT           DATE DEFAULT SYSDATE,
    UPDATED_BY           VARCHAR2(50),
    UPDATED_AT           DATE,

    CONSTRAINT PK_OM_TRANSACTION_CODE
        PRIMARY KEY (TRANSACTION_CODE)
);

18. 마무리말
거래코드는 NSIGHT TCF Framework에서 업무 거래를 운영 관점으로 식별하는 표준 코드이다.
ServiceId는 실행 기준이고,
Transaction Code는 거래 추적 기준이다.

따라서 NSIGHT에서는 다음 원칙을 고정한다.
거래코드는 {업무코드}-{거래유형}-{4자리 일련번호}로 정의하고, ServiceId·거래통제·거래로그·감사로그와 반드시 연결한다.
이 기준을 지키면 운영자는 SV-INQ-0001만 보고도 SV 업무의 조회성 거래임을 알 수 있고, ServiceId, Handler, SQL ID, 거래로그, 오류코드까지 일관되게 추적할 수 있다.
