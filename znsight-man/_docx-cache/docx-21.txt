16. ServiceId 설계 기준
16.1 도입 전 안내말
NSIGHT TCF Framework에서 ServiceId는 단순한 프로그램 이름이 아니다.ServiceId는 온라인 거래를 실행하고, 통제하고, 추적하고, 로그와 감사 기준으로 연결하는 핵심 실행 식별자이다.
NSIGHT는 일반 REST 방식처럼 URL마다 Controller Method를 직접 매핑하지 않는다. 기본 진입점은 POST /{businessCode}/online이고, 실제 실행할 업무 프로그램은 JSON Header의 serviceId로 결정한다. 기존 TCF 설계 기준에서도 온라인 진입점은 POST /{businessCode}/online, 실행 식별자는 URL이 아니라 header.serviceId, 공통 처리는 STF → TransactionDispatcher → ETF, 업무 처리는 Handler → Facade → Service → Rule → DAO/Mapper 구조로 정의되어 있다.
따라서 개발자는 화면이나 API를 개발할 때 먼저 다음 질문에 답해야 한다.
이 거래의 ServiceId는 무엇인가?이 ServiceId는 어느 업무코드, 거래코드, Handler, Timeout, 권한, 거래로그와 연결되는가?

16.5 ServiceId와 Handler 매핑 기준
ServiceId는 반드시 Handler와 연결되어야 한다.
기본 원칙은 다음이다.
ServiceId 1개 = Handler 1개

예시는 다음과 같다.
ServiceId
Handler Class
Bean Name
SV.Customer.selectSummary
SvCustomerSummaryHandler
svCustomerSummaryHandler
SV.Customer.selectDetail
SvCustomerDetailHandler
svCustomerDetailHandler
OM.User.selectList
OmUserListHandler
omUserListHandler
OM.Menu.save
OmMenuSaveHandler
omMenuSaveHandler
TCF 설계 기준에서도 운영 표준은 ServiceId 1개당 Handler 1개를 기본으로 하고, 도메인 응집도가 높은 경우에만 다중 ServiceId Handler를 제한적으로 허용한다.
권장 구조는 다음과 같다.
@Component
@RequiredArgsConstructor
public class SvCustomerSummaryHandler implements TransactionHandler {

    private final SvCustomerFacade svCustomerFacade;

    @Override
    public String serviceId() {
        return "SV.Customer.selectSummary";
    }

    @Override
    public Object handle(StandardRequest<?> request, TransactionContext context) {

        SvCustomerSummaryRequest body =
            convertBody(request.getBody(), SvCustomerSummaryRequest.class);

        return svCustomerFacade.selectCustomerSummary(body, context);
    }
}

Handler는 업무 로직을 직접 처리하지 않는다.Handler는 요청 Body를 업무 DTO로 변환하고 Facade를 호출하는 Adapter 역할만 수행한다.

16.18 마무리말
ServiceId는 NSIGHT TCF Framework의 중심 식별자이다.업무 개발자는 Controller URL을 늘리는 방식이 아니라, 표준 전문 Header의 serviceId를 기준으로 Handler를 개발해야 한다.
ServiceId가 표준화되면 다음이 가능해진다.
효과
설명
실행 통제
등록된 서비스만 실행 가능
권한 통제
사용자·권한그룹별 ServiceId 실행 권한 관리 가능
거래 추적
GUID, ServiceId, 거래코드 기준 장애 추적 가능
운영 관리
OM에서 서비스 사용 여부, Timeout, 감사 여부 관리 가능
배포 안정성
Handler와 Catalog 정합성 점검 가능
감사 대응

16.19 시사점
NSIGHT TCF 개발에서 가장 위험한 것은 ServiceId를 개발자별로 임의 생성하는 것이다.ServiceId가 난립하면 권한, 거래통제, Timeout, 거래로그, 오류코드, 운영관리 기준이 모두 분리되어 운영이 불가능해진다.
따라서 NSIGHT 개발자는 다음 원칙을 지켜야 한다.
ServiceId를 먼저 설계하고, 그 다음 Handler와 업무 로직을 만든다.ServiceId는 프로그램명이 아니라 운영 가능한 거래 식별자이다.
