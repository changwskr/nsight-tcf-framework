56. TCF 거래 테스트 기준
56.1 도입 전 안내말
TCF 거래 테스트는 단순히 POST /sv/online이 성공하는지 확인하는 테스트가 아니다.NSIGHT TCF 거래 테스트는 표준 전문 한 건이 TCF 전처리, Dispatcher, Handler, 업무 계층, DB, 후처리, 거래로그까지 정상적으로 흐르는지 확인하는 거래 단위 검증 절차이다.
NSIGHT TCF 구조는 일반 REST Controller 중심 구조가 아니라, 모든 온라인 거래가 TCF.process() 단일 진입점을 통과하고 STF 전처리 → TransactionDispatcher → Business Handler → ETF 후처리 흐름으로 실행되는 구조다.
TCF 거래 테스트
= StandardRequest 한 건을 넣고
+ Header 검증
+ 세션/권한/거래통제 검증
+ ServiceId Dispatcher 검증
+ Handler 실행 검증
+ 업무 처리 결과 검증
+ StandardResponse 검증
+ 거래로그/감사로그 검증

56.4 TCF 거래 테스트 전체 흐름
[테스트 도구]
 tcf-ui / Postman / curl / RestAssured
        ↓
POST /sv/online
StandardRequest JSON
        ↓
[OnlineTransactionController / TcfGateway]
        ↓
TCF.process()
        ↓
[STF.preProcess]
 - Header 필수값 검증
 - businessCode / serviceId / transactionCode 검증
 - GUID / TraceId 생성 또는 검증
 - 세션 검증
 - 인증 / 권한 검증
 - 거래통제 확인
 - Timeout 정책 조회
 - Idempotency 중복요청 확인
 - 거래로그 PROCESSING 생성
        ↓
[TransactionDispatcher]
 - serviceId 기준 Handler 선택
        ↓
[Business Handler]
 - Handler → Facade → Service → Rule → DAO/Mapper
        ↓
[DB / 외부연계]
        ↓
[ETF.postProcess]
 - StandardResponse 조립
 - 오류코드 매핑
 - 데이터 마스킹
 - 감사로그 기록
 - 성능 메트릭 기록
 - 거래로그 SUCCESS / FAIL / TIMEOUT 갱신
        ↓
[StandardResponse JSON]

TCF 설계 기준에서도 STF는 Header 검증, GUID/TraceId 생성, 세션·인증·권한 검증, 거래통제, 중복요청, Timeout 정책, 거래 시작 로그와 PROCESSING 상태 저장을 수행하고, ETF는 표준 응답 조립, 오류코드 매핑, 마스킹, 감사로그, 성능 메트릭, 거래 종료 상태 저장을 수행한다.

56.21 마무리말
TCF 거래 테스트는 NSIGHT 개발자가 작성한 프로그램이 TCF 표준 거래로 운영 가능한지 증명하는 절차이다.
단순 API 테스트
= 응답이 왔는가?

TCF 거래 테스트
= 표준 전문으로 수신되었는가?
+ Header가 검증되었는가?
+ ServiceId로 Handler가 실행되었는가?
+ 세션/권한/거래통제가 적용되었는가?
+ Timeout과 중복요청이 통제되었는가?
+ 업무 결과가 표준 응답으로 조립되었는가?
+ 거래로그와 감사로그가 남았는가?

56.22 시사점
NSIGHT TCF의 운영 추적성은 거래 테스트에서 완성된다.응답 Body만 확인하면 개발 테스트이고, GUID·TraceId·ServiceId·거래코드·거래로그·오류코드까지 확인해야 TCF 거래 테스트이다.
TCF 거래 테스트의 최종 기준은 “요청 전문 1건이 운영자가 추적 가능한 거래로그 1건으로 남는가”이다.
