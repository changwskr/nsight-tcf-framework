46. OM 운영관리 개발 구조
46.1 도입 전 안내말
OM은 Operation Management의 약자이며, NSIGHT TCF Framework에서 운영 기준정보와 운영 통제 기능을 담당하는 관리 업무이다.
OM은 단순한 관리자 화면이 아니다.NSIGHT에서 OM은 사용자, 권한, 메뉴, ServiceId, 거래통제, Timeout, 오류코드, 공통코드, 세션, 거래로그, 감사로그, 배치, 배포, 파일관리, 환경설정, 운영 대시보드를 통합 관리하는 운영 기준정보 관리 모듈이다. 기존 설계 기준에서도 관리화면은 MG(Message) 업무에 넣지 않고 별도 OM(Operation Management) 업무로 분리하여 om.war, /om Context로 운영하는 것이 적합하다고 정리되어 있다.
OM 개발의 핵심 문장은 다음과 같다.
업무 WAR는 운영 기준정보를 직접 수정하지 않는다.운영 기준정보는 OM에서 관리하고, 업무 WAR는 조회·검증·캐시된 기준정보를 사용한다.

46.7.3 Handler 예시
@Component("OM.User.inquiry")
@RequiredArgsConstructor
public class OmUserInquiryHandler implements TransactionHandler {

    private final OmUserFacade omUserFacade;
    private final ObjectMapper objectMapper;

    @Override
    public Object handle(StandardRequest request, TransactionContext context) {
        OmUserInquiryRequest body = objectMapper.convertValue(
                request.getBody(),
                OmUserInquiryRequest.class
        );

        return omUserFacade.inquiryUsers(body, context);
    }
}

Handler는 업무 로직을 직접 처리하지 않는다.Handler는 표준 전문의 Body를 요청 DTO로 변환하고 Facade를 호출하는 역할만 수행한다.

46.7.4 Facade 예시
@Service
@RequiredArgsConstructor
public class OmUserFacade {

    private final OmUserService omUserService;

    @Transactional(readOnly = true)
    public OmUserInquiryResponse inquiryUsers(
            OmUserInquiryRequest request,
            TransactionContext context
    ) {
        return omUserService.inquiryUsers(request, context);
    }
}

조회 기능은 readOnly = true를 적용한다.등록, 수정, 삭제 기능은 별도 Facade 메서드에서 쓰기 트랜잭션으로 처리한다.

46.7.5 Service 예시
@Service
@RequiredArgsConstructor
public class OmUserService {

    private final OmUserRule omUserRule;
    private final OmUserDao omUserDao;

    public OmUserInquiryResponse inquiryUsers(
            OmUserInquiryRequest request,
            TransactionContext context
    ) {
        omUserRule.validateInquiryCondition(request);

        int totalCount = omUserDao.countUsers(request);
        List<OmUserDto> users = omUserDao.selectUsers(request);

        return OmUserInquiryResponse.builder()
                .totalCount(totalCount)
                .users(users)
                .build();
    }
}

Service는 조회 조건 검증, DAO 호출, 응답 DTO 조립을 담당한다.

46.7.6 Rule 예시
@Component
public class OmUserRule {

    public void validateInquiryCondition(OmUserInquiryRequest request) {
        if (request.getPageSize() != null && request.getPageSize() > 500) {
            throw new BusinessException(
                    "E-OM-USER-0001",
                    "사용자 조회 건수가 허용 범위를 초과했습니다."
            );
        }
    }
}

Rule은 업무 정책과 검증을 담당한다.사용자 조회 최대 건수, 필수값, 상태값, 중복 여부, 삭제 가능 여부 같은 판단은 Rule에 둔다.

46.7.7 DAO / Mapper 예시
@Repository
@RequiredArgsConstructor
public class OmUserDao {

    private final OmUserMapper omUserMapper;

    public int countUsers(OmUserInquiryRequest request) {
        return omUserMapper.countUsers(request);
    }

    public List<OmUserDto> selectUsers(OmUserInquiryRequest request) {
        return omUserMapper.selectUsers(request);
    }
}

@Mapper
public interface OmUserMapper {

    int countUsers(OmUserInquiryRequest request);

    List<OmUserDto> selectUsers(OmUserInquiryRequest request);
}

기능권한은 “권한그룹이 특정 메뉴에서 특정 기능 버튼을 사용할 수 있는지 통제하는 권한”으로 정의하는 것이 적절하다. 예를 들어 운영 조회자는 거래로그 조회와 다운로드만 가능하고, OM 관리자는 사용자관리 메뉴에서 조회·등록·수정·삭제가 가능하도록 설계한다.

46.13 OM 개발 체크리스트
점검 항목
확인
ServiceId가 OM.{대상}.{행위} 형식을 따르는가

Handler에서 업무 로직을 직접 처리하지 않는가

Facade에 트랜잭션 경계가 있는가

조회 거래는 readOnly = true를 적용했는가

변경 거래는 감사로그를 남기는가

등록/수정/삭제 시 Before/After 값이 기록되는가

운영 기준정보 삭제는 물리삭제보다 비활성 처리하는가

권한이 없는 사용자가 OM 기능을 실행할 수 없도록 차단하는가

다운로드 기능은 감사로그를 남기는가

ServiceId, 거래통제, Timeout 변경 시 Cache 갱신이 되는가

오류 발생 시 표준 오류코드로 응답하는가

개인정보와 비밀번호, Token, Secret 값이 마스킹되는가

대량 조회 시 페이징을 적용했는가

거래로그에서 GUID, TraceId, ServiceId 기준 조회가 가능한가

운영 설정값은 화면에서 직접 수정하지 않고 Git/CI-CD 기준 변경으로 통제되는가


46.14 마무리말
OM 운영관리 개발 구조의 핵심은 운영 기준정보의 중앙화이다.
업무 WAR가 각자 사용자, 권한, ServiceId, 거래통제, Timeout, 오류코드, 공통코드를 관리하면 운영 기준이 흩어지고 장애 분석이 어려워진다. 따라서 NSIGHT TCF에서는 OM을 별도 업무로 분리하고, 모든 운영 기준정보를 OM에서 관리해야 한다.
정리하면 다음과 같다.
OM = 운영 기준정보의 원장
TCF = 운영 기준정보를 기준으로 거래 실행 통제
업무 WAR = OM이 관리한 기준정보를 사용하여 업무 처리


46.15 시사점
OM은 개발 편의 기능이 아니라 운영 안정성 확보 장치이다.OM이 제대로 설계되어야 ServiceId 관리, 거래통제, Timeout, 권한, 오류코드, 거래로그, 감사로그가 하나의 흐름으로 연결된다.
따라서 OM 개발 시에는 단순 CRUD 화면을 만드는 관점이 아니라 다음 관점으로 접근해야 한다.
관점
시사점
운영자 관점
장애 발생 시 GUID, ServiceId, 사용자, 세션, 오류코드 기준으로 추적 가능해야 한다
개발자 관점
신규 ServiceId 등록, 거래통제 등록, Timeout 설정이 표준 절차로 가능해야 한다
보안 관점
권한 없는 운영자가 기준정보를 변경하지 못해야 한다
감사 관점
누가, 언제, 무엇을, 어떤 값에서 어떤 값으로 변경했는지 남아야 한다
아키텍처 관점
OM은 업무 WAR와 분리된 운영 기준정보 원장 역할을 해야 한다
