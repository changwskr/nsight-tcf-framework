6. 로컬 개발환경 구성
6.1 도입 전 안내말
본 장은 NSIGHT TCF Framework 개발자가 로컬 PC에서 소스를 내려받고, 빌드하고, 테스트하고, bootRun 방식으로 업무 서비스를 실행하기 위한 개발환경 구성 기준을 정의한다.
NSIGHT TCF Framework는 일반적인 단일 Spring Boot 애플리케이션이 아니다.공통 TCF 모듈, 업무 서비스 WAR, OM 운영관리, Gateway, Batch, JWT, Cache, UI, CI/CD 설정이 함께 구성되는 멀티 모듈 구조이다. 따라서 개발자는 단순히 Controller 하나를 실행하는 방식이 아니라, 다음 흐름을 이해한 상태에서 개발해야 한다.
Git Repository
   ↓
Gradle Multi Module
   ↓
tcf-core / tcf-web / tcf-cache / tcf-om / 업무 서비스
   ↓
bootRun
   ↓
POST /{businessCode}/online
   ↓
TCF.process()
   ↓
STF 전처리
   ↓
ServiceId Dispatcher
   ↓
Handler → Facade → Service → Rule → DAO/Mapper

NSIGHT 구축·운영 표준은 Apache, Tomcat, Spring Boot, Spring Session JDBC, RDW/ADW, GitLab CI/CD를 적용 범위로 두고 있다. 또한 NSIGHT Java 코딩 스타일 가이드는 Spring Boot, MyBatis, 예외처리, 로그, 트랜잭션, 보안, 테스트 기준을 통합 개발 표준으로 정의하고 있다.

6.4 로컬 개발환경 기본 구조

개발자 PC

 ├─ Git
 ├─ IDE
 ├─ Gradle Wrapper
 ├─ 로컬 DB 또는 개발 DB 접속정보
 ├─ nsight-tcf-framework 소스
 │
 ├─ tcf-util
 ├─ tcf-core
 ├─ tcf-web
 ├─ tcf-cache
 ├─ tcf-om
 ├─ tcf-gateway
 ├─ tcf-jwt
 ├─ tcf-batch
 ├─ sv-service
 ├─ ic-service
 └─ 기타 업무 서비스

로컬 실행 시에는 운영 Tomcat에 WAR를 배포하지 않고, 각 업무 서비스 모듈을 bootRun으로 실행한다.
로컬 개발 실행
= Spring Boot bootRun

운영 배포 실행
= WAR 생성 후 Tomcat 배포

bootRun과 Tomcat WAR 배포 차이는 뒤의 10장. bootRun / Tomcat WAR 배포 차이에서 상세히 설명한다.

6.6 소스 Clone 기준
개발자는 Git 저장소에서 기준 브랜치를 내려받는다.
git clone <NSIGHT Git Repository URL>
cd nsight-tcf-framework
git checkout develop

브랜치 사용 기준은 뒤의 7장. Git 저장소 및 브랜치 사용 기준에서 정의한다.
로컬 개발자는 원칙적으로 develop 브랜치에 직접 Commit하지 않는다.신규 기능 개발은 feature/* 브랜치를 생성하여 작업한다.
git checkout -b feature/sv-customer-summary

6.8 로컬 Profile 기준
로컬 개발은 반드시 local Profile을 사용한다.
-Dspring.profiles.active=local

또는 IDE 실행 설정에 다음 값을 지정한다.
VM Options:
-Dspring.profiles.active=local
-Dfile.encoding=UTF-8
-Duser.timezone=Asia/Seoul

Spring Boot 표준에서는 application.yml을 Port, Context Path, 세션, DB, HikariCP, MyBatis, Transaction, Security, Log, Actuator 설정을 통제하는 핵심 운영 기준 파일로 본다. 따라서 로컬에서도 운영 설정과 동일한 항목을 사용하되, 값은 로컬 기준으로 분리해야 한다.

6.12 로컬 실행 방법
SV 업무 서비스를 예로 들면 다음과 같이 실행한다.
./gradlew :sv-service:bootRun --args='--spring.profiles.active=local'

Windows에서는 다음과 같이 실행한다.
gradlew.bat :sv-service:bootRun --args="--spring.profiles.active=local"

OM을 실행하는 경우는 다음과 같다.
./gradlew :tcf-om:bootRun --args='--spring.profiles.active=local'

Gateway를 실행하는 경우는 다음과 같다.
./gradlew :tcf-gateway:bootRun --args='--spring.profiles.active=local'


6.13 로컬 표준 전문 테스트
업무 서비스가 실행되면 POST /{businessCode}/online으로 표준 전문을 호출한다.
SV 고객 요약 조회 예시는 다음과 같다.
POST http://localhost:8086/sv/online
Content-Type: application/json

{
  "header": {
    "serviceId": "SV.Customer.selectSummary",
    "transactionCode": "SV-INQ-0001",
    "businessCode": "SV",
    "serviceName": "고객요약조회",
    "user": "testuser",
    "channelId": "WEB",
    "branch": "0001"
  },
  "body": {
    "customerNo": "1234567890"
  }
}

정상 응답은 다음 구조를 따라야 한다.
{
  "header": {
    "serviceId": "SV.Customer.selectSummary",
    "transactionCode": "SV-INQ-0001",
    "businessCode": "SV"
  },
  "result": {
    "code": "SUCCESS",
    "message": "정상 처리되었습니다."
  },
  "body": {
    "customerNo": "1234567890",
    "customerName": "홍길동"
  }
}

NSIGHT TCF는 REST Resource 방식이 아니라 HTTP/JSON 표준 전문과 ServiceId Dispatcher 방식을 기준으로 한다. 따라서 로컬 테스트도 URL별 Controller 테스트가 아니라, serviceId가 올바른 Handler로 매핑되는지 확인하는 방식으로 수행해야 한다.

6.15 환경변수 관리 기준
로컬에서도 DB 비밀번호, API Key, JWT Secret 등은 소스에 직접 작성하지 않는다.
권장 방식은 환경변수 사용이다.
export NSIGHT_DB_USERNAME=nsight
export NSIGHT_DB_PASSWORD=local-password

application-local.yml에서는 다음처럼 참조한다.
spring:
  datasource:
    username: ${NSIGHT_DB_USERNAME}
    password: ${NSIGHT_DB_PASSWORD}

6.17 로컬 빌드 기준
전체 프로젝트 빌드는 다음 명령으로 수행한다.
./gradlew clean build

특정 업무 서비스만 빌드할 경우 다음과 같이 수행한다.
./gradlew :sv-service:clean :sv-service:build

WAR 생성은 다음 기준을 따른다.
./gradlew :sv-service:bootWar

빌드 산출물은 일반적으로 다음 위치에 생성된다.
sv-service/build/libs/sv-service.war

CI/CD 운영 매뉴얼도 GitLab, GitLab Runner, Gradle, 17 WAR 배포를 기준으로 빌드 자동화, 배포 자동화, 롤백 자동화, 운영 안정성 확보를 목표로 둔다. 따라서 로컬 빌드는 CI/CD에 올라가기 전 1차 품질 확인 단계로 본다.

개인정보 로그 출력이 없는가
□
DB 비밀번호가 소스에 포함되지 않았는가
□
운영 Profile을 사용하지 않았는가
□
로컬 빌드가 성공했는가
□

6.20 마무리말
로컬 개발환경은 단순히 개발자 개인 PC에서 프로그램을 실행하기 위한 환경이 아니다.NSIGHT TCF Framework에서는 로컬 개발환경도 운영 구조의 축소판이어야 한다.
즉, 개발자는 로컬에서도 다음 기준을 지켜야 한다.
POST /{businessCode}/online
   ↓
StandardRequest
   ↓
TCF.process()
   ↓
ServiceId Dispatcher
   ↓
Handler → Facade → Service → Rule → DAO/Mapper
   ↓
StandardResponse

로컬에서 이 구조가 지켜지지 않으면 개발 단계에서는 동작하더라도, 운영 배포 시 Gateway, OM, 거래통제, Timeout, 거래로그, 감사로그, CI/CD 기준과 맞지 않게 된다.

6.21 시사점
로컬 개발환경 구성의 핵심은 “내 PC에서만 돌아가는 개발”이 아니라 “운영 표준과 같은 방식으로 작게 실행하는 개발”이다.
따라서 NSIGHT 개발자는 로컬 개발 시점부터 다음 세 가지를 반드시 지켜야 한다.
핵심 기준
의미
표준 전문으로 호출
모든 거래는 StandardRequest로 테스트한다
ServiceId로 실행
URL이 아니라 serviceId가 실행 기준이다
계층 책임 준수
Handler, Facade, Service, Rule, DAO/Mapper 책임을 분리한다
이 기준이 지켜져야 개발 산출물이 OM 등록, 거래통제, Timeout 정책, 거래로그, CI/CD, 운영 전환 체크리스트까지 자연스럽게 연결된다.
