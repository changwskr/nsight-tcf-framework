54. 명명규칙 총정리
54.1 도입 전 안내말
NSIGHT TCF Framework의 명명규칙은 단순히 이름을 통일하는 기준이 아니다.이름 하나가 업무코드, URL, WAR, Package, ServiceId, Handler, Mapper, SQL ID, 거래로그, 오류코드까지 이어져야 한다.
기존 네이밍 원칙에서도 핵심은 업무코드 → Context → WAR → Package → ServiceId → Handler → Facade → Service → Rule → DAO/Mapper → MyBatis XML → SQL ID → 거래로그/오류코드가 하나의 선으로 연결되게 만드는 것으로 정리되어 있다.
업무코드
   ↓
Context Path
   ↓
WAR / Gradle Module
   ↓
Java Package
   ↓
ServiceId
   ↓
Handler
   ↓
Facade / Service / Rule / DAO / Mapper
   ↓
Mapper XML / SQL ID
   ↓
거래로그 / 오류코드 / 감사로그

54.23 개발자 점검 체크리스트
점검 항목
확인
업무코드가 표준 2자리 대문자인가?

Context, WAR, Package가 같은 업무코드를 사용하는가?

ServiceId가 {BusinessCode}.{Domain}.{action} 형식인가?

거래코드가 {업무코드}-{처리유형}-{일련번호} 형식인가?

Handler 이름이 ServiceId와 연결되는가?

Handler에서 DB를 직접 호출하지 않는가?

Facade, Service, Rule, DAO, Mapper 계층명이 명확한가?

Mapper XML namespace가 Mapper Interface와 일치하는가?

SQL ID가 selectList, insertData처럼 모호하지 않은가?

오류코드가 E-{DOMAIN}-{CATEGORY}-{NNNN} 형식인가?

로그에 guid, traceId, serviceId, transactionCode가 남는가?

약어와 임시 명칭이 운영 코드에 남아 있지 않은가?


54.24 마무리말
NSIGHT TCF의 명명규칙은 개발 편의를 위한 단순 규칙이 아니라 운영 추적성, 장애 분석, 거래통제, Timeout 정책, 오류코드, 감사로그까지 연결하는 기준이다.
따라서 개발자는 이름을 지을 때 다음 한 문장을 기준으로 삼아야 한다.
이 이름만 보고 어느 업무의 어떤 거래가 어떤 Handler, 어떤 SQL, 어떤 오류코드와 연결되는지 알 수 있어야 한다.
54.25 시사점
명명규칙이 흔들리면 TCF의 가장 중요한 장점인 ServiceId 기반 실행 통제와 운영 추적성이 약해진다.반대로 명명규칙이 일관되면 신규 업무 추가, 장애 추적, 로그 분석, Mapper 관리, OM 운영관리, CI/CD 배포까지 하나의 기준으로 연결된다.
