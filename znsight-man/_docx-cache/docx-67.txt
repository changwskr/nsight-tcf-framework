71.1 도입 전 안내말

기존 업무 서비스 샘플 설계에서도 SV 고객요약조회는 SV.Customer.selectSummary, 거래코드는 SV-INQ-0001, Endpoint는 POST /sv/online, 실행 구조는 Handler → Facade → Service → Rule → DAO → Mapper → RDW로 정의되어 있다.

71.2 샘플의 핵심 원칙
Controller는 개발하지 않는다.
Controller는 TCF 공통 진입점이 담당한다.

업무 개발자는 ServiceId별 Handler를 만든다.
Handler는 요청 Body를 DTO로 변환하고 Facade를 호출한다.

트랜잭션은 Facade에서 잡는다.
업무 검증과 판단은 Rule에서 처리한다.
DB 접근은 DAO / Mapper에서만 수행한다.
응답 전문 조립은 ETF 후처리에서 수행한다.

71.8 Handler 샘플
package com.nh.nsight.marketing.sv.entry.handler;

import com.fasterxml.jackson.databind.ObjectMapper;
import com.nh.nsight.marketing.sv.dto.SvCustomerSummaryRequest;
import com.nh.nsight.marketing.sv.entry.facade.SvCustomerFacade;
import com.nh.nsight.tcf.core.context.TransactionContext;
import com.nh.nsight.tcf.core.handler.TransactionHandler;
import lombok.RequiredArgsConstructor;
import org.springframework.stereotype.Component;

@Component("SV.Customer.selectSummary")
@RequiredArgsConstructor
public class SvCustomerSummaryHandler implements TransactionHandler {

    private final SvCustomerFacade svCustomerFacade;
    private final ObjectMapper objectMapper;

    @Override
    public String getServiceId() {
        return "SV.Customer.selectSummary";
    }

    @Override
    public Object handle(TransactionContext context, Object body) {
        SvCustomerSummaryRequest request =
                objectMapper.convertValue(body, SvCustomerSummaryRequest.class);

        return svCustomerFacade.selectCustomerSummary(context, request);
    }
}

Handler는 TCF Dispatcher가 serviceId 기준으로 찾는 업무 진입점이며, 요청 Body를 업무 DTO로 변환한 뒤 Facade를 호출한다. 기존 HTTP/JSON 처리 가이드에서도 SvCustomerSummaryHandler가 SV.Customer.selectSummary를 반환하고 Body를 SvCustomerSummaryRequest로 변환하여 Facade를 호출하는 예시가 제시되어 있다.

71.10 Service 샘플
package com.nh.nsight.marketing.sv.application.service;

import com.nh.nsight.marketing.sv.application.rule.SvCustomerRule;
import com.nh.nsight.marketing.sv.dto.SvCustomerSummaryRequest;
import com.nh.nsight.marketing.sv.dto.SvCustomerSummaryResponse;
import com.nh.nsight.marketing.sv.persistence.dao.SvCustomerDao;
import com.nh.nsight.tcf.core.context.TransactionContext;
import lombok.RequiredArgsConstructor;
import org.springframework.stereotype.Service;

@Service
@RequiredArgsConstructor
public class SvCustomerService {

    private final SvCustomerRule svCustomerRule;
    private final SvCustomerDao svCustomerDao;

    public SvCustomerSummaryResponse selectCustomerSummary(
            TransactionContext context,
            SvCustomerSummaryRequest request) {

        svCustomerRule.validateCustomerSummaryRequest(request);

        SvCustomerSummaryResponse response =
                svCustomerDao.selectCustomerSummary(request);

        svCustomerRule.validateCustomerExists(response);

        return response;
    }
}

Service는 Rule과 DAO를 조합하는 계층이다. 고객번호 검증은 Rule, DB 조회는 DAO, 결과 존재 여부 판단은 다시 Rule에서 수행한다.

71.11 Rule 샘플
package com.nh.nsight.marketing.sv.application.rule;

import com.nh.nsight.marketing.sv.dto.SvCustomerSummaryRequest;
import com.nh.nsight.marketing.sv.dto.SvCustomerSummaryResponse;
import com.nh.nsight.tcf.core.exception.BusinessException;
import org.springframework.stereotype.Component;

@Component
public class SvCustomerRule {

    public void validateCustomerSummaryRequest(SvCustomerSummaryRequest request) {
        if (request == null) {
            throw new BusinessException("E-SV-VAL-0001", "고객번호는 필수입니다.");
        }

        if (request.getCustomerNo() == null || request.getCustomerNo().isBlank()) {
            throw new BusinessException("E-SV-VAL-0001", "고객번호는 필수입니다.");
        }

        if (request.getCustomerNo().length() > 20) {
            throw new BusinessException("E-SV-VAL-0002", "고객번호 형식이 올바르지 않습니다.");
        }

        if (request.getBaseDate() != null
                && !request.getBaseDate().matches("\\d{8}")) {
            throw new BusinessException("E-SV-VAL-0003", "기준일자 형식이 올바르지 않습니다.");
        }
    }

    public void validateCustomerExists(SvCustomerSummaryResponse response) {
        if (response == null) {
            throw new BusinessException("E-SV-BIZ-0001", "조회된 고객정보가 없습니다.");
        }
    }
}

71.14 Mapper XML 샘플
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE mapper
  PUBLIC "-//mybatis.org//DTD Mapper 3.0//EN"
  "http://mybatis.org/dtd/mybatis-3-mapper.dtd">

<mapper namespace="com.nh.nsight.marketing.sv.persistence.mapper.SvCustomerMapper">

    <select id="selectCustomerSummary"
            parameterType="com.nh.nsight.marketing.sv.dto.SvCustomerSummaryRequest"
            resultType="com.nh.nsight.marketing.sv.dto.SvCustomerSummaryResponse">

        SELECT
              CUSTOMER_NO           AS customerNo
            , CUSTOMER_NAME         AS customerName
            , CUSTOMER_GRADE        AS customerGrade
            , BRANCH_CODE           AS branchCode
            , BRANCH_NAME           AS branchName
            , TOTAL_BALANCE         AS totalBalance
            , LOAN_BALANCE          AS loanBalance
            , PRODUCT_COUNT         AS productCount
            , LAST_TRANSACTION_DATE AS lastTransactionDate
        FROM RDW_CUSTOMER_SUMMARY
        WHERE CUSTOMER_NO = #{customerNo}

    </select>

</mapper>

Mapper XML의 namespace는 Mapper Interface의 FQCN과 동일하게 하고, SQL id는 Mapper Method명과 동일하게 한다. 기존 샘플 설계에서도 namespace는 Mapper 인터페이스 FQCN, id는 Mapper Method명과 동일하게 하며 SQL ID가 로그와 성능분석에서 추적 가능해야 한다고 정리되어 있다.

71.21 마무리말
SV 고객요약조회 샘플의 핵심은 업무 개발자가 새로운 구조를 만들지 않고, 정해진 TCF 표준 구조를 그대로 복제해서 개발하는 것이다.
Controller는 공통 TCF 진입점
실행 기준은 ServiceId
업무 시작점은 Handler
트랜잭션 경계는 Facade
업무 판단은 Rule
DB 접근은 DAO / Mapper
SQL은 Mapper XML
오류 응답은 ETF 후처리
거래 추적은 GUID + TraceId + ServiceId + SQL ID


71.22 시사점
SV 고객요약조회 샘플은 단순 예제가 아니라, NSIGHT 업무 개발의 기준 패턴이다.
신규 업무 거래는 SV.Customer.selectSummary 샘플에서 ServiceId, DTO, Rule, Mapper SQL만 업무에 맞게 바꾸면 동일한 구조로 개발할 수 있다.
