10. bootRun / Tomcat WAR 배포 차이
10.1 도입 전 안내말
본 장은 NSIGHT TCF Framework에서 개발자가 로컬에서 실행하는 bootRun 방식과 운영환경에서 사용하는 Tomcat WAR 배포 방식의 차이를 정의한다.
NSIGHT 개발자는 로컬에서는 빠른 개발과 테스트를 위해 bootRun으로 업무 서비스를 실행한다. 그러나 운영환경에서는 Apache, Tomcat, Spring Boot, Spring Session JDBC, RDW/ADW, SESSIONDB, CI/CD가 함께 동작하므로 WAR를 생성하여 외부 Tomcat에 배포하는 방식을 기준으로 한다. NSIGHT Tomcat 운영 기준에서도 Tomcat은 Spring Boot Runtime, Session Cookie 처리, Transaction 처리, Thread Pool 관리, Connection Pool 연계 역할을 수행한다고 정의되어 있다.
핵심 차이는 다음 한 문장으로 정리할 수 있다.
bootRun은 개발자가 로컬에서 Spring Boot를 직접 실행하는 방식이고, Tomcat WAR 배포는 운영 Tomcat이 업무 WAR를 적재하여 실행하는 방식이다.

10.3 bootRun 방식
bootRun은 Gradle이 Spring Boot 애플리케이션을 직접 실행하는 방식이다.
개발자 PC
   ↓
Gradle bootRun
   ↓
Spring Boot Application 실행
   ↓
내장 Tomcat 기동
   ↓
/sv/online 호출
   ↓
TCF.process()
   ↓
Handler → Facade → Service → Rule → DAO/Mapper

SV 업무 실행 예시는 다음과 같다.
./gradlew :sv-service:bootRun --args='--spring.profiles.active=local'

Windows에서는 다음과 같이 실행한다.
gradlew.bat :sv-service:bootRun --args="--spring.profiles.active=local"

OM 실행 예시는 다음과 같다.
./gradlew :tcf-om:bootRun --args='--spring.profiles.active=local'

Gateway 실행 예시는 다음과 같다.
./gradlew :tcf-gateway:bootRun --args='--spring.profiles.active=local'

10.5 Tomcat WAR 배포 방식
Tomcat WAR 배포는 Gradle로 업무 WAR를 생성한 뒤 외부 Tomcat에 배포하는 방식이다.
Gradle bootWar
   ↓
sv.war 생성
   ↓
CI/CD Artifact 저장
   ↓
Tomcat webapps 배포
   ↓
Tomcat 기동 또는 Reload
   ↓
/sv/online 호출
   ↓
Spring Boot Application 초기화
   ↓
TCF.process()

WAR 생성 명령은 다음과 같다.
./gradlew :sv-service:bootWar

생성 결과는 다음과 같다.
sv-service/build/libs/sv.war

Tomcat 배포 위치 예시는 다음과 같다.
$CATALINA_BASE/webapps
 ├─ sv.war
 ├─ ic.war
 ├─ cm.war
 ├─ mg.war
 └─ om.war

NSIGHT CI/CD 운영 기준도 GitLab, GitLab Runner, Gradle, WAR 배포를 기준으로 빌드 자동화, 배포 자동화, 롤백 자동화, 운영 안정성 확보를 목표로 둔다.

10.8 build.gradle 차이
업무 WAR 모듈은 로컬에서는 bootRun이 가능해야 하고, 운영에서는 bootWar로 WAR를 생성해야 한다.
plugins {
    id 'java'
    id 'war'
    id 'org.springframework.boot'
    id 'io.spring.dependency-management'
}

dependencies {
    implementation project(':tcf-web')
    implementation project(':tcf-core')
    implementation project(':tcf-cache')
    implementation project(':tcf-integration')

    implementation 'org.springframework.boot:spring-boot-starter-web'
    implementation 'org.springframework.boot:spring-boot-starter-validation'
    implementation 'org.mybatis.spring.boot:mybatis-spring-boot-starter'
    implementation 'com.zaxxer:HikariCP'

    providedRuntime 'org.springframework.boot:spring-boot-starter-tomcat'

    testImplementation 'org.springframework.boot:spring-boot-starter-test'
}

bootWar {
    archiveFileName = 'sv.war'
}

설정
의미
war plugin
WAR 생성 가능하게 함
org.springframework.boot
Spring Boot 실행 및 bootWar 지원
providedRuntime spring-boot-starter-tomcat
운영 Tomcat에서 제공하는 Tomcat과 중복 방지
bootWar.archiveFileName
표준 WAR 파일명 지정
bootRun
로컬 실행 Task
bootWar
운영 배포 산출물 생성 Task

10.9 SpringBootServletInitializer 기준
외부 Tomcat에 WAR로 배포하려면 Spring Boot Application Class가 SpringBootServletInitializer를 상속해야 한다.
@SpringBootApplication
public class SvServiceApplication extends SpringBootServletInitializer {

    public static void main(String[] args) {
        SpringApplication.run(SvServiceApplication.class, args);
    }

    @Override
    protected SpringApplicationBuilder configure(SpringApplicationBuilder builder) {
        return builder.sources(SvServiceApplication.class);
    }
}

실행 방식
필요한 이유
bootRun
main()으로 실행
Tomcat WAR
Tomcat이 configure()를 통해 Spring Application 초기화
이 구조를 사용하면 하나의 업무 서비스가 로컬에서는 bootRun, 운영에서는 WAR 배포 방식으로 모두 실행될 수 있다.

10.10 Context Path 차이
bootRun
로컬 bootRun에서는 application-local.yml의 server.servlet.context-path가 적용된다.
server:
  port: 8086
  servlet:
    context-path: /sv

호출 URL은 다음과 같다.
http://localhost:8086/sv/online

Tomcat WAR
Tomcat WAR 배포에서는 WAR 파일명 또는 Tomcat Context 설정이 Context Path가 된다.
sv.war → /sv
om.war → /om
ic.war → /ic

호출 URL은 다음과 같다.
https://nh.marketing.com/sv/online

운영에서는 WAR 파일명, Context Path, Apache Routing, Gateway Routing, Header businessCode가 모두 일치해야 한다.

10.20 개발 완료 후 WAR 검증 절차
개발자는 bootRun 성공 후 반드시 WAR 생성과 Tomcat 배포 검증을 수행해야 한다.

10.24 마무리말
bootRun과 Tomcat WAR 배포는 서로 경쟁하는 방식이 아니다.두 방식은 목적이 다르다.
bootRun
= 개발자가 빠르게 실행하고 디버깅하기 위한 방식

Tomcat WAR 배포
= 운영과 동일한 구조로 배포, 검증, 모니터링하기 위한 방식

개발자는 bootRun으로 기능을 빠르게 만들고, 반드시 bootWar와 Tomcat 배포 방식으로 운영 적합성을 확인해야 한다.

10.25 시사점
NSIGHT TCF Framework에서 개발 완료의 기준은 “내 PC에서 bootRun으로 실행된다”가 아니다.최종 기준은 다음이다.
WAR로 패키징되고, Tomcat에 배포되며, Context Path가 맞고, ServiceId가 실행되고, 거래로그와 Health Check가 정상인 상태이다.
따라서 개발자는 다음 순서를 습관화해야 한다.
bootRun으로 개발 확인
   ↓
단위 테스트
   ↓
Gradle build
   ↓
bootWar 생성
   ↓
Tomcat 배포 확인
   ↓
Health Check
   ↓
ServiceId Smoke Test
   ↓
Merge Request

결론적으로 bootRun은 개발 속도를 높이는 도구이고, Tomcat WAR 배포는 운영 가능성을 증명하는 절차이다.
