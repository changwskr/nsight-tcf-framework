맞습니다. 클라이언트가 tcf-gateway를 통과하지 않고 업무 WAR/Tomcat으로 직접 들어올 가능성이 있다면, 업무 WAR 쪽에서도 JWT 유효성 검증을 해야 합니다.
다만 정확히 말하면, tcf-core 안에 직접 JwtDecoder를 넣기보다는, 업무 WAR의 공통 진입 계층인 tcf-web 또는 공통 Filter/Interceptor에서 토큰을 검증하고, tcf-core의 STF에서는 검증 결과와 전문 Header가 맞는지 2차 확인하는 구조가 가장 좋습니다.

1. 결론
상황
JWT 검증 위치

Gateway 검증 중심으로 충분
업무 WAR 직접 호출 차단이 불완전
Gateway + 업무 WAR 이중 검증 필요
즉, Gateway 우회 가능성이 있다면 업무 WAR도 스스로 방어해야 합니다.

2. 왜 필요한가
현재 tcf-core는 표준 전문 모델, 거래 컨텍스트, STF/TCF/ETF 처리기, Dispatcher를 제공하는 핵심 프레임워크 모듈로 설명되어 있고, 처리 흐름도 TCF.process() 안에서 STF.preProcess()가 Header 검증, GUID/TraceId, 세션·권한, 멱등성, 거래통제, Timeout 정책을 수행하는 구조로 되어 있습니다. (GitHub)
현재 STF.preProcess() 실제 흐름도 Header 검증 후 GUID/TraceId를 만들고, SessionValidator, AuthorizationValidator, 거래통제, Timeout, 멱등성, 거래로그 시작 순서로 실행됩니다. (GitHub)
그런데 현재 SessionValidator는 header.userId가 있는지만 확인하고, AuthorizationValidator도 branchId 존재 여부만 확인하는 단순 구조입니다. (GitHub) 즉, 업무 WAR로 직접 들어온 요청이 Header에 userId와 branchId만 임의로 넣으면, 지금 구조만으로는 “진짜 인증된 사용자냐”를 충분히 확인하기 어렵습니다.

왜 tcf-core에 직접 JwtDecoder를 넣는 것은 비추천이냐면, tcf-core는 HTTP Header를 직접 다루는 보안 필터 계층이 아니라, 표준 전문과 거래 실행을 처리하는 코어 엔진이기 때문입니다. tcf-core의 Header 모델인 StandardHeader는 businessCode, serviceId, transactionCode, channelId, userId, branchId 같은 전문 Header 값을 담는 구조입니다. (GitHub) JWT 원문인 Authorization Header는 이 모델의 본래 책임이 아닙니다.

10. 결론
네, Gateway를 통과하지 않고 업무 WAR/Tomcat으로 직접 들어올 수 있다면 TCF 영역에서도 토큰 유효성 처리가 필요합니다.
다만 설계상 더 정확한 표현은 다음입니다.
JWT 원문의 서명·만료·issuer·audience 검증은 tcf-web의 공통 JWT Filter에서 수행하고, tcf-core의 STF에서는 검증된 사용자 정보와 전문 Header의 userId·branchId·channelId가 일치하는지 확인한다.
따라서 최종 권장안은 이렇습니다.
1. 업무 WAR 직접 접근은 인프라에서 최대한 차단한다.
2. 그래도 직접 접근 가능성이 있으면 업무 WAR 앞단에 JWT Filter를 둔다.
3. JWT Filter는 Gateway와 동일한 기준으로 토큰을 검증한다.
4. TCF Core는 JwtDecoder를 직접 갖기보다 인증 결과와 전문 Header를 비교한다.
5. 권한, 거래통제, Timeout, 거래로그는 기존 TCF 흐름에서 처리한다.

한 줄로 정리하면:
직접 호출 가능성이 있다면 업무 WAR도 JWT를 검증해야 하고, 그 검증은 tcf-core 안보다는 tcf-web 앞단 Filter에 두는 것이 가장 합당합니다.
