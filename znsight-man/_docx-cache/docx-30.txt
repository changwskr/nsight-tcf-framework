25. Service 개발 기준
25.1 도입 전 안내말
NSIGHT TCF Framework에서 Service는 실제 업무 처리를 수행하는 핵심 계층이다.
Handler가 표준 전문을 업무 DTO로 변환하고, Facade가 유스케이스 흐름을 조정한다면, Service는 해당 유스케이스 안에서 실제 업무 처리를 수행한다.
TransactionHandler
   ↓
Facade
   ↓
Service
   ↓
Rule
   ↓
DAO / Mapper

Service는 업무 개발자가 가장 많이 작성하는 계층이다.따라서 Service에 모든 것을 넣기 쉽지만, NSIGHT에서는 Service의 책임을 명확히 제한한다.
Service는 업무 처리를 수행한다.
Rule은 업무 규칙을 판단한다.
DAO는 DB 접근을 담당한다.
Mapper는 SQL을 실행한다.

25.4 Service 기본 작성 예시
SV 고객 요약 조회 Service 예시는 다음과 같다.
@Service
@RequiredArgsConstructor
public class SvCustomerService {

    private final SvCustomerRule svCustomerRule;
    private final SvCustomerDao svCustomerDao;

    @Transactional(readOnly = true)
    public SvCustomerSummaryResponse selectCustomerSummary(
            SvCustomerSummaryRequest request,
            TransactionContext context
    ) {
        svCustomerRule.validateCustomerInquiry(
                request.getCustomerNo(),
                context
        );

        SvCustomerSummaryResult result =
                svCustomerDao.selectCustomerSummary(
                        SvCustomerSummaryCriteria.from(request, context)
                );

        return SvCustomerSummaryResponse.from(result);
    }
}

    private final SvProductService svProductService;

    @Transactional(readOnly = true)
    public SvCustomerDetailResponse selectCustomerDetail(
            SvCustomerDetailRequest request,
            TransactionContext context
    ) {

                svCustomerService.selectCustomerBasicInfo(
                        request.getCustomerNo(),

Rule 예시는 다음과 같다.
@Component
@RequiredArgsConstructor
public class CmCampaignRule {

    private final CmCampaignDao cmCampaignDao;

    public void validateCampaignExists(String campaignId) {
        if (!cmCampaignDao.existsCampaign(campaignId)) {
            throw new BusinessException(
                    "E-CM-VAL-0001",
                    "캠페인 정보가 존재하지 않습니다."
            );
        }
    }

    public void validateUpdatePeriod(LocalDate startDate, LocalDate endDate) {
        if (startDate.isAfter(endDate)) {
            throw new BusinessException(
                    "E-CM-VAL-0002",
                    "시작일자는 종료일자보다 클 수 없습니다."
            );
        }
    }
}

25.13 조회 Service 개발 기준
조회 Service는 readOnly = true를 적용하고, 대량조회 제한을 반드시 고려한다.
@Transactional(readOnly = true)
public SvCustomerSearchResponse selectCustomerList(
        SvCustomerSearchRequest request,
        TransactionContext context
) {
    svCustomerRule.validateSearchCondition(request, context);

    SvCustomerSearchCriteria criteria =
            SvCustomerSearchCriteria.from(request, context);

    int totalCount = svCustomerDao.countCustomerList(criteria);

    List<SvCustomerSearchItem> list =
            svCustomerDao.selectCustomerList(criteria);

    return SvCustomerSearchResponse.of(list, PageResponse.of(request, totalCount));
}

@Transactional
public CmCampaignCreateResponse createCampaign(

    cmCampaignRule.validateCreatePeriod(command.getStartDate(), command.getEndDate());

@Transactional
public OmMenuDeleteResponse deleteMenu(

25.23 Service 개발 예시: SV 고객 목록 조회
@Service
@RequiredArgsConstructor
public class SvCustomerService {

    private final SvCustomerRule svCustomerRule;
    private final SvCustomerDao svCustomerDao;

    @Transactional(readOnly = true)
    public SvCustomerSearchResponse selectCustomerList(
            SvCustomerSearchRequest request,
            TransactionContext context
    ) {
        svCustomerRule.validateSearchCondition(request, context);

        SvCustomerSearchCriteria criteria =
                SvCustomerSearchCriteria.from(request, context);

        int totalCount = svCustomerDao.countCustomerList(criteria);

        List<SvCustomerSearchItem> list =
                svCustomerDao.selectCustomerList(criteria);

        return SvCustomerSearchResponse.builder()
                .list(list)
                .page(PageResponse.of(request.getPageNo(), request.getPageSize(), totalCount))
                .build();
    }
}

25.29 마무리말
Service는 NSIGHT 업무 개발의 중심 계층이다.하지만 중심 계층이라고 해서 모든 책임을 가져서는 안 된다.
Service는 업무 처리를 수행하되, 업무 규칙 판단은 Rule에 위임하고, DB 접근은 DAO에 위임해야 한다. 또한 표준 전문, Header, 응답 전문 조립은 TCF 공통 영역에 맡겨야 한다.
Service는 업무 처리의 중심이다.
그러나 Service는 모든 것을 처리하는 곳이 아니다.
Service가 얇고 명확해야 전체 구조가 유지된다.
