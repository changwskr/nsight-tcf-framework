59. 성능 테스트 기준
59.1 도입 전 안내말
NSIGHT TCF Framework에서 성능 테스트는 단순히 “TPS가 몇 건 나오는가”를 확인하는 시험이 아니다.성능 테스트는 표준 전문 기반 거래가 실제 운영 부하에서 p95 3초 이내로 처리되고, Tomcat Thread, JVM Heap, Hikari Pool, DB SQL, SESSIONDB, Gateway, Apache가 함께 안정적으로 버티는지 검증하는 절차이다.
NSIGHT 기준 성능 목표는 6,000지점 × 6명 = 36,000명, 설계 세션 43,200개, 피크 동시요청률 10% 기준 1,200 TPS, 목표 응답시간 p95 3초, Busy Thread 70% 이하, Heap 70% 이하, Hikari Pool 70~80% 이하로 정리되어 있다.
성능 테스트
= TPS 검증
+ p95 / p99 응답시간 검증
+ 오류율 검증
+ Tomcat Thread 검증
+ JVM Heap / GC 검증
+ Hikari Pool 검증
+ MyBatis SQL Time 검증
+ DB Session 검증
+ SESSIONDB 검증
+ Gateway / Apache / L4 분산 검증
+ 거래로그 / 성능로그 추적 검증

59.25 마무리말
NSIGHT 성능 테스트의 핵심은 TPS 숫자 하나가 아니다.TPS가 나와도 p95가 깨지거나, Thread가 고갈되거나, Hikari Pool이 대기하거나, DB SQL이 느리거나, 거래로그가 누락되면 운영 가능한 성능이라고 볼 수 없다.
성능 테스트 완료
= 1,200 TPS 달성
+ p95 3초 이하
+ 오류율 1% 이하
+ Thread / Heap / Pool 안정
+ SQL Time 안정
+ GC 영향 없음
+ Timeout 정상 작동
+ GUID 기준 거래 추적 가능

59.26 시사점
NSIGHT는 Apache + Gateway + Tomcat + TCF + MyBatis + HikariCP + RDW/ADW + SESSIONDB가 결합된 구조다.따라서 성능 테스트는 어느 한 계층의 부하 테스트가 아니라 계층 간 정합성 검증이어야 한다. NSIGHT 통합 운영 매뉴얼도 17개 WAR, 다중 WAS, RDW/ADW/SESSIONDB, L4/GSLB가 결합된 구조에서는 한 계층만 튜닝해서 안정성을 확보하기 어렵고, Timeout·Thread·Pool·Session 기준을 하나의 운영 표준으로 묶어야 한다고 정리한다.
성능 테스트의 최종 기준은 “1,200 TPS에서 p95 3초를 만족하면서도 Thread, Heap, Pool, SQL, 세션, 거래로그가 모두 운영 기준 안에 있는가”이다.
