업무코드 / Context / WAR / Package 명명규칙 설계기준
1. 도입 전 안내말
본 절은 NSIGHT TCF Framework에서 사용하는 업무코드, Context Path, WAR 파일명, Java Package의 명명규칙을 정의한다.
이 네 가지는 별도 기준이 아니라 하나의 기준으로 묶어야 한다.
업무코드
   ↓
Context Path
   ↓
WAR 파일명
   ↓
Gradle Module
   ↓
Java Package
   ↓
ServiceId / Handler / Mapper / SQL ID / 오류코드

NSIGHT 명명규칙 문서에서도 이름 하나가 업무코드, Context Path, WAR, Gradle Module, Java Package, ServiceId, Handler, Mapper XML, SQL ID, 거래로그, 오류코드, 감사로그까지 연결되어야 한다고 정의하고 있다.

9. Package 설계기준
Package는 소스의 책임 경계를 표현한다. 업무코드가 Package에 포함되어야 개발자가 소스만 보고 어느 업무의 코드인지 알 수 있다.
항목
기준
표준 Root
com.nh.nsight.marketing
업무 Package
com.nh.nsight.marketing.{업무코드소문자}
예시
com.nh.nsight.marketing.sv
대소문자
소문자만 사용
업무코드 누락
금지
업무 간 Package 혼재
금지
일부 개발자 가이드에는 Package Root를 com.nh.nsight.{업무코드}로 표현한 예시가 있으나, NSIGHT 마케팅플랫폼 기준으로는 com.nh.nsight.marketing.{업무코드}를 표준으로 고정하는 것이 좋다. 패키지 Root가 혼재되면 소스 이동, Scan 범위, Mapper Scan, Component Scan, 운영 추적 기준이 흔들릴 수 있다.

개발자 가이드에서도 Handler, Facade, Service, Rule, DAO, Mapper, DTO, Client, Config, Support의 패키지 위치를 계층별로 분리하는 기준을 제시하고 있다.

Context는 업무코드 소문자 기준인가
□
WAR 파일명은 Context와 일치하는가
□
Gradle Module은 {업무코드소문자}-service 규칙을 따르는가
□
OM은 tcf-om 모듈 예외 기준을 따르는가
□
Package Root는 com.nh.nsight.marketing.{업무코드}로 통일되었는가
□
Java Prefix는 업무코드 PascalCase 기준인가
□
ServiceId Prefix는 업무코드 대문자와 일치하는가
□
거래코드 Prefix는 업무코드와 일치하는가
□
오류코드 Prefix는 업무코드와 일치하는가
□
Mapper XML 위치가 mapper/{업무코드} 기준인가
□
Apache / Gateway 라우팅과 Context가 일치하는가
□
Spring context-path와 WAR Context가 일치하는가
□

18. 마무리말
업무코드 / Context / WAR / Package 명명규칙은 NSIGHT 개발 표준의 시작점이다.
업무코드가 정해지면
Context가 정해지고,
WAR가 정해지고,
Package가 정해지고,
ServiceId와 Mapper와 SQL ID와 오류코드까지 정해진다.

따라서 NSIGHT에서는 다음 원칙을 고정한다.
업무코드는 대문자, Context와 WAR는 소문자, Package는 업무코드 소문자 기준, 실행은 ServiceId 기준으로 관리한다.
이 기준을 지키면 개발, 배포, 로그 추적, 장애 분석, PMO 보고, 운영 인수인계까지 같은 이름 체계로 설명할 수 있다.
