# 5. 개발자 역할과 책임

> **NSIGHT TCF 개발 Manual** · 원본: [`znsight-guide-word`](../znsight-guide-word/) · 갱신: 2026-07-05

## 5. 개발자 역할과 책임

### 5.1 도입 전 안내말

본 장은 NSIGHT TCF Framework 기반 개발에 참여하는 각 개발자와 관련 담당자의 역할과 책임을 정의한다.
NSIGHT 개발은 단순히 업무 프로그램을 작성하는 일이 아니다. 하나의 ServiceId를 개발하더라도 표준 전문, Header, Handler, Facade, Service, Rule, DAO, Mapper, SQL, 오류코드, 거래로그, 권한, Timeout, 테스트, 배포 기준이 함께 맞아야 한다.
NSIGHT Java 코딩 스타일 가이드도 적용 대상을 마케팅플랫폼, SingleView, 공통모듈, 배치·연계성 Java 모듈, MyBatis SQL Mapper까지 포함하고 있다. 또한 전처리·후처리 요구사항에서는 적용 계층을 Controller, Filter, Interceptor, Facade, Service, DAO, Integration Adapter까지 포함하고, 공통기능으로 전문, GUID, 인증, 권한, 검증, 오류, 로그, 감사, Timeout, 재처리, 모니터링을 요구한다.
따라서 NSIGHT 개발자의 책임은 다음 한 문장으로 정리할 수 있다.
업무 개발자는 업무 로직을 구현하고, 공통 프레임워크는 거래 실행을 통제하며, 운영관리와 DevOps는 개발 산출물이 운영 가능한 서비스로 배포되도록 보장한다.

### 5.2 역할과 책임 결론

NSIGHT 개발 조직의 역할은 다음 10개로 구분한다.
| No | 역할 | 핵심 책임 |
| --- | --- | --- |
| 1 | 업무 개발자 | 업무 ServiceId, Handler, Facade, Service, Rule, DAO, Mapper 개발 |
| 2 | 공통 프레임워크 개발자 | TCF Core, Web, Cache, Integration 공통 모듈 개발 |
| 3 | OM 개발자 | 운영관리 화면과 ServiceId, 거래통제, 권한, Timeout, 오류코드 관리 기능 개발 |
| 4 | Gateway 개발자 | 업무코드 기반 라우팅, 세션/JWT 확인, Downstream Relay 개발 |
| 5 | JWT / SSO 개발자 | SSO 인증 연계, JWT 발급·검증·갱신·폐기 기능 개발 |
| 6 | Batch 개발자 | Scheduler, Batch Job, 실행이력, 실패 재처리 기능 개발 |
| 7 | UI 개발자 | WebTopSuite, tcf-ui, OM 화면에서 표준 전문 호출 구현 |
| 8 | SQL / DB 개발자 | MyBatis SQL, RDW/ADW 접근, SQL 성능, 테이블 설계 지원 |
| 9 | 테스트 담당자 | 단위, 통합, 거래, 성능, 장애, 보안 테스트 수행 |
| 10 | DevOps / 배포 담당자 | Git, Gradle, CI/CD, WAR 배포, Health Check, Rollback 관리 |
| 11 | 아키텍트 / 리뷰어 | 개발 표준 준수, 구조 적합성, 품질 게이트 검토 |
| 12 | PMO / 품질 담당자 | 산출물, 일정, 표준 준수, 검토 이력, 운영 전환 준비 관리 |

### 5.4 업무 개발자 역할과 책임

업무 개발자는 NSIGHT에서 실제 업무 거래를 구현하는 담당자이다.업무 개발자는 하나의 ServiceId를 기준으로 Handler부터 Mapper까지 개발한다.
| 책임 영역 | 주요 책임 |
| --- | --- |
| ServiceId 설계 | 업무코드, 업무도메인, 행위 기준으로 ServiceId 정의 |
| 거래코드 정의 | 조회, 등록, 수정, 삭제, 실행, 다운로드 등 거래 유형에 맞는 거래코드 정의 |
| Handler 개발 | StandardRequest.body를 업무 Request DTO로 변환하고 Facade 호출 |
| Facade 개발 | 유스케이스 조립, 트랜잭션 경계 설정 |
| Service 개발 | 업무 처리 절차 구현, Rule/DAO/Client 호출 조정 |
| Rule 개발 | 업무 규칙, 조건 판단, 계산, 검증 로직 구현 |
| DAO 개발 | Mapper 호출 캡슐화 |
| Mapper 개발 | MyBatis Mapper Interface와 XML 작성 |
| 오류 처리 | BusinessException, 표준 오류코드 사용 |
| 로그 처리 | GUID, TraceId, ServiceId 기준 로그 추적 가능하게 작성 |
| 테스트 | Rule, Service, Mapper, TCF 거래 테스트 작성 |
| 운영 등록 확인 | ServiceId, 거래통제, Timeout, 권한, 오류코드 등록 확인 |

업무 개발자가 직접 구현해야 하는 표준 구조는 다음과 같다.
```text
SvCustomerSelectSummaryHandler
↓
SvCustomerFacade
↓
SvCustomerService
↓
SvCustomerRule
↓
SvCustomerDao
↓
SvCustomerMapper
↓
SvCustomerMapper.xml
```

업무 개발자가 하지 말아야 할 일은 다음과 같다.
| 금지 사항 | 사유 |
| --- | --- |
| 업무별 Controller를 임의로 다수 생성 | /online 공통 진입 구조가 깨진다. |
| Handler에서 SQL 실행 | DAO/Mapper 계층 책임을 침범한다. |
| Service에서 표준 응답 JSON 직접 생성 | ETF 후처리 기준을 우회한다. |
| 오류코드 없이 문자열 오류 반환 | OM 오류코드 관리와 연결되지 않는다. |
| 세션에 대량 데이터 저장 | SESSIONDB와 WAS Heap에 부담을 준다. |
| Paging 없는 대량 조회 작성 | DB와 WAS 자원 고갈 위험이 있다. |
| Timeout 없는 외부연계 호출 | 장애 전파와 Thread 고갈 위험이 있다. |

### 5.5 공통 프레임워크 개발자 역할과 책임

공통 프레임워크 개발자는 TCF 공통 실행 구조를 개발하고 유지한다.
| 책임 영역 | 주요 책임 |
| --- | --- |
| 표준 전문 | StandardRequest, StandardHeader, StandardResponse, Result 구조 관리 |
| TCF 처리 | STF → Dispatcher → ETF 공통 파이프라인 개발 |
| Header 검증 | 필수 Header, 업무코드, ServiceId, 거래코드 검증 |
| GUID / TraceId | 거래 추적 ID 생성과 MDC 연계 |
| Dispatcher | serviceId → TransactionHandler 매핑 구조 관리 |
| 예외 처리 | BusinessException, SystemException, 오류코드 매핑 구조 관리 |
| 공통 응답 | ETF 표준 응답 조립, 오류 응답, 마스킹 처리 |
| 거래로그 | PROCESSING, SUCCESS, FAIL, TIMEOUT 상태 기록 구조 제공 |
| 공통 Web | OnlineTransactionController, Global Exception Handler 관리 |
| Cache | 공통코드, ServiceId, 정책정보 Cache 구조 제공 |
| Integration | 업무 간 HTTP/JSON 표준 전문 호출 Client 제공 |

공통 프레임워크 개발자의 핵심 원칙은 다음과 같다.
업무 로직을 공통 프레임워크에 넣지 않는다. 공통 프레임워크는 거래 실행 방식과 운영 통제 방식만 제공한다.

### 5.6 OM 개발자 역할과 책임

OM 개발자는 NSIGHT 운영관리 기능을 개발한다.OM은 관리자 화면이 아니라 운영 기준정보 관리 시스템이다.
| 책임 영역 | 주요 책임 |
| --- | --- |
| 사용자 관리 | 사용자 정보, 상태, 로그인 정책, 중복 로그인 관리 |
| 권한 관리 | 권한그룹, 메뉴권한, 기능권한, 데이터권한 관리 |
| 메뉴 관리 | 메뉴 트리, 화면 URL, ServiceId 매핑 관리 |
| ServiceId 관리 | Service Catalog 등록, 수정, 중지, 조회 |
| 거래통제 관리 | 허용 거래, 차단 거래, 사용자·채널·지점 기준 통제 |
| Timeout 정책 관리 | ServiceId별 Online, DB, 외부연계 Timeout 관리 |
| 오류코드 관리 | 사용자 메시지, 운영자 메시지, 오류 등급 관리 |
| 공통코드 관리 | 코드그룹, 코드값, 사용 여부, Cache Evict 관리 |
| 세션 관리 | 로그인 세션 조회, 강제 종료, 만료 관리 |
| 거래로그 조회 | GUID, TraceId, ServiceId, 사용자 기준 거래 추적 |
| 감사로그 조회 | 중요 행위, 권한 변경, 다운로드 이력 추적 |
| 배치 관리 | Job 등록, 실행, 중지, 이력 조회 |
| 배포 관리 | 배포 요청, 승인, Health Check, 롤백 이력 조회 |
| 환경설정 조회 | Runtime 설정, Profile, Hikari, MyBatis, Actuator 정보 조회 |

OM 개발자는 다음 기준을 지켜야 한다.
| 기준 | 설명 |
| --- | --- |
| OM도 TCF 거래로 개발 | POST /om/online + OM.* ServiceId 기준 |
| 운영 변경 이력 필수 | 권한, 거래통제, Timeout, 오류코드 변경은 감사로그 대상 |
| 조회와 변경 분리 | 운영 기준정보 조회와 변경 권한을 분리 |
| Cache 반영 | 공통코드, ServiceId, 정책 변경 시 Cache Evict 또는 Reload 연계 |
| 보안 강화 | OM 기능은 높은 권한과 감사 추적을 기본 적용 |

### 5.7 Gateway 개발자 역할과 책임

Gateway 개발자는 업무 요청을 적절한 업무 WAR 또는 업무 WAS로 전달하는 Application Gateway를 개발한다.
| 책임 영역 | 주요 책임 |
| --- | --- |
| 업무코드 라우팅 | URL Path와 Header의 businessCode 기준 라우팅 |
| Route Table 조회 | TCF_GATEWAY_ROUTE 기준 대상 서비스 결정 |
| 세션 Cookie 확인 | JSESSIONID 또는 표준 세션 Cookie 존재 여부 확인 |
| JWT 확인 | Gateway 호출 시 Access Token 검증 또는 전달 |
| Header 검증 | Path 업무코드와 Header 업무코드 불일치 차단 |
| Downstream Relay | 대상 업무 WAR의 /online으로 표준 전문 전달 |
| Timeout 적용 | Downstream 호출 Timeout 설정 |
| 장애 처리 | 대상 서버 장애, Timeout, 응답 오류 처리 |
| Gateway 로그 | 요청, 대상 URL, 응답시간, 결과 상태 기록 |
| Health Check | Downstream 대상 상태 확인 |

Gateway 개발의 핵심 기준은 다음과 같다.
Gateway는 업무 로직을 처리하지 않는다.
Gateway는 인증 확인, Header 검증, 업무코드 라우팅, Relay만 담당한다.

### 5.8 JWT / SSO 개발자 역할과 책임

JWT / SSO 개발자는 NSIGHT 인증 연계와 토큰 생명주기를 관리한다.
| 책임 영역 | 주요 책임 |
| --- | --- |
| SSO Token 검증 | SSO 서버 또는 IdP에서 전달된 Token, Code, Assertion 검증 |
| OM 사용자 조회 | SSO 인증 결과와 OM_USER 매핑 |
| 세션 생성 | 로그인 성공 시 Spring Session JDBC 기반 세션 생성 |
| JWT 발급 | Access Token, Refresh Token 발급 |
| JWT 검증 | 서명, 만료, 발급자, 사용자, 권한 Claim 검증 |
| Refresh Token 관리 | 재발급, 만료, 폐기 관리 |
| Token DenyList | 로그아웃, 강제종료, 탈취 의심 토큰 차단 |
| 인증 감사로그 | 로그인, 발급, 갱신, 폐기, 실패 이력 기록 |
| 보안 설정 | Secure Cookie, HttpOnly, SameSite, CSRF 정책 연계 |
| Gateway 연계 | Gateway에서 JWT 확인 또는 Downstream 전달 구조 제공 |

JWT / SSO 개발 원칙은 다음과 같다.
JWT는 사용자가 누구인지 증명하는 인증 수단이고, TCF는 그 사용자가 해당 거래를 실행해도 되는지 판단하는 거래통제 프레임워크이다.

### 5.9 Batch 개발자 역할과 책임

Batch 개발자는 Scheduler와 Batch Job을 개발한다.
| 책임 영역 | 주요 책임 |
| --- | --- |
| Job 정의 | 배치 ID, 배치명, 실행 목적, 처리 주기 정의 |
| Scheduler 설정 | Cron, 주기, 수동 실행 여부 설정 |
| 실행 이력 | 시작시간, 종료시간, 처리건수, 성공/실패 상태 기록 |
| 실패 재처리 | 실패 원인, 재실행 가능 여부, 재처리 방식 정의 |
| 운영 상태 수집 | AP, DB, 세션, 배포, Cache 상태 수집 |
| 로그 정리 | 거래로그, 감사로그, 다운로드로그 보관기간 관리 |
| 파일 정리 | 파일 보관기간 만료, 삭제 대상 처리 |
| Timeout 적용 | 장시간 배치 제한시간 관리 |
| 중복 실행 방지 | 동일 Job 중복 실행 Lock 관리 |
| OM 연계 | 배치 등록, 실행, 중지, 이력 조회 화면 연계 |

Batch 개발자는 다음 원칙을 따른다.
Scheduler = 언제 실행할 것인가
Batch     = 무엇을 실행할 것인가
OM        = 실행과 이력을 어떻게 관리할 것인가

### 5.10 UI 개발자 역할과 책임

UI 개발자는 WebTopSuite, tcf-ui, OM Admin 화면에서 NSIGHT 표준 전문을 생성하고 호출한다.
| 책임 영역 | 주요 책임 |
| --- | --- |
| 표준 전문 생성 | header + body 구조로 요청 JSON 생성 |
| Header 구성 | businessCode, serviceId, transactionCode, user, channelId, branch 설정 |
| Endpoint 호출 | POST /{businessCode}/online 또는 Gateway Endpoint 호출 |
| 응답 처리 | StandardResponse.result 기준 성공/실패 처리 |
| 오류 메시지 표시 | 사용자 메시지 기준으로 화면 표시 |
| 권한 연계 | 메뉴, 버튼, 기능권한 기준 UI 제어 |
| 입력값 검증 | 화면 1차 검증, 서버 검증과 중복 보완 |
| 파일 처리 | 업로드/다운로드 권한, 크기, 확장자 기준 반영 |
| 페이징 처리 | pageNo, pageSize, 검색조건 전달 |
| Trace 지원 | 장애 문의 시 GUID, TraceId, ServiceId 확인 가능하게 표시 또는 보관 |

UI 개발자는 다음을 지켜야 한다.
| 기준 | 설명 |
| --- | --- |
| URL에 업무 기능을 직접 표현하지 않음 | 실제 기능은 ServiceId로 구분 |
| Header 필수값 누락 금지 | TCF 전처리에서 차단됨 |
| 응답 Result 기준 처리 | HTTP 200이어도 업무 실패일 수 있음 |
| 민감정보 표시 주의 | 마스킹 기준 적용 |
| 중복 클릭 방지 | 중복요청, Idempotency 정책과 연계 |

### 5.11 SQL / DB 개발자 역할과 책임

SQL / DB 개발자는 MyBatis SQL 품질과 DB 접근 기준을 관리한다.
| 책임 영역 | 주요 책임 |
| --- | --- |
| Mapper SQL 작성 | 업무 Mapper XML SQL 작성 또는 검토 |
| SQL ID 관리 | ServiceId와 추적 가능한 SQL ID 작성 |
| RDW/ADW 구분 | 실시간 조회와 분석성 조회 분리 |
| Paging 적용 | 목록 조회에 pageNo, pageSize, offset 기준 적용 |
| Count SQL 작성 | 목록 조회 시 totalCount SQL 분리 |
| 인덱스 검토 | 조회 조건과 정렬 기준에 맞는 인덱스 검토 |
| 실행계획 확인 | Full Scan, Cartesian Join, 과도한 Sort 방지 |
| Query Timeout | SQL Timeout 기준 반영 |
| 대량 조회 통제 | 온라인 대량 조회 금지 또는 다운로드/Batch 분리 |
| 테이블 변경 관리 | 테이블, 컬럼, 인덱스 변경 시 영향 분석 |

SQL / DB 개발자는 다음 원칙을 지킨다.
DAO/Mapper는 업무 로직을 처리하지 않고, SQL 실행 책임만 가진다. SQL은 빠르게 끝나고, 추적 가능하고, Timeout 기준 안에서 수행되어야 한다.

### 5.12 테스트 담당자 역할과 책임

테스트 담당자는 개발 결과가 TCF 표준 구조와 운영 기준을 만족하는지 검증한다.
| 테스트 영역 | 주요 책임 |
| --- | --- |
| 단위 테스트 | Rule, Service, DAO 단위 검증 |
| Mapper 테스트 | SQL 실행, ResultMap, Paging, Count SQL 검증 |
| Handler 테스트 | Body 변환, Facade 호출 검증 |
| TCF 거래 테스트 | StandardRequest → StandardResponse 전체 흐름 검증 |
| 오류 테스트 | 오류코드, 사용자 메시지, 운영자 메시지 검증 |
| 권한 테스트 | 권한 없는 사용자 차단 확인 |
| 거래통제 테스트 | 미등록 ServiceId, 미허용 거래 차단 확인 |
| Timeout 테스트 | 지연 SQL, 지연 외부연계, Downstream Timeout 검증 |
| 세션 테스트 | 로그인 세션, 만료, 중복 로그인, 강제 종료 검증 |
| 보안 테스트 | 마스킹, 민감정보 로그, SQL Injection 방지 검증 |
| 성능 테스트 | 주요 거래 응답시간, TPS, SQL 시간 검증 |
| 장애 테스트 | DB 지연, SessionDB 장애, Gateway 오류, WAS 장애 검증 |

테스트 완료 기준은 다음과 같다.
기능 정상
+ 표준 전문 정상
+ 오류 응답 정상
+ 권한/거래통제 정상
+ Timeout 정상
+ 로그/감사 정상
+ SQL 성능 정상
+ 배포 후 Health Check 정상

### 5.13 DevOps / 배포 담당자 역할과 책임

DevOps / 배포 담당자는 개발 산출물이 안정적으로 빌드, 배포, 롤백될 수 있도록 관리한다.
NSIGHT CI/CD 운영 매뉴얼은 GitLab, GitLab Runner, Gradle, 17 WAR 배포를 기준으로 개발 표준화, 형상관리 표준화, 빌드 자동화, 배포 자동화, 롤백 자동화, 운영 안정성 확보를 목표로 한다.
| 책임 영역 | 주요 책임 |
| --- | --- |
| Git 관리 | Branch, Merge Request, Tag, Release 관리 |
| Build 관리 | Gradle Build, Test, Artifact 생성 |
| 품질 게이트 | Checkstyle, PMD, SonarQube, Test Coverage 연계 |
| WAR 패키징 | 업무별 WAR 생성 기준 관리 |
| 환경 설정 | local/dev/stg/prd Profile과 외부 설정 관리 |
| 배포 자동화 | Tomcat 배포 Script, 배포 순서, 권한 관리 |
| Health Check | 배포 후 Liveness, Readiness, Smoke Test 확인 |
| 롤백 | 이전 WAR 복구, 설정 복구, 트래픽 복귀 |
| 이력 관리 | 배포자, 배포시간, 버전, 결과, 승인 이력 기록 |
| 장애 대응 | 배포 실패, 기동 실패, Health Check 실패 시 조치 |

DevOps 담당자는 다음 기준을 지킨다.
| 기준 | 설명 |
| --- | --- |
| 운영 직접 수정 금지 | 운영 서버에서 임의 수정하지 않고 Git 기준 반영 |
| 배포 전 테스트 필수 | 빌드와 테스트 실패 시 배포 금지 |
| WAR 단위 명확화 | 업무코드와 WAR명이 일치해야 함 |
| 설정 분리 | 환경별 설정 파일과 Secret 분리 |
| 롤백 가능성 확보 | 항상 이전 버전 복구 가능해야 함 |

아키텍트와 리뷰어는 개발 산출물이 NSIGHT 표준 구조에 맞는지 검토한다.

| 배포와 롤백이 가능한가? | 운영 안정성 확인 |

### 5.15 PMO / 품질 담당자 역할과 책임

PMO와 품질 담당자는 개발 활동이 표준 프로세스와 산출물 기준에 맞게 진행되는지 관리한다.
| 책임 영역 | 주요 책임 |
| --- | --- |
| 산출물 관리 | 개발자 가이드, 설계서, 프로그램 목록, 테스트 결과 관리 |
| 일정 관리 | 개발, 리뷰, 테스트, 배포 일정 추적 |
| 표준 준수 확인 | 개발 표준 체크리스트 수집과 검토 |
| 품질 지표 관리 | 결함, 테스트 통과율, 코드 리뷰 결과, 빌드 성공률 관리 |

PMO 관점에서 중요한 것은 다음이다.
개발 완료 보고는 코드 완료가 아니라
표준 준수 + 테스트 완료 + 운영 등록 + 배포 준비 완료를 의미한다.

### 5.18 개발 완료의 책임 기준

개발 완료
= 소스 작성 완료
+ 단위 테스트 완료
+ Mapper SQL 검증 완료
+ TCF 거래 테스트 완료
+ 오류/권한/Timeout 검증 완료
+ ServiceId 등록 확인
+ 거래통제 등록 확인
+ 권한/메뉴 매핑 확인
+ 오류코드 등록 확인
+ CI/CD 빌드 통과
+ 배포 후 Health Check 통과

역할별 개발 완료 책임은 다음과 같다.
| 역할 | 완료 책임 |

| OM 개발자 | 운영 기준정보 등록과 조회 가능 상태 확인 |
| SQL / DB 개발자 | SQL 성능과 DB 반영 확인 |
| 테스트 담당자 | 테스트 결과와 결함 종료 확인 |
| DevOps 담당자 | 배포와 Health Check 성공 확인 |
| 아키텍트 | 표준 준수와 운영 가능성 승인 |

### 5.21 마무리말

NSIGHT 개발에서 역할과 책임을 명확히 하는 이유는 개발자를 통제하기 위해서가 아니다.각자가 맡은 책임을 명확히 해야 하나의 거래가 개발 단계에서 운영 단계까지 끊기지 않고 연결될 수 있기 때문이다.
핵심은 다음이다.
업무 개발자
= 업무 로직 구현 책임

공통 프레임워크 개발자
= 거래 실행 표준 제공 책임

OM 개발자
= 운영 기준정보 관리 책임

Gateway / JWT / Batch 개발자
= 공통 실행 인프라 기능 제공 책임

SQL / DB 개발자
= 데이터 접근 안정성 책임

테스트 / DevOps
= 검증과 배포 안정성 책임

아키텍트 / PMO
= 표준 준수와 운영 전환 책임

### 5.22 시사점

NSIGHT 개발자 역할과 책임의 최종 시사점은 다음과 같다.
| 시사점 | 설명 |
| --- | --- |
| 개발 완료 기준이 명확해진다 | 코드 작성뿐 아니라 테스트, 운영 등록, 배포 검증까지 완료 기준이 된다. |
| 장애 책임 추적이 쉬워진다 | Handler, SQL, Gateway, Session, 배포 중 어느 영역 문제인지 구분할 수 있다. |
| 협업 비용이 줄어든다 | 각 역할의 산출물과 검토 기준이 명확해진다. |
| 운영 전환 품질이 높아진다 | ServiceId, 거래통제, 권한, Timeout, 오류코드가 누락되지 않는다. |
| PMO 대응력이 높아진다 | RACI와 체크리스트 기반으로 진행률과 품질을 설명할 수 있다. |

최종 문장은 다음과 같다.
NSIGHT에서 개발자의 책임은 프로그램을 만드는 것에서 끝나지 않는다. 운영 가능한 거래 단위를 완성하는 것까지가 개발자의 책임이다.
