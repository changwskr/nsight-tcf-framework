NSIGHT TCF 개발방법론

신규 프로젝트 아키텍처 구축 방법론 설계기준

1. 도입 전 안내말

NSIGHT 프로젝트를 처음 시작할 때 가장 먼저 해서는 안 되는 일은 다음과 같다.

```
서버 사양부터 결정
→ Tomcat부터 설치
→ 업무 WAR부터 생성
→ ServiceId부터 채번
→ 화면별 프로그램 개발
→ 개발 완료 후 용량·보안·운영 검토
```

이 방식은 개발은 빨리 시작할 수 있지만, 프로젝트 후반에 다음 문제가 발생한다.

- 사용자·TPS 근거 없이 VM, JVM, Thread, DB Pool이 결정된다.
- 업무 경계가 정해지지 않은 상태에서 WAR와 패키지가 생성된다.
- 화면, ServiceId, 프로그램, SQL 간 추적성이 끊어진다.
- Gateway, JWT, 세션, TCF 검증 책임이 중복된다.
- 거래통제와 Timeout 정책이 개발 후 OM에 사후 등록된다.
- 성능시험에서 문제가 발견돼 배포구조를 다시 설계한다.
- 운영 직전에 로그, 모니터링, 장애 대응 기능을 추가한다.
- 설계서와 실제 소스·설정·배포구조가 서로 달라진다.
NSIGHT 프로젝트는 단순한 업무 애플리케이션 구축이 아니다.

```
업무 요구사항
+ 데이터 플랫폼
+ 온라인 거래 처리
+ TCF 공통 통제
+ Gateway·JWT·권한
+ 업무 WAR
+ OM 운영관리
+ 배치·파일·연계
+ 용량·가용성·DR
+ CI/CD·배포
+ 로그·감사·장애 대응
```

이 모든 항목을 하나의 생명주기로 연결해야 하는 운영 통제형 엔터프라이즈 플랫폼 구축 프로젝트다.

NSIGHT의 아키텍처 검증 범위도 시스템 구성도만 확인하는 수준이 아니라 다음 정합성을 함께 확인해야 한다.

```
비전·요구사항
↔ 논리·물리·런타임 구조
↔ 상세설계
↔ 소스코드
↔ OM 기준정보
↔ 환경설정
↔ 배포구조
↔ 테스트 결과
↔ 운영 증적
```

특히 ServiceId ↔ Handler ↔ OM Catalog, JWT Claim ↔ StandardHeader, GUID·TraceId ↔ 거래로그·감사로그 정합성이 유지되어야 한다.

따라서 NSIGHT 신규 프로젝트는 다음 순서로 진행해야 한다.

```
0. 거버넌스와 의사결정체계 수립
1. 프로젝트 비전·목표·범위 확정
2. 현행·이해관계자·제약조건 분석
3. 업무기능·도메인·데이터 소유권 정의
4. 기능·비기능 요구사항 기준선 확정
5. 목표 아키텍처와 대안 의사결정
6. TCF 플랫폼 및 개발표준 설계
7. 애플리케이션·데이터·연계·보안 상세설계
8. 물리구조·용량·가용성·DR 설계
9. 개발환경·CI/CD·운영환경 기반 구축
10. 선도개발과 아키텍처 실행 가능성 검증
11. 본 개발·OM 등록·자동 품질 Gate 운영
12. 통합·성능·보안·장애·DR 시험
13. 운영전환·초기 안정화
14. 지속적인 변경·호환성·폐기 관리
```

2. 문서 개요

2.1 목적

본 기준서의 목적은 NSIGHT 프로젝트를 처음 시작할 때 무엇을 어떤 순서로 정의하고 검증해야 하는지를 아키텍처 구축 방법론 관점에서 규정하는 것이다.

| 목적 | 설명 |
| --- | --- |
| 진행 순서 표준화 | 요구사항부터 운영 안정화까지 선후관계 정의 |
| 조기 의사결정 | 개발 이후 변경비용이 큰 항목을 선행 확정 |
| 책임 명확화 | SA·AA·DA·TA·보안·개발·운영 책임 구분 |
| 추적성 확보 | 요구사항·설계·코드·OM·시험·운영 증적 연결 |
| 운영 가능성 확보 | 성능·보안·장애·배포·복구를 설계 단계에서 반영 |
| 품질 객관화 | 단계별 Architecture Gate와 완료기준 정의 |
| 변경 통제 | ADR·Risk·Gap·예외·기술부채의 지속 관리 |

2.3 대상 독자

- 프로젝트 책임자와 PMO
- 시스템·애플리케이션·데이터·기술 아키텍트
- 프레임워크·업무·UI 개발팀
- 보안·인프라·DBA·DevOps 담당자
- 테스트·품질 담당자
- 운영·장애 대응 담당자
2.4 선행조건

프로젝트 착수 단계에서 다음이 확보되어야 한다.

- 공식 프로젝트 책임자와 아키텍처 책임자
- 업무 및 시스템 담당 조직
- 설계·소스·시험 산출물 저장소
- 의사결정 및 승인체계
- 요구사항 식별번호 관리기준
- 아키텍처 검토 일정과 Gate
- 이슈·Risk·Gap·ADR 관리대장
2.5 용어 정의

| 용어 | 정의 |
| --- | --- |
| Architecture Gate | 다음 단계 진입 가능 여부를 결정하는 공식 검토지점 |
| Baseline | 승인되어 변경통제를 받는 기준 버전 |
| ADR | 아키텍처 의사결정의 대안·근거·영향을 기록한 문서 |
| RTM | 요구사항과 설계·구현·시험을 연결하는 추적 매트릭스 |
| SoT | 공식 기준정보의 단일 원천 |
| Gap | 목표 상태와 현재 상태의 차이 |
| Blocker | 다음 단계 진행을 중지해야 하는 치명적 결함 |
| 선도개발 | 핵심 구조의 실행 가능성을 조기에 검증하는 대표 구현 |
| 운영 준비도 | 배포·감시·장애 대응·복구가 가능한 정도 |

3. NSIGHT 아키텍처 구축 방법론 전체 구조

3.1 단계별 전체 흐름

```
[G0] 거버넌스
      ↓
[G1] 비전·목표·범위
      ↓
[G2] 현행·제약 분석
      ↓
[G3] 업무·도메인·데이터 경계
      ↓
[G4] 기능·비기능 요구사항
      ↓
[G5] 목표 아키텍처·ADR
      ↓
[G6] TCF 플랫폼·표준
      ↓
[G7] 상세 아키텍처
      ↓
[G8] 물리·용량·가용성·DR
      ↓
[G9] 개발·배포 기반
      ↓
[G10] 선도개발
      ↓
[G11] 본 개발·품질 Gate
      ↓
[G12] 통합 검증
      ↓
[G13] 운영전환
      ↓
[G14] 안정화·지속 개선
```

3.2 단계별 요약

| 단계 | 핵심 질문 | 주요 결과 |
| --- | --- | --- |
| G0 | 누가 어떤 기준으로 결정하는가 | RACI, Gate, ADR, 산출물 체계 |
| G1 | 무엇을 왜 구축하는가 | 목표, 범위, KPI, 로드맵 |
| G2 | 현재 구조와 제약은 무엇인가 | AS-IS, 문제점, 기술부채 |
| G3 | 업무와 데이터 책임을 어떻게 나눌 것인가 | 업무기능, 도메인, 데이터 소유권 |
| G4 | 어느 수준의 품질을 만족해야 하는가 | 기능·비기능 요구사항 Baseline |
| G5 | 어떤 구조로 구축할 것인가 | Big Picture, 논리·물리·런타임 구조 |
| G6 | TCF가 무엇을 표준 통제하는가 | TCF·ServiceId·전문·계층·OM 표준 |
| G7 | 각 영역을 어떻게 상세 구현하는가 | 애플리케이션·데이터·연계·보안 설계 |
| G8 | 어느 규모와 장애 수준을 감당해야 하는가 | 용량, JVM, Pool, HA, DR |
| G9 | 개발부터 배포까지 어떻게 통제하는가 | 저장소, 환경, CI/CD, 배포체계 |
| G10 | 설계가 실제로 동작하는가 | 선도개발, 표준 샘플, Gap |
| G11 | 전체 개발이 표준을 지키는가 | 코드, OM 등록, 자동 품질검증 |
| G12 | 정상·실패·피크·장애 상황을 견디는가 | 종합 시험결과와 결함현황 |
| G13 | 운영자가 배포·감시·복구할 수 있는가 | Go/No-Go, 전환·롤백 계획 |
| G14 | 운영 결과가 설계에 반영되는가 | 안정화, 기술부채, 기준 개정 |

각 단계는 단순 문서 완료가 아니라 다음 형식으로 관리한다.

```
입력자료
→ 분석·설계 활동
→ 필수 의사결정
→ 산출물
→ 검증 증적
→ Gate 통과
```

4. G0 — 아키텍처 거버넌스 및 프로젝트 통제체계 수립

4.1 목적

프로젝트 시작 전에 의사결정, 검토, 변경, 예외, 위험 관리체계를 확정한다.

아키텍처 거버넌스를 나중에 만들면 팀별로 서로 다른 기준으로 개발을 시작하게 된다.

4.2 최초 확정 항목

| 구분 | 확정 내용 |
| --- | --- |
| 의사결정 | 최종 아키텍처 승인자와 대리자 |
| 역할 | SA, AA, DA, TA, 보안, DevOps, 운영 RACI |
| Gate | 분석·설계·개발·시험·전환 검토지점 |
| 문서 | 필수 산출물과 작성 책임 |
| 저장소 | 문서·소스·설정의 공식 SoT |
| ADR | 주요 기술결정 기록 방식 |
| 예외 | 표준 미준수 승인·만료·개선 절차 |
| Risk·Gap | 등록, 담당자, 기한, 종료기준 |
| 변경관리 | Baseline 변경 승인과 영향분석 |
| 품질등급 | Blocker, Critical, Major, Minor 기준 |

거버넌스 단계에서는 검토 범위, 역할, 산출물, ADR, 예외 승인, 통합 Risk·Gap 체계를 가장 먼저 정해야 한다.

4.3 필수 산출물

- 아키텍처 관리계획서
- 프로젝트 RACI
- Architecture Gate 정의서
- 산출물 목록 및 템플릿
- ADR 템플릿
- Risk·Gap·기술부채 관리대장
- 아키텍처 예외 승인서
- 변경관리 절차서
4.4 Gate G0 완료 기준

```
아키텍처 책임자 지정
+ 검토조직 구성
+ Gate 일정 승인
+ 문서 SoT 지정
+ ADR·예외·변경절차 승인
+ Blocker 0건
```

5. G1 — 프로젝트 비전·목표·범위 정의

5.1 목적

“무엇을 구축하는가”보다 먼저 “왜 구축하는가”와 “어떤 결과를 성공으로 보는가”를 확정한다.

5.2 정리해야 할 내용

5.2.1 프로젝트 비전

예:

```
분산된 상호금융 정보와 마케팅 기능을
표준화된 NSIGHT 정보계 플랫폼으로 통합하고,
업무 거래의 통제·추적·운영 안정성을 확보한다.
```

5.2.2 정량적 성공기준

| 구분 | 정의 예 |
| --- | --- |
| 사용자 | 지점·본부·운영관리자 규모 |
| 응답시간 | 평균, p95, 최대 허용시간 |
| 처리량 | 평시·피크·스트레스 TPS |
| 가용성 | 업무등급별 목표 가용성 |
| 복구 | RTO·RPO |
| 배포 | 배포시간, 무중단 또는 중단 허용시간 |
| 품질 | 결함밀도, 테스트 통과율 |
| 추적성 | GUID·TraceId·ServiceId 검색 가능 |
| 보안 | 인증·권한·감사·개인정보 기준 |
| 운영 | 장애 탐지·원인 식별·복구 목표 |

5.3 범위 정의

```
포함 업무
제외 업무
1차·2차 구축 범위
채널 범위
데이터 범위
외부연계 범위
배치·파일 범위
운영관리 범위
DR 범위
```

5.4 필수 산출물

- 프로젝트 비전 및 목표서
- 시스템 범위도
- 대상·제외 업무 목록
- 이해관계자 목록
- 성공 KPI
- 단계별 구축 로드맵
- 상위 요구사항 목록
5.5 Gate G1 완료 기준

- 프로젝트 목표가 정량적으로 표현되어 있다.
- 업무·시스템·데이터·채널 범위가 구분되어 있다.
- 1차 구축과 후속 구축 범위가 분리되어 있다.
- 주요 이해관계자가 범위를 승인했다.
6. G2 — 현행 구조·제약조건·기술부채 분석

6.1 목적

신규 구축이라도 기존 조직, 데이터, 인증, 네트워크, 운영절차와 완전히 독립적이지 않다.

따라서 현행 분석은 기존 프로그램 목록만 조사하는 활동이 아니다.

6.2 현행 분석 범위

| 영역 | 확인 내용 |
| --- | --- |
| 채널 | 사용자, 화면, 전용브라우저, BI포털 |

7. G3 — 업무기능·도메인·데이터 소유권 설계

7.1 목적

WAR, ServiceId, 패키지, 테이블을 정하기 전에 업무 책임 경계를 먼저 정의한다.

```
업무기능
→ 업무영역
→ 업무코드
→ 도메인
→ 유스케이스
→ 화면·이벤트
→ ServiceId
→ 프로그램
→ 데이터
```

7.2 업무기능 분해

권장 분해 순서는 다음과 같다.

```
Level 1. 업무영역
Level 2. 업무그룹
Level 3. 업무기능
Level 4. 유스케이스
Level 5. 화면 이벤트·거래
```

예:

```
마케팅
 └─ 고객 단일뷰
     └─ 고객 종합정보
         ├─ 고객요약 조회
         ├─ 상품보유 조회
         └─ 접촉이력 조회
```

7.3 도메인 정의

도메인은 패키지 이름이 아니라 다음 요소가 함께 변경되는 업무 경계다.

```
업무 용어
+ 업무 규칙
+ 사용 사례
+ ServiceId
+ 프로그램
+ 데이터 소유권
+ 권한·감사
+ 담당 조직
```

NSIGHT 업무 WAR 내부는 업무코드 → 도메인 → 계층 순서로 구성하는 것이 적절하다. 계층만 먼저 나누면 한 도메인의 Handler, Service, DAO가 여러 디렉터리에 흩어지고 변경 책임이 불명확해진다.

7.4 데이터 소유권

각 데이터는 다음 중 하나로 분류한다.

| 유형 | 의미 |
| --- | --- |
| 소유 데이터 | 해당 도메인만 생성·변경·삭제 가능 |
| 참조 데이터 | 다른 도메인이 조회할 수 있으나 변경 불가 |
| 공유 기준정보 | 명시된 공통 관리주체가 관리 |
| 통합 조회모델 | 여러 원천을 조회 목적으로 조합 |
| 이벤트 데이터 | 상태변화를 다른 도메인에 전달 |
| 운영 데이터 | OM, 거래로그, 감사로그, 배포이력 |

25. 시사점

25.1 핵심 아키텍처 판단

첫째, NSIGHT 개발 매뉴얼은 프로젝트의 출발점이 아니라, 목표 아키텍처와 업무 경계가 확정된 이후 적용하는 구현 기준이다.

둘째, 다음 순서는 바뀌어서는 안 된다.

```
요구사항
→ 도메인
→ 목표 아키텍처
→ TCF 표준
→ 상세설계
→ 용량·물리구조
→ 선도개발
→ 본 개발
→ 시험
→ 운영전환
```

셋째, ServiceId와 프로그램만 연결해서는 부족하다.

```
요구사항
↔ 화면 이벤트
↔ ServiceId
↔ Handler
↔ 프로그램
↔ SQL·DB
↔ OM 정책
↔ 테스트
↔ 운영로그
```

전체 관계가 연결되어야 한다.

넷째, 용량산정은 인프라팀의 독립 활동이 아니다. 업무 사용자, 거래량, 응답시간, SQL, 배포구조, 장애격리 정책이 함께 확정되어야 한다.

다섯째, 아키텍처는 문서가 아니라 코드·설정·OM·배포·운영에서 실행되는 통제체계여야 한다.

| 위험 | 영향 |
| --- | --- |
| 개발 선착수 | 구조 재작업과 표준 불일치 |

| 도메인 미정의 | WAR·패키지·데이터 책임 혼선 |

25.3 우선 보완 과제

- 프로젝트 전체 Architecture Gate 기준을 공식화한다.
- 요구사항–도메인–ServiceId–프로그램–SQL 추적체계를 만든다.
- 업무코드·WAR·도메인·데이터 소유권을 먼저 확정한다.
- 비기능 요구사항과 용량산정 전제를 Baseline으로 관리한다.
- TCF·OM·Gateway·JWT 책임 경계를 확정한다.
- 대표 선도개발로 전체 실행 흐름을 검증한다.
- 구조·등록정보·보안·SQL 검사를 CI/CD Gate에 포함한다.
- 운영진단·Runbook·롤백을 개발 산출물에 포함한다.

26. 마무리말

NSIGHT 프로젝트를 처음 시작할 때 가장 중요한 것은 많은 설계서를 한꺼번에 작성하는 것이 아니다.

```
무엇을 먼저 결정해야 하는가
어떤 결정이 다음 단계의 전제인가
누가 승인해야 하는가
무엇으로 검증할 것인가
어떤 증적이 있어야 완료인가
```

이 다섯 가지를 단계별로 명확히 하는 것이 중요하다.

최종적으로 NSIGHT 아키텍처 구축 방법론은 다음 한 문장으로 정리할 수 있다.

```
요구사항과 업무 경계를 먼저 확정하고,
목표 아키텍처와 TCF 표준을 설계한 뒤,
선도개발과 자동 품질 Gate로 실행 가능성을 검증하고,
성능·보안·장애·운영전환 시험을 통과한 구조만 운영에 반영한다.
```

NSIGHT의 성공 기준은 시스템이 한 번 정상 실행되는 것이 아니다.

```
표준에 맞게 개발할 수 있고,
용량을 예측할 수 있으며,
장애 원인을 추적할 수 있고,
안전하게 배포·복구할 수 있으며,
변경 이후에도 정합성을 유지할 수 있는가
```

이 조건을 만족할 때 비로소 아키텍처 구축이 완료된 것으로 판단한다.
