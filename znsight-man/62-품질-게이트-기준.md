# 62. 품질 게이트 기준

> **NSIGHT TCF 개발 Manual** · 원본: [`znsight-guide-word`](../znsight-guide-word/) · 갱신: 2026-07-05

## 62. 품질 게이트 기준

### 62.1 도입 전 안내말

품질 게이트는 개발자가 만든 소스를 다음 단계로 넘겨도 되는지 판단하는 공식 통과 기준이다.
NSIGHT TCF Framework에서는 단순히 “컴파일이 된다”는 이유만으로 개발 완료로 보지 않는다.온라인 거래는 HTTP/JSON 표준 전문 + ServiceId Dispatcher + STF/TCF/ETF 공통 처리 + Handler → Facade → Service → Rule → DAO/Mapper 흐름으로 실행되므로, 품질 게이트도 코드, 전문, 세션, 권한, 거래통제, Timeout, 로그, SQL, 배포까지 함께 확인해야 한다.
NSIGHT Java 코딩 스타일 가이드에서도 품질 통제 항목을 코드 인스펙션, 단위 테스트, SQL 검증, SonarQube/Checkstyle/PMD/JUnit/MyBatis SQL 검증으로 정의하고 있으며, 품질 게이트를 개발 착수 전 교육, 코드 리뷰, 선도개발 검증, 운영 전환 점검 기준으로 함께 사용해야 한다고 정리하고 있다.

| --- | --- | --- |
| Gate 1 | 로컬 개발 검증 | 컴파일 성공, 단위 테스트 성공, 주요 로직 자체 검증 완료 |
| Gate 2 | 코드 리뷰 | 명명규칙, 계층 책임, 예외, 로그, 보안, SQL 기준 중대 위반 0건 |
| Gate 3 | 정적 분석 | Blocker / Critical 이슈 0건 |
| Gate 4 | 단위 테스트 | 핵심 Service, Rule, DAO 단위 테스트 성공 |
| Gate 5 | SQL 검증 | Mapper SQL 문법, 실행계획, 인덱스, Full Scan 여부 확인 |
| Gate 6 | 통합 테스트 | 표준 전문, Header, GUID, 세션, 권한, 오류코드 시나리오 통과 |
| Gate 7 | 보안 테스트 | 인증, 권한, 마스킹, SQL Injection, 민감정보 노출 검증 통과 |
| Gate 8 | 성능 테스트 | p95 응답시간, 오류율, Thread, Pool, GC, SQL Time 기준 충족 |
| Gate 9 | 배포 검증 | Profile, 환경변수, Health Check, Rollback, Smoke Test 성공 |

NSIGHT CI/CD 구조도 GitLab → Merge Request → GitLab Runner → Gradle Build → Unit Test / Code Inspection → WAR 생성 → 배포 승인 → Tomcat 배포 → Health Check → Apache/L4 트래픽 복귀 → OM 배포이력 기록 흐름으로 정의되어 있으므로, 품질 게이트는 CI/CD 파이프라인 안에 포함되어야 한다.

### 62.4 Gate 1. 로컬 개발 품질 기준

개발자는 소스를 Push하기 전에 반드시 로컬 품질 기준을 만족해야 한다.
| 점검 항목 | 기준 | 반려 조건 |
| --- | --- | --- |
| Compile | gradle clean build 또는 모듈별 build 성공 | 컴파일 오류 |
| 단위 테스트 | 작성한 JUnit 테스트 성공 | 테스트 실패 |
| 애플리케이션 기동 | local profile 기동 성공 | Spring Context 기동 실패 |
| ServiceId 등록 | 신규 거래는 ServiceId 등록 확인 | Dispatcher에서 Handler 미식별 |
| DTO 변환 | StandardRequest Body → 업무 DTO 변환 명확 | Map 직접 남용 |
| 예외 처리 | BusinessException / SystemException 기준 사용 | catch(Exception) 후 무시 |
| 로그 | GUID, TraceId, ServiceId 기준 로그 확인 | 민감정보 평문 로그 |
| SQL | Mapper XML 문법 오류 없음 | SQL 실행 오류 |

### 62.5 Gate 2. 코드 리뷰 품질 기준

코드 리뷰는 단순 오타 검사가 아니라 운영 가능한 코드인지 확인하는 절차이다.

| 리뷰 영역 | 확인 기준 | 계층 책임 |
| --- | --- | --- |
| Handler, Facade, Service, Rule, DAO/Mapper 책임이 섞이지 않았는가 | 명명 규칙 | 업무코드, ServiceId, Class, Method, Mapper, SQL ID가 표준을 따르는가 |
| 전문 처리 | Header, Body, Result 구조가 표준 전문 구조와 맞는가 | 트랜잭션 |

Facade 또는 Service 상위 거래 단위에서 트랜잭션 경계를 잡았는가
예외 처리
업무 예외와 시스템 예외를 구분했는가

| 오류코드 | 표준 오류코드를 사용했는가 | 로그 |
| --- | --- | --- |
| GUID, TraceId, ServiceId, transactionCode가 추적 가능한가 | 보안 | 개인정보, 계좌번호, 고객번호, 토큰이 평문 노출되지 않는가 |
| SQL | SELECT *, 동적 SQL 남용, 조건 없는 대량조회가 없는가 | 테스트 |

정상, 오류, 권한 실패, 입력값 오류 테스트가 있는가

### 62.6 Gate 3. 정적 분석 품질 기준

정적 분석은 CI/CD 파이프라인에서 자동 수행한다.
| 항목 | 기준 |
| --- | --- |
| SonarQube Blocker | 0건 |
| SonarQube Critical | 0건 |
| 보안 취약점 | 신규 High 이상 0건 |
| 중복 코드 | 신규 중복 과다 발생 시 수정 또는 사유 등록 |
| Checkstyle | 프로젝트 표준 위반 0건 또는 승인된 예외만 허용 |
| PMD | 명백한 버그 가능성 항목 0건 |
| Secret Scan | Password, Token, API Key, DB 계정정보 노출 0건 |
| Dependency Scan | 운영 반영 전 취약 라이브러리 확인 및 조치 |

정적 분석 실패는 개발자 PC에서는 경고일 수 있지만, Merge Request 단계에서는 품질 게이트 실패로 처리한다.

### 62.8 Gate 5. SQL / MyBatis 품질 기준

NSIGHT에서는 SQL 품질이 곧 운영 품질이다.SQL 검증은 반드시 MyBatis Mapper 기준으로 수행한다.
| 점검 항목 | 기준 | Mapper 위치 |
| --- | --- | --- |
| 업무별 Mapper XML 위치 표준 준수 | SQL ID | 업무코드 + 기능 + 행위 기준 명명 |
| SQL 문법 | local/dev DB 기준 실행 가능 | Result Mapping |
| DTO 필드와 컬럼 매핑 오류 없음 | Full Scan | 온라인 주요 SQL Full Scan 금지 |
| Index | 조회 조건에 적절한 인덱스 확인 | Paging |
| 목록 조회는 pageSize 제한 | 대량조회 | 다운로드 또는 배치성 거래로 분리 |
| Timeout | Query Timeout 기준 적용 | 동적 SQL |
| 조건 누락으로 전체 조회되지 않도록 방어 | SQL Injection | ${} 직접 사용 금지, #{} 사용 원칙 |

온라인 주요 SQL은 실행계획을 확인하고, 대량 조회 가능성이 있는 SQL은 운영 반영 전에 DBA 또는 성능 담당자 검토를 받는다.

### 62.9 Gate 6. TCF 통합 테스트 품질 기준

TCF 통합 테스트는 업무 기능만 검증하지 않는다.전문, Header, 세션, 권한, 거래통제, Timeout, 거래로그, 오류코드까지 함께 검증한다.
전처리·후처리 요구사항에서도 모든 온라인 거래는 표준 전문으로 수신되고, Header 검증, GUID/Trace 생성, 세션 검증, 인증/권한 확인, 입력값 검증, 중복요청 확인, 거래 시작 로그, PROCESSING 저장을 수행해야 하며, 후처리에서는 오류코드 매핑, 응답 전문 조립, 마스킹, 거래 종료 로그, 감사 로그, 성능 메트릭을 수행해야 한다고 정의되어 있다.
| 검증 항목 | 정상 기준 | Endpoint |
| --- | --- | --- |
| POST /{businessCode}/online 정상 호출 | ServiceId | 등록된 ServiceId만 실행 |
| Header | 필수 Header 누락 시 표준 오류 응답 | GUID |
| 요청~응답~로그까지 동일 GUID 유지 | 세션 | 유효 세션만 업무 실행 |
| 권한 | 권한 없는 사용자는 차단 | 거래통제 |
| 미등록 거래는 실행 전 차단 | Timeout | 초과 거래는 표준 Timeout 오류 처리 |
| 오류코드 | 업무 오류와 시스템 오류가 표준 오류코드로 변환 | 마스킹 |
| 권한에 따라 민감정보 마스킹 | 거래로그 | PROCESSING → SUCCESS / FAIL / TIMEOUT 상태 갱신 |

감사로그
고객정보 조회, 다운로드, 권한 위반 기록

### 62.12 Gate 9. 배포 품질 기준

배포 품질 게이트는 운영 반영 직전 마지막 통제 지점이다.

| Apache/L4 | 배포 중 트래픽 제외 및 복귀 절차 확인 | 로그 |

| 품질 담당 | 품질 게이트 결과 확인, 미준수 항목 관리 |
| 운영 담당 | 배포 전 설정, Health Check, Rollback 계획 확인 |
| PMO | 품질 게이트 통과 여부, 예외 승인 이력 관리 |

### 62.15 개발 완료 전 품질 체크리스트

| 영역 | 점검 질문 | 확인 |
| --- | --- | --- |
| ServiceId | 신규 ServiceId가 표준 명명규칙에 맞는가 | □ |
| Handler | Handler가 업무 로직을 직접 처리하지 않는가 | □ |
| Facade | 트랜잭션 경계가 명확한가 | □ |
| Service / Rule | 업무 흐름과 규칙이 분리되어 있는가 | □ |
| DAO / Mapper | DAO가 Mapper 호출만 담당하는가 | □ |
| SQL | SQL ID, 실행계획, 인덱스가 검증되었는가 | □ |
| 예외 | 표준 예외와 오류코드를 사용하는가 | □ |
| 로그 | GUID, TraceId, ServiceId로 추적 가능한가 | □ |
| 보안 | 민감정보가 응답과 로그에 노출되지 않는가 | □ |
| 세션 | 유효 세션과 권한 검증이 적용되는가 | □ |
| 거래통제 | 미등록 거래가 차단되는가 | □ |
| Timeout | 거래/SQL/연계 Timeout 기준이 적용되는가 | □ |
| 테스트 | 정상/오류/권한실패/입력값오류 테스트가 있는가 | □ |
| 배포 | Health Check와 Smoke Test가 정의되어 있는가 | □ |

### 62.16 마무리말

품질 게이트는 개발자를 통제하기 위한 절차가 아니라, NSIGHT TCF Framework를 운영 가능한 수준으로 유지하기 위한 최소 안전장치이다.
개발자는 기능이 정상 동작하는지만 확인해서는 안 된다.NSIGHT에서는 다음 조건을 모두 만족해야 개발 완료로 본다.
컴파일된다
+ 테스트된다
+ 표준 구조를 지킨다
+ 보안 기준을 통과한다
+ SQL이 검증된다
+ 거래로그로 추적된다
+ 배포 후 Health Check가 가능하다
+ 장애 시 Rollback 가능하다

따라서 품질 게이트는 개발 완료 기준, Merge Request 승인 기준, 배포 승인 기준, 운영 전환 기준으로 함께 사용해야 한다.
