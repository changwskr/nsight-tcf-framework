# 30. 페이징 처리 기준

> **NSIGHT TCF 개발 Manual** · 원본: [`znsight-guide`](../znsight-guide/) · 갱신: 2026-07-05

## 30. 페이징 처리 기준

### 30.1 도입 전 안내말

페이징은 목록 데이터를 한 번에 모두 조회하지 않고, 필요한 범위만 잘라서 조회하는 처리 방식이다.
NSIGHT TCF Framework에서 페이징은 TCF Core가 직접 관리하지 않는다.TCF Core와 Dispatcher는 전문 수신, Header 검증, 세션, 권한, 거래통제, Timeout, Handler 호출을 담당하고, 페이징은 각 업무 WAR 내부에서 처리한다. 기존 페이징 설계 기준에서도 TCF Core와 Dispatcher는 페이징을 관리하지 않고, 각 업무 WAR 내부의 Rule → Service → DAO → MyBatis 흐름에서 처리한다고 정리되어 있다.
기본 흐름은 다음과 같다.
```text
화면
  pageNo, pageSize, 검색조건 전달
        ↓
Handler
  요청 Body 전달
        ↓
Facade
  ReadOnly Transaction 경계
        ↓
Service
  Rule 호출
  DAO search/count 호출
  응답 조립
        ↓
Rule
  pageNo/pageSize 검증
  offset 계산
        ↓
DAO
  Mapper 호출
        ↓
MyBatis XML
  Count SQL
  List SQL
  OFFSET/FETCH
        ↓
DB
  필요한 범위만 조회
```

핵심 문장은 다음이다.
TCF는 페이징을 모른다.페이징은 업무 서비스 안에서 처리하고, MyBatis SQL에서 DB 레벨 페이징을 수행한다.

### 30.2 페이징 처리 결론

| 구분 | 기준 |
| --- | --- |
| 페이징 관리 주체 | 각 업무팀 / 각 업무 WAR |
| TCF 역할 | 페이징 미관여 |
| 처리 계층 | Rule → Service → DAO → MyBatis |
| 요청값 | pageNo, pageSize, 검색조건 |
| 서버 계산값 | offset |
| 클라이언트 offset 전달 | 금지 |
| SQL 방식 | DB 레벨 페이징 |
| 기본 방식 | Offset Paging |
| 목록 SQL | search* 또는 select*List |
| Count SQL | count* 별도 작성 |
| 정렬 | 고정 ORDER BY 원칙 |
| 응답값 | list, pageNo, pageSize, totalCount, totalPage |
| 대량 다운로드 | 일반 목록 조회와 분리 |
| 공통 Page 정책 테이블 | 만들지 않음 |
| OM 페이징 관리화면 | 만들지 않음 |

### 30.3 적용 범위

| 구분 | 적용 대상 | 업무 WAR |
| --- | --- | --- |
| sv-service, ic-service, (미포함·확장 예정), mg-service, om-service 등 | 거래 유형 | 목록 조회, 검색 결과 조회, 로그 조회, 운영관리 목록 |
| 대표 ServiceId | SV.Sample.inquiry, SV.Customer.inquiry, OM.TransactionLog.inquiry | Mapper |
| 업무별 Mapper XML | DB | RDW, ADW, OMDB, LOGDB |

제외 대상
단건 조회, 저장/수정/삭제 거래, 파일 다운로드 본문
페이징은 목록 조회 거래에만 적용한다.단건 조회는 페이징 대상이 아니며, 대량 다운로드는 목록 조회와 별도 ServiceId로 분리한다.

### 30.4 페이징 기본 정책

| 항목 | 기준 |
| --- | --- |
| pageNo 기본값 | 1 |
| pageSize 기본값 | 100 |
| 일반 목록 최대 pageSize | 500 |
| 로그·대량 이력 목록 최대 pageSize | 1000 |
| offset 계산 | 서버 Rule에서 계산 |
| Count SQL | 목록 SQL과 동일 WHERE 사용 |
| 정렬 | SQL에서 고정 ORDER BY |
| 대량 다운로드 | 별도 ServiceId |
| Java 메모리 페이징 | 금지 |
| DB 레벨 페이징 | 필수 |

기존 페이징 설계에서도 pageNo 기본값은 1, pageSize 기본값은 100, 일반 목록 상한은 500, offset은 (pageNo - 1) × pageSize로 Rule에서 계산하며, 클라이언트 offset은 수신하지 않는다고 정의되어 있다.

### 30.5 페이징 계산 기준

offset 계산식은 다음과 같다.
offset = (pageNo - 1) × pageSize

예시는 다음과 같다.

| pageNo | pageSize | offset |
| --- | --- | --- |
| 의미 | 1 | 100 |
| 0 | 처음부터 100건 조회 | 2 |
| 100 | 100 | 100건 건너뛰고 다음 100건 조회 |
| 3 | 100 | 200 |
| 200건 건너뛰고 다음 100건 조회 | 2 | 10 |
화면은 pageNo, pageSize만 보낸다.offset은 서버가 계산해야 한다.

| 10 | 10건 건너뛰고 다음 10건 조회 |

### 30.6 요청 전문 기준

화면 또는 WebTopSuite는 업무 검색조건과 함께 pageNo, pageSize를 전달한다.
```json
{
  "header": {
    "businessCode": "SV",
    "serviceId": "SV.Customer.inquiry",
    "transactionCode": "SV-INQ-0002",
    "user": "user001",
    "channelId": "WEB",
    "branch": "001"
  },
  "body": {
    "customerName": "홍길동",
    "customerGrade": "A",
    "pageNo": 1,
    "pageSize": 100
  }
}
```

| 필드 | 필수 | 기본값 |
| --- | --- | --- |
| 설명 | pageNo | N |
| 1 | 조회할 페이지 번호 | pageSize |
| N | 100 | 한 페이지 조회 건수 |
| 검색조건 | N | 없음 |
| 업무별 검색조건 | offset | 금지 |
| 서버 계산 | 클라이언트에서 보내지 않음 | sortColumn |
| 원칙 금지 | 고정 정렬 | 동적 정렬 필요 시 whitelist 검증 |

### 30.7 응답 전문 기준

목록 조회 응답은 다음 항목을 포함한다.
{
  "list": [
    {
      "customerNo": "C0000001",
      "customerName": "홍길동",
      "customerGrade": "A",
      "branchCode": "001"
    }
  ],
  "pageNo": 1,
  "pageSize": 100,
  "totalCount": 352,
  "totalPage": 4
}

| 필드 | 설명 |
| --- | --- |
| list | 현재 페이지 목록 |
| pageNo | 현재 페이지 번호 |
| pageSize | 한 페이지 조회 건수 |
| totalCount | 전체 검색 결과 건수 |
| totalPage | 전체 페이지 수 |
| 주의할 점은 다음이다. | list.size() ≠ totalCount |

list.size()는 현재 페이지에 담긴 건수이고, totalCount는 검색조건에 해당하는 전체 건수다.
totalPage 계산식은 다음과 같다.
int totalPage = totalCount == 0
        ? 0
        : (totalCount + pageSize - 1) / pageSize;

### 30.8 계층별 책임 기준

| 계층 | 페이징 책임 | 금지 사항 |
| --- | --- | --- |
| TCF Core | 없음 | pageNo/pageSize 검증 금지 |
| Dispatcher | 없음 | ServiceId 외 페이징 판단 금지 |
| Handler | 요청 Body 전달 | offset 계산 금지 |
| Facade | ReadOnly Transaction 경계 | SQL 호출 금지 |
| Service | Rule 호출, DAO search/count 호출, 응답 조립 | SQL 문자열 조립 금지 |
| Rule | pageNo/pageSize 검증, 최대값 제한, offset 계산 | DB 접근 금지 |
| DAO | Mapper 호출 | 업무 판단 금지 |
| Mapper XML | Count SQL, List SQL, DB 페이징 | Java 페이징 금지 |
| DB | 필요한 범위만 조회 | 전체 조회 후 AP 전달 금지 |

기존 설계에서도 Handler에는 페이징 로직이 없고, Facade는 트랜잭션 경계, Rule은 검증·페이징 정규화, Service는 DAO 호출과 응답 조립, Mapper XML은 OFFSET … FETCH NEXT …와 COUNT(1)을 담당한다고 정리되어 있다.

### 30.9 Rule 작성 기준

Rule은 페이징에서 가장 중요한 계층이다.
Rule의 책임은 다음이다.
| 항목 | 기준 |
| --- | --- |
| pageNo 검증 | 1보다 작으면 1 |
| pageSize 검증 | 1보다 작으면 기본값 |
| pageSize 상한 | 일반 목록 500 |
| offset 계산 | (pageNo - 1) × pageSize |
| 검색조건 정리 | 공백 제거, 기본값 보정 |
| 정렬값 검증 | 동적 정렬 시 whitelist 검증 |

예시는 다음과 같다.
```java
@Component
public class SvCustomerInquiryRule {
    private static final int DEFAULT_PAGE_NO = 1;
    private static final int DEFAULT_PAGE_SIZE = 100;
    private static final int MAX_PAGE_SIZE = 500;
    public SvCustomerSearchCriteria buildCriteria(SvCustomerInquiryRequest request) {
        int pageNo = request.getPageNo() == null ? DEFAULT_PAGE_NO : request.getPageNo();
        int pageSize = request.getPageSize() == null ? DEFAULT_PAGE_SIZE : request.getPageSize();
        if (pageNo < 1) {
            pageNo = DEFAULT_PAGE_NO;
        }
        if (pageSize < 1) {
            pageSize = DEFAULT_PAGE_SIZE;
        }
        if (pageSize > MAX_PAGE_SIZE) {
            pageSize = MAX_PAGE_SIZE;
        }
        int offset = (pageNo - 1) * pageSize;
        return SvCustomerSearchCriteria.builder()
                .customerName(trim(request.getCustomerName()))
                .customerGrade(trim(request.getCustomerGrade()))
                .pageNo(pageNo)
                .pageSize(pageSize)
                .offset(offset)
                .build();
    }
    private String trim(String value) {
        return value == null ? null : value.trim();
    }
}
```

### 30.10 Service 작성 기준

Service는 목록 조회와 전체 건수 조회를 조합한다.
```java
@Service
@RequiredArgsConstructor
public class SvCustomerInquiryService {
    private final SvCustomerInquiryRule rule;
    private final SvCustomerDao dao;
    public SvCustomerInquiryResponse inquiry(SvCustomerInquiryRequest request) {
        SvCustomerSearchCriteria criteria = rule.buildCriteria(request);
        List<SvCustomerRow> list = dao.searchCustomers(criteria);
        int totalCount = dao.countCustomers(criteria);
        int totalPage = totalCount == 0
                ? 0
                : (totalCount + criteria.getPageSize() - 1) / criteria.getPageSize();
        return SvCustomerInquiryResponse.builder()
                .list(list)
                .pageNo(criteria.getPageNo())
                .pageSize(criteria.getPageSize())
                .totalCount(totalCount)
                .totalPage(totalPage)
                .build();
    }
}
```

Service 기준은 다음과 같다.
| 항목 | 기준 |
| --- | --- |
| 목록 조회 | dao.search* 호출 |
| 전체 건수 | dao.count* 호출 |
| 응답 조립 | list, pageNo, pageSize, totalCount, totalPage |
| Count 생략 | 화면 요건상 필요 없을 때만 가능 |
| list.size() | totalCount로 사용 금지 |

### 30.11 DAO / Mapper 기준

DAO는 Mapper를 호출한다.
```java
@Repository
@RequiredArgsConstructor
public class SvCustomerDao {
    private final SvCustomerMapper mapper;
    public List<SvCustomerRow> searchCustomers(SvCustomerSearchCriteria criteria) {
        return mapper.searchCustomers(criteria);
    }
    public int countCustomers(SvCustomerSearchCriteria criteria) {
        return mapper.countCustomers(criteria);
    }
}
```

Mapper Interface는 다음과 같이 작성한다.
```java
@Mapper
public interface SvCustomerMapper {
    List<SvCustomerRow> searchCustomers(SvCustomerSearchCriteria criteria);
    int countCustomers(SvCustomerSearchCriteria criteria);
}
```

### 30.12 MyBatis XML 작성 기준

검색조건은 목록 SQL과 Count SQL이 동일하게 사용해야 한다.
<sql id="customerSearchWhere">
    FROM RDW_CUSTOMER_SUMMARY C
    WHERE 1 = 1
    <if test="customerName != null and customerName != ''">
        AND C.CUSTOMER_NAME LIKE '%' || #{customerName} || '%'
    </if>
    <if test="customerGrade != null and customerGrade != ''">
        AND C.CUSTOMER_GRADE = #{customerGrade}
    </if>
</sql>

목록 조회 SQL은 다음과 같다.
```xml
<select id="searchCustomers"
        parameterType="com.nh.nsight.marketing.sv.dto.SvCustomerSearchCriteria"
        resultMap="SvCustomerRowMap">
    /* SQL_ID: SV.Customer.searchCustomers */
    SELECT
          C.CUSTOMER_NO
        , C.CUSTOMER_NAME
        , C.CUSTOMER_GRADE
        , C.BRANCH_CODE
        , C.TOTAL_BALANCE
    <include refid="customerSearchWhere"/>
    ORDER BY C.CUSTOMER_NO
    OFFSET #{offset} ROWS FETCH NEXT #{pageSize} ROWS ONLY
</select>
```

전체 건수 조회 SQL은 다음과 같다.
<select id="countCustomers"
        parameterType="com.nh.nsight.marketing.sv.dto.SvCustomerSearchCriteria"
        resultType="int">
    /* SQL_ID: SV.Customer.countCustomers */
    SELECT COUNT(1)
    <include refid="customerSearchWhere"/>
</select>

SQL 기준은 다음과 같다.
| 항목 | 기준 |
| --- | --- |
| search*와 count* | 동일 WHERE 사용 |
| 목록 SQL | 반드시 ORDER BY 포함 |
| DB 페이징 | OFFSET/FETCH 또는 DB별 표준 방식 |
| 컬럼 | SELECT * 금지 |
| 정렬 | 고정 정렬 권장 |
| Count | 필요 시 별도 SQL |
| 대량 조회 | 온라인 목록 SQL로 처리 금지 |

### 30.13 Count SQL 기준

Count SQL은 전체 검색 결과 건수를 계산한다.
| 구분 | 기준 |
| --- | --- |
| 일반 목록 | Count SQL 작성 |
| 로그 목록 | 기간 조건 필수 |
| 대량 테이블 | Count 비용 확인 |
| Count 지연 | 총건수 생략 또는 hasNext 방식 검토 |
| 화면 요건상 불필요 | Count 생략 가능 |
| Count와 List WHERE | 반드시 동일 조건 |
Count SQL이 너무 느린 경우에는 다음 방식 중 하나를 검토한다.

| 대안 | |
| 설명 | Count 생략 |
| totalCount 없이 hasNext만 제공 | 제한 Count |
| 최대 N건까지만 Count | 집계 테이블 |
| 사전 집계 결과 사용 | 검색조건 강화 |
| 기간, 업무코드, 지점코드 필수화 | ADW 분리 |
| 분석성 목록은 ADW로 분리 |  |

### 30.14 정렬 기준

페이징 SQL에는 반드시 ORDER BY가 있어야 한다.
ORDER BY C.CUSTOMER_NO
OFFSET #{offset} ROWS FETCH NEXT #{pageSize} ROWS ONLY

정렬 기준은 다음과 같다.
| 구분 | 기준 |
| --- | --- |
| 기본 정렬 | 업무 Key 또는 생성일시 |
| 로그 조회 | START_TIME DESC, GUID DESC |
| 고객 목록 | CUSTOMER_NO |
| 캠페인 목록 | CAMPAIGN_ID 또는 CREATED_AT DESC |
| 동적 정렬 | 원칙 금지 |
| 동적 정렬 예외 | whitelist 검증 후 제한 허용 |

사용자 입력값을 그대로 ORDER BY ${sortColumn}에 넣으면 SQL Injection 위험이 있으므로 금지한다.

### 30.15 대량 다운로드 분리 기준

대량 다운로드는 일반 목록 조회와 분리한다.
일반 목록 조회
= 화면에 보여줄 100건 또는 500건 조회

대량 다운로드
= 별도 ServiceId
+ 별도 Timeout
+ 감사로그
+ 파일관리
+ 다운로드 이력

| 구분 | 일반 목록 조회 | 대량 다운로드 |
| --- | --- | --- |
| ServiceId | SV.Customer.inquiry | SV.Customer.download |
| 조회 건수 | 제한 | 업무 승인 기준 |
| Timeout | 짧게 | 별도 정책 |
| 응답 | JSON list | 파일 또는 다운로드 ID |
| 감사로그 | 선택 | 필수 |
| 파일관리 | 불필요 | 필요 |
| 처리 방식 | 동기 | 비동기 또는 파일 생성 후 다운로드 권장 |

대량 다운로드를 화면 목록 조회 SQL로 처리하면 Timeout, WAS Heap 증가, DB 부하, Hikari Pool 고갈 위험이 커진다.

### 30.17 금지 규칙

| 금지 항목 | 이유 | 페이징 없는 목록 조회 |
| --- | --- | --- |
| DB와 WAS 부하 증가 | SELECT * | 불필요한 컬럼 조회 |

Java에서 전체 조회 후 subList()

| WAS 메모리 낭비 | 클라이언트 offset 신뢰 | 조작 가능성 |
| --- | --- | --- |
| ORDER BY 없는 페이징 | 페이지 결과 흔들림 | search와 count WHERE 불일치 |

| 목록과 총건수 불일치 | 대량 다운로드를 목록 조회로 처리 | Timeout 및 장애 위험 |
| --- | --- | --- |
| 동적 정렬 컬럼 직접 사용 | SQL Injection 위험 | Count SQL을 무조건 실행 |
| 대량 테이블 성능 저하 | pageSize 무제한 허용 | DB/WAS 자원 고갈 |

### 30.18 개발 체크리스트

| 점검 항목 | 확인 |
| --- | --- |
| 목록 조회 요청에 pageNo, pageSize가 있는가 | □ |
| pageNo 기본값을 1로 보정하는가 | □ |
| pageSize 기본값을 100으로 보정하는가 | □ |
| pageSize 최대값을 제한하는가 | □ |
| offset을 서버 Rule에서 계산하는가 | □ |
| 클라이언트가 offset을 보내지 않는가 | □ |
| Handler에 페이징 로직이 없는가 | □ |
| Facade가 readOnly Transaction을 사용하는가 | □ |
| Service가 search*와 count*를 분리 호출하는가 | □ |
| Mapper XML에서 search와 count가 동일 WHERE를 사용하는가 | □ |
| 목록 SQL에 고정 ORDER BY가 있는가 | □ |
| 목록 SQL에 DB 페이징이 적용되어 있는가 | □ |
| SELECT *를 사용하지 않았는가 | □ |
| list.size()를 totalCount로 사용하지 않는가 | □ |
| 응답에 list, pageNo, pageSize, totalCount, totalPage가 있는가 | □ |
| 대량 다운로드가 별도 ServiceId로 분리되어 있는가 | □ |
| 주요 SQL 실행계획을 확인했는가 | □ |

### 30.19 마무리말

페이징 처리 기준의 핵심은 복잡한 공통 프레임워크를 만드는 것이 아니다.
NSIGHT에서는 다음처럼 단순하게 정리한다.
TCF
= 거래가 정상 요청인지 확인하고 Handler를 찾아주는 역할

업무 프로그램
= pageNo, pageSize, offset, list, totalCount, totalPage 처리

MyBatis
= 실제 DB 페이징 수행

즉, 페이징은 업무팀이 자기 목록 조회 안에서 책임지고 처리한다.TCF는 페이징 정책 테이블, 공통 Page DTO, OM 페이징 관리화면을 만들지 않는다.

### 30.20 시사점

페이징은 단순 화면 편의 기능이 아니라 DB, WAS, HikariCP, Tomcat Thread를 보호하는 성능 통제 기준이다.
페이징 없는 목록 조회는 처음에는 정상처럼 보이지만, 데이터가 증가하면 다음 장애로 이어진다.
페이징 없는 목록 조회
```text
  ↓
```

DB 대량 Scan
```text
  ↓
```

대량 ResultSet 생성
```text
  ↓
```

Hikari Connection 장시간 점유
```text
  ↓
```

Tomcat Thread 대기
```text
  ↓
```

응답 지연 / Timeout

따라서 업무 개발자는 목록 조회를 만들 때 반드시 세 가지를 확인해야 한다.
첫째, pageNo/pageSize를 받는가?
둘째, DB에서 필요한 범위만 조회하는가?
셋째, 대량 다운로드를 별도 거래로 분리했는가?
