## 82.1 한 줄 요약

**Prometheus 없이** 각 업무 WAR의 Thread·JVM·DB Pool·실행 중 ServiceId·SQL 상태를 수집하고, **tcf-om**이 통합 조회·원인 판정을 수행하는 **단순 런타임 진단** 기능이다.

운영자는 OM Admin **`/om/admin/runtime-diagnostics.html`** 한 화면에서 “지금 어디가 병목인지”를 확인한다.

---

## 82.2 설계 배경

### 82.2.1 해결하려는 질문

별도 APM·모니터링 제품 없이, 운영 중 아래 질문에 답한다.

| 질문 | 진단 근거 |
| --- | --- |
| 지금 거래가 느린가? | Active Transaction, Slow Transaction |
| Tomcat Thread가 부족한가? | Thread busy ratio, Active TX |
| JVM CPU·GC·Heap 문제인가? | JVM 스냅샷 |
| DB Connection이 부족한가? | HikariCP Active / Idle / Pending |
| 특정 업무 WAR가 자원을 많이 쓰는가? | 업무별 Active TX 점유율 |
| 어떤 ServiceId·SQL이 원인인가? | ActiveTransactionRegistry, Slow SQL |

### 82.2.2 설계 원칙

1. **현재 상태 중심** — 시계열 DB·Prometheus 없이 “지금” 스냅샷을 우선 표시한다.
2. **업무 영향 최소** — Monitor 오류는 로그만 남기고, 업무 거래는 계속 처리한다.
3. **JVM 공유 vs WAR 구분** — 동일 Tomcat의 CPU·Heap·GC는 **공유 JVM** 값이며, WAR별로는 **실행 중 거래·Pool·SQL**로 영향을 추정한다.
4. **SQL·개인정보 미노출** — SQL 문장·Parameter·Body는 표시하지 않고 **Mapper ID·SQL ID·실행 시간**만 노출한다.
5. **내부 API** — `/internal/runtime/*`는 운영·관리망에서만 접근한다.

## 82.3 전체 아키텍처

```text
[업무 WAR: IC / PC / MS / SV / PD / EB / EP / SS / MG / OM]
  └─ TcfRuntimeMonitor (tcf-web)
       ├─ Thread / JVM / DB Pool
       ├─ ActiveTransactionRegistry (tcf-core)
       └─ SlowSqlTracker / SlowTransactionTracker
              │
              │  GET /internal/runtime/status
              │  GET /internal/runtime/active-transactions
              │  GET /internal/runtime/slow-transactions
              │  GET /internal/runtime/slow-sql
              │  GET /internal/runtime/threads
              ▼
[tcf-om]
  RuntimeStatusCollector  →  각 WAR HTTP 병렬 수집
  RuntimeCauseAnalyzer    →  원인 코드·메시지 통합
  OmRuntimeService        →  OM.Runtime.inquiry 응답
              │
              ▼
[tcf-ui]  /om/admin/runtime-diagnostics.html
```

### 82.3.1 모듈별 책임

| 구성요소 | 모듈 | 역할 |
| --- | --- | --- |
| `ActiveTransactionRegistry` | tcf-core | 실행 중 TCF 거래 등록·해제 |
| `TcfRuntimeTransactionHook` | tcf-core | 거래 단계(STF~SQL) 추적 |
| `TcfSqlMonitorInterceptor` | tcf-web | MyBatis SQL 실행 시간·Mapper ID 수집 |
| `TcfRuntimeMonitor` | tcf-web | JVM·Thread·Pool·스냅샷 조립 |
| `TcfRuntimeDiagnosticsController` | tcf-web | `/internal/runtime/*` REST API |
| `RuntimeStatusCollector` | tcf-om | 등록된 WAR 병렬 HTTP 수집 |
| `RuntimeCauseAnalyzer` | tcf-om | 업무 간 원인·점유율 통합 판정 |
| `OmRuntimeHandler/Service` | tcf-om | `OM.Runtime.inquiry` 거래 처리 |
| `runtime-diagnostics.html` | tcf-ui | 운영 단일 화면 |

## 82.4 활성화 조건

### 82.4.1 업무 WAR 설정

각 업무 WAR·tcf-om·tcf-batch의 `application.yml`에 다음을 설정한다.

```yaml
nsight:
  tcf:
    runtime-monitor-enabled: true
    runtime-slow-sql-threshold-ms: 1000   # Slow SQL 기준 (ms)
    runtime:
      business-code: IC   # 업무별 코드 (IC, SV, OM …)
```

`runtime-monitor-enabled: false`이면 `/internal/runtime/*` API가 등록되지 않는다.

### 82.4.2 tcf-om 수집 대상

`nsight.om.runtime-diagnostics.targets`에 **업무 WAR만** 등록한다.

**ztomcat dev 기준 (10개):**

| businessCode | baseUrl (예) |
| --- | --- |
| IC | `http://127.0.0.1:8080/ic` |
| PC | `http://127.0.0.1:8080/pc` |
| MS | `http://127.0.0.1:8080/ms` |
| SV | `http://127.0.0.1:8080/sv` |
| PD | `http://127.0.0.1:8080/pd` |
| EB | `http://127.0.0.1:8080/eb` |
| EP | `http://127.0.0.1:8080/ep` |
| SS | `http://127.0.0.1:8080/ss` |
| MG | `http://127.0.0.1:8080/mg` |
| OM | `http://127.0.0.1:8080/om` |

**모니터링 대상에서 제외한 모듈**

| 모듈 | 제외 이유 |
| --- | --- |
| tcf-ui | `/internal/runtime/*` 미제공 (정적·Relay UI) |
| tcf-uj | 동일 |
| tcf-jwt | 인증 전용, 런타임 진단 API 없음 |
| tcf-gateway | Gateway 라우팅 전용 |
| tcf-batch | 배치 전용 (대시보드 AP 수집은 별도) |

설정 파일: `tcf-om/src/main/resources/application-dev.yml`, `application-local.yml`

```yaml
nsight:
  om:
    runtime-diagnostics:
      enabled: true
      connect-timeout-ms: 2000
      read-timeout-ms: 8000
      max-parallel-requests: 4
      targets:
        - { businessCode: IC, baseUrl: '${nsight.gateway.base-url}/ic', enabled: true }
        # … (위 10개)
```

Online Timeout·3계층 정책 상세는 [80장](./80-TCF-타임아웃-3계층-예시.md) 참고.

## 82.6 WAR 내부 진단 API

모든 API는 **`GET`** 이고, context-path 아래 `/internal/runtime`에 매핑된다.

| API | 설명 |
| --- | --- |
| `/internal/runtime/status` | Thread·JVM·Pool·요약·Active/Slow 목록 통합 |
| `/internal/runtime/active-transactions` | 실행 중 거래 |
| `/internal/runtime/slow-transactions?limit=50` | 최근 Slow 거래 |
| `/internal/runtime/slow-sql?limit=50` | 최근 Slow SQL |
| `/internal/runtime/threads` | TCF 거래와 연결된 Thread 목록 |

### 82.6.1 status 응답 구조 (요약)

```json
{
  "instance": {
    "hostName": "...",
    "port": 8080,
    "applicationName": "nsight-ic-service",
    "businessCode": "IC",
    "version": "1.0.0-SNAPSHOT"
  },
  "summary": {
    "status": "NORMAL",
    "primaryCauseCode": "NORMAL",
    "message": "현재 주요 병목이 없습니다.",
    "activeTransactionCount": 12,
    "slowTransactionCount": 0,
    "slowSqlCount": 0
  },
  "thread": { "max": 200, "current": 64, "busy": 52, "busyRatio": 26.0, "deadlock": false },
  "jvm": { "processCpuRatio": 4.2, "heapRatio": 31.4, "gcTimeLastMinuteMs": 120 },
  "dbPools": [
    { "poolName": "HikariPool-IC", "maximum": 10, "active": 3, "idle": 7, "pending": 0 }
  ],
  "activeTransactions": [ ... ],
  "slowTransactions": [ ... ],
  "slowSql": [ ... ]
}
```

WAR가 내려가 있거나 API가 없으면 tcf-om 수집 결과에 `reachable: false`, `causeCode: UNKNOWN`이 기록된다.

## 82.7 거래 추적 (Active Transaction)

### 82.7.1 등록 정보

TCF 거래가 STF에 진입하면 `ActiveTransactionRegistry`에 등록된다.

| 필드 | 설명 |
| --- | --- |
| guid / traceId | 거래 추적 ID |
| businessCode / serviceId / transactionCode | Header 기준 |
| threadName / threadId | 처리 Thread |
| currentStep | 현재 처리 단계 |
| currentSqlId | 실행 중 Mapper SQL ID |
| dbWaitStartMillis | Connection 대기 시작 시각 |
| externalSystemCode | 외부 연동 대기 대상 |

### 82.7.2 처리 단계 (currentStep)

| 단계 | 의미 |
| --- | --- |
| `STF` | 공통 전처리 |
| `WAIT_HANDLER` | Handler 대기 |
| `HANDLER` | Handler 처리 |
| `FACADE` | Facade 처리 |
| `SERVICE` | Service 처리 |
| `RULE` | Rule 검증 |
| `WAIT_DB_CONNECTION` | DB Connection 대기 |
| `EXECUTING_SQL` | SQL 실행 |
| `WAIT_EXTERNAL` | 외부 시스템 대기 |
| `ETF` | 후처리 |
| `COMPLETED` | 완료 |

**반드시 `finally`에서 Registry 제거** — 미제거 시 “유령 거래”로 Slow·점유율이 왜곡된다.

## 82.8 원인 판정 (Cause Code)

설계서 **14. 자동 원인판정**과 동일한 우선순위·조건·운영 메시지를 사용한다.

각 WAR의 `TcfRuntimeMonitor`가 1차 판정하고, tcf-om `RuntimeCauseAnalyzer`가 **업무 간 우선순위**로 통합한다.

### 82.8.1 판정 우선순위

1. Deadlock
2. DB Pool 고갈
3. GC 압박
4. CPU 과부하
5. Tomcat Thread 포화
6. Slow SQL
7. 외부 연계 지연
8. 특정 업무·ServiceId 독점
9. 원인 불명
10. 정상 (`NORMAL`)

### 82.8.2 WAR 1차 판정표

| 조건 | Cause Code | 운영 메시지 | status |
| --- | --- | --- | --- |
| Deadlock 존재 | `THREAD_DEADLOCK` | Thread Deadlock이 발견되었습니다. | CRITICAL |
| Active=Max, Idle=0, Pending>0 | `DB_POOL_EXHAUSTED` | DB Connection 대기가 발생했습니다. | CRITICAL |
| Heap≥80%, GC 시간 증가 | `GC_PRESSURE` | GC 증가로 JVM 응답이 지연되고 있습니다. | WARN |
| CPU≥90% | `CPU_OVERLOAD` | JVM CPU가 과부하 상태입니다. | WARN |
| Busy≥85% **AND** Slow TX≥5 (최근 1분) | `THREAD_SATURATION` | Tomcat Thread 부족 또는 장기 거래 점유 상태입니다. | WARN |
| Slow SQL 다수 (최근 1분 ≥3) | `SLOW_SQL` | 특정 Mapper SQL이 장시간 실행 중입니다. | WARN |
| WAIT_EXTERNAL 다수 (≥3) | `EXTERNAL_WAIT` | 외부 시스템 응답을 기다리고 있습니다. | WARN |
| 위 조건 미해당 | `NORMAL` | 현재 주요 병목이 없습니다. | NORMAL |
| API 조회 실패 | `UNKNOWN` | 진단 API 조회 실패 (별도 메시지) | UNKNOWN |

구현: `TcfRuntimeMonitor.buildSummary()`

### 82.8.3 tcf-om 통합 판정표

WAR 1~7번 조건은 각 WAR `summary.primaryCauseCode` 중 **우선순위가 가장 높은 값**을 채택한다.

| 조건 | Cause Code | 운영 메시지 |
| --- | --- | --- |
| 업무 점유율≥60% | `BUSINESS_DOMINANCE` | 특정 업무가 실행 자원을 과다 점유 중입니다. |
| ServiceId 점유율≥40% | `SERVICE_DOMINANCE` | 특정 ServiceId가 자원을 과다 점유 중입니다. |
| 1~8 미해당, 이상 징후 존재 | `UNKNOWN` | 추가 로그와 DB 상태 확인이 필요합니다. |
| 이상 없음 | `NORMAL` | 현재 주요 병목이 없습니다. |

**8번(독점)** 은 WAR 1~7번에 해당하지 않을 때만 적용한다. `BUSINESS_DOMINANCE`가 `SERVICE_DOMINANCE`보다 우선한다.

**9번(원인 불명)** 은 통합 결과가 `NORMAL`이지만 Slow TX·Slow SQL·Busy≥70% 등 **경계 징후**가 있을 때 tcf-om이 설정한다.

구현: `RuntimeCauseAnalyzer.applyDominanceCause()`, `applyUnknownFallback()`

### 82.8.4 화면 표시 영역

| 영역 | 내용 |
| --- | --- |
| 상단 배너 | `overallStatus` — NORMAL / WARN / CRITICAL / UNKNOWN |
| 원인 요약 | `primaryMessage`, `primaryCauseCode` |
| 영향 업무 | `dominantBusinessCode`, `dominantServiceId`, `dominantSqlId` |
| 상태 카드 | Thread busy %, JVM CPU, Heap, DB Active/Pending, Slow TX/SQL, 주요 업무 점유 |
| 원인 판정表 | 업무별 status / causeCode / message |
| 업무 점유 | `businessOwnership` — Active TX · 점유율 |
| 실행 중 거래 | serviceId, elapsedMs, currentStep, sqlId |
| Slow SQL | Mapper ID, sqlId, elapsedMs |

상세 진단 설계(Thread·JVM·DB·SQL·독점)는 [82.18~82.23](#8218-thread-분석-단순-설계) 참고.

## 82.9 JVM 공유 vs WAR별 정보

ztomcat에 여러 WAR가 올라가면 **JVM·Heap·GC·Process CPU는 Tomcat 프로세스 전체** 값이다.

| Tomcat 공유 (WAR별 분리 불가) | WAR별로 구분 가능 |
| --- | --- |
| JVM CPU | 실행 중 Active Transaction |
| Heap / Old Gen / Metaspace | ServiceId별 실행 건수·시간 |
| GC Count / Time | HikariCP Pool (WAR별 DataSource) |
| Live Thread (전체) | Mapper SQL / Slow SQL |
| | WAIT_EXTERNAL 대상 |
| | 업무별 Cause Code |

**WAR별 Heap 사용량을 정확히 표시하지 않는다** — 설계서·구현 모두 동일한 제한이다. 대신 **업무별 거래·Pool·SQL**로 영향 업무를 추정한다.

## 82.10 운영 대시보드 AP 상태와의 차이

| 구분 | AP 상태 (dashboard.html) | 런타임 진단 (runtime-diagnostics.html) |
| --- | --- | --- |
| 수집 주체 | **tcf-batch** (5분 cron) | **tcf-om** (화면 조회 시) |
| 데이터 저장 | `OM_AP_STATUS` 테이블 | 저장 없음 (실시간 스냅샷) |
| 수집 소스 | Actuator `/actuator/metrics/*` | `/internal/runtime/*` |
| 표시 정보 | CPU·Heap·Thread (숫자) | Thread·JVM·Pool·**ServiceId·SQL·원인** |
| tcf-om FAIL 시 | Actuator 연결 실패 (재배포·기동 지연) | `/internal/runtime` 미응답 |

두 화면은 **목적이 다르다**. AP 상태는 “서버가 살아 있는지”, 런타임 진단은 “**왜 느린지**”를 본다.

## 82.11 보안

### 82.11.1 접근 제한

- `/internal/runtime/*` — **일반 브라우저·외부망·업무 사용자 차단**
- 허용: tcf-om 서버, 운영 관리망, 승인된 관리자
- Gateway Route에 `/internal/runtime` **외부 등록 금지**

### 82.11.2 JWT 필터 예외

`tcf-web`의 `TcfJwtAuthenticationFilter`는 아래 URI를 인증 없이 통과시킨다.

- `/actuator/*`
- `/internal/runtime/*`

운영 환경에서는 **IP Allow List·관리자 JWT** 등 추가 통제를 권장한다.

### 82.11.3 미수집·미표시 항목

- SQL Parameter, 요청 Body, Session ID, JWT 원문
- 고객 식별 정보
- GUID는 필요 시 Masking

## 82.12 장애 확인 절차 (운영자)

```text
1. runtime-diagnostics.html 열기
2. overallStatus · primaryMessage 확인
3. 원인 판정表에서 CRITICAL/WARN 업무 식별
4. 상태 카드 — DB Pending, Thread busy, Slow SQL 확인
5. 실행 중 거래 — serviceId · currentStep · sqlId 확인
6. 해당 WAR 로그·DB·외부 연동 추가 조사
```

### 82.12.1 증상별 1차 조치

| primaryCauseCode | 1차 확인 |
| --- | --- |
| `DB_POOL_EXHAUSTED` | 해당 WAR Pool 크기, Slow SQL, DB Lock |
| `THREAD_SATURATION` | Active TX 다수 ServiceId, Pool·외부 대기 |
| `SLOW_SQL` | Mapper ID·실행 시간, DB 실행 계획 (DBA) |
| `EXTERNAL_WAIT` | EAI·HTTP 연동 대상 상태 |
| `GC_PRESSURE` | Heap, GC 로그, 메모리 leak 여부 |
| `CPU_OVERLOAD` | CPU 사용 ServiceId·Thread (JVM 공유) |
| `UNKNOWN` | WAR 기동·`/internal/runtime/status` 직접 호출 |

## 82.13 장애 전파 흐름 (참고)

### DB Pool 고갈

```text
Connection 부족
  → WAIT_DB_CONNECTION 증가
  → DB Pending 증가
  → Tomcat Busy Thread 증가
  → tcf-om: DB_POOL_EXHAUSTED
```

### Slow SQL

```text
Connection 정상 획득
  → EXECUTING_SQL 장시간
  → Pool Active 증가
  → tcf-om: SLOW_SQL
```

## 82.14 제한 사항

설계서·현행 구현 공통 한계:

## 82.18 Thread 분석 단순 설계

> 설계서 7장 · 수집: `TcfRuntimeMonitor.collectThreadStatus()` · 화면: `runtime-diagnostics.html` Thread 카드·Thread 상세表

### 82.18.1 표시 항목

| 항목 | 설명 | API 필드 (`thread`) | 구현 |
| --- | --- | --- | --- |
| Max Thread | Tomcat 최대 Thread | `max` | ○ Tomcat JMX (`TomcatThreadPoolProbe`), 미조회 시 추정 |
| Current Thread | 현재 생성된 Thread | `current` | ○ JMX `currentThreadCount` 또는 `ThreadMXBean` |
| Busy Thread | 요청 처리 중 Thread | `busy` | ○ JMX `currentThreadsBusy`, 미조회 시 Active TX 추정 |
| Busy Ratio | Busy / Max × 100 | `busyRatio` | ○ |
| Active Transaction | 현재 TCF 거래 수 | `activeTransactionCount` | ○ `ActiveTransactionRegistry` |
| Slow Transaction | 기준시간 이상 실행 거래 (최근 1분) | `slowTransactionCount` | ○ `SlowTransactionTracker` |
| Blocked Thread | Java Lock 대기 Thread | `blocked` | ○ `http-*` Thread 중 `BLOCKED` 카운트 |
| Deadlock | 교착상태 여부 | `deadlock` | ○ `findDeadlockedThreads()` |
| Max 출처 | JMX·추정 구분 | `maxSource` | ○ `tomcat-jmx` / `estimated` |

**Tomcat `maxThreads` JMX:** `Catalina:type=ThreadPool,*` / `Tomcat:type=ThreadPool,*`에서 `http` Connector Pool을 조회한다. JMX 미가용 시 Active Transaction + Live Thread로 추정한다.

### 82.18.2 Thread 부족 판단

```text
조건:
  thread.busyRatio ≥ 85%
  AND
  summary.slowTransactionCount ≥ 5   (최근 1분, Slow TX 증가)

판정 (WAR 1차):
  Cause Code : THREAD_SATURATION
  status     : WARN
  메시지     : Tomcat Thread 부족 또는 장기 거래 점유 상태입니다.
```

**주의:** `THREAD_SATURATION`만으로 `server.xml maxThreads` 부족을 **확정하지 않는다**. 반드시 **DB Pool Pending·JVM GC**를 함께 본다.

| 연쇄 패턴 | 실제 1차 원인 후보 |
| --- | --- |
| Pending > 0, Busy ↑ | DB Pool (`DB_POOL_EXHAUSTED`) |
| EXECUTING_SQL 다수, Pending = 0 | Slow SQL (`SLOW_SQL`) |
| WAIT_EXTERNAL 다수 | 외부 연동 (`EXTERNAL_WAIT`) |
| Busy ↑, Pending = 0, Slow TX 적음 | CPU·GC (`CPU_OVERLOAD`, `GC_PRESSURE`) |

구현: `TcfRuntimeMonitor.buildSummary()` — `busyRatio >= 85 && slowTx >= 5`

### 82.18.3 Thread 상세 목록

**“현재 거래가 왜 느린가”를 판단하는 핵심 화면** — TCF Active Transaction과 1:1 매핑된 Thread만 표시한다 (전체 Thread Dump 아님).

| Thread | 업무 | ServiceId | 실행시간 | 현재 단계 | SQL·외부 |
| --- | --- | --- | --- | --- | --- |
| http-nio-8080-exec-135 | MG | MG.Message.sendBulk | 7.2초 | EXECUTING_SQL | insertSendHistory |
| http-nio-8080-exec-204 | SV | SV.Customer.selectSummary | 4.1초 | WAIT_DB_CONNECTION | - |
| http-nio-8080-exec-318 | IC | IC.Credit.selectResult | 6.8초 | WAIT_EXTERNAL | CREDIT-EAI |

| API | 필드 |
| --- | --- |
| `GET /internal/runtime/threads` | `threadName`, `businessCode`, `serviceId`, `elapsedMs`, `currentStep`, `sqlId`, `externalSystemCode` |
| OM 화면 | **Thread 상세**表 + **실행 중 거래**表 (동일 Registry 소스) |

정렬: `elapsedMs` 내림차순 (가장 오래 걸린 거래 우선).

## 82.19 JVM 상태 단순 설계

> 설계서 8장 · 수집: `TcfRuntimeMonitor.collectJvmStatus()` · 화면: JVM CPU·Heap 카드

### 82.19.1 표시 항목

| 항목 | 설명 | API 필드 (`jvm`) | 구현 |
| --- | --- | --- | --- |
| Process CPU | 현재 JVM CPU 사용률 (%) | `processCpuRatio` | ○ `OperatingSystemMXBean.getProcessCpuLoad()` |
| Heap Used | Heap 사용량 (MB) | `heapUsedMb` | ○ |
| Heap Max | 최대 Heap (MB) | `heapMaxMb` | ○ |
| Heap Ratio | Heap 사용률 (%) | `heapRatio` | ○ |
| Old Gen | Old Generation 사용률 (%) | `oldGenRatio` | ○ MemoryPool `"Old"` |
| Metaspace | Metaspace 사용량 (MB) | `metaspaceMb` | ○ |
| GC Count | 최근 측정 구간 GC 횟수 | `gcCountLastMinute` | ○ GC MXBean delta |
| GC Time | 최근 측정 구간 GC 소요 (ms) | `gcTimeLastMinuteMs` | ○ |
| Live Thread | JVM 전체 Thread 수 | `liveThreadCount` | ○ |
| Deadlock | Deadlock 존재 | `deadlock` | ○ thread와 동일 |
| Uptime | JVM 기동 시간 (ms) | `uptimeMs` | ○ `RuntimeMXBean` |

**ztomcat 공유 JVM:** 위 값은 **Tomcat 프로세스 전체**이며 WAR별로 분리되지 않는다.

### 82.19.2 CPU 문제 판단

```text
조건:
  jvm.processCpuRatio ≥ 90%

판정:
  Cause Code : CPU_OVERLOAD
  status     : WARN
  메시지     : JVM CPU가 과부하 상태입니다.
```

**참고:** DB Pending이 있으면 CPU 높음이 Pool 대기의 2차 결과일 수 있다. 우선순위상 `DB_POOL_EXHAUSTED`(2번)가 먼저 판정된다.

**ServiceId별 Thread CPU 표** (설계서 예시):

| ServiceId | 실행 Thread | 누적 Thread CPU | 실행시간 |
| --- | --- | --- | --- |
| MG.Message.sendBulk | 215 | 68% 상당 | 7.2초 |
| SV.Customer.selectSummary | 185 | 12% 상당 | 2.1초 |

| 구분 | 상태 |
| --- | --- |
| 2차 구현 | ○ `TcfRuntimeMonitor.collectServiceCpuUsage()` — Active TX Thread CPU share |
| OM 화면 | **ServiceId CPU 점유**表 — `serviceCpuUsage[]` (`serviceId`, `threadCount`, `cpuSharePct`, `maxElapsedMs`) |

### 82.19.3 GC 문제 판단

```text
조건:
  jvm.heapRatio ≥ 80%
  AND
  jvm.gcTimeLastMinuteMs ≥ 3000

판정:
  Cause Code : GC_PRESSURE
  status     : WARN
```

**운영 메시지 예:**

```text
JVM Heap 사용률 87%
최근 1분 GC 소요시간 8.2초
모든 WAR의 응답 지연 가능성이 있습니다.
```

## 82.20 DB Pool 상태 단순 설계

> 설계서 9장 · 수집: `TcfRuntimeMonitor.collectDbPoolStatus()` · HikariCP MXBean

### 82.20.1 표시 항목

| 항목 | 설명 | API 필드 (`dbPools[]`) | 구현 |
| --- | --- | --- | --- |
| Pool Name | HikariCP 이름 | `poolName` | ○ |
| Maximum | 최대 Connection | `maximum` | ○ |
| Active | 사용 중 Connection | `active` | ○ |
| Idle | 사용 가능 Connection | `idle` | ○ |
| Pending | Connection 대기 Thread | `pending` | ○ `getThreadsAwaitingConnection()` |
| Usage Ratio | Active / Maximum × 100 | `usageRatio` | ○ |
| Connection Timeout | 최근 획득 실패 건수 | `connectionTimeoutCount` | △ 2차 (Hikari metrics) |
| Long Usage | 오래 점유 중 Connection | `longUsageConnections` | △ 2차 |

WAR당 DataSource 1개 기준 **Pool 1행**. OM 통합 화면 카드는 **전 WAR 중 max Active·max Pending**을 표시한다.

### 82.20.2 DB Connection 대기 판단

```text
조건 (Pool 1개 기준):
  active  = maximum
  AND idle    = 0
  AND pending > 0

판정:
  Cause Code : DB_POOL_EXHAUSTED
  status     : CRITICAL
```

**운영 메시지 예:**

```text
MG HikariPool
  Max: 50 / Active: 50 / Idle: 0 / Pending: 48
현재 MG 거래 48건이 DB Connection을 기다리고 있습니다.
```

구현: `TcfRuntimeMonitor.isPoolExhausted()`

### 82.20.3 Connection 대기시간 측정 (거래 단계)

DB 접근 직전·직후에 `ActiveTransactionRegistry` 단계를 기록한다.

```text
DAO / MyBatis 호출 직전  →  WAIT_DB_CONNECTION
Connection 획득 후       →  EXECUTING_SQL
SQL 종료                 →  (다음 단계 또는 COMPLETED)
```

| 상황 | currentStep 패턴 | 의미 |
| --- | --- | --- |
| Connection 획득 **전** 지연 | `WAIT_DB_CONNECTION` 장시간 | **DB Pool 부족** |
| Connection 획득 **후** SQL 지연 | `EXECUTING_SQL` 장시간 | **Slow SQL** 또는 **DB Lock** |

구현: `TcfSqlMonitorInterceptor` — `registry.markDbWait(guid)` → `registry.updateSqlId(guid, sqlId)`

## 82.21 SQL 분석 단순 설계

> 설계서 10장 · 수집: `TcfSqlMonitorInterceptor` + `SlowSqlTracker`

### 82.21.1 MyBatis Interceptor 수집 항목

| 항목 | 설명 | 저장 |
| --- | --- | --- |
| Mapper ID | `MappedStatement.getId()` 전체 | `SlowSqlInfo.mapperId` |
| SQL ID | Mapper ID 마지막 segment | `SlowSqlInfo.sqlId` |
| ServiceId | 현재 `TransactionContext` Header | `SlowSqlInfo.serviceId` |
| 실행 시작·종료 시각 | ms | `startTimeMillis`, `endTimeMillis` |
| 실행시간 | ms | `elapsedMillis` |
| 성공·실패 | boolean | `success` |
| 영향 행 수 | update count | △ 2차 (`Executor.update` 반환값) |

**수집하지 않음:** SQL 원문, Bind Parameter, Result 데이터.

```java
@Intercepts({
    @Signature(type = Executor.class, method = "query", args = { ... }),
    @Signature(type = Executor.class, method = "update", args = { ... })
})
public class TcfSqlMonitorInterceptor implements Interceptor { ... }
```

### 82.21.2 Slow SQL 기준

| 처리 유형 | 설계 기준 | 현행 구현 |
| --- | --- | --- |
| 단건 조회 | 1초 | `nsight.tcf.runtime-slow-sql-threshold-ms: 1000` (공통) |
| 목록 조회 | 2초 | △ ServiceId별 미분리 |
| 등록·변경 | 2초 | △ |
| 대량 처리 | ServiceId별 별도 | △ 2차 — Catalog `TIMEOUT_SEC` 연동 검토 |

1차는 **환경설정 단일 threshold**로 단순화. 2차에서 `OM_SERVICE_CATALOG`·ServiceId별 threshold 연동 가능.

## 82.22 특정 WAR·ServiceId 자원 독점 판단

> 설계서 11장 · 통합: `RuntimeCauseAnalyzer` (tcf-om)

### 82.22.1 판단 가능한 항목

동일 JVM에서 WAR별 Heap은 분리 불가. 아래 **WAR·ServiceId 단위 지표**로 독점을 추정한다.

| 지표 | API / 계산 |
| --- | --- |
| 업무코드별 실행 중 거래 수 | `businessOwnership[].activeTransactions` |
| 업무 Thread 점유율 | `ownershipPct` = 업무 Active TX / 전체 Active TX |
| ServiceId별 실행 중 건수 | `activeTransactions` 집계 |
| 업무별 DB Active·Pending | `findings[].dbPools` |
| 업무별 Slow SQL 수 | `summary.slowSqlCount` (WAR별) |
| 외부 호출 대기 | `currentStep = WAIT_EXTERNAL` 건수 |

### 82.22.2 업무 독점 판단식

```text
업무 Thread 점유율 = 해당 업무 Active Transaction / 전체 Active Transaction × 100

예:
  전체 실행 거래: 700건
  MG 실행 거래: 490건
  MG 점유율: 70%
```

```text
경고 조건 (tcf-om 통합, WAR 1~7 미해당 시):
  특정 업무 ownershipPct ≥ 60%

판정:
  Cause Code : BUSINESS_DOMINANCE
              (설계서 명칭 BUSINESS_RESOURCE_DOMINANCE — 동일 의미)
  메시지     : 특정 업무가 실행 자원을 과다 점유 중입니다.
```

구현: `RuntimeCauseAnalyzer.applyDominanceCause()` — 점유 ≥ 60% 시 `primaryCauseCode` 설정.

### 82.22.3 ServiceId 독점 판단

```text
조건:
  특정 ServiceId 실행 중 건수 / 전체 실행 중 건수 ≥ 40%

예:
  MG.Message.sendBulk 실행 중: 310건 / 전체: 700건 → 44.3%

판정:
  Cause Code : SERVICE_DOMINANCE
  메시지     : 특정 ServiceId가 자원을 과다 점유 중입니다.
```

구현: `RuntimeCauseAnalyzer.applyDominanceCause()` — ≥ 40% ServiceId.

### 82.22.4 OM 통합 화면 매핑

| 설계서 영역 | OM 응답 필드 | HTML |
| --- | --- | --- |
| 업무 점유表 | `businessOwnership[]` | 업무 점유 |
| 주요 업무 카드 | `cards.dominantBusinessCode`, `dominantBusinessOwnershipPct` | 주요 업무 점유 |
| 영향 ServiceId | `dominantServiceId` | 상단 `causeMeta` |
| 영향 SQL | `dominantSqlId` | 상단 `causeMeta` |
| ServiceId CPU | `serviceCpuUsage[]` | **ServiceId CPU 점유**表 |

## 82.17 마무리

런타임 진단은 **“지금 거래가 느린 이유”**를 Prometheus 없이 TCF·OM만으로 답하기 위한 기능이다.

운영자에게 제공해야 할 최종 메시지 형태는 다음과 같다.

> 현재 거래가 느린 이유는 **MG 업무의 `MG.Message.sendBulk` 거래가 `MessageMapper.insertSendHistory` SQL을 장시간 실행**하여 MG DB Pool의 Connection을 모두 점유했기 때문입니다. Tomcat Thread 부족이 1차 원인은 아니며, **DB Connection 대기**에 따른 결과입니다.

상세 Timeout·수집 시간 제약은 [80장 TCF 타임아웃 3계층 예시](./80-TCF-타임아웃-3계층-예시.md)를 함께 참고한다.
