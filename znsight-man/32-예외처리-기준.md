# 32. 예외처리 기준

> **NSIGHT TCF 개발 Manual** · 원본: [`znsight-guide-word`](../znsight-guide-word/) · 갱신: 2026-07-05

## 32. 예외처리 기준

### 32.1 도입 전 안내말

예외처리는 단순히 try-catch를 많이 작성하는 것이 아니다.NSIGHT TCF Framework에서 예외처리는 오류를 표준 오류코드로 변환하고, 표준 응답 전문으로 조립하며, 거래로그·감사로그·모니터링 지표에 남기는 공통 처리 기준이다.
NSIGHT 예외처리 설계 기준에서도 모든 예외는 TCF 공통 후처리에서 표준 오류코드로 변환하고, StandardResponse로 조립하며, 거래로그·감사로그·모니터링 지표에 남기도록 정의되어 있다. 또한 예외처리 위치는 tcf-core의 ETF와 tcf-web의 Global Exception Handler로 분리하고, 업무 예외는 BusinessException, 시스템 예외는 SystemException 또는 미처리 RuntimeException으로 처리하는 구조를 기준으로 한다.
기본 흐름은 다음과 같다.
```text
업무 처리 중 오류 발생
        ↓
BusinessException / SystemException 발생
        ↓
TCF / ETF / GlobalExceptionHandler에서 수신
        ↓
```

표준 오류코드 매핑
```text
        ↓
```

사용자 메시지 / 운영자 메시지 분리
```text
        ↓
StandardResponse 조립
        ↓
거래로그 상태 갱신
        ↓

```

감사로그 / 성능메트릭 / 오류율 기록
```text
        ↓
```

화면에 표준 오류 응답 반환

핵심 문장은 다음이다.
업무 개발자는 임의 문자열로 오류를 반환하지 않는다.모든 오류는 표준 예외와 표준 오류코드로 표현한다.

### 32.2 예외처리 기준 결론

| 구분 | 기준 |
| --- | --- |
| 예외처리 중심 | ETF 후처리 + Global Exception Handler |
| 업무 예외 | BusinessException |
| 시스템 예외 | SystemException |
| 미처리 예외 | RuntimeException → 공통 시스템 오류로 변환 |
| 입력값 오류 | ValidationException 또는 BusinessException |
| 권한 오류 | BusinessException, 감사로그 필수 |
| DB 오류 | SystemException, SQL ID 운영로그 기록 |
| Timeout 오류 | 일반 시스템 오류와 분리 |
| 연계 오류 | 통신 오류, Timeout, 업무 오류로 구분 |
| 응답 구조 | StandardResponse.header + result + body |
| 오류코드 | E-{DOMAIN}-{CATEGORY}-{NNNN} |
| 사용자 메시지 | 기술정보 제거 |
| 운영자 메시지 | GUID, TraceId, ServiceId, SQL ID, Exception Type 포함 |
| 거래로그 | PROCESSING → SUCCESS / FAIL / TIMEOUT / UNKNOWN |
| 보안 원칙 | StackTrace, SQL, DB 정보, 서버 IP, 개인정보 응답 노출 금지 |
| 운영관리 | OM_ERROR_CODE 기준 오류코드 관리 |

### 32.3 예외처리 목표

| 목표 | 설명 |
| --- | --- |
| 표준 응답 보장 | 성공·실패와 관계없이 항상 같은 응답 구조를 반환 |
| 장애 추적성 확보 | GUID, TraceId, ServiceId, SQL ID 기준으로 원인 추적 |
| 사용자 보호 | 사용자에게 기술 오류 상세를 노출하지 않음 |
| 운영자 지원 | 운영자는 오류코드와 거래로그로 원인 파악 |
| 감사 대응 | 권한 위반, 고객정보 조회 실패, 관리자 오류를 감사로그로 남김 |
| 장애 확산 방지 | Timeout, DB 오류, 연계 오류를 구분해 빠르게 차단 |
| 개발 표준화 | 개발자가 임의 응답·임의 오류코드·임의 로그 포맷을 사용하지 못하게 함 |

전처리·후처리 요구사항에서도 표준 응답 전문 생성, 결과코드 매핑, 오류코드 변환, 사용자 메시지와 운영 메시지 분리, 거래 종료 로그, 성능 로그, 감사 로그, Timeout 상태 처리, 모니터링 메트릭 전송, MDC 정리를 후처리 필수 기준으로 정의한다.

### 32.5 계층별 예외처리 책임

| 계층 | 예외처리 책임 | 금지 사항 |
| --- | --- | --- |
| Controller | TCF로 요청 전달, 직접 예외처리 최소화 | 업무별 try-catch 남발 |
| STF | Header, 세션, 인증, 권한, 거래통제 오류 발생 | 업무 로직 오류 처리 |

| Service | 업무 흐름 중 업무 예외 발생 | SQL 오류 상세 응답 |
| Rule | 입력값·업무 규칙 오류 발생 | DB 접근 |
| DAO | Mapper 오류를 상위로 전달 또는 SystemException 변환 | 사용자 메시지 생성 |
| Mapper | SQL 실행 | 예외 직접 처리 불가 |

| ETF | 표준 응답 조립, 오류코드 매핑, 거래로그 갱신 | 업무 로직 수행 |
| GlobalExceptionHandler | TCF 밖 미처리 예외를 표준 응답으로 변환 | StackTrace 응답 노출 |

### 32.6 오류코드 기준

오류코드는 다음 형식을 사용한다.
E-{DOMAIN}-{CATEGORY}-{NNNN}

| 구분 | 의미 | 예시 |
| --- | --- | --- |
| E | Error | E |
| DOMAIN | 오류 발생 영역 | TCF, COM, SV, OM, GW, JWT, UD |
| CATEGORY | 오류 분류 | HDR, VAL, AUTHN, AUTHZ, CTL, DB, IF, TIME, SYS |
| NNNN | 일련번호 | 0001 |

오류코드 설계안에서도 오류코드와 메시지는 TCF 공통 후처리 ETF에서 일원화하고, 운영관리는 OM 오류코드 관리 화면에서 수행하는 구조가 적합하다고 정의한다. 오류코드 형식도 E-{DOMAIN}-{CATEGORY}-{NNNN}을 기준으로 한다.

### 32.8 BusinessException 기준

BusinessException은 업무적으로 예측 가능한 실패에 사용한다.
예시는 다음과 같다.

| 상황 | 예외 | 조회 결과 없음 |
| --- | --- | --- |
| BusinessException | 입력값 오류 | BusinessException |

| 업무 조건 불일치 | BusinessException | 권한 없음 |
| --- | --- | --- |
| BusinessException | 미등록 거래 | BusinessException |
| 다운로드 권한 없음 | BusinessException | 예시 코드: |

```java
public class BusinessException extends RuntimeException {
    private final String errorCode;
    private final String userMessage;
    private final String operatorMessage;
    private final boolean auditRequired;
    private final boolean retryAllowed;
    public BusinessException(
            String errorCode,
            String userMessage,
            String operatorMessage) {
        super(operatorMessage);
        this.errorCode = errorCode;
        this.userMessage = userMessage;
        this.operatorMessage = operatorMessage;
        this.auditRequired = false;
        this.retryAllowed = false;
    }
    public String getErrorCode() {
        return errorCode;
    }
    public String getUserMessage() {
        return userMessage;
    }
    public String getOperatorMessage() {
        return operatorMessage;
    }
    public boolean isAuditRequired() {
        return auditRequired;
    }
    public boolean isRetryAllowed() {
        return retryAllowed;
    }
}
```

Rule에서 사용하는 예시는 다음과 같다.
```java
@Component
public class SvCustomerRule {
    public void validateCustomerNo(String customerNo) {
        if (customerNo == null || customerNo.isBlank()) {
            throw new BusinessException(
                    "E-SV-VAL-0001",
                    "고객번호를 입력해 주십시오.",
                    "customerNo is blank"
            );
        }
        if (customerNo.length() > 20) {
            throw new BusinessException(
                    "E-SV-VAL-0002",
                    "고객번호 형식이 올바르지 않습니다.",
                    "customerNo length exceeds 20"
            );
        }
    }
}
```

### 32.9 SystemException 기준

SystemException은 기술적 실패 또는 시스템 장애성 오류에 사용한다.
예시는 다음과 같다.

| 상황 | 예외 | DB Connection 오류 |
| --- | --- | --- |
| SystemException | SQL Timeout | SystemException |

| 외부 연계 Timeout | SystemException | 파일시스템 오류 |
| --- | --- | --- |
| SystemException | Cache 장애 | SystemException |
| 시스템 설정 오류 | SystemException | 예상하지 못한 RuntimeException |

SystemException으로 변환
예시 코드:
```java
public class SystemException extends RuntimeException {
    private final String errorCode;
    private final String operatorMessage;
    private final String sqlId;
    private final String interfaceId;
    public SystemException(
            String errorCode,
            String operatorMessage,
            Throwable cause) {
        super(operatorMessage, cause);
        this.errorCode = errorCode;
        this.operatorMessage = operatorMessage;
        this.sqlId = null;
        this.interfaceId = null;
    }
    public String getErrorCode() {
        return errorCode;
    }
    public String getOperatorMessage() {
        return operatorMessage;
    }
    public String getSqlId() {
        return sqlId;
    }
    public String getInterfaceId() {
        return interfaceId;
    }
}
```

DAO에서 변환하는 예시는 다음과 같다.
```java
@Repository
@RequiredArgsConstructor
public class SvCustomerDao {
    private final SvCustomerMapper mapper;
    public SvCustomerSummaryRow selectCustomerSummary(SvCustomerQuery query) {
        try {
            return mapper.selectCustomerSummary(query);
        } catch (DataAccessException e) {
            throw new SystemException(
                    "E-COM-DB-0001",
                    "DB 조회 중 오류가 발생했습니다. sqlId=SV.Customer.selectSummary",
                    e
            );
        }
    }
}
```

### 32.11 권한 예외 기준

권한 예외는 일반 업무 예외보다 더 엄격하게 관리한다.
| 항목 | 기준 |
| --- | --- |
| 사용자 메시지 | “접근 권한이 없습니다.” 수준으로 단순화 |
| 운영자 메시지 | userId, branchId, menuId, functionId, serviceId 포함 |
| 감사로그 | 필수 |
| 거래로그 | FAIL |
| 알림 | 반복 위반 또는 중요정보 접근 시 보안 담당 알림 |
| 응답 보안 | 내부 권한 구조 상세 노출 금지 |
| 예시: | throw new BusinessException( |
| "E-TCF-AUTHZ-0001", | "접근 권한이 없습니다.", |
| "userId=U123456 has no data permission for branchId=009999" | ); |

권한 실패는 사용자에게 내부 정보를 노출하지 않아야 하며, 고객정보 조회, 다운로드, 권한 위반은 감사 대상으로 식별되어야 한다.

### 32.12 DB 예외 기준

DB 예외는 사용자에게 SQL이나 테이블명, 서버 정보를 노출하지 않는다.
| 구분 | 사용자 메시지 | 운영자 로그 |
| --- | --- | --- |
| SQL 오류 | 일시적으로 처리할 수 없습니다. | SQL ID, Mapper ID, ErrorCode, SQLException |
| Query Timeout | 조회가 지연되고 있습니다. | SQL ID, Timeout Sec, DB, elapsedTime |
일시적으로 조회가 지연되고 있습니다.

| Connection 오류 | DataSource, Hikari Pool 상태, Exception Type |
| Deadlock | 일시적으로 처리할 수 없습니다. | Lock 정보, Transaction ID, SQL ID |
| Constraint 오류 | 입력값을 확인해 주십시오. | Constraint 명, Key 정보 |
| 예시: | catch (QueryTimeoutException e) { | throw new SystemException( |
| "E-COM-TIME-DB-0001", | "SQL Timeout occurred. sqlId=SV.Customer.selectList, timeout=3s", | e |
| ); | } |  |

주의할 점은 사용자 응답에는 SQL ID도 노출하지 않는 것이다. SQL ID는 운영 로그와 거래로그에 남긴다.

### 32.13 Timeout 예외 기준

Timeout은 일반 시스템 오류와 분리한다.

| Timeout 유형 | 발생 위치 | 오류코드 예시 |
| --- | --- | --- |
| 거래상태 | Online Transaction Timeout | TCF / Facade |

### 32.14 연계 예외 기준

오류 유형
| 예시 | 예외 | 처리 기준 | 통신 오류 |
| --- | --- | --- | --- |
| Connection refused | IntegrationConnectionException | 시스템 예외 | Connect Timeout |
| 대상 연결 지연 | IntegrationTimeoutException | Timeout 예외 | Read Timeout |
| 응답 지연 | IntegrationTimeoutException | Timeout 예외 | 대상 업무 오류 |
| 대상 서비스 FAIL 응답 | IntegrationBusinessException | 업무 예외 | 응답 파싱 오류 |
| JSON 구조 오류 | SystemException | 시스템 예외 | 인증 오류 |
| 대상 인증 실패 | SystemException 또는 BusinessException | 보안 오류 | 예시: |

### 32.15 Global Exception Handler 기준

TCF 파이프라인 밖에서 발생한 예외는 GlobalExceptionHandler에서 표준 응답으로 변환한다.
@RestControllerAdvice
```java
@RequiredArgsConstructor
public class GlobalStandardExceptionHandler {
    private final StandardErrorResponseFactory responseFactory;
    @ExceptionHandler(BusinessException.class)
    public StandardResponse<Void> handleBusinessException(BusinessException e) {
        return responseFactory.fromBusinessException(e);
    }
    @ExceptionHandler(SystemException.class)
    public StandardResponse<Void> handleSystemException(SystemException e) {
        return responseFactory.fromSystemException(e);
    }
    @ExceptionHandler(Exception.class)
    public StandardResponse<Void> handleUnknownException(Exception e) {
        return responseFactory.fromUnknownException(
                new SystemException(
                        "E-COM-SYS-0001",
                        "Unhandled exception occurred.",
                        e
                )
        );
    }
}
```

주의할 점은 @RestControllerAdvice가 있더라도 업무 개발자가 Controller에서 임의 응답을 만들면 안 된다는 것이다.온라인 거래의 기본 예외처리는 TCF / ETF 흐름을 우선한다.

### 32.17 HTTP Status 기준

NSIGHT는 업무 성공·실패 판단을 HTTP Status만으로 하지 않는다.기본적으로 HTTP는 정상 수신되었고, 업무 결과는 표준 응답의 result로 판단한다.

| --- | --- | --- |
| 설명 | 정상 처리 | 200 |
| SUCCESS | 업무 정상 | 업무 실패 |

200
FAIL

입력값 오류
200 또는 400

| FAIL | 내부 표준에 따라 결정 | 인증 오류 |
| --- | --- | --- |

| FAIL | 감사로그 필수 | 시스템 오류 |
| --- | --- | --- |

### 32.18 거래로그 상태 기준

예외 발생 시 거래로그 상태를 반드시 갱신한다.
상황
거래로그 상태
| 설명 | 요청 수신 | PROCESSING | STF 전처리에서 생성 | 정상 처리 |
| --- | --- | --- | --- | --- |
| SUCCESS | ETF 성공 후처리 | 업무 예외 | FAIL | BusinessException |
| 시스템 예외 | FAIL | SystemException | Timeout | TIMEOUT |

### 32.20 예외 발생 위치별 개발 기준

#### 32.20.1 Rule

Rule은 입력값과 업무 규칙 오류를 발생시킨다.
if (request.getPageSize() > 500) {
    throw new BusinessException(
            "E-SV-VAL-0007",
            "조회 건수가 허용 범위를 초과했습니다.",
            "pageSize exceeds max value. pageSize=" + request.getPageSize()
    );
}

#### 32.20.2 Service

Service는 업무 흐름상 실패를 표현한다.
SvCustomerSummaryRow row = dao.selectCustomerSummary(query)
        .orElseThrow(() -> new BusinessException(
                "E-SV-BIZ-0001",
                "조회 결과가 없습니다.",
                "customer summary not found. customerNo=" + query.getCustomerNo()
        ));

#### 32.20.3 DAO

DAO는 DB 오류를 시스템 예외로 변환한다.
try {
    return mapper.searchCustomers(criteria);
} catch (QueryTimeoutException e) {
    throw new SystemException(
            "E-COM-TIME-DB-0001",
            "Query timeout. sqlId=SV.Customer.searchCustomers",
            e
    );
} catch (DataAccessException e) {
    throw new SystemException(
            "E-COM-DB-0001",
            "Data access error. sqlId=SV.Customer.searchCustomers",
            e
    );
}

#### 32.20.4 Integration Adapter

연계 오류를 구분한다.
try {
    return integrationClient.call(...);
} catch (SocketTimeoutException e) {
    throw new SystemException(
            "E-COM-TIME-IF-0001",
            "Integration read timeout. targetServiceId=SV.Customer.selectSummary",
            e
    );
} catch (IOException e) {
    throw new SystemException(
            "E-COM-IF-0001",
            "Integration connection error. targetBusinessCode=SV",
            e
    );
}

### 32.25 마무리말

예외처리 기준의 핵심은 오류를 숨기는 것이 아니다.오류를 운영 가능한 정보로 바꾸는 것이다.
NSIGHT에서는 다음 구조를 표준으로 삼는다.
```text
Exception
↓
ErrorCode
↓
UserMessage / OperatorMessage
↓
StandardResponse
↓
TransactionLog
↓
AuditLog / Metrics
↓
OM ErrorCode Management
```

업무 개발자는 오류가 발생했을 때 임의 메시지를 반환하지 않고, 반드시 표준 예외와 표준 오류코드를 사용해야 한다. TCF는 이를 ETF와 Global Exception Handler에서 표준 응답, 거래로그, 감사로그, 메트릭으로 변환한다.

### 32.26 시사점

예외처리가 표준화되면 운영자는 장애 발생 시 다음 순서로 추적할 수 있다.
```text
GUID
  ↓
거래로그
  ↓
ServiceId / TransactionCode
  ↓
ErrorCode
↓
OM_ERROR_CODE
↓
운영자 메시지 / 조치 가이드
↓

```

담당팀 조치

따라서 NSIGHT 예외처리 기준은 단순 개발 편의 기능이 아니라 운영 안정성, 장애 추적성, 감사 대응, PMO 설명 가능성을 확보하는 핵심 아키텍처 기준이다.
