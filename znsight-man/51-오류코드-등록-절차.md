# 51. 오류코드 등록 절차

> **NSIGHT TCF 개발 Manual** · 원본: [`znsight-guide`](../znsight-guide/) · 갱신: 2026-07-05

## 51. 오류코드 등록 절차

### 51.1 도입 전 안내말

오류코드는 NSIGHT TCF Framework에서 오류 발생 원인을 식별하고, 사용자에게 표준 메시지를 반환하며, 운영자가 장애 원인을 추적하기 위한 기준정보이다.
NSIGHT에서는 개발자가 임의로 "오류가 발생했습니다" 같은 문자열을 직접 반환하지 않는다. 모든 예외는 TCF 공통 후처리 영역에서 표준 오류코드로 변환하고, OM_ERROR_CODE에 등록된 사용자 메시지·운영자 메시지·조치 가이드를 조회하여 StandardResponse로 조립한다. 기존 오류코드 설계에서도 오류코드와 메시지는 TCF 공통 후처리 ETF에서 일원화하고, 운영관리는 OM 오류코드 관리 화면에서 수행하는 구조가 적합하다고 정리되어 있다.
핵심 문장은 다음이다.
오류코드는 소스에 흩어진 문자열이 아니라 OM에서 관리되는 운영 기준정보이다.

### 51.2 오류코드 등록 절차 결론

| 구분 | 기준 | 등록 주체 |
| --- | --- | --- |
| 개발자 신청 → 업무 PL 검토 → OM 관리자 / 아키텍트 승인 | 등록 위치 | OM 오류코드 관리 화면 |
| 기준 테이블 | OM_ERROR_CODE | 적용 위치 |
| ETF.error() / GlobalExceptionHandler | 코드 형식 | E-{DOMAIN}-{CATEGORY}-{NNNN} |
| 메시지 구분 | 사용자 메시지 / 운영자 메시지 / 조치 가이드 분리 | 사용 주체 |
| TCF, 업무 WAR, OM, Gateway, JWT, Batch, File | 변경 이력 | 등록·수정·중지 이력은 감사로그 기록 |
| 삭제 원칙 | 물리 삭제보다 비활성 처리 | Cache 적용 |
| 오류코드 조회는 Cache 적용 가능 | 보안 원칙 | 사용자 메시지에 StackTrace, SQL, 서버정보, 개인정보 노출 금지 |

예외처리 설계에서도 모든 예외는 TCF 공통 후처리에서 표준 오류코드로 변환하고, StandardResponse로 조립하며, 거래로그·감사로그·모니터링 지표에 남기는 구조를 기준으로 한다.

### 51.3 오류코드 등록 전체 흐름

```text
[1. 오류 발생 가능 지점 식별]

        ↓
[2. 오류코드 명명]
```

```text
        ↓
[4. 운영자 메시지 작성]

        ↓
```

```text
        ↓

        ↓
```

```text
        ↓
[10. 운영 추적]
 거래로그 / 감사로그 / 알림 / 모니터링 연계

```

### 51.5 오류코드 명명 규칙

오류코드는 다음 형식을 사용한다.
E-{DOMAIN}-{CATEGORY}-{NNNN}

| 구분 | 의미 | 예시 |
| --- | --- | --- |
| E | Error | E |
| DOMAIN | 오류 발생 영역 | TCF, COM, SV, OM, GW, JWT, UD |
| CATEGORY | 오류 분류 | HDR, VAL, AUTHN, AUTHZ, CTL, TIME, DB, IF, SYS |
| NNNN | 일련번호 | 0001 |

오류코드 설계 기준에서도 권장 형식은 E-{DOMAIN}-{CATEGORY}-{NNNN}이며, 예시는 E-TCF-HDR-0001, E-TCF-CTL-0001, E-SV-BIZ-0001, E-OM-CODE-0001, E-UD-FILE-0001처럼 정리되어 있다.

### 51.8 오류코드 등록 항목

OM_ERROR_CODE에는 최소한 다음 항목을 등록한다.

| --- | --- | --- | --- |
| ERROR_CODE | Y | 표준 오류코드 | E-TCF-HDR-0001 |
| ERROR_NAME | Y | 오류명 | ServiceId 누락 |

### 51.9 ResultCode와 ErrorCode 분리 기준

오류코드 등록 시 resultCode와 errorCode를 혼동하면 안 된다.
| 구분 | 의미 | 예시 |
| --- | --- | --- |
| resultStatus | 처리 상태 | SUCCESS, FAIL, TIMEOUT, UNKNOWN |
| resultCode | 응답 결과 요약 | S0000, E0001, E0002, T0001 |
| errorCode | 상세 오류 식별자 | E-TCF-HDR-0001 |
| messageCode | 메시지 식별자 | MSG-TCF-HDR-0001 |
| userMessage | 사용자 표시 메시지 | 요청 정보를 확인해 주십시오. |
| operatorMessage | 운영자 상세 메시지 | header.serviceId is blank |
| actionGuide | 조치 가이드 | OM ServiceId 관리 화면을 확인하십시오. |
권장 결과코드는 다음과 같이 단순화한다.

| resultStatus | resultCode |
| 의미 | SUCCESS | S0000 |
| 정상 | FAIL | E0001 |
| 일반 실패 | VALIDATION_FAIL | E0002 |
| 입력값 오류 | AUTH_FAIL | E0003 |
| 인증·권한 오류 | TIMEOUT | T0001 |
| Timeout | UNKNOWN | U0001 |
| 처리상태 불명확 | SYSTEM_ERROR | E9999 |

시스템 오류
기존 오류코드 설계에서도 resultStatus, resultCode, errorCode, messageCode, userMessage, operatorMessage, actionGuide를 분리하는 것을 중요한 기준으로 정의하고 있다.

### 51.13 오류코드 사용 소스 예시

#### 51.13.1 BusinessException 사용 예시

```java
@Component
public class SvCustomerRule {
    public void validateCustomerExists(SvCustomer customer) {
        if (customer == null) {
            throw new BusinessException(
                    "E-SV-BIZ-0001",
                    "조회된 고객 정보가 없습니다.",
                    "Customer summary not found"
            );
        }
    }
}
```

업무적으로 예상 가능한 오류는 BusinessException을 사용한다.예를 들어 조회 결과 없음, 업무 조건 불일치, 입력값 오류, 권한 없음, 거래통제 차단 등이 여기에 해당한다.

#### 51.13.2 SystemException 사용 예시

```java
@Repository
@RequiredArgsConstructor
public class SvCustomerDao {
    private final SvCustomerMapper svCustomerMapper;
    public SvCustomer selectCustomerSummary(String customerNo) {
        try {
            return svCustomerMapper.selectCustomerSummary(customerNo);
        } catch (DataAccessException e) {
            throw new SystemException(
                    "E-COM-DB-0002",
                    "SQL execution failed. sqlId=SvCustomerMapper.selectCustomerSummary",
                    e
            );
        }
    }
}
```

DB, 외부연계, 시스템 설정, Timeout 같은 기술적 오류는 SystemException으로 변환한다.

```text

        ↓
catch
        ↓

```

```text
[ETF.error()]
        ↓
```

### 51.19 오류코드 변경 원칙

| 변경 유형 | 처리 기준 |
| --- | --- |
| 신규 등록 | 승인 후 USE_YN = Y |
| 사용자 메시지 변경 | 업무 영향 검토 후 변경 |
| 운영자 메시지 변경 | 운영팀 검토 후 변경 |
| 조치 가이드 변경 | 운영 절차 변경 시 반영 |
| 심각도 변경 | 알림 정책 영향 검토 |
| 코드값 변경 | 원칙적으로 금지 |
| 오류코드 삭제 | 원칙적으로 금지 |
| 사용 중지 | USE_YN = N 처리 |
| 폐기 | RETIRED 상태 관리 권장 |

오류코드 값 자체를 변경하면 과거 거래로그와 감사로그의 해석이 달라질 수 있다. 따라서 운영 중 오류코드 값은 변경하지 않고, 필요한 경우 신규 오류코드를 추가한 뒤 기존 코드를 비활성화한다.

### 51.20 오류코드 테스트 기준

| 테스트 유형 | 확인 내용 | 정상 등록 |
| --- | --- | --- |
| OM_ERROR_CODE에 오류코드 정상 등록 | 중복 등록 | 동일 오류코드 중복 등록 차단 |
| 형식 검증 | E-{DOMAIN}-{CATEGORY}-{NNNN} 형식 검증 | 메시지 누락 |
| 사용자 메시지, 운영자 메시지 누락 차단 | BusinessException | 오류코드 기준 표준 응답 변환 |
| SystemException | 시스템 오류코드 기준 표준 응답 변환 | 미등록 오류코드 |
| 기본 오류코드로 대체 처리 | 비활성 오류코드 | 사용 차단 또는 기본 메시지 반환 |
| 사용자 메시지 보안 | StackTrace, SQL, 개인정보 미노출 | 거래로그 연계 |
| TCF_TX_LOG.ERROR_CODE 기록 | 감사로그 연계 | 오류코드 등록·수정 이력 기록 |
| Cache 반영 | 변경 후 즉시 응답 메시지 반영 | 알림 연계 |

### 51.21 오류코드 금지 사항

| 금지 사항 | 사유 |
| --- | --- |
| 소스에 사용자 메시지 하드코딩 | 메시지 운영관리 불가 |
| Exception.getMessage()를 사용자에게 직접 노출 | 기술정보 유출 위험 |
| StackTrace를 응답 전문에 포함 | 보안 위험 |
| SQL 전문을 사용자 메시지에 포함 | DB 구조 노출 |
| 오류코드 없이 RuntimeException 발생 | 운영 추적 어려움 |
| 같은 의미의 오류코드 중복 생성 | 운영 기준 혼선 |
| 운영 중 오류코드 값 변경 | 과거 로그 해석 오류 |
| 오류코드 물리 삭제 | 거래로그 추적 불가 |
| 모든 오류를 E9999로 처리 | 원인 분석 불가 |

### 51.22 개발자 체크리스트

| 점검 항목 | 확인 |
| --- | --- |
| 오류코드가 E-{DOMAIN}-{CATEGORY}-{NNNN} 형식을 따르는가 |  |
| 기존 오류코드와 중복되지 않는가 |  |
| 사용자 메시지가 쉽고 기술정보를 노출하지 않는가 |  |
| 운영자 메시지에 원인 추적 정보가 있는가 |  |
| 조치 가이드가 실제 운영자가 따라 할 수 있는가 |  |
| 심각도와 알림 대상이 적절한가 |  |
| 감사로그 대상 여부가 정의되어 있는가 |  |
| 재시도 가능 여부가 정의되어 있는가 |  |
| BusinessException / SystemException에서 해당 오류코드를 사용하는가 |  |
| ETF 후처리에서 표준 응답으로 변환되는가 |  |
| 거래로그에 ERROR_CODE가 기록되는가 |  |
| Cache Evict / Reload가 정상 동작하는가 |  |
| 미등록 오류코드 발생 시 기본 오류 처리 기준이 있는가 |  |
| 운영환경 반영 전 승인이 완료되었는가 |  |

### 51.23 마무리말

오류코드 등록 절차의 핵심은 예외 처리의 운영관리화이다.
오류코드는 개발자가 임의로 작성하는 문자열이 아니라, OM에서 관리하고 TCF 후처리에서 표준 응답으로 변환하는 기준정보이다. 오류코드가 표준화되어야 사용자는 일관된 메시지를 받고, 운영자는 GUID, TraceId, ServiceId, ErrorCode 기준으로 장애를 추적할 수 있다.
정리하면 다음과 같다.
오류 발생
```text
  ↓
BusinessException / SystemException
  ↓
ETF 오류 변환
  ↓
OM_ERROR_CODE 메시지 조회
  ↓
StandardResponse 조립
  ↓
거래로그 / 감사로그 / 모니터링 기록

```

### 51.24 시사점

오류코드는 단순 개발 규칙이 아니라 운영·보안·감사 기준이다.
| 관점 | 시사점 | 개발 관점 |
| --- | --- | --- |
| 신규 오류 상황이 생기면 오류코드를 먼저 정의하고 예외에 적용해야 한다 | 운영 관점 | 오류코드 기준으로 장애 원인, 조치 가이드, 알림 대상을 확인할 수 있어야 한다 |
| 보안 관점 | 사용자 응답에는 내부 기술정보가 노출되지 않아야 한다 | 감사 관점 |
| 오류코드 등록·수정·중지 이력이 남아야 한다 | 아키텍처 관점 | 오류코드는 ETF, 거래로그, 감사로그, OM 운영관리와 연결되는 공통 기준정보이다 |
