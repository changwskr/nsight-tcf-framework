# 14. 명명 규칙

> **NSIGHT TCF 개발 Manual** · 원본: [`znsight-guide`](../znsight-guide/) · 갱신: 2026-07-05

## 14. 명명 규칙

### 14.1 도입 전 안내말

본 장은 NSIGHT TCF Framework에서 사용하는 업무코드, Context, WAR, Package, ServiceId, 거래코드, Java Class, Method, DTO, Mapper, SQL ID, DB 객체, 오류코드의 명명 규칙을 정의한다.
명명 규칙은 단순히 이름을 예쁘게 짓기 위한 기준이 아니다. NSIGHT에서는 이름 하나가 다음 항목을 모두 연결한다.
업무코드
```text
   ↓
Context Path
↓
WAR 파일명
↓
Gradle Module
↓
Java Package
↓
ServiceId
↓
Handler
↓
Facade / Service / Rule / DAO / Mapper
↓
Mapper XML
↓
SQL ID
↓

```

거래로그 / 오류코드 / 감사로그

기존 NSIGHT 네이밍 원칙에서도 핵심 기준을 “업무코드 → Context → WAR → Package → ServiceId → Handler → Facade → Service → Rule → DAO/Mapper → MyBatis XML → SQL ID → 거래로그/오류코드가 하나의 선으로 이어지게 만드는 것”으로 정의하고 있다.

### 14.2 명명 규칙의 목적

| 목적 | 설명 |
| --- | --- |
| 업무 경계 명확화 | 이름만 보고 어느 업무의 소스인지 알 수 있어야 한다 |
| 실행 추적성 확보 | ServiceId, 거래코드, 로그, 오류코드가 연결되어야 한다 |
| 유지보수성 향상 | 신규 개발자도 파일명과 클래스명만 보고 역할을 이해할 수 있어야 한다 |
| 배포 기준 통일 | Context, WAR, Gradle Module명이 동일 업무코드로 연결되어야 한다 |
| SQL 관리 표준화 | Mapper, XML, SQL ID가 일관되어야 한다 |
| 운영관리 연계 | OM의 ServiceId, 거래통제, Timeout, 오류코드와 연결되어야 한다 |
| 품질 점검 용이 | 코드 리뷰 시 명명 위반 여부를 쉽게 확인할 수 있어야 한다 |

### 14.3 최상위 명명 원칙

| 원칙 | 기준 | 예시 |
| --- | --- | --- |
| 업무코드 우선 | 모든 이름은 업무코드를 기준으로 시작한다 | SV, OM, MG |
| Context와 WAR 일치 | URL Context와 WAR 파일명이 같아야 한다 | /sv, sv.war |
| Package와 업무코드 일치 | Package에 업무코드를 포함한다 | com.nh.nsight.sv |
| 실행 기준은 ServiceId | 실제 업무 실행은 URL이 아니라 serviceId 기준이다 | SV.Customer.selectSummary |
| Class는 PascalCase | 첫 글자 대문자, 의미 있는 이름 사용 | SvCustomerService |
| Method는 lowerCamelCase | 동사 + 업무대상 형태 | selectCustomerSummary() |
| Package는 lowercase | 소문자만 사용 | com.nh.nsight.sv.entry.handler |
| DB 객체는 UPPER_SNAKE_CASE | 테이블, 컬럼, 인덱스 통일 | TCF_TRANSACTION_LOG |
| 약어 남용 금지 | 의미 없는 축약어 사용 금지 | Svc 금지, Service 사용 |
| 운영 추적성 우선 | 로그와 오류코드에서 원 업무를 알 수 있어야 한다 | E-SV-BIZ-0001 |

### 14.7 ServiceId 명명 규칙

ServiceId는 TCF Dispatcher가 Handler를 찾는 실행 기준이다.
{BUSINESS_CODE}.{BusinessObject}.{action}

| 구성요소 | 규칙 | 예시 |
| --- | --- | --- |
| BUSINESS_CODE | 대문자 업무코드 | SV |
| BusinessObject | 업무대상 PascalCase | Customer, Campaign, User |
| action | 처리행위 lowerCamelCase | selectSummary, create, update |

예시는 다음과 같다.
| 업무 | ServiceId |

### 14.8 거래코드 명명 규칙

거래코드는 거래로그, 오류 추적, 운영관리, 감사로그에서 사용하는 온라인 거래 식별자이다.
{BUSINESS_CODE}-{TYPE}-{NNNN}

| 구성요소 | 설명 | 예시 |
| --- | --- | --- |
| BUSINESS_CODE | 업무코드 | SV |
| TYPE | 거래유형 | INQ |
| NNNN | 4자리 일련번호 | 0001 |

| TYPE | 의미 | 예시 |
| --- | --- | --- |
| INQ | 조회 | SV-INQ-0001 |
| REG | 등록 | CM-REG-0001 |
| UPD | 수정 | OM-UPD-0001 |
| DEL | 삭제 | OM-DEL-0001 |
| EXE | 실행 | BP-EXE-0001 |
| DWN | 다운로드 | UD-DWN-0001 |
| UPL | 업로드 | UD-UPL-0001 |
| AUTH | 인증 | JWT-AUTH-0001 |

ServiceId와 거래코드는 1:1 또는 1:N으로 연결할 수 있으나, 운영 추적을 위해 대표 거래코드는 반드시 정의한다.

| 구분 | 권장 | 업무 내부에서만 사용하는 DTO |

### 14.13 Mapper XML / SQL ID 명명 규칙

Mapper XML은 업무코드별 디렉터리에 둔다.
resources/mapper/{업무코드소문자}/{MapperName}.xml

예시는 다음과 같다.

| Mapper Interface | Mapper XML | SvCustomerMapper |
| --- | --- | --- |
| mapper/sv/SvCustomerMapper.xml | SvCustomerRdwMapper | mapper/sv/SvCustomerRdwMapper.xml |
| CmCampaignMapper | mapper/cm/CmCampaignMapper.xml | OmUserMapper |
Mapper XML의 namespace는 Java Mapper Interface의 FQCN과 일치해야 한다.

| mapper/om/OmUserMapper.xml | <mapper namespace="com.nh.nsight.sv.persistence.mapper.SvCustomerMapper"> |

SQL 유형

### 14.14 DB 객체 명명 규칙

### 14.22 마무리말

명명 규칙은 개발자 개인의 취향을 제한하기 위한 규칙이 아니다.NSIGHT TCF Framework에서는 명명 규칙이 곧 업무 경계, 실행 경로, 운영 추적, 장애 분석, 배포 단위, SQL 관리를 연결하는 기준이다.
개발자는 신규 ServiceId 하나를 만들 때도 다음 연결을 항상 유지해야 한다.

### 14.23 시사점

NSIGHT에서 좋은 이름은 짧은 이름이 아니다.운영자가 로그만 보고도 어느 업무, 어느 거래, 어느 SQL, 어느 오류인지 추적할 수 있는 이름이 좋은 이름이다.
결론적으로 명명 규칙은 다음 한 문장으로 정리할 수 있다.
모든 이름은 업무코드에서 시작하고, ServiceId를 중심으로 실행되며, Mapper와 SQL ID를 거쳐 거래로그와 오류코드까지 추적 가능해야 한다.
