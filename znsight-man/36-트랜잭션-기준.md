# 36. 트랜잭션 기준

> **NSIGHT TCF 개발 Manual** · 원본: [`znsight-guide`](../znsight-guide/) · 갱신: 2026-07-05

## 36. 트랜잭션 기준

### 36.1 도입 전 안내말

트랜잭션은 하나의 업무 처리를 성공 또는 실패 단위로 묶는 기준이다.NSIGHT TCF Framework에서 트랜잭션은 단순히 @Transactional을 붙이는 문제가 아니라, 업무 정합성, DB Connection 점유시간, Timeout, Rollback, 거래로그 상태, 재처리 가능성을 함께 통제하는 개발 기준이다.
NSIGHT의 기본 처리 흐름은 다음과 같다.
```text
Client / WebTopSuite / tcf-ui
  ↓
Controller / Gateway
  ↓
TCF.process()
  ↓
STF.preProcess()
  - Header 검증
  - 세션 / 권한 / 거래통제
  - 거래로그 PROCESSING 생성
  ↓
Dispatcher
↓
Handler
↓
Facade
- 트랜잭션 경계
↓
Service
↓
Rule
↓
DAO / Mapper
↓
RDW / ADW / OMDB / LOGDB
↓
ETF.postProcess()

```

  - SUCCESS / FAIL / TIMEOUT / UNKNOWN 상태 갱신

기존 DAO 설계 기준에서도 트랜잭션 경계는 DAO가 아니라 Facade 기준으로 두고, DAO와 Mapper는 Facade 트랜잭션에 참여만 하도록 정리되어 있다. AOP 설계 기준에서도 AOP의 주 역할은 DB 트랜잭션 경계와 캐시 처리이며, @Transactional 적용 계층은 Facade 중심으로 정리되어 있다.
핵심 문장은 다음이다.
트랜잭션은 DAO가 아니라 Facade에서 시작한다.Service는 업무 흐름을 조립하고, DAO/Mapper는 이미 시작된 트랜잭션 안에서 SQL을 실행한다.

### 36.2 트랜잭션 기준 결론

| 구분 | 기준 |
| --- | --- |
| 트랜잭션 시작 위치 | Facade |
| 장시간 거래 | 온라인 거래 금지, 배치·비동기·파일 처리로 분리 |
| DAO 트랜잭션 | 원칙 금지 |
| Mapper 트랜잭션 | 금지 |
| Service 트랜잭션 | 예외적 허용, 기본은 Facade |
| Rule 트랜잭션 | 금지 |
| 외부연계 트랜잭션 | 원격 서비스와 로컬 DB를 하나의 트랜잭션으로 묶지 않음 |
| 다중 DB 트랜잭션 | 원칙적으로 최소화 |
| 분산 트랜잭션 | 기본 금지, 특별 승인 필요 |
| Rollback 기준 | 시스템 오류와 변경성 업무 오류는 Rollback |
| Timeout 기준 | SQL Timeout < Transaction Timeout < Client Timeout |
| 거래로그 | 본거래와 분리 또는 후처리 보장 방식 검토 |
| 재처리 | Timeout / UNKNOWN 거래는 상태조회 후 처리 |

```java
트랜잭션 선언
@Transactional
조회 거래
@Transactional(readOnly = true, timeout = 3~5)
등록/수정/삭제 거래
@Transactional(timeout = 5~10, rollbackFor = Exception.class)
```

### 36.3 트랜잭션 적용 위치

| 계층 | 트랜잭션 적용 | 기준 |
| --- | --- | --- |
| Controller | X | HTTP 요청 수신만 담당 |
| STF / TCF / ETF | X | 공통 전처리·후처리 담당 |
| Handler | X | TCF와 업무 Facade 사이 Adapter |
| Facade | O | 유스케이스 단위 트랜잭션 경계 |
| Service | △ | 특별한 내부 유스케이스가 아니면 금지 |
| Rule | X | 업무 규칙 판단, 입력값 검증 |
| DAO | X | Mapper 호출, 트랜잭션 참여만 |
| Mapper XML | X | SQL 실행 |
| Integration Adapter | X | 원격 호출, 자체 트랜잭션 없음 |

NSIGHT Java 코딩 스타일 기준에서도 Facade는 거래 흐름 조립과 트랜잭션 경계를 담당하고, DAO/Mapper는 MyBatis Mapper 호출과 SQL 파라미터·결과 매핑을 담당하며 업무 조건 판단과 사용자 메시지 생성을 금지한다.

### 36.4 트랜잭션 유형

| 유형 | 설명 | 예시 | 기준 |
| --- | --- | --- | --- |
| 조회 트랜잭션 | 데이터 변경 없이 조회 | 고객요약조회, 거래로그조회 | readOnly = true |
| 변경 트랜잭션 | 등록·수정·삭제 | 사용자 등록, 캠페인 상태 변경 | Rollback 명확화 |
| 복합 트랜잭션 | 여러 DAO 호출 | 고객 메모 저장 + 이력 저장 | Facade 1개 경계 |
| 운영관리 트랜잭션 | OM 기준정보 변경 | ServiceId 등록, 권한 변경 | 감사로그 필수 |
| 로그 트랜잭션 | 거래로그·감사로그 저장 | TCF_TX_LOG, OM_AUDIT_LOG | 본거래와 분리 검토 |
| 연계 포함 트랜잭션 | 내부/외부 서비스 호출 포함 | IC → SV 호출 후 IC 저장 | 원격 트랜잭션 묶기 금지 |
| 배치 트랜잭션 | 대량 데이터 처리 | 로그 삭제, 상태 수집 | Chunk 또는 건별 Commit |
| 파일 트랜잭션 | 파일 메타 + Storage 처리 | 업로드 메타 저장 | 파일과 DB 불일치 보상 필요 |

### 36.5 조회성 거래 기준

조회성 거래는 readOnly = true를 기본으로 한다.
```java
@Service
@RequiredArgsConstructor
public class SvCustomerFacade {
    private final SvCustomerService service;
    @Transactional(readOnly = true, timeout = 3)
    public SvCustomerSummaryResponse selectCustomerSummary(SvCustomerSummaryRequest request) {
        return service.selectCustomerSummary(request);
    }
}
```

| 항목 | 기준 | 적용 대상 |
| --- | --- | --- |
| Timeout | 3~5초 | DB |
| RDW / ADW / OMDB / LOGDB | SQL | MyBatis Query Timeout 준수 |
| 금지 | 조회 중 DB Update, 임시 테이블 남용 | 대량 조회 |

```java
단건 조회, 목록 조회, 로그 조회
Annotation
@Transactional(readOnly = true)
```

일반 조회 금지, 다운로드·배치로 분리
조회성 거래라도 트랜잭션을 아예 쓰지 않는 것보다 readOnly = true를 명시하는 것이 좋다.이유는 개발자가 “이 거래는 변경하지 않는다”는 의도를 코드에 남길 수 있고, 일부 DB/ORM/Connection 설정에서 읽기 전용 최적화와 안전성을 기대할 수 있기 때문이다.

### 36.6 등록 / 수정 / 삭제 거래 기준

변경성 거래는 데이터 정합성을 보호해야 하므로 Rollback 기준을 명확히 한다.
```java
@Service
@RequiredArgsConstructor
public class OmUserFacade {
    private final OmUserService service;
    @Transactional(timeout = 5, rollbackFor = Exception.class)
    public OmUserSaveResponse saveUser(OmUserSaveRequest request) {
        return service.saveUser(request);
    }
}
```

| 항목 | 기준 | 적용 대상 |
| --- | --- | --- |
| Rollback | 시스템 오류, 업무 실패 시 Rollback | 영향 건수 |
| Update/Delete 후 영향 건수 확인 | 감사로그 | 관리자 변경, 권한 변경, 고객정보 변경 시 필수 |
| 논리삭제 | 물리삭제보다 우선 | 대량 변경 |
변경성 거래는 다음 순서로 처리한다.

| 온라인 일반 거래 금지 | Rule 검증 |
| ↓ | 기존 데이터 조회 | ↓ |

```text
업무 조건 확인
        ↓
Insert / Update / Delete
        ↓
영향 건수 확인
        ↓
변경 이력 / 감사로그 기록
        ↓
Commit
등록, 수정, 삭제, 상태변경
Annotation
@Transactional(timeout = 5~10, rollbackFor = Exception.class)
```

### 36.10 SQL Timeout과 Transaction Timeout 관계

SQL Timeout은 Transaction Timeout보다 짧아야 한다.
좋은 기준
MyBatis Query Timeout = 3초
Spring Transaction Timeout = 5초

나쁜 기준
MyBatis Query Timeout = 10초
Spring Transaction Timeout = 5초

나쁜 기준에서는 트랜잭션은 이미 5초에 만료되었는데 SQL은 10초까지 실행될 수 있어 상태 판단이 어려워진다.
| 항목 | 기준 |
| --- | --- |
| SQL Timeout | SQL 1건 실행 제한 |
| Transaction Timeout | 업무 DB 트랜잭션 전체 제한 |
| Online Timeout | 업무 전체 처리 제한 |
| Client Timeout | 사용자 또는 앞단 시스템 대기 제한 |
| 원칙 | 안쪽 Timeout이 바깥 Timeout보다 짧아야 함 |

AOP 운영 기준에서도 장시간 SQL은 MyBatis Timeout 3초를 우선 적용하고, Service 직접 트랜잭션과 DAO 트랜잭션은 금지하며, Timeout 값은 조회·CUD·예외성 거래별로 관리하도록 정리되어 있다.

### 36.11 Propagation 기준

Propagation은 트랜잭션 전파 방식이다.NSIGHT에서는 복잡한 전파 옵션을 남용하지 않는다.
Propagation
| 사용 기준 | 설명 |
| --- | --- |
| REQUIRED | 기본 |

| 제한 사용 | 본거래와 분리해야 하는 이력·감사·로그 |
| SUPPORTS | 조회 유틸성 |

| NOT_SUPPORTED | 제한 사용 |
| 트랜잭션 없이 수행 | NESTED |
| 원칙 금지 | 운영 복잡도 증가 |
| propagation = Propagation.REQUIRED, | timeout = 5, |
| rollbackFor = Exception.class | ) |

```java
기본은 다음이다.

```

감사로그를 본거래와 분리할 경우에는 제한적으로 REQUIRES_NEW를 검토한다.

### 36.12 Isolation 기준

Isolation은 DB 격리 수준이다.NSIGHT 온라인 업무에서는 기본 DB 격리수준을 그대로 사용한다.
Isolation
| 사용 기준 | 설명 |
| --- | --- |
| DEFAULT | 기본 |
| DB 기본값 사용 | READ_COMMITTED |
| 일반 온라인 권장 | Oracle 기본 성격과 유사 |
| REPEATABLE_READ | 제한 |
| 장시간 Lock 가능성 검토 | SERIALIZABLE |
| 원칙 금지 | 성능 저하와 Lock 증가 |
| READ_UNCOMMITTED | 금지 |
권장 코드는 다음처럼 Isolation을 명시하지 않는 것이다.

| Dirty Read 위험 | |

```java
@Transactional(readOnly = true, timeout = 3)
public SvCustomerSummaryResponse selectCustomerSummary(SvCustomerSummaryRequest request) {
return service.selectCustomerSummary(request);
}
```

특별한 업무 정합성 요건이 있을 때만 DBA와 협의해 Isolation을 변경한다.

### 36.14 서비스 간 연동과 트랜잭션 기준

서비스 간 연동에서는 로컬 DB와 원격 서비스 DB를 하나의 트랜잭션으로 묶지 않는다.
금지:
IC 트랜잭션 시작
```text
  ↓
IC DB Update
↓
SV 원격 서비스 호출
↓

```

SV DB Update까지 같은 트랜잭션처럼 기대

권장:
IC는 IC DB 트랜잭션만 책임진다.
SV는 SV DB 트랜잭션만 책임진다.
서비스 간 정합성은 상태값, 거래로그, 보상처리로 관리한다.

| 항목 | 기준 |
| --- | --- |
| 원격 호출 | Integration Adapter에서 수행 |
| 원격 트랜잭션 | 피호출 서비스 내부 책임 |
| 로컬 트랜잭션 | 호출 서비스 Facade 책임 |
| 분산 Commit | 기본 금지 |
| Timeout | 원격 호출 Timeout은 짧게 |
| Retry | 변경성 거래 Retry 금지 |
| 상태 불명확 | GUID 기준 상태조회 후 처리 |

### 36.15 외부연계 포함 거래 기준

외부연계가 포함된 거래는 트랜잭션을 길게 잡지 않는다.
| 패턴 | 기준 |
| --- | --- |
| 조회성 연계 | 짧은 Timeout, 실패 시 업무 오류 |
| 변경성 연계 | 상태관리 테이블 필요 |
| DB 저장 후 외부송신 | 송신상태 READY 저장 후 별도 송신 권장 |
| 외부송신 후 DB 저장 | 실패 시 정합성 문제 주의 |
| Timeout | 처리상태 불명확 가능 |
| Retry | 멱등성 Key 없으면 금지 |
| 보상처리 | 필요 시 별도 설계 |
| 권장 패턴은 다음이다. | 요청 수신 |

```text
        ↓
로컬 DB에 요청 상태 저장 READY
        ↓
Commit
        ↓
외부연계 호출
        ↓
연계 결과 수신
        ↓
상태 SUCCESS / FAIL 갱신
```

이 방식은 한 번에 끝나는 것처럼 보이지는 않지만, Timeout이나 연계 장애가 발생했을 때 상태 추적과 재처리가 가능하다.

### 36.16 거래로그 / 감사로그와 트랜잭션 기준

거래로그와 감사로그는 본거래 Rollback 때문에 사라지면 안 되는 경우가 많다.

| 로그 유형 | 트랜잭션 기준 | 거래 시작 로그 |
| --- | --- | --- |
| 본거래 전 또는 별도 트랜잭션 검토 | 거래 종료 로그 | 후처리에서 상태 갱신 |
| 오류 로그 | 실패해도 기록되어야 함 | 감사로그 |

본거래 성공/실패와 별도 보존 필요
성능로그
비동기 가능
배치로그
Job 단위 별도 트랜잭션
예시는 다음과 같다.
```text
STF
  ↓
TCF_TX_LOG PROCESSING insert
  - 별도 트랜잭션 또는 LOGDB 기준
  ↓
```

업무 Facade @Transactional
```text
  ↓
```

업무 처리
```text
  ↓
ETF
  ↓
```

TCF_TX_LOG SUCCESS / FAIL / TIMEOUT update

본거래가 Rollback되어도 거래로그에는 실패 이력이 남아야 한다.따라서 거래로그 DB 작업은 업무 DB 트랜잭션과 분리하거나, 후처리에서 별도 보장 방식으로 처리하는 것이 좋다.

### 36.17 다중 DB 트랜잭션 기준

NSIGHT는 RDW, ADW, SESSIONDB, LOGDB, OMDB 등 여러 DB를 사용할 수 있다.하지만 하나의 온라인 거래에서 여러 DB를 동시에 변경하는 것은 최소화한다.
| 조합 | 기준 |
| --- | --- |
| RDW 조회 + LOGDB 기록 | 허용 |
| OMDB 변경 + AUDIT 기록 | 허용, 감사 별도 트랜잭션 검토 |
| SESSIONDB 조회 + 업무 DB 조회 | 허용 |
| RDW 변경 + ADW 변경 | 원칙 금지 |
| 서로 다른 업무 DB 동시 변경 | 원칙 금지 |
| 두 DB 동시 Commit 필요 | JTA/XA 검토, 기본 금지 |
| 분석 DB 변경 | 온라인 거래에서 금지 |
다중 DB 변경이 꼭 필요한 경우에는 다음 중 하나로 설계한다.

| 방식 | |
| 설명 | 단일 DB로 책임 통합 |
| 하나의 DB에서 처리하도록 업무 경계 조정 | 상태관리 + 비동기 처리 |
| 먼저 상태 저장 후 다른 DB 반영 | 보상 트랜잭션 |
| 실패 시 반대 거래 수행 | JTA/XA |
| 운영 복잡도 높음, 별도 승인 필요 | 배치 정합성 보정 |
| 주기적으로 차이 보정 |  |

### 36.18 배치 트랜잭션 기준

배치는 온라인과 다르게 긴 처리시간을 가질 수 있으므로 Chunk 또는 건별 Commit 기준을 둔다.
| 항목 | 기준 |
| --- | --- |
| 대량 처리 | Chunk 단위 Commit |

| 재처리 | 실패건 기준 재처리 가능해야 함 |
| Lock | 장시간 Lock 금지 |
| Commit 주기 | 업무량과 DB 부하 기준 조정 |

| Timeout | 온라인 기준과 분리 |
| 중복 실행 | Scheduler Lock 또는 Job Lock 필요 |

```yaml
      online-transaction-seconds: 5
```

MyBatis와 HikariCP 설정도 함께 맞춘다.

### 36.22 Self-invocation 주의 기준

Spring @Transactional은 프록시 기반이므로 같은 클래스 내부 메서드 호출에는 적용되지 않을 수 있다.
나쁜 예
```java
@Service
public class OmUserFacade {
    public void saveAll(List<OmUserSaveRequest> requests) {
        for (OmUserSaveRequest request : requests) {
            saveOne(request); // 같은 클래스 내부 호출
        }
    }
    @Transactional
    public void saveOne(OmUserSaveRequest request) {
        // 트랜잭션 적용이 기대와 다를 수 있음
    }
}
```

좋은 예
```java
@Service
@RequiredArgsConstructor
public class OmUserBulkFacade {
    private final OmUserSingleSaveFacade singleSaveFacade;
    public void saveAll(List<OmUserSaveRequest> requests) {
        for (OmUserSaveRequest request : requests) {
            singleSaveFacade.saveOne(request);
        }
    }
}
@Service
@RequiredArgsConstructor
class OmUserSingleSaveFacade {
    @Transactional(timeout = 5, rollbackFor = Exception.class)
    public void saveOne(OmUserSaveRequest request) {
        // 개별 트랜잭션 적용
    }
}
```

AOP 운영 기준에서도 내부 메서드 호출, 즉 Self-invocation에 주의해야 한다고 정리되어 있다.

### 36.25 마무리말

트랜잭션 기준의 핵심은 “어디에 @Transactional을 붙일 것인가”가 아니다.업무 유스케이스 하나를 어떤 단위로 성공·실패 처리할 것인지 정하는 것이다.
NSIGHT에서는 다음 구조를 표준으로 삼는다.
Facade
  = 트랜잭션 경계

Service
  = 업무 처리 흐름

Rule
  = 업무 규칙 판단

DAO / Mapper
  = DB 접근 참여

ETF
  = 결과 응답과 거래로그 상태 갱신

따라서 트랜잭션은 Facade에서 시작하고, Service·Rule·DAO·Mapper는 각자의 책임 안에서 참여한다.
