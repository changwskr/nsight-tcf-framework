# 61. 코드 리뷰 기준

> **NSIGHT TCF 개발 Manual** · 원본: [`znsight-guide`](../znsight-guide/) · 갱신: 2026-07-05

## 61. 코드 리뷰 기준

### 61.1 도입 전 안내말

NSIGHT TCF Framework에서 코드 리뷰는 단순히 문법 오류나 취향을 확인하는 절차가 아니다.코드 리뷰는 개발자가 작성한 코드가 NSIGHT 표준 아키텍처, TCF 거래 구조, 계층 책임, 보안, 로그, SQL, 테스트 기준을 지키는지 확인하는 품질 통제 절차이다.
NSIGHT Java 코딩 스타일 가이드도 코드 스타일을 단순 들여쓰기 기준이 아니라 업무 책임 경계, 데이터 접근 통제, 오류 추적성, 보안 감사, 운영 가능성을 코드 수준에서 보장하는 표준으로 정의한다. 또한 품질 통제 기준으로 코드 인스펙션, 단위 테스트, SQL 검증, SonarQube/Checkstyle/PMD/JUnit/MyBatis SQL 검증을 제시한다.
코드 리뷰
= 표준 구조 검토
+ 계층 책임 검토
+ ServiceId 거래 흐름 검토
+ 예외/오류코드 검토
+ 로그/GUID/MDC 검토
+ 보안/마스킹 검토
+ MyBatis SQL 검토
+ 트랜잭션/Timeout 검토
+ 테스트 코드 검토
+ 운영 추적성 검토

### 61.10 TCF 구조 리뷰 기준

NSIGHT TCF는 일반 REST Resource 방식이 아니라 HTTP/JSON 표준 전문 + ServiceId Dispatcher 방식이다. 온라인 진입점은 POST /{businessCode}/online이며, 실제 실행 기준은 URL이 아니라 header.serviceId이다. 또한 업무 처리는 Handler → Facade → Service → Rule → DAO/Mapper 흐름을 따른다.
| 리뷰 항목 | 기준 |
| --- | --- |
| URL 구조 | /sv/customer/list 같은 REST URL 우회가 없는가 |
| ServiceId | header.serviceId 기준으로 Handler가 실행되는가 |
| 등록 기준 | Service Catalog 또는 Handler Registry에 등록되어 있는가 |
| 업무코드 | URL Path, Header businessCode, ServiceId Prefix가 일치하는가 |
| 거래코드 | ServiceId와 transactionCode가 매핑되는가 |
| Handler | TCF 공통 처리 이후 업무 진입점 역할만 하는가 |
| STF 중복 구현 | Header, 세션, 권한, 거래통제를 Handler에서 반복 구현하지 않는가 |
| ETF 우회 | 표준 응답 조립을 직접 하지 않고 ETF를 사용하는가 |
| 거래로그 | GUID, TraceId, ServiceId 기준 추적 가능한가 |

### 61.11 계층 책임 리뷰 기준

업무 WAR 내부는 Handler, Facade, Service, Rule, DAO, Mapper 구조를 따르며, 각 계층은 책임과 금지사항이 분명하다. Handler는 요청 Body 변환과 Facade 호출, Facade는 유스케이스와 트랜잭션 경계, Service는 업무 흐름, Rule은 업무 규칙, DAO는 Mapper 호출, Mapper는 SQL 실행을 담당한다.
| 계층 | 리뷰 기준 | 금지 사항 |
| --- | --- | --- |
| Handler | ServiceId 진입, Body 변환, Facade 호출 | DB 직접 접근, 복잡한 업무 로직 |
| Facade | 유스케이스 조립, 트랜잭션 경계 | SQL 직접 호출, 세부 Rule 직접 구현 과다 |
| Service | 업무 흐름 제어, Rule/DAO/Client 조합 | 화면/HTTP 의존, SQL 문자열 작성 |
| Rule | 검증, 계산, 정책 판단 | DB Connection 직접 사용, 외부 API 직접 호출 |
| DAO | Mapper 호출 캡슐화, 조회 조건 보정 | 업무 판단, 메시지 조립 |
| Mapper | MyBatis SQL 실행 | Java 업무 로직 포함 |
| Client | 외부/타업무 호출 캡슐화 | 업무 규칙 판단 |
| Config | 기술 설정 | 업무 로직 포함 |

### 61.13 ServiceId / 거래코드 리뷰 기준

| 리뷰 항목 | 기준 |
| --- | --- |
| ServiceId 형식 | {BusinessCode}.{Domain}.{action} |
| ServiceId 등록 | OM 또는 Service Catalog 기준 등록 필요 |
| Handler 매핑 | ServiceId와 Handler가 1:1 또는 명확한 매핑 |
| 업무코드 일치 | ServiceId Prefix와 businessCode 일치 |
| 거래코드 형식 | {업무코드}-{처리유형}-{일련번호4자리} |
| 처리유형 | INQ, REG, UPD, DEL, EXE 등 표준 코드 |
| 권한 연결 | ServiceId 기준 메뉴/기능권한 연결 가능 |
| 거래통제 연결 | ServiceId 기준 Allow-List 관리 가능 |
| Timeout 연결 | ServiceId 기준 Timeout 정책 적용 가능 |
| 로그 추적 | 거래로그에 ServiceId/transactionCode 저장 가능 |

### 61.14 Handler 리뷰 기준

| 리뷰 항목 | 기준 |
| --- | --- |
| 단일 책임 | Body 변환과 Facade 호출 중심 |
| 공통 검증 | Header, 세션, 권한, 거래통제는 STF에 위임 |
| Body 변환 | StandardRequest.body를 업무 Request DTO로 변환 |
| Facade 호출 | Handler는 Facade만 호출 |
| 예외 처리 | 예외를 삼키지 않고 TCF로 전달 |
| 응답 | 업무 DTO만 반환하고 StandardResponse 직접 조립 금지 |
| 로깅 | 필요 시 ServiceId/GUID 중심, 개인정보 제외 |
| 테스트 | Handler 단위 테스트 포함 |
| // Handler에서 DB 직접 접근 | return customerMapper.selectCustomerSummary(request.getBody().get("customerNo")); |

```java
나쁜 예
public Object handle(StandardRequest request) {
}
```

좋은 예
```java
public CustomerSummaryResponse handle(StandardRequest request, TransactionContext context) {
    CustomerSummaryRequest body = request.convertBody(CustomerSummaryRequest.class);
    return svCustomerFacade.selectCustomerSummary(body, context);
}
```

### 61.15 Facade / Service / Rule 리뷰 기준

| 계층 | 리뷰 기준 | Facade |
| --- | --- | --- |
| 트랜잭션 경계가 적절한가 | Facade | 여러 Service 호출 순서가 유스케이스 기준으로 명확한가 |
| Facade | SQL 직접 호출이 없는가 | Service |
| Rule, DAO, Client 조합이 명확한가 | Service | 화면/HTTP 객체에 의존하지 않는가 |
| Service | 외부연계 오류를 표준 예외로 변환하는가 | Rule |
| 업무 규칙이 Service에서 분리되어 있는가 | Rule | DB/HTTP 의존 없이 테스트 가능한가 |

### 61.16 DAO / Mapper 리뷰 기준

| 운영 | 거래로그, 감사로그, 장애 추적 가능 |
| 승인 | 필수 리뷰어 승인 완료 |

### 61.31 마무리말

NSIGHT 코드 리뷰의 핵심은 코드가 동작하는지가 아니라,운영 가능한 TCF 거래로 배포해도 되는지를 판단하는 것이다.
코드 리뷰 완료
= 계층 책임 준수
+ ServiceId 흐름 정상
+ SQL 안전성 확보
+ 예외/오류코드 표준화
+ 로그/GUID 추적 가능
+ 보안/마스킹 적용
+ 트랜잭션/Timeout 적정
+ 테스트 자동화 통과
+ 운영 장애 시 설명 가능

### 61.32 시사점

NSIGHT TCF Framework는 여러 업무 WAR, 공통 TCF 모듈, Gateway, MyBatis, RDW/ADW, SESSIONDB, LOGDB가 결합된 구조다.따라서 코드 리뷰는 개발자 개인의 코드 스타일 점검이 아니라 전체 운영 안정성을 지키는 아키텍처 통제 활동이다.
코드 리뷰의 최종 기준은 “이 코드가 운영에서 장애가 났을 때 GUID, TraceId, ServiceId, SQL ID, 오류코드만으로 원인을 추적할 수 있는가”이다.
