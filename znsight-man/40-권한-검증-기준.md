# TCF 로그인 정리

NSIGHT TCF의 **OM 포털 로그인**은 **TCF 표준 거래(`OM.Auth.login`) + Spring Session JDBC** 조합으로 동작합니다.  
로그인도 일반 온라인 거래와 동일하게 `serviceId` → Handler → Facade → Service 파이프라인을 탑니다.

| 관점 | 역할 | 핵심 |
|------|------|------|
| **인증 거래** | 로그인·로그아웃·세션 조회 | `OM.Auth.login/logout/session` |
| **인증 로직** | ID/PW 검증·세션 생성 | `OmAuthService` |
| **세션 저장** | HttpSession → JDBC | `SPRING_SESSION` + `OmAuthSessionSupport` |
| **UI** | 로그인 화면·Relay | `login.html`, `om-admin.js` |
| **쿠키** | JSESSIONID 전파 | `TransactionRelayService` |
| **Gateway** | SESSIONDB 등록 | `TCF_USER_SESSION` (GEF) |
| **JWT (별도)** | 토큰 발급 | `JWT.Auth.login` (`tcf-jwt`) |

세션 유지·검증·운영 모니터링: [세션관리.md](세션관리.md)

---

## 1. 개요 — 로그인도 TCF 거래

```text
로그인 ≠ 별도 REST API
로그인 = POST /om/online + StandardRequest { OM.Auth.login }
```

| 특징 | 설명 |
|------|------|
| 거래 식별 | `serviceId: OM.Auth.login`, `transactionCode: OM-AUT-0002` |
| processingType | `EXECUTE` |
| 파이프라인 | STF → Handler → Facade → Service → ETF |
| 세션 | 성공 시 `HttpSession` + `SPRING_SESSION` 행 생성 |
| 브라우저 | `JSESSIONID` 쿠키 + `sessionStorage` 사용자 정보 |

---

## 2. 전체 흐름

```text
Browser (login.html)
  │ userId / password
  ▼
om-admin.js login()
  │ POST /api/relay/OM/online
  │ { header: OM.Auth.login, body: { userId, password } }
  ▼
tcf-ui TransactionRelayService
  │ Cookie 전달 · Set-Cookie 역전달
  ▼
tcf-om POST /om/online
  ▼
TCF.process()
  STF → OmAuthHandler (OM.Auth.login) → OmAuthFacade.login()
       → OmAuthService.login()
       → OmAuthSessionSupport.createSession()
  ETF → StandardResponse
  ▼
Browser: JSESSIONID 쿠키 + sessionStorage(nsight.om.session)
```

**Gateway 경유 시 (GEF):**  
`OM.Auth.login` 성공 응답의 `sessionId`·Set-Cookie로 **`TCF_USER_SESSION`** 에 등록 — Gateway는 HttpSession을 소유하지 않음.

---

## 3. serviceId · 카탈로그

| serviceId | transactionCode | Handler | 용도 |
|-----------|-----------------|---------|------|
| `OM.Auth.login` | `OM-AUT-0002` | `OmAuthHandler` | 로그인 |
| `OM.Auth.logout` | `OM-AUT-0003` | `OmAuthHandler` | 로그아웃 |
| `OM.Auth.session` | `OM-AUT-0004` | `OmAuthHandler` | 현재 세션 조회 |

- `OM_SERVICE_CATALOG` 등록 — `seed-auth-login.sql`, `ServiceCatalogSeedData`
- 거래통제 면제: `TransactionControlExemptions.isAuthBootstrap()` — 로그인 전 단계 허용

## 4. UI 레이어 (tcf-ui)

### 4.1 화면

| 환경 | URL |
|------|-----|
| bootRun | http://localhost:8099/om/admin/login.html |
| Tomcat | http://localhost:8080/ui/om/admin/login.html |

- 진입 시 `authSession` 호출 → 이미 로그인이면 대시보드로 이동
- 제출 → `OmAdmin.login(userId, password)`

### 4.2 요청 조립 (`om-admin.js`)

**Header**

| 필드 | 값 |
|------|-----|
| `businessCode` | `OM` |
| `serviceId` | `OM.Auth.login` |
| `transactionCode` | `OM-AUT-0002` |
| `processingType` | `EXECUTE` |
| `channelId` | `WEBTOP` |

**Body**

```json
{ "userId": "admin01", "password": "..." }
```

**성공 판별**

```text
1. relay.httpStatus < 400
2. result.resultCode === "S0000"
3. body.loggedIn === true
```

→ `sessionStorage`에 `userId`, `userName`, `branchId`, `authGroupId`, `sessionId` 저장

### 4.3 Relay · 쿠키

| 방향 | 처리 |
|------|------|
| 요청 | Browser `Cookie` → tcf-om |
| 응답 | tcf-om `Set-Cookie` → Browser |

`credentials: 'include'` — 이후 OM Admin 거래에서 동일 `JSESSIONID` 사용.

**참고:** `OM.Auth.*` 거래는 **Gateway JWT 경로를 타지 않음** (`isAuthTransaction` → Relay 직통).

## 5. 서버 — Handler · Facade · Service

### 5.1 호출 체인

```text
OmAuthHandler (serviceId=OM.Auth.login)
  → OmAuthFacade.login()     @Transactional(timeout=5)
  → OmAuthService.login()
```

### 5.2 OmAuthService.login 처리 순서

```32:63:tcf-om/src/main/java/com/nh/nsight/marketing/om/application/service/OmAuthService.java
    public Map<String, Object> login(Map<String, Object> body, TransactionContext context) {
        rule.validateOperation(context);
        rule.requireField(body, "userId");
        rule.requireField(body, "password");
        // 1. 사용자 조회 (dao.selectUserForLogin)
        // 2. useYn == Y 확인
        // 3. PasswordEncoder.matches(password, passwordHash)
        // 4. lastLoginTime 갱신
        // 5. sessionSupport.createSession(user)
        // 6. loggedIn=true, sessionId 포함 body 반환
```

| 단계 | 내용 |
|------|------|
| 검증 | `OmOperationRule.requireField` |
| 사용자 | `OM_USER` — `selectUserForLogin` |
| 비밀번호 | `PasswordEncoder` vs `PASSWORD_HASH` |
| 실패 | `E-OM-AUTH-0001` — **통일 메시지** (계정 존재 여부 비노출) |
| 성공 | `lastLoginTime` UPDATE + HttpSession 생성 |

### 5.3 logout · session

| 메서드 | 동작 |
|--------|------|
| `logout` | `sessionSupport.invalidateSession()` → `loggedOut=true` |
| `session` | `currentUser()` → `toSessionBody(loggedIn)` — UI 자동 로그인 확인 |

## 6. 세션 생성 — OmAuthSessionSupport

```18:25:tcf-om/src/main/java/com/nh/nsight/marketing/om/support/OmAuthSessionSupport.java
    public HttpSession createSession(Map<String, Object> user) {
        HttpSession session = request.getSession(true);
        session.setAttribute(FindByIndexNameSessionRepository.PRINCIPAL_NAME_INDEX_NAME, userId);
        session.setAttribute(SESSION_USER, toSessionUser(user));  // OM_SESSION_USER
        return session;
    }
```

| 세션 속성 | 내용 |
|-----------|------|
| `OM_SESSION_USER` | `OmSessionUser` — userId, userName, branchId, authGroupId … |
| `PRINCIPAL_NAME_INDEX_NAME` | Spring Session 사용자 인덱스 (userId) |

**응답 body (로그인 성공)**

```json
{
  "businessCode": "OM",
  "screen": "OM 로그인",
  "loggedIn": true,
  "sessionId": "...",
  "userId": "admin01",
  "userName": "...",
  "branchId": "...",
  "authGroupId": "...",
  "authGroupName": "..."
}
```

## 8. TCF 파이프라인과 로그인

```text
OM.Auth.login
  → STF (Header, guid, 거래통제 면제, Timeout)
  → Dispatcher → OmAuthHandler
  → Facade @Transactional
  → ETF (거래로그, result S0000)
```

### STF 세션 검증

`nsight.tcf.session-validation-enabled=true` 시 `SessionValidator`가 Header `userId`를 검사합니다.

- 로그인 거래는 **인증 전**이므로 Header `userId`는 입력값·GUEST일 수 있음
- 운영에서 세션 검증 on 시 **로그인/비로그인 거래 경계** 설계 필요

로그인·로그아웃·세션 조회는 **거래통제 면제** (`TransactionControlExemptions.isAuthBootstrap`).

---

## 9. Gateway (GEF) 연계

Gateway는 세션을 **소유하지 않습니다**. `OM.Auth.login` 성공 시:

```text
GatewaySessionRegistry.tryRegisterOmLogin()
  → 응답 body.loggedIn == true
  → sessionId (body 또는 Set-Cookie)
  → UserSessionService.registerLogin()
  → TCF_USER_SESSION INSERT/UPDATE
```

이후 Gateway 경유 downstream 요청은 **Bearer 시 JWT JWKS 검증**, **쿠키 시 SESSIONDB 4단계 검증** — [세션관리.md](세션관리.md)

---

## 10. JWT 로그인 (별도 모듈)

OM **쿠키 세션**과 별도로 `tcf-jwt`가 **Bearer 토큰** 인증을 제공합니다.

| serviceId | 용도 |
|-----------|------|
| `JWT.Auth.login` | Access/Refresh 토큰 발급 |
| `JWT.Auth.refresh` | 토큰 갱신 |
| `JWT.Auth.logout` / `revoke` | 토큰 무효화 |

| 구분 | OM 로그인 | JWT 로그인 |
|------|-----------|------------|
| 모듈 | `tcf-om` | `tcf-jwt` |
| UI | `/om/admin/login.html` | `/jwt/admin/login.html` |
| 상태 | `JSESSIONID` + Spring Session | `accessToken` (sessionStorage) |
| Gateway | `TCF_USER_SESSION` 등록 | Authorization Bearer |

JWT 로그인 후 OM Admin 일부 거래는 **tcf-ui** `om-admin.js`의 `callViaGateway` + Bearer 사용 (`omGatewayEnabled`, `loginType !== 'SSO'`).

## 11. 오류 · 보안

### 11.1 오류 코드

| 상황 | errorCode |
|------|-----------|
| ID/PW 불일치·비활성 계정 | `E-OM-AUTH-0001` |
| STF 세션 검증 실패 | `E-COM-AUTH-0001` |
| 시스템 오류 | `E-COM-SYS-0001` |

### 11.2 보안 포인트

1. 비밀번호 — DB `PASSWORD_HASH` + `PasswordEncoder` (평문 저장 X)
2. 실패 메시지 통일 — 계정 열거 방지
3. 쿠키 — HttpOnly, SameSite=Lax
4. 운영 — HTTPS + `secure cookie=true` 권장
5. 탈취 대응 — `OM.Session.delete` 강제 종료

### 11.3 개발용 계정

- `OmUserPasswordInitializer` — `PASSWORD_HASH` NULL 사용자에 초기 비밀번호 `nsight01!` 설정

## 14. 개발자 체크리스트

1. 로그인은 **`OM.Auth.login`** TCF 거래로만 구현 (별도 REST 남용 X)
2. 세션 생성은 **`OmAuthSessionSupport`** 경유
3. UI는 **Relay + credentials:include** 로 쿠키 유지
4. 성공 판별 — **`resultCode=S0000` + `body.loggedIn=true`**
5. **`OM_SERVICE_CATALOG`** 에 Auth 거래 등록
6. 비밀번호 검증 실패 — **`E-OM-AUTH-0001`** 통일 메시지
7. Gateway 사용 시 **`OM.Auth.login` 성공 → TCF_USER_SESSION** 등록 확인
