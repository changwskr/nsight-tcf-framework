# DB 객체 명명규칙

> **NSIGHT TCF 명명규칙 상세** · 원본: [`znsight-guide-word`](../znsight-guide-word/) · 갱신: 2026-07-05

아래는 NSIGHT TCF 개발가이드 - 11차에 바로 붙여 넣을 수 있는 DB 객체 명명규칙 설계기준입니다.

DB 객체 명명규칙 설계기준
## 1. 도입 전 안내말

DB 객체 명명규칙은 단순히 테이블명과 컬럼명을 보기 좋게 정하는 기준이 아니다.NSIGHT TCF에서는 DB 객체명이 다음 항목과 직접 연결된다.
업무코드
```text
  ↓
ServiceId
  ↓
Handler / Facade / Service / Rule
  ↓
DAO / Mapper
  ↓
Mapper XML
  ↓
SQL ID
  ↓
Table / Column / Index / Constraint
  ↓
거래로그 / 감사로그 / 운영관리

```

NSIGHT TCF 구조에서는 업무 처리가 Handler → Facade → Service → Rule → DAO / Mapper → MyBatis XML → DB 흐름으로 이어지며, DAO/Mapper는 업무 로직이 아니라 SQL 실행 책임을 갖는다. 따라서 DB 객체명도 업무코드, 데이터 성격, 운영 목적을 명확히 드러내야 한다.
또한 NSIGHT 테이블 구조는 TCF 실행/통제, OM 운영관리, Session, Gateway, JWT, Batch, File/Message 영역으로 구분되어 있으므로 DB 객체명은 어느 영역의 객체인지 식별 가능해야 한다.

## 2. DB 객체 명명 최상위 원칙

| 원칙 | 설계 기준 | 예시 |
| --- | --- | --- |
| 대문자 사용 | DB 객체명은 모두 대문자 사용 | OM_USER, TCF_TX_LOG |
| 언더스코어 사용 | 단어 구분은 _ 사용 | SERVICE_ID, TRANSACTION_CODE |
| 업무코드 우선 | 업무 테이블은 업무코드 또는 영역 Prefix로 시작 | SV_CUSTOMER_SUMMARY |
| 영역 식별 | 공통/운영/로그/세션/게이트웨이 등 영역 Prefix 사용 | OM_, TCF_, GW_, JWT_ |
| 의미 우선 | 축약보다 의미가 분명한 이름 사용 | CUSTOMER_NO, BRANCH_ID |
| 물리명 일관성 | Table, Column, Index, Constraint가 동일한 용어 사용 | USER_ID, IDX_OM_USER_01 |
| 약어 통제 | 표준 약어만 사용 | TX, AUTH, CFG, HIST |
| Java 매핑 고려 | MyBatis map-underscore-to-camel-case와 매핑 가능해야 함 | CUSTOMER_NO → customerNo |
| 운영 추적성 | 로그, 감사, Timeout, 거래통제와 연결 가능해야 함 | SERVICE_ID, GUID, TRACE_ID |
| 임시명 금지 | 테스트성, 개인명, 의미 없는 이름 금지 | TMP1, TEST_TABLE, KIM_TABLE 금지 |

| OM_ | 운영관리 | 사용자, 권한, 메뉴, 공통코드, 오류코드, 환경설정 | OM_USER, OM_COMMON_CODE |

### 7.1 기본 형식

FK_{자식테이블명}_{부모테이블약어}_{순번}

| --- | --- | --- |
| OM_FUNCTION_AUTH | OM_AUTH_GROUP | FK_OM_FUNCTION_AUTH_AUTH_GROUP_01 |
| OM_FUNCTION_AUTH | OM_MENU | FK_OM_FUNCTION_AUTH_MENU_01 |

| 기준 | 설명 |

## 12. View 명명규칙

### 12.1 기본 형식

VW_{업무코드}_{조회대상}

또는 운영 View는 다음 형식을 사용한다.
VW_OM_{조회대상}
VW_TCF_{조회대상}

| View 명 | 설명 |
| --- | --- |
| VW_TCF_TX_LOG_SUMMARY | 거래로그 요약 조회 |
| VW_OM_USER_AUTH | 사용자 + 권한그룹 조회 |
| VW_OM_MENU_AUTH | 메뉴 + 기능권한 조회 |

View는 화면 조회 편의를 위해 사용할 수 있으나, 복잡한 업무 로직을 View에 숨기면 안 된다. 업무 판단은 Rule/Service에서 수행하고, View는 조회 편의와 보안 마스킹 목적에 한정한다.

## 13. Synonym 명명규칙

### 13.1 기본 형식

SYN_{원천스키마}_{객체명}

| Synonym 명 | 설명 |
| --- | --- |
| SYN_RDW_CUSTOMER_MASTER | RDW 고객 마스터 접근 |
| SYN_ADW_CAMPAIGN_SUMMARY | ADW 캠페인 집계 접근 |
| SYN_SESSION_SPRING_SESSION | SESSIONDB 세션 테이블 접근 |

Synonym은 스키마 경계를 숨기는 기능이므로 남용하지 않는다. RDW, ADW, SESSIONDB, LOGDB 접근 책임은 DataSource와 Mapper 기준으로 분리한다.

## 14. Procedure / Function / Package 명명규칙

NSIGHT 온라인 업무에서는 가능한 한 Java Service + MyBatis SQL 중심으로 처리하고, DB Procedure는 제한적으로 사용한다.

| --- | --- | --- |
| Procedure | PRC_{업무코드}_{처리대상}_{행위} | PRC_CM_CAMPAIGN_CLOSE |

Function
FN_{업무코드}_{계산대상}
FN_SV_MASK_CUSTOMER_NAME
Package
PKG_{업무코드}_{업무영역}
PKG_CM_CAMPAIGN

TRG_{테이블명}_{시점}_{행위}

| BD | Before Delete | AI |

## 18. MyBatis / Java 매핑 기준

DB 컬럼은 UPPER_SNAKE_CASE, Java 필드는 lowerCamelCase로 매핑한다.

| DB Column | Java Field |
| --- | --- |
| SERVICE_ID | serviceId |
| TRANSACTION_CODE | transactionCode |
| BUSINESS_CODE | businessCode |
| USER_ID | userId |
| BRANCH_ID | branchId |
| CREATED_DTM | createdDtm |
| USE_YN | useYn |
| RESULT_STATUS | resultStatus |

MyBatis 운영 기준에서는 Mapper XML 위치, SQL ID, Query Timeout, RDW/ADW 접근 경계, SQL 추적성을 통제해야 한다. 따라서 DB 객체명은 Mapper XML과 SQL ID에서도 동일한 업무 의미를 유지해야 한다.

## 20. 마무리말

DB 객체 명명규칙은 개발 편의 기준이 아니라 운영 가능성을 확보하기 위한 기준이다.NSIGHT TCF에서는 모든 온라인 거래가 ServiceId, 거래코드, 업무코드, GUID, TraceId를 중심으로 추적되므로 DB 객체명도 이 흐름을 방해하지 않아야 한다.
핵심 기준은 다음과 같다.
DB 객체명은
업무코드와 영역을 드러내고,
Java / MyBatis / SQL ID와 연결되며,
운영자가 장애 상황에서 바로 추적할 수 있어야 한다.

따라서 NSIGHT DB 객체는 UPPER_SNAKE_CASE, 업무/영역 Prefix, 표준 suffix, 제약조건/인덱스 표준명을 일관되게 적용한다. 이렇게 해야 개발, 운영, DBA, 아키텍트, PMO가 같은 이름을 보고 같은 의미로 이해할 수 있다.
