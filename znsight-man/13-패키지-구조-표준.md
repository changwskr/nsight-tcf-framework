# 13. 패키지 구조 표준

> **NSIGHT TCF 개발 Manual** · 원본: [`znsight-guide`](../znsight-guide/) · 갱신: 2026-07-05

## 13. 패키지 구조 표준

### 13.1 도입 전 안내말

본 장은 NSIGHT TCF Framework에서 업무 서비스, 공통 모듈, OM, Gateway, JWT, Batch를 어떤 Java Package 구조로 구성해야 하는지 정의한다.
패키지 구조는 단순한 폴더 정리가 아니다.NSIGHT에서는 패키지 구조가 곧 계층구조, 업무 경계, 공통/업무 분리, ServiceId 개발 위치, Mapper 관리 위치, 테스트 범위를 결정한다.
따라서 개발자는 개인 취향에 따라 패키지를 만들면 안 되고, 아래 표준 구조를 따라야 한다.
업무코드
```text
   ↓
Base Package
↓
계층 Package
↓
Class Naming
↓
ServiceId 구현
↓

```

Mapper / SQL 연결

### 13.2 패키지 구조 결론

NSIGHT 업무 서비스 패키지는 다음 구조를 표준으로 한다.
com.nh.nsight.{업무코드소문자}
```text
 ├─ entry
 │   ├─ handler
 │   └─ facade
 ├─ application
 │   ├─ service
 │   └─ rule
 ├─ persistence
 │   ├─ dao
 │   └─ mapper
 ├─ dto
 │   ├─ request
 │   ├─ response
 │   ├─ command
 │   └─ result
 ├─ client
 ├─ support
 ├─ config
 ├─ exception
 └─ constant
```

한 문장으로 정리하면 다음과 같다.
패키지는 업무코드 아래에 두고, 계층은 Handler, Facade, Service, Rule, DAO, Mapper 순서로 분리한다.

### 13.3 기본 패키지 명명 원칙

| 구분 | 기준 | 예시 |
| --- | --- | --- |
| 최상위 Package | com.nh.nsight | 고정 |
| 업무 Package | com.nh.nsight.{업무코드소문자} | com.nh.nsight.sv |
| 공통 TCF Package | com.nh.nsight.tcf.{모듈} | com.nh.nsight.tcf.core |
| OM Package | com.nh.nsight.om | 운영관리 |
| Gateway Package | com.nh.nsight.gateway | 라우팅 |
| JWT Package | com.nh.nsight.jwt | 토큰 |
| Batch Package | com.nh.nsight.batch | 배치 |
| 패키지명 | 소문자 사용 | customer, service, mapper |
| 클래스명 | PascalCase | SvCustomerService |
| 메서드명 | camelCase | selectCustomerSummary |

| 패키지 | 역할 | 대표 클래스 |
| --- | --- | --- |
| entry.handler | TCF Dispatcher가 호출하는 업무 진입점 | SvCustomerSummaryHandler |
| entry.facade | 유스케이스 조립, 트랜잭션 경계 | SvCustomerFacade |
| application.service | 업무 처리 흐름 제어 | SvCustomerService |
| application.rule | 업무 규칙, 검증, 계산 | SvCustomerRule |
| persistence.dao | Mapper 호출 캡슐화 | SvCustomerDao |
| persistence.mapper | MyBatis Mapper Interface | SvCustomerMapper |
| dto.request | 외부 요청 Body | SvCustomerSummaryRequest |
| dto.response | 외부 응답 Body | SvCustomerSummaryResponse |
| dto.command | 내부 처리 요청 | SvCustomerSummaryCommand |
| dto.result | 내부 처리 결과 | SvCustomerSummaryResult |
| client | 다른 업무 또는 외부 서비스 호출 | SvIntegrationClient |
| support | 업무 내부 지원 기능 | SvCustomerSupport |
| config | 업무별 Spring 설정 | SvMyBatisConfig |
| exception | 업무 예외 | SvBusinessException |
| constant | 업무 상수 | SvConstants |

### 13.6 패키지와 계층 호출 관계

패키지 호출 방향은 다음과 같이 위에서 아래로만 흐른다.
```text
entry.handler
   ↓
entry.facade
   ↓
application.service
   ↓
application.rule
   ↓
persistence.dao
   ↓
persistence.mapper
```

허용되는 호출 관계는 다음과 같다.
| 호출 관계 | 허용 여부 | 설명 |
| --- | --- | --- |
| handler → facade | 허용 | 표준 진입 흐름 |
| facade → service | 허용 | 유스케이스 처리 |
| service → rule | 허용 | 업무 판단 |
| service → dao | 허용 | DB 조회·저장 |
| dao → mapper | 허용 | SQL 실행 |
| service → client | 허용 | 서비스 간 연동 |
| handler → dao | 금지 | 계층 우회 |
| handler → mapper | 금지 | SQL 직접 접근 |
| rule → mapper | 금지 | Rule이 DB에 종속됨 |
| dao → service | 금지 | 역방향 호출 |
| mapper → service | 금지 | 역방향 호출 |
| 업무 패키지 → 다른 업무 패키지 직접 참조 | 금지 | WAR 독립성 훼손 |

### 13.7 업무 도메인별 하위 패키지 분리 기준

업무가 커질 경우 계층 아래에 도메인 단위 하위 패키지를 둘 수 있다.
예를 들어 SV 업무에 고객, 상품, 거래내역 도메인이 있다면 다음처럼 구성할 수 있다.

```text
com.nh.nsight.sv
 ├─ entry
 │   ├─ handler
 │   │   ├─ customer
 │   │   ├─ product
 │   │   └─ transaction
 │   └─ facade
 │       ├─ customer
 │       ├─ product
 │       └─ transaction
 │
 ├─ application
 │   ├─ service
 │   │   ├─ customer
 │   │   ├─ product
 │   │   └─ transaction
 │   └─ rule
 │       ├─ customer
 │       ├─ product
 │       └─ transaction
 │
 └─ persistence
     ├─ dao
     │   ├─ customer
     │   ├─ product
     │   └─ transaction
     └─ mapper
         ├─ customer
         ├─ product
         └─ transaction
```

| 분리 기준 | 설명 |
| --- | --- |
| 업무 기능이 많을 때 | 고객, 상품, 거래, 캠페인 등으로 분리 |
| ServiceId가 많을 때 | 도메인별 Handler 관리 |
| Mapper가 많을 때 | SQL 파일 관리 편의 |
| 담당자가 나뉠 때 | 개발 충돌 최소화 |
| 공통 Rule이 많을 때 | 도메인별 Rule 분리 |

단, 처음부터 과도하게 세분화하지 않는다.작은 업무는 계층 패키지만 두고, 업무가 커질 때 도메인 하위 패키지를 추가한다.

### 13.8 DTO 패키지 표준

DTO는 목적별로 분리한다.

```text
dto
 ├─ request
 ├─ response
 ├─ command
 ├─ result
 └─ entity
```

| DTO 유형 | 사용 구간 | 설명 |
| --- | --- | --- |
| request | Handler 입력 | 표준 전문 Body를 Java 객체로 변환 |
| response | Handler 출력 | 표준 전문 Body 응답 |
| command | Facade → Service | 내부 업무 처리 요청 |
| result | Service → Facade | 내부 업무 처리 결과 |
| entity | DAO / Mapper | DB 조회 결과 또는 테이블 구조 |

예시는 다음과 같다.
com.nh.nsight.sv.dto.request.SvCustomerSummaryRequest
com.nh.nsight.sv.dto.response.SvCustomerSummaryResponse
com.nh.nsight.sv.dto.command.SvCustomerSummaryCommand
com.nh.nsight.sv.dto.result.SvCustomerSummaryResult
com.nh.nsight.sv.dto.entity.SvCustomerEntity

DTO 사용 원칙은 다음과 같다.
| 원칙 | 설명 |
| --- | --- |
| Request를 Service까지 끌고 가지 않음 | 외부 요청 구조와 내부 처리 구조 분리 |
| Response를 DAO에서 만들지 않음 | DB 결과와 외부 응답 구조 분리 |
| Entity를 화면 응답으로 직접 반환하지 않음 | DB 구조 노출 방지 |
| Command는 내부 처리 중심 | userId, branchCode, 권한정보 등 Context 포함 가능 |
| Result는 응답 조립 전 단계 | 여러 DAO 결과를 모은 내부 결과 |

### 13.9 Mapper XML 경로 표준

Java Mapper Interface는 persistence.mapper에 두고, XML은 resources/mapper/{업무코드} 아래에 둔다.

```text
src/main/java
 └─ com/nh/nsight/sv/persistence/mapper
    └─ SvCustomerMapper.java
```


```text
src/main/resources
 └─ mapper
    └─ sv
       └─ SvCustomerMapper.xml
```

Mapper XML의 namespace는 Java Mapper Interface와 일치해야 한다.
<mapper namespace="com.nh.nsight.sv.persistence.mapper.SvCustomerMapper">

| 항목 | 기준 |
| --- | --- |
| Java Mapper 위치 | persistence.mapper |
| XML 위치 | resources/mapper/{업무코드} |
| XML 파일명 | {업무}{도메인}Mapper.xml |
| namespace | Mapper Interface FQCN |
| SQL ID | Mapper 메서드명과 동일 |
| 업무별 SQL 분리 | 업무코드 디렉터리 기준 |

### 13.11 공통 TCF 모듈 패키지 구조

공통 모듈은 업무코드를 포함하지 않고 com.nh.nsight.tcf 아래에 둔다.
com.nh.nsight.tcf
```text
 ├─ util
 ├─ core
 ├─ web
 ├─ cache
 ├─ integration
 ├─ exception
 ├─ logging
 ├─ context
 └─ security
```

| 공통 패키지 | 역할 |
| --- | --- |
| tcf.util | 공통 유틸, 마스킹, 날짜, GUID |
| tcf.core | 표준 전문, TCF 처리, Dispatcher |
| tcf.web | 공통 Controller, Filter, Interceptor |
| tcf.cache | Cache Manager, Cache Policy |
| tcf.integration | 서비스 간 호출 Client |
| tcf.exception | 공통 예외 |
| tcf.logging | MDC, 거래로그, 감사로그 |
| tcf.context | TransactionContext |
| tcf.security | 인증, 권한, 세션 보조 기능 |
공통 모듈에는 특정 업무 로직을 넣지 않는다.

| 금지 예 | |
| 사유 | com.nh.nsight.tcf.core.sv |
| 공통 Core가 SV 업무에 종속됨 | tcf.util.CustomerGradeRule |
| 고객등급 업무 규칙은 업무 Rule이어야 함 | tcf.web.SvController |
| 공통 Web에 특정 업무 Controller 금지 | tcf.cache.SvCustomerCache |
| 업무 데이터 Cache 금지 |  |

### 13.13 Gateway 패키지 구조

Gateway는 업무를 직접 처리하지 않고 라우팅과 인증·검증을 담당한다.
com.nh.nsight.gateway
```text
 ├─ entry
 │   └─ controller
 ├─ application
 │   ├─ service
 │   └─ rule
 ├─ routing
 │   ├─ resolver
 │   ├─ repository
 │   └─ policy
 ├─ client
 ├─ filter
 ├─ dto
 ├─ config
 └─ support
```

| 패키지 | 역할 |
| --- | --- |
| entry.controller | Gateway HTTP 진입점 |
| routing.resolver | 업무코드 기준 라우팅 대상 결정 |
| routing.repository | 라우팅 테이블 조회 |
| routing.policy | 환경별 라우팅 정책 |
| client | Downstream 업무 WAR 호출 |
| filter | 인증, Header, TraceId 처리 |
| config | Gateway 설정 |
Gateway에는 업무 Service, 업무 Rule, 업무 DAO를 두지 않는다.

### 13.14 JWT 패키지 구조

JWT 모듈은 토큰 발급, 검증, 저장, 폐기를 담당한다.
com.nh.nsight.jwt
```text
 ├─ entry
 │   ├─ handler
 │   └─ controller
 ├─ application
 │   ├─ service
 │   └─ rule
 ├─ token
 │   ├─ issuer
 │   ├─ verifier
 │   ├─ store
 │   └─ policy
 ├─ persistence
 │   ├─ dao
 │   └─ mapper
 ├─ dto
 ├─ config
 └─ support
```

| 패키지 | 역할 |
| --- | --- |
| token.issuer | Access Token, Refresh Token 발급 |
| token.verifier | 토큰 검증 |
| token.store | 토큰 저장·조회 |

### 13.18 Import 금지 기준

다음 Import는 금지한다.
| 금지 Import | 사유 |
| --- | --- |
| com.nh.nsight.sv.*를 ic-service에서 직접 Import | 업무 WAR 간 직접 결합 |
| com.nh.nsight.om.*를 업무 서비스에서 직접 Import | OM 구현 종속 |
| persistence.mapper.*를 Handler에서 Import | 계층 우회 |
| persistence.mapper.*를 Rule에서 Import | Rule이 DB 종속 |
| entry.handler.*를 Service에서 Import | 역방향 의존 |
| 운영 Secret 관련 Class 직접 참조 | 보안 책임 혼재 |

업무 간 호출은 Java Import가 아니라 tcf-integration 기반 HTTP/JSON 호출로 처리한다.

### 13.19 패키지 구조 금지사항

| 금지사항 | 사유 |
| --- | --- |
| controller 패키지에 업무 Controller 남발 | TCF 표준 진입점 우회 |
| Handler에서 Mapper 직접 호출 | 계층 책임 위반 |
| common 패키지에 모든 것을 넣기 | 책임 불명확 |
| util 패키지에 업무 로직 넣기 | 테스트·유지보수 어려움 |
| dto 하나에 Request/Response/Entity 혼재 | 외부·내부·DB 구조 결합 |
| 업무별 Mapper XML을 공통 폴더에 혼재 | SQL 관리 어려움 |
| 공통 TCF 모듈에 업무 Package 생성 | 공통 오염 |
| 업무 WAR 간 Package 직접 참조 | 독립 배포 불가 |
| 도메인 없는 service1, service2 패키지 | 의미 파악 어려움 |
| 개인명 또는 임시명 패키지 사용 | 표준 위반 |

### 13.20 패키지 구조 점검 체크리스트

| 점검 항목 | 확인 |
| --- | --- |
| Base Package가 com.nh.nsight.{업무코드} 기준인가 | □ |
| 업무코드는 소문자로 사용했는가 | □ |
| entry.handler와 entry.facade가 분리되어 있는가 | □ |
| application.service와 application.rule이 분리되어 있는가 | □ |
| persistence.dao와 persistence.mapper가 분리되어 있는가 | □ |
| DTO가 request/response/command/result로 분리되어 있는가 | □ |
| Mapper XML이 resources/mapper/{업무코드} 아래에 있는가 | □ |
| Mapper namespace가 Java Mapper FQCN과 일치하는가 | □ |
| Handler가 DAO/Mapper를 직접 Import하지 않는가 | □ |
| Rule이 Mapper를 직접 Import하지 않는가 | □ |
| 업무 WAR 간 Java 직접 참조가 없는가 | □ |
| 공통 모듈에 업무 로직이 들어가지 않았는가 | □ |
| 테스트 패키지가 운영 패키지 구조와 일치하는가 | □ |

### 13.21 마무리말

패키지 구조는 개발자가 자유롭게 정하는 폴더 구조가 아니다.NSIGHT TCF Framework에서는 패키지 구조가 곧 아키텍처 규칙을 코드로 강제하는 수단이다.
패키지 구조가 표준화되면 다음 효과가 있다.
| 효과 | 설명 |
| --- | --- |
| 신규 개발자 이해 용이 | 어디에 어떤 클래스를 작성해야 하는지 명확 |
| 코드 리뷰 효율 향상 | 계층 위반 여부를 쉽게 확인 |
| 장애 추적 용이 | 오류 발생 계층을 빠르게 식별 |
| 테스트 작성 용이 | Rule, Service, Mapper 단위 테스트 분리 |
| 업무 WAR 독립성 확보 | 업무 간 직접 참조 방지 |
| 공통 Framework 보호 | 업무 로직이 공통 모듈에 섞이지 않음 |

### 13.22 시사점

NSIGHT의 패키지 구조 표준은 다음 원칙을 지키기 위한 것이다.
업무는 업무 Package에 둔다.
공통은 tcf Package에 둔다.
진입은 entry에 둔다.
업무 흐름은 application에 둔다.
DB 접근은 persistence에 둔다.
데이터 전달은 dto에 둔다.
외부 호출은 client에 둔다.

결론적으로 다음 한 문장으로 정리할 수 있다.
패키지 구조는 계층구조를 코드에 고정시키는 표준이며, 개발자는 패키지를 보면 해당 클래스의 책임을 즉시 알 수 있어야 한다.
