# 42. 보안 코딩 기준

> **NSIGHT TCF 개발 Manual** · 원본: [`znsight-guide`](../znsight-guide/) · 갱신: 2026-07-05

## 42. 보안코딩 기준

### 42.1 도입 전 안내말

보안코딩은 개발 마지막에 취약점만 점검하는 활동이 아니다.NSIGHT TCF Framework에서는 입력값 검증, 인증·권한, 세션/JWT, SQL 작성, 로그 마스킹, 오류응답, 파일처리, 외부연계, 설정정보 관리가 모두 코드 작성 단계에서 통제되어야 한다.
NSIGHT Java 코딩 스타일 기준에서도 코딩 스타일은 단순 문법 규칙이 아니라 업무 책임 경계, 데이터 접근 통제, 오류 추적성, 보안 감사, 운영 가능성을 코드 수준에서 일관되게 보장하는 표준이며, 최상위 원칙 중 하나로 민감정보를 코드·로그·응답에 노출하지 않고, 마스킹·접근권한·입력검증·SQL Injection 방지를 적용해야 한다고 정리되어 있다.
핵심 문장은 다음이다.
NSIGHT 보안코딩은 “취약점 제거”가 아니라, 모든 온라인 거래가 TCF 전처리·업무계층·후처리를 통과하면서 안전하게 검증·실행·기록되도록 만드는 개발 기준이다.

### 42.2 보안코딩 기준 결론

| 구분 | 기준 |
| --- | --- |
| 기본 원칙 | 입력은 신뢰하지 않고, 출력은 마스킹하며, 권한은 서버에서 검증한다 |
| 적용 위치 | Controller, STF, Handler, Facade, Service, Rule, DAO/Mapper, Adapter, ETF |
| 인증 기준 | Spring Session JDBC, JWT, SSO 인증 결과를 서버에서 재검증 |
| 권한 기준 | ServiceId + 메뉴 + 기능권한 + 데이터권한 + 거래통제 기준 |
| 입력 검증 | DTO Bean Validation + Rule 업무검증 + Header 검증 |
| SQL 보안 | MyBatis #{} 사용, ${} 금지 또는 Whitelist 제한 |
| 로그 보안 | GUID/TraceId 중심 기록, 개인정보·Token·전문 원문 로그 금지 |
| 응답 보안 | StackTrace, SQL, 서버 IP, 내부 테이블명 노출 금지 |
| 파일 보안 | 확장자, MIME, 크기, 경로, 바이러스, 다운로드 권한 검증 |
| 외부연계 보안 | 허용된 URL, Timeout, 인증 Header, 응답 검증 적용 |
| Secret 관리 | application.yml 직접 저장 금지, 환경변수/Secret 저장소 사용 |
| 검증 도구 | SonarQube, Checkstyle, PMD, Dependency Check, 단위·통합 테스트 |

### 42.3 보안코딩 적용 구조

```text
[Client / WebTopSuite / API]
        ↓
[Controller / Gateway]
  - Content-Type 검증
  - 요청 크기 제한
  - 표준 전문 수신
        ↓

```

```text
[STF.preProcess()]
  - Header 필수값 검증
  - GUID / TraceId 생성
  - 세션 검증
  - JWT Claim 검증
  - 권한 검증
  - 거래통제
  - Idempotency
  - Timeout 정책 조회
        ↓
[Handler]
  - 요청 Body → 업무 DTO 변환
  - 직접 권한 판단 금지
        ↓

```

```text
[Facade / Service / Rule]
  - 업무 입력값 검증
  - 데이터권한 검증
  - 업무 규칙 검증
        ↓
[DAO / Mapper]
  - MyBatis 파라미터 바인딩
  - SQL Injection 방지
  - Query Timeout
        ↓

```

```text
[ETF.postProcess()]
  - 표준 응답 조립
  - 오류코드 매핑
  - 개인정보 마스킹
  - 감사로그 / 거래로그
```

TCF 전문처리 구조는 STF.preProcess()에서 Header 검증, GUID/TraceId 생성, 세션·인증·권한 검증, 거래통제, Idempotency, Timeout, 거래 시작 로그를 수행하고, 후처리에서 표준 응답 조립, 오류코드 매핑, 데이터 마스킹, 감사로그, 성능 메트릭, 거래상태 갱신을 수행하는 구조로 정의되어 있다.

### 42.5 입력값 검증 기준

입력값 검증은 Controller에서 끝나지 않는다.NSIGHT는 전문 Header 검증 + DTO 형식 검증 + Rule 업무 검증을 분리한다.
| 검증 단계 | 위치 |
| --- | --- |
| 검증 내용 | Header 검증 |
| STF | serviceId, businessCode, transactionCode, user, channelId, branch |
| 형식 검증 | DTO |
| 필수값, 길이, 날짜형식, 숫자범위, 코드값 | 업무 검증 |
| Rule | 업무 상태, 권한범위, 데이터범위, 처리 가능 여부 |
| DB 검증 | DAO/Mapper |
| 존재 여부, 중복 여부, 참조 무결성 | 외부연계 검증 |
| Adapter | 요청값, 응답값, 상태코드, 전문 구조 |

### 42.6 Header 보안 검증 기준

전문 Header는 업무 실행의 핵심 기준이므로 반드시 서버에서 재검증한다.
| Header 항목 | 보안 검증 기준 | serviceId |
| --- | --- | --- |
| OM_SERVICE_CATALOG 등록 여부, 사용 여부 확인 | businessCode | URL Path 업무코드와 일치하는지 확인 |
| transactionCode | ServiceId에 등록된 거래코드와 일치하는지 확인 | user |
| SessionContext 또는 JWT Claim과 일치하는지 확인 | branch | SessionContext 또는 데이터권한 범위와 일치하는지 확인 |
| channelId | 허용 채널인지 확인 | menuId |
| 메뉴-ServiceId 매핑 확인 | idempotencyKey | 등록·변경·삭제 거래는 필수 |

guid, traceId
없으면 서버에서 생성, 있으면 형식 검증
JWT 적용 시에도 Header 값을 그대로 신뢰하면 안 된다. JWT 설계 기준에서도 JWT 인증 이후 TCF STF 전처리에서 JWT 인증 결과 확인, 전문 Header 검증, Token Claim과 Header 비교, ServiceId 권한 확인, 거래통제 확인을 수행하도록 정리되어 있다.

### 42.7 SQL Injection 방지 기준

MyBatis Mapper에서는 문자열 결합을 금지하고, 기본적으로 #{} 파라미터 바인딩을 사용한다.
| 항목 | 기준 |
| --- | --- |
| 일반 조건 | #{} 사용 |
| 동적 정렬 | Whitelist 검증 후 제한적으로만 허용 |
| 테이블명 동적 변경 | 원칙적 금지 |
| 컬럼명 동적 변경 | Whitelist 방식만 허용 |
| LIKE 검색 | 입력값 Escape 처리 |
| IN 조건 | Collection 바인딩 사용 |
| SQL 로그 | 바인딩값 중 개인정보 마스킹 |
| Query Timeout | 온라인 조회는 짧게 통제 |
| 좋은 예 | <select id="selectCustomerSummary" |
| parameterType="SvCustomerSummaryQuery" | resultType="SvCustomerSummary"> |
| SELECT | CUSTOMER_NO, |
| CUSTOMER_NAME, | BRANCH_ID, |
| CUSTOMER_GRADE | FROM SV_CUSTOMER_SUMMARY |
| WHERE CUSTOMER_NO = #{customerNo} | AND BRANCH_ID = #{branchId} |
| </select> |  |
| 나쁜 예 | <select id="selectCustomerSummary" |
| parameterType="map" | resultType="SvCustomerSummary"> |
| SELECT * | FROM SV_CUSTOMER_SUMMARY |
| WHERE CUSTOMER_NO = '${customerNo}' | </select> |

'${customerNo}'는 문자열 치환이므로 SQL Injection 위험이 있다.업무 Mapper에서는 기본적으로 금지한다.
동적 정렬 허용 예시
```java
public class SortWhitelist {
    private static final Map<String, String> ALLOWED_COLUMNS = Map.of(
            "customerName", "CUSTOMER_NAME",
            "baseDate", "BASE_DATE",
            "branchId", "BRANCH_ID"
    );
    public String resolveColumn(String sortKey) {
        String column = ALLOWED_COLUMNS.get(sortKey);
        if (column == null) {
            throw new BusinessException(
                    "E-SEC-SQL-0001",
                    "허용되지 않은 정렬 조건입니다."
            );
        }
        return column;
    }
}
```

### 42.8 XSS / 응답 데이터 보안 기준

NSIGHT는 주로 HTTP/JSON 전문 방식이지만, 화면에 표시되는 문자열은 XSS 위험을 고려해야 한다.
| 구분 | 기준 |
| --- | --- |
| JSON 응답 | HTML 실행 가능 문자열을 그대로 화면에 출력하지 않도록 UI에서 Escape |
| 서버 응답 | 사용자 입력값을 메시지에 그대로 포함하지 않음 |
| 오류 메시지 | 표준 메시지만 반환 |
| 파일명 | 다운로드 Header에 들어가는 파일명 검증 |
| HTML 생성 | 서버에서 HTML 조립 지양 |
| 스크립트 | 사용자 입력으로 Script 생성 금지 |
| 금지 예시 | throw new BusinessException( |
| "E-SV-VAL-0001", | "입력값 오류: " + request.getMemo() |
| ); |  |
| 권장 예시 | throw new BusinessException( |
| "E-SV-VAL-0001", | "입력값이 올바르지 않습니다." |
| ); |  |
운영 로그에는 필요한 경우 원문 대신 마스킹 또는 요약값을 남긴다.

### 42.9 인증 / 권한 보안 기준

인증과 권한은 업무 Service에 흩뿌리지 않고 TCF 공통 전처리에서 수행한다.
| 보안 항목 | 기준 |
| --- | --- |
| 세션 검증 | Spring Session JDBC 기준 |
| JWT 검증 | 서명, 만료, issuer, audience, denylist 확인 |
| Header-Session 비교 | 사용자, 지점, 채널 비교 |
| Header-Token 비교 | sub, branchId, channelId 비교 |
| ServiceId 권한 | OM_SERVICE_CATALOG 기준 |
| 메뉴권한 | OM_MENU, OM_MENU_SERVICE_MAP 기준 |
| 기능권한 | 조회, 등록, 수정, 삭제, 다운로드, 업로드, 실행, 승인 |
| 데이터권한 | 지점, 고객등급, 채널, 다운로드 권한 |
| 거래통제 | TCF_TRANSACTION_CONTROL 기준 |

NSIGHT TCF는 JWT를 인증 수단으로 사용하더라도 TCF가 거래통제·권한·로그·감사·Timeout·표준응답을 계속 담당하는 구조이다.

### 42.11 JWT / Token 보안 기준

| 항목 | 기준 |
| --- | --- |
| Access Token | 짧게 유지, 기본 15분 |
| Refresh Token | 서버 DB에 Hash 저장 |
| Token 원문 저장 | 금지 |
| JWT Claim | 최소 정보만 포함 |
| 고객정보 Claim | 금지 |
| Token 로그 | 금지 |
| Token 폐기 | jti 기준 Denylist 또는 Token DB 상태 관리 |
| 내부 발급 API | 외부 직접 호출 금지 |
| 알고리즘 | 운영 표준 알고리즘 고정, none 허용 금지 |
| Header 비교 | Token Claim과 전문 Header 반드시 비교 |

JWT 설계 기준에서도 Access Token은 짧게, Refresh Token은 서버에서 Hash로 관리하며, JWT에는 사용자ID·지점·채널·권한그룹·만료시간 같은 최소 정보만 저장하고 고객명·고객번호·계좌·조회결과는 저장하지 않도록 정리되어 있다.

### 42.12 로그 / 감사 보안 기준

로그는 디버깅 출력이 아니라 거래 증적이다.NSIGHT 로그는 GUID 기준으로 추적 가능해야 하지만, 개인정보 원문이나 Token을 남기면 안 된다.

| 로그 유형 | 필수 필드 | 보안 주의사항 |
| --- | --- | --- |
| 거래로그 | guid, traceId, serviceId, userId, branchId, elapsedMs, resultCode | 개인정보 원문 금지 |

오류로그
guid, errorCode, exceptionType, message

| 사용자 메시지와 운영 메시지 분리 | SQL 로그 |
| --- | --- |
| guid, sqlId, elapsedMs, rowCount | 바인딩값 개인정보 마스킹 |
| 연계로그 | guid, interfaceId, targetSystem, timeout, retryCount |
| 요청/응답 전문 원문 저장 통제 | 감사로그 |
| userId, roleId, accessPurpose, dataGrade, maskingLevel | 고객정보 조회/다운로드 필수 |

NSIGHT Java 코딩 스타일 기준에서도 사용자 응답에는 SQL, 서버 IP, Stack Trace, 개인정보를 노출하지 않고, 운영 로그에는 GUID, traceId, errorCode, interfaceId, sqlId를 함께 남기며, SQL 로그의 바인딩값 중 개인정보는 마스킹하도록 정리되어 있다.
로그 작성 예시
MDC.put("guid", context.getGuid());
MDC.put("serviceId", context.getServiceId());
MDC.put("userId", MaskingUtils.maskUserId(context.getUserId()));

try {
    log.info("SV customer summary inquiry started");
    return service.selectCustomerSummary(request, context);
} finally {
    MDC.clear();
}

금지 예시
log.info("request body = {}", request);
log.info("accessToken = {}", accessToken);

### 42.20 개발자 구현 예시

#### 42.20.1 Security Validator

```java
@Component
@RequiredArgsConstructor
public class SecurityValidator {
    private final ServiceCatalogService serviceCatalogService;
    private final AuthorizationValidator authorizationValidator;
    public SecurityValidationResult validate(
            StandardRequest<?> request,
            SessionContext session,
            JwtPrincipal jwtPrincipal
    ) {
        StandardHeader header = request.getHeader();
        validateHeaderConsistency(header, session, jwtPrincipal);
        validateServiceCatalog(header);
        AuthorizationResult authorization =
                authorizationValidator.validate(header, session);
        return SecurityValidationResult.builder()
                .authorizationResult(authorization)
                .build();
    }
    private void validateHeaderConsistency(
            StandardHeader header,
            SessionContext session,
            JwtPrincipal jwtPrincipal
    ) {
        if (!header.getUser().equals(session.getUserId())) {
            throw new BusinessException(
                    "E-SEC-HDR-0001",
                    "사용자 정보가 일치하지 않습니다."
            );
        }
        if (jwtPrincipal != null && !header.getUser().equals(jwtPrincipal.getUserId())) {
            throw new BusinessException(
                    "E-SEC-TOKEN-0001",
                    "Token 사용자 정보가 일치하지 않습니다."
            );
        }
    }
    private void validateServiceCatalog(StandardHeader header) {
        ServiceCatalog service =
                serviceCatalogService.getEnabledService(header.getServiceId());
        if (!service.getBusinessCode().equals(header.getBusinessCode())) {
            throw new BusinessException(
                    "E-SEC-HDR-0001",
                    "업무코드가 일치하지 않습니다."
            );
        }
    }
}
```

### 42.24 마무리말

보안코딩 기준의 핵심은 개발자가 각자 보안 로직을 흩어서 구현하는 것이 아니라, TCF 공통 파이프라인에서 인증·권한·거래통제·마스킹·오류응답·로그 기준을 일관되게 적용하는 것이다.
업무 개발자는 기능 구현과 함께 다음 질문을 항상 확인해야 한다.
이 입력값은 검증되었는가?
이 사용자는 이 ServiceId를 실행할 권한이 있는가?
이 데이터는 이 사용자에게 보여도 되는가?
이 오류가 사용자에게 내부정보를 노출하지 않는가?
이 로그가 개인정보나 Token을 남기지 않는가?
