농협 상호금융 NSIGHT 프로젝트

Architecture Gate 단계별 체크리스트

1. 도입 전 안내말

Architecture Gate는 산출물 작성 여부를 확인하는 문서검수 회의가 아니다.

각 단계에서 결정해야 할 사항이 충분히 확정되었는지, 다음 단계로 넘어갔을 때 구조를 다시 설계할 위험이 없는지, 운영 가능한 시스템으로 이어질 수 있는지를 판단하는 공식 의사결정 절차다.

```
Architecture Gate가 확인해야 하는 것

요구사항이 존재하는가
→ 설계에 반영되었는가
→ 코드와 설정으로 구현되었는가
→ OM 기준정보에 등록되었는가
→ 배포구조에 반영되었는가
→ 시험으로 검증되었는가
→ 운영자가 탐지·대응·복구할 수 있는가
```

NSIGHT TCF는 다음 구조 전체가 연결되는 온라인 거래 플랫폼이다.

```
사용자·UI
  ↓
GSLB·L4·Apache·Gateway
  ↓
업무 WAR
  ↓
OnlineTransactionController
  ↓
TCF → STF → TimeoutExecutor → Dispatcher
  ↓
Handler → Facade → Service → Rule → DAO → Mapper
  ↓
ETF
  ↓
거래로그·감사로그·OM 운영관리
```

따라서 Architecture Gate에서는 다음 정합성을 핵심으로 확인해야 한다.

```
ServiceId
↔ Handler
↔ 화면 이벤트
↔ 거래코드
↔ OM Service Catalog
↔ 거래통제·Timeout
↔ 프로그램·SQL
↔ 테스트케이스
↔ 거래로그·감사로그
```

TCF는 표준 전문 검증, 인증 문맥 확인, 거래통제, Timeout, 중복요청, ServiceId 기반 Handler 실행, 오류 표준화와 거래로그 종료를 공통으로 통제해야 한다.

2. 문서 개요

2.1 목적

본 체크리스트의 목적은 NSIGHT 프로젝트의 G0~G14 단계별 Architecture Gate를 객관적으로 수행하고 다음 단계 진입 여부를 결정하는 것이다.

| 목적 | 설명 |
| --- | --- |
| 조기 위험 제거 | 개발·시험·운영 이전에 구조적 결함 발견 |
| 단계 통제 | 선행조건이 충족되지 않은 상태의 다음 단계 진입 방지 |
| 책임 명확화 | 작성·검토·조치·승인 책임 구분 |
| 증적 중심 검토 | 설명이나 구두 합의가 아닌 검증 가능한 증적 확인 |
| 추적성 확보 | 요구사항부터 운영까지 연결관계 확인 |
| 운영 가능성 검증 | 배포·관측·장애대응·복구 가능 여부 확인 |
| 변경 통제 | 승인 이후 변경의 영향도와 예외 관리 |
| 프로젝트 품질관리 | Gate 결과를 PMO 진척과 위험관리에 반영 |

3. Gate 판정체계

3.1 점검 결과

| 결과 | 의미 |
| --- | --- |
| Y | 충족, 증적 확인 완료 |
| P | 부분 충족, 보완 진행 중 |
| N | 미충족 |
| N/A | 적용 대상 아님. 사유 기록 필수 |
| V | 별도 검증 또는 재시험 필요 |

3.2 결함 심각도

| 등급 | 명칭 | 기준 | Gate 영향 |
| --- | --- | --- | --- |
| B | Blocker | 다음 단계 진행 시 프로젝트 실패·보안사고·데이터 손상 가능 | Gate 통과 불가 |
| C | Critical | 주요 구조·성능·가용성·운영 위험 | 원칙적으로 통과 불가 |
| M | Major | 품질·유지보수·운영 효율 저하 | 조건부 통과 가능 |
| m | Minor | 문서 보완·표현·경미한 누락 | 후속조치 등록 |
| O | Observation | 개선 권고 | 통과 영향 없음 |

3.3 Gate 판정

| 판정 | 조건 |
| --- | --- |
| 적합 | Blocker·Critical 0건, 필수항목 충족, 증적 확인 완료 |
| 조건부 적합 | Blocker 0건, Critical 0건, Major에 담당자·완료일·검증자 지정 |
| 재검토 | 핵심 증적 미제출 또는 주요 검증 미완료 |
| 부적합 | Blocker 또는 미조치 Critical 존재 |
| 보류 | 범위·요구사항·의사결정권이 확정되지 않아 판단 불가 |

3.4 필수 증적 원칙

다음 중 하나 이상이 제출되어야 Y로 판정할 수 있다.

```
승인된 설계서
소스코드 또는 Pull Request
환경설정·배포 Manifest
OM 등록 화면 또는 등록 SQL
CI/CD 실행결과
테스트 결과
로그·거래 추적 결과
성능 측정 결과
장애·복구 시험 결과
운영 Runbook
공식 승인서
```

회의 설명이나 담당자의 구두 답변만으로는 충족 처리하지 않는다.

25. 시사점

25.1 핵심 아키텍처 판단

Architecture Gate의 핵심은 문서를 많이 만드는 것이 아니다.

```
좋은 Gate
= 다음 단계에서 되돌릴 구조적 위험을 제거하는 것

나쁜 Gate
= 문서 목차와 서명 여부만 확인하는 것
```

NSIGHT에서는 특히 다음 세 개의 연결관계를 Gate 전체에서 반복 검증해야 한다.

거래 식별 정합성

```
화면 이벤트
↔ ServiceId
↔ Handler
↔ 거래코드
↔ OM Service Catalog
```

인증 문맥 정합성

```
SSO·JWT Claim
↔ AuthenticationContext
↔ StandardHeader
↔ 권한
↔ 감사로그
```

운영 추적 정합성

```
GUID·TraceId
↔ Gateway 로그
↔ TCF 거래로그
↔ ServiceId
↔ Mapper SQL
↔ 장애 Runbook
```

25.2 주요 위험

| 위험 | 결과 |
| --- | --- |
| Gate를 일정 승인회의로만 운영 | 구조적 위험이 다음 단계로 이월 |
| 증적 없이 담당자 설명을 신뢰 | 실제 구현과 문서 불일치 |
| 조건부 적합 남용 | Critical 결함이 운영까지 잔존 |
| OM 등록을 개발 외 작업으로 분리 | 배포되었으나 실행되지 않는 거래 발생 |
| 성능시험만 수행하고 장애시험 생략 | Thread·Pool·Failover 위험 미발견 |
| 운영팀의 늦은 참여 | Runbook과 모니터링의 실효성 저하 |
| Gate 결과 미추적 | 동일한 지적 반복 발생 |
| 아키텍처 예외 만료 미관리 | 임시구조가 영구 기술부채로 전환 |

25.3 우선 적용과제

- 본 체크리스트를 G0~G14 공식 Gate 템플릿으로 등록한다.
- 각 Gate 점검항목을 WBS와 산출물 ID에 연결한다.
- Blocker·Critical 자동 차단규칙을 PMO 절차에 반영한다.
- ServiceId–Handler–OM–Timeout 대조를 자동화한다.
- G10 선도개발 통과 전 본 개발 확대를 금지한다.
- G12 Blocker·Critical 0건을 G13 진입조건으로 강제한다.
- 조건부 적합 항목은 다음 Gate의 첫 번째 안건으로 확인한다.
- 운영 안정화 결과를 G6 개발표준과 G8 용량기준에 재반영한다.
26. 마무리말

NSIGHT Architecture Gate는 다음 질문에 명확하게 답해야 한다.

```
지금 단계에서 반드시 결정해야 할 것은 결정되었는가?

정의한 구조가 실제 설계·코드·설정·OM에 반영되었는가?

성능·보안·장애·운영 관점에서 검증되었는가?

다음 단계로 진행했을 때 되돌릴 수 없는 위험이 남아 있는가?
```

최종 Gate 운영 흐름은 다음과 같다.

```
체크리스트 배포
→ 증적 제출
→ 영역별 사전검토
→ 통합 Gate 심의
→ 결함·잔여위험 판단
→ 적합성 판정
→ 조건부 조치 등록
→ Baseline 승인
→ 다음 단계 진입
```

이 체크리스트를 적용하면 프로젝트 진척률을 문서 작성률이나 개발률로만 판단하지 않고, 아키텍처 의사결정과 운영 가능성이 실제로 확보되었는지를 기준으로 관리할 수 있다.
