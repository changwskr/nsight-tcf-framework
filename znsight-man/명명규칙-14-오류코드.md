# 오류코드 명명규칙

> **NSIGHT TCF 명명규칙 상세** · 원본: [`znsight-guide-word`](../znsight-guide-word/) · 갱신: 2026-07-05

오류코드 명명규칙 설계기준
## 1. 도입 전 안내말

오류코드는 단순히 실패를 표시하는 코드가 아니다. NSIGHT TCF에서는 사용자 응답, 운영자 조치, 거래로그, 감사로그, 장애 분석, OM 오류코드 관리를 연결하는 핵심 식별자다.
기존 금융권 에러코드 표준도 에러를 업무 에러, 시스템 에러, DB I/O 에러로 구분하고, Framework를 통해 에러처리와 Rollback/Commit을 통제하는 것을 원칙으로 둔다. 또한 기존 8자리 체계는 에러구분, 에러유형구분, 일련번호 조합으로 관리했다.NSIGHT TCF에서는 이 원칙을 유지하되, 운영자가 코드를 보고 즉시 영역과 원인을 알 수 있도록 가독성 있는 도메인 기반 오류코드 체계를 사용한다.

## 2. 오류코드 명명 최상위 원칙

| 원칙 | 설계 기준 |
| --- | --- |
| 업무코드 기반 | 업무 오류는 SV, CM, MG, OM 등 업무코드를 포함한다 |
| 공통영역 분리 | TCF, Gateway, JWT, 파일, 배치 등 공통영역은 별도 DOMAIN을 사용한다 |
| 오류분류 명확화 | Header, Validation, 인증, 권한, DB, Timeout 등 오류 유형을 CATEGORY로 구분한다 |
| 일련번호 고정 | 오류별 상세 식별을 위해 4자리 일련번호를 사용한다 |
| 사용자/운영자 메시지 분리 | 화면 사용자 메시지와 운영자 조치 메시지를 분리한다 |
| 결과코드와 오류코드 분리 | resultCode는 응답 결과, errorCode는 상세 오류 원인으로 사용한다 |
| OM 관리 | 오류코드는 OM_ERROR_CODE 기준정보로 등록·조회·변경 관리한다 |
| ETF 일원화 | 모든 예외는 TCF 후처리 ETF에서 표준 오류코드로 변환한다 |

NSIGHT 오류코드 설계도 오류 발생 후 Exception을 TCF/ETF에서 표준 오류코드로 변환하고, OM_ERROR_CODE에서 사용자 메시지와 운영자 메시지를 조회한 뒤 StandardResponse로 조립하는 구조를 권장한다.

## 6. ResultCode / ErrorCode / MessageCode 분리

오류코드 체계에서 가장 중요한 것은 결과코드와 오류코드를 섞지 않는 것이다.
| 구분 | 역할 | 예시 |
| --- | --- | --- |
| resultStatus | 처리 상태 | SUCCESS, FAIL, TIMEOUT, UNKNOWN |
| resultCode | 응답 결과 코드 | S0000, E0001, T0001 |
| errorCode | 상세 오류 식별자 | E-TCF-HDR-0001 |
| messageCode | 메시지 식별자 | MSG-TCF-HDR-0001 |
| userMessage | 사용자 표시 메시지 | 요청 정보가 올바르지 않습니다. |
| operatorMessage | 운영자 상세 메시지 | Header.serviceId is blank |
| actionGuide | 운영 조치 가이드 | 요청 전문 Header를 확인하십시오. |

| 구분 | 채번 범위 | 설명 |

| 업무 기본 오류 | 0100~0999 | 업무별 기본 오류 |
| 업무 상세 오류 | 1000~4999 | 업무 Rule, 조건, 상태 오류 |

| DB 오류 | 6000~6999 | DB, SQL, Pool 오류 |
| Timeout 오류 | 7000~7999 | 온라인, SQL, 연계 Timeout |
| 보안 오류 | 8000~8999 | 인증, 권한, 위변조, 접근차단 |
| 시스템 오류 | 9000~9999 | 미처리 예외, 인프라, 설정 오류 |

채번 원칙은 다음과 같다.
| 원칙 | 기준 |
| --- | --- |
| 재사용 금지 | 한 번 사용한 오류코드는 폐기 후에도 재사용하지 않는다 |
| 의미 변경 금지 | 기존 코드의 의미를 바꾸지 않는다 |
| 업무별 독립 채번 | E-SV-BIZ-0001과 E-CM-BIZ-0001은 서로 다른 업무 오류다 |
| 공통 오류 우선 사용 | Header, 인증, 거래통제, Timeout 등은 TCF 공통 오류코드를 우선 사용한다 |
| 신규 채번 전 검색 | OM 오류코드 관리 화면에서 중복 여부를 확인한다 |
| 메시지와 함께 등록 | 오류코드만 만들지 않고 사용자 메시지, 운영자 메시지, 조치가이드를 함께 등록한다 |

## 14. 오류코드 코드리뷰 체크리스트

| 점검 항목 | 확인 |
| --- | --- |
| 오류코드 형식이 E-{DOMAIN}-{CATEGORY}-{NNNN}인가? | □ |
| DOMAIN이 업무코드 또는 공통영역 표준에 맞는가? | □ |
| CATEGORY가 오류 원인을 명확히 나타내는가? | □ |
| 공통 오류를 업무 오류로 중복 채번하지 않았는가? | □ |
| 사용자 메시지와 운영자 메시지를 분리했는가? | □ |
| 사용자 메시지에 SQL, StackTrace, 서버정보, 개인정보가 노출되지 않는가? | □ |
| OM_ERROR_CODE에 오류코드, 메시지, 조치가이드가 등록되어 있는가? | □ |
| 거래로그에 guid, traceId, serviceId, errorCode가 남는가? | □ |
| Timeout, DB, 연계 오류는 재시도 가능 여부가 정의되어 있는가? | □ |
| 감사 대상 오류는 AUDIT_YN = Y로 관리되는가? | □ |

## 15. 마무리말

NSIGHT 오류코드 명명규칙의 핵심은 다음 한 문장으로 정리할 수 있다.
오류코드는 사용자가 볼 메시지가 아니라,
운영자가 장애 원인을 추적하고,
개발자가 수정 위치를 찾고,
PMO와 감사가 처리 이력을 확인할 수 있게 하는 표준 식별자다.

따라서 NSIGHT TCF의 오류코드는 E-{DOMAIN}-{CATEGORY}-{NNNN} 형식을 사용하고, resultCode, messageCode, userMessage, operatorMessage, actionGuide와 분리해서 관리한다. 이 기준을 지켜야 오류 발생 시 어느 업무, 어느 계층, 어떤 원인, 어떤 조치가 필요한지를 즉시 판단할 수 있다.
