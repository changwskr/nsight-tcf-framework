# 80. TCF 타임아웃 3계층 예시 — OM.Runtime.inquiry

## 80.1 한 줄 요약

**정책의 단일 출처는 `TCF_SERVICE_TIMEOUT_POLICY` 테이블**이다.  
STF가 거래 시작 시 이 테이블에서 3개 값을 읽어 `TimeoutContextHolder`에 넣고, 각 계층이 **서로 다른 구간**에 적용한다.

| 계층 | 정책 컬럼 | 기본값 | 감싸는 구간 |
| --- | --- | --- | --- |
| **TCF Online** | `ONLINE_TIMEOUT_SEC` | 5초 | Handler → Facade → Service **전체** (외부 HTTP 포함) |
| **Facade (Spring TX)** | `TX_TIMEOUT_SEC` | 5초 | Facade `@Transactional` 안의 **DB 트랜잭션** |
| **DB Query** | `DB_QUERY_TIMEOUT_SEC` | 3초 | MyBatis **SQL 1건** 실행 |

Facade에 `@Transactional(timeout = 30)`을 적어 둔 것과, 카탈로그 `TIMEOUT_SEC = 60`은 **별도 설정**이며 Online Timeout과는 직접 연결되지 않는다.

## 80.2 예시 거래 — OM.Runtime.inquiry (런타임 진단)

| 항목 | 값 |
| --- | --- |
| serviceId | `OM.Runtime.inquiry` |
| transactionCode | `OM-RTM-0001` |
| businessCode | `OM` |
| Handler | `OmRuntimeHandler` |
| Facade | `OmRuntimeFacade` |
| Service | `OmRuntimeService` |
| 주요 작업 | 15개 WAR에 HTTP status/threads 조회 |

요청 경로:

```text
브라우저 → tcf-ui → POST /om/online
Header: serviceId=OM.Runtime.inquiry, transactionCode=OM-RTM-0001
```

---

## 80.3 전체 처리 흐름

```text
① STF.preProcess()
   └─ TCF_SERVICE_TIMEOUT_POLICY 조회
      OM.Runtime.inquiry / OM-RTM-0001 / OM
      → ONLINE=5초, TX=5초, DB_QUERY=3초  (seed 수정 전)
      → TimeoutContextHolder에 바인딩

② TCF.process()
   └─ OnlineTransactionTimeoutExecutor  ← ONLINE_TIMEOUT_SEC
        └─ OmRuntimeHandler
             └─ OmRuntimeFacade.inquiry()  ← TX_TIMEOUT_SEC (30 덮어씀)
                  └─ OmRuntimeService.inquiry()
                       └─ RuntimeStatusCollector → 15개 WAR HTTP 호출
                       (DB 조회 거의 없음 → DB Query timeout 거의 안 걸림)
```

## 80.4 계층 1 — TCF Online Timeout (바깥 끈)

**역할:** “이 온라인 거래 전체가 N초 안에 끝나야 한다.”

**적용 코드:** `TCF.process()` → `OnlineTransactionTimeoutExecutor.execute()`

```java
// tcf-core .../processor/TCF.java
Object body = onlineTransactionTimeoutExecutor.execute(
        () -> dispatcher.dispatch(request, dispatchContext));
```

**적용 구간:** Dispatcher 진입부터 Handler · Facade · Service · **외부 HTTP**까지 모두 포함.

**런타임 진단에서:** 15개 WAR에 status/threads HTTP 호출 → Online 제한(5초)을 넘기면 **여기서 끊김**.

**에러 메시지:** `"온라인 거래 처리 시간(5초)을 초과했습니다."` — 이 계층에서만 발생한다.

```java
// tcf-core .../timeout/OnlineTransactionTimeoutExecutor.java
TimeoutPolicy policy = TimeoutContextHolder.get();
int timeoutSec = policy != null && policy.getOnlineTimeoutSec() > 0
        ? policy.getOnlineTimeoutSec()
        : TcfServiceTimeoutConstants.DEFAULT_ONLINE_TIMEOUT_SEC; // 5
return future.get(timeoutSec, TimeUnit.SECONDS);
```

## 80.5 계층 2 — Facade Timeout (Spring TX, DB 트랜잭션 끈)

**역할:** “Facade가 연 DB 트랜잭션이 N초 안에 끝나야 한다.”

```java
// tcf-om .../facade/OmRuntimeFacade.java
@Transactional(readOnly = true, timeout = 30)
public Map<String, Object> inquiry(Map<String, Object> body, TransactionContext context) {
    return service.inquiry(body, context);
}
```

**적용 구간:** `@Transactional` 메서드 안에서 JDBC로 열린 **DB 커넥션/트랜잭션**만.

**정책 ON 시:** `PolicyDrivenTransactionAttributeSource`가 `TX_TIMEOUT_SEC`(5초)로 **30을 덮어씀**.

```java
// tcf-web .../rule/PolicyDrivenTransactionAttributeSource.java
TimeoutPolicy policy = TimeoutContextHolder.get();
int policyTimeout = policy.getTxTimeoutSec();
return copyWithTimeout(rule, policyTimeout);
```

**런타임 진단 예시에서의 의미**

- `OmRuntimeService.inquiry()`는 대부분 **HTTP로 WAR 상태 수집** → DB는 거의 안 씀.
- Facade TX timeout(5초)은 **5초 에러의 직접 원인이 아님**.
- Online(5초)이 HTTP 수집 전체를 감싸므로 **먼저** Online에서 실패.

**비유:** Online = “택배 전체 배송 시간”, Facade TX = “창고에서 포장하는 시간”.  
런타임 진단은 창고(DB)보다 **배송(HTTP)** 이 오래 걸린다.

---

## 80.6 계층 3 — DB Query Timeout (SQL 1건 끈)

**역할:** MyBatis Mapper **한 번 실행**이 N초를 넘기면 해당 SQL만 중단.

**적용 지점:** `PolicyDrivenQueryTimeoutInterceptor` → `TimeoutContextHolder.get().dbQueryTimeoutSec`

**런타임 진단 예시:** inquiry 경로에 SQL이 거의 없어 **보통 관여하지 않음**.

**다른 거래 예:** `OM.TransactionLog.inquiry`처럼 DB 조회가 많은 거래에서는 Online > TX > Query 순으로 안쪽부터 각각 적용된다.

## 80.7 세 계층 중첩 구조

```text
┌─ Online Timeout (ONLINE_TIMEOUT_SEC) ─────────────────────────────┐
│  "거래 전체" — HTTP, Service 로직, DB 모두 포함                      │
│  ┌─ Spring TX Timeout (TX_TIMEOUT_SEC) ────────────────────────┐  │
│  │  "DB 트랜잭션" — @Transactional 구간만                        │  │
│  │  ┌─ DB Query Timeout (DB_QUERY_TIMEOUT_SEC) ──────────────┐  │  │
│  │  │  "SQL 1건" — select/insert/update 각각                  │  │  │
│  │  └────────────────────────────────────────────────────────┘  │  │
│  └──────────────────────────────────────────────────────────────┘  │
└────────────────────────────────────────────────────────────────────┘
```

### 80.7.1 설계 원칙

1. **바깥(Online) ≥ 안쪽(TX) ≥ Query** 가 이상적 — TX가 5초인데 SQL이 10초면 “거래는 끝났는데 SQL은 계속 도는” 불일치가 생긴다.
2. **Online이 HTTP·외부 연동까지 포함**하므로, DB를 안 쓰는 거래도 Online만으로 타임아웃 가능.
3. **Facade `@Transactional(timeout=N)`** 은 TCF 정책이 켜져 있으면 **참고용/폴백** — 실제 TX는 `TX_TIMEOUT_SEC`.

## 80.8 설정 출처 vs 실제 적용 (수정 전)

| 출처 | 설정값 | 실제 적용 |
| --- | --- | --- |
| `OmRuntimeFacade` | `timeout = 30` | ❌ → `TX_TIMEOUT_SEC=5`로 덮어씀 |
| `OM_SERVICE_CATALOG.TIMEOUT_SEC` | 60 | ❌ → (구) seed가 Online에 반영 안 함 |
| `TCF_SERVICE_TIMEOUT_POLICY.ONLINE_TIMEOUT_SEC` | 5 | ✅ **에러 발생 지점** |
| `TCF_SERVICE_TIMEOUT_POLICY.TX_TIMEOUT_SEC` | 5 | ✅ (이 거래에선 거의 무관) |
| `TCF_SERVICE_TIMEOUT_POLICY.DB_QUERY_TIMEOUT_SEC` | 3 | ✅ (이 거래에선 거의 무관) |

### 80.8.1 타임라인 (수정 전, includeDetails=Y)

```text
0s   ─ STF: 정책 로드 (Online=5)
0s   ─ Online 타이머 시작
0~5s ─ 15개 WAR HTTP 순차/병렬 수집
5s   ─ Online Timeout → "온라인 거래 처리 시간(5초)을 초과"
       (Facade 30초, 카탈로그 60초는 아직 의미 없음)
```

### 80.8.2 원인 — seed와 카탈로그 불일치

구 `TimeoutPolicySeedData`는 INQUIRY를 **항상 5초**로 MERGE했다.

```sql
-- 개념 (수정 전)
CASE WHEN UPPER(c.PROCESSING_TYPE) IN ('EXECUTE',...) THEN 10 ELSE 5 END
-- 카탈로그 TIMEOUT_SEC=60 이 ONLINE_TIMEOUT_SEC에 반영되지 않음
```

---

## 80.9 수정 후 기대 동작

`TimeoutPolicySeedData`가 카탈로그 `TIMEOUT_SEC`을 `ONLINE_TIMEOUT_SEC`에 반영하도록 수정되었다.

| 계층 | OM.Runtime.inquiry |
| --- | --- |
| Online | **60초** — 15 WAR 수집 여유 |
| TX | 5초 (INQUIRY seed 기본) — DB 거의 없어 영향 적음 |
| DB Query | 3초 — SQL 없으면 영향 없음 |

확인 SQL:

```sql
SELECT SERVICE_ID, ONLINE_TIMEOUT_SEC, TX_TIMEOUT_SEC, DB_QUERY_TIMEOUT_SEC
  FROM TCF_SERVICE_TIMEOUT_POLICY
 WHERE SERVICE_ID = 'OM.Runtime.inquiry';
```

OM 기동 시 `TimeoutPolicySeedData.mergeAll()`이 실행되면 DB 정책이 갱신된다.

## 80.10 다른 거래와 비교

| 거래 | Online | TX | Query | 주로 걸리는 계층 |
| --- | --- | --- | --- | --- |
| `OM.Runtime.inquiry` | 5→60초 | 5초 | 3초 | **Online** (HTTP 15건) |
| `OM.TransactionLog.inquiry` | 5~10초 | 5초 | 3초 | **Query** (대량 SELECT) |
| `OM.User.update` | 5초 | 5초 | 3초 | **TX + Query** (UPDATE 여러 건) |

## 80.11 정리

| 관점 | 내용 |
| --- | --- |
| 마스터 | `TCF_SERVICE_TIMEOUT_POLICY` |
| Online / TX / Query | 같은 정책의 3개 컬럼을 **서로 다른 레이어**에 적용 |
| Facade `timeout=30` | TCF 정책 ON 시 TX 정책값으로 덮어씀 — Online과 무관 |
| 런타임 진단 5초 에러 | Online 5초가 HTTP 수집 전체를 먼저 끊은 것 |
| 해결 방향 | 카탈로그 `TIMEOUT_SEC` → `ONLINE_TIMEOUT_SEC` 반영 + HTTP 수집 병렬화 |

## 80.12 개발자 체크리스트

| 항목 | 확인 |
| --- | --- |
| 장시간 HTTP/외부 연동이 있는 거래는 `ONLINE_TIMEOUT_SEC`을 충분히 등록했는가 | □ |
| Facade `@Transactional(timeout=N)`만 늘리고 정책 테이블을 안 바꾸지 않았는가 | □ |
| `OM_SERVICE_CATALOG.TIMEOUT_SEC`과 `TCF_SERVICE_TIMEOUT_POLICY`가 일치하는가 | □ |
| DB-heavy 거래는 Query timeout(3초)도 함께 점검했는가 | □ |
| Timeout 오류 메시지가 Online인지 TX인지 Query인지 구분할 수 있는가 | □ |

## 80.13 마무리말

TCF Timeout은 **한 곳의 숫자**가 아니라 **3계층 중첩**이다.  
Facade에 적은 `timeout = 30`과 카탈로그 `TIMEOUT_SEC = 60`을 보고 “30초/60초까지 기다릴 것”이라 기대하면, Online 5초에서 먼저 실패할 수 있다.  
신규 거래·장애 분석 시 **어느 계층에서 끊겼는지**를 먼저 확인하는 것이 기준이다.
