# 업무코드 표준표

> **NSIGHT TCF 명명규칙 상세** · 원본: [`znsight-guide-word`](../znsight-guide-word/) · 갱신: 2026-07-05

NSIGHT 9개 업무코드 기준
## 1. 도입 전 안내말

NSIGHT TCF Framework에서 업무코드는 모든 명명규칙의 출발점이다.업무코드가 정해지면 다음 항목이 자동으로 연결된다.
업무코드
```text
  ↓
Context Path
↓
WAR 파일명
↓
Gradle Module
↓
Java Package
↓
ServiceId
↓
거래코드
↓
Mapper / SQL ID
↓

```

오류코드 / 거래로그 / 감사로그

기존 NSIGHT 기준에서도 9개 WAR 배포 구조를 전제로 하고 있으며, MG는 Message 의미로 사용하므로 운영관리 업무는 별도 OM(Operation Management)으로 분리하는 것이 적합하다.

## 4. 업무코드 설계 원칙

| 원칙 | 기준 |
| --- | --- |
| 2자리 대문자 사용 | SV, CM, OM처럼 영문 대문자 2자리 사용 |
| 중복 금지 | 하나의 업무코드는 하나의 업무 의미만 가져야 함 |
| Context와 일치 | SV는 /sv, CM은 /cm으로 고정 |
| WAR와 일치 | SV는 sv.war, CM은 cm.war로 고정 |
| Package와 일치 | SV는 com.nh.nsight.marketing.sv로 고정 |
| ServiceId Prefix와 일치 | SV 업무 ServiceId는 반드시 SV.로 시작 |
| 거래코드 Prefix와 일치 | SV 업무 거래코드는 반드시 SV-로 시작 |
| 오류코드 Prefix와 일치 | SV 업무 오류코드는 반드시 E-SV-로 시작 |
| 운영 반영 후 변경 금지 | 업무코드는 배포, 로그, 오류코드, DB 기준정보와 연결되므로 변경 금지 |
| OM 분리 | 운영관리는 MG가 아니라 OM으로 분리 |

## 5. 9개 업무코드 기준 사용 위치

| 사용 위치 | 사용 방식 | 예시 |
| --- | --- | --- |
| Apache Routing | URL Prefix 기준 라우팅 | /sv/* → sv.war |
| Gateway Routing | businessCode 기준 대상 업무 결정 | SV → /sv/online |
| Tomcat 배포 | WAR 파일명 기준 Context 생성 | sv.war → /sv |
| Spring 설정 | server.servlet.context-path | /sv |
| Java Package | 업무별 Component Scan 기준 | com.nh.nsight.marketing.sv |
| ServiceId | Dispatcher 실행 기준 | SV.Customer.selectSummary |
| 거래코드 | 거래로그 / 감사로그 추적 기준 | SV-INQ-0001 |
| Mapper XML | 업무별 SQL 위치 | mapper/sv/SvCustomerMapper.xml |
| SQL ID | SQL 추적 기준 | SV.Customer.selectSummary |
| 오류코드 | 오류 응답 / OM 오류관리 기준 | E-SV-BIZ-0001 |
| 권한관리 | 메뉴/기능/ServiceId 권한 기준 | businessCode = SV |
| 거래통제 | 허용 거래 등록 기준 | businessCode = SV |
| Timeout 정책 | 서비스별 제한시간 기준 | serviceId = SV.Customer.selectSummary |

## 7. OM 업무코드 특별 기준

OM은 17번째 업무코드이지만 일반 업무 모듈과 성격이 다르다.
| 항목 | OM 기준 |
| --- | --- |
| 업무코드 | OM |
| 업무명 | Operation Management |
| Context | /om |
| WAR | om.war |
| Module | tcf-om |
| Package | com.nh.nsight.marketing.om |
| 역할 | 운영 기준정보 관리 |
| 관리 대상 | 사용자, 권한, 메뉴, ServiceId, 거래통제, Timeout, 오류코드, 공통코드, 세션, 거래로그, 감사로그 |
| 분리 이유 | 일반 업무 로직과 운영 기준정보 관리를 분리하기 위함 |
| MG와 구분 | MG는 Message 업무, OM은 Operation Management 업무 |

## 8. 마무리말

NSIGHT 9개 업무코드는 개발 표준의 최상위 기준이다.
업무코드가 흔들리면
Context가 흔들리고,
WAR가 흔들리고,
Package가 흔들리고,
ServiceId가 흔들리고,
거래로그와 오류코드 추적도 흔들린다.

따라서 NSIGHT에서는 다음 기준을 고정한다.
9개 업무코드는 업무 경계, 배포 경계, URL 경계, Package 경계, 운영관리 기준정보의 최상위 식별자로 사용한다.
