## Master 해설

목록 조회는 count SQL과 select SQL 분리·offset/limit 페이징(znsight-man 72)이 표준이며, Mapper XML dynamic where와 Facade에서 PageDto 조립 패턴을 따릅니다. 등록·변경(save/update)은 CREATE/UPDATE processingType과 Facade @Transactional 쓰기 경계·감사로그 대상입니다.

TcfServiceClient를 통한 WAR 간 호출(znsight-man 74)은 StandardRequest POST·Header 전파·Timeout 중첩을 이해하는 실습입니다. ic↔sv 데모는 tcf-eai contract docs/46-service-integration-contract와 맞춥니다.

Validation Rule + BusinessException(E-SV-*) 패턴, 파일 다운로드·Batch Job 샘플(75~76)은 온라인 Handler 패턴의 변형이므로 동일 STF~ETF 전제를 유지합니다.

리뷰: 페이징 total count 정합, duplicate key business error, EAI failure rollback, Idempotency on save. SQL injection in dynamic sort/order by whitelist.

### TcfServiceClient

```java
package com.nh.nsight.tcf.eai.client;

import com.nh.nsight.tcf.core.support.context.TransactionContext;
import com.nh.nsight.tcf.eai.model.IntegrationCallRequest;
import com.nh.nsight.tcf.eai.model.IntegrationCallResult;
import java.util.Map;

/**
 * 서비스 간 표준 전문 호출 공통 Client.
 *
 * <p>업무 서비스는 다른 업무 서비스의 Java 코드를 직접 참조하지 않고,
 * 이 Client 를 통해 {@code POST /{businessCode}/online} + serviceId 방식으로 호출한다.
 *
 * <p>대상별 Adapter 없이 한 줄로 호출하려면 {@link #call(String, String, String, Map, TransactionContext)}
 * 또는 {@link #callForBody(String, String, String, Map, TransactionContext)} 를 사용한다.
 */
public interface TcfServiceClient {

    /**
     * 대상 서비스를 호출한다. 호출 측 컨텍스트의 GUID/TraceId/사용자 정보를 전파한다.
     *
     * @param request       대상 식별 정보와 body
     * @param callerContext 호출 측 거래 컨텍스트 (null 허용 — 신규 GUID 생성)
     * @return 정규화된 연동 결과
     */
    IntegrationCallResult call(IntegrationCallRequest request, TransactionContext callerContext);

    /**
     * 대상별 Adapter 없이 한 줄로 호출하는 편의 메서드 (INQUIRY 기본).
     *
     * <pre>
     * IntegrationCallResult r = tcfServiceClient.call(
     *         "SV", "SV.Customer.selectSummary", "SV-INQ-0002",
     *         Map.of("customerNo", customerNo), context);
     * </pre>
     */
    default IntegrationCallResult call(String targetBusinessCode,
                                       String targetServiceId,
                                       String targetTransactionCode,
                                       Map<String, Object> body,
                                       TransactionContext callerContext) {
        IntegrationCallRequest request = IntegrationCallRequest.builder()
                .targetBusinessCode(targetBusinessCode)
                .targetServiceId(targetServiceId)
                .targetTransactionCode(targetTransactionCode)
                .body(body)
                .build();
        return call(request, callerContext);
    }

    /**
     * 성공 시 대상 응답 body(Map)만 바로 반환하는 편의 메서드.
     *
     * <p>업무 실패/Timeout/시스템 오류는 {@code IntegrationException} 계열로 전파된다.
     *
     * <pre>
     * Map&lt;String, Object&gt; summary = tcfServiceClient.callForBody(
     *         "SV", "SV.Customer.selectSummary", "SV-INQ-0002",
     *         Map.of("customerNo", customerNo), context);
     * </pre>
     */
    default Map<String, Object> callForBody(String targetBusinessCode,
                                            String targetServiceId,
                                            String targetTransactionCode,
                                            Map<String, Object> body,
                                            TransactionContext callerContext) {
        return call(targetBusinessCode, targetServiceId, targetTransactionCode, body, callerContext)
                .getBody();
    }
}
```

원본: [`tcf-eai/src/main/java/com/nh/nsight/tcf/eai/client/TcfServiceClient.java`](../tcf-eai/src/main/java/com/nh/nsight/tcf/eai/client/TcfServiceClient.java)

## 23.1 목록조회 + 페이징

### 거래 정의

| 항목 | 값 |
| --- | --- |
| ServiceId | `SV.Customer.selectList` |
| 거래코드 | `SV-INQ-0002` |
| 거래명 | 고객 목록 조회 |
| 페이징 | Offset Paging |
| 기본 pageNo | 1 |
| 기본 pageSize | 100 |
| 최대 pageSize | 500 |

### TCF와 페이징의 역할 분리

**TCF Core는 페이징에 관여하지 않습니다.** STF/ETF는 Header 검증·로그·표준 응답만 담당하고, pageNo/pageSize 검증·offset 계산·count/list SQL은 **업무 WAR 내부 Rule → DAO → Mapper**에서 처리합니다.

```text
StandardRequest.body
  ├─ searchCondition  (검색 조건)
  └─ page
       ├─ pageNo
       └─ pageSize
        ↓
Rule: pageNo≥1, 1≤pageSize≤500, offset=(pageNo-1)*pageSize
        ↓
DAO: countQuery + listQuery
        ↓
Response body
  ├─ list[]
  └─ page { pageNo, pageSize, totalCount, totalPages }
```

### Rule 구현

```java
public void validatePaging(PageRequest page) {
    if (page.getPageNo() < 1) throw new BusinessException("E-SV-PAGE-0001", "pageNo 오류");
    if (page.getPageSize() < 1 || page.getPageSize() > 500)
        throw new BusinessException("E-SV-PAGE-0002", "pageSize 오류");
    page.setOffset((page.getPageNo() - 1) * page.getPageSize());
}
```

### Mapper — count + list

```xml
<select id="countCustomerList" resultType="int">
    SELECT COUNT(*) FROM sv_customer
    <include refid="customerSearchCondition"/>
</select>

<select id="selectCustomerList" resultType="CustomerListItem">
    SELECT customer_id, customer_name, grade_code
    FROM sv_customer
    <include refid="customerSearchCondition"/>
    ORDER BY customer_id
    LIMIT #{pageSize} OFFSET #{offset}
</select>
```

Service는 count → list 순으로 호출하고 `totalPages = ceil(totalCount / pageSize)`를 계산합니다.

### UI 연동

tcf-ui 샘플: `http://localhost:8099/sv/sample-list.html` — 페이징 UI와 Relay 호출 예시.

## 23.2 등록/변경 거래

### 등록 거래

| 항목 | 값 |
| --- | --- |
| ServiceId | `SV.Customer.register` |
| 거래코드 | `SV-REG-0001` |
| 트랜잭션 | Facade `@Transactional` (readOnly=false) |
| Idempotency | clientRequestId 중복 검사(권장) |

```java
@Transactional
public CustomerRegisterResponse register(CustomerRegisterRequest req) {
    rule.validateRegister(req);
    rule.checkDuplicate(req.getCustomerId());
    dao.insertCustomer(req);
    auditLogService.log("SV.Customer.register", req.getCustomerId());
    return CustomerRegisterResponse.of(req.getCustomerId());
}
```

### 변경 거래

| 항목 | 값 |
| --- | --- |
| ServiceId | `SV.Customer.update` |
| 거래코드 | `SV-UPD-0001` |
| 낙관적 잠금 | `updateVersion` 또는 `lastModifiedDate` (필요 시) |

변경 거래는 **존재 여부 확인(Rule) → UPDATE Mapper → 영향 row=0이면 동시 수정 오류** 패턴을 따릅니다.

### 트랜잭션·롤백

- Facade에서 `@Transactional` 시작
- Rule에서 `BusinessException` → 롤백
- DAO 실패 → `SystemException` → 롤백 + TCF_TX_LOG FAIL

## 23.3 외부 서비스 호출 (EAI)

SV에서 IC WAR를 호출하는 샘플: `SV.Integration.icSample`.

| 항목 | 기준 |
| --- | --- |
| 연동 방식 | tcf-eai HTTP/JSON Client |
| 금지 | WAR 간 Java 클래스 직접 import |
| Header | serviceId, businessCode, guid 전달 |
| Timeout | OM TimeoutPolicy + Client Timeout |
| 오류 | 대상 WAR 다운 → 연동 오류코드 |

```java
@Service
@RequiredArgsConstructor
public class SvIntegrationService {
    private final TcfEaiClient eaiClient;

    public IcSampleResponse callIc(IcSampleRequest req) {
        return eaiClient.post("/ic/online", "IC.Sample.inquiry", req, IcSampleResponse.class);
    }
}
```

통합 테스트: IC·SV 모두 bootRun 또는 ztomcat 배포 후 End-to-End 호출.

## 23.5 Batch Job

| 항목 | 값 |
| --- | --- |
| 모듈 | `tcf-batch` + 업무 Scheduler |
| Job명 | `SV.Customer.dailySync` (예시) |
| 실행 | Spring `@Scheduled` 또는 OM Batch 등록 |

Batch는 Online Endpoint가 아닌 **Scheduler → Service → DAO** 흐름입니다. 중복 실행 방지(Lock), 실행 이력, 실패 재처리를 tcf-batch 모니터링과 연동합니다.

```java
@Scheduled(cron = "0 0 2 * * *")
public void dailyCustomerSync() {
    batchLockService.executeWithLock("SV.Customer.dailySync", () -> service.sync());
}
```

## 23.6 오류처리·테스트 코드

### 오류 처리 패턴

| 유형 | 예외 | resultCode |
| --- | --- | --- |
| 입력값 오류 | BusinessException | E-SV-* |
| 데이터 없음 | BusinessException | E-SV-CUSTOMER-0001 |
| 권한 없음 | BusinessException | E-AUTH-* |
| DB/시스템 | SystemException | E-SYS-* |
| Timeout | TimeoutException | E-TMO-* |

Handler·ETF가 BusinessException을 표준 Result로 변환합니다. **Rule에서 BusinessException, DAO에서 SystemException** 위치를 지킵니다.

### 테스트 코드 샘플

```java
// Rule 단위
@Test
void pageSize_초과() {
    var page = new PageRequest(1, 501);
    assertThatThrownBy(() -> rule.validatePaging(page))
        .isInstanceOf(BusinessException.class)
        .hasFieldOrPropertyWithValue("errorCode", "E-SV-PAGE-0002");
}

// Mapper (H2)
@Test
void selectCustomerList_페이징() {
    var criteria = new CustomerSearchCriteria();
    criteria.setPage(new PageRequest(1, 10));
    assertThat(mapper.countCustomerList(criteria)).isGreaterThan(0);
    assertThat(mapper.selectCustomerList(criteria)).hasSizeLessThanOrEqualTo(10);
}
```

## 장 요약 (Master)

목록·페이징은 TCF가 아닌 **업무 Rule·Mapper**에서 count+list로 처리합니다. 등록·변경은 Facade `@Transactional`과 Idempotency·감사로그를 적용하고, WAR 간 연동은 **tcf-eai**만 사용합니다. 파일·Batch는 별도 패턴이며, BusinessException 중심 오류 처리와 계층별 단위·통합 테스트로 [제22장](../제08편/22-조회-거래-SV-고객요약.md) 조회 샘플을 확장한 End-to-End 실습을 완성합니다.

> Master Edition: **아키텍처 뷰** → **Master 해설** → **구현 샘플** → **Master Deep Dive** → **심화 참고** 순으로 본문과 함께 읽는다.
