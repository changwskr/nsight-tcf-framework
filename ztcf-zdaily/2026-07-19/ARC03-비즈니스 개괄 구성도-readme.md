# ARC03 - NSIGHT TCF 비즈니스 개괄 구성도

## 1. 문서 개요

### 1.1 목적
- NSIGHT TCF Framework 프로젝트의 **비즈니스 관점** 통합 아키텍처 구성도 제공
- 업무 도메인(고객·마케팅·실시간·운영), 거래 프로세스, 데이터 흐름, 업무 간 연동을 종합 표현
- 이해관계자가 비즈니스 구조를 한눈에 파악할 수 있는 개괄 구성도 제공

### 1.2 범위
- 비즈니스 도메인 모델 (업무코드 기준)
- 핵심 온라인 거래·연계 프로세스
- 데이터 흐름 및 연관 관계
- 시스템 간 통합 및 연동 (Gateway / EAI / Client)
- 비즈니스 규칙 및 운영 정책 (Rule / OM / Timeout)

### 1.3 대상 독자
- 비즈니스 분석가
- 시스템 아키텍트
- 프로젝트 매니저
- 개발팀 리더
- 운영·경영 이해관계자

### 2.2 도메인별 핵심 엔티티·업무

#### 2.2.1 고객 도메인
| 코드 | 명칭 | 핵심 개념 |
|------|------|-----------|
| IC | Integration Customer | 통합 고객 조회·연계 |
| PC | Private Customer | 개인 고객 |
| BC | Business Customer | 기업 고객 |
| MS | Mini Single View | 경량 Single View |

- **Customer / User**: 고객·사용자 식별 정보
- **CustomerSummary**: 연계 조회 요약 (예: SV→IC)
- **Activity**: 고객 관련 조회·변경 이력(거래로그)

#### 2.2.2 마케팅 도메인
| 코드 | 명칭 | 핵심 개념 |
|------|------|-----------|
| SV | Single View | 고객 통합 뷰·마케팅 조회 |
| PD | Product | 상품 |
| EB | EBM | 이벤트/캠페인 연계 마케팅 |
| CM | Campaign | 캠페인 |

- **Event**: EB 이벤트·Outbox 상태
- **Product**: 상품 정보
- **Campaign**: 캠페인 정의·대상
- **Sample/SystemTx**: 샘플·시스템 거래(개발·운영 검증)

#### 2.2.3 실시간·이벤트 도메인
| 코드 | 명칭 | 핵심 개념 |
|------|------|-----------|
| EP | Event Processing | 이벤트 수신 처리 |
| BP | Behavior Processing | 행동 처리 |
| BD | Behavior Data | 행동 데이터 |

- **UserEvent**: EP로 전달되는 사용자 이벤트 페이로드
- **PublishStatus**: Outbox 발행 상태 (PENDING/DONE/FAIL)

#### 2.2.4 지원 도메인
| 코드 | 명칭 | 핵심 개념 |
|------|------|-----------|
| SS | Sales Support | 영업 지원 |
| MG | Message | 메시지 |
| CT | Contents | 콘텐츠 |

#### 2.2.5 운영 도메인
| 코드 | 명칭 | 핵심 개념 |
|------|------|-----------|
| OM | Operation Management | 운영관리·거래통제·오류코드 |
| OC | Operation Capacity | 용량·환경 |
| UD | UpDownload | 파일 업다운로드 |

- **TimeoutPolicy / ErrorCode / AuthHistory**
- **Updownload 파일 메타·바이너리**

#### 2.2.6 공통·인증 도메인
| 코드 | 명칭 | 핵심 개념 |
|------|------|-----------|
| CC | Common | 공통 |
| JWT | JWT Auth | 인증 토큰 |

- **GatewayRoute** (`TCF_GATEWAY_ROUTE`): ENV + BUSINESS_CODE 라우팅
- **TransactionLog**: GUID / TraceId / ServiceId 기반 거래로그

### 3.2 프로세스별 상세 흐름

#### 3.2.1 온라인 거래 프로세스 (표준)
```
1. UI 요청 → 2. Gateway 라우팅 → 3. JWT/세션 검증(필요 시)
→ 4. OnlineController → 5. STF → 6. ServiceId로 Handler
→ 7. Facade→Service→Rule→DAO→Mapper → 8. ETF(응답·거래로그)
```

#### 3.2.2 고객 조회·연계 프로세스
```
1. SV/MS 고객 조회 요청 → 2. Gateway → SV/MS WAR
→ 3. (필요 시) IC Integration Client 호출 → 4. 통합 결과 조립 → 5. 표준 응답
```

#### 3.2.3 마케팅 이벤트 발행 프로세스 (EB→EP)
```
1. EB 이벤트 등록/갱신 → 2. Outbox 적재
→ 3. Scheduler Publish → 4. EP /ep/online 호출
→ 5. EP 처리·상태 반영 → 6. 거래/감사 로그
```

#### 3.2.4 운영 통제 프로세스
```
1. OM 정책(Timeout/오류코드/인증) 설정
→ 2. 온라인 거래에 정책 적용 → 3. 위반·만료 시 표준 오류
→ 4. 인증 이력·관측 지표 확인
```


## 6. 비즈니스 규칙 및 정책

### 6.1 핵심 비즈니스 규칙

#### 6.1.1 고객·조회 규칙
- 업무코드·ServiceId 필수
- 연계 호출(IC 등) 실패 시 표준 오류 매핑
- 조회 조건(Rule) 검증 후 DAO 접근

#### 6.1.2 마케팅·이벤트 규칙
- EB 이벤트 상태 전이 규칙 (등록→발행→완료/실패)
- Outbox 배치 크기·주기 정책
- EP 페이로드 필수 항목 검증

#### 6.1.3 거래 공통 규칙
- STF에서 요청 스키마·필수값 검사
- Timeout 정책 초과 시 거래 중단
- ETF에서 표준 응답코드·메시지 통일

#### 6.1.4 운영 규칙
- Gateway 미등록 BUSINESS_CODE → 404
- OM 오류코드·Timeout 정책 우선 적용
- PRD 라우트 변경은 승인된 배포 절차만 허용

### 6.2 보안 정책

#### 6.2.1 인증 정책
- JWT 발급·검증 (`tcf-jwt`)
- Gateway 세션·헤더 규칙
- OM 인증 이력 기록

#### 6.2.2 권한 정책
- 관리 화면(OM/Gateway Admin) 접근 통제
- 최소 권한·운영 역할 분리
- Secret/ConfigMap 분리 (K8s)

#### 6.2.3 데이터 보호 정책
- DB 계정 Secret 관리
- 거래로그·업다운로드 경로 통제
- 환경별 프로필(`local`/`dev`/`prod`) 분리

## 11. 결론

### 11.1 아키텍처 특징
- **업무코드 중심 도메인 분리**: IC/SV/EB/EP/OM 등 독립 WAR
- **TCF 표준 거래 엔진**: STF~ETF·ServiceId·거래로그 일원화
- **Gateway 단일 라우팅 원칙**: `TCF_GATEWAY_ROUTE`만 사용
- **이벤트 연계**: EB Outbox → EP 실시간 처리
- **운영 가시성**: OM + GUID/TraceId/ServiceId

### 11.2 기대 효과
- **업무 민첩성**: 도메인별 독립 개발·배포
- **운영 효율**: 표준 오류·Timeout·거래 추적으로 장애 대응 가속
- **확장성**: Docker/K8s 기반 스케일 아웃
- **일관성**: 화면·Gateway·업무·로그가 동일 식별체계 공유

### 11.3 향후 발전 방향
- 미구현/연계 업무코드(CM/BP/BD 등) 모듈 확장
- `tcf-docker`·`tcf-k8s` 전 모듈 표준화
- 관측성(메트릭·대시보드) 고도화
- CI/CD(`tcf-cicd`)와 배포 파이프라인 일체화
