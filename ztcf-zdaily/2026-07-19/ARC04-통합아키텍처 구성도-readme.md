# ARC04 - NSIGHT TCF 통합 아키텍처 구성도

## 1. 문서 개요

### 1.1 목적
- NSIGHT TCF Framework의 **통합 아키텍처** 관점 종합 구성도 제공
- AA(Application), DA(Data), TA(Technology), IA(Interface), SA(Security) 5개 관점 정의
- UI → Gateway → 업무 WAR → TCF 6계층 → DB 전 구간의 구조와 관계를 명확히 표현

### 1.2 범위
- **AA**: 애플리케이션 구조·컴포넌트·TCF 계층
- **DA**: 데이터 모델·저장소·거래로그·라우트
- **TA**: 기술 스택·런타임·Docker/K8s·CI/CD
- **IA**: Gateway·업무 간 Client·온라인/관리 API
- **SA**: JWT·세션·Secret·운영 보안 정책

### 1.3 대상 독자
- 엔터프라이즈·시스템·기술·보안 아키텍트
- 프로젝트 매니저 / 개발·운영 리더

### 1.4 관련 문서
| 문서 | 내용 |
|------|------|
| ARC02 | 애플리케이션 구성도 |
| ARC03 | 비즈니스 개괄 구성도 |
| `tcf-gateway/docs/ROUTING_TABLE.md` | Gateway 라우팅 |

### 3.2 애플리케이션 컴포넌트 상세

#### 3.2.1 Presentation Layer
- **tcf-ui**: 업무 정적 UI + Relay (`:8099`)
- **tcf-uj**: 통합 시험 UI·sample-requests
- **OM Admin**: 운영관리·Timeout·오류코드·업다운로드
- **Gateway Admin**: `TCF_GATEWAY_ROUTE` 관리

#### 3.2.2 Business Layer (업무 도메인)
- **고객**: IC / PC / BC / MS
- **마케팅**: SV / PD / EB / CM
- **실시간**: EP / BP
- **지원·운영**: SS / MG / OM / OC / UD
- **공통·인증**: CC / JWT

#### 3.2.3 TCF Service Layer
- **Handler**: ServiceId 진입
- **Facade**: 유스케이스·트랜잭션 경계
- **Service / Rule**: 업무 처리·규칙 검증
- **STF / ETF**: 요청 검사·표준 응답·거래로그

#### 3.2.4 Data Access Layer
- **DAO + MyBatis Mapper/XML**
- **Persistence DTO/Row**
- **tcf-cache** (환경별 캐시)

### 4.3 데이터 관리 전략

#### 4.3.1 데이터 분류
- **Master Data**: 고객·상품·캠페인 등 업무 마스터
- **Transaction Data**: 온라인 거래 결과·이벤트 Outbox
- **Reference Data**: 공통코드·Gateway Route·오류코드
- **Audit / Ops Data**: 거래로그, 인증 이력, OM 관측

#### 4.3.2 환경별 저장소
| 환경 | DB | 비고 |
|------|-----|------|
| local | H2 (파일/메모리) | gateway-route, txlog 로컬 |
| dev | Oracle (또는 지정 DB) | `tcf-cicd/dev` |
| prod | Oracle | `tcf-cicd/prod`, 직접 변경 최소화 |

#### 4.3.3 파일·백업
- **Updownload**: `data/updownload/`
- **백업**: DB 백업 + 설정/시드 SQL(`TCF_GATEWAY_ROUTE`) 버전관리
- **복구**: GUID 단위 거래 추적 후 모듈별 재처리

### 6.2 API·엔드포인트 설계

#### 6.2.1 온라인 거래 (표준)
```
POST /{businessCode}/online     # Gateway 또는 업무 WAR
  body: ServiceId + 업무 요청 JSON
  headers: 세션/JWT/추적 헤더 (정책에 따름)
```

주요 LOCAL Target 예:

| BUSINESS_CODE | Port | Target |
|---------------|------|--------|
| IC | 8082 | `http://127.0.0.1:8082/ic/online` |
| SV | 8086 | `http://127.0.0.1:8086/sv/online` |
| EB | 8089 | `http://127.0.0.1:8089/eb/online` |
| EP | 8090 | `http://127.0.0.1:8090/ep/online` |
| OM | 8097 | `http://127.0.0.1:8097/om/online` |
| JWT | 8110 | `http://127.0.0.1:8110/online` |

#### 6.2.2 관리·운영 API
- Gateway: `/api/admin/routes`
- Actuator: `/actuator/health`
- OM / JWT Admin UI·API

#### 6.2.3 인터페이스 표준

- **식별**: ServiceId, GUID, TraceId, BUSINESS_CODE
- **라우팅**: `ENV_CODE + BUSINESS_CODE` → `TCF_GATEWAY_ROUTE`만 사용
- **오류**: 표준 응답코드 + HTTP Status (미등록 라우트 404)

### 7.2 보안 레이어별 상세

#### 7.2.1 Perimeter / Platform
- K8s Namespace 격리 (`nsight`)
- Ingress / TLS (환경별)
- 이미지 레지스트리·배포 승인 (`deploy.sh` + commit-sha)

#### 7.2.2 Application Security
- **Authentication**: `tcf-jwt` 발급·검증
- **Session/Header**: Gateway Session Header Rules
- **Authorization**: OM·Admin 화면 접근 통제
- **Input Validation**: Rule 계층 필수

#### 7.2.3 Data Security
- DB 자격증명 Secret
- 거래로그 GUID/TraceId로 감사 추적
- 업다운로드 파일 경로·권한 통제
- PRD 라우트 DB 직접 수정 금지

### 7.3 보안 정책 및 절차

#### 7.3.1 인증 정책
```
로그인/토큰 요청 → tcf-jwt 발급
→ Gateway/업무에서 토큰·세션 검증
→ 유효 시 BUSINESS_CODE 라우팅 허용
→ OM 인증 이력 기록
```

#### 7.3.2 권한·운영 정책
- Admin(OM/Gateway)과 일반 온라인 거래 권한 분리
- Timeout·오류코드는 OM 정책 우선
- 최소 권한, Secret/환경변수로 민감정보 분리

#### 7.3.3 데이터 보호 정책
- 환경별 프로필 분리 (`local` / `dev` / `prod`)
- 설정은 Git/`tcf-cicd` 기준 배포
- 로그에 민감정보 마스킹(운영 표준)
