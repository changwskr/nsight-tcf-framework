## 핵심 결론

> 세션 원천 = **SESSIONDB** (Spring Session JDBC). WAS 메모리 ❌.

---

## 세션 아키텍처 (10.3)

```
브라우저 (JSESSIONID/NSIGHTSID)
→ Apache → [Gateway] → 업무 WAR
→ Spring Session JDBC Filter
→ STF.preProcess() (Header ↔ 세션 비교)
→ SESSIONDB: SPRING_SESSION, SPRING_SESSION_ATTRIBUTES, TCF_USER_SESSION
```

## 세션 생성 (10.4)

1. `OM.Auth.login` / `OM.Auth.ssoLogin` (tcf-om)  
2. OM_USER·권한 조회  
3. HttpSession → Spring Session JDBC 저장  
4. Cookie 응답  

## 업무 요청 검증 (10.5)

```
POST /sv/online + Cookie + header(user, branch, channelId)
→ Session Filter: SESSIONDB 조회, 만료
→ STF: header user ↔ session user, branch, channel, 권한, 중복로그인, Absolute Timeout
→ Dispatcher
```

## 테이블 (10.7)

| 테이블 | 용도 |
|--------|------|
| SPRING_SESSION | Spring Session 기본 |
| SPRING_SESSION_ATTRIBUTES | 속성 Blob |
| TCF_USER_SESSION | TCF/Gateway 확장 (지점·채널·상태) |

## Timeout (10.9)

| 유형 | 설명 |
|------|------|
| Idle | 미사용 시간 |
| Absolute | 로그인 후 최대 시간 |

## Gateway·Sticky (10.10~10.11)

- Gateway: Cookie **전달** (Sticky는 Apache/L4)
- DeltaManager(WAS 메모리) ❌ — JDBC Session ✅

## 다중 WAR (10.12)

모든 WAR 동일 SESSIONDB → **세션 공유**

## application.yml (10.13)

```yaml
spring.session.store-type: jdbc
```

## 세션 오류 (10.17)

- 세션 없음 / 만료
- Header user ≠ session user → E-TCF-SES-*
