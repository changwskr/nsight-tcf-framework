---

## 21.1 TCF 테스트 = serviceId 1건 검증

일반 REST 테스트와 다릅니다.  
**“이 URL이 200이냐?”** 보다 **“이 serviceId 거래가 표준대로 끝났냐?”** 가 중심입니다.

| 확인 | 어디서 |
| --- | --- |
| `resultCode` | JSON 응답 |
| `guid` | Header |
| 거래 이력 | `TCF_TX_LOG` (OM 조회) |

---

## 21.2 테스트 피라미드 (외울 것 4층)

```text
        Smoke (배포 후 curl 5건)
      ─────────────────────
        통합 (POST /sv/online)
      ─────────────────────
        Mapper/SQL (H2)
      ─────────────────────
        단위 (JUnit + Mock)  ← 가장 많이
```

| 층 | 언제 | 도구 |
| --- | --- | --- |
| **단위** | Handler·Rule 짤 때 | JUnit, Mockito |
| **Mapper** | XML SQL 바꿀 때 | H2 |
| **통합** | 기능 끝났을 때 | curl, tcf-ui |
| **Smoke** | 배포 후 | 대표 serviceId 3~5건 |

---

## 21.3 단위 vs 통합 — 구분법

| | 단위 | 통합 |
| --- | --- | --- |
| Spring 켜나? | **아니오** (Mock) | **예** (bootRun/H2) |
| DB | Mock | H2 또는 local |
| HTTP | 안 함 | `POST /{bc}/online` |

Handler 단위 예: Facade를 Mock → `doHandle()` 결과만 검증.

---

## 21.4 MR 올리기 전 최소 테스트

| # | 할 일 |
| --- | --- |
| 1 | 신규 Handler **단위 테스트** 1개 이상 |
| 2 | curl 또는 tcf-ui로 **SUCCESS** |
| 3 | 필수값 빼고 **오류코드** 확인 |
| 4 | `gradlew build` 통과 |

부록 H 체크리스트와 **같은 맥락**입니다.

---

## 21.5 ⚠️ 초보자 실수

| 실수 | |
| --- | --- |
| Controller 테스트만 작성 | TCF는 **`/online` 공통** — Handler·통합이 핵심 |
| DB 없이 Mapper “단위” | Mapper는 **H2 통합**으로 분리 |
| Catalog 없이 통과만 확인 | **미등록 serviceId** 차단도 테스트 |
| 로그에 고객번호 전체 출력 | **마스킹** 확인 |

---

## 요약

- 테스트 중심 = **serviceId 1건**
- **단위(많이)** + **curl 통합(필수)** + **Smoke(배포 후)**
- `resultCode`, `guid`, 거래로그까지 보면 **TCF 거래 테스트**
