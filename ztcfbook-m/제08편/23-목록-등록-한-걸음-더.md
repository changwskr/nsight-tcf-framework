## 23.1 22장 다음 단계

22장은 **한 건 조회**였습니다. 실무에서는 보통:

| 거래 유형 | serviceId 예 | 거래코드 예 |
| --- | --- | --- |
| 목록+페이징 | `CM.Campaign.selectList` | CM-INQ-0001 |
| 등록 | `CM.Campaign.create` | CM-CRT-0001 |
| 수정 | `CM.Campaign.update` | CM-UPD-0001 |

**패턴은 같습니다.** Handler → 6계층 → JSON. **body만** 달라집니다.

---

## 23.2 목록 조회 body

```json
"body": {
  "searchCondition": {
    "campaignName": "우대",
    "fromDate": "20260701"
  },
  "page": {
    "pageNo": 1,
    "pageSize": 100
  }
}
```

응답:

```json
"body": {
  "list": [ { "campaignId": "...", "campaignName": "..." } ],
  "page": {
    "pageNo": 1,
    "totalCount": 250,
    "hasNext": true
  }
}
```

SQL은 Mapper XML에 **LIMIT/OFFSET** 또는 DB별 페이징.

---

## 23.3 등록·수정

- **processingType**: `CREATE` / `UPDATE`
- **idempotencyKey** (header): 같은 등록 버튼 **두 번** 눌림 방지
- Facade `@Transactional` — **readOnly = false**
- Rule에서 **필수값·중복** 검사

등록 성공 body 예: `{ "campaignId": "CMP2026070003" }`

---

## 23.4 SV에서 복제하는 법

22장 `selectSummary`를 **복사**해서:

| 새 거래 | 바꿀 것 |
| --- | --- |
| selectDetail | serviceId, SQL, Rule |
| selectList | body.page, list SQL |
| create | INSERT SQL, processingType |

**Handler·Facade·Service·Rule·Dao·Mapper** 파일 구조는 **그대로**.

---

## 23.5 ⚠️ 초보자 실수

| 실수 | |
| --- | --- |
| 목록인데 processingType INQUIRY 안 맞춤 | Catalog·Header **일치** |
| 등록인데 readOnly 트랜잭션 | **쓰기** 트랜잭션 |
| 페이징 없이 전체 SELECT | **성능** 문제 |

---

## 요약

- 목록 = **searchCondition + page**
- 등록/수정 = **processingType + 트랜잭션**
- 22장 패턴 **복제**가 정석
