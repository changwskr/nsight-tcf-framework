## 2.1 큰 그림 — 사용자에서 DB까지

로컬 개발할 때 자주 보는 흐름만 그립니다. (운영의 L4·GSLB는 나중에)

```text
[브라우저 / tcf-ui]
        │  JSON POST
        ▼
[tcf-gateway]  ← 업무코드(sv, om…) 보고 어디로 보낼지 결정 (8100)
        │
        ▼
[sv-service WAR]  ← Context /sv, 포트 8086
        │  POST /sv/online
        ▼
[TCF] STF → Handler → ETF
        │
        ▼
[DB / H2]  ← Mapper SQL
```

**기억할 숫자 (로컬 bootRun):**

| 무엇 | 포트 | 주소 예 |
| --- | --- | --- |
| Gateway | 8100 | 업무로 전달 |
| SV 업무 | 8086 | `http://localhost:8086/sv/online` |
| OM 운영 | 8097 | `http://localhost:8097/om/online` |
| tcf-ui | 8099 | 테스트 화면 |

전체 표는 원본 [부록 K](../ztcfbook/부록/K-모듈-포트-Context-WAR-매핑표.md)에 있습니다.

---

## 2.2 업무 WAR가 여러 개인 이유

NSIGHT에는 **업무별로 WAR 파일**이 나뉩니다. 예: `sv.war`(고객 조회), `mg.war`(메시지), `tcf-om.war`(운영 화면).

| 장점 (쉬운 말) |
| --- |
| SV만 수정했으면 **sv.war만** 다시 배포 |
| 팀별로 **코드·DB 책임**이 나뉨 |
| 모든 WAR가 **같은 TCF 방식**으로 동작 |

지금 코드베이스에 배포되는 업무 WAR는 **9개**입니다: `ic`, `pc`, `ms`, `sv`, `pd`, `eb`, `ep`, `ss`, `mg` + 운영용 **`tcf-om`**.

---

## 2.3 Gateway는 왜 있나요?

클라이언트가 매번 “SV는 8086, OM은 8097…”을 외울 필요 없게 **한 곳(Gateway)** 으로 보냅니다. Gateway는 JSON의 **businessCode**(또는 경로)를 보고 해당 WAR로 **그대로 전달(Relay)** 합니다.

초보 단계에서는 **로컬에서 sv-service에 직접** `POST /sv/online` 해도 TCF 학습에는 충분합니다. Gateway는 **나중에** 붙여도 됩니다.

---

## 2.4 tcf-ui는 뭐예요?

**브라우저에서 표준 JSON을 보내 보는 테스트 도구**입니다. curl 대신 버튼으로 `serviceId`·`body`를 채워 보낼 수 있습니다. SV 고객요약 실습(8편)에서 다시 씁니다.

---

## 2.5 ⚠️ 초보자 실수

| 실수 | 올바른 이해 |
| --- | --- |
| “WAR 하나에 모든 업무” | 업무 **도메인별 WAR** |
| Gateway URL과 WAR URL 혼동 | 학습 시 **WAR 직접 호출** OK |
| tcf-ui = 업무 서버 | tcf-ui는 **클라이언트/릴레이** |

---

## 요약

- **Gateway → 업무 WAR → TCF → DB** 순으로 요청이 갑니다.
- 업무는 **WAR 단위**로 나뉘지만, **호출 방식은 모두 동일** (`POST /업무코드/online`).
- 로컬에서는 **8086/sv/online** 같은 주소만 알아도 첫 실습이 가능합니다.
