# 업무 페이징 처리 (신규 WAR 표준)

NSIGHT **신규 업무 WAR**(EP, SV, EB 등)의 목록 조회는 **업무 Service·Rule·DAO**에서 페이징을 처리합니다.  
TCF Core는 페이징 정책, Page DTO, STF 검증, `TCF_PAGE_POLICY`를 **제공하지 않습니다.**

| 문서 | 대상 | 비고 |
|------|------|------|
| **본 문서** | 신규 업무 WAR 목록 INQUIRY | `list` + `totalPage` 응답 |
| [화면페이징.md](화면페이징.md) | OM Admin **현행** UI·서버 | `rows` + `OmOperationRule.normalizePaging` |
| [DAO처리.md](DAO처리.md) §8 | DAO search/count 쌍 | MyBatis WHERE 공유 |
| [mybatisNaming.md](mybatisNaming.md) | SQL id | `search*`, `count*` |

관련: [어플리케이션계층.md](어플리케이션계층.md) · [applicationNaming.md](applicationNaming.md)

---

## 1. 설계 원칙

| 원칙 | 설명 |
|------|------|
| **TCF 무관** | Dispatcher·Handler는 body를 그대로 Service에 전달. offset 계산·상한은 업무 Rule |
| **서버 주도** | 클라이언트는 `pageNo`·`pageSize`만 전송. **`offset`은 보내지 않음** |
| **DB slice** | `ORDER BY` 고정 후 `OFFSET … ROWS FETCH NEXT … ROWS ONLY` |
| **정확한 total** | `totalCount`는 count 쿼리. `list.size()`로 대체 금지 |
| **다운로드 분리** | 대량 엑셀/CSV는 목록 INQUIRY와 **별도 serviceId** |

### TCF 공통 페이징 설계안 (보류)

`body.page` + `body.searchCondition`, STF 검증, `countYn`, Cursor, `TCF_PAGE_POLICY` 등 **프레임워크 공통화안은 Phase 1~3 보류**입니다.  
당분간 업무팀이 Rule/Service에서 아래 5규칙을 따릅니다.

---

## 2. 업무팀 5규칙

| # | 규칙 |
|---|------|
| 1 | 목록 INQUIRY body에 `pageNo`, `pageSize` 수신 |
| 2 | 기본값: `pageNo = 1`, `pageSize = 100` |
| 3 | 상한: 일반 목록 **500**, 로그·관리 목록 **1000** |
| 4 | DB: `OFFSET #{offset} ROWS FETCH NEXT #{pageSize} ROWS ONLY` (ORDER BY 필수) |
| 5 | 대량 다운로드는 목록 조회와 API·쿼리 분리 |

---

## 3. 계층별 책임

```text
[Client]
  body: { pageNo, pageSize, ...검색조건 }
       │
       ▼ POST /{bc}/online  (serviceId: BC.Domain.inquiry)
[Handler]  — 라우팅만, 페이징 로직 없음
       │
       ▼
[Facade]   @Transactional(readOnly=true)
       │
       ▼
[Service]  criteria 조립 → Rule.normalizePaging → DAO search + count → 응답 Map
       │
[Rule]     validateInquiry, pageNo/pageSize/offset 정규화, 검색조건 trim
       │
[DAO]      searchXxx(criteria), countXxx(criteria)
       │
[Mapper]   <sql id="searchWhere"> 공유, search* / count*
```

| 계층 | 페이징 관련 책임 |
|------|------------------|
| Handler | 없음 |
| Facade | TX 경계 |
| **Rule** | 기본값·상한·`offset` 계산, 검색 필드 검증 |
| **Service** | criteria → DAO 호출 → `list`·`totalPage` 응답 조립 |
| DAO/Mapper | 동일 WHERE로 slice·count |

## 4. 요청·응답 계약

### 4.1 요청 body

```json
{
  "pageNo": 1,
  "pageSize": 20,
  "userId": "U001",
  "eventType": "USER_CREATED"
}
```

| 필드 | 타입 | 필수 | 기본값 | 설명 |
|------|------|------|--------|------|
| `pageNo` | number | N | `1` | **1-based** 페이지 |
| `pageSize` | number | N | `100` | 페이지당 건수 (업무별 상한 적용) |
| 검색 필드 | varies | N | — | 도메인별 (`userId`, `sampleKey` …) |

- 클라이언트는 **`offset`을 보내지 않습니다.**

### 4.2 응답 body (신규 표준)

```json
{
  "businessCode": "EP",
  "serviceId": "EP.UserEvent.inquiry",
  "guid": "...",
  "list": [ { "eventId": "EVT001", "userId": "U001" } ],
  "pageNo": 1,
  "pageSize": 20,
  "totalCount": 153,
  "totalPage": 8
}
```

| 필드 | 설명 |
|------|------|
| `list` | 현재 페이지 행 (`List<Map>`) |
| `pageNo` | 적용된 페이지 (정규화 후) |
| `pageSize` | 적용된 페이지 크기 |
| `totalCount` | 검색 조건 전체 건수 |
| `totalPage` | `ceil(totalCount / pageSize)`, totalCount=0이면 `0` |

### 4.3 OM Admin 레거시와의 차이

| 항목 | OM Admin ([화면페이징.md](화면페이징.md)) | 신규 업무 (본 문서) |
|------|---------------------------------------------|---------------------|
| 목록 키 | `rows` | `list` |
| 기본 pageSize | 20 | 100 |
| 최대 pageSize | 100 | 500 / 1000 |
| totalPage | 없음 (UI가 계산) | **서버가 내려줌** |
| Rule | `OmOperationRule.normalizePaging` | 업무별 Rule |

EB 등 기존 코드가 `rows`를 쓰는 경우는 **레거시**이며, 신규·개편 시 `list`로 맞춥니다.

## 5. Rule — 페이징 정규화

업무 Rule에 `normalizePaging(Map<String, Object> criteria, int maxPageSize)`를 둡니다.

```java
// 의사 코드 — EpUserEventRule / SvSampleRule 참고
int pageNo = max(1, parseInt(body.get("pageNo"), 1));
int pageSize = parseInt(body.get("pageSize"), 100);
if (pageSize < 1) pageSize = 100;
if (pageSize > maxPageSize) pageSize = maxPageSize;
criteria.put("pageNo", pageNo);
criteria.put("pageSize", pageSize);
criteria.put("offset", (pageNo - 1) * pageSize);
```

| 업무 유형 | `maxPageSize` 예 |
|-----------|------------------|
| 일반 마스터·샘플 | 500 (`SV.Sample.inquiry`) |
| 이벤트·로그·관리 | 1000 (`EP.UserEvent.inquiry`) |

## 6. MyBatis 패턴

### 6.1 WHERE 공유

```xml
<sql id="userEventSearchWhere">
  <where>
    <if test="userId != null and userId != ''">
      AND USER_ID LIKE CONCAT('%', #{userId}, '%')
    </if>
    <if test="eventType != null and eventType != ''">
      AND EVENT_TYPE = #{eventType}
    </if>
  </where>
</sql>

<select id="searchReceivedEvents" resultType="map">
  SELECT EVENT_ID AS eventId, USER_ID AS userId, ...
    FROM EP_USER_EVENT
  <include refid="userEventSearchWhere"/>
   ORDER BY RECEIVED_AT DESC
   OFFSET #{offset} ROWS FETCH NEXT #{pageSize} ROWS ONLY
</select>

<select id="countReceivedEvents" resultType="int">
  SELECT COUNT(1) FROM EP_USER_EVENT
  <include refid="userEventSearchWhere"/>
</select>
```

### 6.2 네이밍

| 용도 | DAO / Mapper id |
|------|-----------------|
| 목록 slice | `searchReceivedEvents` |
| 전체 건수 | `countReceivedEvents` |

[mybatisNaming.md](mybatisNaming.md)의 `selectList`는 신규 표준; 기존 OM `inquiry`/`search*` 혼용은 레거시입니다.

## 7. Service 응답 조립

```java
List<Map<String, Object>> list = dao.searchXxx(criteria);
int totalCount = dao.countXxx(criteria);
int pageNo = (int) criteria.get("pageNo");
int pageSize = (int) criteria.get("pageSize");
int totalPage = totalCount == 0 ? 0 : (totalCount + pageSize - 1) / pageSize;

result.put("list", list);
result.put("pageNo", pageNo);
result.put("pageSize", pageSize);
result.put("totalCount", totalCount);
result.put("totalPage", totalPage);
```

## 8. 참조 구현

| 모듈 | serviceId | max pageSize | 소스 |
|------|-----------|--------------|------|
| **ep-service** | `EP.UserEvent.inquiry` | 1000 | `EpUserEventService.inquiry` |

### EP — 수신 이벤트 목록

- 테이블: `EP_USER_EVENT`
- 검색: `userId`, `eventType`, `eventId` (부분 일치)
- 수신: `EP.UserEvent.receive` (페이징 없음, 등록 전용)

### SV — 샘플 목록

- 테이블: `SV_SAMPLE` (로컬 H2 시드)
- 검색: `sampleKey` (부분 일치)
- `SV.Sample.inquiry` body에 `pageNo`·`pageSize` 추가
- **상세 설계:** [sv-service-페이징.md](sv-service-페이징.md)
